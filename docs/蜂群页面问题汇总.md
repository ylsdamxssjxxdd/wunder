# 蜂群页面问题汇总

更新时间：2026-07-12

## 结论

当前问题不是单一组件渲染故障，而是蜂群页把“蜂群任务投影”“普通聊天会话”“全局智能体运行状态”和“本地缓存的派生消息”交叉写入。它们缺少统一的 `group_id / mission_id / session_id / agent_id` 归属校验，导致状态串扰、重复刷新和错误的完成提示。

## P1：动态头像回退默认头像、卡片工具轨迹串到当前聊天智能体

### 已确认的机制

1. 普通头像可以直接解析为图片 URL，但 companion 动态头像的配置不会产生普通图片 URL；旧卡片只渲染 `avatarImageUrl`，因此动态头像会落入默认头像分支。
2. 卡片工具轨迹来自任务会话事件。旧选择规则先取智能体最新任务，后续又可能从另一条有轨迹的任务取内容；同一工蜂连续执行任务时，显示任务 ID、状态与工具条目可能不属于同一个任务。
3. 运行态工具预览若依赖聊天页当前活动会话，就会表现为只有当前打开的智能体卡片有内容；卡片必须按各自任务的 `target/spawned session_id` 独立补水。

### 本轮处理

- 卡片直接解析成员的 companion 头像配置，从 companion store 加载 spritesheet，并以暂停的 idle 首帧渲染静态帧。
- 每个智能体按自己的任务集合选择最新且确有轨迹的任务，工具条目、状态和 `workflowTaskId` 使用同一个任务来源；母蜂继续使用母会话独立轨迹。
- 工蜂任务与子智能体预览均保持按各自会话补水，不再借用聊天页活动会话作为卡片工具来源。

## P0：完成提示会在任务刚发送或母蜂派发工蜂时出现，并在切换聊天页时刷屏

### 现象

- 发送一条消息或母蜂调用工蜂后，尚未得到最终回复就出现“任务完成”。
- 进入聊天页、刷新运行状态或连续处理多个蜂群事件时，完成提示重复出现。

### 原因

1. 蜂群实时事件的 `team_task_result`、`team_finish` 和 `team_error` 会无条件发送 `emitAgentRuntimeRefresh`；没有带上或检查蜂群任务的会话归属。
2. 全局运行状态控制器将任意智能体从 `running/pending` 变为 `idle/done` 都视为应通知的完成事件；它不知道这是蜂群中间任务、普通聊天任务，还是因刷新获得的空闲快照。
3. 蜂群运行时会对目标会话执行后台预加载；当该会话正是聊天页活动会话时还会以 `syncActive: true` 写入活动消息/运行时。刷新后的终态会再次触发全局完成判断。

### 风险

- 通知语义错误，用户无法区分工蜂阶段完成与整个蜂群任务完成。
- 多个工蜂结果会触发多条全局 Toast/系统通知。
- 对当前聊天会话的后台写入会放大提示和状态跳变。

### 修复建议

- 完成通知改为基于明确的用户发起任务标识（`mission_id + root_session_id + user_turn_id`）结算；工蜂任务只更新蜂群投影，不触发全局“智能体任务完成”。
- 首次加载和纯轮询快照不得产生完成通知；只允许同一权威任务从非终态转为终态时通知一次，并持久化去重键。
- 蜂群后台同步禁止 `syncActive: true`，禁止通过 `loadSessionDetail` 切换或覆盖聊天页活动会话。

## P0：聊天页消息顺序错乱、工具调用缺失

### 现象

- 蜂群发送后切到聊天页，用户气泡可能排到更早的历史之前。
- 工具调用或流式过程偶发消失，最终内容与工具轨迹不一致。

### 原因

1. 同一蜂群发送同时走两条写入路径：本地先写 canonical runtime，再通过蜂群 WebSocket 写 canonical runtime；同时又异步预加载会话详情。
2. `syncDispatchSessionToMessenger` 在目标智能体与当前聊天页智能体相同但会话不同的情况下调用 `loadSessionDetail`，这会改变活动会话内容，而不是仅更新缓存。
3. `readDispatchSessionMessages` 会在缓存比活动数组“新”时，用缓存数组替换活动数组；该替换不以当前聊天页渲染的回合、流式事件序号和未完成工具调用为边界。
4. 蜂群侧又把最终回复单独追加为手工消息，随后用会话历史回填。两套消息主键、时间戳来源和折叠规则不同，极易发生重排、覆盖或遗漏。

### 修复建议

- 聊天页只接受 chat runtime 的单一事件流；蜂群页仅订阅该投影，不直接替换 `chatStore.messages` 或活动会话。
- 发送动作只保留一个 canonical 写入口，并以 `client_message_id / user_turn_id / model_turn_id` 幂等合并；禁止在最终事件外手工再追加同一回复。
- 会话回填必须按稳定事件序号和 turn id 合并，不能只按时间、数量或“缓存更新”覆盖活动数组。

## P0：蜂群右侧栏消息混入无关内容，且逐渐卡顿

### 现象

- 右侧栏并非只展示用户与母蜂的主会话消息，出现工蜂请求、子智能体回复和其他派生内容。
- 运行时间越长越卡，滚动和更新明显变慢。

### 原因

1. 右栏将 `manualChatMessages` 与 `runtimeRelayChatMessages` 直接合并。后者来自 `dispatchPreview.subagents`，会主动生成“母蜂 -> 子智能体”和“子智能体 -> 母蜂”两类消息，因此设计上就会混入非主会话内容。
2. 右栏没有最终的 mission/session 过滤。派生消息的父会话可回退到当前 `dispatchSessionId`，当预览或缓存切换滞后时可能归属到错误会话。
3. 子智能体派生消息每次预览变化都构造、签名、合并、排序并写入 sessionStorage；随后 `displayChatMessages` 再次合并和排序。高频实时事件会反复处理全部历史数组。
4. 右栏使用了最多 120 条缓存加多个来源的全量 computed、字符串全量签名和深度内容比较，未做窗口化或增量 reducer。

### 修复建议

- 明确右栏产品语义：默认仅显示当前蜂群母会话的用户/母蜂消息；工蜂通信改为卡片详情或单独的“协作轨迹”面板，默认折叠。
- 每条投影消息必须携带 `group_id`、`mission_id`、`session_id`、`source_kind`，渲染前按当前作用域严格过滤；不允许通过“当前 dispatch 会话”猜测父归属。
- 运行时消息只保留内存中的有界增量窗口；缓存只存主会话恢复必需数据。右栏采用虚拟列表，签名只针对尾部变更。

## P1：首次进入蜂群时默认智能体混入其他蜂群，刷新后恢复

### 现象

- 首次进入或切换蜂群时，默认智能体偶尔出现在不属于它的蜂群卡片中；刷新详情后正常。

### 已确认的机制

1. 画布成员由 `group.members`、异步到达的 `agents`，以及当前 `dispatchPreview.targetAgentId` 三方合并。
2. `dispatchPreview.targetAgentId` 没有验证其是否为当前蜂群成员，任何恢复中的 dispatch 目标都会被直接插入画布成员集合。
3. 蜂群工作台先渲染 `group`，成员详情 `agents` 后到；这段时间画布会以摘要成员和旧 dispatch 缓存渲染。刷新后详情覆盖旧数据，所以问题消失。

### 修复建议

- 画布成员集合以服务端当前 group detail 的成员列表为唯一权威来源；dispatch 只能更新已存在成员状态，不能创建成员节点。
- group 切换时先清除旧的 dispatch preview、子智能体预览和成员投影，再等待目标 group detail 成功返回后渲染画布。
- 服务端实时事件增加并强制校验 `group_id` 与 `mission_id`；前端拒绝任何不匹配当前作用域的 preview/event。

## P1：当前投影边界不清，形成反馈环

蜂群运行时会：接收团队事件 -> 更新 beeroom store -> 刷新工作流/子智能体 -> 预加载聊天会话 -> 改写 chat store -> 触发全局运行状态与通知 -> 再刷新蜂群。该闭环使一个事件可被多个层重复解释。应将权威状态拆分为：

| 状态 | 权威来源 | 消费者 |
| --- | --- | --- |
| 蜂群/任务/成员状态 | 服务端 mission stream | 蜂群画布、卡片 |
| 主聊天回合与工具调用 | chat runtime stream | 聊天页、蜂群右栏只读投影 |
| 工蜂协作轨迹 | mission task/subagent stream | 画布与独立详情 |
| 完成通知 | 已确认的根用户任务终态 | 通知服务 |

## 建议实施顺序

1. 立即停止蜂群对活动聊天会话的写入，并为完成通知加任务归属和去重。
2. 将右栏改为母会话只读投影，移出子智能体消息合并与持久化。
3. 收口聊天发送/回填为单一 canonical runtime reducer，补充乱序、工具调用先后到达、切页回填的回归测试。
4. 收紧画布成员合并规则和 group/mission 事件过滤，修复首次渲染默认智能体串入。
5. 为实时事件建立 `group_id + mission_id + session_id + event_id` 调试追踪，验证断线重连、快速切群和并发工蜂场景。

## 关键代码位置

- `frontend/src/components/beeroom/useBeeroomMissionCanvasRuntime.ts`：蜂群发送、会话预加载、右栏消息合并、团队实时事件。
- `frontend/src/views/messenger/controller/messengerControllerAgentRuntimeSignals.ts`：全局完成通知触发条件。

## 修复审计（2026-07-12）

上一轮修复有效地移除了蜂群运行时中一部分直接的活动会话覆盖和完成刷新调用，但审计确认仍残留三条关键串扰路径：

1. 共享 chat runtime 仍会把所有 `team_task_result`、`team_finish`、`team_error` 广播为全局智能体运行状态刷新；普通聊天页收到团队事件时仍可触发完成提示。
2. 右栏虽不显示工蜂转发消息，但旧实现仍持续构造、排序并写入 sessionStorage，造成无效计算和缓存污染；此外缓存更“新”时仍会回写活动消息数组。
3. 服务端蜂群摘要曾截断成员，前端又会在切群详情加载期间保留上一蜂群投影；两者叠加使默认智能体或旧成员短暂进入错误画布。

本轮已完成：

- 从共享 chat runtime 删除团队任务终态到全局智能体刷新/通知的路径；运行状态快照本身也不再产生用户可见完成通知。
- 蜂群右栏只保留显式 `sessionId` 的母会话投影，移除工蜂转发缓存、合并、持久化和对活动聊天数组的回写。
- 蜂群缓存格式升级为 v3，清理旧的工蜂转发字段；跨会话恢复会丢弃错误归属消息。
- 切换蜂群时立即清空旧详情投影；只有当前蜂群详情作为工作台渲染来源。
- 后端列表接口兼容保留最多 6 个成员摘要，当前蜂群详情返回完整成员；前端将详情中的成员数组（包括空数组）作为权威集合，过滤卡片和活动成员。

验证：前端类型检查、生产构建、主会话隔离回归、画布成员/工具投影回归与 runtime release 检查均通过。仓库统一回归入口仍被其他聊天测试夹具的既有 TypeScript 类型错误阻断；本次涉及的蜂群用例已绕过该入口独立打包执行并全部通过。

## 右栏跨蜂群串组、第二轮倒插与额外气泡补充审计（2026-07-12）

### 新确认根因

1. 蜂群工作台曾把普通聊天页最后打开的前台会话传给蜂群画布，右栏因此直接发送到无关智能体或无关蜂群的会话。
2. 即使去掉前台会话注入，普通蜂群仍通过母蜂唯一的主会话解析右栏历史；同一母蜂被多个蜂群复用时，各蜂群天然读取同一份消息。
3. 中栏切组调用绕过了 store 的清理入口；同一次作用域变化又被多个 watcher 与 mounted 钩子重复初始化，旧详情和新详情存在交错窗口。
4. 发送结束和恢复结束的 `finally` 读取可变的当前 `dispatchSessionId`，旧组请求结束时可能补水新组会话；流事件也只按 session 处理，没有同时校验发送时的 group。
5. 蜂群发送曾同时创建本地用户气泡、伪造助手轮次、追加最终助手气泡并随后加载服务端历史，产生额外气泡。排序又优先比较不同来源时间戳，第二轮可倒插到第一轮前。

### 已完成根治

- 后端新增蜂群专属母蜂会话绑定，稳定维护 `user_id + group_id + mother_agent_id -> session_id`；同组重复解析幂等，不同蜂群即使共用母蜂也使用不同会话，发送前再将该会话设为母蜂主线程。
- 蜂群摘要/详情返回 `mother_session_id`，前端发送前通过蜂群接口解析权威会话，不再借用聊天页活动会话或猜测最近会话。
- 切组统一走 `selectGroup` 并立即清空旧详情；画布作用域只保留一个初始化 watcher，迟到流事件和补水同时校验发送时的 `group_id + session_id`。
- 发送只创建 canonical optimistic user，并让服务端事件/历史拥有助手内容；删除手工最终回复和伪造助手身份。
- reducer 收到服务端 `user_round` 后把 optimistic turn 提升为 canonical round；右栏按 canonical turn/message order 排序，同轮顺序不再受反向时间戳影响。

### 回归覆盖

- 后端：同组母蜂会话稳定复用、不同蜂群会话隔离。
- 前端：迟到派发事件不能跨 group/session、canonical 轮次覆盖反向时间戳、历史补水不保留其他 session 消息、最终助手去重。
- 聊天 reducer：仅提供 `client_message_id` 时，服务端轮次仍能接管 optimistic turn，并只生成一个助手气泡。

## 真实 Docker 压测复现与补充修复（2026-07-12）

### 复现结果

- 使用隔离测试账号、两个独立蜂群和 Docker 中实际运行的 PostgreSQL/服务端/静态前端执行浏览器用例。
- 每轮覆盖：Alpha 蜂群连续发送两轮；第二轮期间切到 Beta 蜂群发送一轮；随后快速往返切换两组并等待补水收敛。
- 修复前同一场景连续执行 5 次，有 2 次在 Beta 右栏复现同一条用户消息的两个气泡：一个 key 来自 history，另一个 key 来自 runtime 投影。

### 补充根因与处理

1. 切组期间，本地 optimistic 用户轮次仍处于待确认状态；会话详情补水已返回同一轮的 canonical `user_turn_id`，但两者的 message key 分别是本地 key 和 `history:<id>`，runtime projection 将它们视为两个用户消息。
2. 右栏收到 history 与 runtime 两条同轮记录时只按 key 合并，无法收敛这两种服务端 key。
3. runtime snapshot 现在会将唯一匹配的待确认本地用户轮次提升并合并到 canonical `user_turn_id`；匹配同时要求内容一致（兼容显示用 `@目标` 前缀）且不存在歧义。
4. 蜂群右栏会话合并再按 `user_turn_id` 去重，重复来源中优先保留可持久化的 history key，避免 Vue 重新挂载造成视觉跳动。

### 验证

- 新增 runtime 和蜂群会话合并单元回归，覆盖同一用户轮次的 history/runtime 双 key 情况。
- Docker 静态前端重新构建后，真实浏览器用例单次通过，随后 `--repeat-each=5` 连续 5 次全部通过；未再出现跨组消息、轮次倒置或额外用户气泡。

## 蜂群工具状态变红与母蜂/工蜂运行态拖动审计（2026-07-13）

### 真实事件结论

- 核对 PostgreSQL 持久事件后，用户提供的 `team_task_dispatch` 事件是正常的排队快照，不是失败结果。
- 对应的 team run 最终为 `completed`，task 最终为 `success`，工蜂 session 有完整的 `final -> turn_terminal(completed) -> thread_status(idle/not_loaded)` 链路。
- `queued` 只表示已成功受理并等待执行；前端将其显示为红色属于投影错误。

### 根因

1. 持久化 stream event 可能有两层 transport `data` 包装；旧 canonical adapter 只解一层，导致真实 `status/tool_call_id/team_run_id/task_id` 丢失。
2. `team_*` 事件为关联会携带发起蜂群的 `tool_call_id`；旧 reducer 将该关联字段当成工具行主键，使 task 的 queued/running/failed 反向覆盖 `agent_swarm` 工具本身。
3. 工蜂主线程可复用，其 history 中会有上一次调用的 `final/turn_terminal`；仅从尾部搜索任意终态，会让旧完成影子覆盖当前 queued/running task。
4. session runtime 的 `idle/not_loaded/completed` 轮询快照会无条件压过消息上仍在 `loading` 的阻塞式 `agent_swarm` 工具，继而将母蜂/工蜂标记为完成、失败或中止。
5. `team_error` 是协作层工具结果，不是母蜂整个 turn 的终态；旧 reducer 却直接将 session 置为 failed。

### 已完成根治

- canonical adapter 只递归解包 transport envelope，保留真实工具结果内的业务 `data`。
- team run 以 `team_run_id` 建立工作流身份，team task 以 `task_id` 建立身份；`tool_call_id` 仅作相关性信息，不再合并或改色工具行。
- 画布仅把真实 `tool_*` 事件或带工具名的项目识别为工具卡，不再把 `team_*` 上的相关 id 当工具。
- dispatch 预览和工蜂 shadow 状态会优先当前 task 的 active 快照；较旧终态只能结算旧任务。

## 2026-07-13 新主线程与运行态轮询根治

### 现象

- 母蜂在普通聊天页新建对话后，从蜂群右栏发送仍落到旧蜂群会话，用户轮次继续累计而非从第 1 轮开始。
- 母蜂调用智能体蜂群工具时，母蜂和工蜂会被 `/agents/running` 的轮询结果反复刷成“已完成”；打开聊天页补水后短暂正常，下一次轮询又回退。

### 根因

- 蜂群母会话只按 `user + hive + mother agent` 保存永久绑定，`mother-session` 接口随后还会把该旧会话反向设为母蜂主线程，覆盖用户刚创建的新主线程。
- `/agents/running` 只聚合 session lock、monitor 和普通 agent task，没有读取阻塞式蜂群的 `team_runs/team_tasks`；模型动作与工具动作交界处的短暂终态因此被误当成整个蜂群已结束。
- 原蜂群聚合回归脚本先用 CommonJS `tsc` 编译整个 regression 目录，会因无关类型错误与 `import.meta` 失败，导致蜂群测试本身未执行，掩盖了工蜂新旧终态比较缺陷。

### 修复

- 非活动编排下，蜂群会采用用户显式选择的新母蜂主线程并更新当前蜂群绑定；采用前检查该会话是否属于另一蜂群。活动编排继续冻结原 `mother_session_id`，运行中不换线。
- 后端按当前用户查询 `queued/running/merging` 团队运行，将母蜂和活跃工蜂任务作为 `running` 权威证据，优先于短暂 `done/error`；PostgreSQL 与 SQLite 同步增加用户状态查询和组合索引。
- 工蜂历史终态与当前任务状态按更新时间比较：新终态可收敛旧运行，新活动也能挡住旧终态。
- 蜂群回归改为逐用例 esbuild 打包执行，并为 Vite 头像资源 glob 提供测试替身，避免再次出现“脚本启动失败但误以为用例已覆盖”。

### 验证

- Rust：母蜂新主线程接管/跨蜂群隔离 3 项、团队运行态投影 2 项、SQLite 存储回归 1 项通过；`cargo check -p wunder-runtime -j 8` 通过。
- 前端：蜂群调度 51 项、蜂群实时 16 项、聊天 reducer 104 项、busy state 15 项、agent runtime 7 项通过；`typecheck`、`build:check` 通过。
- Docker PostgreSQL 实测：新建母蜂主线程后 `mother-session` 返回同一新会话；活跃母蜂/工蜂连续 3 次 `/agents/running` 轮询均为 `running`；真实浏览器快速多轮及跨蜂群隔离 E2E 通过。

## 2026-07-13 发送气泡延迟、切页空白与整体重置

- 发送气泡延迟：本地 runtime 已产生 session-scoped optimistic 用户消息，但紧随其后的 history 补水尚为空；旧逻辑用空 history 强制替换本地投影，直到最终回复持久化后气泡才重新出现。
- 切页空白：右栏缓存此前只保存 queued/running 等活动状态，完成会话在组件卸载时会清空 session 与消息；重新进入页面虽有权威母蜂绑定，却只读瞬时缓存且不做 durable history 补水。
- 修复：history 合流保留尚未被 durable 用户轮次确认的 session-scoped optimistic 气泡；完成会话也缓存当前母蜂绑定和消息；页面重新激活时主动补水当前会话。
- 行为调整：移除只清理右栏派生态的“清空”按钮，替换为“重置蜂群”。服务端按蜂群停止活动任务、关闭编排、清除任务/右栏投影，并为所有蜂群成员创建全新主线程。

## 2026-07-13 增量投影消失、切页假中止与工蜂误失败

### 日志复盘

- 右栏在助手首个增量到达时会收到只包含助手的部分 runtime 快照；旧的强制刷新直接替换本地消息数组，因此已显示的用户消息短暂消失，终态 history 补水后才重新出现。
- 页面卸载时即使已判断为保留运行任务，仍无条件中止蜂群发送流；本地 AbortError 随后被当作用户中止投影，重新进入时出现“已中止”，但后端任务实际仍在执行。
- 工蜂会话的连续模型动作可能先出现可恢复的 error、后续动作再完成；卡片把任意早期 error 聚合为工蜂失败，并且 `not_loaded` 被当成线程终态，造成运行中或已成功工蜂被误显示为完成/失败。

### 根治

- 所有右栏 session 补水（包括强制刷新）统一经过按稳定消息/用户轮次/模型轮次合流的 reducer；部分快照只补充或更新已有转录，不能删除同会话已经观察到的消息。
- 页面切走时保留在途蜂群流，不再调用本地 abort；用户点击停止仍走明确的 stop/cancel 路径，因此真实停止继续显示“已中止”。
- `not_loaded` 改为“本地尚未加载”而非终态；智能体状态等待服务端明确 `completed/failed/cancelled` 或当前阻塞蜂群结束后再收敛。
- 工蜂卡片在无活动任务时以最新任务时间的终态为准，后续完成可覆盖同一会话内更早的重试错误；最新任务真实失败仍保持红色。

### 回归

## 2026-07-13 右栏用户消息错序、重复与画布状态抖动

### 日志复盘

- 同一轮中，母蜂回复已经取得服务端 `turnOrder`，本地乐观用户消息仍处于未取得轮次号的过渡状态；旧排序器无条件把有权威轮次的消息放在前面，因此回复会短暂跑到用户消息上方。
- 同一次提交会暂时以两个 runtime message key 出现。两条记录的时间、正文和 `client_message_id` 相同，但其中一条可能尚未取得 `user_turn_id`；旧去重只认用户轮次，因而渲染出两个相同用户气泡。
- mission 的 `updated_time` 会随任务派发、状态推进和工蜂完成持续变化。旧 watcher 把它视为画布切换并重播全部节点入场动画；同时节点槽位按实时状态和任务数排序，状态变化还会交换工蜂坐标。

### 根治

- 右栏消息模型贯通 `client_message_id`，同一提交优先按 `user_turn_id`、其次按 `client_message_id` 合并；兼容缺少两种身份的短暂快照时，只在同会话、同时间、同正文和同发送者完全一致时做保守去重。
- 排序只在两侧都具备权威轮次时使用 `turnOrder/messageOrder`；乐观消息和权威消息混合期间按时间及用户优先规则稳定排序，待身份补齐后再自然收敛到服务端轮次。
- 画布重播作用域只包含页面激活、蜂群、mission 和 team run 身份，不再包含状态或更新时间；已有工蜂从空闲进入运行时只更新卡片内容，不重播节点入场动画。
- 工蜂蜂窝槽位改为稳定成员顺序，保留母蜂和入口智能体优先级；任务状态、任务数量和完成时间变化不再改变节点坐标。

### 回归

- 新增“乐观用户消息与权威母蜂回复混排仍保持用户在前”“相同 `client_message_id` 双 key 只保留一个气泡”的消息回归。
- 新增“状态更新时间不进入重播作用域”“工蜂任务状态互换前后坐标完全一致”的画布回归。
- 蜂群派发、蜂群实时、聊天 reducer、busy state、类型检查和构建检查均通过。
