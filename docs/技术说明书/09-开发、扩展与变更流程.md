# 开发、扩展与变更流程

## 1. 开发原则

- **先判职责再选目录** — 按职责边界改动，不按调用方便跨层写。
- **超大文件不堆功能** — 超过 2000 行的文件视为维护态，新增能力拆新文件。
- **新能力优先对齐 eva** — 宿主无关语义放 `crates/eva`，Wunder 适配放 `crates/eva/src/wunder`。
- **关键逻辑英文注释** — 逻辑强的地方要有英文注释，方便后期维护。
- **文档同步** — 结构变化及时更新 `docs/API`、`docs/设计文档` 和本说明书。

## 2. 通用开发步骤

1. 判断能力属于哪一层（接入 / 运行时 / 投影 / 业务服务 / 前端）。
2. 选择正确目录，不跨层。
3. 先实现稳定最小闭环。
4. 补充回归或定向验证。
5. 用脚本更新 `docs/功能迭代.md`：
   ```bash
   python scripts/update_feature_log.py --type <类型> --scope <范围> ...
   ```
   类型仅限：新增/变更/修复/性能/文档/重构/安全/工程/测试/移除/弃用。

## 3. 新增 API

```
src/api/*           ← 路由文件
src/services/*      ← 业务逻辑
src/api/mod.rs      ← 挂载路由
docs/API/*          ← 更新文档
```

不要：在路由文件里堆业务逻辑；改接口不更新 API 文档。

## 4. 新增工具

```
src/services/tools/*.rs     ← 独立文件实现
crates/eva/src/wunder/tooling/ ← 接入目录与 surface builder
docs/API/*                  ← 更新文档
```

必须明确：审批策略、沙盒策略、超时、并行、错误模型。

不要：往旧 `src/services/tools.rs` 增量堆积。

## 5. 新增运行时语义

宿主无关的 thread / turn / session / prompt / context / tools / recovery 语义优先放 `crates/eva/src/*`。如果只是 Wunder 适配，放 `crates/eva/src/wunder/*`。

不要：在 `src/orchestrator` 新造运行时状态机。

## 6. 新增实时投影

```
src/services/runtime/*       ← 产出内部事件
src/services/projection/*    ← 构建 snapshot / delta / replay 视图
src/api/*                    ← WS / API 接线
frontend/src/realtime/       ← 前端消费
```

不要：在单个 WS 路由里私造状态机。

## 7. 新增渠道

```
src/channels/            ← 独立适配文件
                         ← 接入注册表与服务层
                         ← inbound / outbound / 媒体 / 重试 / 限流
config/wunder.yaml       ← 渠道配置
```

## 8. 新增前端能力

**用户侧**：
- 页面编排 → `frontend/src/views/messenger/`
- 组件 → `frontend/src/components/messenger/`
- 实时逻辑 → `frontend/src/realtime/`
- 状态 → `frontend/src/stores/`

不要：继续往 `MessengerView.vue` 堆功能。

**管理员侧**：
- 模块 → `web/modules/*`

不要：在 `web/app.js` 写大段耦合逻辑。

## 9. 校验命令

Rust 编译检查：
```bash
cargo fmt --all
cargo check -j 8
```

必要时：
```bash
cargo clippy -j 8
```

前端检查：
```bash
npm run typecheck --workspace wunder-frontend
npm run build --workspace wunder-frontend
```

## 10. 文档更新要求

| 变化类型 | 需要更新 |
| --- | --- |
| 功能变更 | `docs/功能迭代.md`（通过脚本） |
| 结构/重要接口变化 | `docs/API/` |
| 系统边界/核心设计变化 | `docs/设计文档/` |
| 使用说明书/图示 | 更新 `docs/使用说明书/`（含 `assets/`）后执行 `python scripts/build_docs_site.py` |
