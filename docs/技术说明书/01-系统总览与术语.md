# 系统总览与术语

## 1. 系统定位

wunder 是一个面向组织与个人的智能体调度系统。三种运行形态共享同一套内核：

| 形态 | 入口 | 场景 |
| --- | --- | --- |
| `wunder-server` | 服务端 | 多租户、多用户、多渠道、组织级部署 |
| `wunder-desktop` | 桌面应用 | 个人用户主推形态，本地即用 |
| `wunder-cli` | 命令行 | 脚本化、批处理、自动化流水线 |

系统的核心不是聊天，而是**可持续执行的智能体运行时**。所有入口最终都落到统一的线程运行时、工具治理、投影发布和存储抽象上。

## 2. 核心能力

| 能力 | 职责 | 主要代码 |
| --- | --- | --- |
| 智能体运行时 | 线程生命周期、turn 执行、prompt 冻结、上下文管理、工具治理、恢复链 | `src/orchestrator` `src/services/runtime` |
| 实时投影 | 面向前端和外部观察者的状态发布层 | `src/services/presence` `src/services/runtime` `src/services/beeroom_realtime` |
| 蜂群协作 | hive / mission / subagent 驱动的多智能体协作 | `src/services/swarm` `src/services/hive_pack` |
| 工具化能力 | 文件、命令、浏览器、MCP、技能、A2A | `src/services/tools` `src/orchestrator/tool_exec` |
| 多入口接入 | HTTP、WebSocket、gateway、channels、desktop、CLI | `src/api` `src/gateway` `src/channels` |

## 3. 对象模型

wunder 围绕以下核心对象构建：

```
用户 (User)
 ├── Agent 预设 × N           — 角色、提示词、工具面、模型设置
 │    └── Thread × N          — 单智能体连续执行上下文
 │         ├── 用户轮次 × N   — 一次用户输入到最终回复
 │         │    └── 模型轮次 × N — 一次模型 API 调用
 │         └── Checkpoint × N — 可恢复的执行截面
 │
 ├── Hive × N                 — 业务协作单元
 │    └── Mission × N         — 多智能体协作任务实例
 │         └── Team Run       — 协作执行过程
 │
 ├── Workspace                — 智能体读写的文件与目录
 ├── Memory                   — 跨线程的长期记忆
 └── Knowledge                — 检索型知识资料
```

## 4. 术语表

以下术语在技术文档中使用时有严格含义，不应混用：

| 术语 | 定义 | 常见错误理解 |
| --- | --- | --- |
| **Thread** | 单智能体的连续执行上下文，跨越多轮对话 | ≠ session id，≠ 数据库行 |
| **用户轮次** (User Round) | 用户发来一次输入，到智能体产出最终回复的全过程 | ≠ 一条消息，≠ 一次模型调用 |
| **模型轮次** (Model Round) | 模型一次 API 调用 + 输出解析 | 一轮用户轮次可包含多轮模型轮次 |
| **Turn** | 当代码中使用时，通常指一轮用户轮次 | ≠ 单条消息 |
| **TurnState** | orchestrator 中用户轮次的执行状态机 | 包含 pending / running / completed / failed |
| **Tool Surface** | 当前线程当前模型可见的工具集合 | ≠ 系统全部工具 |
| **Beeroom** | 协作域的实时房间附着点 | ≠ 泛业务容器 |
| **Mission** | 多智能体协作任务实例 | ≠ 单线程本身 |
| **Orchestrator** | wunder 的智能体执行调度引擎，管理 turn / prompt / context / tools 全链路 | ≠ 简单路由器 |

## 5. 硬约束

技术实现必须长期遵守：

1. **Prompt 冻结** — 线程首次确定 system prompt 后冻结，后续用户轮次不得改写。
2. **记忆单次注入** — 长期记忆只在线程初始化时注入一次，后续通过工具检索。
3. **投影单向消费** — 前端和外部观察者只能消费投影，不能反向篡改后端运行态。

## 6. 产品边界

- `server / desktop / cli` 是同一系统的三种运行形态，不是三套独立产品。
- 前端页面是投影消费层，不是运行时真相源。
- 技能是能力组织方式，不是内核原语。
- MCP 是外部资源接入机制，不是独立运行时。
