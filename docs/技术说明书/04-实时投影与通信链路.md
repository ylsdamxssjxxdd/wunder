# 实时投影与通信链路

## 1. 核心原则

wunder 的实时系统遵循一条铁律：**内部运行态先变成公开事件，再由前端消费**。前端、桌面、CLI 或外部观察者都只能通过公开事件流获取运行态，不能直接读后端内部状态。

| `frontend/src/stores/` | 前端状态存储 |

## 3. 事件模型

所有实时目标遵循统一事件模型：

```
Snapshot → 首次接入时获取完整状态视图
Delta    → 运行时变化时发送增量事件
Replay   → 断线后补齐未消费事件
```

`stream_events` 表持久化事件流，支持断线后 replay。

## 4. Presence 与 Watch

**Presence** — 记录哪些用户在线、哪些连接存在。Presence 是进程内观测状态，可丢失可重建，不是 durable business state。

**Watch** — 某个连接正在观察哪个目标。用于控制订阅、计算活跃观察者数量。

## 5. 前端消费流程

```
1. 路由确定观察目标
2. 本地缓存恢复已有视图
3. 首包获取 snapshot
4. WS 发起 watch
5. 接收 snapshot 或承接后续 delta
6. 断线 → replay → 补水 → 重同步
```

前端允许短暂 optimistic 更新，但最终必须服从后端事件回压。

## 6. 会话类实时链路

负责：当前会话消息流、渠道消息即时渲染、线程运行态、waiting/approval/user_input 控制态、resume 后状态收敛。

常见问题：

| 问题 | 原因 |
| --- | --- |
| 控制面事件被当成消息事件 | 事件类型过滤缺失 |
| 会话列表与正文不同步 | 两路消费时序未协调 |
| 冷会话恢复时详情和列表互相回压 | 缺少补水逻辑 |

## 7. 协作类实时链路

负责：mission 公开面板、beeroom 公开摘要、team run 状态和 agent activity。

beeroom 是实时房间附着点，不应承担领域聚合逻辑。领域逻辑在 swarm 层完成，beeroom 只做实时分发。

## 8. 通信原则

- **WebSocket 为主**，SSE 为兜底。
- 页面卸载只取消当前 watch，**不关闭全局共享连接**。
- 工具页、设置页、工作目录默认走补热链路，不抢占消息热路径。

## 9. 排查矩阵

| 现象 | 检查点 |
| --- | --- |
| 页面实时卡住 | 连接在线？watch 仍有效？ |
| 重连后列表不一致 | replay 命中？stream_events 是否连续？ |
| 活跃会话正文不刷新 | 当前 target 是否指向最新 session？ |
| 设置页切换变慢 | 是否重复请求同一份数据？ |
| beeroom 信息混乱 | 是否把 swarm 逻辑堆进了 beeroom 层？ |
