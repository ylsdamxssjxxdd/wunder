# 仓库结构与模块职责

## 1. 顶层目录

```
wunder/
├── crates/eva/       智能体运行时 crate（宿主无关内核 + Wunder 宿主适配）
├── src/              Server 主 crate（API、渠道、存储、配置、观测、业务服务）
├── frontend/         用户侧前端（Vue 3）
├── web/              管理员侧前端（原生 HTML + JS 模块）
├── wunder-cli/       CLI 入口（含 TUI）
├── desktop/tauri/    Tauri 桌面形态
├── desktop/electron/ Electron 打包壳
├── config/           配置样例、i18n、字体、MCP 配置
├── docs/             设计文档、技术说明书、API 文档、方案
├── scripts/          仓库脚本
├── tests/            后端集成/回归测试
├── extra_mcp/        额外 MCP 运行时与工具脚本
└── packaging/        构建/分发附加资源（docker、python、windows）
```

## 2. Rust 后端分层

### 2.1 接入层

把外部请求接入内部运行时。只做鉴权、参数校验、协议适配和接线。

| 目录 | 职责 |
| --- | --- |
| `src/api/` | HTTP / WS 路由、请求编排、payload shaping |
| `src/gateway/` | Gateway 协议、节点调用、管理面消息 |
| `src/channels/` | 外部渠道适配（飞书、微信、QQ 等）、消息收发 |

### 2.2 运行时层

智能体执行的核心引擎。

| 目录 | 职责 |
| --- | --- |
| `crates/eva/` | 宿主无关的运行时内核（thread / session / turn / prompt / context / tools / recovery） |
| `src/agent_kernel/` | 旧内核承接层，与 eva 类型对齐，逐步退出 |
| `src/orchestrator/` | 旧执行主链与兼容逻辑，只维护未迁移部分 |
| `src/core/` | Observed runtime 桥接层 + 配置 + 鉴权 + 基础能力 |

### 2.3 业务服务层

领域逻辑和编排。

| 目录 | 职责 |
| --- | --- |
| `src/services/swarm/` | 协作任务、team run、聚合汇总 |
| `src/services/hive/` | hive 领域和公开摘要 |
| `src/services/projection/` | session / beeroom / user_world 投影 |
| `src/services/runtime/` | thread / mission 运行态与事件 |
| `src/services/tools/` | 工具实现（文件、命令、浏览器等） |
| `src/services/memory*` | 长期记忆与自动提取 |
| `src/services/knowledge.rs` | 知识解析与检索 |
| `src/services/workspace.rs` | 工作区与文件现实 |

### 2.4 基础设施层

| 目录 | 职责 |
| --- | --- |
| `src/storage/` | PostgreSQL / SQLite 统一抽象与实现 |
| `src/sandbox/` | 沙盒服务 |
| `src/ops/` | 监控、性能、吞吐、基准测试 |
| `src/lsp/` | LSP 相关入口 |

## 3. eva crate 内部结构

```
crates/eva/src/
├── core/        内核 schema 与公共契约
├── engine/      EvaEngine 入口、runtime wiring、ports 注入
├── host/        宿主端口 trait（model / tooling / approval / store 等）
├── thread/      线程快照、注册表、watch、resume、execution lease
├── session/     会话运行时、checkpoint、waiting state
├── turn/        用户轮次执行器、生命周期阶段、事件、cursor
├── prompt/      prompt 组装（base / developer / user / agents / memory）
├── context/     上下文历史、压缩、截断、预算
├── tools/       工具 spec / registry / router / batch / approvals / sandbox / retry / ledger
├── recovery/    崩溃恢复、中断恢复、溢出恢复、工具结果缺失
├── model/       模型网关抽象
├── protocol/    通用协议定义
├── store/       持久化抽象
└── wunder/      Wunder 宿主适配层
     ├── adapter/      prompt / tooling / turn 适配器
     ├── prompt/       Wunder prompt 组装、记忆提供者、模板
     ├── tooling/      工具目录、动态工具、legacy/observed 桥接
     ├── runtime_api/  Wunder 运行时 API 入口
     └── store/        Wunder 存储适配
```

## 5. 常见改动落点

| 要做的事 | 放在哪里 | 不要放 |
| --- | --- | --- |
| 新增 API | `src/api/*` + `src/services/*` | 在路由里堆业务逻辑 |
| 新增工具实现 | `src/services/tools/*.rs` | 往旧 `tools.rs` 堆 |
| 新增运行时语义 | `crates/eva/src/*` | 在 `src/orchestrator` 新造 |
| 新增 Wunder 适配 | `crates/eva/src/wunder/*` | 在 API 层手工拼 |
| 新增投影 | `src/services/projection/*` + `src/services/runtime/*` | 在 WS 路由里私造状态机 |
| 新增前端页面能力 | `frontend/src/views/messenger/` + `frontend/src/components/messenger/` | 往 `MessengerView.vue` 堆 |
| 新增管理页能力 | `web/modules/*` | 在 `web/app.js` 写耦合逻辑 |

## 6. 改代码前的判断顺序

1. 这是**接入层**问题、**运行时**问题、**投影**问题还是**业务服务**问题？
2. 这段逻辑是**宿主无关语义**还是 **Wunder 产品适配**？
3. 操作的是 **durable state**、**projection state** 还是**进程内临时状态**？
4. 改动是否需要同步更新设计文档、API 文档或本说明书？
