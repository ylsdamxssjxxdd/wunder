# 运维、排障与验收

## 1. 运维目标

运维的目标不是"服务能启动"，而是：

- 线程、mission、事件流和工具链路**可观测**
- 异常时能快速判断属于**哪一层**
- 部署、升级、配置变更和回归验证有**统一口径**

## 2. 发布边界

当前默认发布场景：**一人一会话、各自分散聊天**。

验收围绕消息主链、连接恢复、会话列表/正文同步、设置页补热。shared hotspot 属于后续容量专项，不阻塞当前发布。

## 3. 观测对象

| 对象 | 关注点 |
| --- | --- |
| Thread Runtime | 当前状态、waiting、resume、失败原因 |
| Mission Runtime | 任务推进、agent activity |
| Stream Events | 事件连续性、replay 覆盖范围、断线恢复 |
| Tool Runtime | 审批等待、超时、失败、重试、日志 |
| Gateway / Channels | 连接状态、路由、认证、限流 |
| Storage | 数据库可用性、读写错误 |

## 4. 日志与监控入口

### 后端

| 位置 | 用途 |
| --- | --- |
| `src/ops/` | monitor、performance、throughput、benchmark |
| `src/services/stream_events.rs` | 事件流持久化与回放 |
| `src/orchestrator/event_stream.rs` | 事件流产出 |

启动日志会打印：配置加载路径、server mode、bind 地址、app state 初始化耗时。

### 管理端

| 位置 | 用途 |
| --- | --- |
| `web/modules/monitor.js` | 监控面板（系统指标、token 趋势、工具热力图、用户管理） |
| `web/modules/performance.js` | 性能面板（prompt build、file ops、command exec 等） |
| `web/modules/throughput.js` | 吞吐面板 |
| `web/modules/debug.js` | 调试面板（单次请求全流程跟踪） |

### API 端点

| 端点 | 用途 |
| --- | --- |
| `/wunder/admin/gateway/status` | Gateway 状态与指标 |
| `/wunder/admin/throughput/status` | 吞吐指标 |
| `/wunder/admin/channel/runtime/probe` | 渠道运行时探测 |

## 5. 排查矩阵

| 现象 | 第一检查点 | 第二检查点 |
| --- | --- | --- |
| 服务起不来 | 配置文件路径、端口占用 | `WUNDER_HOST` / `WUNDER_PORT` |
| 前端打不开数据 | 后端 `18000` 是否可达 | 前端 dev 代理是否正确 |
| 消息流中断 | WS 是否断开 | stream_events 连续性 |
| 工具不执行 | 工具面是否可见 | 审批、沙盒、超时策略 |
| 恢复失败 | session 状态完整性 | stream_events 断档 |
| 渠道消息不进来 | 渠道配置与鉴权 | inbound_queue / outbox / rate_limit |
| MCP 不可用 | endpoint、headers、enabled | tool_specs 获取与超时 |
| 首轮 prompt 漂移 | prompt.rs 是否重复生成 | 配置与 agent 预设 |
| 工具结果丢失 | tool_logs 状态 | stream persist |
| 线程一直 waiting | turn_state 字段 | resume 调用匹配 |

## 7. 验收维度

### 7.1 功能验收

- 单智能体请求能稳定进入线程并完成一轮执行
- 多智能体任务能形成 team run 并完成公开摘要输出
- 工具、技能、MCP 至少有一条闭环路径可用

### 7.2 运行验收

- server、desktop、cli 三种形态都能启动
- 配置文件和环境变量覆盖关系清楚
- WebSocket 主链正常，断线后可恢复

### 7.3 稳定性验收

- waiting / resume / crash recovery 可工作
- 前端路由切换不出现重复订阅和状态回退
- 慢客户端不会拖垮事件分发

### 7.4 文档验收

- 本目录、`docs/设计文档`、`docs/API文档.md` 与实现事实一致
- 变更已记录到 `docs/功能迭代.md`
