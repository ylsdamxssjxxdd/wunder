# 运维、排障与验收

## 1. 运维目标

运维的目标不是"服务能启动"，而是：

- 线程、mission、投影和工具链路**可观测**
- 异常时能快速判断属于**哪一层**
- 部署、升级、配置变更和回归验证有**统一口径**

## 2. 发布边界

当前默认发布场景：**一人一会话、各自分散聊天**。

验收围绕消息主链、watcher 恢复、会话列表/正文同步、设置页补热。`shared-session / shared hotspot` 属于后续容量专项，不阻塞当前发布。

## 3. 观测对象

| 对象 | 关注点 |
| --- | --- |
| Thread Runtime | 当前状态、waiting、resume、失败原因 |
| Mission Runtime | 任务推进、agent activity |
| Projection | watch 数、replay、sync_required、慢客户端 |
| Tool Runtime | 审批等待、超时、失败、重试、ledger |
| Gateway / Channels | 连接状态、路由、认证、限流 |
| Storage | 数据库可用性、读写错误 |

## 4. 日志与监控入口

| 位置 | 用途 |

| 端点 | 用途 |
| --- | --- |

## 5. 排查矩阵

| 现象 | 第一检查点 | 第二检查点 |
| --- | --- | --- |
| 服务起不来 | 配置文件路径、端口占用 | `WUNDER_HOST` / `WUNDER_PORT` |
| 前端打不开数据 | 后端 `18000` 是否可达 | 前端 dev 代理是否正确 |
| 消息流中断 | WS 是否断开 | watch / replay / sync_required 状态 |
| 工具不执行 | tool surface 是否可见 | 审批、沙盒、超时策略 |
| 恢复失败 | checkpoint / thread snapshot | waiting_state / request_id 匹配 |
| 渠道消息不进来 | 渠道配置与鉴权 | inbound_queue / outbox / rate_limit |
| MCP 不可用 | endpoint、headers、enabled | tool_specs 获取与超时 |
| 首轮 prompt 漂移 | frozen prompt 是否重复生成 | host adapter 是否错误走初始化 |
| 工具结果丢失 | tool ledger / call 状态 | recovery 是否标记 discarded |
| 线程一直 waiting | waiting_state 与 request_id | resume 调用是否匹配当前 call |
| 恢复后状态错乱 | checkpoint 内容 | thread snapshot 与 compat 映射 |

## 6. 校验命令

Rust：
```bash
cargo fmt --all
cargo check -j 8
# 必要时
cargo clippy -j 8
```

前端：
```bash
npm run typecheck --workspace wunder-frontend
npm run build --workspace wunder-frontend
npm run test:messenger-projection --workspace wunder-frontend
```

实时投影 smoke 验收：
```bash
python scripts/run_realtime_runtime_acceptance.py --profile smoke
python scripts/render_realtime_runtime_acceptance_report.py \
  temp_dir/realtime_runtime_acceptance/report.json \
```

## 7. 验收维度

### 7.1 功能验收

- 单智能体请求能稳定进入线程并完成一轮执行
- 多智能体任务能形成 mission 并完成公开摘要输出
- 工具、技能、MCP 至少有一条闭环路径可用

### 7.2 运行验收

- server、desktop、cli 三种形态都能启动
- 配置文件和环境变量覆盖关系清楚
- WebSocket 主链正常，断线后可恢复

### 7.3 稳定性验收

- waiting / resume / crash recovery 可工作
- 前端路由切换不出现重复订阅和状态回退
- 慢客户端不会拖垮投影分发
- 关键延迟指标：`thread_accept_latency < 80ms`、`mission_schedule_latency < 120ms`、`projection_live_latency_proxy < 200ms`
- replay 样本 `replay_events >= 500` 时，恢复应尽量保持在 `500ms` 内

### 7.4 文档验收

- 本目录、`docs/设计`、`docs/API` 与实现事实一致
- 变更已记录到 `docs/功能迭代.md`
