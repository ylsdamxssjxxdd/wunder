# 运行时与执行链路

> 本文档与 `docs/设计文档/02-智能体核心设计.md` 互补：那篇讲"为什么这样设计"，这篇讲"代码怎么跑、怎么查、怎么改"。

## 1. 运行时全景

wunder 的执行核心围绕**线程运行时**展开。不管入口是 HTTP、WS、gateway、channel 还是 CLI，最终都映射为线程请求、mission 请求或系统控制请求。

```
外部请求 → 接入层（API / WS / Channel / Gateway）
          → ThreadRuntime（lease、queue、dispatch）
            → ObservedAgentTurn（桥接新旧内核）
              → eva / agent_kernel（turn 执行主链）
                → Prompt → Model → Tools → Output
              → Projection（状态发布）
```

### 关键目录

| 目录 | 职责 |
| --- | --- |
| `crates/eva/src/thread/` | 线程快照、watch、resume、execution lease |
| `crates/eva/src/session/` | checkpoint、waiting state、session runtime |
| `crates/eva/src/turn/` | turn 输入、阶段、输出、事件 |
| `crates/eva/src/prompt/` | prompt 装配和冻结 |
| `crates/eva/src/context/` | history、budget、compaction |
| `crates/eva/src/tools/` | 工具治理全链路 |
| `crates/eva/src/recovery/` | 恢复链 |
| `src/core/observed_*` | observed runtime 桥接（双运行时管理、状态同步） |

## 2. 一次标准请求的执行流程

1. **接入** — API / WS / Channel / Cron 收到请求，鉴权，参数校验。
2. **定位线程** — 根据请求找到或创建线程。主线程忙时自动分叉隔离 session。
3. **获取 lease** — `submit lease` + `thread route lease`，保证执行互斥。忙时入队。
4. **决定模式** — 新建线程（`FreshUserTurn`）？接续活跃轮次（`AttachActiveTurn`）？从检查点恢复（`RestoreFromCheckpoint`）？
5. **组装上下文** — 构建 prompt 输入、初始上下文标签、工具面和记忆快照。
6. **context 预处理** — budget 分配、history normalization、必要 compaction。
7. **模型推理** — 发起模型 API 调用。
8. **输出解析** — 文本回复直接输出；工具调用进入工具循环。
9. **工具执行** — 通过统一 tool runtime 执行，写入 ledger 和 durable state。
10. **循环或结束** — 工具结果注入上下文，回到步骤 7；或到达最终回复。
11. **状态持久化** — 写入 checkpoint、waiting 状态、assistant 输出。
12. **事件发布** — turn events 推送到 watch、projection、前端和日志系统。

## 3. 三层时间尺度

```
线程 (Thread)
 └─ 用户轮次 (User Round)    ← 用户发一次输入到最终回复
 │   ├─ 模型轮次 1 (Model Round)  ← 模型一次 API 调用 + 工具执行
 │   ├─ 模型轮次 2
 │   └─ ...
 ├─ 用户轮次 2
 └─ ...
```

对应代码中 `EvaTurnCursor` 的字段：

- `user_turn_id` — 标识当前用户轮次
- `model_round_index` — 当前第几轮模型轮次
- `tool_call_index` — 当前工具批次中的第几个调用
- `attempt_index` — 重试序号

## 4. 用户轮次流水线

一轮用户轮次经历 19 个显式阶段（`EvaTurnLifecycleStage`），构成严格流水线：

```
接入:     InputAccepted → ListenerAttached → RuntimeAssembled
上下文:   ContextSnapshotInitialized → ContextBudgeted → ContextPrepared → ContextItemsInjected
模型轮次: PromptBuilt → ModelRequested → ModelResponded → OutputParsed
          ↕ ToolBatchPlanned → ToolRunning ──→ PromptBuilt (循环)
终态:     HistoryArchived → Finalizing → CompletionPublished → Completed / Failed
```

其中 `PromptBuilt → … → ToolRunning → PromptBuilt` 是模型轮次循环，每循环一次 `model_round_index` 递增。

## 5. Prompt 链路

### 规则

- 首轮线程初始化时确定 system prompt，之后只能复用（`FrozenPromptSnapshot`）。
- prompt 由 `BaseInstructions` + `DeveloperContext` + `UserContext` + `AgentsSnapshot` + `MemorySnapshot` 组装。
- 宿主相关组装通过 `eva::wunder` 完成，不在 API 层手工拼。

### 排查顺序

1. 线程是否已有 frozen prompt？
2. 是否错误地把恢复逻辑走成了重新初始化？
3. `eva::wunder` prompt adapter 是否按当前语言和宿主状态正确装配？

## 6. Context 链路

Context 链路让长会话可持续运行：

- **history 归一化** — 格式统一、去重、角色标注
- **budget 控制** — 为 system prompt、恢复说明、观察项预留空间
- **compaction** — 历史对话压缩（`Compacting` 阶段）
- **truncation** — 硬截断保底
- **overflow recovery** — 溢出时走明确恢复链，不静默丢上下文

### 排查顺序

1. 是预算配置问题还是恢复链逻辑问题？
2. 是 thread-init context 重注入问题还是普通 history 超载？
3. 是否错误地重复注入 memory snapshot 或初始上下文？

## 7. Tool 链路

工具调用必须经过统一治理链路：

```
spec → registry → router → batch planner → preflight
  → permissions → approvals → sandbox → retry governor → executor → ledger
```

关键原则：

- 模型只能看到当前线程允许的 **tool surface**，不是全部后端能力。
- 调用前检查审批、沙盒、并行策略和超时。
- 执行后写入 ledger 和 durable state。

### 排查区分

| 现象 | 可能原因 |
| --- | --- |
| 工具不可见 | tool surface 构建或模型能力过滤问题 |
| 工具被阻塞 | 审批策略、沙盒限制 |
| 工具执行失败 | 运行时错误、超时 |
| 工具结果丢失 | recovery 标记为 discarded、ledger 与 checkpoint 不一致 |

## 8. Waiting 与 Recovery

### Waiting 类型

| 类型 | 触发条件 |
| --- | --- |
| `Approval` | 工具调用需要用户/管理员审批 |
| `UserInput` | 智能体请求用户补充输入 |
| `Subagent` | 等待子智能体完成 |

每种等待态记录在 `EvaWaitingState` 中，包含 request_id、关联工具名和调用 ID、状态（Pending / Answered / Expired / Superseded / Cancelled / Orphaned）。

### 恢复场景

| 场景 | 设计要求 |
| --- | --- |
| 中断恢复 | 保留 waiting / continuation，不降级为重新开始 |
| 崩溃恢复 | 从 checkpoint 或 thread snapshot 重新装回运行时 |
| 工具结果缺失 | 识别 missing / discarded results，给出后续策略 |
| 上下文溢出 | 先压缩 → 再恢复 → 必要时回退安全历史 |
| 恢复耗尽 | `EvaRecoveryState` 追踪尝试次数，超过限制标记 `ManualIntervention` |

### Execution Lease

线程通过 `EvaExecutionLease` 保证同一时刻只有一个 runtime owner 在执行。Lease 支持 acquire / heartbeat / release，超时自动失效。

### 排查顺序

1. checkpoint 是否存在且内容完整？
2. thread snapshot 是否仍可装回？
3. waiting_state 的 request_id / call_id 是否被错误覆盖？
4. 恢复链是否把续跑错误降级成了"重新开始"？

## 9. Observed Runtime 桥接

当前系统同时维护新旧两套运行时，通过 `src/core/observed_*` 桥接：

| 文件 | 职责 |
| --- | --- |
| `observed_runtime.rs` | `ObservedRuntimeHost` — 双运行时管理（legacy + eva） |
| `observed_runtime_mirror.rs` | `ObservedEvaRuntimeMirror` — EVA → legacy 状态同步 |
| `observed_runtime_snapshot.rs` | 统一快照格式与恢复计划 |
| `observed_agent_turn.rs` | Agent turn 桥接，协调新旧执行器 |
| `observed_turn_executor.rs` | Turn 执行器桥接 |
| `legacy_agent_kernel.rs` | 旧类型 re-export 兼容 |

桥接层确保 EVA 运行时的状态变化能同步回 legacy 格式，前端和 API 层无需感知底层切换。

## 10. 典型排查路径

| 现象 | 第一检查点 | 第二检查点 |
| --- | --- | --- |
| 首轮 prompt 漂移 | frozen prompt 是否重复生成 | host adapter 是否错误走初始化 |
| 工具结果丢失 | tool ledger / call 状态 | recovery 是否标记 discarded |
| 线程一直 waiting | waiting_state 字段与 request_id | 前端或 resume 调用是否匹配当前 call |
| 恢复后状态错乱 | checkpoint 内容 | thread snapshot 与 runtime compat 映射 |
| 上下文溢出后死循环 | recovery_state 尝试计数 | overflow recovery 是否走 compaction 分支 |
