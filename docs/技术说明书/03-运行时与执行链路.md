# 运行时与执行链路

> 本文档与 `docs/设计文档/02-智能体核心设计.md` 互补：那篇讲"为什么这样设计"，这篇讲"代码怎么跑、怎么查、怎么改"。

## 1. 运行时全景

wunder 的执行核心围绕 **orchestrator + ThreadRuntime** 展开。不管入口是 HTTP、WS、gateway、channel 还是 CLI，最终都映射为线程请求、mission 请求或系统控制请求。

```
外部请求 → 接入层（API / WS / Channel / Gateway）
          → ThreadRuntime（lease、queue、dispatch）
            → Orchestrator（turn 执行主链）
              → Prompt → Model → Tools → Output
            → 事件发布（stream events、beeroom）
```

### 关键目录

| 目录 | 职责 |
| --- | --- |
| `src/orchestrator/` | Turn 状态机、prompt 组装、context 管理、工具执行、恢复链 |
| `src/services/runtime/thread/` | 线程生命周期、lease 管理、排队、dispatch |
| `src/services/runtime/mission/` | Mission 调度与 team run 管理 |
| `src/core/state.rs` | AppState 三层装配（kernel / projection / control） |
| `src/core/config.rs` | 运行时配置定义 |

## 2. 一次标准请求的执行流程

1. **接入** — API / WS / Channel / Cron 收到请求，鉴权，参数校验。
2. **定位线程** — 根据请求找到或创建线程。主线程忙时入队。
3. **获取 lease** — ThreadRuntime 分配执行 lease，保证执行互斥。
4. **Turn 启动** — Orchestrator 接管 turn，初始化 turn state。
5. **Prompt 组装** — 从配置和 agent 预设中组装 prompt（身份、指令、上下文、工具面）。
6. **Context 预处理** — budget 分配、history normalization、必要时 compaction。
7. **模型推理** — 发起模型 API 调用。
8. **输出解析** — 文本回复直接输出；工具调用进入工具执行。
9. **工具执行** — 通过 `tool_exec` / `tool_parallel` 执行，写入工具日志。
10. **循环或结束** — 工具结果注入上下文，回到步骤 7；或到达最终回复。
11. **结果持久化** — 写入 stream events、chat history、session 状态。
12. **事件发布** — 推送到 beeroom、user world 等消费方。

## 3. 三层时间尺度

```
线程 (Thread)
 └─ 用户轮次 (User Round)    ← 用户发一次输入到最终回复
 │   ├─ 模型轮次 1 (Model Round)  ← 模型一次 API 调用 + 工具执行
 │   ├─ 模型轮次 2
 │   └─ ...
 ├─ 用户轮次 2
 └─ ...
```

对应 orchestrator 中 `TurnState` 的字段：
- turn ID 与状态
- 当前模型轮次索引
- 当前工具调用批次
- 重试计数

## 4. 用户轮次流水线

一轮用户轮次经历以下阶段：

```
接入:     RequestReceived → Preflight → PromptAssembled
上下文:   ContextBuilt → ContextBudgeted → ContextPrepared
模型轮次: LLMRequested → LLMResponded → OutputParsed
          ↕ ToolCalls → ToolExecuting ──→ PromptAssembled (循环)
终态:     ResultNormalized → StreamPersisted → Completed / Failed
```

其中 `PromptAssembled → … → ToolExecuting → PromptAssembled` 是模型轮次循环。

### 排查关键文件

| 文件 | 关注内容 |
| --- | --- |
| `src/orchestrator/execute.rs` | turn 执行主流程 |
| `src/orchestrator/turn_state.rs` | turn 状态机 |
| `src/orchestrator/preflight.rs` | 执行前预处理 |
| `src/orchestrator/prompt.rs` | prompt 组装与冻结 |

## 5. Prompt 链路

### 规则

- 首轮线程初始化时确定 system prompt，之后保持稳定。
- prompt 由基础指令 + 开发者上下文 + 用户上下文 + 可用智能体列表 + 记忆快照 组装。
- prompt 组装在 orchestrator 内完成，不在 API 层手工拼。

### 排查顺序

1. 线程是否已有缓存的 prompt？
2. 是否错误地把恢复逻辑走成了重新初始化？
3. prompt.rs 是否按当前语言和状态正确装配？

## 6. Context 链路

Context 链路让长会话可持续运行：

- **history 归一化** — 格式统一、去重、角色标注
- **budget 控制** — 为 system prompt、恢复说明、观察项预留空间
- **compaction** — 历史对话压缩（`context_compactor.rs`）
- **truncation** — 硬截断保底

### 排查顺序

1. 是预算配置问题还是 compaction 逻辑问题？
2. 是 context 重注入问题还是普通 history 超载？
3. 是否错误地重复注入 memory？

## 7. Tool 链路

工具调用经过 orchestrator 的统一治理链路：

```
tool_calls.rs (解析) → tool_exec.rs (调度) → tool_parallel.rs (并行)
  → 权限检查 → 审批判定 → 执行 → 结果归一化
```

关键原则：
- 模型只能看到当前线程允许的 **工具面**，不是全部后端能力。
- 调用前检查审批、并行策略和超时。
- 执行后写入 tool_logs 和 stream events。

### 排查区分

| 现象 | 可能原因 |
| --- | --- |
| 工具不可见 | 工具面构建或模型能力过滤问题 |
| 工具被阻塞 | 审批策略、沙盒限制 |
| 工具执行失败 | 运行时错误、超时 |
| 工具结果丢失 | stream persist 异常、tool_log 不一致 |

## 8. Waiting 与 Recovery

### Waiting 类型

| 类型 | 触发条件 |
| --- | --- |
| 审批等待 | 工具调用需要用户/管理员审批 |
| 用户输入 | 智能体请求用户补充输入 |
| 子智能体 | 等待子智能体完成 |

### 恢复场景

| 场景 | 设计要求 |
| --- | --- |
| 中断恢复 | 保留 waiting 状态，不降级为重新开始 |
| 崩溃恢复 | 从 session 状态和 stream events 恢复 |
| 工具结果缺失 | 识别缺失结果，给出后续策略 |
| 上下文溢出 | 先压缩 → 再恢复 |

### Execution Lease

线程通过 ThreadRuntime 的 lease 机制保证同一时刻只有一个 owner 在执行。

### 排查顺序

1. session 状态是否完整？
2. stream events 是否有断档？
3. waiting 状态的 request_id 是否被错误覆盖？
4. 恢复链是否把续跑错误降级成了"重新开始"？

## 9. AppState 三层装配

`src/core/state.rs` 将服务组织为三层：

| Plane | 代表字段 | 主要职责 | 关键模块 |
| --- | --- | --- | --- |
| `kernel` | `AppKernelServices` | 智能体执行、线程推进、任务编排 | `orchestrator`、`runtime/thread`、`runtime/mission`、`swarm` |
| `projection` | `AppProjectionServices` | 用户世界、beeroom 的实时视图 | `services/user_world`、`services/beeroom_realtime` |
| `control` | `AppControlServices` | presence、gateway、channels、cron、审批、命令会话 | `services/presence`、`gateway`、`channels`、`cron`、`approval_registry` |

## 10. 典型排查路径

| 现象 | 第一检查点 | 第二检查点 |
| --- | --- | --- |
| 首轮 prompt 漂移 | prompt.rs 是否重复生成 | 配置与 agent 预设是否正确 |
| 工具结果丢失 | tool_logs 状态 | stream persist 是否异常 |
| 线程一直 waiting | turn_state 字段 | 前端或 resume 调用是否匹配 |
| 恢复后状态错乱 | session 状态内容 | stream events 与 session 对齐 |
| 上下文溢出后死循环 | retry_governor 尝试计数 | compaction 是否正常触发 |
