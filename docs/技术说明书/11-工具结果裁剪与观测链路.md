# 工具结果裁剪与观测链路

## 1. 目的

本章用于澄清 wunder 在工具执行链路中的多层裁剪机制，明确“哪些会影响模型输入，哪些只影响监控或前端显示”。

## 2. 五层裁剪模型

1. 工具自身裁剪：影响模型输入、事件数据、日志。
2. 编排层全局裁剪（`finalize_tool_result`）：影响模型输入、事件数据、日志。
3. 模型观测压缩（`model_observation`）：只影响模型 observation，不改原始 `tool_result.data`。
4. 监控裁剪（monitor）：影响监控记录与时间线查询，不直接改模型输入。
5. 前端显示裁剪：仅影响 UI，可读性优化，不改后端真实数据。

## 3. 端到端链路

一次工具调用的关键处理顺序：

1. 工具执行，返回原始结果（可能已触发工具自身预算裁剪）。
2. `normalize_tool_result_payload` 统一成功/失败语义与错误元信息。
3. `finalize_tool_result` 执行全局字符串/数组/大结果裁剪。
4. 生成 `tool_result` 事件负载（用于前端工作流、排障、日志）。
5. 生成 `model_observation`（给模型继续推理的压缩 observation）。
6. 以 `tool_response` 形式写入模型上下文。

核心结论：`tool_result.data` 与 `model_observation` 是双通道，不能混为同一个排障对象。

## 4. 工具层典型裁剪

### 4.1 `read_file`

- 单文件读取字节上限与默认行窗口限制。
- 支持 `output_budget_bytes`，超预算会在文本中追加 `...(truncated read output, omitted N bytes)...`。
- 会返回 `continuation_required` 及相关 budget meta。
- observation 层不再为 `read_file` 维护独立字符阈值，统一复用全局 observation 预算。

### 4.2 `list_files`

- 主要为分页（`cursor/offset/limit`），不是字符串截断。
- 返回 `has_more/next_cursor/next_offset` 供续取。

### 4.3 `search_content`

- `max_matches/max_candidates` 有默认值与上限。
- 可按 `output_budget_bytes` 裁剪 hits 尾部，并在 summary 标注压缩提示。

### 4.4 `execute_command`

- stdout/stderr 分别按 head+tail 护栏保留。
- 可配置总输出预算；超限时插入 `...(truncated command output, omitted N bytes)...`。

### 4.5 `web_fetch`

- 响应体按字节上限截断。
- 内容按字符上限截断，结果通过 `truncated/warning` 显式标注。

## 5. 编排层全局裁剪（`finalize_tool_result`）

- 字符串按 head/tail 截断，插入统一标记 `...(truncated)...`。
- 数组超过阈值时改为“头 + 省略标记 + 尾”的结构。
- 对带续取信号的分页数据放宽数组上限。
- 结果整体仍过大时，封装为 `truncated/original_chars/preview/truncation_reasons` 结构，并在可续取场景附带 continuation 提示。

## 6. 模型观测压缩（`model_observation`）

- observation 有独立字符与数组预算。
- 会对 `search_content` 做进一步抽样；`read_file` 则统一走全局 observation 预算，不再维护单独的字符截头常量。
- 若输入已经是 `truncated/original_chars/preview` 包装，observation 会尽量保留接近自身字符预算的 `preview`，避免把长结果再次压成过短摘要导致模型失去整体轮廓。
- 密集数组可能转为 `*_jsonl + *_count`。
- 会移除一批与模型决策无关的噪声字段。

该层只服务模型上下文控制，不作为前端“原始结果”展示基线。

## 7. 监控与前端层说明

### 7.1 monitor

- monitor 在 normal 画像下会按 `monitor_payload_max_chars` 裁剪字符串，并追加 `...(truncated)`。
- 时间线事件读取 monitor 记录时，看到的截断可能来自 monitor 层。

### 7.2 frontend

- 工具工作流预览、流式缓冲、结构化视图都可能做显示裁剪。
- 该类裁剪不回写后端结果，不代表模型真实输入被截断。

## 8. 统一排障顺序

1. 先看 `tool_result.data`：确认工具层/全局层是否已截断。
2. 再看 `model_observation`：确认模型实际接收内容。
3. 再看 monitor 记录：判断是否是监控层裁剪。
4. 最后看前端展示：判断是否只是 UI 层裁剪。

按该顺序可快速区分“模型输入问题”与“展示问题”。
