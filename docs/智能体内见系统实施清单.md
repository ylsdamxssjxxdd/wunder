# wunder 智能体内见系统实施清单

- 详细设计以 `docs/智能体内见系统落地方案.md` 为准；本文只保留工程实施节点、责任边界与验收口径。
- 总原则固定为：**复用用户私有工作目录（`container_id=0`）作为内见系统根，主配置统一使用 `WorkerCard-first`，不新增聊天右侧切换入口。**

## 阶段 0：目录与边界定型

- 明确用户私有工作目录中的保留区结构：
  - `skills/`
  - `global/`
  - `global/tooling.json`
  - `agents/<agent_id>/`
  - `.wunder/effective/`
  - `.wunder/diagnostics/`
  - `.wunder/last_good/`
- 明确这些目录属于“用户资产目录”，不是临时工作产物目录；运行时生成物与普通任务产物继续留在既有工作区路径。
- 明确权限边界：智能体允许修改同一用户私有目录下的自身配置、全局配置、其它智能体配置，但不得越权写系统级配置与其它用户目录。
- 验收标准：
  - 新用户首次进入私有目录时可自动补齐最小骨架。
  - 目录结构稳定、可重复初始化、不会覆盖已有用户文件。

## 阶段 1：目录初始化与路径解析

- 后端新增内见系统路径解析模块，统一解析：
  - 用户私有目录根
  - `skills/global/agents/.wunder` 保留区
  - `agents/__default__/` 默认智能体目录
- 在用户私有目录缺少关键目录时执行幂等初始化，并写入最小 `README.md` / 示例配置。
- 所有路径访问必须先做规范化与越界校验，防止 `..`、符号链接或软链接逃逸。
- 验收标准：
  - 任意用户首次触发智能体时，内见系统目录可自动准备完成。
  - 非法路径、损坏路径、越界路径都会被拒绝并产出诊断信息。

## 阶段 2：配置读取、合并与校验

- 后端新增 `worker-card` 合并链路，合并顺序固定为：
  - 系统内建默认
  - `global/defaults.worker-card.json`
  - `agents/<agent_id>/worker-card.json`
- 明确字段语义：
  - 文件层使用 `WorkerCard.abilities.tool_names/skills` 作为声明依赖源
  - 运行态仍同步 `tool_names`、`declared_tool_names`、`declared_skill_names`
  - 不再设计 `extensions.wunder.declared_*` 作为主承载字段
- 保留 `system_prompt.md` 作为长提示词首选编辑源，并在 apply 时回写 `worker-card.json.prompt.system_prompt`。
- 生成并持久化：
  - `.wunder/effective/global.json`
  - `.wunder/effective/agents/<agent_id>.json`
  - `.wunder/diagnostics/...`
  - `.wunder/last_good/...`
- 对坏文件、缺字段、JSON 非法、Prompt 冲突提供诊断与降级，不允许直接把线程跑挂。
- 验收标准：
  - 配置损坏时仍能回退到 last_good 或默认配置继续运行。
  - diagnostics 能明确指出出错文件、字段、原因与回退来源。

## 阶段 3：运行时挂载与生效语义

- 每次智能体新一轮执行时，运行时需同时感知：
  - 当前 agent 工作目录
  - 当前用户私有目录根
- 生效语义固定为：
  - 工具声明、技能声明、运行时参数等配置修改：**下一轮生效**
  - `system_prompt.md` / `WorkerCard.prompt` 修改：**仅新线程生效**
  - 不采用“修改后立即热替换当前线程 system prompt”
- 提供“软重启智能体”语义：创建新会话并切主，不重启后端进程、不重启桌面应用。
- 验收标准：
  - 当前线程的冻结 system prompt 不会被后续文件修改破坏。
  - 软重启后新线程能看到最新配置，旧线程仍保持原提示词快照。

## 阶段 4：文件修改、应用与恢复机制

- 复用现有读写/搜索/补丁工具，不新增专用“修改智能体配置”工具。
- 对内见系统目录的写入增加统一 apply 流程：
  - 读取目标文件
  - 解析与校验
  - 生成 diagnostics/effective
  - 成功后提交
  - 失败则回退 last_good
- 对关键文件建立最小审计字段或事件，至少能追踪：
  - 谁改了
  - 改了哪个文件
  - 是否生效
  - 是否回退
- 验收标准：
  - 智能体可通过普通文件工具自修改配置。
  - 配置写坏后系统可恢复，不会导致后续轮次全部不可用。

## 阶段 5：技能分层与解析链路

- 技能解析优先级固定为：
  - `agents/<agent_id>/skills/`
  - `skills/`
  - 仓库内置技能
- 与当前代码保持一致：
  - 用户自有技能启用以 `skills/` 目录发现为准
  - 不重新发明一套人工维护的 `skills.enabled` 主开关
  - 用户级共享/挂载策略单独落 `global/tooling.json`
- `worker-card.json` 中只声明引用关系；技能正文仍以文件目录为准。
- 当共享技能与私有技能重名时，按优先级覆盖，并在 diagnostics 中记录解析来源。
- 验收标准：
  - 同名技能覆盖关系稳定可预测。
  - 删除或损坏单个技能文件不会拖垮整套技能加载。

## 阶段 6：前端最小支撑

- 不新增聊天右侧“工作/家”切换按钮，不改 `MessengerRightDock.vue` 交互模型。
- 前端只做最小支撑：
  - 确保用户能进入自己的私有目录
  - 确保 `json/md` 文件可预览、可编辑、可保存
  - 必要时为 `skills/global/agents` 自动生成只读 `README.md`
- 若后续需要增强，也优先做目录说明与文件模板，不单独造一个“配置面板”。
- 验收标准：
  - 用户无需额外入口，只靠私有目录即可看懂当前配置状态。
  - 浅色/深色主题下目录浏览、文件预览、文件编辑均无明显样式问题。

## 阶段 7：接口、事件与诊断暴露

- 对外接口最小增量优先，先复用现有 workspace 文件接口与 agent 会话接口。
- 如确有必要，再补充轻量接口：
  - 查询某个 agent 的 effective 配置
  - 查询最近一次 apply diagnostics
  - 触发软重启并切主会话
- 事件语义需与现有会话流统一，避免新开一套前后端协议。
- 验收标准：
  - 无新增接口时主链路已可跑通。
  - 新增接口若落地，必须保持返回体简单稳定，且补齐 `docs/API文档.md`。

## 阶段 8：迁移与默认数据

- 老的智能体配置入口若仍存在，需要补迁移器或首次访问自动导入逻辑，把旧配置映射到：
  - `global/defaults.worker-card.json`
  - `global/tooling.json`
  - `agents/<agent_id>/worker-card.json`
  - `agents/<agent_id>/system_prompt.md`
- 现有 `data/user_tools/<user_id>/config.json` 与用户技能目录语义需要迁移到用户私有目录：
  - 用户技能目录继续作为技能发现源
  - 用户级共享工具/共享技能/MCP 共享选择迁移到 `global/tooling.json`
- 迁移过程必须幂等，可重复执行，不依赖人工手改数据库。
- 对默认智能体 `__default__` 单独准备默认模板，保证新用户开箱即用。
- 验收标准：
  - 新用户能直接使用。
  - 老数据迁移后配置等价，不出现提示词丢失、工具声明丢失、技能声明丢失。

## 阶段 9：测试与上线门槛

- 后端至少覆盖以下测试：
  - 路径解析与越界防护
  - 目录初始化幂等
  - `WorkerCard` 合并与字段覆盖
  - `system_prompt.md` 与 `worker-card.json` 同步
  - 坏配置回退与 diagnostics 生成
  - 软重启后的新旧线程生效差异
- 前端至少覆盖以下验证：
  - 私有目录可进入
  - `json/md` 文件预览编辑正常
  - 无新增切换按钮后交互路径依然清晰
- 上线门槛：
  - `cargo check`
  - 相关 Rust 单测/集成测试通过
  - 前端相关构建或最小范围校验通过

## 推荐落地顺序

- 第一批：阶段 0 ~ 3，先把目录、配置解析、生效语义跑通。
- 第二批：阶段 4 ~ 5，补齐 apply/回退/技能分层。
- 第三批：阶段 6 ~ 9，补齐前端可见性、轻量接口、迁移与测试。

## 完成定义

- 用户私有目录成为智能体可见、用户可见、系统可恢复的统一配置根。
- 智能体可以通过普通文件工具修改自身、全局与其它智能体配置。
- 当前线程 system prompt 冻结约束不被破坏。
- 配置损坏时系统可诊断、可回退、可继续运行。
- 用户不需要进入额外配置页面，只需查看自己的私有目录即可理解当前配置状态。
