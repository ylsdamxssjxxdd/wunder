# 多智能体调控平台设计与实现

## 摘要

随着大语言模型从"问答工具"演进为"行动主体"，如何构建一个既能稳定承载单智能体认知执行、又能支撑多智能体协作调控的运行时平台，成为智能体工程化的核心难题。本文以 wunder（心舰）智能体调度系统为研究对象，系统阐述其设计与实现。该平台以 orchestrator 为执行内核，采用 kernel/projection/control 三层主骨架，使服务端、桌面端、命令行端三种运行形态共享同一套核心语义；以"线程"为认知单元，通过 prompt 冻结与一次性记忆注入保证认知态稳定；以"技能"为流程化经验包，通过渐进式披露与元技能自举实现领域知识的可复用封装；以"蜂群"为协作域，通过子智能体与蜂群工具的二分隔离实现灵活的多智能体调控。本文重点剖析技能系统的三级加载体系、自由度分级设计与自包含资产模型，并完整呈现存储双后端抽象、实时投影契约与渠道适配治理等关键工程实现。

**关键词**：智能体运行时；多智能体协作；技能系统；工具治理；实时投影；prompt 冻结

---

## 1. 引言

传统的大模型应用多以"对话即服务"为基本形态，模型只负责生成文本，外部能力依赖人工编排。这种模式在面对需要持续执行、跨工具协作、长期记忆与多角色协同的真实任务时显得力不从心。wunder 的设计哲学是"对开发者来说一切都是接口，对大模型来说一切皆工具"——将文件、浏览器、记忆、蜂群、渠道、线程都统一为模型可调用的工具语义，使智能体从"说话者"转变为"行动者"。

然而，当智能体具备了强大的行动能力后，平台层面的调控难题随之浮现：单智能体的认知态如何不被协作态与前端态污染？多智能体协作的临时派生与持久协作如何区分？领域专业知识如何在不撑爆上下文窗口的前提下按需注入？高风险能力如何纳入统一治理而非依赖提示词约束？三种运行形态如何共享同一套核心而不各自长成独立系统？这些问题共同构成了"多智能体调控平台"的设计空间。

本文从系统总体设计、智能体核心设计、技能系统设计、多智能体协作设计、实时投影与通信五个维度，完整呈现 wunder 的设计与实现。

## 2. 系统总体设计

### 2.1 架构基线与设计理念

wunder 不是一个带聊天界面的模型调用器，而是以 orchestrator 为核心的智能体运行时系统。其主架构可概括为：

```
Request → Orchestrator (turn) → ThreadRuntime → Stream Events → Frontend
```

架构设计遵循五条基线：智能体优先（线程、turn、工具治理是主系统，页面与房间只是事件流的消费视图）；认知态隔离（线程首次确定后的 system prompt 必须冻结，长期记忆只允许在线程初始化时注入一次）；单写者收敛（线程 lease 与 mission lease 是当前单服务器阶段的权威边界）；实时系统服务运行时（WebSocket、watch、stream events 围绕运行时复制与恢复，而非反向支配运行时）；执行调度集中（orchestrator 负责智能体执行的完整链路，不分散到多个独立状态机）。

### 2.2 三层主骨架：kernel / projection / control

后端最重要的装配锚点是 `AppState`，它被明确收口为三层。**kernel 层**承载智能体执行、线程推进与任务编排，包含 orchestrator、ThreadRuntime、MissionRuntime 与 SwarmService；**projection 层**承载用户世界与实时视图，包含 UserWorldService 与 BeeroomRealtimeService；**control 层**承载 presence、gateway、channels、cron、审批与命令会话等治理能力。

三层分离的必要性在于：kernel 必须可以在没有前端页面、没有房间订阅的情况下继续运行；projection 必须可以从 stream events 重新构建，不应成为线程认知态来源；control 负责"谁有权推进""谁在观看""谁能接入"，不直接承担模型推理语义；frontend 负责消费、恢复和补偿，不应重新定义后端真相。这种分层确保了认知态、投影态与治理态的边界清晰，避免了不同层的债务混在同一模块里修补。

### 2.3 三形态同核

平台通过 `AppRuntimeProfile` 区分三种运行形态：ServerDistributed（组织级服务部署，启动全部后台运行时）、DesktopEmbedded（本地桌面主应用，启动 thread_runtime 与 cron）、CliEmbedded（本地命令行，默认不启动后台运行时）。三种形态共享同一套 `AppState` 核心装配，靠 profile 默认值与显式覆盖差异化。`AppRuntimeCapabilities` 暴露 8 个开关（如 embedded_mode、thread_runtime_active、mission_runtime_active、gateway_maintenance_active 等），由 profile 与 config 共同决定，并支持 `AppStateInitOptions` 显式覆盖。装配过程严格按依赖顺序推进，每阶段打点计时，便于性能优化。

### 2.4 双数据库存储抽象

存储层采用"trait 定义 + 双实现"模式，但远比普通 Repository 模式精细。`StorageBackend` 并非单体 trait，而是被拆分为 24 个领域内聚的子 trait（如 MetaStore、ConversationLogStore、AgentRuntimeStore、MemoryRecordStore、ChannelRuntimeStore、GatewayStore 等），通过 super-trait 聚合。这种组合式 trait 设计让新增存储领域只需扩展子 trait，不破坏现有调用面，业务层只需持有 `Arc<dyn StorageBackend>` 即可访问全部存储能力。

统一状态模型由与后端无关的共享 Record 结构承载，两个后端实现都解析与写入相同的 Record 类型。PostgreSQL 实现使用 deadpool_postgres 连接池，通过封装层将异步调用同步化使 trait 方法保持同步签名，利用 `ON CONFLICT`、事务等原生能力；SQLite 实现每次创建新连接，设置 `busy_timeout=5s`、`journal_mode=WAL`，适合桌面端单机嵌入。迁移采用"幂等 CREATE + 探测式 ALTER + 数据回填"三段式，通过双重检查保证只执行一次，避免了引入额外迁移框架。工厂层支持 `auto` 模式，在 PostgreSQL 不可用时回退到 SQLite，保证桌面端容错启动。

## 3. 智能体核心设计

### 3.1 四层运行时单元

单智能体在 wunder 中不是一个页面对象或聊天室对象，而是带严格边界的线程运行时单元，同时包含四个层面：线程状态（thread_id、当前 turn、waiting state、resume plan、frozen prompt 引用、execution lease）；认知上下文（system prompt、developer/user context、memory snapshot、工具 observation、上下文压缩状态）；执行治理（tool surface、审批策略、沙箱策略、重试治理、动态工具挂载、模型能力约束）；现实外壳（工作区文件、流式事件、公开投影、数据库记忆、运行指标）。

### 3.2 三层时间尺度与线程生命周期

线程内的交互被拆为三个时间层级：线程是单智能体的完整认知容器，贯穿一次会话的整个生命周期；用户轮次是用户发来一次输入到智能体产出最终回复之间的全过程，一轮用户轮次可包含多轮模型轮次；模型轮次是模型一次 API 调用加输出解析，若模型返回工具调用则执行工具后进入下一轮模型轮次。用户轮次与模型轮次独立计量，避免将一次"消息交互"混为一谈。

线程运行时管理线程的完整生命周期状态，包括 Initializing、Idle、Running、Compacting、WaitingApproval、WaitingUserInput、WaitingSubagent、Interrupted、Error、Closed 十种状态。其中 Waiting 系列状态（审批等待、用户输入、子智能体等待）必须是显式运行态，可继续 watch 和 resume，而非隐式约定。线程通过 ThreadRuntime 的 lease 保证执行互斥：同一时刻只有一个 owner 可以执行线程，超时自动释放防止崩溃后死锁。

### 3.3 用户轮次流水线

一轮用户轮次在 orchestrator 中经历接入、上下文准备、模型轮次循环、后处理与终态五个阶段组。其中模型轮次循环构成核心：`PromptAssembled → LLMRequested → LLMResponded → OutputParsed → ToolExecuting → PromptAssembled` 形成循环，每次循环即一轮模型轮次。每次阶段切换产生事件并封装在 stream events 中，是管理侧状态页的核心数据源。流水线中任何阶段都可能转入 Failed，终止当前用户轮次并记录错误上下文。

### 3.4 Prompt 冻结与记忆一次性注入

prompt 组装包含基础指令、开发者上下文、用户上下文、可用智能体列表与记忆快照。首轮线程初始化时确定 system prompt，之后轮次直接复用已冻结的 prompt，任何情况下不允许后续轮次改写。记忆注入只允许在线程初始化阶段通过 snapshot 注入一次，原因有三：线程进入 steady state 后不应因记忆库变化而漂移 system prompt；再次注入会破坏模型侧提示词缓存；记忆后续可通过工具检索，但不应回写线程 system prompt。这一设计从结构上保证了认知态的稳定性，是整个平台最不可破坏的约束之一。

### 3.5 工具面与执行治理

工具系统采用四层模型：模型侧工具面（暴露工具名、短 description、canonical schema）、兼容解析层（吃掉别名、历史字段与兼容输入）、执行治理层（调度、审批、并行、超时、失败治理）、返回归一层（统一成功包、失败包与 compact 结果）。模型侧 schema 的目标不是完整表达后端接口，而是让模型更容易调用正确——name 稳定、description 短、parameters 只保留高频 canonical 字段、`additionalProperties` 尽量关闭、少用复杂的条件分支。

orchestrator 提供完整的治理链路：工具调用解析 → 并行打包 → 权限检查 → 审批判定 → 沙盒策略 → 执行 → 重试 → 结果归一化 → 日志。高风险能力（命令执行、桌面控制、外部渠道、浏览器等）必须受审批、超时、隔离与重试治理，不能只靠提示词约束。工具返回统一向结构化骨架靠拢，成功包包含 ok、action、state、summary、data，失败包包含结构化错误与可重试标记，大对象默认做 compact 不直接回传全量世界。

## 4. 技能系统设计

技能系统是 wunder 最具特色的设计之一，也是本文的重点。它解决的核心问题是：如何将领域专业知识、工作流程与工具集成以模块化、自包含的方式注入智能体，同时在有限的上下文窗口内保持经济性。

### 4.1 定位与三层能力体系

wunder 的能力系统由三层组成：工具是模型直接调用的最小执行单元；技能是对工具和流程的可复用组织；MCP 是把外部资源和外部能力接入 wunder 的扩展边界。三者不是平行孤岛，而是同一能力体系的三个视角——模型首先看到的是工具面，技能负责把多个工具组织成稳定流程，MCP 负责把 wunder 之外的能力接进来。

技能的目标不是增加另一套执行引擎，而是把稳定流程沉淀成可复用资产。它适合做成固定文档生产流程、稳定资料加工流程、团队约定好的任务模板、一类工具组合的最佳实践。技能不绕开工具治理，不替代工具，而是通过 `skill_call` 工具间接调用——模型调用 `skill_call` 后，技能内容进入上下文，模型再按技能里的流程指导使用其他工具。

### 4.2 渐进式披露机制

技能系统最核心的设计哲学是"上下文作为公共资源"的节约意识。技能采用三级加载架构：第一级是元数据（name 与 description），始终驻留在模型上下文中，约 100 词规模，帮助模型判断何时使用该技能；第二级是 SKILL.md 主体，在技能被触发时加载，控制在 5000 词以内；第三级是打包资源（scripts、references、assets），按需加载，无规模限制。

这种渐进式披露确保模型上下文只承载当前任务真正需要的内容。技能创建器明确质问每一条信息："wunder 真的需要这个解释吗？这段话是否值得其 token 成本？" description 是技能的主要触发机制，"何时使用"信息必须放在 description 中而非主体里，因为主体仅在触发后加载。三类打包资源有严格职责边界：scripts 是不进上下文直接执行的可执行代码，references 是按需读入上下文的指导文档，assets 是不进上下文用于输出的文件。三类资源的严格区分体现了对上下文窗口的约束意识。

### 4.3 技能加载与调用流程

技能的核心数据结构 `SkillSpec` 包含 name（唯一运行标识）、description（触发描述）、path（SKILL.md 绝对路径）、input_schema、frontmatter、root（技能根目录）与 entrypoint（可选可执行入口）。加载流程遵循严格的去重与过滤策略，使用 seen_roots、seen_dirs、seen_names 三重 HashSet 保证不会重复加载。frontmatter 解析兼容 BOM 和 CRLF，名称提取支持多语言键名（name/名称/技能名称）并三级回退到 Markdown 标题与目录名，体现了本地化设计。

技能调用通过 `skill_call` 工具实现，执行流程包含参数解析与名称归一化、三级技能查找（用户工具绑定优先 → 全局注册表回退 → 技能根目录扫描兜底）、内容渲染与文件树构建三个阶段。关键设计是 `{{SKILL_ROOT}}` 占位符渲染——SKILL.md 中所有该占位符会被替换为技能根目录的绝对路径，让技能可以引用自身的 scripts、references 与 assets。这一设计让技能"位置无关"，无论安装到哪个用户目录，内部引用都能正确解析，是多租户分发的基础。技能目录还会被加入工具的"可读根"列表，使技能内的 scripts 与 references 能被文件工具合法读取。

对于具备 entrypoint 的可执行技能，平台提供独立的执行能力：通过 skill_runner.py 脚本以子进程方式执行，支持超时控制、JSON 输出解析与国际化错误信息，运行环境变量透传 `WUNDER_LANGUAGE`。

### 4.4 元技能自举

"女娲"与"技能创建器"构成元技能层——技能系统可以自我繁衍。技能创建器定义标准规范，女娲按标准生产领域技能，新生成的技能又可被 `skill_call` 调用。这种自举能力让技能生态可扩展，不依赖平台硬编码。

女娲是一个创建技能的技能，本身是一个复杂的多阶段工作流技能，产出物是另一个完整的 SKILL.md。其设计包含双入口分流（明确人名走直接路径，模糊需求走诊断路径）、6 个智能体并行信息采集（著作、对话、表达、他者、决策、时间线六个维度）、三重验证心智模型提炼（跨域复现、生成力、排他性）、三重检查点（调研质量、提炼确认、质量验证）与质量量化标准。女娲强调"自包含原则"——所有调研文件必须存在 skill 目录内部，复制整个 skill 目录就能独立使用。它还明确排除不可靠信息源，体现了对 LLM 幻觉的系统性防御。

### 4.5 自由度分级与诚实边界

技能创建器提出"将具体程度与任务的脆弱性匹配"的设计哲学：高自由度（文本指令）适用于多种方法都有效的任务；中等自由度（伪代码或带参脚本）适用于有首选模式的任务；低自由度（特定脚本、少参数）适用于操作脆弱、一致性关键的任务。隐喻是"悬崖间的窄桥需要特定护栏，而开阔的田野允许多条路线"。

以四个代表性技能为例：PDF 处理采用高自由度（多种库都有效，内嵌代码示例作为指导）；深度研究采用中等自由度（六阶段标准流程强约束，但工具选择弹性）；PPTX 生成器采用低自由度（量化质量基线加自动检查脚本强制执行）；女娲采用低自由度（多阶段流程加三重检查点加量化标准）。这种分级让技能的约束程度与任务脆弱性匹配，避免了"一刀切"的过度约束或约束不足。

复杂技能普遍采用"暂停-确认"检查点设计，避免模型一路狂奔到终点才发现方向错误。女娲提出"宁可生成诚实标注局限的 60 分技能，也不要生成看起来完美但实际在编造的 90 分技能"。"诚实边界"作为技能的标准组成要素，强制技能声明自身局限，这是对 LLM 幻觉的系统性防御。深度研究的单轮澄清约束（"只允许做一轮澄清问询，要把关键问题一次问全"）避免了模型陷入无限追问。

### 4.6 工具优先级编排与跨技能协作

技能不只是知识文档，更是工具编排策略。深度研究定义了明确的工具使用优先级：先看有没有更专业的技能（skill_call）→ 本地文件工具 → 知识库工具 → write_file → web_fetch → browser 兜底。这将"先本地后网络""先已有技能后重新发明"的节约原则编码进流程。技能间还可组合调用——深度研究明确要求"如果当前环境挂载了公文写作技能，在写最终研究报告前必须先调用或阅读该技能的版式约束"，形成技能生态的网络效应。

### 4.7 多租户归属与发布元数据

`skill_call` 的三级查找与 `owner@skill` 命名约定实现了多租户技能隔离：用户可以拥有私有技能，也可以调用共享技能，归属通过 ownerId 明确。运行态元数据完全来自 SKILL.md 的 frontmatter，而 `_meta.json` 是发布态产物，携带 ownerId、slug、version、publishedAt 字段用于分发与版本追踪。这种"运行元数据与发布元数据分离"的设计，让技能在未发布状态下也能直接运行，符合原型阶段保持轻量的理念。

## 5. 多智能体协作设计

### 5.1 蜂群对象模型

wunder 的蜂群系统把"一个用户目标"拆解为"多个可执行角色与任务实例"。核心对象包括：Hive（协作域容器）、Mother Agent（母蜂，Hive 入口与协调者）、Mission/Team Run（一次协作任务实例）、Team Task（分任务单元）、Subagent（子智能体执行单元）、Beeroom（Hive 在实时层的房间投影）、HivePack（蜂群级资产封装单位）与 WorkerCard（工蜂标准资产描述）。

设计遵循三层状态分离：配置态（hive、agent、角色关系）是 durable 的；运行态（team_run、task、agent activity）是 durable 加实时投影的；公开态（mission summary、board、beeroom projection）是派生态。母蜂通过 `prefer_mother` 标记或显式认领确定，认领结果通过 meta 键持久化。母蜂不一定是任务执行者，而是 Hive 的入口与协调者，派发任务时只注入最小化的协作上下文，刻意降低 prompt 噪音。

### 5.2 子智能体与蜂群的二分

这是 wunder 最具辨识度的设计。子智能体控制（subagent_control）是主智能体临时创建的新工作单元，默认不阻塞；蜂群工具（agent_swarm）是调用已存在的智能体协作，默认阻塞并汇总。两者在目标对象、会话来源、阻塞性、完成语义上完全不同：子智能体总是创建新的 child session，通过 auto-wake 机制在子任务完成后唤醒父会话；蜂群优先复用 agent 的 main session，通过 batch_send 内置 wait 循环阻塞汇总。

关键的隔离机制在于：在列出"父会话的子智能体"时，会排除 `spawned_by="agent_swarm"`、`run_kind="swarm"` 的会话，确保蜂群工蜂会话不污染子智能体面板，反之亦然。子智能体的 auto-wake 通过"等待父会话解锁 + 三层幂等守卫（mark_wake_once、mark_dispatch_once、auto_wake_consumed）+ 观察消息注入"实现真正的非阻塞协作——主智能体派发后可继续工作，子任务完成时自动唤醒父会话收敛结果。

子智能体提供三种完成模式（All、Any、FirstSuccess）与三种剩余动作（Keep、Interrupt、Close），共 9 种协作策略组合，覆盖了"全部完成、任一完成、首个成功"以及"保留其余、中断其余、关闭其余"的常见协作范式。蜂群则提供 collect 与 first_success 两种汇总策略，collect 按 priority 降序、更新时间升序拼接多行汇总。

### 5.3 Mission 运行时调度

MissionRuntime 是蜂群任务的核心调度器，内部用 Arc<Mutex> 管理并发状态。调度主循环通过 tokio::select 在队列唤醒与 600ms 轮询超时之间等待，每轮先回收已结束的 worker 再派发。派发策略的关键设计是"工具托管 vs runner 托管"分流：direct_send 与 batch_send 由蜂群工具同步驱动，parallel 等策略由 MissionRuntime 异步驱动。这避免了 runner 重新派发工具托管任务导致的模型与配置漂移。

单个 Team Run 的执行包含加载记录、置位 running 并发事件、解析父问题、等待 task 对齐、按优先级排序过滤终态 task、用 JoinSet 并发执行（并发上限为 max_parallel_tasks_per_team）等步骤。单个 Team Task 的执行包含重试循环，根据 SessionExecutionOutcome 决定状态：Success 进终态，Timeout 或 Error 在重试次数内重试否则进终态。终结时通过 reconcile_unfinished_tasks 将所有未终态 task 强制收敛，聚合性能度量（context_tokens_total/peak、model_round_total），生成汇总并发 TEAM_MERGE、TEAM_FINISH 事件。

策略守卫（SwarmPolicyGuard）在工具入口层拦截，确保协作不会因模型误判而失控：目标必须在同一 hive 内（防跨域）、递归派发深度上限（防无限嵌套）、单 team 并发 task 上限、全局活跃 team_run 上限、task 重试次数上限、超时必须为正且不超过硬上限。

### 5.4 HivePack 资产化

HivePack 是 wunder 蜂群资产化的标准协议，形式化定义为 `HivePack = HiveMeta + WorkerCard[] + SkillPack[]`。WorkerCard 包含 Metadata、PromptProfile、AbilityProfile、InteractionProfile、RuntimeProfile、HiveRef 与 Extensions，其中 runtime.approval_mode（suggest/auto_edit/full_auto）与 sandbox_container_id 是必填项。关键的安全语义是"声明≠授权"——WorkerCard 只声明工蜂期望配置，目标系统必须根据自身权限模型决定最终可启用的工具、技能与审批级别。

导入作业采用阶段机：validating（校验与解压）→ planning（解析 worker_refs 与冲突模式）→ installing（逐 worker 安装 SkillPack，通过内容寻址的 skill 去重——skill 名加源目录 SHA256——保证相同内容的 skill 只安装一次）→ creating_agents（创建 UserAgentRecord，处理名称冲突）→ activating（认领母蜂）→ completed（持久化绑定）。导出作业对称地构建 worker 目录、复制技能（full 模式完整复制，reference_only 模式只写占位）、写 hive.yaml 与 checksums.sha256 后打包成 zip。冲突处理支持 AutoRenameOnly（追加后缀）与 UpdateReplace（备份后覆盖）两种模式，并提供 rollback_import 在导入失败时尽力回滚保证系统一致性。

## 6. 实时投影与通信

### 6.1 Stream Events 契约

实时投影的基础是 StreamEventService，它通过 per-stream 锁保证事件 ID 在并发追加下严格单调递增，并通过 tail 缓存让绝大多数追加只需一次写库。每个 stream 拥有独立的锁，追加时取 max(cached_tail, storage_tail) + 1 作为 next_event_id，写库后更新缓存。所有 DB 操作通过阻塞线程池执行，避免阻塞异步 runtime。

BeeroomRealtimeService 在其上构建广播与回放：每个 stream_key 对应一个 broadcast channel（容量 512），publish 时先持久化获得单调 event_id 再广播。replay 机制支持断线重连——客户端用本地最后看到的 event_id 调用 list_events 补水，默认回放 200 条、上限 1000 条防止过载。Metrics 用 AtomicU64 统计 publish_total、replay_batch/event/failure_total、lag_recovery_total、push_latency_avg/max_ms，可观测实时投影健康度。PresenceService 实现连接级在线状态，按 connection_id 计数多端登录，断开后 TTL 90s 内仍视为 online 容忍短暂断线。

### 6.2 渠道适配与投递治理

渠道层支持飞书、QQ Bot、微信（公众号、企业微信、iLink Weixin）、WhatsApp Cloud、XMPP 等多渠道接入。ChannelAdapter trait 定义渠道适配器抽象，包含验签、解析入站、发送出站与健康检查四个能力。入站队列基于有界队列实现（容量 512，最大并发处理 8），enqueue_with_timeout 在 80ms 内无法入队则返回 busy，避免 HTTP 入站请求长时间阻塞。入站处理包含去重（TTL 120s）、账号校验、绑定解析、用户绑定、会话策略（MainThread/PerPeer/Hybrid）、限流、流式事件推送、附件下载、模型执行与出站入队等完整流程。

出站投递采用 Outbox 模式：将出站消息序列化为 ChannelOutboxRecord（status=pending），后台 worker 轮询投递，支持指数退避重试（base * 2^retry_count，capped 60s）。关键设计是"Outbox 加同步投递双模"——server 用 worker 异步投递，desktop 关闭 worker 时同步投递，同一套代码适配两种形态。限流采用令牌桶加并发控制双维度，按 channel:account_id 维度隔离。渠道层还深度集成审批：监听 orchestrator 的 approval 请求注册到 PendingApprovalRegistry，用户可在 IM 内直接回复数字审批工具调用。运行时日志实现有界环形缓冲（容量 300）加 flood 去重，防止渠道异常时日志淹没。

### 6.3 Gateway 节点协议

Gateway 实现基于 WebSocket 的网关协议，区分 Operator、Node、Channel、Unknown 四种角色。GatewayHub 持有连接表、node_id 到 connection_id 的索引、pending 调用表与单调递增的 state_version。节点调用（invoke_node）实现 RPC 风格：生成 request_id 创建 oneshot channel，发送调用消息，timeout 等待响应，handle_response 校验 request_id、connection_id、node_id 三重匹配后回填结果。

治理方面，broadcast_event 通过 try_send 检测慢客户端，连续 3 次 Full 或 Closed 加入驱逐列表批量注销；spawn_maintenance 定时广播 tick（15s）与 health（60s，含连接数、在线节点数、state_version）；节点断开时 fail_pending_for_connection 主动失败所有 pending 调用，避免调用方永久等待。state_version 单调递增为 presence snapshot/delta 提供版本基线，支持客户端增量同步。

## 7. 工程实现要点

### 7.1 性能与稳定性保障

平台在性能与稳定性上有多重保障。ThreadRuntime 的 SessionLease 是 RAII 守卫，Drop 实现自动从集合移除，确保 lease 释放不泄漏；主线程忙时自动 fork 隔离会话保证并发响应；USER_BUSY 时指数退避重试（1.5^retry_count，clamp 1-30s）避免雪崩；队列事件实时推送排队状态。存储层 per-method 同步签名加内部 block_on 简化调用方，连接池默认 64（PostgreSQL）或每次新建（SQLite，WAL 模式）。所有缓存与队列都有容量、过期或失效策略——实时缓冲、事件缓存、调试事件与工具结果都不能无界增长。

### 7.2 可观测性与回归

orchestrator 的每次阶段切换都产生事件，封装在 stream events 中，是管理侧状态页的核心数据源。MissionRuntime 的事件三路投递（monitor 时间线、stream_events 父会话流、beeroom_realtime 群组）确保协作过程可被外部观察而不必深入每个线程内部。TeamRunRecord 内建 context_tokens_total/peak 与 model_round_total 字段，让蜂群协作的成本可观测。后端改动完成后至少运行 cargo check，触及共享逻辑时运行定向 cargo test 与 cargo clippy；前端改动运行 npm run typecheck，触及实时消息链路时优先运行 chat 相关回归测试。

### 7.3 模块化与演进策略

平台面向百万行级代码演进，超过 2000 行的文件视为维护状态，新增功能拆到新文件后接回入口。目录落点优先按职责边界而非调用方便决定。orchestrator 负责完整的 turn 执行链路不依赖外部模块完成核心执行语义；ThreadRuntime 负责线程调度不重复 orchestrator 的执行逻辑；工具实现放在 services/tools/ 不在 orchestrator 中堆叠业务逻辑；API 层只做鉴权、校验和接线不承担执行逻辑。这种边界纪律确保系统在快速演进中不丧失结构清晰度。

## 8. 结论

wunder 多智能体调控平台的设计围绕一个核心命题：如何在赋予智能体强大行动能力的同时，保持认知态稳定、协作态清晰、治理态统一。本文阐述了平台的五个关键设计贡献。

第一，三层主骨架（kernel/projection/control）与三形态同核，使智能体执行、实时投影与治理接入边界清晰，服务端、桌面端、命令行端共享同一套核心语义。第二，以线程为认知单元，通过 prompt 冻结与一次性记忆注入从结构上保证认知态稳定，用户轮次与模型轮次独立计量使执行过程可观测可恢复。第三，技能系统通过渐进式披露实现上下文经济性，通过元技能自举实现生态可扩展，通过自由度分级匹配任务脆弱性，通过自包含与位置无关支持多租户分发，是平台最具特色的设计。第四，子智能体与蜂群的二分隔离，配合 auto-wake 非阻塞协作与 HivePack 资产化协议，实现了灵活且可控的多智能体调控。第五，Stream Events 契约、渠道 Outbox 双模投递与 Gateway 节点协议，构成了完整的实时投影与通信基础设施。

技能系统的设计尤其值得关注。它找到了"知识封装"与"上下文经济性"的优雅平衡点：三级加载让上下文只承载当前任务真正需要的内容；`{{SKILL_ROOT}}` 占位符让技能位置无关；女娲与技能创建器构成的元技能层让技能生态自我繁衍；"悬崖窄桥需护栏，开阔田野允多路线"的自由度分级哲学让技能约束与任务脆弱性匹配；诚实边界与检查点设计系统性防御 LLM 幻觉。这些设计使技能不仅是知识文档，更是工具编排策略与流程经验包，形成技能生态的网络效应。

未来工作将围绕分布式 lease 演进、shared hotspot 容量扩展、技能生态的跨实例分发与版本治理等方向推进。平台的核心设计——认知态隔离、单写者收敛、实时系统服务运行时、执行调度集中——为这些演进提供了稳定的基础。
