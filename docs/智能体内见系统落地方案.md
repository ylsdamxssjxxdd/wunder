# wunder 智能体内见系统落地方案

## 1. 定义

“智能体内见系统”指：

- 用户和智能体都能直接看到、读写、理解的一套文件化配置系统
- 它落在**用户私有工作目录**中，而不是额外新建一套独立根目录
- 它既是用户的 AI 资产空间，也是智能体可自我调整的可见配置空间

本方案最终采用：

1. **复用用户私有工作目录**作为内见系统根
2. **主配置统一使用 `WorkerCard-first`**
3. **用户共享层 + 用户全局层 + 智能体专属层** 三层结构
4. **不在聊天右侧工作目录区域增加切换按钮**
5. 用户只需查看自己的私有目录，就能理解当前配置与状态

---

## 2. 为什么改名为“内见系统”

之前“家目录”这个名字更强调“目录形态”，但这次目标不是单纯新增一个目录，而是建立一套完整的：

- 文件布局
- 配置协议
- 生效规则
- 恢复机制
- 智能体自修改机制

所以改为“内见系统”更准确，它强调的是：

> 这是一套对用户和智能体都可见、可理解、可调整的内部配置系统。

---

## 3. 总体设计结论

## 3.1 根目录

智能体内见系统不再单独建立 `agent_home_root`，而是直接复用：

> **用户私有工作目录（container_id=0）**

也就是说，当前用户自己的私有目录，就是内见系统根目录。

## 3.2 主配置协议

主配置统一使用：

```text
worker-card.json
```

理由：

- wunder 已经有成熟的 `WorkerCard` 协议
- 导入导出、前端工具、蜂群资产都围绕它展开
- 内见系统继续发明 `agent.json` 会造成双模型并存

因此本方案明确：

> **内见系统主配置 = WorkerCard**

## 3.3 三层结构

内见系统采用三层：

1. **用户共享层**
   - `skills/`
2. **用户全局层**
   - `global/`
3. **智能体专属层**
   - `agents/<agent_id>/`

## 3.4 灵活性

本方案允许智能体修改：

- 自己的配置
- 用户级全局配置
- 其他智能体配置

前提是：

- 都属于同一用户自己的私有资产空间
- 生效仍走稳定的 apply 机制
- 不允许通过修改文件直接越权到系统级配置

---

## 4. 与 openclaw 的关系

参考项目 `openclaw` 给我们的关键启发，不在于目录名，而在于分层原则：

| 原则 | openclaw | wunder 内见系统 |
|---|---|---|
| 工作产物与配置状态分层 | workspace / agentDir / sessions 分离 | 继续保留工作区体系，但把用户私有目录中的保留区作为内见系统 |
| 每个 agent 要有独立配置面 | `agentDir` 唯一 | `agents/<agent_id>/` 唯一 |
| 共享技能与私有技能分层 | managed / workspace / per-agent | `skills/` + `agents/<id>/skills/` |
| 配置异常不拖垮系统 | fallback / last good | `effective + diagnostics + last_good` |
| watcher 要克制 | 只盯关键文件 | 只对保留目录关键文件做签名与 dirty 检测 |

因此 wunder 不是照搬 openclaw，而是把这些原则映射到当前已有的工作区体系里。

## 5. 目录结构

## 5.1 根目录结构

```text
<用户私有工作目录>/
  skills/
    ...
  global/
    defaults.worker-card.json
    README.md
  agents/
    __default__/
      worker-card.json
      system_prompt.md
      skills/
      README.md
    agent_xxx/
      worker-card.json
      system_prompt.md
      skills/
      README.md
  .wunder/
    effective/
      global.json
      agents/
        __default__.json
        agent_xxx.json
    diagnostics/
      global.json
      agents/
        __default__.json
        agent_xxx.json
    last_good/
      global/
      agents/
```

## 5.2 各层职责

| 路径 | 作用 | 影响范围 |
|---|---|---|
| `skills/` | 用户共享技能 | 当前用户全部智能体 |
| `global/` | 用户级全局配置 | 当前用户全部智能体 |
| `agents/<agent_id>/` | 某个智能体自己的配置 | 单个智能体 |
| `.wunder/effective/` | 生效快照 | 内部 |
| `.wunder/diagnostics/` | 诊断结果 | 内部/可读 |
| `.wunder/last_good/` | 最近可回退版本 | 内部 |

## 5.3 默认智能体

默认智能体目录固定为：

```text
agents/__default__/
```

普通智能体目录固定为：

```text
agents/<agent_id>/
```

## 6. WorkerCard-first 设计

## 6.1 为什么主配置直接使用 WorkerCard

当前仓库已经明确把 `WorkerCard` 定义为单智能体标准资产协议，因此内见系统继续发明另一套主配置格式没有必要。

采用 `WorkerCard-first` 的好处：

- 导入导出与运行态配置统一
- 用户只需要理解一套模型
- 前端工具可复用现有 `WorkerCard` 解析与生成逻辑
- 后续蜂群/单体之间迁移成本更低

## 6.2 用户全局配置文件

用户全局配置文件：

```text
global/defaults.worker-card.json
```

定位：

- 不是一个真实可对话智能体
- 而是当前用户所有智能体共享的默认 `WorkerCard Profile`

它允许只写部分字段，用来提供默认值。

## 6.3 智能体个体配置文件

每个智能体主配置文件：

```text
agents/<agent_id>/worker-card.json
```

它是该智能体的正式个体配置。

## 6.4 wunder 扩展字段

标准 `WorkerCard` 已有：

- `metadata`
- `prompt`
- `abilities`
- `interaction`
- `runtime`
- `hive`
- `extensions`

wunder 现有还需要保留：

- `declared_tool_names`
- `declared_skill_names`

建议放到：

```json
"extensions": {
  "wunder": {
    "declared_tool_names": [],
    "declared_skill_names": []
  }
}
```

这样不会破坏协议主干，也能保住 wunder 现有语义。

## 7. `system_prompt.md` 设计

## 7.1 为什么仍然保留它

虽然 `WorkerCard.prompt.system_prompt` 已有标准字段，但长提示词不适合长期直接编辑 JSON。

因此每个智能体目录仍保留：

```text
agents/<agent_id>/system_prompt.md
```

## 7.2 同步规则

内见系统采用：

> **`system_prompt.md` 是提示词正文的首选可编辑源，`worker-card.json.prompt.system_prompt` 是镜像字段。**

也就是说：

- 用户/模型优先改 `system_prompt.md`
- apply 时同步回 `worker-card.json`
- 导出工蜂卡时仍导出完整 `WorkerCard`

## 7.3 冲突规则

如果 `worker-card.json.prompt.system_prompt` 与 `system_prompt.md` 同时变化：

1. 采用修改时间较新的内容
2. 自动回写同步到另一侧
3. 在 diagnostics 中记录 warning

---

## 8. 技能分层

## 8.1 三层技能

```text
1. agents/<agent_id>/skills/   # 智能体私有技能
2. skills/                     # 用户共享技能
3. 仓库内置/管理员技能          # 系统层技能
```

## 8.2 优先级

建议当前智能体运行时优先级为：

```text
agent private skills
> user shared skills
> builtin/admin skills
```

## 8.3 作用范围

| 修改位置 | 影响对象 |
|---|---|
| `agents/<id>/skills/` | 仅该智能体 |
| `skills/` | 当前用户全部智能体 |

---

## 9. 配置解析与覆盖顺序

每次智能体开始新 turn 时，按下面顺序构建生效配置：

```text
系统内建默认
-> global/defaults.worker-card.json
-> agents/<agent_id>/worker-card.json
-> 当前线程已冻结的 system prompt / 记忆
```

对应规则：

| 维度 | 解析顺序 |
|---|---|
| `metadata/runtime/interaction` | global -> agent |
| `abilities` | global 合并 agent，agent 优先 |
| prompt | `system_prompt.md` / agent WorkerCard -> 新线程冻结 |

同时保留数据库的运行态角色：

- 文件是用户可编辑期望态
- DB/meta 是兼容现有接口的运行时物化层

---

## 10. 生效语义

## 10.1 总原则

本方案不是“重启进程生效”，而是：

1. **能下一轮生效的，下一轮生效**
2. **不能安全热改当前线程的，新线程生效**
3. **绝不因为配置改动去重启后端/桌面进程**

## 10.2 生效表

| 变更 | 当前正在执行的 turn | 当前线程下一次用户消息 | 新线程 |
|---|---|---|---|
| `metadata.name/description` | 不变 | 生效 | 生效 |
| `runtime.model_name` | 不变 | 生效 | 生效 |
| `runtime.approval_mode` | 不变 | 生效 | 生效 |
| `runtime.sandbox_container_id` | 不变 | 生效 | 生效 |
| `abilities.tool_names/skills` | 不变 | 生效 | 生效 |
| `agents/<id>/skills/` | 不变 | 生效 | 生效 |
| `skills/` | 不变 | 受影响智能体下一轮生效 | 生效 |
| `global/defaults.worker-card.json` | 不变 | 受影响智能体下一轮生效 | 生效 |
| `system_prompt.md` | 不变 | 不变 | 生效 |

## 10.3 为什么提示词不是下一轮热更新

因为当前 wunder 已实现线程级 system prompt 冻结：

- `src/orchestrator/prompt.rs`

所以：

> 修改 `system_prompt.md` 不影响当前线程，只影响未来新线程。

---

## 11. 智能体能否“重启自己”

## 11.1 不能重启进程

智能体不应拥有以下能力：

- 重启后端服务
- 重启桌面应用
- 重启整个 runtime

## 11.2 可以“软重启自己”

允许的“自我重启”定义为：

> **创建一个新会话，并把它切成当前智能体的主会话**

也就是：

- 不是重启进程
- 是切新线程

这和当前仓库已有主会话/隔离会话能力是一致的。

## 11.3 何时需要软重启

当智能体修改了：

- `system_prompt.md`
- prompt 相关全局配置

并希望立刻按新提示词工作时，应：

1. 保存配置
2. 告知用户当前线程不变
3. 新建会话并切主

---

## 12. 灵活性边界

## 12.1 允许修改范围

在当前用户私有工作目录中，智能体允许修改：

- `skills/**`
- `global/**`
- `agents/**`

因此它可以：

- 改自己的配置
- 改全局配置
- 改其他智能体配置

## 12.2 为什么这样仍然合理

因为这些文件都属于：

> **同一个用户自己的 AI 私有资产空间**

不是系统公共配置，也不是其他用户空间。

## 12.3 但必须保留的治理能力

为了避免灵活性过高带来混乱，必须补：

1. **审计日志**
2. **diagnostics**
3. **last_good 回退**
4. **按影响面 apply**

也就是说：

> 文件权限可以很灵活，但生效必须稳定、可追踪、可恢复。

---

## 13. 后端落点设计

建议新增：

```text
src/services/inner_visible/
  mod.rs
  layout.rs
  worker_card.rs
  sync.rs
  diagnostics.rs
  impact.rs
```

职责：

| 文件 | 职责 |
|---|---|
| `layout.rs` | 保留目录路径解析 |
| `worker_card.rs` | WorkerCard 读取、校验、默认值合并 |
| `sync.rs` | 文件 -> effective -> DB/meta |
| `diagnostics.rs` | 问题收集、落盘、恢复 |
| `impact.rs` | 计算一次修改影响哪些 agent |

同时保留内部快照：

```text
.wunder/effective/
.wunder/diagnostics/
.wunder/last_good/
```

## 13.1 apply 流程

每次新 turn 开始前，建议：

1. 检测用户私有目录下保留区是否变化
2. 计算影响面
3. 解析 `global/defaults.worker-card.json`
4. 解析 `agents/<id>/worker-card.json`
5. 同步 `system_prompt.md`
6. 构建 effective
7. 回写 DB/meta
8. 清理缓存
9. 落 diagnostics / last_good

## 13.2 DB / meta 同步目标

| 文件内容 | 同步目标 |
|---|---|
| `metadata.name` | `user_agents.name` / default agent meta |
| `metadata.description` | `description` |
| `prompt.system_prompt` | `system_prompt` |
| `abilities.tool_names/skills` | `tool_names` + declared data |
| `runtime.model_name` | `model_name` |
| `runtime.approval_mode` | `approval_mode` |
| `runtime.sandbox_container_id` | `sandbox_container_id` |
| `interaction.preset_questions` | `preset_questions` |

## 13.3 默认智能体特殊处理

默认智能体 `__default__` 继续走现有 default agent meta 逻辑，不强行伪装成普通 `user_agents` 记录。

## 14. 工具链改造

当前 agent turn 默认只在当前 `workspace_id` 下解析文件。  
这意味着如果智能体运行在 agent 工作目录中，它看不到用户私有工作目录中的内见系统。

因此必须补：

1. 当前 agent 工作目录
2. 当前用户私有工作目录

这两个根都挂入当前 request 的文件工具可读写根中。

这样智能体才能通过现有：

- 读文件
- 写文件
- 编辑文件
- 搜索内容
- apply_patch

直接操作：

- `skills/`
- `global/`
- `agents/`

不需要新增专用“修改智能体配置”工具。

## 15. 前端设计

## 15.1 明确取消的设计

本方案**明确取消**以下设计：

- 在聊天右侧工作目录区域增加“工作 / 家”切换按钮
- 在 `MessengerRightDock.vue` 中增加新的文件面切换控件

原因：

1. 会让工作目录区承担过多概念
2. 会增加聊天页主界面认知负担
3. 用户已经有自己的私有目录，只要看私有目录就能理解当前配置

## 15.2 新的前端原则

前端不为“内见系统”再造一个聊天侧边栏入口，而是采用：

> **用户通过自己的私有目录直接查看内见系统。**

也就是说：

- `skills/`
- `global/`
- `agents/`

这些目录就直接存在于用户私有目录中。

## 15.3 用户如何查看当前配置

用户进入自己的私有目录后，就可以直接看到：

- 共享技能：`skills/`
- 全局配置：`global/defaults.worker-card.json`
- 每个智能体的个体配置：`agents/<agent_id>/worker-card.json`

因此：

> 不需要额外切换按钮，目录结构本身就是界面。

## 15.4 前端最小改动建议

前端层面只需要：

1. 确保用户能顺畅进入自己的私有目录
2. 确保私有目录中文件可正常预览与编辑
3. 必要时补充一个只读帮助文档，如：
   - `global/README.md`
   - `agents/<id>/README.md`

也就是说，前端重点不是“增加切换控件”，而是“让目录本身足够可理解”。

## 16. 异常恢复机制

## 16.1 需要处理的异常

| 异常 | 处理 |
|---|---|
| `global/defaults.worker-card.json` 缺失 | 自动补空模板 |
| `agents/<id>/worker-card.json` 缺失 | 从当前 DB/meta 重新种子 |
| `worker-card.json` JSON 非法 | 回退 `last_good` |
| `system_prompt.md` 缺失 | 用当前 prompt 镜像重建 |
| `skills/<...>/SKILL.md` 非法 | 跳过该技能 |

## 16.2 diagnostics

建议分别落：

```text
.wunder/diagnostics/global.json
.wunder/diagnostics/agents/<agent_id>.json
```

## 16.3 last_good

建议分别保留：

```text
.wunder/last_good/global/defaults.worker-card.json
.wunder/last_good/agents/<agent_id>/worker-card.json
.wunder/last_good/agents/<agent_id>/system_prompt.md
```

## 17. 与现有设置页 / 导入导出的关系

## 17.1 设置页/API 改动后

当用户通过设置页/API修改智能体时，应同步回：

- `agents/<id>/worker-card.json`
- `agents/<id>/system_prompt.md`

## 17.2 文件改动后

当用户或智能体直接修改内见系统文件时，应同步回：

- `user_agents`
- default agent meta
- 其他运行态 meta

## 17.3 导入导出工蜂卡

导入时应直接种子：

- `worker-card.json`
- `system_prompt.md`

导出时优先从 effective 或最新同步结果导出，保证导出的卡片与内见系统一致。

## 18. 分阶段实施建议

## Phase 1：文档与模型统一

- 将方案正式命名为“智能体内见系统”
- 明确根为用户私有工作目录
- 主配置统一到 `WorkerCard-first`
- 删除聊天右侧切换设计

## Phase 2：后端 apply 骨架

- 新增 `inner_visible` 服务模块
- 解析 `global/` 与 `agents/`
- 构建 effective / diagnostics / last_good

## Phase 3：工具可见性

- 当前 agent turn 同时挂载用户私有目录根

## Phase 4：目录可读性

- 种子 `README.md`
- 保证用户进入私有目录即可理解内见系统结构

## Phase 5：软重启能力

- 支持修改提示词后新建会话切主

## 19. 最终建议

最终建议如下：

> **将 wunder 的智能体内见系统定义为“用户私有工作目录中的 AI 资产保留区”，统一采用 `WorkerCard-first`：共享技能放 `skills/`，用户全局默认放 `global/`，智能体个体配置放 `agents/<agent_id>/`，不再在聊天右侧工作目录区域加入切换按钮。**

并明确：

- 智能体可以修改自己的、全局的、其他智能体的文件
- 生效以“下一轮 / 新线程”为准，不通过进程重启实现
- 用户只要查看自己的私有目录，就能知道当前配置情况

这套方案比“单独新建家目录 + 聊天侧边栏切换”更贴合 wunder 当前结构，也更符合“目录本身就是配置界面”的目标。
