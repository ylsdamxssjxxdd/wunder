# 基础工具优化方案（命令/读取/写入/补丁/搜索）

## 1. 背景与目标

wunder 在编程场景下的核心体验问题，不是单一工具能力不足，而是“工具执行链路一致性”不够：

- 同一工具在本地路径与沙盒路径行为不一致（流式、超时、输出截断、权限判断语义不同）。
- 关键工具缺少统一编排层（审批、重试、降级、观测指标未统一）。
- 搜索与编辑路径没有形成“快路径 + 精准路径”的分层协作。

本方案目标：

1. 让命令、读取、写入、补丁、搜索五类基础工具形成一致的执行体验与返回语义。  
2. 提升“首字节速度、整体耗时、内存占用、token 占用稳定性”。  
3. 建立可持续 10 年维护的模块化结构与回归测试基线。

---

## 2. 现状差距（Wunder vs Codex）

## 2.1 命令执行

- Wunder 现状
  - 非沙盒路径支持流式输出与 output guard（头尾保留+截断元信息）。
  - 沙盒路径以 `wait_with_output` 为主，完成后一次性返回，流式体验不一致。
  - 命令按 `content.lines()` 逐行执行，多行脚本/控制流不稳定。
  - `allow_commands` 基于字符串前缀匹配，语义粗糙。
  - 超时参数本地/沙盒类型不一致（浮点秒 vs 整数秒）。
- Codex 参考
  - 统一 runtime + orchestrator，审批、沙盒、重试、输出流都走同一控制面。
  - 命令审批键做 canonicalization，可缓存审批结果，减少重复确认。
  - 输出既有 delta 事件，也有上限截断，避免大输出卡死/撑内存。

## 2.2 读取

- Wunder 现状
  - 支持多文件+行范围读取，含 binary/image 守卫，基础能力扎实。
  - 但读取模式偏“切片”，缺少代码块/缩进感知读取。
- Codex 参考
  - `read_file` 支持 slice + indentation 模式，便于低 token 成本读取结构化代码块。
  - `read_file/list_dir/grep_files` 在 spec 明确并行支持。

## 2.3 写入

- Wunder 现状
  - `write_file` 主要是直接覆盖写，缺少原子写（tmp+rename）和前置条件保护（mtime/hash）。
  - 与补丁路径并行存在，默认编辑路径不够收敛。
- Codex 参考
  - 主要通过 `apply_patch` 承担结构化编辑；命令内触发补丁会被拦截回补丁链路。

## 2.4 补丁

- Wunder 现状
  - 已具备：补丁语法解析、输入/文件操作数限制、失败回滚。
  - 但补丁入口未与命令执行深度统一（命令中补丁语义未强拦截并复用同一审批/编排）。
- Codex 参考
  - `apply_patch` 可被 shell/unified_exec 拦截并统一进入补丁 runtime + orchestrator。
  - 对 direct/heredoc 形态有更强校验与 verified parse。

## 2.5 搜索（新增专项）

- Wunder 现状
  - `search_content` 采用 Walk + Rust 正则逐文件扫描，返回命中行和上下文，信息丰富。
  - 当前问题：
    - 大仓库下缺少“外部高性能检索快路径”（如 `rg`），扫描成本高。
    - 缺少全局时间预算/中断策略，复杂目录可能长时间占用线程。
    - `file_pattern` 非法时会静默失效（退化为不过滤），结果不可预期。
    - `max_files` 计数发生在过滤前，可能“提前触顶”导致漏结果。
    - 对二进制文本污染缺少专门守卫（读取阶段以 lossy 解码为主）。
- Codex 参考
  - `grep_files` 直接调用 `rg`，有超时、结果上限、错误显式返回，文件级检索非常快。
  - 搜索工具并行属性明确，调度更稳定。

## 3. 目标架构（统一工具运行内核）

建设统一 `Tool Runtime Orchestrator`，五类工具共用：

1. **统一输入归一化层**
   - 参数类型统一（尤其 timeout/workdir/path）。
   - 明确错误码与 hint，禁止静默降级。

2. **统一执行编排层**
   - 审批、沙盒选择、重试策略、降级策略在同一层处理。
   - 本地与沙盒运行时共享同一输出协议（delta + summary + meta）。

3. **统一输出与观测层**
   - 标准化 `meta`：耗时、截断、扫描文件数、命中数、是否触顶、错误分类。
   - 指标上报：P50/P95 首字节延迟、总耗时、峰值内存、token 占用。

4. **统一安全策略层**
   - 命令权限采用“规范化命令 + 前缀规则 + 缓存审批键”。
   - 文件写入采用 allow/deny + 原子写 + 冲突检测。

## 4. 分阶段实施节点（明确排期）

## M0（第 0-3 天）：基线测量与开关准备

- 交付
  - 建立工具性能基线脚本：命令、搜索、补丁各 3 组典型场景。
  - 增加开关：`tool_runtime_v2_enabled`、`search_engine_v2_enabled`。
- 验收
  - 输出 baseline 报告（P50/P95、失败率、峰值内存、token 占用）。
  - 新开关默认关闭，不影响现网行为。

## M1（第 1-2 周）：命令执行一致性（P0）

- 交付
  - 合并本地/沙盒执行返回协议：统一流式 delta、统一截断元信息。
  - 沙盒命令执行改为流式读取（替代纯 `wait_with_output`）。
  - `content.lines()` 逐行执行改为单脚本执行；保留 `commands[]` 显式多命令模式。
  - 统一 timeout 语义（统一为毫秒或统一为浮点秒，需全链路一致）。
- 验收
  - 相同命令在本地/沙盒返回字段一致（结构一致率 100%）。
  - 10MB 输出场景不 OOM，且有明确截断标记。
  - 首字节延迟较基线下降 30%+（沙盒模式）。

## M2（第 2-3 周）：搜索引擎分层（P0）

- 交付
  - 新增双引擎：
    - Fast Path：`rg` 文件级/行级候选检索。
    - Rich Path：在候选集上补充高亮与上下文片段。
  - 增加 `timeout_ms`、`max_matches`、`max_candidates`、`engine=auto|rg|rust` 参数。
  - 修复 `file_pattern` 非法静默问题，改为显式错误。
  - `max_files` 计数改为“过滤后文件”计数，语义可预期。
  - 增加二进制守卫，避免乱码内容进入上下文。
- 验收
  - 中大型仓库（10 万+ 文件）检索 P95 耗时下降 50%+。
  - 参数非法时返回标准错误码，不再静默退化。
  - 命中质量与旧版一致或更优（抽样回归通过率 95%+）。

## M3（第 3-4 周）：写入与补丁收敛（P1）

- 交付
  - `write_file` 增加原子写（tmp+fsync+rename）与冲突检测（mtime/hash 可选）。
  - 规范编辑主路径：默认推荐 `apply_patch`，`write_file` 作为兜底。
  - 命令执行中识别 `apply_patch` 语义并强制路由到补丁工具。
  - 补丁审批与执行接入统一 orchestrator。
- 验收
  - 并发写入冲突可检测并返回可恢复错误。
  - 命令中补丁请求不再绕行写文件路径。
  - 补丁失败回滚成功率 >= 99.9%（压测场景）。

## M4（第 4-6 周）：读取语义增强（P1）

- 交付
  - 新增 `read_file` 的 indentation/block 模式。
  - 增加“函数/类锚点读取”与 `max_lines` 保护。
  - 统一读取输出元信息（read_lines/total_lines/complete 等）。
- 验收
  - 代码阅读类任务 token 占用下降 20%+。
  - 读取失败错误码标准化，前端可稳定消费。

## M5（第 6-8 周）：测试与治理固化（P2）

- 交付
  - 扩展回归测试矩阵：命令/搜索/写入/补丁/读取全链路。
  - 增加混沌场景：超时、巨量输出、权限拒绝、并发写冲突、沙盒抖动。
  - 补充运维看板：失败率、超时率、触顶率、平均 token 占用。
- 验收
  - `cargo check`、`cargo clippy`、关键测试集全绿。
  - 连续压测 72h 无崩溃、无明显内存泄漏趋势。

## 5. 关键设计细节（实施要求）

## 5.1 命令执行

- 权限匹配从 `starts_with` 升级为：
  - 解析后的命令前缀规则（程序 + 前置参数）。
  - canonical command 作为审批缓存键。
- 输出策略统一：
  - stdout/stderr 独立 cap + 聚合 cap。
  - delta 事件数量上限，防止事件洪泛。
  - 截断元信息进入 `meta.output_guard`。

## 5.2 搜索工具

- 统一输出模型：
  - `matches`（兼容）
  - `hits`（结构化）
  - `meta.search`（engine、scanned_files、timeout_hit、limit_hit、elapsed_ms）
- 默认策略：
  - `engine=auto`：优先 `rg`，失败回退 rust。
  - 先候选、后精排，避免全量逐文件读入。

## 5.3 写入与补丁

- 写入必须支持：
  - 原子写、目录自动创建、回滚策略、冲突前置条件。
- 补丁必须支持：
  - 命令通道拦截（避免双写路径）。
  - 与审批、沙盒、重试走统一 orchestrator。

---

## 6. 指标与验收门槛

- 体验指标
  - 命令首字节延迟（P95）下降 >= 30%（沙盒）。
  - 搜索耗时（P95）下降 >= 50%（中大型仓库）。
  - 读代码 token 占用下降 >= 20%。
- 稳定性指标
  - 工具失败率下降 >= 40%。
  - 超时率下降 >= 30%。
  - 大输出/大补丁场景无 OOM。
- 工程指标
  - 五类工具统一错误码与元信息字段。
  - 回归测试覆盖关键路径并纳入 CI。

## 7. 风险与回滚策略

- 风险
  - 新旧执行引擎切换期间行为不一致。
  - `rg` 在部分环境不可用导致功能降级。
  - 命令审批规则收紧可能引发历史提示词失配。
- 控制
  - 全部改造走特性开关，支持灰度用户与一键回滚。
  - `rg` 不可用时自动回退 rust 引擎并上报告警。
  - 对外 schema 保持兼容，新增字段只增不减。

## 8. 模块落点建议

- `src/services/tools/runtime/*`：统一运行时抽象（新增）。
- `src/services/tools/command/*`：命令执行具体实现（从 `tools.rs` 拆出）。
- `src/services/tools/search/*`：搜索双引擎与结果聚合（从 `search_content_tool.rs` 细分）。
- `src/services/tools/patch/*`：补丁编排、审批适配层（保持 `apply_patch_tool.rs` 轻入口）。

说明：`src/services/tools.rs` 已处于大文件维护态，新增能力应以“拆模块 + 保留最小入口”方式推进。

## 9. 当前落地状态（2026-03-16）

- 已落地：`execute_command` 预算护栏（`time_budget_ms/output_budget_bytes/max_commands`，支持 `budget` 对象）；预算会回传到 `budget` 与 `meta.output_guard.effective_total_bytes`。
- 已落地：`execute_command/write_file/apply_patch` 的 `dry_run` 预演模式。  
  `dry_run=true` 时只返回执行计划/变更摘要，不进行真实落盘或命令执行。
- 已落地：`search_content/read_file` 同步接入预算协议与 `dry_run` 预演。  
  搜索支持 `budget.max_files/max_matches/max_candidates + time_budget_ms/output_budget_bytes`，读取支持 `max_files/time_budget_ms/output_budget_bytes`；预算触发会在 `meta.search/meta.read` 显式标记。
- 已落地：基础工具失败结果统一补充 `error_meta`（`code/hint/retryable/retry_after_ms`），便于前端与模型基于错误码恢复。
- 已落地：sandbox 命令路径支持预算输出护栏与 `dry_run`，并与本机命令执行结果结构保持一致。
