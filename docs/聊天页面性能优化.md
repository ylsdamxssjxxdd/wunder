# 聊天页面性能优化方案

> 适用范围：`frontend/src/views/MessengerView.vue` 及其聊天消息、实时投影、Markdown、工作流和历史加载链路。
>
> 核心约束：一个用户可能在同一个会话中持续执行很长时间；消息数量、单条正文、工具输出、引用、附件和子智能体记录都可能持续增长。系统必须保证会话增长不会把首开、滚动、输入、流式输出或内存占用拖入不可用状态。

## 1. 目标与非目标

### 1.1 体验目标

1. 打开长会话时，先可见并可操作，再逐步补齐非关键内容；不能等待全部历史 Markdown、工作流和附件解析完成。
2. 滚动时每帧只做常数级位置计算和少量 DOM 更新；不得扫描整段历史、解析 Markdown、读取全部行高或重建大组件树。
3. 流式正文只影响当前消息正文，不重渲染消息列表、头像、工作区、会话列表和历史工具卡片。
4. 单会话历史无限增长时，浏览器中实际挂载的 DOM、已展开的复杂子树、缓存和异步任务必须有严格上限。
5. 所有受支持浏览器使用同一套聊天行为与视觉规则，并在集成显卡、软件渲染等较弱环境中保持可读、可输入、可滚动；不得按浏览器或设备降低功能、内容或装饰。

### 1.2 不以性能名义破坏的语义

- 线程 system prompt、长期记忆和后端事件投影语义不由前端缓存决定。
- 消息顺序、轮次归属、工具状态和终态仍以 canonical runtime projection 为权威；前端只优化投影消费。
- 历史消息不能因裁剪而丢失：裁剪的是浏览器热数据和 DOM，不是服务端 durable state。
- 不允许以全局关闭动画、强制重建聊天页或全量截断历史来掩盖性能问题。

## 2. 当前瓶颈模型

聊天页的成本不是单一的“消息条数”，而是下列乘积：

```
会话成本 = 可挂载气泡数
         x 每条气泡的正文 DOM / Markdown 复杂度
         x 工作流、子智能体、引用、附件等扩展子树
         x 流式与实时事件频率
         x 滚动、布局测量、图片解码和样式重绘次数
```

因此仅有虚拟列表不够。若一个可视窗口内有多条超长 Markdown、展开的终端输出或大量图片，仍会卡；若每个 token 都触发父页面更新，即使只有一条消息也会卡。

当前已有的关键基础包括：稳定渲染 key、消息虚拟窗口、高度缓存、尾部活跃消息保留、正文 content clock、Markdown 流式节流和 `ResizeObserver`。后续优化应复用这些边界，不新建第二套聊天状态或渲染源。

## 3. 分层设计

### 3.1 Durable 层：无限历史留在后端，前端按页取用

- 服务端历史接口必须使用稳定游标和确定排序，不返回未分页的完整会话。
- 初次打开只取“尾部首屏页”：当前 Web 默认 80 条、桌面默认 64 条，用户向上滚到阈值才加载更早一页；后续继续以消息数与序列化字节双重限制收敛到 20 至 40 条。
- 历史记录至少提供摘要元数据：稳定消息 ID、角色、时间、状态、正文摘要、是否含工具/附件/引用、内容版本。列表阶段不内联大工具结果、图片数据或完整调试数据。当前历史页仍返回完整可见消息以保持旧客户端兼容，后续以摘要字段逐步替代。
- 单条超大正文、工具输出和附件预览使用独立按需接口或已有 workspace 引用；消息列表不把大二进制和完整结果塞进热路径。已提供按 `history_id` 读取完整可见消息的接口，供摘要/分段展开接入。
- 前端保存有限的已加载页窗口。离开会话或超过容量时，释放远端页的 Markdown HTML、图片 object URL、工具详情和测量缓存；保留游标、摘要和滚动锚点即可重新补水。

### 3.2 投影层：消息外壳稳定，内容版本局部更新

- `runtimeProjection` 是唯一消息真相；任何 send、watch、replay 和历史 hydration 都不得另建会导致全列表同步的消息数组。
- 每条消息的 shell 使用稳定 `messageId` key。正文、工具项、引用和附件各自维护内容版本；内容更新时原地更新对应子组件，不替换整条 renderable message。
- 流式 delta 先在非响应式缓冲合并，按帧或短定时器写入当前消息的 content clock；终态强制 flush。不得把每个 delta 写成全局 projection version 或全数组 clone。
- 高频事件仅处理可见会话；非当前会话只更新最小的未读数、摘要和运行态，切换时再按 event cursor 补水。
- debug、shadow 对照、统计汇总禁止进入 token 热路径；仅在显式诊断开关下、采样后或空闲时间运行。

### 3.3 DOM 层：窗口化、分级挂载、可恢复锚定

消息虚拟窗口应按“渲染成本”而不是只有消息数决定是否启用。即使消息数较少，只要存在大 Markdown、图片或工具树，也可以提前窗口化；同时避免短且稳定列表被不必要的虚拟化复杂度影响。

窗口必须包含：

- 当前滚动可见范围；
- 有限 overscan，建议按屏幕高度折算，不随历史长度增长；
- 尾部正在流式或刚结束的有限活跃消息；
- 用户定位、搜索命中或展开详情时的临时 pin 项。

每种 pin 都必须有数量和超时上限。不要把“最后 N 条消息永久常驻”作为无限会话策略：随着任务持续数小时，这仍会变成长期泄漏。

滚动锚定使用稳定消息 key 加相对偏移，不依赖数组下标或墙钟时间。上拉补历史时记录补入前的锚点高度，DOM 更新后只补偿高度差；用户阅读历史时绝不强制贴底。自动贴底只在用户本来距底部很近且未主动向上浏览时启用。

### 3.4 气泡内容：按复杂度分级，而不是一律完整挂载

| 内容类型 | 首次进入可视区 | 非可视/历史状态 | 展开后的限制 |
| --- | --- | --- | --- |
| 普通短文本 | 直接文本或已缓存 HTML | 可保留轻量 HTML | 无额外处理 |
| 长 Markdown | 先渲染正文骨架或纯文本，空闲时生成增强 HTML | 不解析或只保留摘要 | 代码高亮、KaTeX 按块懒处理 |
| 超长正文 | 首屏片段 + 明确“展开剩余内容” | 仅摘要和原文引用 | 分段增量渲染，不能一次插入巨型 DOM |
| 工具工作流 | 只挂载简短状态摘要 | 不挂载详情组件 | 一次最多展开有限条，旧详情自动折叠 |
| 命令/日志输出 | 显示尾部、行数和状态 | 不保存完整 DOM | 使用行窗口；原始全文通过详情/文件读取 |
| 引用、子智能体、思考过程 | 默认折叠 | 仅摘要 | 展开后按需获取和虚拟化 |
| 图片、音频、附件 | 固定尺寸占位 | 不请求资源 | `loading=lazy`、异步解码、离屏取消请求 |

展开不是免费操作。每个会话要维护“已展开复杂块”LRU，例如只保留最近有限个展开工作流/日志视图；超出后自动收起最久未操作的块，但保留用户可再次展开的状态摘要。这样长任务不会因为用户曾打开过很多工具详情而累积数千个节点。

Markdown 缓存以 `(messageId, contentVersion, renderMode)` 为键，缓存值必须设条目数和字节数上限。淘汰时优先释放非当前会话、非可视区、非流式消息的 HTML。缓存命中只复用安全的已净化结果；不能因缓存绕过内容净化。

### 3.5 布局与绘制：默认低成本并保证滚动帧

- `scroll` 使用 passive 监听和 `requestAnimationFrame` 合并；滚动帧只更新滚动位置、按钮状态和虚拟窗口边界。
- 不能在每一帧调用 `querySelectorAll`、`offsetHeight`、`getBoundingClientRect` 或同步读取 `scrollHeight` 后又写样式。行高由 `ResizeObserver` 在实际内容变化时更新；不支持时在停止滚动后的短暂静默期补测。
- 高度缓存缺失时用保守估算值；测得真实高度后只修正受影响窗口和锚点，避免全历史重算。
- 气泡默认使用纯色/低成本背景和边框。避免在消息滚动层中使用大面积透明渐变、`filter`、`backdrop-filter`、多重大阴影、无限动画和无界 `will-change`。
- 图片、表格、代码块设置宽高/最大尺寸和内部滚动，防止内容加载后反复撑开整页。
- 消息区默认不使用无限动画；交互动效限定为短时、局部且只使用低成本属性。所有浏览器均使用相同规则，不根据浏览器、设备性能或帧率切换聊天行为、内容形态或视觉装饰。

## 4. 单一长期会话的容量策略

以下是必须实现的硬边界，而不是可选优化：

1. **DOM 上限**：仅保留可视窗口、有限 overscan、有限活跃/pin 项的消息壳。超出窗口的气泡 DOM 必须卸载。
2. **详情上限**：工具卡、终端、引用、子智能体和思考区按需挂载并使用 LRU 回收。展开状态是 UI 派生状态，不等于永久 DOM。
3. **内容上限**：正文、日志和工具结果超过阈值时分块展示；完整内容保留在服务端事件/文件引用，前端不创建同等大小的 HTML 副本。
4. **缓存上限**：Markdown HTML、高度、附件 URL、语法高亮结果、图片预览均需容量与失效策略。高度缓存只保留当前会话已加载窗口与少量邻近页的稳定 key。
5. **任务上限**：每条消息的图片解码、Markdown 增强、资源水合、摘要生成均应可取消；气泡离开窗口、会话切换或组件卸载时取消对应任务。
6. **事件上限**：流式缓冲按消息/会话限量，最终事件可覆盖或补齐缓冲；不能让不可见会话持续堆积完整 delta 文本。

当会话达到较大规模时，前端不应尝试“完整恢复”。恢复顺序应为：会话摘要和尾部消息 -> 当前运行消息与其最小 workflow 状态 -> 当前视口附近页 -> 用户明确请求的历史详情。这样任务持续很久也能秒开到可用状态。

## 5. 首开、切换、滚动和流式的执行时序

```
打开会话
  -> 获取会话摘要、cursor、尾部首屏页和当前运行态
  -> 立即挂载消息 shell / 已缓存纯文本
  -> 恢复稳定 key 锚点或贴底状态
  -> 首屏可交互
  -> 空闲时增强可视气泡 Markdown，并按需水合附件/工具详情

用户滚动
  -> rAF 合并 scrollTop
  -> 更新虚拟窗口与导航按钮
  -> 不做全量测量、Markdown 解析或资源扫描
  -> 停止滚动后补测旧浏览器行高；接近顶部再请求上一页

模型流式输出
  -> 仅写当前 message content buffer
  -> 节流推进该 message content clock
  -> 当前用户贴底时小批量滚到底；阅读历史时保持锚点
  -> final 时 flush、记录终态，再按需增强 Markdown
```

## 6. 分阶段实施顺序

### P0：建立真实基线与防回退

- 增加真实 `MessengerView` 的 Playwright 场景：长会话打开、向上分页、快速往返滚动、持续流式输出、展开多个工具详情、切换会话再返回。
- 场景至少包含数百条消息、长 Markdown、工具输出、图片占位和运行中消息；不能只使用简化的全量 `v-for` 测试页。
- 采集首屏可交互时间、滚动最大帧间隔、长任务数量、可挂载消息数、已展开复杂块数、JS heap、DOM 节点数和网络请求数。

当前已建立直接挂载生产 `MessengerView` 的 Playwright 防回退场景，覆盖长历史、补页、快速滚动、连续正文更新、工具详情容量和会话切换返回，并采集浏览器可用的上述指标。真实性能百分比结论仍以同一真实设备的优化前后重复采样为准。

### P1：首开与历史分页

- 把会话打开收敛为尾部首屏页 + cursor，禁止一次性 hydration 全量历史。
- 建立消息摘要 DTO 与大内容按需获取边界；补齐 scroll anchor 和取消机制。
- 将非关键 Markdown 增强、资源水合和详情预取排到首屏可交互之后的 idle 阶段。

### P2：复杂气泡按需挂载

- 将 `MessageToolWorkflow`、子智能体、引用、思考和大结果改为摘要优先、展开挂载、LRU 自动回收。
- 为命令输出和超长正文采用分段/行窗口，不再把完整内容直接写入页面。
- 缓存所有 HTML/预览/对象 URL，并建立可观测的容量与淘汰计数。

### P3：长期会话内存治理

- 定义并落实 DOM、详情、HTML 缓存、高度缓存、资源 URL、异步任务和事件缓冲的上限。
- 会话切换和页面卸载时统一释放 observer、timer、AbortController、object URL 和非当前会话热缓存。
- 以默认有界 DOM、按需挂载和低成本绘制覆盖不同设备性能，不引入设备或浏览器专属模式。

### P4：持续回归与灰度

- 每次修改 Markdown、实时事件、工作流或虚拟窗口，都运行定向性能回归和类型检查。
- 首先观察长会话打开、长任务持续输出、不同系统浏览器和桌面端 SQLite 场景；发现长帧时记录阶段、消息复杂度和已挂载子树数量，而不是只记录总消息数。

## 7. 验收与基线

性能结论必须遵守 [性能标准](性能标准.md)：同环境、同数据规模、预热后至少多次采样并保留前后对比报告到 `docs/性能基线/`。未采集真实对比数据前，本文中的措施只能称为设计目标，不得宣称已提升百分比。

建议记录以下门槛，并在首次基线后按真实设备校准：

| 链路 | 指标 | 验收方向 |
| --- | --- | --- |
| 打开长会话 | 首个消息 shell 可见、可输入时间 | 不等待全历史和非关键详情 |
| 长会话滚动 | 最大帧间隔、长任务次数 | 持续滚动不出现成片长帧 |
| 流式输出 | delta 到可见文本延迟、父级重渲染次数 | 只刷新当前正文，不阻塞输入 |
| 上拉历史 | 请求次数、锚点误差、重复页数 | 分页、幂等、阅读位置稳定 |
| 长期任务 | DOM 节点、HTML 缓存、heap 峰值 | 随历史增长保持受控而非线性累积 |
| 跨设备浏览器 | 首开、滚动帧与功能一致性 | 使用同一行为与视觉规则，弱环境仍可用 |

回归测试还必须覆盖：断线重连后的历史补水、工具展开后高度变化、图片加载、流式末尾补齐、向上分页与用户不贴底状态。这些是最容易破坏滚动锚点和虚拟高度缓存的边界。

## 8. 禁止事项

- 不在 `computed`、`watch` 或 scroll 回调中遍历全量消息并解析正文、统计工具项或深比较对象。
- 不使用数组 index 作为消息 key，不通过强制修改大面板 `:key` 解决不同步。
- 不让全局 store 订阅每个 token，也不让非当前会话接收完整 Markdown 和工具详情。
- 不用 CSS 隐藏大量已挂载复杂 DOM 当作虚拟化；隐藏节点仍会占内存、样式和事件成本。
- 不用无限缓存、无限 pin、无限 observer 或不取消的异步任务换取“再次打开更快”。
- 不为任何浏览器或设备引入另一套消息语义、内容形态或视觉模式；性能问题应通过通用的数据、渲染、布局和资源生命周期优化解决。

## 9. 结论

聊天页要在单一会话长期增长时保持流畅，关键不是把“更多内容塞进更快的列表”，而是持续区分 durable history、运行时投影、可视 DOM 和可丢弃 UI 派生状态。历史可以无限，浏览器热集必须有限；正文可以很长，单次渲染和单帧工作必须有限；实时事件可以高频，受影响的组件范围必须有限。所有实现都应围绕这三个有限性约束验收。
