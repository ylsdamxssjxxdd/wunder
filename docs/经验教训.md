# 经验教训（每次一条）
- 2026-07-08 聊天页刷新冷启动时不能为了“保持当前会话温热”直接启动 watch；如果本地没有 warm detail/cache 或可靠 last event cursor，WebSocket watch 会以 `after_event_id=0` 接收历史事件，表现成整段聊天历史被逐条重放。冷会话必须先 hydrate detail 确认 `running` 和 transcript，再决定是否 watch；watch cursor 也要取 projection、runtime local、runtime remote 和消息尾部事件 id 的最大值。
- 2026-07-08 聊天智能体循环区域、计划面板这类用户手动展开状态不能绑定到流式内容、投影合并或弱 id 替换时会变化的 render key；应使用会话 + 消息身份作为主状态 key，并保留模型轮次、旧渲染 key、首个工具项等窄别名做迁移。多别名同时命中缓存时必须按最近写入状态恢复，不能简单取第一个命中项，否则用户刚展开后会被早期默认折叠状态覆盖。
- 2026-07-08 聊天历史会话冷打开时，已结束线程的消息权威来源必须是后端 canonical transcript，不能先把 stream events snapshot 灌入实时投影再等待 transcript 覆盖；长会话里事件快照可能按到达/补水顺序创建 turn 和 `createdSeq`，即使后续刷新也会把旧顺序残留放大成老消息跑到底部、新消息插到中间。运行中会话才允许用事件快照恢复投影，已结束会话应直接用 transcript authoritative snapshot，并在 canonical 覆盖时重写 message/turn 排序字段。
- 2026-07-08 长 Markdown 气泡和消息列表上的高频交互不要用父级 `.bubble:hover 子按钮` 这类选择器控制子元素显隐；鼠标经过正文会让整块大 DOM 反复参与 hover 匹配、opacity transition 和指针命中切换，滚动时容易造成可感知卡顿。复制、反馈等小按钮应保留稳定布局和低调常驻态，只让按钮自身 hover/focus 改变视觉状态。
- 2026-07-08 Vue 单文件组件里如果通过内联 `style` 动态拼接 CSS `animation` 名称，对应 `@keyframes` 不能放在 `scoped` 样式中；`scoped` 会改写 keyframes 名称而内联字符串不会被编译器同步改写，表现为雪碧帧图永远停在第一帧。动态帧数、时长可以继续走 CSS 变量或内联属性，但关键帧命名必须保持全局或改为 class 驱动。
- 2026-07-08 聊天气泡流式阶段不能因为正文里出现列表、强调、行内代码等 Markdown 标记就退出轻量文本路径；正在生成时应优先用纯文本预览，终态再渲染完整 Markdown。头像或浮动形象这类持续动画也不能用 Vue/JS 定时器逐帧更新响应式状态，应该尽量用 CSS transform/steps 动画，避免气泡排版和 token flush 把动画一起拖卡。
- 2026-07-07 智能体循环条目的上下文快照必须按 `model_turn_id` 绑定和回填，message 级快照只能作为同模型轮次兼容兜底；`llm_output/token_usage` 的 `usage.total_tokens` 可以作为该模型步输出后的占用快照，但不能让后续 `round_usage` 或上一轮最终快照覆盖已有工具条目。否则第二轮第一条工具会显示上一轮最终上下文，或每条工具都变成同一个最终值。
- 2026-07-07 智能体循环条目聚合器必须同时支持前端内部归一事件名和后端/投影原始事件名，例如 `tool_call_started/tool_call_delta/tool_call_completed/tool_call_failed` 与旧的 `tool_call/tool_output_delta/tool_result`。如果只识别旧事件名，首个工具调用 started 事件会被过滤，直到第二个工具事件到达后面板才出现，造成实时反馈晚一拍。
- 2026-07-07 工具结果展示链路不能各自手写 `result.data || result` 这类浅层剥壳；读取、写入和 observation 预览都可能收到 `{data:{action, data:{content}}}` 这类双层工具信封，必须先走共享 normalization 再进入结构化视图和文本兜底。否则文件内容明明在 `content` 中，界面仍会退回渲染外层 JSON。
- 2026-07-07 智能体循环工具结果展示不能把 runtime envelope、stream event detail 或包含 `client_message_id` 的原始事件包当作用户可读内容兜底；读/写文件应只展示文件正文，其他工具优先展示实际发给模型的 `model_observation`/observation，并统一做头尾保留压缩。否则排障时会被事件元数据、路径、字节数和内部轮次字段淹没。
- 2026-07-07 聊天智能体循环条目的工具调用原文和单次上下文占用属于 per-workflow item 元数据，实时投影 upsert、hydrate 迁移、render adapter 和 render-version key 必须同时携带并纳入失效条件；不能只依赖消息最终 stats、结果 detail 或刷新后的历史快照兜底，否则会出现实时右键无内容、刷新后才恢复，以及所有条目显示同一个最终上下文占用。
- 2026-07-06 聊天页流式增量不能把“用户展开/折叠/目录操作”绑定到实时投影对象生命周期；智能体循环、工具调用明细和工作目录这类可交互面板必须按会话与稳定消息/路径 key 记忆本地 UI 状态，后台刷新也不能拉起全局 loading 或覆盖右侧 dock 点击，否则模型输出期间会表现为面板自动收起、目录点不开和整页卡顿。
- 2026-07-06 Win7 legacy console 使用 crossterm WinAPI backend 时不能启用 ratatui `underline-color` feature；该 feature 会在每帧 reset 阶段发送 `SetUnderlineColor`，WinAPI backend 明确不支持并会让 TUI 退出。CLI 这类 Win7 产物应显式关闭 ratatui default features，只保留 `crossterm` 和实际需要的 unstable feature。
- 2026-07-06 Win7 legacy console 不支持 terminal bracketed paste；TUI 初始化时不能把 `EnableBracketedPaste` / `DisableBracketedPaste` 当作硬启动条件，遇到 `legacy windows api` 不支持错误应降级并保留 Ctrl+V/Shift+Insert 显式粘贴路径，否则真实 Win7 会在进入界面前直接退出。
- 2026-07-05 CLI/TUI 普通按键路径必须保持纯内存、低常数开销：不要为了识别粘贴在每个字符附近同步读系统剪贴板、启动 PowerShell/pbpaste/xclip，也不要把高速输入先缓存成“疑似粘贴”再延迟展示。粘贴应走 bracketed paste 事件或 Ctrl+V/Shift+Insert 显式路径，redraw 请求也要合并，避免快速输入时 UI 追帧。
- 2026-07-05 Win7 GNU CLI 构建不能只复用 target 名称，还必须固化为脚本化 SOP：`target_vendor="win7"` 下的 `reqwest` 要关闭默认特性并使用 `rustls-no-provider`，workspace `tokio-rustls` 要显式使用 `ring` provider，避免系统代理/TLS 链路拉入 WinRT API-set 或 aws-lc 体积；release 分发前要 strip 去掉 COFF 符号，并用 objdump 同时检查 `pei-i386`、`api-ms/winrt` 导入和 SHA256，否则很容易产出看似能启动但 Win7 不兼容或体积异常的 exe。
- 2026-07-05 蜂群右侧栏和聊天页头像这类二级实时视图必须与主聊天使用同一套终态 busy 判定和稳定渲染键；初始 hydrate 的远端历史若替换本地 optimistic key，会让 Vue 把整列消息当成删除重建，表现为发送后顺序先乱再排。等价远端消息应保留本地渲染 key 并记录 remoteKey，同时间戳消息必须显式按用户先于助手排序，头像运行态也要读取 effective busy 而不是原始 busy getter。
- 2026-07-05 聊天断线恢复后的终态收敛不能只看 `stream_incomplete`、`workflowStreaming` 或 waiting flags；当 `thread_status/thread_closed` 已确认运行终态时，`status` 仍停在 `placeholder/waiting_first_output/streaming/tooling` 的空助手占位也必须被结算为 final/failed。否则 projection 已经有最终回复，legacy/renderable 的状态型占位仍会让气泡底部残留“断线重连中”。
- 2026-07-03 聊天渲染根治不能只把气泡列表切到 projection；watch 在 send/resume controller 存在时也必须继续消费 canonical 事件，`event_seq` 小缺口超时必须立即触发 replay，后端所有本轮对象型流事件都要回带 `client_message_id`。否则旧的 send/watch 互斥闸门、gap 只置 `syncRequired` 不补水、乐观消息无精确键三者叠加，仍会表现为重复气泡、顺序漂移或假死。
- 2026-07-02 聊天实时链路必须把队列事件、模型流事件和终态事件放在同一条可恢复的 `stream_events` 顺序边界里；异步流式持久化在写入 `queue_finish/queue_fail` 前必须 flush，WS 排队恢复必须从本轮 `queue_enter` 前一位或失败时的当前 tail 开始。否则 replay 会观察到旧事件回灌、终态抢跑或 event_id 缺口，前端再强的去重也只能缓解，无法根治重复、乱序和假死。
- 2026-06-30 智能体工具选择不能混用“目录展示名、英文别名、运行时中文名”作为前端选中值和后端保存值；桌面监视/控制这类本地工具尤其容易因为目录接口显示别名、保存接口按运行时名过滤而出现“保存成功但刷新丢选中”。新增工具入口时必须同时检查 catalog、保存过滤、详情回显和运行时解析四条链路，保证同一工具的别名可往返且不会被权限过滤误删。
- 2026-06-29 桌面端设置页或 API 保存智能体配置后，不能在同一次同步里再允许本地 worker-card 文件反向导入；旧文件的 mtime 可能与数据库 `updated_at` 非常接近，从而覆盖刚保存的新配置。保存后的同步应只做 runtime -> files 单向落盘，读取或用户手动编辑文件入口再执行双向导入，并用回归测试固定“旧文件不能覆盖新设置”。
- 2026-06-18 聊天可见历史分页必须按用户可见 transcript 计数，不能按原始 `chat_history` 行数计数；`system`、`tool`、上下文压缩摘要和隐藏内部消息只能推进原始游标，不能吃掉可见页额度。模型上下文压缩是运行时上下文替换，不等于 durable 聊天记录截断，刷新和回放必须继续从追加历史中补水。
- 2026-06-18 聊天实时链路不能把 `thread_status`、`thread_closed` 这类运行时状态帧当作模型回合终态；真正关闭 send/resume 流应依赖 `final`、`error`、`queue_fail`、`turn_terminal` 或带完成标记的 `llm_output`。同一最终回复从流式投影、历史快照和渠道侧边消息多路到达时，要按用户轮次合流并保留 stats/workflow，跨用户轮次即使文本相同也不能回填到旧气泡。
- 2026-06-14 Cargo build script 里的 `PROFILE` 通常是 `debug`/`release`，不是 Cargo 输出里显示的 `dev`；如果用 `PROFILE=dev` 判断轻量检查分支，分支不会生效，Tauri build.rs 仍会递归追踪前端 dist、字体和打包资源，导致普通 `cargo check` 变慢或因本地资源权限失败。
- 2026-06-12 Tauri `bundle.resources` 不应直接包含整个 `config` 目录；其中的 `config/data`、工作区、数据库和 benchmark 产物会被 build script 递归监听，既拖慢编译也容易在本地数据权限上失败。桌面资源应使用显式白名单，只打包启动必需的静态配置和内置资源。
- 2026-06-12 后端入口包不能继续挂桌面 Tauri build script 或 CLI 二进制；服务端、CLI、桌面应作为独立 workspace crate 依赖 runtime，否则一次普通 `cargo check` 会被无关运行形态拖慢，且桌面打包配置变化会污染服务端编译链路。
- 2026-06-11 后端重构验证时不要并行启动多个 `cargo test`/`cargo check` 进程；它们会争用 package cache 与 artifact directory 文件锁，反而让单测等待时间翻倍。批量改完后应优先用单个 cargo 命令过滤多个测试，或按顺序执行关键测试，再跑一次 `cargo check -j 8`。
- 2026-06-12 蜂群等待结果被压缩成 preview 后，不能只依赖 `all_finished` 或 `done` 字段判断完成；如果结构化 preview 只保留 `run_ids` 与 `state=completed/partial`，解析器也要从状态推导已完成，否则母蜂会误判仍在运行并重复等待。
- 2026-06-12 会初始化 LSP、Tokio 定时器或异步资源管理器的 Rust 单测必须使用 `#[tokio::test]` 或显式 runtime；普通 `#[test]` 在并行过滤测试中会因无 reactor 直接失败，容易被误判为业务回归。

- 2026-06-09 Messenger 内部接管动画形象层是为了替代全局层、避免重复挂载，不等于只显示当前聊天智能体；聊天页也应显示所有已开启形象，否则切换到其它智能体会让非当前智能体形象被误过滤。

- 2026-06-09 智能体形象帧图在头像、设置页预览和选择列表中必须共用同一个静态帧常量，不能在局部组件里分别硬编码 `idle`/`waiting`；否则修复一个场景会把另一个场景带到不同动作行的第一帧。

- 2026-06-05 悬浮动画形象去重不能按 `/chat` 这类粗路由直接关闭全局层；MessengerView 内部 section 与独立路由不是同一层边界。应明确“消息 section 由 active-only 层接管、其他 Messenger section 由同一 Messenger 层兜底、非 Messenger 页面再由 UserLayout 全局层兜底”，否则切出聊天消息区后形象会被两边同时卸载。

- 2026-06-05 智能体头像与动画形象的动效要分层处理：头像状态角标的 spinner 属于状态可见性，不能跟头像本体/形象动作的 `animated` 开关绑定；companion 的 done/jumping 也不能直接长期跟随后端 done 状态，必须有“用户确认/拖动后 settled 到 waiting”的本地状态，否则中栏已确认或用户交互后仍会停留在完成动画。

- 2026-06-03 预设智能体同步不能只覆盖存量用户；默认智能体这类模板态必须在所有新用户入口执行同一套 bootstrap，并保存 last_applied 基线，否则管理员强制同步后新注册用户会继续读取内置默认配置，后续安全同步也无法判断是否仍未被用户改动。

- 2026-05-25 桌面端聊天页如果在 `messenger-realtime-pulse-start` 之后才出现 `render-process-gone: oom`，排查不能只盯 pulse 自身的请求数量；还要同步检查 pulse 之后的 watcher 是否又触发前 4 个会话详情预热、未读统计是否在后台拉整条 transcript。对 Win7 ia32 这类 renderer，真正的根因常是“轻量轮询 + 重量级自动预热”叠加，而不是单个 API 本身过大。
- 2026-05-25 桌面端聊天页 `render-process-gone: oom` 不能只靠前端渲染后裁剪消息缓解；会话详情和事件快照必须在 API/存储层按尾窗限载，且前端事件快照缓存要按窗口维度隔离。否则旧的全量快照或“取最早 N 条”的错误限流仍会在打开会话时把 Electron renderer 推爆。
- 2026-05-24 Electron 桌面端聊天页出现 `render-process-gone: oom` 时，不能只按普通前端白屏或 main 进程内存泄漏处理；要同时收敛首屏 `loadSessionDetail`、watcher、realtime pulse、工作区预热和图形兼容路径。桌面端 idle active session 不应仅因“当前会话”就保温启动 watcher，后台轮询应等首屏稳定后再启动；同时需要 renderer 阶段日志和 OOM 后自动低图形开销兼容模式，避免下一次启动重复白屏。
- 2026-05-24 桌面端聊天白屏如果伴随 render-process-gone: oom，优先排查 session detail hydrate 是否会反向触发 watcher 重建，以及工作区/图标预热是否在首屏同步执行；这类问题不是单个页面状态错误，而是递归恢复链路叠加重资源预热把 Electron renderer 推爆。
- 2026-05-23 消息页的“保留上下文”判断和“消息类型归类”必须使用同一套事实源；如果只保留了 activeConversation 却让 kind 仍然为空，切换聊天页和智能体页时就会落入既不显示空态也不显示内容的白屏状态。
- 2026-05-23 companion 双击进入聊天不能把“目标会话已经 active”当作完成条件；如果当前路由停在设置、广场或其他 section，仍必须把主界面恢复到 `/desktop/chat` 并确认 MessengerView 可见，否则用户会看到白屏或误以为双击无效。
- 2026-05-23 桌面端 companion 的拖动结束不应再同步回发主窗口状态，也不应把展示层是否可用和页面内兜底渲染绑成一个门闩；一旦 `webContents.send` 落到已释放的渲染帧，Win11 就会出现白屏、卡顿和拖拽异常，应该优先让原生窗口只负责显示，前端只负责驱动，不要互相反向回写。
- 2026-05-23 桌面端外置形象、透明窗口和硬件加速的降级判断必须按真实运行系统和显式环境变量控制，不能复用分发包标签做包型推断，否则现代 Windows 会被错误限制在应用窗口内并引发卡顿或白屏。
- 2026-05-23 Win7 透明桌面窗口兼容修复不能把 Win11 的 companion 展示路径改成额外 render-ready/nativeVisible 握手；透明窗口首帧、原生回执和前端兜底层很容易形成竞态，表现为形象出现后白屏或卡住。Win7 应按真实系统检测直接禁用桌面绘制，现代 Windows 保持原先外置形象路径。

- 2026-05-23 用户正在查看的聊天页不能在空闲时把 `watch` 退化成“只有本地发送或恢复流时才订阅”；活跃 session 应保持轻量持久在线订阅，否则微信、渠道、后台任务这类外部入站消息只能等刷新或下一次 hydrate 才出现。实时链路优化要区分“非当前会话降载”和“当前会话保活”，不能用同一条 idle 规则处理。

- 2026-05-23 聊天运行时错误不能让 `error` 事件和随后的 `turn_terminal` 事件各自生成一条失败 assistant；错误正文应先落到对应 `model_turn_id` 的同一条消息，终态只负责结算状态。同样，运行时工具名和模型侧 function name 必须分层：运行时可以用带命名空间的路由名，送入模型 API 前必须清洗成协议允许的 ASCII 安全名，并保留映射回原始运行时名，否则一个非法工具名会触发模型请求失败，再被前端重复渲染成多个错误气泡。

- 2026-07-04 聊天发送结束或停止时不能只清 `loadingBySession`、controller 和 busy getter；本地 optimistic assistant 占位、runtime projection 的 model turn、legacy raw 消息数组必须一起收到终端事件并清除 streaming 标志。后端事件若只带 `model-turn:<session>:user:<round>` 这类弱模型轮次，也必须折叠到同一用户轮次已有的强本地 turn，否则停止后再发送会出现一直转圈或额外空气泡。
- 2026-07-04 聊天投影合并弱模型轮次和强最终轮次时，不能只保留最终正文气泡；速度、工具调用次数、workflowItems 和 subagents 都是用户可见投影元数据，必须按稳定业务 id 合并到保留气泡，并让渲染物化缓存按元数据内容失效。否则实时过程中看似去重成功，刷新或回到历史会话后气泡底部统计和智能体循环信息会丢失。

- 2026-07-04 智能体工具调用可能发生在同一用户轮次的强 `model_round=1`，最终回复和平均速度发生在强 `model_round=2`，而 canonical transcript 只保留一个 assistant 气泡；刷新迁移投影元数据时不能只迁移弱/legacy 轮次。单 assistant 用户轮次应合并同轮所有带 workflow/stats/subagents 的投影源，多 assistant 用户轮次再按 model turn 保守分配，否则历史会话会丢工具调用但保留最终正文。
- 2026-07-04 聊天实时渲染链路里，后到的 legacy/session snapshot 若带空 `workflowItems`、空 `subagents`、空 `stats` 或旧别名零值，只能视为“未携带投影元数据”，不能清空 canonical events 已恢复的智能体循环和速度统计；工具结果行也必须保留模型原始 tool call 参数，右键调试优先读 `toolCallRawDetail` 而不是后来覆盖的结果明细。

- 2026-07-04 蜂群派发流不能只更新蜂群本地运行状态和最终补水；它收到的 chat stream 事件也必须进入统一 `chatStore.runtimeProjection`，并在用户提交时先建立本地 user/model turn 占位。否则右侧栏和智能体卡片只能等 session hydrate 才恢复，表现为消息不同步、工具调用条目实时缺失但刷新后正常。
- 2026-07-04 蜂群画布这类二级实时视图不能靠自身 dispatch 缓存反推当前聊天上下文；进入蜂群时必须显式绑定当前消息会话 id 和所属 agent id，并让统一缓存接口优先物化 runtime projection、再兜底当前 active messages。智能体卡片如果产品语义是“工具调用情况”，就不能继续使用任务摘要、session id、模型事件或非工具 workflow fallback，否则会出现实时不渲染、必须再问一句才同步、节点条目杂乱和刷新后才恢复的假象。
- 2026-07-04 蜂群页、右侧栏、实时 watch 这类二级视图的组件卸载只能表示“本地脱离监听”，不能等同于用户停止运行；本地 `AbortController.abort()` 必须显式关闭 WS 自动 cancel，用户点击停止再走统一 stop/cancel 协议。否则从蜂群页切到聊天页会把仍在运行的会话误停止，表现为页面切换触发后端 cancel 或本地状态被写成 stopped。
- 2026-07-05 蜂群派发在创建本地 optimistic user/model turn 后，后续 chat stream、工具调用和最终回复必须继续携带同一组 `client_message_id/user_turn_id/model_turn_id/assistant_message_id`；不能只依赖后端 `user_round/model_round` 回推。否则前端实时投影会先生成 `round:N` 的额外空 assistant 气泡并排到用户消息前，刷新后才被 canonical transcript 纠正。
- 2026-07-05 execute_command 同时存在 command_session_delta 与 legacy tool_output_delta 时，前端只能选择一个源写入命令会话 tail；command_session_delta 应作为实时终端权威增量，tool_output_delta 只保留工作流兼容投影，否则聊天工具条目会把同一段 stdout/stderr 追加两次。
- 2026-07-05 命令终端实时流不能只覆盖本地 execute_command 路径；server/sandbox 执行路径也必须有独立流式协议，并把 stdout/stderr 增量桥接为 command_session 事件。前端对这类一次性在线增量应按 tool_call_id 立即关联工作流条目，但不能用较大的 command_session event_seq 推进严格顺序水位，否则后到的普通模型或工具事件会被误判为 stale，表现为终端或正文仍像阻塞到最终结果才出现。
- 2026-07-05 命令会话事件和原始工具调用必须以 `tool_call_id` 作为主关联键，`command_session_id` 只能作为别名；命令会话事件也不能用 `event_id` 兜底生成假的 tool call id。刷新补水可能先到 summary 再到 tool_call，前端应保持同一终态行并只补充原始 `toolCallRawDetail`，否则会出现重复 execute_command 条目或丢失 `timeout_s` 等原始参数。
- 2026-07-05 智能体设置页的工具选中态必须在工具目录、表单、保存 payload 和历史配置之间共用同一套 alias canonical 规则；网页抓取/网页搜索这类同时有中文运行时名和英文别名的内置工具不能只在后端识别别名，前端也要在选项 value 与已保存 tool_names 比较前归一化，否则保存成功后刷新会表现为未选中。
- 2026-07-05 智能体设置页的工具目录不能使用“配置启用”口径而保存接口使用“运行时可用”口径；带 Cargo feature 或 provider 开关的内置工具必须在 catalog 和保存过滤中共享同一可用性判断。否则会出现“页面能选、保存无报错、刷新后丢选中”的假保存。
- 2026-07-05 命令终端流式展示要区分“工具调用条目”和“进程会话条目”：聊天工作流里同一个工具调用的原始参数可用 `tool_call_id` 关联，但 CLI/TUI 的实际终端进程必须以 `command_session_id` 作为主键；`tool_call_id` 只能作为启动前占位别名。否则一次 `execute_command` 拆出多条命令时，会把多个进程输出合并到同一张卡片，或者最终 `tool_result` 覆盖前一条会话。
- 2026-07-06 聊天用户轮次不能只依赖 MonitorState 的内存计数。冷启动、历史未 warm 完、很久以前的会话继续发送或异步写队列乱序时，必须先按 session 精确补水 monitor/stream events 中的最大 `user_round`，并让存储 upsert 保持 `updated_time` 单调，否则后续发送会重新从 1 开始，前端投影顺序和工具调用归属都会被污染。

- 2026-07-06 智能体循环右键展示的“工具调用原文”只能来自真实 `tool_call` 或显式保存的 `toolCallRawDetail`；`tool_result`、`command_session_summary` 和运行时结果快照不能反推成模型调用参数。命令状态也必须读取 `exit_code/returncode/timed_out/error`，不能只相信 `status=completed`，否则失败命令会显示绿色且右键看到的是结果。
- 2026-07-07 聊天普通文本流式输出不能让每个 token 增量驱动父页面大模板、Markdown 解析、资源卡片 DOM 扫描和全局投影刷新一起跑；正文渲染应隔离成小组件，普通文本流式阶段走轻量文本路径，最终再补 Markdown，资源补水只在确实出现资源卡片时触发。否则即使后端流式很快，头像动画、工作目录树和右侧面板也会被主线程重活拖卡。

- 2026-07-07 本地模型流式输出比网络 API 更容易暴露前端主线程问题：网络延迟会天然摊平 token 到达，而本地推理可能同机抢占 CPU/GPU 并把多个 delta 突发塞进同一渲染窗口。前端处理本地模型流时必须同时做 reducer 快路径、按帧合并 content clock、动画组件隔离订阅和必要的本地模型线程/资源限制，否则会表现为前几个字卡住后突然刷出一大段。
- 2026-07-07 用户可见的 `llm_output_delta` 不能作为 lossy WebSocket 事件处理，也不能在队列低水位时返回错误触发流转发循环退出；本地模型高频输出会比网络 API 更快填满内存队列，一旦中间 delta 被静默丢弃或转发中断，前端就只能显示开头并等待最终 `llm_output` 快照一次性补全。
- 2026-07-07 聊天流式卡顿排查时，debug 本身不能成为热路径负担；`chat.stream.perf`、projection shadow、render source、terminal-debug 这类大 payload 日志应保留在内存导出历史里，避免直接 `console.info` 完整对象。浏览器 console 展开大对象、序列化 payload size、以及逐 token 布局测量都会和动画、工作目录交互抢同一个主线程，导致用户看到头像和浮动形象卡顿。
- 2026-07-07 本地模型流式体验必须同时观察后端 `ttft/prefill`、上游 chunk 间隔和前端 flush 延迟；如果 `prefill_ms` 或 `max_chunk_gap_ms` 已经很大，前端优化只能减少放大效应，不能制造真实 token 流。后端 `llm_output` 需要携带轻量 `stream_timing` 诊断字段，前端 debug 需要记录 content clock 慢 flush，才能区分“模型没吐”“服务端攒包”和“前端主线程掉帧”。

- 2026-07-08 流式正文气泡不能订阅全局 `runtimeProjectionVersion`；普通文本增量应只订阅当前 `messageId/turn` 的 content clock，并直接更新局部文本节点。普通 debug 也不能在流式热路径构造全历史消息摘要或 projection shadow，这类重型对比必须放到 verbose，否则即使刚开始对话，也会让输入框、动画和目录交互被气泡渲染拖慢。
- 2026-07-09 工具轮次中模型在工具调用前后输出的正文只能作为临时可见文本，真正 `final` 事件才是整轮最终答案；进入真实工具调用边界时应立即清空 assistant 可见正文，再让后续 delta 重新显示，否则多个中间片段会累积进最终气泡，表现为“每次输出没有清掉”。
- 2026-07-09 聊天流式正文小组件只订阅单个 message id 时，工具边界、最终回复工具或弱 turn 折叠可能让后续 delta 写到同一 user/model turn 的另一个 assistant 投影上；应在正文小组件内部保留会话级 content clock 兜底，同时避免恢复父级 `runtimeProjectionVersion` 订阅，否则会出现“只显示前几个字，后面终态才一口气出现”。
- 2026-07-09 流式正文组件的响应式订阅仍可能因为父级 renderable 对象原地同步、content clock 合并或组件挂载时机而漏掉后续 delta；对正在 streaming 的正文小组件，可以用低频本地轮询直接读 runtime projection 并更新文本节点作为兜底，但轮询必须限定在单个正在输出的正文组件内，卸载或非流式时立即停止，不能扩大到消息列表或全局状态。
