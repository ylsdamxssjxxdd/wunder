# 经验教训（每次一条）

- 2026-05-23 用户正在查看的聊天页不能在空闲时把 `watch` 退化成“只有本地发送或恢复流时才订阅”；活跃 session 应保持轻量持久在线订阅，否则微信、渠道、后台任务这类外部入站消息只能等刷新或下一次 hydrate 才出现。实时链路优化要区分“非当前会话降载”和“当前会话保活”，不能用同一条 idle 规则处理。

- 2026-05-23 聊天运行时错误不能让 `error` 事件和随后的 `turn_terminal` 事件各自生成一条失败 assistant；错误正文应先落到对应 `model_turn_id` 的同一条消息，终态只负责结算状态。同样，运行时工具名和模型侧 function name 必须分层：运行时可以用带命名空间的路由名，送入模型 API 前必须清洗成协议允许的 ASCII 安全名，并保留映射回原始运行时名，否则一个非法工具名会触发模型请求失败，再被前端重复渲染成多个错误气泡。

- 2026-05-21 聊天页流式输出不能让每个正文或 reasoning delta 都直接写入响应式气泡对象；这会级联触发 renderable 列表、Markdown、虚拟列表测量、头像状态和工作目录扫描，表现为模型输出时整页卡顿或短暂假死。delta 应先进入普通缓冲，按 30-50ms 与 `requestAnimationFrame` 合并写入；终态事件再强制 flush，同时无可见变化的 snapshot 不应推进消息列表版本、projection reconcile、窗口计算或快照落盘。
- 2026-05-21 聊天实时事件不能在发现小范围 `event_seq` 缺口后继续抢跑应用后续 delta/final；否则缺失的早期 delta 后到时会被 stale 规则吞掉，表现为智能体回复片段丢失、最终内容跳变或页面看似假死。严格 WS/test 事件应先进入有界顺序缓冲，缺口补齐后按序排空；超过缓冲能力的大缺口必须立即触发 events/snapshot 恢复，而不是只记录 `syncRequired` 等 watchdog 兜底。后端未保证连续 canonical seq 前，大缺口仍需兼容合法跳号事件，不能直接硬阻塞。
- 2026-05-21 聊天 projection 渲染的“空列表”不能简单等同于“投影不可用”；投影已经知道当前 session 但可见消息为空时，应渲染空列表而不是回退 legacy `messages`。否则空会话、清空面板、空快照或终态收敛会被旧数组残留气泡重新污染，表现为消息清不掉、切会话回来又出现旧回复。
- 2026-05-21 聊天输入区、停止按钮和发送日志这类周边 UI，不能在气泡列表切到 projection 后继续直接读取旧 `chatStore.messages`；上下文占用、忙闲快照和滚动计数都属于显示态派生，必须由同一个 renderable source 向下传递。否则气泡来自新投影、composer/stop guard 来自旧数组，会在停止确认、上下文占用和发送后居中上继续出现抖动、误判或假实时。

- 2026-05-21 聊天 projection 渲染不能复用旧 `messageMutationVersion` 或依赖深层响应式追踪来刷新；projection reducer 是原地更新，高频 token delta 如果直接被 Vue 深度追踪，会同时带来漏刷和整页重算。应使用独立 `runtimeProjectionVersion` 作为显式渲染时钟，流式事件按帧合并，提交、终态、snapshot 和 runtime status 这类边界事件再立即刷新。
- 2026-05-21 聊天 projection 渲染不能只同步正文、reasoning 和 streaming flag；工具调用、审批、子智能体这类 workflow timeline 也是消息事实的一部分。只要 render adapter 在正文一致时复用旧 raw message，就可能把投影中新产生的 `workflowItems` 静默丢掉，表现为 projection 模式下工具卡片缺失或忙碌状态漂移。投影层必须按稳定调用 id upsert 工作流条目，且一旦投影工作流与 raw 不同就要物化新对象。
- 2026-05-21 聊天页渲染源迁移不能只把 `agentRenderableMessages` 本身接到 projection；身份识别、会话保留、忙碌快照、模型名、预览刷新和发送后滚动计数这类展示派生也必须在安装顺序上晚于 renderable source，并通过同一个 helper 读取当前 UI 消息记录。否则 projection 开关开启后，气泡列表和周边状态会在同一帧读到两套来源，继续造成抖动、误清空或运行态误判。
- 2026-05-21 聊天渲染迁移不能只比较 reducer projection 和旧 `messages`，还必须比较真正送入模板的 renderable 列表；很多抖动来自 vnode key、过滤规则、占位气泡和 streaming flag 在最后一层发生分歧。推进默认切换前应先提供 `shadow` 模式，在不改变 UI 来源的情况下记录 render-source drift。
- 2026-05-21 聊天渲染从 legacy `messages` 切到 canonical projection 时，不能只改气泡列表本身；头像运行态、最新助手消息、stats、TTS/Markdown 缓存 key 等周边读取也必须跟随同一个 renderable selector，否则会形成“气泡来自新链路、状态来自旧数组”的新分裂。迁移期要用显式开关、projection 为空回退和稳定 runtime key 分阶段落地。
- 2026-05-21 聊天页不能在 canonical projection 仍与旧 `messages` 存在不可见漂移时直接切换渲染源；渲染重构前必须先做影子对比，把缺失、重复、顺序、内容、状态和 busy 分歧以稳定指纹记录下来，并用回归测试固定规则，否则一次性改渲染层只会把旧链路竞态迁移成更难定位的新问题。
- 2026-05-21 聊天实时链路不能让 send、watch、resume、HTTP snapshot 各自直接推导消息与 busy 状态；`queued` ack 也不能作为清理 request-scoped 事件的默认边界。必须先把本地提交、WS 事件和快照统一转成带 `event_id/event_seq/turn_id/message_id` 的 canonical 投影，再由 selector 驱动 UI，否则丢消息、重复、假死和刷新后状态漂移会反复出现。
- 2026-05-20 前端列表选中态不能使用用户正在编辑的名称、标识或标题作为唯一 key；输入过程会改变 key，触发列表重建和选中回退。应使用不可编辑的 uid/id 维护组件内部选中态，展示名只用于显示和保存，跨组件同步也要优先传递稳定 id。
- 2026-05-20 反复回归的页面耦合问题不能只靠 `v-if=false`、`detailOnly` 或入口绕行隐藏旧分支；当模型配置、系统设置这类职责已经拆分时，必须删除旧模板、旧状态、旧 watch、旧保存逻辑和旧样式，确保旧入口即使被误传参数也没有可执行路径。
- 2026-05-20 录音转写链路不能把 WebM/Ogg 这类容器只改文件名当成 WAV 发给 whisper.cpp；必须真实转码为 PCM WAV，再让后端对非 WAV 音频继续兜底转换，调试信息只保留在控制台，避免把错误详情塞进弹窗打断录音流程。

- 2026-05-20 透明桌面悬浮窗不能只依赖 CSS 透明和 DOM pointer-events 来避免挡点击；系统层仍会先把事件交给窗口。像素角色这类非矩形交互必须优先在原生窗口层设置 shape/命中区域，运行时不支持时再用 setIgnoreMouseEvents 做回退，否则透明区域仍会吞掉下层窗口的首次点击。

- 2026-05-20 桌面悬浮窗在触摸屏上拖拽时不要长期依赖增量 dx/dy 累加或 touch contextmenu 的 screen 坐标；应按按下时的抓取偏移计算绝对窗口目标位置，并给长按菜单使用最近 pointer client 坐标兜底，否则高频触摸事件、窗口移动延迟和系统长按事件会叠加成抖动或菜单跑偏。

- 2026-05-20 桌面端外置动态形象这类悬浮交互组件必须区分基础运行态和本地短交互态：running/review/done/error 可由业务运行态同步，拖拽方向、打招呼、等待过渡等只能做临时渲染，不能写入持久状态；普通点击也不能触发拖拽结束落盘，否则会出现“鼠标交互一次后长期停留在运行动画，只能刷新恢复”的状态污染。

- 2026-05-17 OpenAI-compatible 模型进入原生 function tools 后，稳定性要优先于盲目执行坏参数；即使不是 vLLM/Ascend，本地或三方兼容后端也可能在流式阶段丢失 `tool_calls.function.arguments` 尾部，表现为 `{"raw":"{"}`、空 arguments 或不可解析 JSON。首轮仍应保留流式体验，但响应后必须做坏工具调用校验；命中截断/包裹原始坏参数时不要继续执行工具，而要记日志后按模型不可用进入恢复重试，后续重试再回退非流式。

- 2026-05-17 管理端/用户端这类成对存在的 multipart 上传接口不能只修一侧的 `DefaultBodyLimit`；即使业务逻辑、文件格式校验和提示文案完全一致，只要路由层漏配包体上限，较大的压缩包就会在 `multipart` 解析前被框架默认限制拦截，前端通常只会看到泛化的 `Failed to fetch`。上传能力联调时要把路由层限制、业务层大小校验和前端错误提示三处一起对齐。

- 2026-05-16 管理端用户列表里的 `online` 语义不能偷换成“当前 WebSocket 连接数大于 0”；登录完成到实时连接建立之间存在正常空窗，列表应复用 presence 的在线窗口语义，强制下线时再显式清空该用户 presence，避免出现“刚登录始终离线”和“被踢后仍在线”两种相反误判。

- 2026-05-16 强制下线/会话替换通知不能无条件清理同源 `localStorage` token；旧标签页收到 `SESSION_REPLACED` 时，另一个标签页可能已经完成新登录并写入新 token，清理逻辑必须只删除触发失效的旧 token，避免表现为登录页提交成功后又立刻回到未登录状态。

- 2026-05-16 管理端免登录用户侧页面应走专用管理端 token 签发接口，继续保留管理员/单位范围校验，不要为了免登 admin 放开公开外链登录的 admin 保护；同时 `window.open` 携带 `noopener,noreferrer` 时可能成功打开但返回 `null`，不能把返回值空直接视为打开失败。

- 2026-05-16 免登/外链登录写入同源浏览器存储时，必须把会话态与持久态分离；如果把目标用户 token 直接落到 localStorage，管理员同浏览器再次登录本账号时就会被旧 token 污染，且被踢下线后还可能回读到别的本地登录态。

- 2026-05-16 外部工作流的超时默认值、上限、示例页和文档示例必须同步更新；只改后端常量会让旧的 600/1800 示例继续覆盖请求参数，最终仍可能按旧阈值触发 `external_workflow_timeout`。

- 2026-05-16 OpenAI-compatible 模型服务可能返回 HTTP 200 但没有任何 content、reasoning 或 tool_calls，尤其在流式链路只给出结束帧时更容易被误判为正常完成；编排层不能把这种空载荷写成 final completed，必须先补拉非流式请求，补拉仍为空时按模型不可用进入退避重试或失败态，否则会表现为任务中途终止但只留下“未返回可展示答复”的兜底文案。

- 2026-05-15 网页搜索工具不能在 provider 空结果时只返回“下一步抓取具体 URL”的泛化提示；这会诱导模型在没有候选链接时自行猜百科、官网或文档 URL。搜索工具返回 0 条时必须明确提示不要猜 URL、不要编造来源，并把问题归因到搜索 provider/网络/查询条件，后续再考虑更换搜索 provider 或调整查询。

- 2026-05-13 放开文件工具路径权限时，不能只改 allow_paths、workspace.resolve_path 或 sandbox 根挂载；还要逐个核对 read_file/list_files/search_content 是否共享同一条解析链，以及 sandbox 请求是否把 workspace_root/container_root 原样传到执行端。否则表面上已经“全放开”，实际仍会因为 public path 被错误重写成其他真实路径而报 not_found，模型也容易把这种路径映射错误误判成工作区白名单限制。
- 2026-05-13 OnlyOffice AI 属于后台插件，镜像内有插件目录不等于每次编辑器都会显示 AI 菜单；必须同时在 Document Server 的 `services.CoAuthoring.plugins.autostart` 和 DocEditor 的 `editorConfig.plugins` 中显式加载 AI 插件，避免可见性依赖浏览器/用户上次插件状态。
- 2026-05-13 新增工具文案或说明书章节时，不能接受源文件里出现 `????` 这类占位乱码再继续提交；必须直接在 `config/i18n.messages.json`、说明书 Markdown 源文件和相关源码常量层做 UTF-8 校验，构建静态文档站点后再全文复扫 `????`，并明确区分真实 `??` 运算符与编码损坏文本，避免源文案、生成产物和回归脚本同时被污染。
- 2026-05-13 排查沙盒文件写入的 `Read-only file system (os error 30)` 时，不能只看 Wunder 进程里的 `WUNDER_SANDBOX_READONLY_ROOTFS` 或请求 payload；Docker Compose 的服务级 `read_only` 会在容器运行时直接把根文件系统挂成只读，优先级更底层。涉及沙盒权限默认值调整时，必须同步检查 compose/runtime flags，并在启动期加入实际写探针日志，避免把容器只读误判为工具路径限制或业务配置未生效。
- 2026-05-13 聊天消息里的 workspace 资源链接可能经过 Markdown、URL、DOM dataset、Axios 参数序列化等多段链路，同一个中文文件名在不同部署入口下可能出现 `%25E5...` 形式的双重编码。资源卡片预览不能只在展示层解码标题，必须在资源 URL 解析层多轮解码并重新规范 publicPath，后端预览配置接口也要兜底解码路径，避免 OnlyOffice/draw.io 按编码字面量查找文件导致 404。
- 2026-05-13 文件类工具的安全边界不要只看工具白名单；列出/读取/写入/搜索/补丁如果在 server 进程执行，就会继承 server 的 `/app` 挂载视图。需要收敛智能体文件系统视野时，应让 server 只做转发，在 sandbox 容器内复用同一份稳定工具实现执行；如果文件工具本身要求不做路径限制，就以 sandbox 容器 `/` 为可访问根，把权限收口放在外层调用链路与容器挂载边界，避免维护两套文件工具逻辑。
- 2026-05-13 OnlyOffice 自定义字体排障不能只看 Docker volume 是否挂载成功；不同 Document Server 版本的字体生成脚本扫描范围可能变化，启动脚本应把 custom-fonts 同步到系统字体目录并每次按当前目录重建字体索引，不要再恢复旧字体索引缓存，否则目录内有字体但 AllFonts/font_selection 仍可能沿用旧结果。
- 2026-05-13 ComfyUI 的 `/upload/mask` 不是通用灰度遮罩上传接口，而是把上传图片的 alpha 通道复制到 `original_ref` 原图上；灰度 PNG 转 RGBA 后 alpha 通常全 255，会导致遮罩退化成整图/无有效局部编辑。OpenAI/vllm 兼容层接收 `mask_image` 时应先规范化为白色=编辑区域的灰度图，再用普通 `/upload/image` 保存，并用 `LoadImageMask` 的颜色通道读取，同时把外部 `strength` 显式映射到采样器 `denoise`。

- 2026-05-13 用户点击停止后，前端不能等待取消接口、子智能体终止接口或 watcher 终止帧返回后才恢复 UI；停止动作本身必须同步执行本地 runtime 结算，清理 send/resume/watch/compaction controller、activeTurn、approval、waiting、loading 和全局 watcher 标记，并把线程投影收敛到 idle。否则后端已经停止时，聊天区、头像和发送按钮仍会被残留本地锁拖在“运行中”，只能刷新恢复。

- 2026-05-13 聊天气泡 footer、头像和发送按钮不能各自维护运行态语义；消息字段在乐观占位、watcher/hydrate 合并或长时间无输出窗口里可能短暂缺失。展示层必须把“当前会话 busy + 最新可见助手气泡”作为同一权威运行态上下文传给 footer，并把该上下文纳入缓存签名，否则会偶发显示“第 N 轮/已完成”而头像或按钮仍在运行。

- 2026-05-13 聊天实时链路不能只依赖前台发送流自己的 `final/turn_terminal` 回调来清理运行态；长时间无模型输出时，浏览器 WebSocket 或中间链路可能漏掉终止帧但服务端已经完成。前端必须有基于服务端事件快照/线程空闲态的权威结算路径，统一清理 send/resume controller、loading、pending assistant 和投影 busy，且本地恢复中断要与用户手动停止区分，避免误显示“已中止”。

- 2026-05-12 线程冻结不只包含 system prompt，还应包含影响 chat template 的工具调用方式；不要在旧线程中静默把 `function_call` 改成 `tool_call` 或重建 system prompt。对 llama.cpp 这类本地服务，native `tools` 可能被服务端模板注入到最新用户消息附近，message JSON 前缀一致也不代表 raw prompt 前缀一致，必须在日志中显式标注 native tools 缓存风险。
- 2026-05-12 用户取消正在生成的模型请求时，模型服务端可能已经进入 assistant 生成段；如果业务账本下一轮直接在 `user/tool` 尾部追加新 user，message 对象虽然是 append-only，但 chat template 层不再与已缓存 raw prompt 对齐，llama.cpp 会回退到很早的 checkpoint 并擦掉后续缓存。取消链路应写入隐藏 assistant 收口标记，保护下一轮提示词缓存前缀。
- 2026-05-12 排查本地模型 prompt cache 命中率时，先用实际发送请求的稳定哈希证明 Wunder 侧 system/tools/messages 前缀是否 append-only，再看 llama.cpp 日志里的 prompt cache eviction、checkpoint 数量和 `forcing full prompt re-processing due to lack of cache data`；如果请求前缀一致但服务端提示缺少 cache data，优先调整模型服务端缓存容量/会话竞争，而不是继续重写业务上下文。
- 2026-05-12 模型上下文缓存与聊天可见历史必须分账本处理：中间模型轮次、内部 follow-up、终止工具前的说明文字可以进入 append-only 模型上下文，但不能默认写入用户可见 chat_history；只有最终回复、等待用户输入和显式 yield 才应形成普通助手气泡，否则工具循环会把同一用户轮次拆成多个重复气泡，并干扰下一轮缓存命中判断。
- 2026-05-10 线程 system prompt 首次确定后必须保持字节级稳定；上下文压缩或产物索引这类动态信息不能追加到主请求 system，否则运行中压缩与下一轮普通请求之间会出现 system 长度/hash 抖动，破坏模型提示词缓存并造成越聊越慢。
