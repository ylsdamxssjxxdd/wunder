# 经验教训（每次一条）

- 2026-05-13 聊天消息里的 workspace 资源链接可能经过 Markdown、URL、DOM dataset、Axios 参数序列化等多段链路，同一个中文文件名在不同部署入口下可能出现 `%25E5...` 形式的双重编码。资源卡片预览不能只在展示层解码标题，必须在资源 URL 解析层多轮解码并重新规范 publicPath，后端预览配置接口也要兜底解码路径，避免 OnlyOffice/draw.io 按编码字面量查找文件导致 404。
- 2026-05-13 文件类工具的安全边界不要只看工具白名单；列出/读取/写入/搜索/补丁如果在 server 进程执行，就会继承 server 的 `/app` 挂载视图。需要收敛智能体文件系统视野时，应让 server 只做转发，在 sandbox 容器内复用同一份稳定工具实现执行；如果文件工具本身要求不做路径限制，就以 sandbox 容器 `/` 为可访问根，把权限收口放在外层调用链路与容器挂载边界，避免维护两套文件工具逻辑。
- 2026-05-13 OnlyOffice 自定义字体排障不能只看 Docker volume 是否挂载成功；不同 Document Server 版本的字体生成脚本扫描范围可能变化，启动脚本应把 custom-fonts 同步到系统字体目录并每次按当前目录重建字体索引，不要再恢复旧字体索引缓存，否则目录内有字体但 AllFonts/font_selection 仍可能沿用旧结果。
- 2026-05-13 ComfyUI 的 `/upload/mask` 不是通用灰度遮罩上传接口，而是把上传图片的 alpha 通道复制到 `original_ref` 原图上；灰度 PNG 转 RGBA 后 alpha 通常全 255，会导致遮罩退化成整图/无有效局部编辑。OpenAI/vllm 兼容层接收 `mask_image` 时应先规范化为白色=编辑区域的灰度图，再用普通 `/upload/image` 保存，并用 `LoadImageMask` 的颜色通道读取，同时把外部 `strength` 显式映射到采样器 `denoise`。

- 2026-05-13 用户点击停止后，前端不能等待取消接口、子智能体终止接口或 watcher 终止帧返回后才恢复 UI；停止动作本身必须同步执行本地 runtime 结算，清理 send/resume/watch/compaction controller、activeTurn、approval、waiting、loading 和全局 watcher 标记，并把线程投影收敛到 idle。否则后端已经停止时，聊天区、头像和发送按钮仍会被残留本地锁拖在“运行中”，只能刷新恢复。

- 2026-05-13 聊天气泡 footer、头像和发送按钮不能各自维护运行态语义；消息字段在乐观占位、watcher/hydrate 合并或长时间无输出窗口里可能短暂缺失。展示层必须把“当前会话 busy + 最新可见助手气泡”作为同一权威运行态上下文传给 footer，并把该上下文纳入缓存签名，否则会偶发显示“第 N 轮/已完成”而头像或按钮仍在运行。

- 2026-05-13 聊天实时链路不能只依赖前台发送流自己的 `final/turn_terminal` 回调来清理运行态；长时间无模型输出时，浏览器 WebSocket 或中间链路可能漏掉终止帧但服务端已经完成。前端必须有基于服务端事件快照/线程空闲态的权威结算路径，统一清理 send/resume controller、loading、pending assistant 和投影 busy，且本地恢复中断要与用户手动停止区分，避免误显示“已中止”。

- 2026-05-12 线程冻结不只包含 system prompt，还应包含影响 chat template 的工具调用方式；不要在旧线程中静默把 `function_call` 改成 `tool_call` 或重建 system prompt。对 llama.cpp 这类本地服务，native `tools` 可能被服务端模板注入到最新用户消息附近，message JSON 前缀一致也不代表 raw prompt 前缀一致，必须在日志中显式标注 native tools 缓存风险。
- 2026-05-12 用户取消正在生成的模型请求时，模型服务端可能已经进入 assistant 生成段；如果业务账本下一轮直接在 `user/tool` 尾部追加新 user，message 对象虽然是 append-only，但 chat template 层不再与已缓存 raw prompt 对齐，llama.cpp 会回退到很早的 checkpoint 并擦掉后续缓存。取消链路应写入隐藏 assistant 收口标记，保护下一轮提示词缓存前缀。
- 2026-05-12 排查本地模型 prompt cache 命中率时，先用实际发送请求的稳定哈希证明 Wunder 侧 system/tools/messages 前缀是否 append-only，再看 llama.cpp 日志里的 prompt cache eviction、checkpoint 数量和 `forcing full prompt re-processing due to lack of cache data`；如果请求前缀一致但服务端提示缺少 cache data，优先调整模型服务端缓存容量/会话竞争，而不是继续重写业务上下文。
- 2026-05-12 模型上下文缓存与聊天可见历史必须分账本处理：中间模型轮次、内部 follow-up、终止工具前的说明文字可以进入 append-only 模型上下文，但不能默认写入用户可见 chat_history；只有最终回复、等待用户输入和显式 yield 才应形成普通助手气泡，否则工具循环会把同一用户轮次拆成多个重复气泡，并干扰下一轮缓存命中判断。
- 2026-05-10 线程 system prompt 首次确定后必须保持字节级稳定；上下文压缩或产物索引这类动态信息不能追加到主请求 system，否则运行中压缩与下一轮普通请求之间会出现 system 长度/hash 抖动，破坏模型提示词缓存并造成越聊越慢。
- 2026-05-10 上下文压缩的续跑决策应由摘要里的结构化 `resume_action` 和当前轮状态共同决定，不要从工具 stdout/stderr、脚本内容或任意文本里用正则提取产物路径来猜任务是否完成；这类语义侦探会让压缩系统变脆，已完成任务应由摘要标记 `final` 后直接收口。
- 2026-05-10 上下文压缩后的成功工具续跑不能写成“可直接 final”的泛化提示；工具成功只是一条 observation，系统必须保留最新工具摘要和下一步提示，并把前端本地 Abort 与用户显式停止分开记录 cancel_source，否则压缩后容易自然停住或被 stale recovery 误取消。
- 2026-05-10 聊天停止按钮这类确认框不能在用户确认后再读取“当前 activeSession/当前运行”直接执行；弹框出现时必须捕获目标会话和用户轮次快照，确认返回后复核仍是同一轮且仍在运行，否则模型已自然结束、用户切换会话或同会话新一轮开始时，晚点点击确认会误停无关任务。
- 2026-05-10 上下文压缩后的 `context_tokens_after=0` 只能配合 `context_usage_source_after=unobserved_after_compaction` 理解为“等待下一次模型 usage 回写”，前端压缩卡片、分割线和统计入口都不能把它展示成真实 0 tokens；展示层要优先识别未观测状态。
- 2026-05-10 上下文压缩触发必须只使用模型服务端已观测的上下文占用，不能把本地消息 token 估算、工具 schema 估算或压缩后重建消息估算回写成 `session_context_tokens`；压缩会改变请求体，压缩后的占用在下一次模型 `usage` 返回前应标记为未观测，否则会把估算值与真实值重复相加，导致低于阈值时提前压缩。
- 2026-05-10 聊天上下文显示必须把“真实观测值”和“估算兜底值”彻底分离：`context_occupancy_tokens/context_tokens` 只能代表已观测上下文占用，`contextPreviewTokens` 只能在完全没有观测值时给输入区临时兜底，`usage.total/input` 与 `roundUsage.total` 都不能再回写成真实上下文，否则会在切页、重连、完成态与运行态之间反复出现错位。
- 2026-05-10 聊天 token 字段必须按展示目的拆开：`context_occupancy_tokens/context_tokens` 表示当前上下文占用，`request_consumed_tokens/round_usage.total_tokens` 表示单次请求消耗，多模型轮次会累加；输入区、消息统计、监控导出和文档必须同时对齐这个优先级，不能为了兼容旧事件再把累计消耗回填成上下文占用。
- 2026-05-10 聊天 WebSocket 的 queued ack 必须按具体协议区分：`/wunder/chat/ws` 的 start 被排队时只返回 queued 确认，后续 queue_start/模型输出来自会话事件流和 watcher，不应继续占用原 request pending；只有明确会在同一 request 上继续发送 delta/final 的生产者，才允许显式保留 queued 后监听。否则前端会在发送后过早 finalize，表现为没有请求中状态或直接跳到终态。
- 2026-05-10 上下文压缩后的当前轮续跑提示不能作为 `assistant` 尾消息追加；即使语义上是在“模型继续做事”，协议层也必须用 `user` 指令承接，否则保留尾部若已经以 assistant 结束，会触发模型服务的连续 assistant 消息校验并导致 400。

- 2026-05-06 聊天上下文显示必须严格区分三类字段：当前上下文占用只读 `context_occupancy_tokens/context_tokens` 等显式占用字段，模型 `usage/roundUsage.total_tokens` 只能作为请求消耗或配额统计，`context_total_tokens/context_max_tokens/max_context` 只表示模型窗口上限。前端缓存合并时还要优先后端显式占用别名，避免旧版本污染过的 `contextTokens` 再盖掉刷新后的真值。
- 2026-05-06 聊天输入区上下文与消息运行态问题回退时，不能只改显示层或只改测试：`composer/store/messageStats/runtime` 四处必须一起对齐同一版本语义，否则会出现“当前值变累计值”“刷新后上下文变 --”“刚发送先显示任务完成”这类交叉回归。

- 2026-05-06 形象包、蜂群包、知识库包这类 zip/multipart 上传接口必须在路由层显式配置 `DefaultBodyLimit`，不要只依赖业务层的包体大小校验；否则请求会在 multipart 解析阶段先被默认限制拦截，前端只能看到泛化的解析失败。同时，用户侧只读公共资源接口不要沿用需要用户身份的 `resolve_user`，鉴权边界和前端请求客户端要一起确认，避免反复出现 401。

- 2026-04-30 模型侧工具标识、用户展示名、运行时路由名必须分层处理：展示名可以中文友好，运行时名可以携带 server@tool 唯一前缀，但送入 function-call API 的 name 必须是可读且协议安全的 ASCII 名，并保留稳定映射回运行时名；不要让中文展示名清洗后退化成 tool_<hash>，否则模型难以选对工具。

- 2026-04-30 聊天页这类实时 UI 不能同时由 optimistic、WS/SSE、watch、hydrate、组件私有状态多入口直接改消息数组和运行态；否则乱序、重复事件、旧快照覆盖 live 状态时必然反复出现气泡错序、头像状态残留和按钮假死。后续实时链路应以服务端事件日志为事实源，前端统一 reducer 投影，组件只消费 selector，并用乱序/重连/旧 snapshot 回归测试守住边界。
- 2026-04-30 实时聊天终态不能只清旧消息对象上的 `stream_incomplete/workflowStreaming`，还必须在统一投影里同时收口 message、model turn、session busyReason；否则即使气泡内容已完成，会话级 busy 仍可能把头像、按钮和中栏拖成“运行中”。旧 hydrate/watch 适配器进入 projection 时应显式携带 running/loading，并在 false 时强制归 idle。
- 2026-04-30 聊天旧消息缺少稳定 `message_id/turn_id` 时，不能用当前时间生成 key，也不能只按数组原始顺序渲染；hydrate、watch 或弱网重放可能把 assistant 放在 user 前面，造成气泡倒置和重复最终消息。legacy 适配必须先用 `stream_round/created_at/邻近用户轮次` 推导语义顺序，并保证缺时间时 key 稳定。
- 2026-04-30 面向模型的工具返回值不能为了服务端后续便利塞入不透明句柄或本地索引 token；模型难以稳定复述这类值，前端调试也会被内部字段误导。若导出类工具需要复用查询，应优先让模型显式传入可读 SQL，并保证 MCP schema、保存过的 tool_specs、技能提示和文档同步删除旧句柄描述，否则请求体仍会从缓存规格里复活旧参数。
- 2026-04-30 模型生成 SQL 时常会混入中文输入法标点；如果只在某一个工具或编排层做预检改写，其他数据库工具或 MCP 服务直连仍会复现同类失败。SQL 规范化应放在所有数据库 SQL 工具的统一入口，并在底层 MCP 服务再兜底一次，测试同时覆盖 query 与 export。
- 2026-04-30 SQL 只读安全校验不能只靠关键词黑名单扫描表达式文本；同一个词可能既是写语句动词也是只读函数名，例如 MySQL 的 REPLACE()。禁用规则应匹配语句级模式（如 REPLACE INTO），并用函数调用回归测试防止误拦安全查询。
- 2026-05-01 聊天输入区的实时上下文占用不能只等待最终 `round_usage`，也不能在加载中简单沿用上一轮最大值；后续轮次可能缺少前置 `context_usage` 或入模值低于上一轮最终值，前端必须从 `llm_request/token_usage` 建立当前轮次基线，并在新 assistant 出现时重置组件稳定显示。
- 2026-05-01 聊天输入区这类当前轮次指标在 loading 期间不能与 session 级历史缓存取全局最大值；session 缓存适合空闲态兜底，运行态必须优先当前尾部 assistant 的实时 stats，否则当前轮次入模值低于上一轮最终值时，UI 会一直显示旧值直到最终用量超过旧值才跳变。
- 2026-05-01 聊天输入区的运行态上下文占用既要能显示当前轮次增长，也不能在新一轮开始时回退到本地估算值；`llm_request` 的估算、`token_usage` 的输入基线和最终 usage 都必须以上一轮已确认上下文为显示下限，输出增量从稳定基线递增，否则用户会看到数字先抖低再回升。
- 2026-05-01 流式工具调用字段拼接不能做基于字符串内容的去重、补全或快照替换；OpenAI-compatible 片段可能按 `for`、` i`、` in` 这样的词边界切分，若把 `" i"` 与 `" in"` 误判为重复前缀，会把合法代码拼成 `for in range`。`name` 与 `arguments` 都应按 API 返回片段顺序追加，避免系统层误吞模型输出。
- 2026-05-01 聊天气泡底部状态不能只依赖 `llm_request/llm_output` 这类强类型工作流项；服务端可能先推送无 eventType 的通用 progress 项。只要当前 assistant 仍在等待首个输出且未终止，状态层就要维持“请求中”，否则状态行会在早期事件窗口短暂消失造成布局抖动。
- 2026-05-02 聊天实时链路里的上下文占用与模型用量/计费消耗必须按 UI 场景拆分字段和合并策略；输入区运行态不能在 normalize、snapshot merge、composer fallback 中把 `usage.total_tokens`、`roundUsage.total_tokens` 回填到 `contextTokens`，否则工具调用后的多轮模型输出会污染成累计 token 并中断后续流式递增；消息气泡完成态则应允许读取最终 `usage/roundUsage` 修正展示值，避免停留在流式估算值。
- 2026-05-02 聊天输入区的请求期上下文估算不能只看 messages 文本；真实入模还包含 system prompt、工具定义、tool_choice、chat_template_kwargs 等完整 payload，调试日志可能只保留 payload_summary。估算器必须同时支持完整 payload 与 summary-only 结构，完成态再用后端最终 usage/roundUsage 校准，避免用户看到几千估算值而气泡最终是两万多。
- 2026-05-07 聊天这类实时界面只要同时存在 watcher、hydrate、历史补载和本地缓存，就不能让活跃会话的 `messages` 在不同流程里被替换成不同数组引用；实时事件处理、前台响应式数组和会话缓存必须始终复用同一可变数组，或在切换后显式重绑，否则会出现“中途不刷新、最终态一次性出现、刷新后恢复”的假断流现象。
- 2026-05-08 `apply_patch` 这类 grammar 工具要想让能力较弱的模型也稳定使用，不能只优化底层匹配器；至少要同时补三层：工具描述里明确“先读精确片段、一次一个小 patch、必要时先 dry_run、真正修改必须写成 -旧行/+新行”，系统提示里约束不要把大量分散修改塞进同一次补丁，失败提示里直接指出“不要把旧行和新行都写成空格上下文行”。否则模型会反复在同一种格式错误上空转。
- 2026-05-08 `apply_patch` 的 `PATCH_CONTEXT_NOT_FOUND` 不能一律解释成“上下文缺失”；真实失败里经常混有结构性错误，例如前一个变更块其实是空 hunk、后一个变更块复用了同一组锚点、或把真实改动拆成两个连续 `@@`。这类情况如果只返回邻近源码，弱模型仍会误判成“多补一行上下文”；必须把诊断翻译成直接可执行的修正建议，如“删除前一个空 hunk”“把插入内容并回同一个 hunk”“不要让两个连续 hunk 复用同一锚点”。
- 2026-05-08 对 `apply_patch` 的结构性修正建议要继续细分：纯插入 hunk 如果只有单侧锚点，模型很容易把插入点放歪；应该明确提示“同时保留头部和尾部未修改原文”，而不是笼统地说上下文太少。这样弱模型在处理背景色/插入效果一类补丁时，能更快知道该补哪一侧锚点。
- 2026-05-08 聊天前台在会话仍处于 running 时，不能把 `loadSessionDetail/getSession` 这类“历史消息 + 运行态投影”结果直接回写覆盖当前 live 消息数组；实时现场和历史 hydrate 必须分层，运行中的前台只允许增量补洞或保留 live pending assistant，否则就会出现“中途消息消失、上下文占用不再刷新、刷新页面后缓解、最终完成时又突然恢复”的假死现象。
- 2026-05-09 聊天页右侧消息面板不能把“左侧会话列表/排序投影瞬时为空”当成“当前会话真实不存在”；会话轮询、智能体列表刷新和排序回放期间，`activeConversation/activeSession/messages/route query` 任何一处仍保留当前上下文时，都必须优先保活当前聊天面板，否则就会出现“发送后头像和气泡不立刻出现、整页短暂空白、切页或刷新后恢复”的竞态闪断。
