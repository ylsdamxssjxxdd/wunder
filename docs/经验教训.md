# 经验教训（每次一条）

- 2026-05-06 聊天上下文显示必须严格区分三类字段：当前上下文占用只读 `context_occupancy_tokens/context_tokens` 等显式占用字段，模型 `usage/roundUsage.total_tokens` 只能作为请求消耗或配额统计，`context_total_tokens/context_max_tokens/max_context` 只表示模型窗口上限。前端缓存合并时还要优先后端显式占用别名，避免旧版本污染过的 `contextTokens` 再盖掉刷新后的真值。

- 2026-05-06 形象包、蜂群包、知识库包这类 zip/multipart 上传接口必须在路由层显式配置 `DefaultBodyLimit`，不要只依赖业务层的包体大小校验；否则请求会在 multipart 解析阶段先被默认限制拦截，前端只能看到泛化的解析失败。同时，用户侧只读公共资源接口不要沿用需要用户身份的 `resolve_user`，鉴权边界和前端请求客户端要一起确认，避免反复出现 401。

- 2026-04-30 模型侧工具标识、用户展示名、运行时路由名必须分层处理：展示名可以中文友好，运行时名可以携带 server@tool 唯一前缀，但送入 function-call API 的 name 必须是可读且协议安全的 ASCII 名，并保留稳定映射回运行时名；不要让中文展示名清洗后退化成 tool_<hash>，否则模型难以选对工具。

- 2026-04-30 聊天页这类实时 UI 不能同时由 optimistic、WS/SSE、watch、hydrate、组件私有状态多入口直接改消息数组和运行态；否则乱序、重复事件、旧快照覆盖 live 状态时必然反复出现气泡错序、头像状态残留和按钮假死。后续实时链路应以服务端事件日志为事实源，前端统一 reducer 投影，组件只消费 selector，并用乱序/重连/旧 snapshot 回归测试守住边界。
- 2026-04-30 实时聊天终态不能只清旧消息对象上的 `stream_incomplete/workflowStreaming`，还必须在统一投影里同时收口 message、model turn、session busyReason；否则即使气泡内容已完成，会话级 busy 仍可能把头像、按钮和中栏拖成“运行中”。旧 hydrate/watch 适配器进入 projection 时应显式携带 running/loading，并在 false 时强制归 idle。
- 2026-04-30 聊天旧消息缺少稳定 `message_id/turn_id` 时，不能用当前时间生成 key，也不能只按数组原始顺序渲染；hydrate、watch 或弱网重放可能把 assistant 放在 user 前面，造成气泡倒置和重复最终消息。legacy 适配必须先用 `stream_round/created_at/邻近用户轮次` 推导语义顺序，并保证缺时间时 key 稳定。
- 2026-04-30 面向模型的工具返回值不能为了服务端后续便利塞入不透明句柄或本地索引 token；模型难以稳定复述这类值，前端调试也会被内部字段误导。若导出类工具需要复用查询，应优先让模型显式传入可读 SQL，并保证 MCP schema、保存过的 tool_specs、技能提示和文档同步删除旧句柄描述，否则请求体仍会从缓存规格里复活旧参数。
- 2026-04-30 模型生成 SQL 时常会混入中文输入法标点；如果只在某一个工具或编排层做预检改写，其他数据库工具或 MCP 服务直连仍会复现同类失败。SQL 规范化应放在所有数据库 SQL 工具的统一入口，并在底层 MCP 服务再兜底一次，测试同时覆盖 query 与 export。
- 2026-04-30 SQL 只读安全校验不能只靠关键词黑名单扫描表达式文本；同一个词可能既是写语句动词也是只读函数名，例如 MySQL 的 REPLACE()。禁用规则应匹配语句级模式（如 REPLACE INTO），并用函数调用回归测试防止误拦安全查询。
- 2026-05-01 聊天输入区的实时上下文占用不能只等待最终 `round_usage`，也不能在加载中简单沿用上一轮最大值；后续轮次可能缺少前置 `context_usage` 或入模值低于上一轮最终值，前端必须从 `llm_request/token_usage` 建立当前轮次基线，并在新 assistant 出现时重置组件稳定显示。
- 2026-05-01 聊天输入区这类当前轮次指标在 loading 期间不能与 session 级历史缓存取全局最大值；session 缓存适合空闲态兜底，运行态必须优先当前尾部 assistant 的实时 stats，否则当前轮次入模值低于上一轮最终值时，UI 会一直显示旧值直到最终用量超过旧值才跳变。
- 2026-05-01 聊天输入区的运行态上下文占用既要能显示当前轮次增长，也不能在新一轮开始时回退到本地估算值；`llm_request` 的估算、`token_usage` 的输入基线和最终 usage 都必须以上一轮已确认上下文为显示下限，输出增量从稳定基线递增，否则用户会看到数字先抖低再回升。
- 2026-05-01 流式工具调用字段拼接不能做基于字符串内容的去重、补全或快照替换；OpenAI-compatible 片段可能按 `for`、` i`、` in` 这样的词边界切分，若把 `" i"` 与 `" in"` 误判为重复前缀，会把合法代码拼成 `for in range`。`name` 与 `arguments` 都应按 API 返回片段顺序追加，避免系统层误吞模型输出。
- 2026-05-01 聊天气泡底部状态不能只依赖 `llm_request/llm_output` 这类强类型工作流项；服务端可能先推送无 eventType 的通用 progress 项。只要当前 assistant 仍在等待首个输出且未终止，状态层就要维持“请求中”，否则状态行会在早期事件窗口短暂消失造成布局抖动。
- 2026-05-02 聊天实时链路里的上下文占用与模型用量/计费消耗必须按 UI 场景拆分字段和合并策略；输入区运行态不能在 normalize、snapshot merge、composer fallback 中把 `usage.total_tokens`、`roundUsage.total_tokens` 回填到 `contextTokens`，否则工具调用后的多轮模型输出会污染成累计 token 并中断后续流式递增；消息气泡完成态则应允许读取最终 `usage/roundUsage` 修正展示值，避免停留在流式估算值。
- 2026-05-02 聊天输入区的请求期上下文估算不能只看 messages 文本；真实入模还包含 system prompt、工具定义、tool_choice、chat_template_kwargs 等完整 payload，调试日志可能只保留 payload_summary。估算器必须同时支持完整 payload 与 summary-only 结构，完成态再用后端最终 usage/roundUsage 校准，避免用户看到几千估算值而气泡最终是两万多。
