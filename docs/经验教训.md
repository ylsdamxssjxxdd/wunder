# 经验教训（每次一条）

- 2026-04-04 preserveWatcher 只应保护 sending/resuming 交互流，不应把仅 watch 存活当成阻断前台 hydration 的条件；否则 watch 的 reconcile 详情会被全部拦截，出现“页面看起来不再更新，必须手动刷新才看到消息”的假死。
- 2026-04-04 聊天页存在 `preserveWatcher` 时，`loadSessionDetail` 不能在 `sending/resuming/watching` 生命周期覆盖前台消息数组；否则会把实时增量暂存到后台引用，直到终态才一次性补渲染。治理上必须同时做两点：流式结束后无条件重启活跃会话 watch，且仅在生命周期回到 `idle` 后才允许前台 detail hydration。
- 2026-04-03 聊天页若允许 `loadSessionDetail(..., { preserveWatcher: true })` 与实时 watcher 并存，就不能把详情重建结果替换成新的消息数组实例；必须原地同步到 watcher 正在持有的那份数组，否则会出现“消息跳来跳去、实时内容回退到旧轮次、刷新后才正常”的双缓冲分叉。
- 2026-04-03 聊天页实时链路必须区分“远端已观察到的最大 event id”和“当前消息列表已物化的最大 event id”；若把 `thread_status/reconcile` 返回的远端 `last_event_id` 直接当作本地继续消费游标，再叠加同会话 `loadSessionDetail` 中断 watcher，就会把已落库但尚未进入 history 的 `channel_message` 永久跳过，并表现为“状态先变 running、消息要刷新才出现”。
- 2026-04-02 Messenger 响应式断点不能优先依赖父容器宽度；当消息器自身与外层 shell 在收起/覆盖态下不同步时，应以组件本体及宿主链路中的“最窄实际渲染宽度”触发布局切换，否则会在特定窗口区间误判为宽屏并出现页面畸变。
- 2026-04-02 桌面端安装包不能直接整包复制仓库 `config/`；Electron 资源准备必须走显式白名单，只带桌面桥接实际需要的 `prompts/`、`skills/` 和少量桌面种子文件，否则网页端/服务端配置、字体与数据目录会一并混入安装包，导致安装包体积暴涨且配置串味。
- 2026-04-02 渠道 `channel_message` 的 assistant 回包是离散整包，不是单调追加的流式 delta；前端若沿用“保留更长内容”的流式合并策略，上一轮残留占位消息会把下一轮较短回复覆盖成旧答案，必须以当前回包内容直接替换 pending assistant。
- 2026-04-02 Chat watch 不能把“服务端运行态已结束”当成无需补拉的充分条件；一旦服务端 `last_event_id` 已领先本地游标，即使 `running=false` 也必须立刻 reconcile。否则在 WS 背压或请求静默中断后，会出现“消息已落库但页面必须手动刷新才显示”。
- 2026-04-02 事件时间线/监控标题提取不能直接用 `summary/error/reason` 的 truthy 对象做字符串回退；这类字段常是结构化对象，前端必须优先递归提取 `message/code/status` 等标量，否则会导出并显示成 `[object Object]`。
- 2026-04-02 路径放开必须统一到三层：`allow_paths` 语义、工具侧 `resolve_tool_path`、以及 `list_files/search_content` 自带的默认噪声过滤；只放开命令执行或只改一层，会继续出现“shell 能看到，文件工具却路径越界/搜不到”的割裂现象。
- 2026-03-23 默认技能只能用于首次空配置兜底，后续保存、同步、列表/详情读取都不能再隐式注入，且显式 declared_skill_names 必须高于推断逻辑，并用“关闭后不被重新注入”的回归测试长期守住。
- 2026-03-24 桌面端 `user_agents.worker_cards_root` 若指向缺失或不完整目录且 `presets` 为空，会使有效预置列表降为 0/1 个并表现为“只剩一个默认智能体”；发布前必须校验打包后 `resources/config/preset_worker_cards` 数量与可解析性。
- 2026-03-24 桌面端设置页在旧 Chromium/WebView 下，纵向 flex 子项仅设 `min-height: 0` 仍可能不滚动，需配合 `flex: 1 1 auto + height: 0 + overflow-y: auto`，否则“本地运行时/开机自启”会被挤出可视区。
- 2026-03-24 Win7 NSIS 打包默认会把 warning 当 error，安装脚本新增变量或宏时必须确保“定义即使用”，否则会在打包阶段直接失败并中断产物生成。
- 2026-03-24 electron-builder 的 NSIS 自定义宏里，卸载确认与清理逻辑应放在 `customUnInstall` 主路径；仅放在 `customUnInstallSection` 可能出现弹窗不触发与清理不执行。
- 2026-03-24 Win7 NSIS 卸载脚本中 `RMDir /REBOOTOK` 会触发“需要重启”提示且掩盖真实清理问题；应优先前置 `taskkill` + 普通 `RMDir/rd` 清理，并在安装阶段先删除旧 `*-updater/installer.exe` 与 updater 目录防止用户误判为新版本生成。
- 2026-03-24 Weixin iLink 二维码在前端显示异常时，优先判断 qrcode_url 是否真实图片流（image/*）；像 liteapp H5 链接必须经后端渲染为 PNG 后再给 <img>。另外二维码启动在新建场景不要强传 account_id，否则会触发账号归属校验导致“权限不足”。
- 2026-03-24 AppImage 分发默认不应启用 electron-updater，需在主进程按 APPIMAGE 环境显式禁用，否则会生成 updater 缓存目录并引发用户对残留与更新来源的疑惑。
- 2026-03-24 Weixin QR troubleshooting: always provide a raw qrcode_url open-in-new-window entry in frontend; do not rely on preview image only when diagnosing scan/connect issues.
- 2026-03-24 Weixin QR open-link should use explicit qrcode_open_url (prompt page) instead of preview/data URI; preview is for inline render, open-link is for user scan experience and troubleshooting.
- 2026-03-24 Weixin QR consistency: preview QR and open-link QR must share the same source payload (qrcode_open_url) to avoid behavior mismatch between inline scan and new-window scan.
- 2026-03-24 Chat channel realtime must avoid over-reusing old pending assistant messages in round binding; stale pending entries can hijack new inbound rounds so UI shows running state without new messages until refresh.
- 2026-03-24 预览能力过滤禁止“缺省回退全量”：当 tool_names 缺失或未加载完成时，展示应为空或声明集，不能直接回退到 allowed 全量，否则会造成“看似全启用”的假象并误导排障。
- 2026-03-24 前后端能力覆盖语义必须一致：前端预览与后端 apply_tool_overrides 需共享同一规则，禁止 desktop 模式单独补全技能，否则会出现设置页、预览与实际执行三者不一致。
- 2026-03-24 Ability preview must never fallback to full allowed set when agent defaults are missing; show empty or declared-only to avoid false "all enabled" states.
- 2026-03-24 Frontend preview filter and backend apply_tool_overrides must use identical override semantics; do not add desktop-only implicit skill injection.
- 2026-03-24 Ability preview must not depend on list-mode agent cache; list payload can lag/flatten fields. Preview paths should fetch detail profile (force) before rendering capability selection.
- 2026-03-24 会话级 tool_overrides 可能携带历史脏数据（如旧版误写的全量集合）；预览与执行都必须用“智能体声明能力集”做二次限界，不能直接信任会话覆盖。
- 2026-03-24 系统提示词预览必须显式区分 frozen system prompt 与 session.tool_overrides：前者是线程冻结的认知基线，后者只是会话级工具裁剪；两者混算会把历史脏覆盖误显示成提示词本体。
- 2026-03-25 Ability preview filtering must include unified payload keys (items/itemList); filtering only grouped arrays still leaks full-capability UI because renderer prioritizes unified items.
- 2026-03-25 Dock 收起态的呼出逻辑不能只依赖 Dock 本体 hover；当按钮位于 host/overlay 边界之外时，必须提供独立的右侧边缘热区与状态同步，否则 hover 会反复抖动并表现为按钮时隐时现。
- 2026-03-25 折叠 Dock 的展开按钮必须采用“隐藏即不可点击（pointer-events:none）”，并确保 edge-active/hitbox 的显示规则在 host/overlay 变体下具有更高优先级；否则会出现“按钮看不见但仍可点击”，导致误判为 hover 逻辑失效。
- 2026-03-25 Win7 打包链中“补充包说明文件”不能默认并入 setup 资源目录；应在 build 脚本显式排除并在打包前清理残留（README-win7-supplement.txt / wunder-win7-supplement.json），避免安装目录出现无意义文件并引发用户困惑。
- 2026-03-25 聊天主视图在折叠导航/中栏场景不能依赖单一容器宽度与硬编码最小列宽；必须同时采样顶层壳宽度并避免 minmax(360px,1fr) 这类硬下限，否则会触发宽度反馈回路并表现为页面左侧挤压。
- 2026-03-26 渠道文件流必须采用“文件消息缓冲、文本消息统一触发”模式：文件上传轮次禁止直接触发模型，并在触发轮次显式附带累计文件清单与工作目录路径；否则会出现无意义自动回复、会话状态不同步和排障困难。
- 2026-03-26 渠道侧异步写入聊天历史（如仅文件上传）若不同时产出可消费的 stream event，会导致前端 watch 在无运行态看不到新消息；此类‘不触发模型’路径必须同步落一条轻量流事件供实时渲染。
- 2026-03-26 实时消息链路必须把“持久化追加 + cursor 回放”作为主恢复路径，不能只靠内存广播和 sync_required 后全量刷新；否则断线/lag 场景会反复出现“必须手动刷新才看到新内容”。
- 2026-03-26 chat watch 对 channel_message 不能绕开 event_id 游标推进；若只插消息不推进游标，重连 replay 会重复消费并造成消息重复渲染、实时状态滞后。
- 2026-04-02 渠道入站消息“已落库但页面未及时显示”的残余根因通常不是渠道没进来，而是聊天页把 `channel_message` 实时事件当成唯一渲染触发；若 `thread_status` 已进入 running/waiting 而当前轮用户气泡仍未出现，watch 必须立刻做一次受控 reconcile，从会话详情补齐消息，避免再次出现“状态动了但消息没显示”的分叉。
- 2026-03-26 微信媒体出站要按 item 单独发送（item_list 单元素）并保持 CDNMedia.aes_key 编码与渠道实现一致；混发 text+media 或 key 编码不一致会触发客户端“图片已过期”类故障。
- 2026-03-26 上下文压缩恢复不能只依赖静态 `max_context` 配置；当供应商返回上下文溢出时，必须提取并持久化真实窗口提示并配合逐步降档预算进行强制压缩，否则会出现“溢出后每轮都失败、用户无法继续对话”的死循环。
- 2026-03-26 后台智能体执行若走 `orchestrator.run`（非流式）会导致会话 `stream_events` 缺失，前端表现为“运行中无增量、结束后刷新才可见”；凡是需要实时可见的链路（Runner、sessions_spawn、agent_swarm）必须统一走 `orchestrator.stream + drain`。
- 2026-03-26 聊天 Token 占用展示不能长期停留在 pre-call context_usage 估算值；当模型返回 usage 后必须按产品约定口径（优先 total_tokens）回写并覆盖展示，同时要避免把 round_usage 的多轮聚合值当作当前轮占用。
- 2026-03-26 Chat watch 在新轮启动阶段不能复用“无 round 且已有内容”的旧 pending assistant；否则会先回显上一轮气泡/工作区并在结束后才被覆盖，同时渠道消息线程切换后也会出现实时错乱。
- 2026-04-02 聊天页“发送按钮卡在中止态”排查不能只盯 `loadingBySession`；前端 busy 判定还依赖 assistant 消息上的 `stream_incomplete/workflowStreaming/reasoningStreaming`。只清其中一个标记或只清末尾消息，都会让旧轮残留 running 状态把整会话长期锁成 busy；终态收敛必须统一清理 trailing 与 superseded assistant 的全部 streaming 标记。
- 2026-03-26 Token 展示口径必须与产品约定保持一致：当约定为 total_tokens 时，前后端都要统一优先使用 usage.total_tokens，并将 usage.input_tokens 仅作为缺失兜底，避免同一轮消息出现多口径不一致。
- 2026-03-27 配置体系必须坚持单一真源：Docker、server、CLI、desktop 与管理端保存都统一围绕同一个 `wunder.yaml`；一旦再引入额外 override 文件，工具、MCP、技能和知识库的显示与实际生效配置就很容易脱节。
- 2026-04-02 线程首条消息确定后，必须同时冻结“默认工具基线”与 system prompt：后续智能体设置改动只能影响新线程；老线程若仍回退读取当前 agent.tool_names，预览与实际执行都会漂移，而会话级 `tool_overrides` 只能作为独立裁剪层，不能回写冻结 prompt。
- 2026-03-27 蜂群画布中的节点若已在卡片上使用 `pointer capture`，拖拽更新不要只绑定到 `window`；节点拖拽应优先走卡片自身的 `pointermove/pointerup`，画布平移再用文档级兜底监听，否则会出现“画布能拖、卡片拖不动”的错觉。
- 2026-03-27 蜂群画布的背景网格不要跟节点世界层一起放大裁切；网格应作为覆盖整个 viewport 的独立固定层，并按 `offset/scale` 计算背景步长与偏移，否则用户会看到“只有局部有网格线”。
- 2026-03-27 蜂群画布拖拽要分离三件事：世界原点稳定、拖拽缓冲区只服务坐标空间、fitView 只按真实节点内容计算；把缓冲区或原点补偿混进 viewport，会同时引出回弹、整画布跟随和节点被缩没三类问题。
- 2026-04-02 Vue `setup` 中凡是会被 `watch/watchEffect/computed` 在初始化阶段触发的辅助函数，必须使用函数声明或定义在调用点之前；若用 `const fn = () => {}` 且由 `immediate: true` watcher 提前触发，会直接命中 TDZ 并让整棵组件 setup 崩溃。
