# 经验教训（每次一条）

- 2026-04-10 `subagent_control.status/wait` 这类“跟在 spawn 后面消费子会话结果”的工具，不能只在 `send/history` 做单子会话纠错；模型一旦把刚返回的 `session_id` 抄错一位，若 `status/wait` 直接按原值查，就会稳定落入 `SUBAGENT_TARGET_NOT_FOUND -> 再次 spawn -> 父线程 hidden auto_wake 补跑` 的连锁反应。对“当前父会话下仅一个直接子会话、且只给了一个 session_id”的安全场景，应在入口先自动纠正到真实 child session。
- 2026-04-10 上下文压缩分割线这类“历史终态 marker + 当前会话又开始忙”的 UI，状态判断不能把全局 `sessionBusy` 直接映射回旧 marker；否则手动压缩明明已经 `completed`，下一轮一发送消息又会被误刷成 `running`。这类状态必须优先看 marker 自己的终态快照，只有在 marker 尚无终态事件时才允许用会话忙碌态兜底。
- 2026-04-10 上下文压缩排障不能把“交互组数”和“归一化 block 数”混成一个字段名；事件日志至少要同时给出目标 `retained_*_exchange_count_target`、目标 `retained_*_block_count_target` 和实际 `retained_*_messages_debug`，否则看到一个裸的 count 根本无法判断保留的是 2 组交互、4 个 block，还是只剩 1 组旧内容。
- 2026-04-10 上下文压缩里看到 `retained_tail_message_count=0` 时，不能立刻断定是首尾拼接丢尾；必须先同时看 `source_interaction_block_count` 和 `source_interaction_messages_debug`。若压缩输入本身只剩 1 组或 2 组归一化交互块，那么 tail 为 0 是数据事实，不是重建 bug。
- 2026-04-10 子智能体完成后的 `auto_wake` 不能只检查“结果是否已被 wait 显式消费”和“父会话锁是否释放”；还必须在入队前与真正执行前同时复核父会话是否仍处于可续跑状态。只要父线程已经被用户取消、monitor 处于 `cancelling/cancelled`，或父会话本身不再 `active`，就必须直接放弃隐藏续跑，否则会稳定复现“用户已中止，但工蜂晚点完成后又把主智能体补唤醒并产出最终回复”。
- 2026-04-09 `read_file` 这类面向模型的文件工具，输入上限和对外 schema 不能脱节，更不能把超限文件静默截成前几项；否则模型明明请求了多文件，执行层却悄悄只处理一部分，最终会被误诊成“工具没读到”或“模型胡乱重试”。同时 `start_line` 单独出现时必须把默认窗口语义在工具描述里写死；否则模型很容易把递增 `start_line` 误当成另一种翻页协议。
- 2026-04-09 蜂群页这类需要在画布上回放子智能体生命周期的视图，不能只依赖 `getSessionSubagents(session_id)` 这类“当前仍存活子会话”接口；一旦子智能体带 `cleanup=delete` 结束，live 列表就会归零，前端会误表现成“子智能体一会有一会没有”。正确做法是把父会话 timeline 里的 `subagent_dispatch_item_update/subagent_status` 终态快照并入实时列表，按父轮次过滤后再投影到画布。
- 2026-04-09 聊天消息状态条这类“运行中状态”不能只看当前 assistant 自己的 `workflowItems`；上下文压缩、手动压缩标记等事件可能会以独立 assistant/marker 消息插入同一会话。状态判定若不带会话级视角，就会稳定出现“实际已经在压缩，但页面仍显示请求中/模型输出中”的假状态。
- 2026-04-09 蜂群画布若直接消费父会话的 `getSessionSubagents(parentSessionId)` 结果，不能只靠 `run_kind/requested_by` 识别真实子智能体；母蜂刚派出去的工蜂 session 在 run 记录尚未落库的时窗里会先以“只有 session、没有 run metadata”的影子项出现。投影层必须再拿蜂群任务自己的 `spawned_session_id/target_session_id/session_run_id` 做身份排除，否则就会把工蜂误画成子智能体卡片。
- 2026-04-09 聊天页智能体循环区这类“实时流事件先到、历史 hydration 后到”的组件，运行时归并层不能只识别单一字段形态；`eventType/toolName/toolCallId/commandSessionId` 与 `event/tool/tool_call_id/command_session_id` 必须统一归一化，并在出现“raw items 已有但 entryCount 仍为 0”时直接打出样本 keys/字段值，否则就会反复表现成“工具明明调用了，循环区要切页或刷新才出现”。
- 2026-04-09 蜂群页的 dispatch 流即使已经收到 `final`，也不能假设会话历史快照会同步立刻包含最终 assistant；右栏若只在流结束瞬间 hydrate 一次，很容易出现“母蜂明明答完了，但蜂群页要手动刷新才看见最终回复”。终态后必须再做一小段带基线比较的补偿 hydration，直到 session history 里真的出现新的 assistant 终态。
- 2026-04-09 蜂群页这类“聊天右栏 + 任务画布”双视图共享同一身份信息的界面，发送者名称与头像不能分别靠本地字符串和局部默认图猜测；必须先把 `Default Agent/__default__/默认智能体` 收敛到单一规范名，再让右栏消息和画布子节点共用同一套头像解析入口，否则就会出现同一智能体在不同区域名字不一致、子智能体卡片退回固定占位头像的问题。
- 2026-04-09 手动上下文压缩分割线的落点不能按 `user_round` 或“本轮最后一条 assistant”去绑定；压缩是独立事件，水合时必须按 `compaction` 事件真实时间插入消息流。否则只要同一轮后面又出现新的用户/助手消息，分割线就会被错误挂到压缩后的首条回复下面。
- 2026-04-09 工具/子会话/蜂群这类会再次回注给模型或被父任务继续消费的结果文本，不能在各业务模块里各自维护 1500/2000 之类的小截断上限；结果预算必须统一复用全局工具结果截断策略，否则上游明明允许 20k 级结果，链路中途却会先把工蜂输出裁短，最终表现成“母蜂拿到的 observation 被莫名截断、前端和 timeline 看起来都像工具自己只返回了这么多”。
- 2026-04-09 子智能体完成结果若已经被 `subagent_control.wait` 显式消费并回给父线程，后端就必须把这次 `run_id/dispatch_id` 标记为“已消费”，后续 `auto-wake` 调度在入队前和真正执行前都要再次拦截；否则同一份子智能体结果会先被 `wait` 用一次、再被父线程隐藏补跑用一次，前端最终会误表现成“默认智能体多回了一条、右栏顺序错乱甚至像又调用了一次子智能体”。

- 2026-04-08 聊天页这类强实时区域里，首个工具事件需要立刻可见的组件不能继续走异步懒加载；否则即便 `workflowItems/workflowStreaming` 已经进入 store，虚拟列表首帧仍可能先按“无循环区”测量并把真正的智能体循环区域延后到终态才出现。此类链路至少要做到两点：关键实时组件同步装载，且在首个 live tool entry 到达时自动展开并输出父/子两层可见性 debug。

- 2026-04-08 蜂群右侧栏若同时混排真实 session 消息与派生子智能体消息，发送者身份不能再回退到“当前全局 dispatchTarget”；必须按 `sessionId` 持久缓存/恢复 target identity。并且消息自动滚动监听不能写成“getter 每次返回新数组”的 `watch(() => [..])`，否则轮询刷新会被误判成新消息，最终一起表现成角色串位、默认智能体头像丢失和上翻后仍被强制拉到底部。

- 2026-04-08 子智能体工具若把单次 `spawn` 的子会话默认配置成 `auto_wake=true`，父线程会在子会话晚于父回复完成时被系统悄悄补起隐藏新轮次；前端只隐藏内部 user、不隐藏后续 assistant 时，就会表现成“母蜂又回复了一次、还顺手又调了一次子智能体”。单子会话默认应只发父事件不自动续跑，`accepted` 回包也必须显式标注 `reply_pending`，避免模型把排队成功误判为任务完成。

- 2026-04-08 蜂群页这类跨页面共享会话态的排障，不能临时散落 `console.log`；必须统一接入可开关的共享 debug 通道，并同时覆盖“会话选择/强制水合/子智能体拉取/右栏滚动”四类关键节点，默认静默、复现时再开启导出，否则问题一旦依赖切页与实时竞态就很难复盘。

- 2026-04-08 蜂群页右侧栏若绑定母蜂主线程连续对话，发送链路不能每轮重新按 `listSessions` 漂移选会话，也不能把蜂群内部的非主线程 `remember` 回写到全局 `lastSessionByAgent`；应优先复用当前 dispatch session，并只让真正 `is_main` 的线程影响主线程映射，否则第二条消息就会被切到新会话并把右栏内容清空。

- 2026-04-08 蜂群页右侧栏若命中的正是聊天 store 当前活跃会话，回放消息源不能盲目信任 `chatStore.messages`；蜂群页自发流不会天然驱动消息页前台数组，必须在“当前活跃会话优先”之外再比较缓存新鲜度，并在缓存更近时把它回写到活跃数组，否则会出现“回复完成了但右栏/聊天页都要刷新才出现”的假延迟。

- 2026-04-08 蜂群页右侧栏若要展示子智能体与父智能体之间的真实往返，subagent 预览层不能只保留 `summary`；必须把 `user_message / assistant_message / error_message` 原始字段一路保留下来，再在右栏按时间线并入可见消息流，否则画布能看到子智能体节点变化，右栏却永远缺少“父派发给子”和“子回传给父”的真实消息。

- 2026-04-08 蜂群页拿共享聊天会话做右栏回放时，`preloadSessionDetail` 不能直接复用“warm 且已有缓存就短路返回”的策略；这类视图承担的是跨页面的最新态同步，必须允许带 `minLastEventId` 的强制刷新并在命中当前活跃会话时同步回填 `chatStore.messages`，否则就会稳定复现“蜂群里已完成、聊天页切回仍像旧线程，刷新才恢复”。

- 2026-04-08 本地 `openai_compatible/llama.cpp` 做上下文压缩摘要时，不能把原始消息栈里的 observation 大包、tool_calls 结构和多条 system 原样透传给摘要模型；压缩摘要请求必须先转成纯文本转录并压低单条 observation 体积，否则很容易触发模板解析异常、服务重载窗口或连接中断，且编排层还要把这类连接失败统一按长退避重试而不是立刻终止整轮。

- 2026-04-08 蜂群页右侧栏若承载某个 dispatch session 的连续对话，就不能再并行维护一套独立的 beeroom 注释消息流；必须直接复用真实 session message source，并允许 dispatch 预览在 `targetAgentId` 缺失时仅凭 `sessionId` 回放，否则会同时出现“回复不进右栏”和“切页返回子智能体消失”的双重假状态。

- 2026-04-08 子智能体完成后若要 auto-wake 父会话，不能立即用同一 `parent_session_id` 再发一轮请求；应先等待父会话 session lock 释放，且 `send/history` 在当前父会话下只有一个子会话时要允许自动回落命中，否则模型一旦抄错长 `session_id` 就会把“已完成的 spawn”误演化成多轮无效重试。

- 2026-04-07 蜂群画布的派生节点布局不能用固定像素偏移拍脑袋定；父卡、子卡尺寸和分支方向一旦变化，固定前向 gap 很容易让新卡几何上仍压在父卡上。分支坐标应按轴向/侧向投影后的卡片半尺寸动态计算安全间距，再叠加少量 padding。

- 2026-04-07 蜂群画布若依赖 `dispatchSessionId -> getSessionEvents/getSessionSubagents` 做回放，组件卸载时不能用通用 reset 逻辑把 dispatch 状态清空后再写回 `sessionStorage`；离开页面应只停止本地 controller/watch，保留回放指针，否则切页返回会把本来可恢复的母蜂/工蜂/子智能体投影直接擦掉。

- 2026-04-07 蜂群页、消息页与聊天 store 只要共享同一会话域，就不能各自手写“会话摘要 merge”与“新建线程复用”规则；必须统一经过 store 同步 `is_main/缓存/最近会话映射`，且显式“新建线程”只能复用当前空白线程，否则就会出现“蜂群派发后母蜂主线程漂移”和“非主线程下点一次先跳旧会话”的跨入口状态分叉。

- 2026-04-06 政策长文（尤其 OCR 来源条约）若 H1 标题混入噪声或缺少中英关键词锚点，模型会在 `read_file` 与 `search_content` 间反复低价值翻页；治理应优先在文档顶部补“检索导航锚点”，并把高价值章节（如 `PART XV/第十五部分`）标题统一为可稳定命中的中英双语形式。

- 2026-04-06 工具结果要区分“模型决策必需字段”和“排障/审计字段”：模型 observation 必须在数组转 JSONL 前做行级去噪（如 command_session_id/output_meta 等），否则会把高噪声元信息放大进上下文；同时要保留 `duration_ms` 这类前端状态依赖字段，避免 UI 功能（耗时）随压缩策略一起被误删。

- 2026-04-05 聊天页“刷新/新建线程”这类会触发会话重建或重水合的操作，必须进入统一前台阻塞态（遮罩+焦点回收+发送门禁）；仅靠按钮 `disabled` 无法阻止输入框回车抢发，容易产生“消息被吃掉”与线程切换竞态。
- 2026-04-05 会话刷新自愈不能先验信任 `runtime=running`；针对“消息已终态但运行态脏住”的场景，刷新流程必须先做至少一次强制 detail hydration 校准，再决定是否保持 busy，否则会出现“页面内刷新无效、只能整页刷新恢复”。
- 2026-04-05 当前用户轮次的 pending assistant 清理不能只处理“最后一条命中的 assistant”；尾部若存在“完成态 assistant + 更早的运行态 assistant”混排，`findPending` 不能被最后一条非运行态提前截断，终态收敛需对 latest user turn 后的 assistant 运行标记做全量清理。
- 2026-04-05 聊天 busy 聚合要有“线程终态优先”兜底：当 runtime 已终态且 send/resume/compact 控制器都已清空时，应抑制消息层残留 running 标记对按钮态的反向劫持，否则会出现“终态事件齐全但发送按钮仍停留中止”。
- 2026-04-05 聊天发送按钮态必须由“会话 loading + runtime + 消息 streaming”三信号联合收敛，并提供终态强制回落保护；任何单信号先行（仅看 loading 或仅看消息标记）都会导致“看起来结束了但按钮不还原”的分叉。
- 2026-04-05 聊天流终态判定不能只依赖 `final/error`；`turn_terminal`、`queue_fail` 与 `thread_status` 的非运行态都必须参与前端收敛，并且 WS `closeOnFinal` 也要同步识别这些终态，否则会出现“消息已完成但发送按钮仍是停止、刷新按钮无效”的假运行态。
- 2026-04-04 网页抓取工具不能把“拿到 HTML/文本片段”直接视为成功；若正文抽取兜底回落到原始 HTML，会把前端壳页脚本当正文返回，进而让模型误判网页可读。治理上至少要做到：兜底基于净化后的 DOM、识别动态壳页/验证页，并在可用时明确升级到浏览器读取。
- 2026-04-04 流式失败（如 429）会同时触发本地失败兜底与后续回放/详情补齐；若 assistant 缺少稳定 stream_round 且去重不识别 request_id，同一失败会被渲染成两条气泡。应在发送时预设本地轮次并在去重阶段按 request_id 合并。
- 2026-04-04 工具失败元信息不能只挂在顶层返回包；如果 `error_meta` 没有同步进入 `data.error_meta`，前端工作流、结果归一化和重试治理会各自看到不同的失败语义，最终表现为“条目显示失败但看不出原因”“明明是超时却没被当成可重试超时”。
- 2026-04-04 文件工具的“行范围兼容”和“结果压缩”必须成对设计：若 `read_file` 把模型常用的 `start_line=0` 视为非法并悄悄退回默认大范围读取，再叠加外层 20k 字符裁剪，模型就会误以为自己只读了小片段，实际却被整段长文前部信息带偏；治理上至少要同时做到零起始归一化、同文件相邻切片合并，以及在 `search_content` 中对单长文件命中做跨区段分散暴露。
- 2026-04-04 `read_file` 不能把反向区间或“默认大窗口 + 外层截断”伪装成普通成功；`start_line > end_line` 必须 fail fast，让现有失败治理接管，而长文默认窗口/截断结果则要显式回传 `continuation_required` 和“先 search_content 再定点读”的提示，否则模型会在同一文件上重复低价值读取并误以为工具还能继续吞全文。

- 2026-04-04 交互流自愈依赖 startedAt/lastEventAt 时，必须兼容“历史残留 controller 无时间戳”的情况；在远端 last_event_id 已领先本地时应立即清理，否则会出现 watch 从未真正启动、聊天页持续空白。
- 2026-04-04 聊天页凡是用 `sendController/resumeController` 作为门禁（阻断 watcher、watchdog 或前台 hydration）前，必须先做“残留自愈清理”（aborted、远端已 idle、远端 event id 已前进但本地长时间无活动）；否则一旦控制器卡死就会把整条实时渲染链长期锁住，表现为“页面无消息直到手动刷新”。
- 2026-04-04 preserveWatcher 只应保护 sending/resuming 交互流，不应把仅 watch 存活当成阻断前台 hydration 的条件；否则 watch 的 reconcile 详情会被全部拦截，出现“页面看起来不再更新，必须手动刷新才看到消息”的假死。

- 2026-04-08 蜂群右侧栏给母蜂续聊时，不能把页面缓存的 `dispatchSessionId` 当成最高优先级真源；母蜂必须始终跟随当前 `main session`，并在页面恢复/路由切换后按最新会话列表自动纠偏，否则会出现“蜂群继续发到旧线程、Messenger 已切到新主线程、两边消息各走各的”的状态分叉。
- 2026-04-08 蜂群 `direct_send/batch_send` 若未显式指定 `sessionKey`，必须强制为工蜂新建干净线程并把它绑定为新的 `main session`；只有调用方明确 pin 到某个 `sessionKey` 时才允许复用旧线程。同时 team task 终态仍要由底层 `session_run` 完成时异步回写，不能只依赖工具同步返回，否则外层工具一旦超时或模型重试，就会出现同一工蜂被重复唤起、旧分支长期卡在 running 的分叉状态。
- 2026-04-08 蜂群等待超时要区分三种语义：省略参数表示走系统默认、显式正数表示限时等待、显式 `0` 表示立即返回快照；若默认语义需要“母蜂阻塞等工蜂全部返回”，就必须同时取消蜂群内部等待上限和 orchestrator 外层工具总超时，否则仍会在约 300 秒处被截断成假超时。
- 2026-04-08 手动上下文压缩的前端渲染不能等收到 `compaction/progress` 事件后才把占位 assistant 改成压缩分割线；用户主动触发时必须在 watcher 创建轮次占位消息的第一帧就打上手动压缩标记，否则会先闪出普通气泡，后续再被 hydrate/marker 合并放大成重复展示。
- 2026-04-08 Messenger 智能体循环区的实时显隐不能只依赖消息内容 watcher 或终态 hydration；当同一条 assistant 消息在运行中只增长 workflow/子面板高度时，虚拟列表必须主动重测最新消息行并给可见消息挂 `ResizeObserver`，否则就会出现“工具已调用但循环区要等刷新或终态才出现”的假死。
- 2026-04-08 Messenger 主消息区承载的是 reasoning、工具工作流、压缩分割线、子智能体面板这类高动态高度内容，不适合继续使用虚拟列表；这类会话页优先保证稳定的实时渲染与贴底滚动，超长历史性能问题应改用旧消息分页/折叠，而不是复杂行高虚拟化。
- 2026-04-08 蜂群页若要回放某个父会话“当前轮次”派生出的子智能体，子智能体查询不能只按 `parent_session_id` 全量拉取；必须支持 `parent_turn_ref / parent_user_round / latest_turn_only` 这类父轮次过滤，否则旧轮次分支会混进当前右栏与画布，切页恢复时尤其容易误判成“又派生了一次”或“历史子卡突然复活”。
- 2026-04-09 上下文压缩要区分“可落库摘要”和“请求阶段二次裁剪”：摘要落库前必须校验是否为空、占位或半截片段；后续若请求预算不足，应优先裁边缘保留消息或直接移除摘要，不能把摘要前缀裁成 `[上下文` 这类坏片段继续发给模型。
- 2026-04-09 上下文压缩不能把“手动压缩”和“智能体循环内自动压缩”共用同一套保留策略：手动压缩可偏向可读性，保留首尾轮次；自动压缩和溢出恢复应走 `summary-first`，优先保住已提交摘要，再牺牲边缘历史与 recent-user window。
- 2026-04-09 面向模型的工具在参数校验失败时，不能只返回一句自然语言报错；必须同时返回结构化 `error_meta`、缺失字段、可直接照抄的 `example` 和下一步提示，并且像 `batch_send` 这类显式缺参错误要在创建 run/task 之前前移拦截，否则模型会在同一 schema 错误上空转重试，前端还会看到伪开始状态。


- 2026-04-09 `agent_swarm.spawn` 不能复用 `sessions_spawn/subagent_control` 这条子智能体派生链路；蜂群工蜂虽然同样会新建子 session，但语义上是“蜂群工蜂主线程”而不是真子智能体。若后端复用子智能体 run metadata 与 `auto_wake` 默认值，前端就会把工蜂误投影成子智能体卡片，且工蜂完成后还会隐藏补唤母蜂，最终撞上父线程当前继续执行而报 `USER_BUSY`。
- 2026-04-09 蜂群右侧栏的母蜂会话绑定不能被“最新活跃 session”或全量会话 `updated_at` 牵着走；前端必须优先跟随显式 `is_main` 主线程，在主线程信息暂未落稳时保持当前已绑定母蜂 session，否则二次发送会漂移到别的线程，右侧栏消息与 `subagent_control` 子卡也会因为 session 被重绑而一起清空或消失。
- 2026-04-09 聊天页这类流式增量很多的折叠面板，不能只把 `<details>` 当成“样式容器”而不显式受控；外层循环区和内层工具条目的 `open` 必须都由响应式状态驱动，并在列表增量更新后回写 DOM，同步消除浏览器原生折叠态漂移，否则就会反复出现“运行中自动展开、结束后又收起”这类看似随机的状态错乱。
- 2026-04-09 聊天页消息状态条这类“时间型运行状态”不能只靠纯函数里直接读 `Date.now()`；界面若没有独立的秒级 tick，倒计时文本就永远不会刷新。同时，像 `llm_stream_retry` 这种瞬时流事件，不能只躺在 `workflowItems` 里指望 hydration 时再现，必须把 attempt/delay/next_retry_at 这类关键重试元数据同步提升到 assistant 消息顶层并写入本地快照，否则一旦页面刷新就会回退成“请求中”的假状态。
- 2026-04-09 手动上下文压缩分割线这类“消息局部状态 + 会话整体运行态”混合组件，不能只看 marker 自己的 `workflowStreaming/stream_incomplete`；页面刷新或 hydration 后，局部 streaming 标记可能先丢，但线程实际上仍在运行。manual marker 至少要叠加会话级 busy/runtime 作为运行态兜底，否则就会出现“后台还在压缩，分割线却先变成已完成”的假终态。
- 2026-04-10 蜂群页右侧栏与画布恢复不能只依赖慢轮询或本地 `final` 占位：母蜂/工蜂终态要同时由 `dispatch preview` 终态变化、team 关键 realtime 事件和缓存恢复三条链路主动追平；否则最容易复发“最终回复要刷新才出现、切页回来子卡先空、节点状态长期滞后”的假不同步。
- 2026-04-10 蜂群前端这类“滚动粘底 + 画布投影 + 动态子卡”的问题，不能只靠纯函数回归；至少要保留一套固定视口的浏览器 harness 做 Playwright 级验证，否则像“右栏不该强制回底”和“工蜂阴影会话误投影成子智能体”这类问题很容易回归。
- 2026-04-10 蜂群右侧栏里由 `dispatch preview` 派生出来的运行消息，不能直接把“当前 live subagents 列表”当最终展示源；像工蜂 shadow run 这类派生消息在 preview 收口时会瞬间消失，必须改成按消息 key 增量归档到当前会话缓存，再由母蜂主线程消息与派生运行消息做稳定合并。
- 2026-04-10 蜂群主智能体的终态不能由子智能体终态反推；`dispatch preview` 若直接用 `subagents.some(failed)` 上卷主状态，会把“子智能体失败但主智能体仍在处理/已正常收尾”的场景错误标成主失败。主失败只能依据主线程自己的 `error/final/turn_terminal` 事件判定。
- 2026-04-10 蜂群里的 `agent_swarm` 工蜂影子会话必须一分为二地处理：画布投影层要继续把它过滤掉，避免误显示成“子智能体卡片”；但右侧栏消息、工蜂节点运行态与工具工作流又必须把同一个影子会话当成实时真源去水合最终回复和 workflow。只做过滤不做回灌，就会复发“工蜂消息缺失、节点卡在等待空闲、工具流要刷新才出来”三连问题。
- 2026-04-10 `subagent_control.wait` 这类会先做结果压缩/包装的等待接口，若还要顺手登记 `auto_wake` 已消费，就必须在压缩前基于原始 wait 结果执行，或让抑制函数显式兼容 `data.items` 包装结构；一旦压缩阶段丢掉 `parent_session_id`，父线程就会误判“结果尚未被内联消费”，随后又被子智能体完成回调补唤一次，最终出现隐藏 internal user + 重复 assistant 回复。
