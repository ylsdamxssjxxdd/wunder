# 经验教训（每次一条）

- 2026-04-24 hive_pack 这类导入导出与打包模块里的单元测试、测试夹具和文档记录，不能再混入带场景暗示的具体组织名、阵营名、岗位名或业务技能名；即便只在测试里出现，也会把不必要的业务语义扩散进仓库并反复回流。后续占位值统一使用中性标识，例如 Worker Alpha、skill_alpha、技能甲、蜂群甲 这类不携带场景意图的名字。

- 2026-04-23 编排这种“前端本地预占轮次 + 后端再按母蜂主线程历史回算正式用户轮次”的链路，涉及消息时间窗口时不能让前后端各自默认秒/毫秒单位，更不能只信前端本地 `Date.now()` 去抑制已取消轮次；一旦取消接口写入的抑制区间单位漂移，后端重新统计母蜂用户消息时就会把已停止的尝试再次算成真实轮次，表现为“停止后重开直接跳下一轮”。这类轮次撤销逻辑必须统一时间单位，并尽量回锚到后端真实落库的用户消息时间。
- 2026-04-23 编排/蜂群右侧栏这类“把母蜂主线程消息重新投影为简化聊天记录”的界面，不能直接把 session 里的所有 assistant 片段都当成对用户可见回复；母蜂在派发工蜂时会短暂产生只有 `stream_event_id`/流式工作流状态、但尚未形成正式 `history_id` 的中间 assistant 段，这类消息如果被右侧栏读取，会表现成“母蜂 @ 你”却实际是在给工蜂派单。同时 relay 缓存也不能做只增不减的 append-only 合并，否则就算后续过滤生效，旧的 `subagent:*` 脏消息仍会跨刷新复活。右侧栏投影必须只保留稳定终态消息，relay 持久化要按当前派生结果收敛。
- 2026-04-23 编排页与蜂群页这类共享同一批工蜂运行态语义的画布，不能各自复制一套 `queued/pending/awaiting_idle` 到 UI 状态的映射；一旦编排页私有判定漂移，就会出现同一任务在蜂群页显示“执行中”、在编排页显示“排队中”。这类节点状态必须复用共享 helper，并做跨页面一致性回归测试。
- 2026-04-22 编排页这种“运行态跨母蜂流式、工蜂 team run、预览卡片三套信号”的界面，不能再靠聊天内容或局部消息出现与否去推断“本轮已结束”；运行按钮、状态灯和卡片忙闲必须优先绑定统一 runtime status，并且“终止本轮”与“清空本轮态势”要解耦，否则就会出现母蜂仍在跑但页面先回空闲、以及用户终止后态势被误删的回归。
- 2026-04-22 编排/蜂群画布这类“默认节点布局依赖成员数组顺序”的界面，成员输入顺序绝不能直接吃数据库 `updated_at DESC` 或实时回包原序；否则用户只要改过某个智能体卡片，刷新后默认布局就会按“最近更新顺序”重排，表现成工蜂位置漂移。成员顺序必须在后端输出和前端消费两侧都稳定排序，并用刷新回归测试守住。
- 2026-04-21 编排时间线这类“多条 run 会出现相同业务 round_id（如 `round_01`/`round_02`）”的树状列表，前端渲染 key 绝不能只用轮次号；一旦主链和分支节点共用同一个 Vue key，切换选中态时就会把不同分支的 DOM 误复用，表现成主链节点点击后前序节点消失、分支挂到错误列、再点一次又恢复。时间线节点 key 必须至少包含 `orchestrationId + roundId`，并配套多分支同轮次号回归测试。
- 2026-04-21 编排态势的“预设态势”与“正式轮次已采用态势”不能混成只靠 `plannedSituations` 一份数据展示；已完成轮次的追溯真相必须始终能从 `round_state.rounds[].situation` 恢复，并在前端运行态合并与设置态势弹窗里回填，否则轮次一完成就会表现成当前态势被清空、历史态势无法追溯。
- 2026-04-21 编排页消费后端 `round_state` 时，不能把“远端快照里同 index 轮次的 `user_message`/`finalized_at` 暂时缺失”直接当成真相覆盖本地运行态；一旦已完成轮次被短暂降级为空消息，时间线就会把中间节点吃掉，只在切换分支后靠另一轮 hydration 偶然恢复。前端必须对同 run 的 rounds 做稳定合并：允许合法清空最前沿 pending 轮次，但不允许已完成中间轮次被坏快照回退。
- 2026-04-21 编排页这类“节点选中 + 右侧编辑器”联动界面，编辑器草稿不能只监听 `activeRoundId` 切换；如果同一个轮次对象在 finalize/restore 后内容被异步补全而 id 不变，输入区就会停留在旧值或空值，表现成“当前态势偶发清空，点别的节点再回来恢复”。选中态同步至少要覆盖 `orchestrationId + roundId + round.situation` 这类内容维度。
- 2026-04-21 编排提示词注入必须保持单一职责：角色说明只说身份和协作，目录说明只出现在每轮产物目录块，态势只出现在态势块；母蜂不要再替系统转述产物路径，工蜂首次说明也不要重复目录约束，否则模型会在多处相似指令之间自行合并、沿用旧目录或忽略真正的后端注入目录。
- 2026-04-22 编排工蜂的产物目录提示只能说明“文件应该写到哪里”，不能暗示“系统已自动生成文件”或要求无条件汇报“已产出文件”；否则模型会把交付目标误报为落盘事实。提示词应柔性提醒：确认写入后再汇报文件，纯文本结论则不要声明产物文件。
- 2026-04-21 编排历史记录一旦支持分支与分支的分支，任何“关闭当前运行/切换到别的运行”的路径都不能再用 `build_history_record_from_state` 从活跃 state 重新生成历史并覆盖旧记录；那会把 `parent_orchestration_id / branch_root_orchestration_id / branch_depth / branch_from_round_index` 全部洗回根分支默认值，前端时间线就会在嵌套分支场景下丢失中间链路。关闭历史时必须优先在已有 record 上更新状态与轮次，并用“root -> branch -> branch-child”回归测试守住。
- 2026-04-21 编排里的产物目录提示不能再让前端母蜂注入、后端工蜂自动注入、实际文件工具解析各用一套路径语义；一旦母蜂看到的是相对 `round_xx/工蜂名`、工蜂拿到的是另一种目录、工具实际又按 `/workspaces/.../orchestration/...` 工作，就会稳定出现分支后目录失配、母蜂转述与系统注入不一致。以后凡是改编排目录规则，必须同时核对“母蜂每轮注入”“工蜂首次/每轮注入”“工具可直接写入路径”三层，并优先统一到可直接使用的绝对工作区路径。
- 2026-04-20 编排停止不能只停母蜂会话或只清前端状态；只要存在 team run 和工蜂子会话，就必须级联取消实际 worker 会话，并同步清空当前轮次的态势文件、plannedSituations 和本地临时草稿，否则刷新或重进后旧轮次会重新复活。
- 2026-04-20 蜂群页这类依赖 `after_event_id` 续订实时事件的画布，不能在切换或进入蜂群时无条件把前端 realtime cursor 清零；即使当前没有活跃 dispatch 会话，也要按 group 维度持久化并恢复最后消费到的事件游标，否则服务端会把整段历史 `team_*` 事件重新回放，前端就会把旧任务链路再次当成实时过程播放，表现为卡片和连线抽动。

- 2026-04-19 编排发送链路里，母蜂注入消息必须在后端 reserve 用户轮次成功后，基于后端返回的真实 round_index 拼接；不能先用前端本地 latestRound 推断后再 reserve，否则一旦本地轮次缓存过期或被回看态影响，母蜂会持续收到错误的旧轮次提示。

- 2026-04-19 编排提示词和工具路径能力必须成对落地：如果提示词要求模型使用短相对产物路径，就必须让文件工具在编排态能直接解析规范目录（如 `round_01/...`）；同时要明确“一次性列本轮目录、批量读取相关文件、不要猜 `orchestration/orch_*`、不要改写轮次目录名”，否则模型会在长 run id 和错误轮次目录上反复重试。

- 2026-04-20 编排轮次目录这种会被模型直接抄写的约定，优先使用低认知负担的短 canonical 形式；本次统一收敛为 `round_xx`。只改提示词不改工具解析，或前后端/画布卡片未同步切到同一 canonical 命名，都会把同一轮产物拆进两套目录。

- 2026-04-19 编排分支的语义是“保留分支点，后续从分支点的下一用户轮次继续”，不能把分支点轮次清空成待重写轮次；否则从第 1 轮回看后继续发送会再次写入第 1 轮，时间线和产物切片都会表现成始终停在第一轮。

- 2026-04-19 编排页这种“全量母蜂聊天消息 + 按用户轮次切片展示”的界面，切片结果只能作为视图派生数据，不能再反向写回蜂群聊天运行时缓存；否则全量消息源会被当前轮次覆盖，后续轮次匹配只能看到旧切片，最终表现为“用户继续发送但始终停在第 1 轮”。

- 2026-04-19 编排这类“右侧栏消息按轮次回看”的界面，不能只靠消息时间戳窗口切片；本地插入用户消息、流式回填消息、后端 round.created_at 经常存在秒/毫秒精度差或时序抖动，极易把后续用户消息继续吸进上一轮。前端应优先按“正式用户轮次对应的母蜂用户消息顺序”切片，时间戳只做兜底。

- 2026-04-19 编排这类“提示词注入”和“产物目录读取”双链路并存的功能，凡是目录命名规则调整，必须同时修改前端提示词拼装、前端产物目录读写、后端工蜂派发注入三层；只改显示文案或只改后端一侧，都会立刻出现“母蜂看到的新目录”和画布实际读取目录不一致的问题。

- 2026-04-19 编排/分支这类既允许“回看旧轮次”又允许“继续向前发送”的系统，不能把前端当前激活轮 `activeRoundId` 当成“业务上的最新正式轮次”直接参与下一轮发送判定；否则用户只要回看过旧节点，后续发送就可能错误复用旧轮次甚至持续卡在第 1 轮。发送侧的下一轮计算必须优先基于 round_state 中最后一个正式轮次，而展示侧的选中轮只负责回看。

- 2026-04-18 OpenAI-compatible/Qwen 这类 reasoning 模型的工具调用不能假设一定落在 `message.tool_calls` 或 assistant `content` 里；它们可能把 `<tool_call><function=...><parameter=...>` 直接塞进 reasoning/thinking 通道。编排层若只兼容 JSON tool_calls 和 `<name>/<arguments>` 变体，就会把真实最终回复误判成“空答复”。后端必须同时从 reasoning 解析 Qwen/XML 工具协议，并在可见文本清洗层剥离这类内部 XML，避免再次出现“工具已调用但用户只看到兜底报错/原始协议文本”。

- 2026-04-17 编排/分支这类“回看历史再继续”的系统，不能让历史节点上的态势编辑直接落回当前 run 的 situation.txt；历史查看态只能维护前端分支草稿，必须等用户真正发送消息、系统创建新分支后，再把草稿写入新 run。否则就会稳定出现“还没分支，原主线已被污染”的语义错位。

- 2026-04-15 帮助手册这类 `/docs` 静态站点若同时被 `server/web`、用户侧 iframe、桌面桥接和 Electron 资源副本复用，修复不能只停留在 `scripts/docs_site` 或 `web/docs` 源产物；必须同时检查“实际运行入口”是否仍指向过期的 `frontend-dist/docs`，并给入口 URL 与 `site.js/site.css` 加版本指纹，否则就会出现“代码已修、用户侧仍整页刷新”的缓存/副本错位假回归。
- 2026-04-15 使用说明书这类嵌入在管理员侧/用户侧里的静态文档站，若用户抱怨“点侧栏页面整体刷新、滚动条回顶部”，就不能只补 `sessionStorage` 式滚动恢复；整页导航仍会重建 iframe/DOM，体验照样割裂。真正的修复应把同站内链接统一收敛到一套 History API + 局部内容切换导航器，并保留当前 query（如 `embed=user/admin`）与侧栏滚动位置。
- 2026-04-14 聊天气泡里的 `Token 消耗` 不能再从 `usage.total_tokens` 或 `contextTokens` 兜底；这些字段在中止、重试、多轮工具链里经常只是“最后一次模型调用快照”或“当前上下文占用”。前端统计必须只认显式扣费字段和 `round_usage/request_consumed_tokens`，否则用户一中止就会稳定看到“Token 消耗退化成最后一次上下文占用”的假值。
- 2026-04-14 蜂群这类“先 list 出目标、再由模型二次派发”的工具提示不能鼓励模型手抄长 `agent_id`；opaque ID 一旦被截断、漏字或被模型自行规范化，就会直接派发失败。默认提示应优先引导使用稳定且可读的 `agent_name`，只有在重名、缺名或用户明确指定时才回退到 `agent_id`。
- 2026-04-14 蜂群右侧栏这类“Markdown 渲染后再二次 hydrate 图片资源卡”的消息区，不能只在 `renderedHtml` 变化时加载图片；切页返回、DOM 复用或组件重新挂载时，旧的 `workspaceState=loading` 也可能原样残留。恢复路径必须先重置未完成图片卡的状态和 `src`，再重新 hydrate，否则就会出现“图片一直显示加载中，刷新整页才恢复”。
- 2026-04-14 聊天页、消息页、用户世界页、蜂群页这类都会渲染工作区图片资源卡的前端入口，不能各自复制一套 `loading label / filename 解析 / svg blob 规范化 / 图片卡重置` 基础逻辑；这种重复代码一旦某处补了“恢复时重置 stale loading”，别处没补就会继续分叉。共用的卡片基础能力要尽快收敛到统一 util，业务页只保留权限、缓存键和刷新过滤等差异逻辑。
- 2026-04-14 复用聊天页 `chat-shell`/工作目录这类依赖容器级 CSS 变量的组件时，不能只复制 DOM class 或组件本身；如果新落点在 `.messenger-view` 外（如蜂群画布弹层），必须同步接入同一套主题变量入口，否则文件名、图标按钮 hover/focus 等前景色会回退到旧默认值，出现白字白底这类伪随机视觉问题。
- 2026-04-14 前端凡是“导入资源/手动刷新资源目录”这类会改变技能、工具、知识库可用性的动作，不能只更新当前面板自己的局部列表；必须同时失效对应缓存并广播全局资源变更事件，否则像智能体设置页这类依赖另一份 catalog 缓存的界面就会继续把新资源判成缺失，最终表现为“明明导入成功却还得按 F5 才恢复”。
- 2026-04-14 “同账号新登录顶旧登录”这类会话治理不能只按 `user_id` 做全局世代；后台线程允许离线继续运行，不代表不同前端入口应该互相挤掉。登录态至少要带上明确的 `session_scope`（如 `user_web/admin_web`），并让 token 校验与在线 WS 强退统一按 `user_id + session_scope` 生效，否则管理员侧和用户侧会被同一套顶号逻辑误伤。
- 2026-04-13 工具条目的 token 消耗若来自触发该工具的模型轮次，就不能只从 `tool_result/detail` 反推；必须把 `tool_call` 与 `modelRound` 绑定，并在 `round_usage` 到达时按轮次回填 usage。还要兼容 `round_usage` 与 `tool_call` 的到达顺序不稳定，否则就会出现“同一条智能体循环始终只有耗时没有 tok”的现象。
- 2026-04-13 智能体循环条目的 token 展示不能依赖 detail 文本本身；当 tool_result 优先展示 `model_observation` 或其它面向模型的摘要时，真实的 `request_consumed_tokens` 可能已不在 detail 里。工作流条目创建时必须额外携带一份轻量 usage 快照元数据，UI 再基于该快照取数，避免“明明后端有真实消耗但右侧循环不显示”的分叉。
- 2026-04-13 智能体循环条目的耗时与右侧元信息布局，不能继续依赖 `detail/model_observation` 临时反解析，也不能让可选字段直接决定横向排版；工作流项创建时要显式固化 `durationMs` 这类展示元数据，前端右侧 token/耗时区域则要用固定列布局渲染，否则一旦某列缺失就会表现成“有的工具有耗时、有的没有，且右侧信息整体错位”。
- 2026-04-13 前台会话在保留 live watcher 的同时做 detail hydration 时，assistant 合并不能只靠 event_id、stream_round 或时间窗；同一用户轮次里的同内容回复在前后端 `created_at` 漂移后仍要允许按“用户轮次锚点 + 内容相似”兜底匹配，否则会把同一条正式回复误当成未匹配 live assistant 再插入一次，表现为第二次用户消息前多出一条重复气泡，刷新后才恢复。
- 2026-04-12 记忆复刻这类“跨作用域复制记录”的链路，不能复用源记录的全局唯一 ID；否则目标侧 upsert 会把复制误写成迁移，直接把源记录挪走。复制时必须为目标侧生成新 ID，并同步重映射 supersedes/superseded 这类内部关联。
- 2026-04-12 蜂群画布这类“任务树投影 + 实时事件 + 惰性子状态抓取”混合链路，不能只让工蜂子智能体卡片依赖任务签名变化或轮询；凡是 `team_task_dispatch/update/result/finish/error` 已经进入前端 realtime 的路径，都要同步触发子智能体预览刷新。否则画布会出现“状态数据实际上已到，但子卡片要刷新页面才出现”的假滞后。
- 2026-04-11 `prepare_child_session` 这类“创建临时子智能体会话”的基础链路，不能顺手把 `agent_thread` 也改绑到子会话；否则一旦蜂群工具先把工蜂主线程切到新 worker session，工蜂内部再调用子智能体工具时，就会把同一个 `agent_id` 的主线程再次劫持到子智能体会话上，表现成“工蜂回到错误线程/子智能体钻进工蜂主线程”。主线程重绑必须只保留给蜂群工具首次唤起工蜂这类明确需要刷新一等现实状态的场景。
- 2026-04-11 `subagent_control.status/wait` 这类“父会话已经拿到子结果，但返回摘要为省 token 被裁掉 `parent_session_id`”的链路，不能把 auto-wake 抑制完全建立在结果项自带父会话字段上；调用点既然天然知道当前父会话，就必须把当前 `context.session_id` 作为 consumed key 的兜底来源，否则后台 `subagent_auto_wake` 会在父线程解锁后把同一个子结果又补跑成第二轮。
- 2026-04-11 子智能体工具链里只要父线程可能先 `wait`、后 `status`、再由后台 `auto_wake` 接管，就不能把“结果已被父线程消费”的判定只挂在 `wait(completed)` 一条路径上；`status` 命中 terminal child run 时也必须同步标记 consumed，并且 `wait` 的参数别名要兼容模型常写的 `timeout_seconds`，否则父线程会先自己回一条，再被 `subagent_auto_wake` 隐藏补跑第二轮。
- 2026-04-11 `subagent_control.history` 这类在工具内二次包装通用会话工具结果的链路，不能假设上游返回的是裸 payload；若 `sessions_history` 已经返回 `{ ok, data: { session_id, messages } }`，下游再直接从根节点取 `session_id/messages` 就会稳定把真实子会话历史吞成空数组，进而诱发模型继续多打一轮“确认/总结”动作。包装层必须先解 `data`，并给 `history + mistyped session_id` 补回归测试。
- 2026-04-11 上下文压缩分割线这类“终态 manual marker + 旧的 running 快照/前台 live 消息”混合回放的链路，不能把 streaming 标记做无脑 OR 合并，也不能在远端终态 marker 已存在后再把本地 running marker 重新插回去；否则就会稳定出现“已完成压缩又变回压缩中”或重复分割线。必须优先保护同一压缩实例的终态 marker，并用手动压缩 running->completed->下一轮 busy->rehydrate 的回归测试守住。
- 2026-04-11 蜂群/子智能体这类“父会话 timeline 投影到画布卡片”的前端链路，不能把 `SUBAGENT_TARGET_NOT_FOUND` 之类工具级错误直接当成新的子智能体实例；凡是 `status=not_found` 且缺少 `sessionId/runKind/requestedBy` 的条目，都应在收集、合并、投影三层至少拦一层，并用“真实 spawn + wait/run_id 输错”回归测试守住，否则一次真实子智能体会被误画成两次派生。
- 2026-04-11 聊天页、消息页、头像态和失败提示这类“assistant 是否仍在运行”的 UI 不能各自手写 `stream_incomplete/workflowStreaming/questionPanel/state` 判定；一旦有压缩恢复、待确认或前台 hydration 介入，就会稳定出现“头像还在转、状态条已结束”之类分叉。运行态必须收敛到一套共享 helper，并用回归测试同时覆盖 pending、compaction、failure 三类边界。
- 2026-04-10 `subagent_control.status/wait` 这类“跟在 spawn 后面消费子会话结果”的工具，不能只在 `send/history` 做单子会话纠错；模型一旦把刚返回的 `session_id` 抄错一位，若 `status/wait` 直接按原值查，就会稳定落入 `SUBAGENT_TARGET_NOT_FOUND -> 再次 spawn -> 父线程 hidden auto_wake 补跑` 的连锁反应。对“当前父会话下仅一个直接子会话、且只给了一个 session_id”的安全场景，应在入口先自动纠正到真实 child session。
- 2026-04-10 上下文压缩分割线这类“历史终态 marker + 当前会话又开始忙”的 UI，状态判断不能把全局 `sessionBusy` 直接映射回旧 marker；否则手动压缩明明已经 `completed`，下一轮一发送消息又会被误刷成 `running`。这类状态必须优先看 marker 自己的终态快照，只有在 marker 尚无终态事件时才允许用会话忙碌态兜底。
- 2026-04-10 上下文压缩排障不能把“交互组数”和“归一化 block 数”混成一个字段名；事件日志至少要同时给出目标 `retained_*_exchange_count_target`、目标 `retained_*_block_count_target` 和实际 `retained_*_messages_debug`，否则看到一个裸的 count 根本无法判断保留的是 2 组交互、4 个 block，还是只剩 1 组旧内容。
- 2026-04-10 上下文压缩里看到 `retained_tail_message_count=0` 时，不能立刻断定是首尾拼接丢尾；必须先同时看 `source_interaction_block_count` 和 `source_interaction_messages_debug`。若压缩输入本身只剩 1 组或 2 组归一化交互块，那么 tail 为 0 是数据事实，不是重建 bug。
- 2026-04-10 子智能体完成后的 `auto_wake` 不能只检查“结果是否已被 wait 显式消费”和“父会话锁是否释放”；还必须在入队前与真正执行前同时复核父会话是否仍处于可续跑状态。只要父线程已经被用户取消、monitor 处于 `cancelling/cancelled`，或父会话本身不再 `active`，就必须直接放弃隐藏续跑，否则会稳定复现“用户已中止，但工蜂晚点完成后又把主智能体补唤醒并产出最终回复”。
- 2026-04-09 `read_file` 这类面向模型的文件工具，输入上限和对外 schema 不能脱节，更不能把超限文件静默截成前几项；否则模型明明请求了多文件，执行层却悄悄只处理一部分，最终会被误诊成“工具没读到”或“模型胡乱重试”。同时 `start_line` 单独出现时必须把默认窗口语义在工具描述里写死；否则模型很容易把递增 `start_line` 误当成另一种翻页协议。
- 2026-04-09 蜂群页这类需要在画布上回放子智能体生命周期的视图，不能只依赖 `getSessionSubagents(session_id)` 这类“当前仍存活子会话”接口；一旦子智能体带 `cleanup=delete` 结束，live 列表就会归零，前端会误表现成“子智能体一会有一会没有”。正确做法是把父会话 timeline 里的 `subagent_dispatch_item_update/subagent_status` 终态快照并入实时列表，按父轮次过滤后再投影到画布。
- 2026-04-09 聊天消息状态条这类“运行中状态”不能只看当前 assistant 自己的 `workflowItems`；上下文压缩、手动压缩标记等事件可能会以独立 assistant/marker 消息插入同一会话。状态判定若不带会话级视角，就会稳定出现“实际已经在压缩，但页面仍显示请求中/模型输出中”的假状态。
- 2026-04-09 蜂群画布若直接消费父会话的 `getSessionSubagents(parentSessionId)` 结果，不能只靠 `run_kind/requested_by` 识别真实子智能体；母蜂刚派出去的工蜂 session 在 run 记录尚未落库的时窗里会先以“只有 session、没有 run metadata”的影子项出现。投影层必须再拿蜂群任务自己的 `spawned_session_id/target_session_id/session_run_id` 做身份排除，否则就会把工蜂误画成子智能体卡片。
- 2026-04-09 聊天页智能体循环区这类“实时流事件先到、历史 hydration 后到”的组件，运行时归并层不能只识别单一字段形态；`eventType/toolName/toolCallId/commandSessionId` 与 `event/tool/tool_call_id/command_session_id` 必须统一归一化，并在出现“raw items 已有但 entryCount 仍为 0”时直接打出样本 keys/字段值，否则就会反复表现成“工具明明调用了，循环区要切页或刷新才出现”。
- 2026-04-09 蜂群页的 dispatch 流即使已经收到 `final`，也不能假设会话历史快照会同步立刻包含最终 assistant；右栏若只在流结束瞬间 hydrate 一次，很容易出现“母蜂明明答完了，但蜂群页要手动刷新才看见最终回复”。终态后必须再做一小段带基线比较的补偿 hydration，直到 session history 里真的出现新的 assistant 终态。
- 2026-04-09 蜂群页这类“聊天右栏 + 任务画布”双视图共享同一身份信息的界面，发送者名称与头像不能分别靠本地字符串和局部默认图猜测；必须先把 `Default Agent/__default__/默认智能体` 收敛到单一规范名，再让右栏消息和画布子节点共用同一套头像解析入口，否则就会出现同一智能体在不同区域名字不一致、子智能体卡片退回固定占位头像的问题。
- 2026-04-09 手动上下文压缩分割线的落点不能按 `user_round` 或“本轮最后一条 assistant”去绑定；压缩是独立事件，水合时必须按 `compaction` 事件真实时间插入消息流。否则只要同一轮后面又出现新的用户/助手消息，分割线就会被错误挂到压缩后的首条回复下面。
- 2026-04-09 工具/子会话/蜂群这类会再次回注给模型或被父任务继续消费的结果文本，不能在各业务模块里各自维护 1500/2000 之类的小截断上限；结果预算必须统一复用全局工具结果截断策略，否则上游明明允许 20k 级结果，链路中途却会先把工蜂输出裁短，最终表现成“母蜂拿到的 observation 被莫名截断、前端和 timeline 看起来都像工具自己只返回了这么多”。
- 2026-04-09 子智能体完成结果若已经被 `subagent_control.wait` 显式消费并回给父线程，后端就必须把这次 `run_id/dispatch_id` 标记为“已消费”，后续 `auto-wake` 调度在入队前和真正执行前都要再次拦截；否则同一份子智能体结果会先被 `wait` 用一次、再被父线程隐藏补跑用一次，前端最终会误表现成“默认智能体多回了一条、右栏顺序错乱甚至像又调用了一次子智能体”。
- 2026-04-08 聊天页这类强实时区域里，首个工具事件需要立刻可见的组件不能继续走异步懒加载；否则即便 `workflowItems/workflowStreaming` 已经进入 store，虚拟列表首帧仍可能先按“无循环区”测量并把真正的智能体循环区域延后到终态才出现。此类链路至少要做到两点：关键实时组件同步装载，且在首个 live tool entry 到达时自动展开并输出父/子两层可见性 debug。
- 2026-04-08 蜂群右侧栏若同时混排真实 session 消息与派生子智能体消息，发送者身份不能再回退到“当前全局 dispatchTarget”；必须按 `sessionId` 持久缓存/恢复 target identity。并且消息自动滚动监听不能写成“getter 每次返回新数组”的 `watch(() => [..])`，否则轮询刷新会被误判成新消息，最终一起表现成角色串位、默认智能体头像丢失和上翻后仍被强制拉到底部。
- 2026-04-08 子智能体工具若把单次 `spawn` 的子会话默认配置成 `auto_wake=true`，父线程会在子会话晚于父回复完成时被系统悄悄补起隐藏新轮次；前端只隐藏内部 user、不隐藏后续 assistant 时，就会表现成“母蜂又回复了一次、还顺手又调了一次子智能体”。单子会话默认应只发父事件不自动续跑，`accepted` 回包也必须显式标注 `reply_pending`，避免模型把排队成功误判为任务完成。
- 2026-04-08 蜂群页这类跨页面共享会话态的排障，不能临时散落 `console.log`；必须统一接入可开关的共享 debug 通道，并同时覆盖“会话选择/强制水合/子智能体拉取/右栏滚动”四类关键节点，默认静默、复现时再开启导出，否则问题一旦依赖切页与实时竞态就很难复盘。
- 2026-04-08 蜂群页右侧栏若绑定母蜂主线程连续对话，发送链路不能每轮重新按 `listSessions` 漂移选会话，也不能把蜂群内部的非主线程 `remember` 回写到全局 `lastSessionByAgent`；应优先复用当前 dispatch session，并只让真正 `is_main` 的线程影响主线程映射，否则第二条消息就会被切到新会话并把右栏内容清空。
- 2026-04-08 蜂群页右侧栏若命中的正是聊天 store 当前活跃会话，回放消息源不能盲目信任 `chatStore.messages`；蜂群页自发流不会天然驱动消息页前台数组，必须在“当前活跃会话优先”之外再比较缓存新鲜度，并在缓存更近时把它回写到活跃数组，否则会出现“回复完成了但右栏/聊天页都要刷新才出现”的假延迟。
- 2026-04-08 蜂群页右侧栏若要展示子智能体与父智能体之间的真实往返，subagent 预览层不能只保留 `summary`；必须把 `user_message / assistant_message / error_message` 原始字段一路保留下来，再在右栏按时间线并入可见消息流，否则画布能看到子智能体节点变化，右栏却永远缺少“父派发给子”和“子回传给父”的真实消息。
- 2026-04-08 蜂群页拿共享聊天会话做右栏回放时，`preloadSessionDetail` 不能直接复用“warm 且已有缓存就短路返回”的策略；这类视图承担的是跨页面的最新态同步，必须允许带 `minLastEventId` 的强制刷新并在命中当前活跃会话时同步回填 `chatStore.messages`，否则就会稳定复现“蜂群里已完成、聊天页切回仍像旧线程，刷新才恢复”。
- 2026-04-08 本地 `openai_compatible/llama.cpp` 做上下文压缩摘要时，不能把原始消息栈里的 observation 大包、tool_calls 结构和多条 system 原样透传给摘要模型；压缩摘要请求必须先转成纯文本转录并压低单条 observation 体积，否则很容易触发模板解析异常、服务重载窗口或连接中断，且编排层还要把这类连接失败统一按长退避重试而不是立刻终止整轮。
- 2026-04-08 蜂群页右侧栏若承载某个 dispatch session 的连续对话，就不能再并行维护一套独立的 beeroom 注释消息流；必须直接复用真实 session message source，并允许 dispatch 预览在 `targetAgentId` 缺失时仅凭 `sessionId` 回放，否则会同时出现“回复不进右栏”和“切页返回子智能体消失”的双重假状态。
- 2026-04-08 子智能体完成后若要 auto-wake 父会话，不能立即用同一 `parent_session_id` 再发一轮请求；应先等待父会话 session lock 释放，且 `send/history` 在当前父会话下只有一个子会话时要允许自动回落命中，否则模型一旦抄错长 `session_id` 就会把“已完成的 spawn”误演化成多轮无效重试。
- 2026-04-07 蜂群画布的派生节点布局不能用固定像素偏移拍脑袋定；父卡、子卡尺寸和分支方向一旦变化，固定前向 gap 很容易让新卡几何上仍压在父卡上。分支坐标应按轴向/侧向投影后的卡片半尺寸动态计算安全间距，再叠加少量 padding。
- 2026-04-07 蜂群画布若依赖 `dispatchSessionId -> getSessionEvents/getSessionSubagents` 做回放，组件卸载时不能用通用 reset 逻辑把 dispatch 状态清空后再写回 `sessionStorage`；离开页面应只停止本地 controller/watch，保留回放指针，否则切页返回会把本来可恢复的母蜂/工蜂/子智能体投影直接擦掉。
- 2026-04-07 蜂群页、消息页与聊天 store 只要共享同一会话域，就不能各自手写“会话摘要 merge”与“新建线程复用”规则；必须统一经过 store 同步 `is_main/缓存/最近会话映射`，且显式“新建线程”只能复用当前空白线程，否则就会出现“蜂群派发后母蜂主线程漂移”和“非主线程下点一次先跳旧会话”的跨入口状态分叉。
- 2026-04-06 政策长文（尤其 OCR 来源条约）若 H1 标题混入噪声或缺少中英关键词锚点，模型会在 `read_file` 与 `search_content` 间反复低价值翻页；治理应优先在文档顶部补“检索导航锚点”，并把高价值章节（如 `PART XV/第十五部分`）标题统一为可稳定命中的中英双语形式。
- 2026-04-06 工具结果要区分“模型决策必需字段”和“排障/审计字段”：模型 observation 必须在数组转 JSONL 前做行级去噪（如 command_session_id/output_meta 等），否则会把高噪声元信息放大进上下文；同时要保留 `duration_ms` 这类前端状态依赖字段，避免 UI 功能（耗时）随压缩策略一起被误删。
