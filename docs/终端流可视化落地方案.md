# 终端流可视化落地方案

## 1. 目标

本方案聚焦 `execute_command` 在用户侧聊天工作流区域的“终端级实时感知”落地，目标不是简单继续堆 `tool_output_delta`，而是把命令执行升级为可持续观测、可恢复、可交互、可控成本的终端流能力。

目标体验：

- 用户在工具开始后 100ms 级看到稳定的“正在执行”命令块，而不是空白等待。
- 命令输出按终端语义持续推进，进度条、颜色、交互提示、逐行日志都能自然呈现。
- 页面刷新、切换、短时断线后，可以重新接管仍在运行的命令会话。
- 工作流卡片负责概览，真实终端内容在工作区终端面板里呈现，避免大段文本把聊天流压垮。
- 默认部署下也成立。当前仓库默认 `sandbox.mode=sandbox`，所以方案必须优先覆盖 sandbox 路径，而不是只做本机直连。

## 2. 当前现状

当前链路已经具备“基础增量输出”能力，但还不是终端流：

- 本机 `execute_command` 会并行读取 `stdout/stderr`，通过 `tool_output_delta` 推送文本分块。
- 输出分块受 `server.stream_chunk_size` 控制，默认 `1024` 字节，小块输出会被攒批后再发。
- 执行模式是 `stdout/stderr` pipe，不是 PTY，TTY 感知程序、交互程序、进度条程序体验不完整。
- 前端工作流区域把命令输出当作工具详情字符串处理，会压缩、截断、节流，不是终端视图。
- 多命令 `execute_command` 顺序执行时，前端实时区只稳定保留第一条命令文本，子命令粒度不够。
- 默认 sandbox 分支当前基本是“执行完成后一次性返回结果”，这会直接破坏实时体感。

结论：当前实现可以做到“边跑边看日志”，但做不到 Codex 那种“命令会话正在活着”的感觉。

## 3. 核心设计原则

### 3.1 命令会话一等公民

不要再把命令流仅视为某个工具详情字符串。要把一次命令执行抽象成独立的 `command_session`，拥有自己的：

- 会话 id
- 生命周期
- 输出流
- 控制面
- 恢复语义
- 资源上限

### 3.2 工作流卡片与终端视图分层

工作流卡片只承担：

- 命令摘要
- 运行状态
- 退出码
- 最后若干行预览
- 打开终端面板入口

真实终端流只放到“单独终端面板”里，避免：

- 大段 ANSI 文本进入 Vue 响应式树
- 多个并发命令同时挂多个重型终端实例
- 历史消息滚动性能被拖垮

### 3.3 原始流不全量落库

高频终端字节流不适合逐条写入 `stream_events` 或 monitor：

- 写放大严重
- 回放成本高
- 数据库与前端都扛不住

因此：

- `stream_events` 只记录命令会话元事件和最终摘要
- 活跃命令原始 scrollback 仅保存在内存态环形缓冲区
- 页面刷新后的“接管”优先从内存快照恢复
- 完成后沉淀摘要、尾部输出和元信息，不追求永久保存完整 ANSI 历史

### 3.4 协议先统一，再做 UI

必须先统一本机与 sandbox 的命令会话协议，否则前端做得再漂亮也只是局部方案。

## 4. 目标架构

整体上收敛为四层：

1. `CommandSessionBroker`
   - 统一管理活跃命令会话、订阅者、环形缓冲、控制指令和清理。
2. `CommandSessionRunner`
   - 负责真正启动命令，可落到本机 PTY 或 sandbox PTY。
3. `Chat Realtime Bridge`
   - 负责把命令会话事件桥接到聊天 WS/SSE 通道，并接收前端控制消息。
4. `Terminal View Runtime`
   - 前端终端面板与工作流卡片的双层展示。

## 5. 协议设计

### 5.1 标识模型

新增以下稳定标识：

- `tool_call_id`
  - 继续表示一次工具调用。
- `command_session_id`
  - 表示一个真实命令会话。
- `command_index`
  - 表示 `execute_command` 内第几条子命令。

关系：

- 一次 `execute_command` 可以对应多个 `command_session_id`
- 每个 `command_session_id` 必须绑定一个 `tool_call_id`
- 前端工作流按 `tool_call_id` 聚合，终端面板按 `command_session_id` 订阅

### 5.2 服务端推送事件

新增事件命名空间，避免继续滥用 `tool_output_delta`：

- `command_session_start`
- `command_session_snapshot`
- `command_session_delta`
- `command_session_status`
- `command_session_exit`
- `command_session_closed`
- `command_session_summary`

推荐字段：

```json
{
  "event": "command_session_start",
  "tool_call_id": "tool_123",
  "command_session_id": "cmd_456",
  "command_index": 0,
  "command": "pnpm dev",
  "cwd": "/workspaces/u1/project",
  "tty": true,
  "interactive": true,
  "shell": "bash",
  "started_at": "2026-03-27T09:00:00Z"
}
```

```json
{
  "event": "command_session_delta",
  "command_session_id": "cmd_456",
  "seq": 128,
  "stream": "pty",
  "encoding": "base64",
  "delta": "Li4u",
  "byte_len": 384,
  "truncated": false
}
```

```json
{
  "event": "command_session_exit",
  "command_session_id": "cmd_456",
  "exit_code": 0,
  "signal": null,
  "duration_ms": 8241,
  "stdout_bytes": 120334,
  "stderr_bytes": 0
}
```

### 5.3 客户端控制事件

新增控制面：

- `command_session_attach`
- `command_session_write`
- `command_session_resize`
- `command_session_terminate`
- `command_session_detach`

其中：

- WS 模式下直接复用聊天实时连接发送控制消息。
- SSE 兜底时，输出继续走 SSE，控制通过独立 HTTP POST 接口发送。

SSE 兜底接口建议：

- `POST /wunder/chat/command_sessions/{id}/attach`
- `POST /wunder/chat/command_sessions/{id}/write`
- `POST /wunder/chat/command_sessions/{id}/resize`
- `POST /wunder/chat/command_sessions/{id}/terminate`

### 5.4 与现有 `tool_output_delta` 的兼容

迁移期保留兼容层：

- 新客户端优先消费 `command_session_*`
- 老客户端继续接收精简版 `tool_output_delta`
- `tool_result` 仍保留最终结果与输出摘要

这样可以灰度升级，不需要一次性切断旧链路。

## 6. 后端执行面设计

### 6.1 本机执行改为 PTY 优先

本机执行不再默认 `stdout/stderr` pipe，而是：

- 默认走 PTY
- PTY 不可用时退回 pipe
- 非交互命令也允许在 PTY 中运行，以获得更接近真实终端的输出行为

实现建议对齐 Codex 的思路，直接引入 `portable-pty`：

- Linux/macOS 用 Unix PTY
- Windows 用 ConPTY

这样可以获得：

- 逐字符/逐块终端输出
- ANSI 颜色
- 终端尺寸感知
- stdin 写入能力

### 6.2 sandbox 执行必须同步升级

由于默认配置是 `sandbox.mode=sandbox`，所以 sandbox 不是附加项，而是主路径。

sandbox 侧新增独立命令会话服务：

- `POST /sandboxes/command_sessions`
- `GET /sandboxes/command_sessions/{id}/events` 作为 SSE 兜底
- `POST /sandboxes/command_sessions/{id}/write`
- `POST /sandboxes/command_sessions/{id}/resize`
- `POST /sandboxes/command_sessions/{id}/terminate`
- `GET /sandboxes/ws` 作为多会话复用的 WS 主通道

推荐优先落地 sandbox WS 多路复用：

- API 服务与 sandbox 服务之间维持长连接
- 单连接承载多个命令会话
- 避免每个命令单独建立 HTTP 轮询链路

服务端内部抽象为统一 trait：

```rust
trait CommandSessionRunner {
    async fn spawn(&self, spec: CommandSessionSpec) -> Result<SpawnedCommandSession>;
    async fn write(&self, session_id: &str, bytes: Vec<u8>) -> Result<()>;
    async fn resize(&self, session_id: &str, size: TerminalSize) -> Result<()>;
    async fn terminate(&self, session_id: &str) -> Result<()>;
}
```

这样本机与 sandbox 对上层完全同构。

### 6.3 Broker 与缓冲策略

每个活跃 `command_session` 在 Broker 中维护：

- 基础元信息
- 进程句柄
- 订阅者列表
- 单调递增 `seq`
- 原始字节环形缓冲区
- 纯文本预览摘要
- 统计信息

推荐默认预算：

- `ring_buffer_bytes`: 256 KB
- `hard_cap_bytes`: 1 MB
- `summary_tail_lines`: 80
- `idle_gc_after_exit_ms`: 300000

说明：

- 256 KB 足够支撑页面刷新后的快速接管
- 1 MB 作为单会话绝对硬上限，避免异常命令把内存撑爆
- 会话结束后保留 5 分钟，便于晚到的前端接管最终态

### 6.4 推送节流与批处理

替换当前“仅按字节门限攒批”的策略，改为“时间窗 + 字节窗 + 行结束”联合触发：

- `flush_bytes >= 512`
- 或 `flush_interval >= 16ms`
- 或检测到换行/光标控制序列结束

这样可以兼顾：

- 小输出及时可见
- 高频输出不把 WS 包打太碎
- 进度条类程序仍然平滑

### 6.5 慢客户端与回放

使用有界队列，禁止无限堆积：

- 每个订阅者一个 bounded channel
- 落后过多时不继续堆 delta，直接发送 `command_session_snapshot_required`
- 客户端收到后发起重新 attach，服务端返回最新快照和当前 `seq`

这比静默丢字节更可控。

## 7. 前端终端可视化设计

### 7.1 视图结构

Phase 1 只做单活动终端面板，不做多终端并排：

- 工作流卡片：列表态摘要
- 终端面板：当前聚焦命令的真实终端

交互规则：

- 点击运行中的命令卡片，右侧或下方打开终端面板
- 若已存在终端面板，则切换绑定的 `command_session_id`
- 命令结束后卡片保留摘要，面板可继续查看最近 scrollback

### 7.2 终端渲染组件

前端建议采用 `xterm.js`：

- `xterm`
- `xterm-addon-fit`
- 可选 `xterm-addon-web-links`

不建议一开始就开 `webgl`：

- 工作流页面实例数不多时，canvas 已足够
- `webgl` 在旧驱动和远程桌面环境下问题更多
- 先把协议与缓冲模型做对，再做 renderer 优化

### 7.3 Store 分层

不要把完整终端文本塞进 chat store 的响应式字符串。

新增独立 `commandSessions` store：

- 非响应式保存原始会话缓冲
- 响应式只暴露状态、退出码、最后更新时间、摘要尾部
- `xterm.write()` 直接消费 delta，不经过大字符串拼接

工作流卡片只消费轻量摘要：

- `last_lines`
- `exit_code`
- `status`
- `duration_ms`

### 7.4 卡片与面板职责

卡片展示：

- 首行命令
- cwd
- 运行中/成功/失败
- 退出码
- 最近 6 到 12 行预览

面板展示：

- 真正的 ANSI 终端
- 可选复制、清屏、重新连接、终止
- 窗口 resize 同步到底层 PTY

### 7.5 多命令场景

一次 `execute_command` 多条命令时：

- 工作流父卡片显示命令总数与整体状态
- 每个子命令有独立 `command_session_id`
- 终端面板按子命令切换

不再把多条命令混成同一段实时详情字符串。

## 8. 高性能专项设计

### 8.1 后端

- 原始输出只保存在 Broker 环形缓冲，不进入大对象 JSON 历史。
- `command_session_delta` 默认不写 monitor 明细，不写 `stream_events` 全量正文。
- 所有订阅队列必须 bounded，避免慢消费者放大内存。
- Broker 用共享只读快照和增量 seq，不做重复字符串拷贝。
- 命令结束后只保留摘要和尾部文本，及时释放 PTY 与大缓冲。

### 8.2 前端

- 页面只允许一个活跃 `xterm` 实例，避免十几个命令各挂一个终端。
- 终端写入按 `requestAnimationFrame` 或 16ms 批量 flush。
- 工作流卡片只保留纯文本摘要，不保留完整 scrollback。
- 长会话切换时复用终端实例，禁止通过 `:key` 暴力重建整个聊天主体。
- 面板不可见时暂停高频 UI 刷新，只保留状态更新。

### 8.3 协议

- 浏览器侧优先 WS，多路复用命令流和控制流。
- 输出增量优先按字节传输，不做重复结构化包装。
- `delta` 建议统一 base64，服务端同时维护 stripped preview，避免浏览器重复做 ANSI 清洗。
- attach 快照设上限，例如首次最多回放最近 128 KB，超出部分只给“已截断”提示与摘要。

### 8.4 安全

终端 ANSI 序列不能完全无过滤直通：

- 保留颜色、光标、清屏等常规控制序列
- 禁止或忽略高风险 OSC 能力，例如剪贴板写入类序列
- 前端链接打开必须走显式确认或受控处理

交付物：

- 指标面板

验收：

- 可以按用户或环境灰度
- 出现异常时可回落到摘要流模式

## 10. 验收指标

建议使用以下口径：

- `tool_call -> command_session_start visible`：P95 < 120ms
- `delta emit -> xterm paint`：P95 < 80ms，本机；P95 < 150ms，sandbox
- 活跃命令页面刷新后重新接管时间：P95 < 500ms
- 单会话服务端额外内存：默认 < 512 KB，硬上限 1.5 MB
- 100 个并发活跃命令时，命令流相关额外内存可控在 80 MB 级以内
- 长命令运行 10 分钟以上，聊天页面 FPS 和交互不明显劣化

## 11. 风险与回滚

主要风险：

- PTY 跨平台行为差异
- sandbox 长连接代理复杂度提升
- 终端原始字节流引入新的内存压力
- 前端若仍把全文放进响应式 store，会出现卡顿回归

回滚策略：

- 使用 feature flag 控制 `command_session_*` 新链路
- 新链路异常时退回当前 `tool_output_delta + tool_result` 模式
- 前端终端面板可以独立开关，先上工作流摘要化，再上真实终端

## 12. 推荐实施顺序

按当前仓库现状，推荐顺序不是“先做前端”，而是：

1. 先冻结协议与会话模型
2. 先做本机 PTY 验证
3. 立即做 sandbox 对齐
4. 再做前端单终端面板
5. 最后补交互控制与灰度

原因很直接：默认部署走 sandbox，如果 sandbox 仍是一锤子返回，前端再像 Codex 也只是视觉假象。

## 13. 结论

要让 wunder 的工作流区域真正具备 Codex 那种命令执行“实时流”体感，关键不是继续优化 `tool_output_delta` 卡片，而是把命令执行从“工具输出文本”升级为“可接管的终端会话”。

这次落地的最小闭环应当是：

- 后端命令会话化
- sandbox 同构流式化
- 前端单终端面板
- 工作流摘要与终端分层
- bounded queue + snapshot 恢复

做到这五点，用户才会明显感觉到命令真的在实时运行，而不是过一会儿刷出一坨日志。
