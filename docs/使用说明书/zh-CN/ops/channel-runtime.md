# 渠道管理

## 原理

渠道是 Wunder 与外部平台的桥梁。通过渠道，用户可以在飞书、企业微信等熟悉的平台直接与智能体对话。

### 渠道架构

```
外部平台（飞书/微信/QQ/WhatsApp/XMPP）
        ↓ Webhook / 长连接
    统一接入层（ChannelHub）
        ↓
    入站队列（快速接收）
        ↓
    调度执行（处理消息）
        ↓
    出站队列（投递回复）
        ↓
    渠道适配器（发送到平台）
```

### 关键组件

| 组件 | 作用 |
|------|------|
| **渠道账号** | 绑定到外部平台的身份 |
| **绑定关系** | 渠道用户与系统用户的映射 |
| **入站链路** | 外部消息进入系统 |
| **出站链路** | 系统回复发送到外部 |
| **运行日志** | 渠道运行状态记录 |

## 作用

- 让用户在熟悉的平台使用智能体
- 统一管理所有渠道的接入和运行
- 集中监控渠道健康状态
- 快速定位和解决渠道问题

## 实践

### 接入新渠道

**通用步骤**：

1. **在外部平台创建应用**
   - 如飞书自建应用、企业微信应用
   - 获取 AppID、AppSecret 等凭证

2. **在管理端添加渠道账号**
   - 管理端 → 渠道管理 → 添加账号
   - 填写渠道类型和凭证信息

3. **配置 Webhook**
   - 设置消息接收地址
   - 在外部平台配置 Webhook URL
   - 确保地址可从外部访问

4. **测试连接**
   - 在外部平台发送测试消息
   - 检查是否正常收到回复
   - 查看运行日志确认

5. **绑定用户**
   - 将渠道用户绑定到系统用户
   - 确认消息路由正确

### 监控渠道运行

**管理端查看**：
- 渠道账号状态（在线/离线）
- 消息收发统计
- 异常告警记录

**用户端查看**：
- 运行日志面板
- 连接状态
- 最近活动记录

### 排查渠道问题

当渠道出现问题时，按这个顺序排查：

**1. 账号是否在线**
- 管理端查看账号状态
- 如果离线，检查凭证和网络

**2. Webhook 是否正常**
- 检查 Webhook 地址是否可访问
- 查看平台侧的消息推送日志
- 确认回调地址没有变更

**3. 入站是否正常**
- 外部消息是否进入了系统
- 检查入站日志

**4. 出站是否卡住**
- 回复是否在队列中积压
- 检查出站队列状态
- 查看是否有发送失败的记录

**5. 文件附件是否正常**
- 渠道客户端是否能打开附件链接
- 检查临时目录下载服务是否正常

### 常见渠道问题

| 问题 | 可能原因 | 排查方向 |
|------|----------|----------|
| 收不到消息 | Webhook 未通 | 检查地址和配置 |
| 消息发不出去 | 出站队列积压 | 查看出站队列 |
| 附件打不开 | 下载链接失效 | 检查临时目录 |
| 连接时断时续 | 网络或凭证问题 | 检查账号和网络 |
| 回复重复 | 消息重复投递 | 检查幂等处理 |

## 渠道日常维护

### 定期检查

- 账号在线状态
- 消息收发是否正常
- 是否有积压或失败

### 账号管理

- 批量管理渠道账号
- 查看绑定影响范围
- 清理无效绑定

### 日志管理

- 查看运行日志了解异常
- 定期归档历史日志
- 关注告警和重连记录

## 常见误区

- **渠道问题 = 模型问题？** 不是。大部分渠道问题出在接入层，先查接入再查模型。
- **消息发出去了就完事了？** 不是。要确认对方收到了，还要确认出站队列正常。
- **渠道配置好就不用管了？** 不是。需要持续监控和维护。
