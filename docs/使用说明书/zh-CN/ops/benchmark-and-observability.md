# 监控与评估

## 四类观测能力

Wunder 的观测不是单一监控页，而是四类各有分工的能力：

| 能力 | 看什么 | 回答什么问题 |
|------|--------|-------------|
| **会话监控** | 线程运行细节 | 线程在干什么？卡在哪？ |
| **工具统计** | 工具使用情况 | 哪个工具最忙？有没有异常？ |
| **性能采样** | 系统链路基线 | 系统扛得住吗？慢在哪？ |
| **吞吐压测** | 并发承受能力 | 最大能扛多少？ |
| **能力评估** | 任务完成质量 | 答得好不好？有没有退化？ |

## 会话监控

### 看什么

- 活跃线程列表
- 每个线程的当前状态和阶段
- Token 占用统计
- 工具调用记录
- 事件时间线

### 什么时候用

- 排查线程卡住或异常
- 查看某个会话的执行详情
- 强制取消异常线程
- 触发上下文压缩

## 工具统计

### 看什么

- 各工具的调用次数
- 工具调用成功率
- 哪些用户在使用哪些工具
- 工具调用与线程状态的关系

### 什么时候用

- 某个工具成了性能瓶颈
- 工具调用异常频繁
- 评估工具使用模式

## 性能与吞吐

### 性能采样

测量单请求的延迟基线：
- 模型响应速度
- 工具执行速度
- 端到端延迟

### 吞吐压测

测试系统在高并发下的表现：
- 最大并发会话数
- 队列处理速度
- 资源使用率

### 两者区别

- **性能采样**：单请求级别的延迟
- **吞吐压测**：系统级别的承压能力

## 能力评估（Benchmark）

### 看什么

- 任务完成质量评分
- 不同模型的能力对比
- 版本间的能力回归检测

### 与压测的区别

- **吞吐压测**：关注"快不快、扛不扛得住"
- **能力评估**：关注"答得好不好、有没有退化"

### 什么时候用

- 新版本发布前评估
- 对比不同模型的能力
- 验证优化是否有效

## 关键指标

### 健康指标

| 指标 | 正常范围 | 告警阈值 |
|------|----------|----------|
| 活跃会话数 | < 最大限制的 80% | > 80% |
| 队列深度 | < 50 | > 100 |
| 错误率 | < 1% | > 5% |
| 平均延迟 | < 10s | > 30s |

### 资源指标

| 指标 | 关注点 |
|------|--------|
| CPU 使用率 | 是否有异常峰值 |
| 内存使用率 | 是否持续增长（可能有泄漏） |
| 磁盘使用率 | 工作区和日志是否过大 |
| 网络流量 | 是否有异常流量 |

### 业务指标

| 指标 | 关注点 |
|------|--------|
| 日活用户数 | 系统使用趋势 |
| 每日 Token 消耗 | 资源使用趋势 |
| 工具调用分布 | 用户使用模式 |
| 渠道消息量 | 各渠道使用情况 |

## 服务端日志

Server 模式下，运行日志保存在 `config/data/logs/server/`：
- 按天轮转的 JSONL 文件
- 默认保留 14 天
- 包含请求、工具调用、错误等记录
- 出问题时优先查看最新日志

## 实践

### 日常巡检

每天看一眼：
- 系统健康状态
- 错误率趋势
- 队列积压情况
- 磁盘空间

### 周期评估

每周或每月：
- Token 消耗趋势
- 工具使用统计
- 性能基线变化
- 能力评估结果

### 问题排查

遇到问题时：
1. 看会话监控找到异常线程
2. 查看工具调用记录
3. 检查性能指标
4. 查看服务端日志

## 常见误区

- **Benchmark = 压测？** 不是。Benchmark 评估能力质量，压测评估承压能力。
- **日志多 = 可观测性好？** 不是。要有结构、有分层、有指标。
- **只看一层就够？** 不够。很多问题需要把监控、日志、统计一起看。
