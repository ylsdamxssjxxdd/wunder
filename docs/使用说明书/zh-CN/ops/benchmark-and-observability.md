# 监控与 WunderBench

## 四类观测能力

Wunder 的观测分为四类各有分工的能力：

| 能力 | 看什么 | 回答什么问题 |
|------|--------|-------------|
| **会话监控** | 线程运行细节 | 线程在干什么？卡在哪？ |
| **工具统计** | 工具使用情况 | 哪个工具最忙？有没有异常？ |
| **性能采样** | 系统链路基线 | 系统扛得住吗？慢在哪？ |
| **吞吐压测** | 并发承受能力 | 最大能扛多少？ |
| **WunderBench** | 任务完成质量 | 模型在 Wunder 中表现如何？有没有退化？ |

## 会话监控

### 观察内容

- 活跃线程列表
- 每个线程的当前状态和阶段
- Token 占用统计
- 工具调用记录
- 事件时间线

### 适用场景

- 排查线程卡住或异常
- 查看某个会话的执行详情
- 强制取消异常线程
- 触发上下文压缩

## 工具统计

### 观察内容

- 各工具的调用次数
- 工具调用成功率
- 哪些用户在使用哪些工具
- 工具调用与线程状态的关系

### 适用场景

- 某个工具成了性能瓶颈
- 工具调用异常频繁
- 评估工具使用模式

## 性能与吞吐

### 性能采样

测量单请求的延迟基线：
- 模型响应速度
- 工具执行速度
- 端到端延迟

### 吞吐压测

测试系统在高并发下的表现：
- 最大并发会话数
- 队列处理速度
- 资源使用率

### 两者区别

- **性能采样**：单请求级别的延迟
- **吞吐压测**：系统级别的承压能力

## WunderBench 模型评测

### 观察内容

- 任务完成质量评分
- 工具调用成功率、稳定性、效率与上下文消耗
- 不同模型的能力对比
- 版本间的能力回归检测
- `full` 全量题库评测，也可按任务组或任务 ID 手动筛选复查

### 与压测的区别

- **吞吐压测**：关注"快不快、扛不扛得住"
- **WunderBench**：关注"是否能在真实智能体链路中完成任务、有没有退化"

### 适用场景

- 新版本发布前评估
- 对比不同模型的能力
- 验证优化是否有效
- 为默认模型、裁判模型、工具启用策略提供依据

### 关键输出

- `summary.scorecard.readiness`：可用性结论，包含 `production_ready`、`usable`、`risky`、`not_ready`
- `overall_score`：任务平均得分
- `reliability_score`：通过率口径
- `tool_success_score`：工具结果成功率
- `stability_score`：完成率与多轮波动
- `weakest_suites` / `top_failures`：最需要复查的任务组和失败项

### 导出评测记录

- 在 WunderBench 页面点击“导出评测记录”，或调用 `GET /wunder/admin/wunderbench/runs/{run_id}/export`。
- 导出的 JSON 包含运行信息、任务聚合、attempt 明细、题目规格、产物摘要、transcript，以及每个模型 attempt 对应的 monitor 持久化日志。
- 新启动的 WunderBench 会使用管理员调试日志记录评测线程，导出文件可用于复盘模型请求、模型输出、工具调用、工具结果、工作区更新与 token/速度统计。
- 详细使用、评分口径和题库扩展说明见 [WunderBench 模型评测](/docs/zh-CN/ops/wunderbench/)。

## 关键指标

### 健康指标

| 指标 | 正常范围 | 告警阈值 |
|------|----------|----------|
| 活跃会话数 | < 最大限制的 80% | > 80% |
| 队列深度 | < 50 | > 100 |
| 错误率 | < 1% | > 5% |
| 平均延迟 | < 10s | > 30s |

### 资源指标

| 指标 | 关注点 |
|------|--------|
| CPU 使用率 | 是否有异常峰值 |
| 内存使用率 | 是否持续增长（可能有泄漏） |
| 磁盘使用率 | 工作区和日志是否过大 |
| 网络流量 | 是否有异常流量 |

### 业务指标

| 指标 | 关注点 |
|------|--------|
| 日活用户数 | 系统使用趋势 |
| 每日 Token 消耗 | 资源使用趋势 |
| 工具调用分布 | 用户使用模式 |
| 渠道消息量 | 各渠道使用情况 |

## 服务端日志

Server 模式下，运行日志保存在 `config/data/logs/server/`：
- 按天轮转的 JSONL 文件
- 默认保留 14 天
- 包含请求、工具调用、错误等记录
- 出问题时优先查看最新日志

## 实践

### 日常巡检

每天看一眼：
- 系统健康状态
- 错误率趋势
- 队列积压情况
- 磁盘空间

### 周期评估

每周或每月：
- Token 消耗趋势
- 工具使用统计
- 性能基线变化
- WunderBench 评测结果

### 问题排查

遇到问题时：
1. 看会话监控找到异常线程
2. 查看工具调用记录
3. 检查性能指标
4. 查看服务端日志

## 常见误区

- **WunderBench = 压测？** 不是。WunderBench 评估模型能力质量，压测评估承压能力。
- **日志多 = 可观测性好？** 不是。要有结构、有分层、有指标。
- **只看一层就够？** 不够。很多问题需要把监控、日志、统计一起看。
