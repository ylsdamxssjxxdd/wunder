# 定时任务

定时任务让 Wunder 不只会响应“现在收到的请求”，还能沿着系统时间轴主动执行治理和业务动作。它解决的是时间驱动任务如何进入统一系统能力，而不是如何写一个外部 cron 脚本。

![定时任务图：调度规则进入调度器，再到后台执行和执行记录](/docs/assets/manual/core-scheduled-tasks.svg)

## 先给结论

- 定时任务不是 `sleep` 的升级版，而是系统级异步调度能力。
- 时间本身也是系统输入，所以调度、执行、记录和失败恢复都要进入正式治理。
- 只要任务会长期运行、会影响用户或系统状态，就不该放在外部脚本黑箱里。

## 为什么它必须单列为核心

如果没有定时任务这个核心，系统里的周期行为通常会散落到：

- 外部 cron 脚本。
- 前端定时轮询。
- 某个管理员手工触发按钮。

这样做的共同问题是：触发、执行、失败、追踪和权限边界都不再统一，后台任务会逐渐失控。

## 这个核心真正保护什么

- 保护调度语义：任务为什么会在这个时间启动，需要有明确规则。
- 保护执行边界：后台任务不能和在线会话语义混成一条链。
- 保护追踪能力：每次计划执行都要有状态、记录和失败原因。

## 关键约束

| 约束 | 作用 |
|------|------|
| 必须与在线线程语义区分 | 避免把后台调度和即时会话混在一起 |
| 执行记录必须可追踪 | 回答“跑没跑、为什么失败、下一次何时执行” |
| 失败重试要有策略 | 防止一次偶发失败直接导致任务长期失效 |
| 周期任务要有边界 | 避免无限堆积、互相抢资源 |

## 设计重点

### 重点一：把“时间”收进系统，而不是放到系统外面

一旦系统需要定时巡检、定时提醒、定时维护，就说明时间已经成为业务的一部分。那它就应该进系统治理。

### 重点二：定时任务是后台执行链，而不是同步等待链

`sleep` 更像当前流程里的短暂等待；定时任务更像“系统承诺未来某个时间再做一件事”。

### 重点三：调度能力必须天然带执行记录

一个没有执行历史、下一次时间、最后状态的定时系统，运维上基本不可用。

## 最容易搞错的点

- `sleep` 不是定时任务，前者偏同步等待，后者偏异步调度。
- 定时任务不是逃离主系统的脚本口子，而是正式治理能力。
- 周期执行越自动，越需要清晰记录和兜底治理。

## 和其他核心的边界

- 和 [智能体循环](/docs/zh-CN/concepts/core-agent-loop/) 的区别：智能体循环处理“当前回合怎么跑”，定时任务处理“未来何时启动一轮运行”。
- 和 [稳定性](/docs/zh-CN/concepts/core-stability/) 的区别：稳定性负责失败恢复，定时任务负责让周期执行有正式入口。
- 和 [可观测性](/docs/zh-CN/concepts/core-observability/) 的区别：可观测性记录计划执行结果，定时任务定义计划本身。
