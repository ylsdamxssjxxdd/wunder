# Token 与配额

## 原理

Wunder 里有两个关于 Token 的概念，经常被混用：

| 概念 | 含义 | 用途 |
|------|------|------|
| **Token 账户** | 用户可支配的资源余额 | 治理和结算 |
| **Token 占用** | 当前上下文的实际大小 | 运行观测 |

### Token 账户

类似"账户余额"：

| 字段 | 含义 |
|------|------|
| 当前余额 | 还可以用的 Token |
| 累计获得 | 总共发放了多少 |
| 累计消耗 | 总共用了多少 |
| 每日发放 | 每天自动获得的量 |

**消耗规则**：按实际使用的 Token 消减余额。余额不足可能无法继续对话。

### Token 占用

类似"当前负载"：告诉你这一轮上下文实际有多大。

**关注什么**：
- `round_usage.total_tokens`：当前上下文占用
- `request_consumed_tokens`：单次请求消耗

## 作用

### Token 账户的作用

- **资源治理**：控制用户能消耗多少资源
- **公平分配**：不同等级用户有不同配额
- **防止滥用**：余额机制限制过度使用

### Token 占用的作用

- **性能观测**：上下文太大会影响响应速度
- **压缩判断**：占用过高会触发压缩
- **排障依据**：某些问题来自上下文过大

## 实践

### 查看账户余额

在"我的概况"或"系统设置"可以看到：
- 当前 Token 余额
- 累计获得和消耗
- 每日发放量

### 查看上下文占用

对话过程中可以看到：
- 当前轮次的 Token 占用
- 累计消耗

### 优化使用

- **及时开新会话**：避免单一会话过长
- **精简输入**：避免大段粘贴重复内容
- **使用 `/compact`**：主动压缩对话历史

### 管理员管理 Token

在管理端 → 用户管理 → 设置：
- 直接设置余额
- 发放额外 Token
- 扣除 Token

## 用户等级与发放

不同等级用户每天自动获得不同量 Token：

| 等级 | 每日发放 |
|------|----------|
| 一级 | 较高配额 |
| 二级 | 中等配额 |
| 三级 | 较低配额 |
| 四级 | 基础配额 |

升级时可能有额外奖励 Token。

## 常见误区

- **Token 账户 = 上下文占用？** 不是。账户是余额，占用是当前负载。
- **Token 统计 = 供应商账单？** 不是。Wunder 的统计是系统内口径，供应商有自己的计费方式。
- **余额够就一切正常？** 不一定。上下文过大也可能影响执行。
