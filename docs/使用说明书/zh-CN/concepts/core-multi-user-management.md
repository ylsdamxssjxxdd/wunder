# 多用户管理

Wunder 面向组织和个人，但一旦进入 server 形态，多用户治理就不再是附属功能，而是系统骨架本身。它决定的不是后台长什么样，而是整套运行边界怎样成立。

![多用户管理图：身份层、治理层、隔离层和结果层依次收口](/docs/assets/manual/core-multi-user-management.svg)

## 先给结论

- 多用户管理不是“登录系统”这么简单，而是运行边界、资源边界和治理边界的总和。
- 如果数据隔离、权限边界和 Token 账户控制不是前置设计，后面几乎一定要返工。
- 管理后台不是展示页，而是系统治理能力的正式入口。

## 为什么它必须单列为核心

如果没有多用户管理这个核心，系统会在增长后暴露出一系列结构性问题：

- 不知道一个请求到底代表“登录用户”还是“外部映射 user_id”。
- 线程、工作区、工具可见性和渠道都缺乏一致边界。
- Token 账户治理和权限只能在接口层打补丁，无法真正收口到运行时。

## 这个核心真正保护什么

- 保护身份边界：区分登录用户、组织用户、虚拟 `user_id` 和外部映射身份。
- 保护运行边界：线程、工作区、工具和资源都要知道“归谁、谁能碰”。
- 保护治理边界：管理员有管理权限，不等于系统到处都没有边界。

## 关键约束

| 约束 | 作用 |
|------|------|
| 默认按多用户并发设计 | 避免后期才发现系统只适合单人使用 |
| 数据隔离必须前置 | 工作区、会话、配置和日志都要先想清楚边界 |
| 权限和 Token 账户治理不能后补 | 否则很容易出现“能看不能管、能调不能控”的灰区 |
| 后台与运行时共用一套口径 | 管理员看到的治理结果要和实际运行一致 |

## 设计重点

### 重点一：身份和隔离是两层

登录用户、API Key 调用者、外部嵌入映射身份、虚拟 `user_id` 都不完全相同。系统必须分清“谁在认证”和“隔离边界按谁建”。

### 重点二：多用户治理一定会下沉到运行时

它不会停留在后台表单里，最终会落实到：

- 线程归属
- 工作区隔离
- 工具可见性
- Token 账户与资源治理

### 重点三：管理员能力必须强，但不能模糊边界

管理员可以治理系统，不代表系统可以失去边界感。真正健康的做法是“有权治理，但仍可追踪和限制”。

## 最容易搞错的点

- `user_id` 的接入语义和后台注册用户不是同一个层面的概念。
- 数据隔离不是单靠数据库表字段补丁就能兜住的。
- 管理后台不是“有了功能再拼 UI”，而是治理能力的正式入口。

## 和其他核心的边界

- 和 [渠道](/docs/zh-CN/concepts/core-channels/) 的区别：渠道讲入口共核，多用户管理讲入口背后的治理边界。
- 和 [定时任务](/docs/zh-CN/concepts/core-scheduled-tasks/) 的区别：定时任务也要服从多用户归属与权限。
- 和 [可观测性](/docs/zh-CN/concepts/core-observability/) 的区别：可观测性要能按用户与组织切分，切分规则来自多用户管理。
