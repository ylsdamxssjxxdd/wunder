# 多用户管理

## 原理

Wunder 支持个人使用，也支持组织部署。多用户管理就是从架构层面解决"多个用户如何共享系统又不互相干扰"。

它不只是"让用户能登录"，而是：
- 谁属于哪个组织
- 谁能访问什么
- 谁消耗了多少资源
- 管理员如何治理

### 层级结构

```
系统
└── 单位（组织/团队）
    └── 用户
        └── 智能体
            └── 线程
```

每个层级有明确的边界：
- **系统**：全局配置、管理员账户
- **单位**：组织成员、共享资源
- **用户**：个人文件、个人设置、Token 账户
- **智能体**：独立配置、独立线程
- **线程**：对话现场

## 作用

### 对组织

**资源治理**：
- Token 账户：控制每个用户的资源配额
- 单位管理：组织结构和成员管理
- 渠道管理：统一接入和监控

**权限控制**：
- 区分管理员和普通用户
- 控制谁能创建智能体、配置渠道
- 工具审批级别的控制

### 对管理员

**统一治理**：
- 管理端可以查看全局状态
- 管理用户账户和配额
- 监控渠道运行情况
- 配置系统参数

**审计追溯**：
- 用户操作记录
- Token 消耗统计
- 渠道消息日志

## 关键规则

### 数据隔离

每个用户的数据严格隔离：
- 工作区：你只能访问自己的文件
- 会话：你只能看到自己的对话
- 配置：你的设置不影响别人

### 身份分层

| 身份类型 | 说明 |
|----------|------|
| **登录用户** | 注册用户，有账户密码 |
| **外部映射 user_id** | 从外部系统映射的身份（如飞书用户） |
| **虚拟 user_id** | 接口调用时传入的标识（用于隔离） |

### Token 账户

Token 账户是用户可支配的资源余额：

| 字段 | 含义 |
|------|------|
| 当前余额 | 还可用的 Token |
| 累计获得 | 总共发放了多少 Token |
| 累计消耗 | 总共消耗了多少 Token |
| 每日发放 | 每天自动获得的 Token |

**消耗规则**：
- 按实际使用的 Token 消耗
- 余额不足时可能无法继续对话
- 管理员账户不受限制

## 实践

### 创建用户

1. 管理端 → 用户管理
2. 点击"创建用户"
3. 填写：
   - 用户名
   - 邮箱
   - 初始密码
   - 所属单位
4. 可选择性发放初始 Token
5. 保存

### 设置 Token

1. 找到用户 → 点击"设置"
2. Token 操作：
   - **直接设置余额**：覆盖当前余额
   - **发放 Token**：增加余额
   - **扣除 Token**：减少余额
3. 设置每日发放量

### 单位管理

1. 创建单位（部门/团队）
2. 设置单位属性
3. 把用户分配到单位
4. 配置单位级资源配额

### 权限控制

管理员可以控制：
- 哪些用户是管理员
- 哪些用户能创建智能体
- 哪些用户能配置渠道
- 审批模式的严格程度

## 常见误区

- **user_id 就是注册用户？** 不一定。user_id 可能只是隔离标识，不一定是登录用户。
- **数据隔离只靠数据库？** 不是。从工作区、会话到工具调用，都有边界控制。
- **管理员权限无限制？** 不是。管理员有治理权，但系统仍有边界和可追踪性。
