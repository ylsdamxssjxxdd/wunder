# 稳定性

## 原理

Wunder 面对的是复杂场景：长对话、高并发、多工具链、多入口。任何一个环节出错，都可能影响整个系统。

稳定性的设计哲学：**出错不扩散，失败能恢复**。不是指望"不出错"，而是出错时系统能兜住、能续跑、能解释。

### 控扩散

最危险的不是单点失败，而是失败把其他部分一起带崩。系统通过各种隔离机制控制扩散：
- **上下文压缩**：防止超长上下文拖垮线程
- **工具输出限制**：防止超大返回拖垮上下文
- **超时机制**：防止无限等待卡住系统
- **队列控制**：防止请求堆积拖垮服务

### 能恢复

失败不是终点，系统有正式恢复链路：
- **网络中断**：自动重连 + 事件补发
- **模型调用失败**：自动重试 + 降级
- **工具失败**：错误分类 + 重试策略
- **会话异常**：恢复续跑 + 状态重建

### 可复盘

同类问题要能被归类、复现、回归验证：
- 完整的错误日志
- 结构化错误信息
- 关键事件记录

## 作用

### 对普通用户

- **长对话不会断**：上下文压缩让超长对话能继续
- **工具失败有兜底**：自动重试，不直接报错
- **网络波动不怕**：自动重连，不会丢失进度
- **卡住能处理**：重置功能让你从干净状态重新开始

### 对管理员

- **系统扛得住高并发**：队列和限流控制流量
- **异常可观测**：完整的日志和监控
- **问题可追溯**：能回放事件流定位问题
- **能主动治理**：通过配置控制系统行为

## 关键机制

### 上下文超限处理

| 层级 | 处理方式 |
|------|----------|
| Level 1 | 压缩工具结果 |
| Level 2 | 摘要历史对话 |
| Level 3 | 移除早期轮次 |
| Level 4 | 仅保留核心设定和当前请求 |

### 网络中断恢复

```
连接断开 → 自动重连（指数退避）
              ↓
         缓存未推送事件
              ↓
         重连成功 → 补发事件
              ↓
         重连失败 → 提示手动刷新
```

### 工具失败分类

| 失败类型 | 处理方式 |
|----------|----------|
| 参数错误 | 让智能体修正参数 |
| 权限拒绝 | 等待用户审批 |
| 临时错误 | 自动重试 |
| 系统错误 | 有限重试 + 降级 |

### 事务一致性

涉及多个步骤的操作使用数据库事务：
- 要么全部成功
- 要么全部回滚
- 不会出现中间状态

## 实践

### 什么时候耐心等待

- 上下文压缩：系统在处理，等待即可
- 网络重连：自动在恢复，等待即可
- 工具重试：系统在处理，等待即可

### 什么时候主动介入

- **长时间无响应**：超过正常范围，可能需要取消
- **状态明显异常**：界面显示和实际不一致，考虑重置
- **审批请求**：需要你确认的操作

### 系统异常时的排查

1. 检查网络连接
2. 查看界面状态提示
3. 如果卡住，尝试取消或重置
4. 查看错误提示了解原因
5. 联系管理员查看系统日志

## 常见误区

- **只要回答出来了就是稳定？** 不是。稳定性要看长期运行表现，不只是单次。
- **重试就能解决所有问题？** 不是。错误隔离和资源治理同样重要。
- **偶发问题不用管？** 不是。没有复盘的问题可能会反复出现。
