# 稳定性

稳定性不是最后再补的一层保险，而是 Wunder 从一开始就必须围绕的硬目标。因为它面对的不是短平快问答，而是长会话、高并发、多工具、多入口的复合链路。

![稳定性图：风险源经过隔离治理、恢复链路后，避免级联故障并形成可回归系统](/docs/assets/manual/core-stability.svg)

## 先给结论

- 稳定性不是“少报错”，而是“出错时不扩散、能恢复、能复盘”。
- 一个长期运行的智能体系统，最危险的不是偶发错误，而是错误没有被结构性收住。
- 只靠 prompt、人工操作或单次热修，无法支撑真正稳定的系统。

## 为什么它必须单列为核心

如果没有稳定性这个核心，系统在规模上来后通常会这样出问题：

- 某个工具输出过大，直接拖垮整轮上下文。
- 某个网络抖动或模型超时，直接把整条链路打断。
- 某个入口异常，把其他入口也一起带进不一致状态。

这些问题不是靠“多重试几次”就能解决的，必须靠结构性治理。

## 这个核心真正保护什么

- 保护执行连续性：出错后还能继续，而不是整条链直接废掉。
- 保护资源边界：高并发、长会话和大工具输出不能无限放大。
- 保护系统可维护性：同类问题要能被归类、复现、回归验证。

## 关键约束

| 约束 | 作用 |
|------|------|
| 稳定性依赖结构保障 | 不能只靠 prompt 或人工操作兜底 |
| 高风险链路优先控扩散 | 模型、工具、网络、存储都要有隔离思路 |
| 恢复必须有正式链路 | 超时、取消、断线、失败都要有后续动作 |
| 同类问题必须能回归验证 | 避免修一次、忘一次、再犯一次 |

## 设计重点

### 重点一：稳定性首先是“控扩散”

真正危险的不是单点失败，而是失败把线程、工具、连接、队列和数据一致性一起带崩。

### 重点二：恢复链要和主链一样正式

恢复续跑、重试、补发、事务回滚、outbox、原子写入，本质上都不是旁路技巧，而是主系统的一部分。

### 重点三：资源治理是稳定性的组成部分

上下文太大、输出太长、并发过高、队列积压，本质上都属于稳定性问题，不只是性能问题。

## 最容易搞错的点

- 只要模型回答出来了，不代表系统是稳定的。
- 重试不是万能药，错误隔离和资源治理同样关键。
- 没有结构化复盘的“偶发问题修好了”通常只是暂时安静。

## 和其他核心的边界

- 和 [上下文压缩](/docs/zh-CN/concepts/core-context-compression/) 的区别：压缩治理窗口压力，稳定性治理所有高风险链路。
- 和 [实时性](/docs/zh-CN/concepts/core-realtime/) 的区别：实时性讲状态怎么送，稳定性讲异常时状态和执行还能否保持一致。
- 和 [可观测性](/docs/zh-CN/concepts/core-observability/) 的区别：可观测性帮助发现和复盘稳定性问题，但不直接替代稳定性设计。
