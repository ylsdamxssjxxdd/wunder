# 会话与轮次

## 原理

你和智能体的对话不是"你说一句、它回一句"这么简单。Wunder 把一次完整的交互拆成清晰的层级：

```
会话（一次完整对话）
├── 用户轮次 1（你说了一句话）
│   ├── 模型轮次 1（智能体思考）
│   ├── 模型轮次 2（调用了工具）
│   └── 模型轮次 3（给出最终回答）
├── 用户轮次 2（你又说了一句话）
│   └── 模型轮次 1（直接回答）
└── ...
```

### 四个层级

| 层级 | 含义 | 计数方式 |
|------|------|----------|
| **会话** | 一次完整的对话过程 | 你点击"新对话"开始 |
| **线程** | 会话的执行主体，维护状态和上下文 | 随会话创建 |
| **用户轮次** | 你每发送一条消息 | 发一条记一轮 |
| **模型轮次** | 智能体每执行一次动作 | 思考、工具调用、回答各算一轮 |

### 为什么这么拆

如果不拆分，你就搞不清楚：
- 你到底问了几次 vs 智能体走了几步
- 某次工具调用发生在第几步
- Token 消耗应该算在哪一层

拆分后，统计、回放、排障都有了清晰的坐标系。

## 作用

### 对你

- **上下文连续**：同一会话内，智能体记得你之前说了什么
- **过程可追溯**：你能看到智能体为了回答你的问题走了几步、调用了什么工具
- **状态有保障**：线程确保执行过程稳定推进

### 对系统

- **Token 统计精确**：知道每一轮消耗了多少
- **回放可定位**：能精确找到某个时间点发生了什么
- **排队有依据**：用户轮次排队，模型轮次顺序执行

## 线程的重要性

线程是会话的"执行大脑"，维护着：
- 系统提示词（创建时冻结，不再改变）
- 对话上下文（随着对话不断增长）
- 长期记忆（初始化时注入一次）
- 执行状态（运行中、等待、完成等）

**关键规则**：
- 线程的系统提示词一旦确定就冻结
- 长期记忆只在线程初始化时注入一次
- 后续轮次复用同一线程上下文

## Token 统计怎么看

Wunder 记录的是上下文占用量——当前线程实际占了多少 Token。

几个关键概念：

| 概念 | 含义 |
|------|------|
| **上下文占用** | 当前线程对话 + 工具结果的总 Token |
| **单次消耗** | 某一次请求消耗的 Token |
| **累计消耗** | 所有请求消耗的 Token 总和 |

**看什么**：主要关注上下文占用。如果占用持续增长，说明对话在变长，可能触发压缩。

## 实践

### 什么时候开新会话

- 完成了一个话题，想开始全新话题
- 当前会话上下文很长，影响响应质量
- 切换到完全不同的任务

### 什么时候继续老会话

- 还在同一个话题上
- 需要引用之前的对话内容
- 任务还没完成

### 如何判断会话状态

界面上可以看到：
- **活跃**：当前正在运行
- **空闲**：上一轮已完成，等待你的下一条消息
- **等待审批**：智能体在等你确认

## 常见误区

- **会话 = 线程？** 密切相关但不完全相同。会话记录交互历史，线程维护执行状态。
- **一轮 = 一次问答？** 不一定。一次用户轮次可能包含多轮模型轮次（工具调用等）。
- **Token 统计 = 账单？** 不是。Wunder 的 Token 统计是上下文占用，不是供应商的计费账单。
