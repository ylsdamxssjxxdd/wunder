# 流式执行

## 原理

Wunder 的对话不是"黑盒处理"——你发送消息后只能等待最终结果。整个过程是流式的，每一步变化都会实时推送给你。

### 流式不是只有文字

很多人以为流式就是"文字一点一点出来"。实际上，Wunder 的流式包含：

| 类型 | 说明 | 例子 |
|------|------|------|
| **过程事件** | 发生了什么 | 智能体正在思考、正在调用工具 |
| **状态事件** | 当前在什么阶段 | 运行中、等待审批、已完成 |
| **终态事件** | 是否已经结束 | 完成、失败、取消、拒绝 |
| **恢复机制** | 断线后能补发 | 重连后自动补发遗漏的事件 |

### WebSocket vs SSE

Wunder 使用两种传输方式：

| 方式 | 特点 | 用途 |
|------|------|------|
| **WebSocket** | 双向通信，可控制 | 主要方式，适合会话控制 |
| **SSE** | 单向流，浏览器友好 | 兜底方案，兼容性好 |

聊天场景优先 WebSocket，通用执行和兼容场景可用 SSE。

## 作用

### 过程可见

你不用等到最后才知道发生了什么：
- 看到"正在思考" → 知道智能体在处理
- 看到"正在读取文件" → 知道它用了什么资料
- 看到"工具执行完成" → 知道它完成了这一步

### 控制及时

不是发出去就失去控制：
- 可以实时取消执行
- 需要审批时立即收到请求
- 状态变化实时反映

### 断线可恢复

网络波动不会丢失进度：
- 自动重连
- 重连后补发遗漏的事件
- 状态自动同步

## 实践

### 状态判断

如何判断一轮是否结束？看以下事件：

| 状态 | 看什么 |
|------|--------|
| 一轮是否结束 | `turn_terminal` 事件 |
| 线程当前状态 | `thread_status` 事件 |
| 审批是否完成 | `approval_resolved` 事件 |

不要只看"好像不动了"来判断结束——要看明确的状态事件。

### 网络断开

遇到网络断开时：
1. 界面会显示"连接断开"
2. 系统自动重连（指数退避）
3. 重连成功后自动恢复
4. 如果一直失败，提示手动刷新

### 事件解读

常见事件含义：

| 事件 | 含义 |
|------|------|
| `queued` | 请求已进入队列 |
| `progress` | 执行进度更新 |
| `tool_call` | 智能体调用了工具 |
| `tool_result` | 工具返回了结果 |
| `approval_request` | 需要你审批 |
| `turn_terminal` | 当前轮次结束 |

## 常见误区

- **流式输出 = 实时性？** 不完全。流式输出只是实时性的表现形式，状态同步和恢复机制同样重要。
- **final = 完成？** 不一定。要结合终态事件判断，包括失败、取消等。
- **不动了 = 结束了？** 不一定。可能是等待审批或网络问题，要看状态指示。
