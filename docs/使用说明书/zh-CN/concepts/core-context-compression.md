# 上下文压缩

长会话不是例外，而是 Wunder 在真实运行里必然遇到的常态。上下文压缩不是锦上添花，而是保证线程能继续工作的关键机制。

![上下文压缩图：完整上下文经过多级压缩后重新进入窗口，并保留观测证据](/docs/assets/manual/core-context-compression.svg)

## 先给结论

- 压缩不是“删一点历史”，而是系统主动重组上下文。
- 只要系统支持长会话，压缩就不能是临时补丁，必须是正式核心。
- 压缩的质量不仅影响成本，更直接影响线程还能不能继续保持语义连续。

## 为什么它必须单列为核心

如果没有上下文压缩，长会话最终只能走三条坏路：

- 直接超限失败。
- 粗暴截断，导致线程失忆。
- 人工控制长度，把系统复杂度转嫁给用户和接入方。

Wunder 把压缩列为核心，等于承认“长会话治理本身就是系统职责”。

## 这个核心真正保护什么

- 保护线程连续性：不让会话因为上下文过重直接断掉。
- 保护有效事实：保留当前继续执行真正需要的信息。
- 保护可复盘性：让压缩前后差异能够被检查，而不是变成黑盒。

## 关键约束

| 约束 | 作用 |
|------|------|
| 压缩不能伪造事实 | 只能重组已有内容，不能补写不存在的事实 |
| 压缩必须可追溯 | 要能回答删了什么、保留了什么、损失了什么 |
| 压缩必须可观测 | 需要留下事件、层级和结果，而不是静默发生 |
| 压缩不能破坏线程冻结约束 | 压缩历史不等于重写线程底座 |

## 设计重点

### 重点一：压缩不是截断

截断只是在内容太多时硬砍；压缩则是在做信息重排，目标是让线程仍然能工作。

### 重点二：压缩不是越早越好，也不是越晚越好

压太早，会丢掉本来还需要的细节；压太晚，又会让线程直接撞到窗口边界。真正的设计重点是预算治理和渐进层级。

### 重点三：工具 observation 是压缩压力的高发区

长日志、大网页、批量搜索、复合工具结果，都会让上下文迅速膨胀。所以压缩和工具设计必须联动考虑。

## 最容易搞错的点

- 压缩不等于截断。截断只是丢内容，压缩要求保留结构化有效信息。
- 压缩不是记忆系统的替代品，长期资料应走专门的记忆和知识链路。
- 只看“有没有超限”不够，还要看压缩是否改变了任务可执行性。

## 和其他核心的边界

- 和 [记忆](/docs/zh-CN/concepts/core-memory/) 的区别：记忆负责长期资料入口，压缩负责当前线程历史治理。
- 和 [稳定性](/docs/zh-CN/concepts/core-stability/) 的区别：稳定性关心“超限后怎么兜”，压缩关心“如何主动避免撞线”。
- 和 [可观测性](/docs/zh-CN/concepts/core-observability/) 的区别：可观测性记录压缩发生了什么，压缩决定到底怎么压。
