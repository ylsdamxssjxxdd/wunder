# 工具

Wunder 的重点不是“让模型回答得更像人”，而是“让模型能稳定调到正确能力并消费返回结果”。

![工具分层图：工具规格、执行层、入模结果、展示投影四层关系](/docs/assets/manual/core-tools.svg)

## 先给结论

- 工具不是附加能力列表，而是系统给模型开放能力的统一接口。
- 一个工具如果只对人类好读、但对模型难调用，它在 Wunder 里就是坏工具。
- 工具设计必须同时考虑调用正确率、上下文成本、权限治理和前端工作流展示。

## 为什么它必须单列为核心

如果没有“工具”这个核心，Wunder 会出现两种退化：

- 能力散落在提示词、隐式逻辑和临时接口里，模型不知道该怎么稳定调用。
- 前端只能看到“模型说自己做了什么”，却看不到真正执行了什么。

Wunder 把能力统一成工具，本质是在统一三件事：模型心智、执行协议、展示协议。

## 这个核心真正保护什么

- 保护模型调用质量：参数和描述必须让模型少猜。
- 保护执行治理：预算、权限、审批、失败反馈必须进入统一链路。
- 保护前端可见性：工具工作流需要稳定地被展示、追踪和回放。

## 关键约束

| 约束 | 作用 |
|------|------|
| 描述必须清晰 | 降低模型误用和幻觉式调用 |
| 参数必须结构化 | 让模型在调用时更像“填表”而不是“自由发挥” |
| 返回必须精简明确 | 防止 observation 噪声反过来污染上下文 |
| 事实与展示要分离 | UI 卡片是投影，不是工具真实执行结果 |

## 设计重点

### 重点一：Wunder 不是只让模型“会说”，而是让模型“会做”

文件、命令、浏览器、桌面、用户世界、渠道、记忆管理，本质上都是执行能力，不该各走各的协议。

### 重点二：工具结果要先服务调度，再服务展示

一个返回几万字原始日志的工具，对用户看似信息丰富，但对模型通常是灾难。真正好的工具返回会优先交付：

- 当前最关键的事实
- 继续下一步所需的信号
- 失败时可行动的提示

### 重点三：统一工具体系比“工具越多越好”更重要

内置工具、MCP、Skills、知识库、用户工具来源不同，但最终都要进入统一调度视角。否则权限、审计、检索和工作流展示都会散掉。

## 最容易搞错的点

- Skills 不是和工具平行的另一套世界，它们本质上是固化流程能力。
- 工具越多不一定越好，描述模糊和边界重叠会直接拉低模型调用质量。
- “能返回很多内容”不是优点，能被模型低成本消费才是。

## 和其他核心的边界

- 和 [智能体循环](/docs/zh-CN/concepts/core-agent-loop/) 的区别：智能体循环定义主链路，工具定义主链路里可被调用的能力。
- 和 [上下文压缩](/docs/zh-CN/concepts/core-context-compression/) 的区别：压缩解决“结果太大怎么办”，工具解决“能力怎么暴露给模型”。
- 和 [可观测性](/docs/zh-CN/concepts/core-observability/) 的区别：可观测性负责复盘工具轨迹，工具负责产生轨迹。
