# 工具

## 原理

在 Wunder 里，智能体能做的每一件事都被统一抽象为"工具"。不管是读写文件、搜索网页、执行命令还是管理记忆，对智能体来说都是"调用一个工具"。

这种设计的好处是：
- 智能体不需要学习各种不同的调用方式
- 所有能力有一致的权限和审计控制
- 你可以统一管理智能体能用哪些工具

## 工具从哪里来

Wunder 的工具有多种来源，但最终都进入统一调度：

| 来源 | 说明 | 例子 |
|------|------|------|
| **内置工具** | 系统自带的基础能力 | 文件读写、命令执行、网页抓取 |
| **MCP 工具** | 通过 MCP 协议接入的外部能力 | 数据库操作、第三方 API |
| **技能** | 封装了完整流程的能力包 | 数据分析方法、报告生成流程 |
| **知识库** | 提供可检索的参考资料 | 制度文档、技术资料 |
| **用户工具** | 用户自己扩展的能力 | 自定义脚本、个性化工具 |

## 作用

### 对你来说

- **智能体能真正干活**：不只是聊天，而是能操作文件、运行代码、搜索信息
- **能力可控**：管理员可以控制哪些工具可用，敏感工具有审批机制
- **过程可见**：你可以在界面上看到智能体调用了什么工具、得到了什么结果

### 工具设计的原则

好的工具设计有三个关键：

**1. 描述清晰**

工具的说明必须让智能体准确理解"这个工具是干什么的、什么时候该用"。描述模糊的工具会导致智能体误用。

**2. 参数结构化**

工具参数像表格一样有明确的字段和类型，智能体只需"填表"而不是"自由发挥"。这大大降低了调用错误。

**3. 返回精简**

工具返回的结果必须精简明确。动辄几万字的返回不仅浪费上下文，还会干扰智能体的判断。好的工具返回是"当前最关键的事实 + 下一步需要的信号"。

## 实践

### 工具审批

当智能体想要执行敏感操作（如修改重要文件、执行命令）时，系统可能会暂停并要求你确认：

- 你会看到操作说明
- 选择"批准"继续或"拒绝"阻止
- 审批结果会被记录

管理员可以调整审批的严格程度。

### 工具与上下文

每个工具调用的结果都会消耗对话的上下文空间。如果一个工具返回了大量内容，系统会自动压缩以保持对话能继续。这就是为什么工具返回要精简——不然上下文会被撑爆。

### 工具可见性

不是所有工具都始终可见。哪些工具可用取决于：
- 系统配置（管理员启用了哪些）
- 运行形态（桌面应用可能有更多本地工具）
- 智能体设置（给智能体配置了哪些工具）
- 安全限制（某些路径下的操作可能被禁止）

## 技能和工具的关系

技能不是另一种工具，而是"流程化的经验包"。一个技能通常包含：
- 操作说明文档
- 执行脚本
- 示例和资源

技能让智能体在面对某类任务时有一套成熟的方法论可以参考，而不是每次都从零开始。

## 常见误区

- **工具越多越好？** 不是。工具太多且描述不清会反而让智能体更容易选错工具。
- **返回内容越多越好？** 不是。能被智能体高效利用的精简结果才是好结果。
- **技能只是说明文档？** 不是。在 Wunder 里，技能是智能体可以主动读取和使用的运行资源。
