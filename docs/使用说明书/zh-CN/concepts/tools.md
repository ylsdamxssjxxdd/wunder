# 工具体系

## 原理

Wunder 的智能体不只会"说话"，还会"做事"。做事的能力全部来自工具。工具有多种来源，但最终统一调度。

### 五种工具来源

```
内置工具 ← 系统自带，最稳定
    +
MCP 工具 ← 外部服务接入
    +
技能（Skills） ← 流程化经验包
    +
知识库 ← 可检索的参考资料
    +
用户工具 ← 个人扩展能力
    ↓
统一调度层（智能体统一使用）
```

### 内置工具

系统自带的基础能力，覆盖最常见的场景：

| 类别 | 工具举例 | 用途 |
|------|----------|------|
| 文件操作 | 读取、写入、搜索、列表 | 处理工作区文件 |
| 命令执行 | 执行命令 | 运行脚本和代码 |
| 网络操作 | 网页抓取 | 获取网页内容 |
| 桌面操作 | 截图、控制 | 桌面端控制 |
| 会话管理 | 线程控制、记忆管理 | 管理对话状态 |
| 协作 | 蜂群调度、子智能体 | 多智能体协作 |

### MCP 工具

MCP（Model Context Protocol）是接入外部能力的标准协议。通过 MCP 可以接入：
- 数据库操作工具
- 第三方 API 服务
- 专业领域的工具

### 技能（Skills）

技能是"流程化的经验包"，封装了某类任务的完整方法：
- 操作说明文档
- 执行步骤和约束
- 示例文件和资源

技能让智能体面对特定任务时有成熟方法论可以参考。

## 作用

### 为什么必须统一

如果不同来源的工具各自为政：
- 智能体不知道该用哪个工具
- 界面无法统一展示
- 权限和审计被打散
- 排障找不到问题在哪一层

Wunder 的做法：**来源不同，调度统一**。所有工具在同一层面被管理、被调用、被审计。

### 工具描述的重要性

好的工具描述让智能体准确调用。差的描述导致误用和幻觉。工具描述需要：
- 准确说明工具用途
- 明确适用场景
- 说清参数含义
- 标注限制条件

## 实践

### 查看可用工具

在界面左栏"工具"页面可以看到：
- 当前可用的工具列表
- 每个工具的功能说明
- 工具的来源（内置/MCP/技能等）

### 工具权限

不是所有工具对所有用户都可用：
- 管理员控制哪些工具被启用
- 敏感工具有审批机制
- 不同运行形态可能有不同工具集

### 工具审批

当智能体调用敏感工具时：
1. 系统暂停执行
2. 弹出审批请求，说明要做什么
3. 你批准或拒绝
4. 根据你的决定继续或调整

审批模式可以调整严格程度（由管理员配置）。

### 工具结果

工具返回的结果会被智能体用于下一步判断。如果结果太大，系统会自动精简以保持对话能继续。

## 常见误区

- **工具越多越好？** 不是。工具太多且描述不清反而让智能体更容易选错。
- **技能只是文档？** 不是。技能是智能体可以主动读取的运行资源。
- **所有工具始终可用？** 不是。工具可见性受配置、运行形态和安全策略影响。
