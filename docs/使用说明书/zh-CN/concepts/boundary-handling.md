# 边界处理

wunder 的设计目标之一是**在各种边界和异常情况下保持系统稳定运行**，而不是简单崩溃。

这一页详细说明系统在各类边界场景下的处理策略。

---

## 1. 上下文超限处理

### 1.1 什么是上下文超限

当会话历史 + 当前请求 + 工具结果等内容的总 token 数超过模型的上下文窗口限制时，就会触发上下文超限。

### 1.2 处理策略

wunder 采用**渐进式上下文压缩**策略，而不是简单截断：

```
完整上下文 → 压缩策略1 → 压缩策略2 → ... → 适配模型窗口
```

### 1.3 压缩层级

| 压缩层级 | 策略 | 损失程度 | 适用场景 |
|---------|------|---------|---------|
| **Level 0** | 保留完整上下文 | 无 | 上下文充足时 |
| **Level 1** | 移除工具中间结果摘要 | 低 | 工具结果可以重新获取 |
| **Level 2** | 摘要化历史消息 | 中 | 保留语义，损失细节 |
| **Level 3** | 移除早期轮次 | 高 | 仅保留最近 N 轮 |
| **Level 4** | 仅保留 system prompt + 当前请求 | 极高 | 紧急降级 |

### 1.4 关键设计约束

- **线程冻结原则**：即使上下文压缩，system prompt 也保持不变（首次确定后冻结）
- **记忆隔离**：长期记忆只在线程初始化时注入一次，不参与动态压缩
- **可观测性**：每次压缩都会记录 `context_compaction` 事件，包含压缩策略和损失评估

## 2. 网络中断处理

### 2.1 WebSocket 连接中断

#### 客户端侧行为

| 阶段 | 行为 |
|------|------|
| **连接断开** | 立即显示"连接断开"提示，启动自动重连 |
| **重连中** | 显示重连进度，指数退避（1s, 2s, 4s, 8s, 16s, 最大 32s） |
| **重连成功** | 发送 `resume` 请求，获取错过的事件 |
| **重连失败** | 超过最大重试次数后，提示手动刷新 |

#### 服务端侧行为

```
连接断开 → 保持会话状态 300s → 超时后清理内存状态（但保留数据库记录）
              ↓
         缓存未推送的事件
              ↓
         客户端重连时补发
```

### 2.2 SSE 连接中断

SSE 作为 WebSocket 的兜底方案，处理逻辑类似：

- 自动重连（浏览器原生支持）
- 通过 `Last-Event-ID` 恢复事件流
- 服务端保留最近 1000 条事件用于恢复

### 2.3 模型 API 调用失败

| 失败类型 | 重试策略 | 降级方案 |
|---------|---------|---------|
| **网络超时** | 重试 3 次，指数退避 | 提示用户稍后重试 |
| **速率限制** | 等待 `retry-after` 头，最长等待 60s | 进入队列排队 |
| **认证失败** | 立即失败，不重试 | 提示管理员检查配置 |
| **模型错误** | 重试 1 次（如果是临时错误） | 切换到备用模型（如果配置） |
| **上下文超限** | 不重试，触发压缩逻辑 | 自动压缩后重试 |

## 3. 工具执行失败处理

### 3.1 工具失败分类

| 失败类别 | 说明 | 可重试 |
|---------|------|--------|
| **参数错误** | 工具参数验证失败 | ❌ 需要模型修正参数 |
| **权限拒绝** | 工具不在白名单或审批被拒 | ❌ 需要用户授权 |
| **临时错误** | 文件锁、网络波动等 | ✅ 自动重试 |
| **系统错误** | 工具执行引擎异常 | ✅ 有限重试 |

### 3.2 工具结果防爆机制

为了防止工具输出过大导致上下文超限，wunder 实现了**三层防护**：

```
工具执行 → Layer 1: 执行层预算裁剪 → Layer 2: MCP 二次压缩 → Layer 3: 入模前最终检查
```

#### Layer 1: 执行层预算

```rust
// 执行命令工具的预算参数
budget: {
  time_budget_ms: 60000,      // 最大执行时间
  output_budget_bytes: 4194304, // 最大输出字节（4MB）
  max_commands: 200            // 最大命令数
}
```

#### Layer 2: 输出裁剪策略

| 输出大小 | 处理方式 |
|---------|---------|
| ≤ 预算 | 完整返回 |
| 预算 < 大小 ≤ 2×预算 | 保留头部 50% + 尾部 50% |
| > 2×预算 | 保留头部 25% + 尾部 25% + 省略中间 |

#### Layer 3: 信号反馈

裁剪后的结果会附带元数据：

```json
{
  "output": "...裁剪后的内容...",
  "output_meta": {
    "truncated": true,
    "original_size_bytes": 8388608,
    "returned_size_bytes": 4194304,
    "continuation_required": true
  }
}
```

模型可以根据这些信号决定是否：
- 缩小查询范围
- 分页获取结果
- 接受当前信息继续

### 3.3 原子写入策略

涉及文件写入的工具（`写入文件`、`应用补丁`）采用**临时文件 + 原子替换**策略：

```
1. 写入临时文件：file.tmp
2. fsync 确保数据落盘
3. 原子重命名：file.tmp → file
4. 如果失败，删除临时文件，保留原文件
```

这确保了：
- ✅ 不会出现半写入的文件
- ✅ 并发写入不会互相破坏
- ✅ 失败时可以回滚

## 4. 会话与线程异常处理

### 4.1 会话状态机

wunder 的会话有明确的状态转换：

```
idle → running → waiting_approval → waiting_user_input → idle
  ↓        ↓
error  cancelled
```

### 4.2 异常状态恢复

| 异常状态 | 触发条件 | 恢复策略 |
|---------|---------|---------|
| **`system_error`** | 未捕获的系统异常 | 记录错误日志，会话重置为 idle |
| **`cancelled`** | 用户主动取消 | 立即停止执行，清理资源 |
| **`timeout`** | 执行超过配置时间 | 停止执行，返回部分结果（如果有） |
| **`approval_pending`** | 等待用户审批 | 暂停执行，等待用户决定 |

### 4.3 线程状态同步

通过 `thread_status` 事件实时同步线程状态：

```json
{
  "event": "thread_status",
  "status": "running", // running, waiting_approval, waiting_user_input, idle, system_error
  "session_id": "xxx",
  "timestamp": 1234567890
}
```

## 5. 并发与排队处理

### 5.1 主线程排队

每个 `user_id + agent_id` 只有一条主线程，忙时新请求进入队列：

```
新请求 → 主线程空闲？→ 是 → 立即执行
              ↓ 否
         进入 agent_tasks 队列
              ↓
         主线程空闲后依次执行
```

### 5.2 队列配置

| 配置项 | 说明 | 默认值 |
|-------|------|--------|
| `max_queue_size` | 最大队列长度 | 100 |
| `queue_timeout_s` | 排队超时时间 | 3600 |
| `max_concurrent_sessions` | 最大并发会话数 | 10 |

### 5.3 优先级策略

队列中的任务按以下优先级排序：
1. 管理员请求（最高）
2. 带 `priority: high` 的请求
3. 普通请求
4. 带 `priority: low` 的请求

## 6. 数据一致性保证

### 6.1 数据库事务

所有涉及多个表的操作都使用数据库事务：

```rust
// 伪代码示例
begin_transaction();
try {
  update_chat_session();
  insert_message();
  update_agent_thread();
  commit();
} catch {
  rollback();
  throw;
}
```

### 6.2 幂等性设计

| 接口 | 幂等键 | 说明 |
|------|--------|------|
| `/wunder/chat/messages` | `client_msg_id` | 重复消息不会重复处理 |
| 渠道 Webhook | `message_id` | 重复推送不会重复处理 |
| 工具调用 | `tool_call_id` | 重复调用不会重复执行 |

### 6.3 最终一致性

对于异步操作（如渠道消息投递），采用**outbox 模式**保证最终一致性：

```
1. 业务操作 + outbox 记录写入（同一事务）
2. 后台 worker 从 outbox 读取
3. 执行外部操作
4. 成功后标记 outbox 为完成
5. 失败则重试（带指数退避）
```

## 7. 资源泄漏防护

### 7.1 超时机制

| 资源类型 | 超时配置 | 清理行为 |
|---------|---------|---------|
| 工具执行 | `tool_timeout_s` | 终止进程，释放资源 |
| 模型调用 | `llm_timeout_s` | 取消请求，释放连接 |
| WebSocket 连接 | `ws_idle_timeout_s` | 关闭空闲连接 |
| 临时文件 | `temp_dir_ttl_h` | 定期清理过期文件 |

### 7.2 连接池管理

- 数据库连接：使用连接池，自动回收空闲连接
- HTTP 客户端：连接复用，限制最大并发数
- WebSocket：限制单用户最大连接数

### 7.3 内存保护

- 工具输出限制：防止单个工具返回过大结果
- 上下文压缩：自动控制内存占用
- 会话历史裁剪：数据库层面限制历史消息数

## 8. 可观测性与调试

### 8.1 错误分类

所有错误都携带结构化的 `error_meta`：

```json
{
  "error": "操作失败",
  "error_meta": {
    "category": "tool_execution",
    "severity": "error",
    "retryable": true,
    "retry_after_ms": 5000,
    "source_stage": "tool_call",
    "recovery_action": "retry_tool"
  }
}
```

### 8.2 关键事件记录

| 事件 | 记录内容 | 用途 |
|------|---------|------|
| `context_compaction` | 压缩策略、压缩比 | 优化上下文管理 |
| `tool_failed` | 工具名、错误类型、重试建议 | 工具调试 |
| `queue_position` | 当前位置、预计等待时间 | 用户体验 |
| `recovery_attempt` | 恢复策略、结果 | 稳定性分析 |

### 8.3 日志分级

| 级别 | 使用场景 |
|------|---------|
| `ERROR` | 系统异常、需要人工干预 |
| `WARN` | 边界情况、自动恢复但需关注 |
| `INFO` | 正常流程、关键节点 |
| `DEBUG` | 详细调试信息 |

## 9. 常见边界场景速查

| 场景 | 预期行为 | 参考章节 |
|------|---------|---------|
| 会话历史太长 | 自动压缩上下文 | 1. 上下文超限处理 |
| WebSocket 断开 | 自动重连 + 事件补发 | 2. 网络中断处理 |
| 工具输出太大 | 裁剪 + 信号反馈 | 3.2 工具结果防爆机制 |
| 写入文件时崩溃 | 原子写入，不破坏原文件 | 3.3 原子写入策略 |
| 主线程忙 | 新请求排队 | 5.1 主线程排队 |
| 网络波动 | 指数退避重试 | 2.3 模型 API 调用失败 |
| 数据库操作失败 | 事务回滚 | 6.1 数据库事务 |
