# 边界处理

## 原理

系统不可能永远在"理想状态"运行。真实环境中会遇到各种边界情况：

- 对话太长超出上下文限制
- 网络波动甚至断开
- 工具执行失败
- 系统资源紧张

Wunder 的设计目标：**在边界情况下保持稳定，而不是崩溃**。

### 处理原则

**控扩散**：一个环节出错，不能把其他环节带崩

**能恢复**：失败不是终点，系统有恢复链路

**可复盘**：同类问题能归类、复现、验证

## 主要边界情况

### 上下文超限

对话越来越长，工具结果越来越多，总 Token 可能超出模型限制。

**处理策略**：渐进式压缩

| 级别 | 策略 | 损失程度 |
|------|------|----------|
| 1级 | 移除工具中间结果 | 很低 |
| 2级 | 摘要历史对话 | 中等 |
| 3级 | 移除较早的对话 | 较高 |
| 4级 | 紧急降级 | 很高 |

**关键**：系统提示词不受压缩影响，保持冻结。

### 网络中断

WebSocket 或 SSE 连接可能因网络波动断开。

**处理策略**：自动重连 + 事件补发

```
连接断开 → 显示提示 → 自动重连（指数退避）
                ↓
           缓存未推送事件
                ↓
           重连成功 → 补发事件
                ↓
           重连失败 → 提示手动刷新
```

**重连策略**：1s → 2s → 4s → 8s → 16s → 最大32s

### 工具失败

工具调用可能因各种原因失败。

**失败分类**：

| 类型 | 处理 |
|------|------|
| 参数错误 | 让智能体修正参数 |
| 权限拒绝 | 等待用户审批 |
| 临时错误（文件锁、网络波动） | 自动重试 |
| 系统错误 | 有限重试 + 降级 |

### 工具输出过大

某些工具可能返回大量内容，撑爆上下文。

**处理策略**：三层防护

| 层级 | 作用 |
|------|------|
| 执行层预算 | 限制单次输出大小 |
| 结果裁剪 | 超出时保留头部和尾部 |
| 入模检查 | 最终检查并精简 |

裁剪后的结果会告知智能体"已裁剪"，让它可以调整策略。

### 并发排队

同一用户可能同时发起多个请求，主线程忙时新请求排队。

**处理策略**：队列 + 优先级

```
新请求 → 主线程空闲？
           ├── 是 → 立即执行
           └── 否 → 进入队列 → 主线程空闲后依次执行
```

**优先级**：管理员请求 > 高优先级请求 > 普通请求 > 低优先级请求

## 实践

### 长对话管理

- 注意上下文占用指标
- 适时使用 `/compact` 主动压缩
- 完成一个话题后开新会话

### 网络波动应对

- 界面显示"连接断开"时等待自动重连
- 不要频繁刷新页面
- 如果重连失败，按提示手动刷新

### 工具失败处理

- 查看失败提示了解原因
- 根据提示调整（如提供正确的文件路径）
- 如果反复失败，联系管理员查看系统日志

### 系统异常处理

- 长时间无响应：尝试取消或重置
- 状态明显异常：使用"重置工作状态"
- 反复出现问题：联系管理员排障

## 常见误区

- **只要智能体回答了就是正常？** 不一定。要看长期稳定性和异常处理能力。
- **重试能解决一切？** 不是。不同类型的错误有不同处理方式。
- **边界情况很少发生？** 不一定。长对话、复杂任务中边界情况很常见。
