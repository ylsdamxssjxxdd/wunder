# 蜂群协作

## 原理

蜂群是 Wunder 的多智能体协作机制。当任务复杂到需要多个智能体协同完成时，蜂群把协作变成系统内可追踪、可重复、可恢复的正式能力。

### 核心概念

| 角色 | 说明 | 比喻 |
|------|------|------|
| **母蜂** | 当前主持协作的主智能体 | 项目经理 |
| **工蜂** | 已存在的智能体，被调度参与协作 | 团队成员 |
| **Mission** | 一次具体的协作任务 | 工作任务 |
| **任务** | 分配给工蜂的具体子任务 | 待办事项 |

### 工作流程

```
你的需求：复杂的、需要分工的任务
         ↓
    母蜂接收并分析
         ↓
    拆解为多个子任务
         ↓
    ┌────┼────┐
    ↓    ↓    ↓
  工蜂A 工蜂B 工蜂C
  搜集  分析  撰写
    ↓    ↓    ↓
    └────┼────┘
         ↓
    母蜂汇总结果
         ↓
    返回最终成果
```

## 作用

### 协作优势

- **并行执行**：多个智能体同时工作，效率更高
- **专业分工**：不同智能体有不同专长，各司其职
- **过程可见**：你能看到每个工蜂的进度和轨迹
- **结果可靠**：工蜂结果归并到母蜂，不会散落

### 蜂群 vs 子智能体

| | 蜂群 | 子智能体 |
|--|------|----------|
| **调度对象** | 已存在的智能体 | 临时派生的子会话 |
| **线程** | 工蜂独立线程 | 当前会话内派生 |
| **适合** | 复杂的分工协作 | 轻量的临时任务 |
| **资源** | 需要预先创建好智能体 | 不需要额外准备 |

## 工蜂的线程约定

工蜂被派发任务时，默认创建新线程作为主线程。为什么？

- **上下文干净**：不受旧对话干扰
- **专注当前任务**：只看母蜂派发的内容
- **避免污染**：不同任务之间不互相影响

你也可以指定复用工蜂的现有线程，或强制使用全新线程。

## 实践

### 在界面上查看蜂群

**中栏列表**：
- 显示所有蜂群任务
- 有运行中任务的蜂群会有呼吸高亮
- 可以快速了解哪些在运行

**画布视图**：
- 可视化展示协作关系
- 母蜂和工蜂之间的连线表示派发关系
- 每个节点显示当前状态

**工具轨迹**：
- 每个节点显示调用了什么工具
- 显示当前走到了哪一步
- 完成后仍保留轨迹记录

### 状态说明

| 状态 | 含义 |
|------|------|
| 运行中 | 工蜂正在执行任务 |
| 已完成 | 工蜂已完成任务 |
| 失败 | 工蜂执行出错 |
| 等待中 | 等待被调度 |

### 什么时候用蜂群

**适合**：
- 任务可以明确拆分为多个子任务
- 不同子任务需要不同专长
- 需要并行处理提高效率

**不适合**：
- 简单任务，一个智能体就能完成
- 只是需要临时分叉一个小任务

### 主线程保护

界面会阻止在智能体运行中切换线程。这是保护机制，避免：
- 运行中的主线程还没结束就切走
- 会话状态和工作流错位

如果要开新线程，先等当前运行完成或先停止。

## 常见误区

- **蜂群 = 并行请求？** 不是。重点是协作关系和结果归并。
- **工蜂越多越好？** 不是。粒度过粗或过细都降低效率。
- **工蜂能访问母蜂的上下文？** 不能。工蜂只看到被派发的任务内容。
