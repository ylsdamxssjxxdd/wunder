# 实时性

在 Wunder 里，实时性不是“文字一点点出来”这么简单，而是让前端和外部系统持续获得可信、可恢复、可收敛的运行态变化。

![实时性图：线程事实通过事件流、恢复机制投影为客户端状态](/docs/assets/manual/core-realtime.svg)

## 先给结论

- 实时性的重点不是字符流，而是线程状态流。
- 事件流必须能表达过程、终态和恢复，而不是只会一直吐增量文本。
- 客户端看到的是投影，投影必须贴近真相，但不能替代真相。

## 为什么它必须单列为核心

如果没有实时性这个核心，系统会出现三个明显问题：

- 用户只能在很晚的时候拿到最终结果，过程不可见。
- 断线后不知道是否继续执行过，也不知道漏了什么。
- 前端只能自己猜“还在跑还是已经结束”，状态机会非常脆弱。

## 这个核心真正保护什么

- 保护用户感知：让用户知道系统正在做什么，而不是只等最后一句话。
- 保护客户端状态机：让列表、详情、工作流和审批区都能基于稳定事件工作。
- 保护恢复能力：断线、重连、补发、resume 都有统一依据。

## 关键约束

| 约束 | 作用 |
|------|------|
| 前端消费事件流，但不定义真相 | 防止 UI 状态反向污染后端语义 |
| 终态必须有明确事件 | 不能让客户端靠超时猜结束 |
| 恢复机制必须正式存在 | 断线后要能 resume、补发和重建快照 |
| 事件口径要稳定 | 不能为了某个页面临时发明一套状态名 |

## 设计重点

### 重点一：流式只是实时性的一部分

增量文本、工具输出、审批等待、线程状态、终态事件、恢复补偿，这些合起来才叫实时性。

### 重点二：实时系统要能“接着看”，不是只能“从头看”

只支持长连接推送但不支持恢复的系统，本质上还是脆弱的。

### 重点三：事件语义要为多端复用设计

同一条事件既要被聊天面板消费，也要被管理员页、桌面壳和外部系统消费，所以它不能只服务一个页面。

## 最容易搞错的点

- `final` 或最终回复不是唯一真相，终态判断要结合完整事件语义。
- “能流式吐字”不等于“具备实时性”，状态同步和恢复同样关键。
- 前端上的忙闲状态是投影，不是后端状态机的替代物。

## 和其他核心的边界

- 和 [智能体循环](/docs/zh-CN/concepts/core-agent-loop/) 的区别：循环定义线程内部如何推进，实时性定义这些推进如何对外可见。
- 和 [可观测性](/docs/zh-CN/concepts/core-observability/) 的区别：实时性偏在线消费，可观测性偏事后解释与复盘。
- 和 [稳定性](/docs/zh-CN/concepts/core-stability/) 的区别：实时性关心状态能否及时送达，稳定性关心异常时是否还能继续工作。
