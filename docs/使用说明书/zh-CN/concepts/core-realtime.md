# 实时性

## 原理

你和智能体对话时，不只是"发送请求→等待→收到结果"这么简单。智能体的思考过程、工具调用、状态变化都是**持续推送**给你的。

这就是实时性：让前端和外部系统持续获得线程的变化状态。

### 不是简单的字符流

实时性不只是"文字一点点出来"，而是包含：
- **过程事件**：智能体在做什么
- **状态事件**：当前运行到什么阶段
- **终态事件**：一轮是否已经结束
- **恢复机制**：断线后能补发遗漏的事件

### 技术实现

Wunder 使用 **WebSocket** 作为主要传输方式，SSE 作为兜底。WebSocket 能双向通信，适合：
- 持续接收状态更新
- 发送控制命令（如取消）
- 恢复连接后补发事件

## 作用

### 对你

**过程可见**：
- 看到智能体"正在思考"
- 看到智能体"正在读取文件"
- 知道它卡在哪里、还需要多久

**控制及时**：
- 可以实时取消执行
- 可以实时审批或拒绝
- 不会"发出去就失去控制"

**断线可恢复**：
- 网络断了重连后，自动补发遗漏的内容
- 不会错过重要状态变化

### 对系统

**状态同步**：
- 多端状态一致（桌面、网页、飞书看到同样的状态）
- 不会一个端显示"运行中"，另一个显示"已完成"

**事件驱动**：
- 界面更新基于真实事件，不是猜测
- 审计和追踪有完整的事件流

## 关键规则

### 前端是投影，后端是真相

你在界面上看到的状态是后端线程状态的"投影"。投影要贴近真相，但不能替代真相。如果前端显示异常，以实际执行状态为准。

### 终态必须有明确事件

不能靠"好像没动静了"来判断一轮是否结束。系统会推送明确的终态事件（完成、失败、取消、拒绝），客户端据此判断。

### 事件语义稳定

状态事件的名称和含义保持一致，不因页面或渠道而异。这保证了多端状态能对齐。

## 实践

### 连接断开了怎么办

1. 系统会显示"连接断开"
2. 自动开始重连（指数退避）
3. 重连成功后：
   - 自动恢复会话
   - 补发遗漏的事件
4. 如果重连失败，会提示手动刷新

### 如何看懂状态提示

界面上的状态提示通常表示：

| 状态 | 含义 |
|------|------|
| 等待中 | 请求已提交，在排队 |
| 正在思考 | 智能体在处理 |
| 执行工具中 | 智能体调用了工具 |
| 需要审批 | 智能体等待你确认 |
| 已完成 | 任务顺利结束 |
| 已失败 | 执行过程出错 |

### 实时更新 vs 刷新

- **实时更新**：通过 WebSocket 自动推送
- **手动刷新**：断网重连或页面刷新后的恢复

两者结合保证你始终能看到最新状态。

## 常见误区

- **实时性 = 速度快？** 不是。实时性是指状态可见，不是指执行速度快。复杂任务本身需要时间。
- **前端状态 = 后端真相？** 不是。前端是投影，后端是真相。正常情况下一致，但可能因网络延迟有短暂差异。
- **final 事件就是完成？** 不一定。还要看完整的终态判断，包括失败、取消等。
