# 智能体循环

智能体循环是 Wunder 的主链路。它真正解决的不是“模型能不能回答”，而是“线程能不能把一次任务稳定推进到终态”。

![智能体循环图：输入进入线程、模型判断、工具动作、状态收敛、事件投影构成系统主链路](/docs/assets/manual/core-agent-loop.svg)

## 先给结论

- Wunder 的基本执行单位不是单次模型调用，而是一条持续运行的线程。
- 一轮执行是否健康，关键看它能否在线程里完成“判断、行动、观察、继续、收敛”。
- 前端看到的是执行投影，真正的执行真相始终在线程主链路里。

## 为什么它必须单列为核心

如果没有智能体循环这个核心，Wunder 会直接退化成一个“每次请求临时拼上下文、临时调模型、临时给结果”的系统。那会带来三个后果：

- 无法稳定表达一轮执行走到哪一步，因为执行状态不再有固定落点。
- 工具调用、审批等待、恢复续跑都会变成外挂逻辑，难以治理。
- 前端、渠道、监控各自维护一套状态判断，系统口径会越来越乱。

## 这个核心真正保护什么

- 它保护线程语义。线程不是 UI 标签，而是 prompt、记忆、上下文和状态的共同边界。
- 它保护执行顺序。模型调用、工具调用、等待态和终态必须进入同一条事实链。
- 它保护恢复能力。断线、失败、等待审批之后，系统能否继续跑，取决于主链路是否稳定。

## 关键约束

| 约束 | 作用 |
|------|------|
| 线程语义必须稳定 | 不能因为展示层、调试逻辑或临时兼容分支把主链路改坏 |
| 终态必须收敛 | 一轮执行要明确落到 `completed/failed/cancelled/rejected` 这类终态 |
| 恢复必须继续在线程里发生 | 断线续跑、等待审批、后台唤醒都不能跳出线程主语义 |
| 投影不能改写真相 | 前端状态只是快照和投影，不是后端线程真相 |

## 设计重点

### 重点一：线程要比“当前回答”更重要

用户看到的是这一轮回复，但系统真正要维护的是这条线程以后还能不能继续可靠工作。

### 重点二：动作必须回到同一条链上

模型调用、工具 observation、审批等待、会话让出、后台回流，这些动作看起来分散，实际都必须落回线程主链路。否则状态机会被拆碎。

### 重点三：终态不是一个 UI 感觉，而是系统事实

一轮是否结束，不能靠“页面好像不动了”来判断，而要靠明确终态事件和运行态收敛。

## 最容易搞错的点

- 线程不是会话的别名。会话承载交互历史，线程承载执行状态和运行语义。
- 流式输出不是循环本身，流式只是循环对外的连续投影。
- 蜂群、子智能体、定时任务都可以触发执行，但都不能替代线程主语义。

## 和其他核心的边界

- 和 [实时性](/docs/zh-CN/concepts/core-realtime/) 的区别：实时性讲“怎么把线程变化送出去”，智能体循环讲“线程内部怎么稳定推进”。
- 和 [稳定性](/docs/zh-CN/concepts/core-stability/) 的区别：稳定性讲“失败怎么兜”，智能体循环讲“主链路如何组织”。
- 和 [记忆](/docs/zh-CN/concepts/core-memory/) 的区别：记忆讲“线程初始化时带什么”，智能体循环讲“线程之后怎么持续跑”。
