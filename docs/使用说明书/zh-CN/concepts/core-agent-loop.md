# 智能体循环

## 原理

Wunder 的智能体不是"一问一答"模式。它的核心是一个持续运行的循环：

```
你说话 → 智能体思考 → 决定做什么
                            ↓
                    需要用工具？→ 调用工具 → 观察结果
                            ↓              ↓
                    直接回答？  ←  继续思考 ←
                            ↓
                       任务完成？
                       ├── 是 → 给出最终回答
                       └── 否 → 继续循环
```

这个循环可能转一圈就结束（简单问答），也可能转很多圈（复杂任务）。关键是：**智能体自己决定循环多少次**，不需要你一步步指挥。

## 为什么这样设计

如果智能体只是"你问一句、它答一句"，那复杂任务就没法做了。比如你让智能体"分析这份报告并生成摘要文件"，它需要：

1. 先读取报告
2. 分析内容
3. 决定摘要结构
4. 写入文件
5. 确认完成

这每一步都是一个循环。智能体循环让这些步骤可以**在同一次对话中自动完成**。

## 作用

### 对你来说

- **说一次就够了**：你不需要每一步都发指令，智能体会自动推进
- **过程可见**：你可以在界面上看到智能体正在做什么（读取了什么文件、调用了什么工具）
- **可以介入**：遇到需要确认的操作（如修改重要文件），智能体会暂停等你审批

### 对系统来说

- **状态连续**：整个过程记录在一条线程里，不会丢失上下文
- **可以恢复**：如果中途断开，重连后智能体能继续未完成的工作
- **可以追踪**：每一轮做了什么都有记录，方便回顾和排障

## 关键规则

### 线程是执行的主体

每条线程就像一个独立的工作现场，拥有自己的：
- 对话历史
- 执行状态
- 上下文记忆

线程一旦创建，它的核心设定（系统提示词）就保持冻结，不会因为后续对话而改变。这保证了行为的一致性。

### 循环必须有终态

每一轮循环最终必须落到一个明确的结果：

| 终态 | 含义 |
|------|------|
| **已完成** | 任务顺利完成 |
| **失败** | 执行过程出错 |
| **已取消** | 你主动停止了执行 |
| **已拒绝** | 你拒绝了审批请求 |

界面上的状态指示器就是根据这些终态来判断的。

### 前端显示的是投影

你在界面上看到的实时状态（思考中、执行工具中、已完成等）是线程状态的"投影"。真正的执行真相始终在后端的线程里。如果界面显示异常，以实际执行状态为准。

## 实践

### 什么时候耐心等待

智能体在执行复杂任务时可能需要较长时间（读大文件、搜索多个网页等）。看到"正在执行工具"的提示时，耐心等待即可，不需要重复发送相同消息。

### 什么时候主动介入

- **需要审批**：智能体会弹出审批请求，你可以批准或拒绝
- **卡住了**：如果长时间没有响应，可以点击"停止"取消当前执行
- **方向不对**：可以随时补充指令调整方向

### 什么时候用重置

如果线程状态明显异常（界面显示和实际不一致），可以在设置中使用"重置工作状态"功能。这会：
- 停止当前运行
- 清理异常状态
- 重新创建干净的线程

但你的长期资产（知识、技能、配置）不会被删除。

## 常见误区

- **线程不是会话的别名**。会话记录的是对话历史，线程管理的是执行状态。一条会话对应一条线程。
- **流式输出不是循环本身**。你看到的文字一点一点出现，只是循环对外的展示方式，不是循环的核心机制。
- **蜂群任务也遵循循环**。每个工蜂也运行自己的智能体循环，只是由母蜂来协调。
