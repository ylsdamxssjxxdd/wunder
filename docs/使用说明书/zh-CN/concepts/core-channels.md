# 渠道

## 原理

Wunder 支持多种使用入口：
- 桌面应用
- 网页端
- 命令行
- 飞书、企业微信、微信、QQ 等第三方平台

但这些入口不是各自独立的系统。它们共享同一套**执行内核**：调度、线程、工具、事件、治理都是统一的。

### 渠道 = 入口层

渠道只负责：
- 接收你的消息（从什么平台进来）
- 传递系统返回（把结果送回去）

不负责的：
- 不重新定义业务逻辑
- 不改变执行方式
- 不独立存储数据

### 多入口共核

不管从哪个渠道进来，最终都落到同一条线程执行。这意味着：
- 在桌面应用里开的会话，在网页端也能看到
- 从飞书发的消息，和网页端发的消息，智能体用同样的方式处理
- 渠道只是"面子"不同，"里子"是同一套

## 作用

### 对你

**体验一致**：习惯用飞书就继续在飞书用，习惯用网页端就用网页端。能力是一样的。

**无缝切换**：在飞书开了个会话，需要传大文件可以去网页端上传，再回来飞书继续对话。

### 对管理员

**统一治理**：管理端配置对所有渠道统一生效，不用每个渠道单独管理。

**渠道管理**：可以统一查看各渠道的运行状态、消息收发情况、异常告警。

## 渠道接入方式

| 渠道 | 接入方式 | 适合场景 |
|------|----------|----------|
| **桌面应用** | 本地安装 | 个人日常使用 |
| **网页端** | 浏览器访问 | 团队 Server 模式 |
| **飞书** | Webhook + 长连接 | 组织内部协作 |
| **企业微信** | Webhook | 企业通信 |
| **微信** | 官方渠道 | 个人用户 |
| **QQ Bot** | 官方 API | QQ 用户 |

## 关键规则

### 协议适配只到入口

接入层负责协议适配（把飞书消息转成系统能理解的格式，反之亦然），但业务逻辑是统一的。

### 线程语义一致

不管从哪个渠道来的请求，最终都进入同一线程执行。渠道不会创造独立的执行逻辑。

### 状态同步

渠道接入的消息有完整的生命周期管理：
- 接收 → 排队 → 执行 → 结果投递
- 失败会自动重试
- 投递有 outbox 机制保证最终一致性

## 实践

### 配置渠道账号

1. 在对应平台创建应用/机器人（如飞书应用）
2. 获取凭证（AppID、AppSecret 等）
3. 在管理端添加渠道账号
4. 配置 Webhook 回调地址（平台把消息推给 Wunder）
5. 测试消息收发是否正常

### 渠道常见问题排查

如果渠道消息没收到：
- 检查渠道账号状态是否在线
- 查看 Webhook 配置是否正确
- 检查平台侧的消息推送日志
- 在管理端查看渠道运行日志

### 文件附件处理

渠道客户端通常无法直接理解内部文件路径，所以附件会临时转成下载链接。如果渠道里的附件打不开，通常需要检查临时目录服务。

## 常见误区

- **渠道是独立系统？** 不是。渠道只是入口，核心是统一的。
- **不同渠道能力不同？** 核心能力相同，但入口形式可能有差异（如飞书可能不支持某些文件操作）。
- **Desktop 和 Server 是两个系统？** 不是。Desktop 是本地优先的运行形态，Server 是多用户模式，但核心是同一套。
