# 渠道

Wunder 有多种运行形态和接入方式，但这些入口不应该长成互相分裂的系统。渠道核心关注的是“入口怎么多样化”和“内核怎么保持唯一”之间的平衡。

![渠道图：HTTP、WebSocket、Desktop、CLI、第三方渠道共同接入统一运行内核](/docs/assets/manual/core-channels.svg)

## 先给结论

- 渠道只是入口层差异，不是业务内核差异。
- 新入口应该优先复用线程、工具、事件、治理这些已有能力，而不是复制一套逻辑。
- 渠道做得越多，越需要收口，不能越做越例外化。

## 为什么它必须单列为核心

如果没有渠道这个核心，系统很容易演化成这样：

- 聊天 WebSocket 一套状态机。
- `/wunder` SSE 一套状态机。
- desktop/cli 再各自补一层特例。
- 第三方渠道再做一轮协议翻译后顺手重写业务逻辑。

最终结果就是多入口各说各话，线程语义、工具语义和事件语义都不再统一。

## 这个核心真正保护什么

- 保护内核唯一性：所有入口共享同一套线程、工具、事件和治理能力。
- 保护协议边界：接入层负责协议适配，不负责重新定义业务真相。
- 保护演进速度：新入口可以扩展，但不必重新建一套系统。

## 关键约束

| 约束 | 作用 |
|------|------|
| 多入口共核 | 避免一个系统长成多个半独立产品 |
| 接入层只做协议适配 | 防止入口层篡改线程、工具、事件语义 |
| 新渠道优先复用已有能力 | 控制维护成本和行为漂移 |
| 本地形态也要保留内核口径 | desktop 和 cli 不能变成特例系统 |

## 设计重点

### 重点一：渠道多样化不能破坏线程语义

不管消息来自网页、桌面、CLI 还是第三方平台，最终都应落回同一线程和事件体系。

### 重点二：接入差异是“面”，不是“骨”

不同渠道可以有不同鉴权、协议、消息格式、连接方式，但这些都应该止步于接入面。

### 重点三：入口越多，统一口径越重要

渠道不是为了炫技接得更多，而是为了让相同能力在不同入口都可用。真正难的是在扩入口的同时不扩混乱。

## 最容易搞错的点

- Desktop、CLI、Server 是运行形态差异，不是三个独立产品内核。
- 第三方渠道接入是协议适配，不是重新实现一套调度系统。
- 渠道越多越需要收口，而不是越需要例外分支。

## 和其他核心的边界

- 和 [实时性](/docs/zh-CN/concepts/core-realtime/) 的区别：渠道讲“从哪里进来”，实时性讲“状态怎么持续送出去”。
- 和 [多用户管理](/docs/zh-CN/concepts/core-multi-user-management/) 的区别：渠道讲入口统一，多用户管理讲治理统一。
- 和 [稳定性](/docs/zh-CN/concepts/core-stability/) 的区别：渠道负责不分裂，稳定性负责不崩。
