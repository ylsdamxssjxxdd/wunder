# 系统架构

wunder 的架构目标很明确：用一套统一内核承载多种运行形态与多种能力来源。

## 顶层结构

从仓库结构和当前实现看，wunder 可以分成几层：

1. 接入层
2. 调度层
3. 工具与能力层
4. 存储与工作区层
5. 前端与桌面壳层

## 接入层

接入层主要包含：

- `/wunder` 主接口
- 聊天 WebSocket 与 SSE
- 用户侧前端
- 管理端前端
- 渠道入口
- A2A 与 MCP 接入点

这一层的作用是把“外部如何进入系统”统一起来。

## 调度层

调度层是系统核心，主要由这些模块构成：

- `src/api/`
- `src/orchestrator/`
- `src/services/`
- `src/core/`

它负责：

- 解析请求
- 管理会话与线程
- 构建模型上下文
- 发起模型调用
- 处理工具调用
- 记录事件与状态

## 工具与能力层

wunder 的能力不是只来自一个地方，而是多源统一：

- 内置工具
- MCP 工具
- Skills
- 知识库
- 用户工具
- 蜂群与多智能体协作能力

这也是 wunder 和单纯聊天产品最大的区别之一。

## 存储与工作区层

工作区与存储是长期运行的基础：

- 用户工作区用于持久文件与产物
- 会话数据用于历史与回放
- 监控与事件用于可观测性
- 长期记忆用于线程初始化阶段的记忆注入

## 前端与桌面壳层

wunder 当前至少有三套用户可见界面：

- 用户侧前端：`frontend/`
- 管理端前端：`web/`
- 桌面端壳：`desktop/`

它们共享底层能力，但交互目标不同。

## 当前最重要的架构约束

- `server` 是平台核心
- `desktop` 是当前主交付形态
- `cli` 是开发与自动化入口
- 用户侧前端与管理员侧前端必须区分
- WebSocket 优先，SSE 兜底
