# 蜂群

蜂群不是“多开几个聊天窗口”或者“并行跑几个请求”这么简单。它是 Wunder 对多智能体协作关系的正式建模。

![蜂群结构图：母蜂主线程调度多个工蜂线程，工蜂结果再汇回母蜂](/docs/assets/manual/core-swarm.svg)

## 先给结论

- 蜂群的重点不是并行，而是协作关系、职责边界和结果归并。
- 工蜂必须拥有干净线程，否则所谓协作只是把脏上下文复制扩散。
- 母蜂负责编排，工蜂负责执行，这个职责分离不能模糊。

## 为什么它必须单列为核心

如果没有蜂群这个核心，多智能体协作通常会退化成两种糟糕形态：

- 把所有角色硬塞进一个线程，让一个模型“扮演很多人”。
- 靠外围脚本临时派发请求，但没有父子关系、状态同步和结果归并。

这两种做法都无法稳定表达“谁在做什么、谁负责什么、结果怎么回主链路”。

## 这个核心真正保护什么

- 保护协作边界：母蜂和工蜂职责分离，避免上下文和责任互相污染。
- 保护结果归并：工蜂结果必须重新进入母蜂主链路，而不是散落在外部。
- 保护过程可见：协作不是黑盒批处理，而是持续可观察的执行结构。

## 关键约束

| 约束 | 作用 |
|------|------|
| 母蜂只做编排，不吞掉所有执行 | 保持任务拆解和结果归并清晰 |
| 工蜂默认新建线程 | 保持上下文干净，不把旧任务残留带进来 |
| 协作关系必须持续可见 | 让状态同步、工作流和复盘都有依据 |
| 结果最终回主线程 | 保证主智能体的一等现实状态不丢失 |

## 设计重点

### 重点一：蜂群管理的是“已有智能体”

蜂群强调的是调度已经存在的工蜂。它不是临时再造一个执行碎片，而是调用现成角色进入协作。

### 重点二：子智能体和蜂群不是同一件事

子智能体更像当前线程的临时分叉；蜂群更像当前智能体调度其他正式工蜂。两者都能分工，但系统语义不同。

### 重点三：协作质量取决于边界清晰，不取决于工蜂数量

工蜂越多，如果拆解不清、归并不清、状态不可见，协作反而更乱。

## 最容易搞错的点

- 子智能体不等于蜂群。蜂群强调编排语义、节点状态和结果归并。
- “并行”不等于“高效”。拆解粒度过粗或过细都会拉低收益。
- 工蜂线程必须干净，否则上下文污染会直接削弱协作质量。

## 和其他核心的边界

- 和 [智能体循环](/docs/zh-CN/concepts/core-agent-loop/) 的区别：智能体循环是单线程主链路，蜂群是多线程协作结构。
- 和 [实时性](/docs/zh-CN/concepts/core-realtime/) 的区别：实时性解决协作状态怎么对外同步，蜂群解决协作关系怎么建模。
- 和 [稳定性](/docs/zh-CN/concepts/core-stability/) 的区别：稳定性负责控制工蜂失败的扩散，蜂群负责定义工蜂为什么会存在。
