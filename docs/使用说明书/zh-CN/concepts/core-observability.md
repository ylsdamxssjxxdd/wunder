# 可观测性

## 原理

如果系统只能运行、不能解释，出了问题时只能靠猜。可观测性就是让系统"能说话"：
- 发生了什么？
- 为什么这样？
- 从哪里开始不对的？

### 三层架构

| 层级 | 用途 | 面向谁 |
|------|------|--------|
| **事实层** | 原始事件流，记录真实发生了什么 | 开发和运维 |
| **回放层** | 能按时间线重建执行过程 | 排障和审计 |
| **画像层** | 整理后的统计视图，展示趋势和热点 | 管理员 |

**关键原则**：画像不能代替事实。管理员面板看到的是整理后的视角，不是底层真相。出问题时要能从画像层追到事实层。

## 作用

### 排障

出了问题能快速定位：
- 看事件流了解执行步骤
- 看状态变化找到异常点
- 看工具调用记录定位失败

### 审计

操作有记录：
- 谁做了什么
- 什么时候做的
- 结果是什么

### 治理

管理员能看到：
- 哪些工具是热点
- 哪些会话资源消耗高
- 系统整体负载趋势

## 关键区分

### Monitor vs Throughput vs Benchmark

三者都是观测能力，但关注点不同：

| 能力 | 看什么 | 用途 |
|------|--------|------|
| **Monitor** | 线程事实与运行细节 | 排障、实时状态 |
| **Throughput** | 系统承压能力 | 容量规划 |
| **Benchmark** | 能力质量与回归 | 质量评估 |

### 实时性 vs 可观测性

- **实时性**：在线同步，关注"现在"
- **可观测性**：事后解释，关注"过去"

两者结合，既能及时响应，又能复盘分析。

## 实践

### 管理员看什么

**会话监控**：
- 活跃会话列表
- 线程状态和阶段
- Token 消耗统计
- 工具调用记录

**性能观测**：
- 吞吐量和并发数
- 响应延迟分布
- 错误率趋势
- 队列深度

**Benchmark**：
- 任务完成质量
- 能力退化检测
- 版本对比

### 如何排障

1. **看现象**：界面显示什么状态
2. **追事件**：查看事件流找到异常点
3. **查工具**：看工具调用记录
4. **看日志**：查看详细日志
5. **定位根因**：综合判断问题所在

### 指标口径

**最重要的字段**：
- `round_usage.total_tokens`：当前上下文占用
- `thread_status`：线程当前状态
- `turn_terminal`：轮次是否结束

**不要混用口径**：
- 不要混用"单次调用 Token"和"上下文占用"
- 不要混用"会话数"和"线程数"
- 统一口径才能有效比较

## 常见误区

- **日志多 = 可观测性好？** 不是。要有结构、有分层、有口径。
- **画像 = 真相？** 不是。画像是整理后的视图，原始真相在事实层。
- **指标越多越好？** 不是。口径统一的指标才有意义。
