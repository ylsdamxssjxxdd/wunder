# 可观测性

如果系统只能运行、不能解释，它就很难真正可维护。Wunder 把可观测性单列为核心，是因为排障、回放、画像和评测本身就是正式系统能力，不是附属工具。

![可观测性图：事实层、回放层、画像层三层分工](/docs/assets/manual/core-observability.svg)

## 先给结论

- 可观测性不是“多打一堆日志”，而是把系统解释能力做成有分层的结构。
- 事实、回放、画像必须分层，否则前端看到的图、管理员看到的统计、底层真实事件会互相污染。
- 指标口径统一，比再多做一个监控页更重要。

## 为什么它必须单列为核心

如果没有可观测性这个核心，系统通常会陷入两种低效状态：

- 出了问题只能靠猜，知道“不对”，不知道“哪一步开始不对”。
- 数据很多但口径混乱，前端、管理员页、日志和压测报告都在说不同语言。

Wunder 单列可观测性，是为了让“解释系统行为”成为一等能力，而不是临时应急手段。

## 这个核心真正保护什么

- 保护事实清晰度：先知道真正发生了什么。
- 保护复盘能力：不仅能看当前状态，还能按时间线重建过程。
- 保护治理效率：管理员看到的是可行动的画像，而不是一堆原始噪声。

## 关键约束

| 约束 | 作用 |
|------|------|
| 必须区分事实层、回放层、画像层 | 防止监控、回放和仪表盘互相污染 |
| 必须区分请求、结果、观察结果 | 避免把输入、输出和工具 observation 混成一个统计口径 |
| 指标口径要统一 | 不同页面和渠道看到的同一指标必须可对齐 |
| 画像不能代替事实 | 管理员页看到的是整理后的视角，不是底层原始真相 |

## 设计重点

### 重点一：先看事实，再看解释，再看画像

一个成熟系统不应该直接从管理员图表反推底层真相，而应该能从事实层一路追到画像层。

### 重点二：可观测性要服务两类人

- 对开发和运维：回答“为什么出错、卡在哪、怎么复盘”。
- 对管理员：回答“哪里是热点、哪里是风险、哪里该治理”。

### 重点三：benchmark、throughput、monitor 不是一回事

它们都属于可观测能力，但关注点不同：

- `monitor` 看线程事实与运行细节。
- `throughput` 看系统承压。
- `benchmark` 看能力质量和回归。

## 最容易搞错的点

- “日志很多”不等于“可观测性好”。
- 管理员面板看到的是画像，不一定是底层事实的原始形态。
- 没有统一口径的指标越多，排障越容易失真。

## 和其他核心的边界

- 和 [实时性](/docs/zh-CN/concepts/core-realtime/) 的区别：实时性关注在线同步，可观测性关注解释和复盘。
- 和 [稳定性](/docs/zh-CN/concepts/core-stability/) 的区别：稳定性负责让系统少出事，可观测性负责出事后能看懂。
- 和 [多用户管理](/docs/zh-CN/concepts/core-multi-user-management/) 的区别：多用户管理定义切分边界，可观测性在这些边界上形成画像。
