# 长期记忆

## 原理

智能体需要"记住"长期信息才能提供个性化服务。Wunder 的记忆系统围绕一个核心原则：**稳定优于灵活**。

### 核心规则

1. **线程创建时注入一次**：长期记忆在对话开始时一次性注入系统提示词
2. **运行中保持冻结**：后续对话不修改系统提示词
3. **需要时补充检索**：运行中通过 recall 机制补充资料，但不改写底座

### 为什么不每轮都更新

每轮都改写系统提示词会导致：
- 智能体行为不稳定（越聊越"漂移"）
- 缓存无法命中（性能下降）
- 难以追踪问题（不知道基于什么背景在运行）

一次注入 + 冻结 = 稳定的行为 + 可追踪的背景。

## 记忆的形态

Wunder 的记忆采用**结构化碎片**形式，不是一大段自由文本：

| 特点 | 好处 |
|------|------|
| **易于命中** | 智能体能快速找到需要的信息 |
| **易于替换** | 更新单个碎片不影响整体 |
| **易于失效** | 过期信息可以被标记失效 |
| **易于追踪** | 能追踪记忆片段的版本关系 |

## 作用

### 记住你是谁

- 你的偏好和习惯
- 你的工作背景
- 你的历史决策

### 组织知识

- 团队制度和规范
- 项目背景信息
- 领域专业知识

### 个性化服务

- 根据你的偏好调整回答风格
- 参考你的历史上下文
- 提供针对性的建议

## 初始化注入 vs 运行期检索

| | 初始化注入 | 运行期检索（Recall） |
|--|-----------|---------------------|
| **时机** | 线程创建时 | 对话进行中 |
| **方式** | 写入系统提示词 | 作为工具结果补充 |
| **影响** | 成为线程认知底座 | 不改写底座 |
| **持久性** | 整个线程生命周期 | 当前轮次 |

## 实践

### 管理记忆

- **查看记忆**：在智能体设置中查看当前记忆内容
- **添加记忆**：告诉智能体"记住这个偏好"，或在设置中手动添加
- **删除记忆**：删除过时或不需要的记忆碎片
- **更新记忆**：添加新的记忆碎片替换旧的

### 记忆生效时机

- **新对话**：立即使用最新记忆
- **已有对话**：不受影响，保持创建时的记忆

所以更新记忆后需要开新对话才能生效。

### 记忆和知识库的区别

| | 记忆 | 知识库 |
|--|------|--------|
| **内容** | 结构化的个人/组织事实 | 文档、资料、制度 |
| **使用方式** | 初始化时注入 | 运行中检索 |
| **更新频率** | 相对稳定 | 可以频繁更新 |
| **适合** | 偏好、背景、决策记录 | 参考资料、技术文档 |

## 常见误区

- **记忆会每轮更新？** 不会。只在初始化注入一次。
- **Recall 结果会写回系统提示词？** 不会。Recall 是临时补充。
- **记忆越多越好？** 不是。过多记忆挤占上下文空间，可能干扰判断。
