# 记忆

## 原理

智能体需要"记住"长期资料——比如你是谁、你的偏好、组织制度、项目背景等。这些资料如果每次都临时拼凑，智能体就会不稳定。

Wunder 的记忆设计原则：**在线程创建时注入一次，之后保持冻结。不会每轮对话都动态修改。**

为什么这样设计？
1. **稳定性**：线程运行期间认知边界稳定，不会越聊越"漂移"
2. **缓存友好**：系统提示词稳定，利于缓存加速
3. **可追踪**：能明确回答"这条线程是基于什么背景启动的"

### 记忆的两层结构

**第一层：线程初始化注入**

在线程创建时，系统把关键长期资料注入系统提示词。这成为线程的"认知底座"。

**第二层：运行期补充检索**

如果执行中需要更多资料，系统可以通过 "recall" 机制动态检索补充，而不是改写系统提示词。

## 作用

### 对普通用户

- **个人偏好持久化**：你告诉智能体的偏好会被记住，新对话自动生效
- **上下文稳定**：智能体不会因为聊着聊着"忘记"你是谁
- **资料可用**：工作区、知识库的长期资料能被智能体利用

### 对管理员

- **治理可控**：记忆注入规则可控，避免敏感信息泄露
- **行为一致**：同一类对话基于相同的背景启动，结果更可预测
- **可追溯**：能追溯线程的初始化背景

## 记忆的形态

Wunder 的记忆采用结构化碎片的形式，不是一大段自由文本：

- **易于命中**：智能体能快速找到需要的信息
- **易于替换**：更新单个碎片不影响整体
- **易于失效**：过期信息可以被标记失效
- **易于追踪**：能追踪记忆片段的版本关系

## 关键规则

### 冻结原则

线程的系统提示词一旦确定，就保持冻结。后续对话不会动态修改。这是为了保证线程行为稳定。

### 初始化唯一

长期记忆只在线程创建时注入一次。运行中即使触发 recall，结果也只是补充参考，不会改写系统提示词。

### 工作区、知识库、记忆的区别

三者都能提供资料，但职责不同：

| 来源 | 特点 | 用途 |
|------|------|------|
| **工作区** | 当前任务现场 | 存放正在处理的文件 |
| **知识库** | 长期文档来源 | 检索制度、资料、参考文档 |
| **记忆** | 结构化沉淀 | 记住你是谁、你的偏好、历史决策 |

## 实践

### 记忆什么时候更新

- **新线程创建时**：自动应用最新记忆
- **已有线程**：不受影响，保持创建时的记忆

所以如果你更新了个人偏好，需要开启新对话才能生效。

### 记忆和上下文压缩的关系

- **上下文压缩**治理的是对话历史（近场信息）
- **记忆**提供的是长期背景（远场信息）

压缩不触及系统提示词里的记忆，只压缩对话历史。

### 记忆太多会不会有副作用

记忆注入过多确实可能：
- 挤占对话上下文空间
- 引入无关信息干扰判断
- 增加 Token 消耗

所以系统有智能选择和限制机制，不会无限制注入。

## 常见误区

- **记忆会每轮自动更新？** 不会。只在初始化注入，运行期不修改。
- **Recall 会改写系统提示词？** 不会。Recall 是动态补充，不改写底座。
- **工作区文件就是记忆？** 不是。工作区是当前任务现场，记忆是长期沉淀。
