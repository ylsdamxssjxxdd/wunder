# 记忆

Wunder 允许长期资料参与执行，但前提是不能破坏线程的一等现实状态。记忆设计的重点从来不是“尽可能多塞资料”，而是“怎么让资料进入线程又不让线程漂移”。

![记忆图：线程初始化时构造 prompt 并注入一次记忆，运行期通过 recall 补充而不重写底座](/docs/assets/manual/core-memory.svg)

## 先给结论

- 记忆不是每轮动态改写 prompt，而是线程初始化时的长期资料铺底。
- 记忆和 recall 必须分开看：前者是底座，后者是运行期补取。
- 线程一旦开始运行，记忆系统就不能反过来不断重写它的认知边界。

## 为什么它必须单列为核心

如果没有“记忆”这个核心，长期资料通常会落入两种坏做法：

- 每轮都往 prompt 里拼，越跑越漂。
- 把 recall 结果当永久底座写回，导致线程认知边界失控。

Wunder 单列记忆核心，就是为了把“长期事实怎么进入线程”从普通上下文拼接里分离出来。

## 这个核心真正保护什么

- 保护线程稳定性：避免提示词和长期资料在运行中不断漂移。
- 保护长期资料利用质量：让知识库、工作区资料和记忆片段各司其职。
- 保护缓存与复盘能力：能明确回答“这条线程是基于什么底座启动的”。

## 关键约束

| 约束 | 作用 |
|------|------|
| 长期记忆只在线程初始化注入一次 | 防止运行中不断漂移 |
| `system prompt` 首次确定后冻结 | 保持线程认知底座和缓存稳定 |
| recall 不等于重写底座 | 运行期补取资料不应改写线程主设定 |
| 记忆要结构化而不是一大段散文 | 便于命中、替换、失效和追踪 |

## 设计重点

### 重点一：记忆的价值不是“记得多”，而是“记得稳”

对一个长期运行系统来说，稳定的认知边界比临时多塞一点资料更重要。

### 重点二：初始化注入与运行期检索必须分层

线程启动时应该带着清晰的长期背景进入；后续如果还需要更多资料，应通过 recall 或工具检索补取，而不是回头改底座。

### 重点三：工作区、知识库、记忆片段不是同一个东西

它们都能提供资料，但职责不同：

- 工作区更接近当前任务现场。
- 知识库更接近长期文档来源。
- 记忆更接近对长期事实的结构化沉淀。

## 最容易搞错的点

- 长期记忆不是每轮都再次注入。
- 记忆不是万能兜底，错误记忆和过度注入一样会污染执行。
- 知识库、工作区文件、结构化记忆都能提供资料，但职责并不相同。

## 和其他核心的边界

- 和 [上下文压缩](/docs/zh-CN/concepts/core-context-compression/) 的区别：压缩治理近场历史，记忆治理远场长期资料。
- 和 [工具](/docs/zh-CN/concepts/core-tools/) 的区别：工具可以暴露 recall 能力，但记忆核心定义的是时机与边界。
- 和 [智能体循环](/docs/zh-CN/concepts/core-agent-loop/) 的区别：循环定义线程怎么跑，记忆定义线程起跑时带什么。
