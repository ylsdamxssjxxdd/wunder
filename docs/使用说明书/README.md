# 使用说明书维护指南

本目录是 wunder 使用说明书站点的唯一正式源。

## 目录边界

- 站点源：`docs/使用说明书/`
- 中文主文档：`docs/使用说明书/zh-CN/`
- 站点导航：`docs/使用说明书/site.json`
- 站点构建：`python scripts/build_docs_site.py`

以下内容不要放进静态站源目录：

- 临时讨论稿
- 一次性实施清单
- 对标分析草稿
- 与用户阅读无关的内部备忘

## 写作目标（对齐 OpenClaw 风格）

文档要先解决问题，再解释背景。

每一页都应该让读者回答这三个问题：

1. 这页适合我现在看吗。
2. 我下一步该做什么。
3. 我做错时该回哪里排查。

## 页面结构建议

### 入口页（index / hubs）

- 先给角色分流和任务分流
- 再给高频入口卡片
- 最后给误区和延伸阅读

### 接入页（integration）

- 先给选型判断
- 再给最短链路
- 再给关键字段和常见误区

### 工具页（tools）

- 先说“什么时候优先用它”
- 再说“输入输出怎么读”
- 再说“和相邻工具怎么选”

### 运维页（ops）

- 先做部署或排障分流
- 再给检查顺序
- 再给风险点与恢复路径

## 文风约束

- 先判断，后解释。
- 用短句，不堆抽象词。
- 尽量给可执行步骤、接口或命令。
- 少用“你最需要记住”这类口号句，多用“适用场景/常见误区/检查清单”。

## Frontmatter 约定

- 必填：`title`、`summary`
- 推荐：`read_when`、`source_docs`
- 可选：`updated_at: YYYY-MM-DD`

如果没填 `updated_at`，构建脚本会回退到文件修改时间。

## 发布前检查

1. 运行 `python scripts/build_docs_site.py`
2. 抽查入口页、接入页、FAQ 是否可读
3. 通过 `python scripts/update_feature_log.py --type 文档 ...` 记录本次迭代
