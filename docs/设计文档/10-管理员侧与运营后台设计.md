# 管理员侧与运营后台设计

## 1. 设计目标

管理员侧与运营后台是 Wunder 的控制面，负责用户、组织、工具、渠道、知识、监控、性能和桥接能力的统一治理。它的目标不是“把接口都堆成一个管理页”，而是提供可审计、可操作、可定位问题的运营工作台。

当前管理员侧前端位于 `web/`，采用原生 HTML + JS 模块组织。

## 2. 模块组成

| 模块域 | 主要文件 |
| --- | --- |
| 用户与组织 | `web/modules/users.js` `web/modules/user-accounts.js` `web/modules/org-units.js` |
| 工具与技能 | `web/modules/tools.js` `web/modules/user-tools.js` `web/modules/skills.js` |
| MCP 与外部接入 | `web/modules/mcp.js` `web/modules/external-links.js` |
| 提示词与预设 | `web/modules/prompt.js` `web/modules/preset-agents.js` |
| 渠道与桥接 | `web/modules/channels.js` `web/modules/bridge-center.js` |
| 知识与工作区 | `web/modules/knowledge.js` `web/modules/workspace.js` |
| 监控与性能 | `web/modules/monitor.js` `web/modules/performance.js` `web/modules/throughput.js` |
| 评估与模拟 | `web/modules/evaluation.js` `web/modules/sim-lab.js` |

## 3. 设计职责

管理员侧长期应承担以下职责：

- 用户、组织和权限治理。
- 工具、技能、MCP、渠道和外部接入配置。
- 知识库、记忆和预设资产管理。
- 监控线程、任务、吞吐、token、错误与恢复情况。
- 为问题排查提供可复现线索，而不是只给出一个“运行中”状态。

## 4. 架构原则

管理员侧应坚持以下架构原则：

- 以 `web/modules/*` 作为领域拆分单位，不继续向单个入口文件堆逻辑。
- 领域模块通过共享 API 层和共享 UI 组件复用，而不是互相复制。
- 页面状态应围绕运营任务组织，而不是围绕单个接口组织。
- 管理端是控制面，不承担智能体主链逻辑。

## 5. 与后端的边界

管理员侧与后端的边界应该非常清楚：

- 管理端负责读写配置、查看状态、触发治理动作。
- 真正的线程执行、工具治理、投影发布仍发生在后端运行时。
- 管理端不能绕开后端状态模型直接伪造运行态。

## 6. 运营价值

一个合格的运营后台，至少要让管理员能快速回答以下问题：

- 当前谁在线、谁在跑、哪个任务卡住了。
- 哪个渠道、哪个节点、哪个 gateway 连接异常。
- 哪个用户、哪个 agent、哪个 hive 的配置不对。
- 某次错误是 prompt 问题、工具问题、存储问题还是实时链路问题。

## 7. 验收标准

- 管理端模块结构清晰，功能按领域组织。
- 用户、工具、渠道、知识、监控、桥接等核心治理面可独立维护。
- 管理端展示的状态与后端真相源一致，不依赖前端私有推导。
- 新增运营能力优先落到领域模块，不继续做大一体化脚本。
