# 实时投影系统设计

## 1. 设计目标

实时投影系统用于把内部运行态稳定地发布给前端、桌面端、CLI 或外部观察者。它不是“聊天房间系统”，也不是“前端订阅到什么就算什么”的松散推送系统，而是 wunder 的公共状态发布层。

当前设计还需要补充一条交付口径：

- `2026-04-01` 起，实时投影系统的当前发布边界固定为“`一人一会话、各自分散聊天`”。
- 在该边界下，`chat session / user_world / workspace public view / agent profile summary` 已形成可发布主链。
- `shared hotspot` 仍保留为后续容量专项，用于热点共享会话、前门突发、多观察者叠加与 mixed contention 验收。

## 2. 核心定义

实时投影系统的核心定义是：运行时内部状态经过投影、标准化和节流后，对外发布为可消费的公共视图。

这意味着：

- 前端只能消费投影，不直接成为状态源。
- 一个内部对象可以有多个投影面，但只能有一个真相源。
- 房间、列表、详情、摘要都是投影目标，不是底层领域对象本身。

## 3. 投影目标

当前系统中最关键的投影目标如下：

| 投影目标 | 说明 | 主要目录 |
| --- | --- | --- |
| Thread / Session 投影 | 单线程执行态、消息流、waiting 状态 | `src/services/projection/session.rs` `src/services/runtime/thread` |
| Mission Public Board | 多智能体协作任务的公开摘要 | `src/services/mission_public` `src/services/runtime/mission` |
| Hive / Beeroom 投影 | 面向协作域的房间级公共视图 | `src/services/projection/beeroom.rs` `src/services/hive` |
| User World Summary | 面向用户侧的会话、联系人、摘要视图 | `src/services/projection/user_world.rs` |
| Workspace / Agent Profile Public View | 工作目录树版本、产物 hint、智能体摘要首包 | `src/api/realtime_public_view.rs` `src/services/workspace.rs` |
| Projection Watch / Presence | 谁在看、看哪块、是否断线 | `src/services/presence` `src/api/projection_watch.rs` |

## 4. 模块分层

实时投影系统建议长期稳定为四段式：

| 层次 | 职责 | 主要目录 |
| --- | --- | --- |
| Runtime Event 层 | 产出线程和 mission 的内部事件 | `src/services/runtime/thread` `src/services/runtime/mission` |
| Projection Build 层 | 将内部事件转为公共 snapshot / delta | `src/services/projection` `src/services/mission_public` |
| Transport 层 | 通过 WS / SSE 对外分发 | `src/api/chat_ws.rs` `src/api/beeroom_ws.rs` `src/api/user_world_ws.rs` |
| Consumer 层 | 前端、桌面、CLI、外部节点消费并渲染 | `frontend/src/realtime` `frontend/src/stores` |

## 5. 契约设计

实时投影的基础契约应固定为 `Snapshot + Delta + Replay`：

- `Snapshot`：客户端首次接入时获取当前公共视图。
- `Delta`：运行时推进后发送增量变更。
- `Replay`：客户端断线、慢消费或重连后补齐缺失事件。

同时必须提供以下能力：

- `watch`：明确客户端正在观察哪个目标。
- `presence`：记录谁在线、谁在订阅、目标是否活跃。
- `sync_required`：当客户端无法安全追上增量时，明确要求重同步。
- `projection public view`：在 HTTP/WS 首包直接回显 `projection_target + projection_snapshot`，让客户端以统一方式接上 watcher 与 targeted hydrate。

## 5.1 当前热路径分层

当前实现已经按前端价值明确区分热路径，不再把所有状态塞进同一条实时通道：

| 档位 | 当前对象 | 目标 |
| --- | --- | --- |
| `P0` | 用户消息、渠道 `channel_message`、主动流回复、thread/mission/tool 状态、会话 preview | 第一时间可见，正文/按钮/头像/列表同步 |
| `P1` | `user_world` summary、beeroom group summary、agent profile 摘要 | 尽量减少侧栏和摘要面板滞后 |
| `P2` | workspace tree version、artifact hint、active file invalidation、按面板定向 refresh | 提升相关面板新鲜度，但不抢占消息热链 |
| `P3` | 工作目录全文、低频管理配置、大块详情 | 按需拉取，不进入默认热广播 |

这个分层是当前发布可行性的关键前提：实时投影已经不是“更多对象都做成实时”，而是“只保关键变化先可见”。

## 6. 一致性原则

实时投影系统必须坚持以下一致性原则：

- 单写者原则：同一投影目标只允许一个统一发布路径。
- 快慢路径分离：热路径优先小增量，慢路径通过 replay 或 snapshot 补齐。
- 派生状态原则：projection 是派生状态，不是底层真相源。
- 断线恢复优先：任何客户端都可能掉线，协议必须内建恢复能力。
- 慢客户端保护：不能让单个慢连接拖垮整个系统。

## 7. 与协作域和前端的边界

实时投影系统与其他系统的边界必须明确：

- 与智能体核心：投影层消费运行时事件，但不决定线程认知态。
- 与蜂群协作：mission 与 hive 是投影来源，不是投影层自己创造的业务对象。
- 与前端：前端维护本地缓存和 optimistic state，但最终必须被投影校正。
- 与存储：投影快照可持久化加速恢复，但投影存储不是业务主存储。

当前前端边界还应明确两点：

- `frontend/src/realtime/agentWorld/*` 是实时消费和恢复的主落点；`stores/*` 继续收口为状态写回与业务入口装配。
- `settings/tools/system prompt/workspace` 等侧板默认属于补热链路，不能反向挤占消息首屏、首 token 和终态收口。

## 9. 设计风险

实时投影系统最容易退化的方向有三个：

- 让 beeroom 继续承担过多领域职责，导致房间概念再次过载。
- 让前端页面在本地拼自己的状态真相，破坏统一投影契约。
- 让新的实时入口绕开共享 replay / sync_required / watch 基元，形成分叉。
- 让设置页、工具页、工作目录全文或摘要面板重复请求同一份数据，反向拖慢消息热路径和页面切换体感。

另一个必须明确的现实风险是：

- `shared-session / shared hotspot` 在 mixed contention 下仍不是当前阶段的强项。它代表的是“很多流量同时打到同一个 target”，不是普通多用户并发。当前设计允许把它保留为扩容专项，而不要求在当前发布边界内一并解决。

因此新功能应优先写入共享投影基元，而不是在某个页面或某个 WS 接口中单独实现。

## 10. 验收标准

- 关键投影目标都提供 snapshot、delta、replay 能力。
- 断线重连后客户端能恢复到一致状态，而不是要求整页刷新。
- session、mission、hive、user world 使用统一 watch / presence / replay 语义。
- 实时接口以 WebSocket 为主，SSE 只作为兜底或补充。
- beeroom 只保留投影房间职责，不再膨胀为泛业务模块。

当前还应把验收口径拆成两层：

- 发布验收：`一人一会话、各自分散聊天` 下，渠道消息实时渲染、watcher 恢复、会话列表和正文同步、`user_world` 摘要跟随、workspace public view 定向刷新、设置页补热不反向拖慢消息页。
- 容量专项：`shared-session / shared hotspot` 的削峰、前门 burst、mixed contention 与多观察者叠加，留给后续压测和扩容门禁。
