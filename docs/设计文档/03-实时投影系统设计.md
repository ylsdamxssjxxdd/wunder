# 实时投影系统设计

## 1. 设计目标

实时系统用于把内部运行态稳定地发布给前端、桌面端、CLI 或外部观察者。它不是"聊天房间系统"，也不是"前端订阅到什么就算什么"的松散推送系统，而是 wunder 的公开状态发布层。

当前发布边界固定为"一人一会话、各自分散聊天"。在这一边界下，chat session、user world、渠道消息实时投影已形成可发布主链。shared hotspot 仍保留为后续容量专项。

## 2. 核心定义

实时系统的核心定义是：运行时内部状态经过事件化、标准化后，对外发布为可消费的事件流。

这意味着：

- 前端只能消费事件流，不直接成为状态源。
- 一个内部对象可以产生多种事件，但只能有一个真相源。
- 房间、列表、详情、摘要都是事件的消费视图，不是底层领域对象本身。

## 3. 事件目标

当前系统中的关键事件目标：

| 事件目标 | 说明 | 主要目录 |
| --- | --- | --- |
| 会话事件流 | 单线程执行态、消息流、waiting 状态 | `src/services/stream_events.rs` `src/services/runtime/thread` |
| Mission 事件 | 多智能体协作任务的公开事件 | `src/services/runtime/mission` `src/services/swarm` |
| Beeroom 事件 | 面向协作域的房间级实时事件 | `src/services/beeroom_realtime.rs` |
| User World 事件 | 面向用户侧的会话、联系人、摘要事件 | `src/services/user_world.rs` |
| Presence 事件 | 谁在线、看哪块、是否断线 | `src/services/presence/` |

## 4. 模块分层

实时系统建议长期稳定为四段式：

| 层次 | 职责 | 主要目录 |
| --- | --- | --- |
| Runtime Event 层 | 产出线程和 mission 的内部事件 | `src/services/runtime/thread` `src/services/runtime/mission` |
| Event Persist 层 | 持久化事件流、支持 replay | `src/services/stream_events.rs` |
| Transport 层 | 通过 WS / SSE 对外分发 | `src/api/chat_ws.rs` `src/api/beeroom_ws.rs` `src/api/user_world_ws.rs` `src/api/core_ws.rs` |
| Consumer 层 | 前端、桌面、CLI、外部节点消费并渲染 | `frontend/src/realtime/` `frontend/src/stores/` |

## 5. 契约设计

实时系统的基础契约为 `Snapshot + Delta + Replay`：

- `Snapshot`：客户端首次接入时获取当前完整状态。
- `Delta`：运行时推进后发送增量事件。
- `Replay`：客户端断线后补齐未消费事件。

同时提供：
- `watch`：明确客户端正在观察哪个目标。
- `presence`：记录谁在线、谁在订阅。
- `sync_required`：当客户端无法安全追上增量时，要求重同步。

## 5.1 当前热路径分层

| 档位 | 当前对象 | 目标 |
| --- | --- | --- |
| `P0` | 用户消息、渠道消息、流式回复、线程/mission/工具状态 | 第一时间可见 |
| `P1` | `user_world` summary、beeroom group summary | 尽量减少侧栏滞后 |
| `P2` | workspace tree version、artifact hint | 提升面板新鲜度 |
| `P3` | 工作目录全文、低频管理配置 | 按需拉取 |

## 6. 一致性原则

- **单写者原则**：同一事件目标只允许一个发布路径。
- **快慢路径分离**：热路径优先小增量，慢路径通过 replay 或 snapshot 补齐。
- **派生状态原则**：前端视图是派生状态，不是底层真相源。
- **断线恢复优先**：协议内建恢复能力。
- **慢客户端保护**：单个慢连接不能拖垮系统。

## 7. 与其他系统的边界

- 与智能体核心：事件层消费运行时事件，但不决定线程认知态。
- 与蜂群协作：mission 与 hive 是事件来源，不是事件层自己创造的业务对象。
- 与前端：前端维护本地缓存和 optimistic state，但最终必须被事件回压校正。
- 与存储：事件可持久化加速恢复，但事件存储不是业务主存储。

## 9. 设计风险

最容易退化的方向：

- 让 beeroom 承担过多领域职责，导致房间概念过载。
- 让前端在本地拼自己的状态真相，破坏统一事件契约。
- 让新实时入口绕开共享 replay / sync_required / watch 机制。
- 设置页、工具页重复请求拖慢消息热路径。

## 10. 验收标准

- 关键事件目标都提供 snapshot、delta、replay 能力。
- 断线重连后能恢复到一致状态。
- 会话、mission、hive、user world 使用统一事件语义。
- 实时接口以 WebSocket 为主，SSE 作为兜底。
- beeroom 只保留实时分发职责。
