# 编排态系统设计

## 1. 文档定位

本文定义 wunder 用户侧“编排态”的系统语义、运行边界与页面投影规则。

编排态不是一个普通页面开关，也不是母蜂单线程上的弱标签，而是蜂群级运行模式。进入编排态后，母蜂与工蜂会围绕同一个编排 run 持续推进；退出编排态后，蜂群会整体切回新的普通主线程现实。

编排页只是编排态的专用工作台。真正的真相源分布在三个地方：

- 母蜂编排线程与蜂群级编排态对象
- `orchestration/<run_id>/round_xx/` 目录结构
- 编排态下对聊天页、蜂群工具、时间线的统一约束

本文目标是让前端、后端、实时层、目录结构和提示词注入都说同一种“编排态语言”，避免继续把编排实现成页面局部逻辑。

## 2. 核心结论

编排态以母蜂为准，并以蜂群为作用域。

系统级约束如下：

1. 母蜂进入编排态后，整个蜂群进入编排态。
2. 一次编排 run 只在初次“新建编排”时为全员强制新建主线程；后续所有用户轮次必须持续沿用这一组编排线程。
3. 聊天页允许查看编排线程，但必须阻止发送、新建线程、切换线程、改主线程等破坏连续性的操作。
4. 工蜂编排线程丢失不等于退出编排态；只要母蜂仍在编排态，被蜂群工具再次唤起时应自动补回当前编排线程。
5. 编排态的激活与解除以蜂群级状态为准，页面切换不应改变它。
6. 轮次主轴始终是母蜂用户轮次；时间线点击只切换“查看哪个轮次的快照”，不是做线程回溯。
7. 产物展示只看目录结构，不做额外业务标注；态势展示只看各轮 `situation.txt`。
8. 编排提示词不写死在代码里，以 `config/prompts/<locale>/orchestration/*.txt` 为准。
9. 历史轮次允许创建新的编排分支；分支会生成新的 run，不会覆盖旧 run。
10. 当用户切回旧轮次后，只要再次发送消息，就必须自动创建新的分支 run 后继续，不要求用户先显式点击“分支”。
11. 用户可在时间线轮次节点上删除该轮之后的内容；删除语义是保留当前节点，清除其后的轮次与从这些后续轮次派生出的后代分支。

## 3. 系统对象模型

## 3.1 蜂群级状态对象

蜂群级对象建议统一为 `OrchestrationState`，当前实现可对应到蜂群级状态记录：

- `orchestration_id`
- `run_id`
- `group_id`
- `mother_agent_id`
- `mother_agent_name`
- `mother_session_id`
- `active`
- `entered_at`
- `updated_at`

语义说明：

- `mother_session_id` 是当前激活编排态的线程真相源。
- `run_id` 是目录结构、轮次文件和页面投影的统一索引。
- `active=true` 表示当前 run 处于激活编排态；`active=false` 表示当前 run 只是被加载查看，但不再对聊天页和蜂群工具施加激活态约束。

## 3.2 成员线程绑定

成员绑定对象统一为 `OrchestrationMemberBinding`：

- `orchestration_id`
- `run_id`
- `group_id`
- `agent_id`
- `agent_name`
- `role`
- `session_id`
- `title`
- `created_at`

约束：

- `role` 仅允许 `mother / worker`
- 同一 `orchestration_id + agent_id` 同时只保留一条有效绑定
- 母蜂绑定是编排态真相源的一部分；工蜂绑定可在缺失时按需补建

线程命名约束：

- 编排线程自动命名为 `编排+智能体名称`

## 3.3 用户轮次对象

编排态时间线只认母蜂用户轮次，对应 `OrchestrationRound`：

- `id`
- `index`
- `situation`
- `user_message`
- `created_at`
- `finalized_at`

语义说明：

- 用户每向母蜂发送一次消息，最多形成一个母蜂用户轮次。
- 轮次是否算入正式时间线，以 `finalized_at` 是否落定为准。
- 被停止打断的那一轮，只保留必要的抑制范围与临时痕迹，不计入正式轮次。

## 4. 生命周期

## 4.1 新建编排

用户在编排页点击“新建编排”后，系统必须：

1. 为母蜂强制新建主线程。
2. 为全部工蜂强制新建主线程。
3. 将这些线程统一登记为当前编排 run 的成员绑定。
4. 将母蜂和工蜂的主线程全部切换到对应编排线程。
5. 初始化蜂群级编排态对象，并设置为激活。
6. 初始化首轮目录 `orchestration/<run_id>/round_01/`。
7. 为母蜂创建首轮目录，并为所有工蜂创建各自的产物子目录。
8. 清空编排页右侧消息视图，进入新的 run 上下文。

这是一场编排 run 唯一一次“全员新线程”。后续轮次不再重复该动作。

## 4.2 编排态内持续推进

进入编排态后，后续所有轮次都必须保持在同一组编排线程中推进。

明确禁止：

- 每轮重新为母蜂或工蜂新建线程
- 在聊天页把编排线程切走后继续当作当前现实
- 把编排页时间线误当成线程切换器

页面与后端要共同保证“这组线程就是这次编排的当前一等现实”。

## 4.3 关闭编排态

编排页右侧栏提供 `关闭` 按钮，用于退出当前 run 的激活编排态。

关闭时系统必须：

1. 结束当前蜂群级激活态。
2. 清除相关线程上的激活编排上下文与锁定效果。
3. 为母蜂和全部工蜂切换到新的普通主线程。
4. 将当前 run 保留为历史编排记录。
5. 页面仍留在编排页，并继续加载当前 run 的轮次、消息、产物与历史查看上下文。

关闭不是离开编排页，也不是删除 run，而是“退出当前激活态，保留该 run 作为可查看、可再次启动的历史现实”。

运行中约束：

- 当母蜂或工蜂仍在作业中时，关闭按钮必须禁用。

## 4.4 启动编排态

当当前编排页加载的是一个未激活 run 时，右侧栏 `关闭` 按钮位变为 `启动`。

启动的语义是：

- 重新激活当前已加载的 run
- 重新将蜂群主线程切回当前 run 对应的编排线程
- 恢复聊天页锁定与蜂群工具的编排态路由

启动不是新建 run，也不是复制历史 run。

## 4.5 历史切换

编排页右侧栏提供 `历史` 按钮，用于选择母蜂过往的编排 run。

恢复规则：

1. 历史恢复以母蜂历史编排线程为准。
2. 母蜂线程仍存在时应优先复用。
3. 工蜂原编排线程存在则复用，不存在则按当前 run 需求补建新的编排线程。
4. 切到历史 run 时，当前页面的“激活/断开”状态必须继承，而不是强制改写。

继承规则：

- 当前页面处于激活态，切换到别的历史 run 后，目标历史 run 也以激活态加载。
- 当前页面处于断开态，切换到别的历史 run 后，目标历史 run 也以断开态加载。

这条规则的意义是把“看哪次 run”与“当前是否激活编排态”拆开，避免历史切换顺便改变蜂群运行现实。

## 4.6 历史分支

当用户修改了某个旧轮次的态势，并希望从那个轮次重新继续时，系统不应在原 run 上原地回滚，而应创建新的编排分支 run。

分支规则：

1. 分支会生成全新的 `orchestration_id` 与 `run_id`。
2. 原 run 保持不变，继续作为历史现实保留。
3. 新分支继承源 run 的 `1..N-1` 轮正式历史。
4. 目标第 `N` 轮在新分支中被重新打开：
   - 保留该轮态势
   - 清空该轮用户消息
   - 清空该轮旧产物
5. 源 run 中 `N+1` 之后若存在未来轮态势文件，可继续带到新分支作为未来计划，但不带旧产物。
6. 分支后的母蜂线程必须继承分支点之前的可见历史上下文，避免未来轮消息污染新的继续过程。

这意味着“回到旧轮次继续”在系统语义上不是倒带，而是分叉。

自动分支规则：

1. 当前查看轮次不是当前 run 的最新正式轮次时，发送入口仍可用。
2. 用户一旦发送消息，系统必须先从当前查看轮次创建新的分支 run。
3. 新消息落到新分支重新打开的该轮中，而不是写回旧 run。
4. 是否修改过该轮态势，不影响是否分支；只要从旧轮次继续发送，就一定分支。
5. 分支后页面上下文自动切到新分支，并继续按激活编排态推进。

历史态势编辑约束：

1. 当用户在旧轮次查看态下修改态势时，这些修改先只保留在页面的分支草稿中。
2. 旧 run 的 `situation.txt` 与轮次状态在用户真正发送消息前不得被改写。
3. 用户一旦发送消息，系统先创建新分支 run，再将该分支草稿写入新分支对应轮次与未来轮次的 `situation.txt`。
4. 若用户放弃发送、切换 run、关闭编排或新建编排，该分支草稿应直接丢弃，不影响任何既有 run。

## 4.7 删除后续内容

时间线轮次节点允许执行“删除该轮之后的内容”。

删除规则：

1. 删除操作以某个 run 内的指定轮次 `N` 为锚点。
2. 系统保留该 run 的 `1..N` 轮。
3. 系统删除该 run 中 `N+1` 及之后的所有轮次目录、态势和轮次状态。
4. 系统同时删除所有从这些后续路径继续生长出来的后代分支历史条目。
5. 若当前激活 run 就是被裁剪的 run，则编排态保持不变，只是当前 run 的末端被收缩到第 `N` 轮。
6. 删除不影响 `1..N` 轮已有的母蜂上下文连续性。

## 5. 编排线程与聊天页保护

## 5.1 编排线程的定义

编排线程不是普通线程加一个图标，而是受保护线程：

- 它属于某个蜂群级编排 run
- 它受当前编排态激活状态约束
- 它的发送与主线程切换入口不能与普通聊天混用
- 它可能在历史状态下可读但不可写

## 5.2 聊天页的保护规则

当智能体当前主线程处于激活编排态时，聊天页必须进入保护模式。

必须禁用的操作：

- 发送消息
- 新建线程
- 切换线程
- 把别的线程设为主线程
- 通过普通聊天入口继续推进该编排线程

允许的操作：

- 查看线程内容
- 打开历史线程面板
- 查看线程列表中的编排标记

其中“历史线程按钮可点击，但内容不可选、不可切换、不可破坏当前编排态”是重要的交互约束。

## 5.3 线程列表与时间线标识

当蜂群处于编排态时，母蜂与工蜂对应的主线程都应在聊天页时间线与线程列表中带有明显的编排标识。

建议统一表现：

- 专用编排 logo
- 更显著的禁用态按钮
- 只读或锁定提示

这不是视觉装饰，而是避免用户误把编排线程当普通线程操作。

## 6. 工蜂缺失补偿

## 6.1 补偿原则

工蜂线程缺失不自动解除编排态。只要母蜂仍处于激活编排态，蜂群仍被视为在当前 run 中。

## 6.2 补偿触发点

当母蜂在编排态中通过 `agent_swarm` 唤起工蜂时，系统必须：

1. 优先命中当前编排 run 中该工蜂的绑定线程。
2. 若绑定线程不存在，则自动补建新的编排线程。
3. 将补建线程登记回当前 run 的成员绑定。
4. 将该工蜂重新纳入当前编排现实。

补偿触发点在“被唤起时”，而不是后台预补齐。

## 6.3 补偿线程上下文

自动补建的工蜂线程仍然是编排线程，不是普通线程。它必须继承：

- 当前 `orchestration_id`
- 当前 `run_id`
- 当前母蜂轮次 `round_xx`
- 当前轮 `situation.txt`
- 该工蜂本轮产物目录

## 7. 态势与提示词注入

## 7.1 态势文件

每个轮次的态势文本统一保存在：

- `orchestration/<run_id>/round_xx/situation.txt`

规则：

- 可预先填写未来轮次的态势
- 母蜂可修改未来轮次态势文件
- 用户每次给母蜂发送消息时，系统按实际目标轮次读取对应 `situation.txt`
- 不做额外数据库标注，目录内容就是展示与注入真相源

## 7.2 母蜂注入规则

编排页给母蜂发送消息时，注入内容以 `config/prompts/<locale>/orchestration/` 下模板为准。当前母蜂侧至少包含：

- `mother_runtime.txt`
- `situation_context.txt`
- `user_message.txt`

语义要求：

- 母蜂首次收到编排消息时，注入编排运行说明，让它知道自己是母蜂、当前 run 的基本运行方式以及各工蜂目录的默认分工。
- 每一轮发送时，注入当前轮态势与用户消息。
- 不要把仅供工蜂使用的模板内容直接附给母蜂。
- 不要在母蜂消息里注入冗长绝对路径；只保留必要的相对轮次目录信息。

## 7.3 工蜂注入规则

母蜂在编排态内通过 `agent_swarm` 唤起工蜂时，后端必须自动拼接工蜂上下文，而不是依赖母蜂自己手写。

注入来源同样以 `config/prompts/<locale>/orchestration/` 为准，重点包括：

- `worker_first_dispatch.txt`
- `worker_round_artifacts.txt`
- `situation_context.txt`

语义要求：

- 工蜂首次在该 run 中被唤起时，注入“你是工蜂、你负责什么、你本轮产物写到哪里”的首次说明。
- 每一轮都注入当前轮产物目录要求。
- 当前轮存在态势时，将态势附到发给工蜂的消息中。
- 工蜂侧路径使用相对路径，例如 `round_02/<agent_id>`，不要塞入长绝对路径。

## 8. 目录与产物规则

## 8.1 目录结构

当前编排产物目录以 run 和母蜂用户轮次组织：

```text
orchestration/
└─ <run_id>/
   ├─ round_01/
   │  ├─ situation.txt
   │  ├─ <worker_agent_id_a>/
   │  ├─ <worker_agent_id_b>/
   │  └─ ...
   ├─ round_02/
   │  ├─ situation.txt
   │  ├─ <worker_agent_id_a>/
   │  ├─ <worker_agent_id_b>/
   │  └─ ...
   └─ ...
```

补充说明：

- 母蜂侧会维护各轮目录与 `situation.txt`
- 工蜂各自的产物目录位于对应轮次下的工蜂子目录
- 目录结构本身就足以支持轮次快照展示，不需要再额外设计文件级标签系统

分支补充规则：

- 新分支使用新的 `<run_id>` 目录根。
- `1..N-1` 轮目录会复制到新分支下。
- 第 `N` 轮只复制 `situation.txt`，不复制旧工蜂产物目录内容。
- 后续未来轮若已预设态势，只复制各轮 `situation.txt`。

## 8.2 产物展示规则

产物展示只看当前选中轮次目录。

因此：

- 选中第 1 轮，只展示 `round_01/` 下各工蜂目录内的文件
- 选中第 2 轮，只展示 `round_02/` 下各工蜂目录内的文件
- 轮次切换只影响展示快照，不影响线程现实

## 8.3 产物卡片语义

编排页右侧产物节点应被设计成纯容器卡片，而不是带智能体状态、工具调用、标题栏语义的伪智能体卡片。

产物卡片只负责：

- 承载该节点对应目录中的文件格子
- 以文件图标直观展示当前轮可见产物
- 作为流程图中的终端节点参与连线

## 9. 页面投影

## 9.1 画布总原则

编排页画布必须与蜂群页画布保持同一套交互壳层能力：

- 节点卡片可拖拽
- 存在连接线
- 连线与节点状态动画行为一致
- 侧栏行为与蜂群页一致
- 支持规整按钮与全屏按钮

编排页和蜂群页的区别不在画布能力，而在默认布局语义。

## 9.2 默认排布

编排页画布不是死的三列表格，而是“默认排布按三列组织”：

- 左侧一列默认放母蜂卡片
- 中间一列默认放工蜂卡片
- 右侧一列默认放产物容器卡片

约束：

- 母蜂卡片需位于工蜂集合的上下居中位置
- 用户规整时，节点按该默认排布重新整理
- 用户拖拽后仍可自由调整，不应锁死布局

## 9.3 右侧栏

编排页右侧栏顶部动作收敛为图标按钮：

- 新建编排
- 历史
- 启动 / 关闭
- 态势

不再保留：

- 清空
- 刷新
- 独立产物按钮

补充规则：

- “新建编排”等顶部动作样式应与其他面板控制按钮统一。
- 新建编排时自动清空右侧消息视图。
- 运行中关闭按钮禁用。

状态灯规则：

- 灰色：未启动 / 当前 run 处于断开态
- 绿色：当前 run 已激活，等待或可继续编排
- 蓝色：当前 run 正在作业中

状态灯只保留灯体，放在蜂群标题旁。

## 9.4 右侧消息区

右侧消息区展示的是当前选中母蜂用户轮次的消息切片，而不是整个 run 永远连续平铺。

规则：

- 默认查看最新轮次时，可正常发送。
- 点击时间线切到历史轮次后，右侧只显示该轮对应消息，发送区改为禁用。
- 用户回到最新轮次后，发送区才恢复可用。
- 新消息发送时，不得把此前消息清空；应保持本轮消息切片连续可见。

当当前轮正在运行时：

- 发送按钮应切换为停止按钮。
- 停止操作需打断母蜂及当前轮工蜂任务。
- 被停止的这一轮不计入正式编排轮次。

## 9.5 时间线

时间线放在画布内部底部栏，而不是独立页面下方结构。

规则：

- 条目表示的就是母蜂用户轮次
- 支持收起 / 展开
- 收起后底部栏整体隐藏，仅保留贴边的半按钮把手
- 点击轮次只切换查看哪个轮次的消息与产物快照
- 不改变线程绑定，不切换母蜂线程，不触发新的编排运行

可视化规则补充：

- 时间线不是单一平铺线，而是可见分支关系的树形轨道。
- 主干表示当前 run 的轮次推进。
- 从旧轮次创建新 run 后，新的 run 在时间线上以分支轨道显示。
- 分支节点应能看出它来源于哪一轮。

## 10. 后端约束

仅靠前端拦截不够，后端必须统一收口。

## 10.1 聊天接口约束

当某会话处于激活编排态锁定下时，后端必须拒绝普通聊天入口的：

- 发送消息
- 新建线程
- 切换主线程

同时返回明确的编排锁定信息，提示用户应前往编排页继续操作。

## 10.2 编排态接口边界

后端应明确支持以下能力：

- 获取当前蜂群编排态
- 新建编排态
- 关闭当前激活编排态
- 恢复指定历史 run
- 预留轮次
- 完成轮次
- 取消轮次
- 获取编排提示词模板
- 更新编排线程上下文
- 从历史轮次创建新分支 run

## 10.3 蜂群工具路由约束

在母蜂处于激活编排态时，`agent_swarm` 必须：

- 优先命中当前编排绑定线程
- 在工蜂绑定缺失时自动补建编排线程
- 自动注入本轮产物目录与态势上下文

## 11. 验收标准

- 新建编排时，母蜂和全部工蜂只在 run 初次创建时强制新建主线程，且线程标题为 `编排+智能体名称`。
- 关闭编排态后，页面仍停留在编排页，但蜂群整体切回新的普通主线程。
- 启动按钮能够重新激活当前已加载 run，而不是新建 run。
- 历史切换会继承当前页面的激活/断开状态。
- 聊天页在激活编排态下允许查看，但发送、新建线程、切换线程、改主线程等操作被明显禁用并由后端拒绝。
- 工蜂线程缺失时不会自动退出编排态；母蜂再次通过 `agent_swarm` 唤起时能自动补建并重新加入当前 run。
- 每轮态势保存在 `orchestration/<run_id>/round_xx/situation.txt` 中，可提前填写未来轮次。
- 编排提示词以 `config/prompts/<locale>/orchestration/*.txt` 为准，不在代码中写死文案。
- 母蜂注入不包含工蜂专用模板，不包含冗长绝对路径；工蜂注入包含本轮产物目录与态势。
- 时间线只代表母蜂用户轮次；点击轮次只影响消息与产物快照展示。
- 从历史某轮创建分支后，原 run 保留，新 run 从该轮重新继续。
- 时间线可视化能看出主干与分支关系，而不是所有轮次平铺在一条线上。
- 历史轮次右侧栏只读，发送禁用。
- 当前轮运行时，发送按钮切换为停止按钮，停止后该轮不算正式轮次。
- 状态灯能明确区分未启动、已激活、作业中三种状态。

## 12. 结论

编排态的本质是：

- 以母蜂线程为真相源
- 以蜂群为作用域
- 以激活态锁定保护线程连续性
- 以 `round_xx` 目录保存轮次现实

兼容说明：

- 新写入的轮次目录统一采用两位格式，如 `round_01`、`round_02`。
- 后端继续兼容历史 `round_0001` 这类旧目录和旧 round_id，避免旧编排 run、历史恢复与已有产物目录失效。
- 以编排页作为控制台与快照投影

只有把这套系统语义先钉牢，编排页、聊天页、蜂群工具、目录结构和提示词注入才能稳定协同，而不是继续彼此打架。
