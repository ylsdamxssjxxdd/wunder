# 编排态系统设计

## 1. 文档定位

本文不再只描述“编排页面”的视觉方案，而是定义 wunder 用户侧的“编排态”系统语义。

编排态是一种蜂群级运行模式，不是普通页面开关，也不是母蜂单线程上的局部标记。它的目标是把一个蜂群在固定流程下的协作过程，收束成一组受保护、可追踪、可恢复、可解除的专用线程现实。

编排态要解决的核心问题有四个：

- 把母蜂和工蜂的主线程从普通聊天语义中隔离出来，避免随意切换线程破坏编排连续性。
- 把“态势 / 用户轮次 / 工蜂产物目录 / 历史回看”统一绑定到同一个编排 run。
- 让聊天页、编排页、蜂群工具三者对“当前是否处于编排态”拥有一致判断，而不是各管各的。
- 在工蜂线程丢失、用户切页、母蜂恢复历史编排线程等情况下，系统仍能维持编排态的主语义。

因此，后续实现不应再把编排理解为“蜂群页旁边的新页面”，而应理解为“蜂群进入编排态后，页面只是这个运行态的投影与控制台”。

## 2. 核心结论

本设计的第一原则只有一条：

- 编排态以母蜂为准。

展开后是六条系统级约束：

1. 母蜂一旦进入编排态，整个蜂群随之进入编排态。
2. 编排态不是单个 session 的弱标签，而是蜂群级状态，所有成员主线程都升级为编排线程。
3. 编排态期间，聊天页面不得对这些编排线程执行发送、新建线程、切换线程等会破坏连续性的操作。
4. 工蜂的编排线程丢失并不自动解除编排态；只要母蜂仍处于编排态，工蜂会在下次被母蜂通过蜂群工具唤起时自动回到编排线程。
5. 只有母蜂编排态被解除，整个蜂群才算退出编排态。
6. 解除编排态后，蜂群内全部智能体自动创建新的普通主线程，与原编排线程脱钩。

## 3. 与原编排页面设计的关系

“编排页”仍然保留，但它不再是系统定义的主体，而是编排态的专用工作台。

关系收敛如下：

| 维度 | 蜂群页 | 编排页 | 编排态系统 |
| --- | --- | --- | --- |
| 本质 | 通用蜂群观察与派工界面 | 编排态的专用工作台 | 蜂群级运行模式 |
| 真相源 | 蜂群任务与实时事件 | 编排态投影与控制 | 母蜂编排线程及其会话上下文 |
| 线程控制 | 普通主线程语义 | 只操作编排线程 | 统一决定哪些线程被锁定、恢复、解除 |
| 历史恢复 | 普通会话历史 | 编排 run 历史 | 以母蜂历史编排线程为入口恢复 |
| 退出方式 | 无 | 右侧栏“解除”按钮 | 系统级解除并切回普通线程 |

所以，前端、后端和实时层都应从“页面控制”提升为“编排态控制”。

## 4. 编排态对象模型

## 4.1 蜂群级对象

建议把编排态对象收口为一个蜂群级对象：

- `OrchestrationState`

建议字段：

- `orchestration_id`
- `user_id`
- `hive_id`
- `mother_agent_id`
- `mother_session_id`
- `status`
- `enabled`
- `created_time`
- `updated_time`
- `latest_round_index`
- `latest_round_dir`

状态建议：

- `idle`
- `active`
- `recovering`
- `exiting`
- `closed`

这里最关键的是：

- `mother_session_id` 是编排态唯一真相源。
- 其它工蜂线程都只是围绕它维持的附属线程现实。

## 4.2 成员线程绑定

建议新增成员级绑定视图：

- `OrchestrationMemberBinding`

建议字段：

- `orchestration_id`
- `agent_id`
- `role`
- `session_id`
- `is_missing`
- `created_time`
- `updated_time`

约束：

- `role` 仅允许 `mother / worker`
- 同一 `orchestration_id + agent_id` 在任一时刻只保留一条有效绑定
- 母蜂绑定不可为空；一旦母蜂绑定失效，编排态必须进入待恢复或待解除判断

## 4.3 用户轮次

编排态下仍以“母蜂用户轮次”为唯一时间线主轴。

对象：

- `OrchestrationRound`

建议字段：

- `orchestration_id`
- `round_index`
- `user_message`
- `situation_file_path`
- `status`
- `started_time`
- `finished_time`

明确规则：

- 时间线里的“用户轮次”就是母蜂线程上的用户轮次。
- 不按工蜂轮次切片。
- 不按模型轮次切片。

## 5. 编排态生命周期

## 5.1 进入编排态

用户在编排页点击“新建编排”后，系统必须执行以下动作：

1. 为母蜂强制新建主线程。
2. 为全部工蜂强制新建主线程。
3. 将这组主线程全部标记为“编排线程”。
4. 在母蜂线程上写入编排态上下文，明确当前蜂群已进入编排态。
5. 建立蜂群级 `OrchestrationState` 与成员绑定关系。
6. 初始化 `orchestration/<run_id>/round_0001/` 目录。

线程命名约束：

- 自动命名为 `编排+智能体名称`

结论：

- 一次编排里，初次新建时是全员新线程。
- 这是进入编排态的唯一入口。

## 5.2 编排态内持续推进

进入编排态后，后续所有轮次都必须保持在同一组编排线程中推进。

明确禁止：

- 每轮重新新建线程
- 在聊天页把母蜂或工蜂切到别的普通线程继续聊
- 在编排态中把“主线程”概念交回普通聊天页自行决定

也就是说，编排态一旦建立，“这组线程”就是整个蜂群的当前一等现实状态。

## 5.3 解除编排态

编排页右侧栏新增：

- `解除编排态` 按钮

用户点击后，系统必须：

1. 将当前 `OrchestrationState.status` 置为 `exiting`
2. 清除母蜂线程上的编排态上下文
3. 清除各工蜂编排绑定
4. 为母蜂自动新建一个新的普通主线程
5. 为全部工蜂自动新建新的普通主线程
6. 将蜂群整体恢复到普通聊天 / 普通蜂群状态
7. 记录本次编排态为历史可恢复对象

这里必须强调：

- 解除编排态，不是“继续留在旧编排线程但不再显示编排页”
- 而是“整个蜂群退出编排线程现实，切换到新的普通主线程现实”

## 5.4 恢复历史编排态

编排页右侧栏新增：

- `历史` 按钮

作用：

- 选择母蜂过往的编排线程
- 恢复到当时那次编排态

恢复逻辑以母蜂为准：

1. 用户选择一个母蜂历史编排线程。
2. 系统检查该线程仍然存在且可用。
3. 若可用，则以该母蜂线程恢复 `OrchestrationState`。
4. 对应工蜂如果原编排线程仍存在，则恢复绑定。
5. 对应工蜂如果原线程缺失，则自动新建新的编排线程补位。
6. 重新建立蜂群级编排态。

这里的重点不是把每个工蜂历史线程都完美复刻，而是：

- 母蜂历史编排线程必须尽量保真恢复。
- 工蜂线程允许在缺失时新建补位。

## 6. 编排线程语义

## 6.1 什么是编排线程

编排线程不是普通聊天线程加一个 UI 图标，而是一个受保护线程。

受保护的含义：

- 它处于蜂群级编排态内。
- 它的主线程地位不能被聊天页随意覆盖。
- 它的消息发送入口不能与普通聊天混用。
- 它的历史恢复、工蜂派工、态势注入都受编排态约束。

## 6.2 全蜂群升级规则

只要母蜂进入编排态：

- 母蜂主线程升级为编排线程
- 全部工蜂主线程升级为编排线程

在时间线和线程列表中，这些线程都需要特殊标记。

建议：

- 在线程条目中显示专用编排 logo
- 在线程详情、聊天顶部、时间线条目中统一显示编排标识

这不是视觉装饰，而是用户确认“此线程不可随意操作”的关键状态信号。

## 6.3 聊天页保护规则

当智能体当前主线程是编排线程时，其聊天页面必须进入保护模式。

必须禁用的操作：

- 发送消息
- 新建线程
- 切换线程
- 把别的线程设为主线程
- 对该线程执行普通聊天态的线程控制操作

用户可见反馈建议：

- 顶部显示“该智能体当前处于编排态，需在编排页操作”
- 发送框禁用
- 新建线程按钮禁用
- 线程列表切换禁用或拦截

原则：

- 系统不应依赖用户自觉避免误操作
- 必须通过 UI 和接口双层阻断来保护编排态

## 6.4 普通页进入编排线程时的行为

当用户从聊天页或其它入口点进一个编排线程时：

- 允许查看
- 不允许破坏性操作

也就是说：

- 可读
- 不可写
- 不可切
- 不可替换主线程

## 7. 工蜂线程缺失补偿

## 7.1 缺失不等于退出编排态

如果某些工蜂的编排线程因为外部原因丢失：

- 只要母蜂仍处于编排态，整个蜂群仍视为处于编排态
- 不应立即自动退出编排态

原因：

- 编排态的真相源是母蜂
- 工蜂线程属于可补偿成员现实

## 7.2 自动补偿触发点

当母蜂仍处于编排态，并且它通过蜂群工具唤起某个工蜂时：

- 系统先检查该工蜂是否仍有当前编排绑定线程
- 若有，直接命中该线程
- 若没有，自动新建一个新的编排线程
- 该新线程自动加入当前编排态绑定

这意味着：

- 工蜂线程缺失的补偿触发点在“被母蜂唤起时”
- 不是后台无意义地提前补齐所有缺失线程

## 7.3 补偿线程的语义

补偿出来的新工蜂线程仍然是编排线程，不是普通线程。

必须继承的上下文：

- 当前 `orchestration_id`
- 当前轮次 `round_000x`
- 当前轮次 `situation.txt`
- 该工蜂在编排中的角色
- 本轮产物目录位置

也就是说，补偿线程是“加入当前编排现实”，而不是“新开一个普通线程临时帮忙”。

## 8. 态势系统

## 8.1 态势文件

每个轮次的态势文本统一放在：

- `orchestration/<run_id>/round_000x/situation.txt`

规则：

- 可提前创建未来轮次的 `situation.txt`
- 母蜂可以自行修改未来轮次态势文件
- 前端发送到母蜂前，以该轮 `situation.txt` 为准

## 8.2 母蜂注入

编排页向母蜂发送消息时，注入内容只保留母蜂自己需要的部分：

- 编排运行说明
- 当前轮态势
- 用户消息

不应把仅属于工蜂的模板直接注入给母蜂。

## 8.3 工蜂注入

当母蜂在编排态中使用蜂群工具唤起工蜂时，系统必须自动注入：

- 当前轮态势
- 该工蜂本轮产物目录
- 若该工蜂在本次编排中首次被唤起，则再注入工蜂首次说明

关键点：

- 工蜂注入由后端按编排态自动完成
- 不依赖母蜂自己是否手动记得转述

## 9. 画布与页面投影

虽然本文焦点不再是单纯页面设计，但编排页仍然是编排态的主工作台，因此 UI 投影要满足以下约束。

## 9.1 画布

画布继续与蜂群页使用同一套画布壳层能力：

- 卡片可拖拽
- 节点有连接线
- 支持规整
- 支持全屏
- 支持状态动画
- 支持侧栏联动

默认排布语义：

- 左列母蜂
- 中列工蜂
- 右列产物容器

但它不是死板的三列表格，而是画布中的默认组织规则。

## 9.2 时间线

底部时间线是编排态的重要导航器。

时间线条目表示的不是普通会话轮次，而是：

- 母蜂用户轮次

时间线还要承担两个额外职责：

- 标记哪些轮次属于编排线程现实
- 用特殊 logo 标出编排态相关线程与轮次

## 9.3 右侧栏动作

右侧栏顶部动作建议固定为图标按钮：

- 新建编排
- 态势
- 历史
- 解除编排态
- 刷新

说明：

- “历史”用于恢复母蜂历史编排线程
- “解除”用于整体退出编排态

## 10. 目录与产物规则

## 10.1 目录结构

推荐结构：

```text
orchestration/
└─ orch_<id>/
   ├─ round_0001/
   │  ├─ situation.txt
   │  ├─ <worker_agent_id_a>/
   │  ├─ <worker_agent_id_b>/
   │  └─ ...
   ├─ round_0002/
   │  ├─ situation.txt
   │  ├─ <worker_agent_id_a>/
   │  ├─ <worker_agent_id_b>/
   │  └─ ...
   └─ meta.json
```

其中：

- 轮次目录是时间线回放的真相源
- 工蜂子目录是产物展示的真相源

## 10.2 产物展示

产物展示只看当前选中轮次目录。

因此：

- 回看第 1 轮时，只展示 `round_0001/` 下的产物
- 回看第 2 轮时，只展示 `round_0002/` 下的产物

系统不做额外数据库快照标注，也不从别处回补。

## 11. 聊天页与编排态的关系

这是本次重构必须明确的新规则。

## 11.1 在聊天页打开编排线程

如果用户打开的是编排线程：

- 允许查看该线程内容
- 不允许发送消息
- 不允许新建线程
- 不允许切换线程

因此不存在“在聊天页继续给编排母蜂线程发一条普通消息，然后系统自动视为编排消息”的模糊状态。

系统应该明确拒绝，而不是猜测。

## 11.2 在聊天页新建线程

如果母蜂当前仍处于编排态：

- 不允许从聊天页直接为该母蜂新建线程

原因：

- 一旦允许，就会破坏“整个蜂群当前主线程都升级为编排线程”的现实
- 会造成聊天页主线程现实与编排页主线程现实冲突

正确做法只有两个：

- 去编排页继续操作
- 在编排页先解除编排态，再回聊天页操作

## 12. 后端约束建议

仅靠前端拦截是不够的，后端也必须收口。

建议后端增加以下判定：

- 若某 agent 当前主线程处于编排态，则拒绝普通聊天发送请求
- 若某 agent 当前主线程处于编排态，则拒绝普通新建线程 / 切主线程请求
- 若某线程带有编排态上下文但蜂群级编排态已解除，则该线程降级为历史编排线程，只读展示，不再参与当前编排运行

蜂群工具层增加：

- 在母蜂编排态下，`agent_swarm` 必须优先命中当前编排绑定线程
- 若工蜂绑定缺失，则自动补建编排线程

## 13. 实现落点建议

## 13.1 前端

建议新增或强化以下模块：

- 编排态 store：维护蜂群级编排态、历史编排态列表、当前恢复目标
- 聊天页线程保护层：识别“当前线程是否为编排线程”，统一禁用发送、新建、切换
- 线程列表状态装饰：编排 logo、锁定态、只读态
- 编排页右侧栏新增“历史 / 解除”动作

## 13.2 后端

建议明确新增以下边界：

- 蜂群级编排态对象存储
- 母蜂为准的编排态查询接口
- 编排态解除接口
- 编排态历史列表接口
- 历史编排态恢复接口
- 编排线程保护校验
- 工蜂编排线程缺失自动补偿

## 14. 验收标准

- 母蜂进入编排态后，整个蜂群主线程都升级为编排线程。
- 线程列表与时间线中能用特殊 logo 标记编排线程。
- 编排态期间，聊天页面不能对这些线程执行发送、新建线程、切换线程等操作。
- 编排态只以母蜂为准；工蜂线程缺失时，不自动退出编排态。
- 母蜂在编排态中通过蜂群工具唤起工蜂时，缺失工蜂线程会自动补建并重新加入编排态。
- 编排页右侧栏存在“解除编排态”按钮，执行后整个蜂群自动新建普通主线程。
- 编排页右侧栏存在“历史”按钮，可按母蜂历史编排线程恢复当时的编排态。
- 恢复历史编排态时，工蜂缺失线程允许自动补建，不要求完全恢复旧工蜂线程。
- 当前轮态势统一存于 `round_000x/situation.txt`，母蜂和工蜂都能按规则获得正确注入。
- 编排页时间线的“用户轮次”明确等于母蜂用户轮次。

## 15. 结论

编排态的本质是：

- 以母蜂为真相源
- 以蜂群为作用域
- 以线程锁定为保护壳
- 以轮次目录为历史现实层

只有先把这套系统语义立住，编排页、聊天页、蜂群工具和工作区目录才不会各自说不同的话。
