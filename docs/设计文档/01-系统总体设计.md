# 系统总体设计

## 1. 设计目标

本册是 wunder 的系统骨架母本，吸收了原“架构设计 + 模块设计”的正式内容，用于回答四个核心问题：

- wunder 到底是一个什么系统，而不是什么。
- 三种运行形态如何共享同一个核心，而不是各自长成一套系统。
- 核心模块如何分层、装配、协作、演进。
- 后续改动应该落在哪里，哪些改动必须优先收口边界。

目标状态是：无论入口来自网页、桌面、CLI、网关还是外部渠道，最终都落到同一套线程运行时、工具治理、存储抽象和实时投影契约上。

## 2. 架构基线

wunder 当前不是“一个带聊天界面的模型调用器”，而是一个向 **Agent Kernel First** 收敛的智能体运行时系统。

当前架构必须同时满足以下 5 条基线：

1. **智能体优先**
   线程、turn、工具治理、恢复链是主系统；页面、房间、群聊、公共摘要都是投影。
2. **认知态隔离**
   线程首次确定后的 `system prompt` 必须冻结；长期记忆只允许在线程初始化时注入一次。
3. **单写者收敛**
   `submit lease`、`thread/mission/projection route lease` 是当前单服务器阶段的权威边界。
4. **实时系统服务运行时**
   WebSocket / SSE、watch、projection snapshot、lag recovery 围绕运行时复制与恢复，而不是反向支配运行时。
5. **内核宿主解耦**
   `crates/eva` 负责智能体内核（`core/`）与 Wunder 宿主适配（`wunder/`），旧 `src/agent_kernel` 与 `src/orchestrator` 处于迁移过渡期。

一句话概括当前主架构：

`Agent Kernel -> Mission Instance -> Projection Room -> Frontend Runtime`

而不是旧式的：

`Room / UI -> Agent`

### 2.1 当前发布边界

当前系统设计除了讲长期架构，也要明确当前可交付边界：

- `2026-04-01` 起，实时投影系统的发布口径固定为“`一人一会话、各自分散聊天`”。
- 在这一边界下，`chat session`、`user_world`、`workspace public view`、设置页摘要补热和渠道消息实时投影都已进入可发布区间。
- `shared-session / shared hotspot` 仍是容量扩展项，不再阻塞本轮发布；是否进入门禁，取决于后续是否引入“多人同时盯同一 target”的业务形态。

## 3. 系统特色

wunder 的总体特色需要在架构层被明确表达，而不是散落在产品说明里：

| 特色 | 设计含义 | 核心落点 |
| --- | --- | --- |
| 智能体内核先行 | 线程是认知单元，页面只是投影消费层 | `crates/eva` `src/agent_kernel` |
| 实时投影优先 | 面向前端与外部系统的状态同步通过 projection room 完成 | `src/services/projection` `src/services/presence` `src/services/directory` |
| 三形态同核 | `server / desktop / cli` 共用同一 `AppState` 核心装配 | `src/core/state.rs` `desktop/tauri` `wunder-cli` |
| 工具化世界观 | 模型通过工具触达文件、浏览器、MCP、技能、子智能体和外部渠道 | `src/services/tools` `src/services/skills.rs` `src/services/mcp.rs` |
| 蜂群资产化 | Hive 不只是运行时协作域，也是可封装、可导入、可导出的资产单元 | `src/services/hive` `src/services/hive_pack` `src/services/swarm` |
| 双数据库策略 | 服务端用 PostgreSQL，桌面端用 SQLite，但共享统一状态模型 | `src/storage` |

## 4. 仓库与运行形态

### 4.1 工作区结构

| 位置 | 作用 | 当前状态 |
| --- | --- | --- |
| `src/` | Wunder 主后端工程 | 仍是对外主入口 |
| `crates/eva/` | 智能体运行时核心（`core/` 内核 + `wunder/` 宿主适配） | 已承接 thread/session/turn/prompt/context/tools/recovery 及 Wunder 适配 |
| `frontend/` | 用户侧 Vue3 前端 | 正在按 `agentWorld` runtime 下沉实时逻辑 |
| `web/` | 管理端原生 HTML/JS | 维持治理、调试与运营面 |
| `desktop/tauri/` | Desktop 本地 bridge 与桌面壳 | 主交付形态 |
| `wunder-cli/` | CLI/TUI | 本地嵌入式运行形态 |

### 4.2 三种运行形态

`src/core/state.rs` 通过 `AppRuntimeProfile` 明确区分三种运行形态：

| Profile | 入口 | 典型场景 | 默认激活能力 |
| --- | --- | --- | --- |
| `ServerDistributed` | `src/main.rs` | 组织级服务部署 | `thread_runtime`、`mission_runtime`、`cron`、`gateway maintenance` |
| `DesktopEmbedded` | `desktop/tauri/main.rs`、`desktop/tauri/bridge_main.rs` | 本地桌面主应用 | `thread_runtime`、`cron`、本地 bridge、LAN overlay、可选 `remote_gateway` |
| `CliEmbedded` | `wunder-cli/main.rs` | 本地命令行/TUI | 嵌入式 `AppState`，默认不启动后台 `thread_runtime/mission_runtime` |

需要特别注意：

- `ServerDistributed` 是运行画像命名，不代表当前已经进入多节点集群。
- Desktop 与 CLI 都通过 `AppState::new_with_options(...)` 共享同一套后端装配，只是能力启用矩阵不同。
- Desktop/CLI 都会设置 `WUNDER_WORKSPACE_SINGLE_ROOT=1`，意味着本地工作区是长期目录，不应被当作临时目录清理。

## 5. 总体分层与装配

### 5.1 三层主骨架

当前后端最重要的装配锚点是 `src/core/state.rs`。`AppState` 已明确收口为 `kernel / projection / control` 三层。

| Plane | 代表字段 | 主要职责 | 关键模块 |
| --- | --- | --- | --- |
| `kernel` | `AppKernelServices` | 智能体执行、线程推进、任务编排 | `orchestrator`、`runtime/thread`、`runtime/mission`、`agent_kernel`、`eva` |
| `projection` | `AppProjectionServices` | 会话、beeroom、user_world 的实时投影、回放与快照 | `services/projection/*`、`api/realtime_public_view.rs` |
| `control` | `AppControlServices` | presence、route lease、gateway、channels、cron、审批、命令会话 | `services/directory`、`services/presence`、`gateway`、`channels`、`cron` |

其中 `projection` 平面当前已经明确包含两类关键基元：

- `projection public view`：由 `src/api/realtime_public_view.rs` 统一回显 target 与 snapshot 指针，作为 HTTP/WS 首包的 watch 起点。
- `projection snapshot + batch meta read`：由 `src/services/projection/snapshot_store.rs`、`src/services/workspace.rs` 与 `src/storage/*` 统一承担，避免消息页和会话列表再次退化成 per-session N 次存储访问。

### 5.3 为什么必须分成三层

- `kernel` 必须可以在没有前端页面、没有房间订阅的情况下继续运行。
- `projection` 必须可以从 durable event log 与 snapshot 重新构建，不应成为线程认知态来源。
- `control` 负责“谁有权推进”“谁在观看”“谁能接入”，不直接承担模型推理语义。
- `frontend` 侧的 `agentWorld` runtime 负责消费、恢复和补偿，不应重新定义后端真相；像 `channel_message`、watcher restart、summary hydration 这类策略，应继续沉到 `frontend/src/realtime/agentWorld/*`，而不是回流到页面组件。

使用这个图时，优先记住 3 条规则：

- 落在右上象限的改动，要先定边界再写代码，避免把新能力继续堆回旧耦合点。

## 7. 主链路与对象边界

系统总体主链路应长期保持一致：

1. 用户、脚本、渠道或节点从接入层发起请求。
2. 接入层将请求映射为线程请求、mission 请求或系统控制请求。
3. 线程运行时完成 prompt 冻结、context 装配、工具面构建和模型执行。
4. 工具调用写入工作区、访问知识、操作浏览器、触发子智能体或外部系统。
5. 运行时把结果落为 durable state，并生成可观察事件。
6. 实时投影层把内部事件整理为公共 snapshot / delta / replay。
7. 前端、桌面端、CLI 或渠道侧消费投影并呈现给用户。

核心对象层级应长期稳定：

| 对象 | 说明 | 设计重点 |
| --- | --- | --- |
| 组织 / 用户 | 资源归属、隔离、配额与治理主体 | 多租户、多用户、权限与审计 |
| Hive | 围绕业务目标组织的一组协作单元 | 任务域与协作域的稳定容器 |
| Agent | 可持续执行的角色实体 | 角色设定、工具面、模型与权限 |
| Thread | Agent 内部连续执行上下文 | 冻结 prompt、context、recovery |
| Mission / Team Run | 一次多智能体协作任务实例 | 任务编排、分工、汇总 |
| Projection Target | 对外可观察的公开状态视图 | Snapshot、Delta、Replay |
| Workspace / Memory / Knowledge | 智能体面向真实世界的资料现实层 | 文件、长期记忆、知识检索 |

要特别区分三组边界：

- `线程创建用户` 与 `注册用户` 不是一个概念。线程接入时传入的 `user_id` 不要求是已注册用户。
- `单智能体执行态` 与 `蜂群协作态` 不是一个概念。mission 不属于单线程内部对象。
- `认知态` 与 `表现态` 不是一个概念。前端投影和房间状态不能反向污染线程 prompt。

## 8. 当前演进重点

截至 `2026-03-30`，系统有两条最重要的演进主线：

- 智能体核心从旧 `orchestrator` 逐步收口到 `eva`。
- 实时系统从历史 beeroom 过载实现，逐步收口到明确的实时投影系统。
- 当前上线边界是 `unique session / stable fanout / delta-first projection`；shared hotspot 继续作为扩容前置专项，不应反向否定当前主架构的可发布性。

因此后续任何结构性改动，都应先判断它属于：

- 智能体内核问题。
- 投影与协作问题。
- 工具与现实层问题。
- 接入与控制面问题。

不要把不同层的债务混在同一模块里修补。

## 9. 验收标准

系统总体设计的验收关注以下结果：

- 任一入口都能映射到统一 thread 或 mission 主链，而不是自带独立状态机。
- 前端、桌面、CLI、渠道消费的是统一实时投影协议。
- `server / desktop / cli` 使用统一核心语义，只保留接入面差异。
- 数据层明确区分 durable state、projection cache、临时目录和运行时内存态。
- 所有重要高风险能力都能找到明确治理落点，而不是散落在 prompt 或页面逻辑里。
