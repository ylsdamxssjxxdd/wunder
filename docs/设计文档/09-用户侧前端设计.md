# 用户侧前端设计

## 1. 设计目标

用户侧前端是 wunder 面向个人和普通业务用户的主要交互面。它不是静态展示层，而是实时消费线程、mission、user world、workspace 等投影的统一界面。

当前用户侧前端位于 `frontend/`，采用 Vue 3 实现。设计目标是统一视觉风格、稳定实时体验、减少路由与副作用竞态，并持续把超大页面拆分成可维护模块。

## 2. 模块边界

| 模块 | 作用 | 主要目录 |
| --- | --- | --- |
| Views | 页面级入口 | `frontend/src/views` |
| Messenger 子域 | 聊天、消息、会话、蜂群等主交互域 | `frontend/src/views/messenger` `frontend/src/components` |
| Realtime | Beeroom 实时消费者 | `frontend/src/realtime` |
| Stores | 会话、user world、界面状态存储 | `frontend/src/stores` |
| API | 后端请求适配 | `frontend/src/api` |
| Router | 页面与深链路由 | `frontend/src/router` |

## 3. 前端主职责

用户侧前端长期应承担四类职责：

- 呈现线程、mission、user world、workspace 的投影视图。
- 管理路由、页面状态、本地缓存和 optimistic 交互。
- 通过统一实时链路完成 watch、replay、补水和恢复。
- 为桌面端与网页端复用同一用户体验骨架。

前端不应承担以下职责：

- 自己定义线程真相状态机。
- 通过页面特判规避后端状态不一致。
- 以强制重建组件树的方式掩盖副作用边界问题。

## 4. 设计约束

用户侧前端必须长期遵守以下稳定性约束：

- `setup` 阶段禁止先引用后初始化，避免 TDZ 引发整棵组件链崩溃。
- 涉及网络、WS、SSE 的 `watch` 默认不在 `setup` 阶段 `immediate` 启动。
- 卸载时优先取消订阅、取消请求、清理定时器和观察器。
- 路由切换必须幂等，同路径同查询不重复写入。
- 不滥用 `:key` 强制重建大面板来掩盖状态同步问题。
- 页面级卸载不得全局关闭共享连接。
- 异步回调落地前必须检查作用域是否仍有效。

这些约束不是前端风格建议，而是保障实时系统稳定性的硬规则。

## 5. 实时交互主链

用户侧前端的一次标准交互应遵循以下链路：

1. 路由解析出当前目标页面和目标对象。
2. 从 store 与本地缓存恢复已有视图。
3. 与后端建立或复用实时订阅。
4. 消费 snapshot、delta 与 replay。
5. 对发送消息、切换会话、执行动作进行 optimistic 更新。
6. 最终由后端投影结果回压并校正本地状态。

## 6. 模块演进方向

当前需要持续推进的方向包括：

- 继续从超大页面中拆分聊天主体、蜂群工作台和 user world 相关逻辑。
- 让实时消费逻辑尽量下沉到 `frontend/src/realtime` 与 store，而不是散落在页面 watcher 中。
- 统一用户侧视觉风格，保持前端风格一致、协调、美观。
- 避免使用 `backdrop-filter` 等在旧浏览器上高成本的样式方案。

## 7. 与后端的关系

用户侧前端是投影消费者，不是后端状态源。它需要：

- 遵守统一的 session / mission / user world 实时协议。
- 允许短时间 optimistic，但必须接受后端状态回压。
- 在慢网、断线和重连场景下依赖 replay 与 rehydrate，而不是整页刷新。

## 8. 验收标准

- 页面切换和会话恢复不出现大规模重建和显著竞态。
- 实时消息、mission 状态和 user world 摘要能稳定收敛。
- 组件卸载后不会再写响应式状态。
- 页面结构持续模块化，不继续向超大文件堆积新逻辑。
