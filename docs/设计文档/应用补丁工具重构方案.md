# 应用补丁工具重构方案

## 1. 背景

wunder 当前的 `apply_patch` 已经具备可用能力，但在真实模型调用链路里仍然存在几个明显问题：

- 模型心智过重：语法、容错、失败提示、展示结构都堆在一起，模型很难形成稳定使用习惯。
- 执行器职责过多：单文件 `src/services/tools/apply_patch_tool.rs` 已超过 3000 行，解析、匹配、写盘、回滚、diff 生成、提示构造、测试全部耦合。
- 失败恢复不够“收口”：虽然已经补了很多兼容修复，但“自动修复越多，模型越容易继续乱打”。
- 观测负载偏大：前端需要 `diff_blocks` 做精细展示，但模型后续轮次并不需要再次吞下整份 diff。

用户反馈“应用补丁工具模型使用起来老是报错不稳定”，本质不是单点 bug，而是工具协议、执行内核、模型观测三层没有彻底分层。

## 2. 参考项目结论

参考项目 `openclaw-main` 的补丁工具有几个关键优点：

- 协议很窄：只接受 `*** Begin Patch ... *** End Patch` 的单一输入，不鼓励模型发散使用。
- 更新内核很稳：先基于原始文件快照计算 replacements，再倒序应用，避免前块改动污染后块定位。
- 匹配策略克制：优先精确匹配，再做 `trimEnd / trim / 标点归一化` 宽松匹配，不做无限制猜测。
- 成功结果很短：默认只回摘要和文件列表，不把大段 patch 结果重新喂给模型。
- 路径边界默认收紧：默认只允许工作区内改动，减少额外失败面与安全噪音。

wunder 之前已经吸收了其中一部分“replacement + 倒序应用”的思路，但仍缺少完整的结构收口。

## 3. 重构目标

### 3.1 模型侧目标

- 让模型把 `apply_patch` 理解成“少量文件、少量 hunk、明确上下文”的精确编辑工具。
- 让失败后的下一步足够明确：重新读取局部、缩小范围、必要时切换 `write_file`。
- 避免模型在后续轮次反复看到大段 diff、再次抄错展示文本或行号文本。

### 3.2 工程侧目标

- 把更新内核从超大文件中拆出，形成独立模块。
- 保留现有 API 兼容，不破坏前端工作流展示。
- 明确区分三层数据：
  - 执行层：完整 patch、完整 diff、完整错误细节
  - 前端展示层：保留 `files[].diff_blocks`
  - 模型观测层：只保留摘要、文件列表、必要 hint

## 4. 新设计原则

### 4.1 窄协议

模型侧只鼓励两类输入：

- `input`
- `dry_run`

兼容字段保留在后端归一化层，但不再继续强化其存在感。

### 4.2 小批量优先

`apply_patch` 继续只服务于：

- 少量文件
- 少量变更块
- 明确上下文

超出这个范围，明确引导切换 `write_file`，而不是继续在 `apply_patch` 上堆容错。

### 4.3 执行与观测分层

同一份结果在不同消费方应有不同形态：

- 写盘与回滚：需要完整信息
- 前端工作流：需要文件级 diff 细节
- 模型 observation：只需要短摘要和少量结构化文件信息

### 4.4 容错克制

仅保留“高收益、低歧义”的兼容恢复：

- 嵌套 JSON `input` 解包
- `\r\n` 归一化
- `read_file` 展示行号剥离
- `trimEnd / trim / 标点归一化` 匹配

不继续扩张成“任意非标准 diff 都尽量猜着修”。

## 5. 目标结构

第一阶段结构调整如下：

```text
src/services/tools/
├─ apply_patch_tool.rs              # 工具入口、参数归一化、结果封装
├─ apply_patch_update.rs            # Update File 匹配与 replacement 内核
```

后续阶段继续拆分为：

```text
src/services/tools/apply_patch/
├─ mod.rs
├─ input.rs                         # input 解包与轻量修复
├─ parser.rs                        # patch grammar -> ops
├─ update.rs                        # update chunk matching / replacement engine
├─ execute.rs                       # staged apply / rollback
├─ diff.rs                          # diff_blocks 生成
├─ error.rs                         # patch error code + hint builder
```

## 6. 执行链路

新的补丁执行链路分四步：

1. 入口归一化
2. patch 解析与范围校验
3. Update File replacement 计算
4. staged 写盘、原子替换、回滚

其中第 3 步使用参考项目风格的更新内核：

- 基于原始文件快照计算 replacements
- 允许精确匹配失败后做有限宽松匹配
- 所有 replacements 倒序应用

这样可以避免：

- 先改标题、后改正文时定位漂移
- 多块 patch 因前块已生效而导致后块误报上下文找不到

## 7. 模型观测收口

这是本次重构的关键补强。

当前 `apply_patch` 成功结果里包含：

- 总体摘要
- 文件列表
- 每文件 `diff_blocks`

这对前端是对的，但对模型后续轮次是噪音。新的策略：

- 事件 payload 继续保留 `diff_blocks`，前端展示不受影响
- observation payload 对 `apply_patch` 单独压缩，只保留：
  - `changed_files`
  - `added / updated / deleted / moved`
  - `hunks_applied`
  - `no_effect_updates`
  - `files[].action/path/to_path/hunks`

不再把完整 `diff_blocks` 重新喂给模型。

预期收益：

- 减少后续轮次上下文膨胀
- 降低模型再次误抄 diff 行号、hunk 头、展示标记的概率
- 让模型更容易决定“继续读文件”而不是“继续瞎补丁”

## 8. 失败治理策略

失败时继续沿用现有 error code，但策略更统一：

- `PATCH_FORMAT_INVALID`
  - 说明语法错
  - 强制回到最小 patch 形态
- `PATCH_CONTEXT_NOT_FOUND`
  - 给出期望旧片段、邻近源码、部分匹配窗口
  - 引导先 `read_file` 读局部，不整篇重读
- `PATCH_NO_EFFECT`
  - 明确提示新旧内容实质相同
- `PATCH_SCOPE_TOO_BROAD`
  - 明确要求拆批或切换 `write_file`

原则是：

- 让模型“换策略”
- 而不是“继续重复同一工具”

## 9. 分阶段落地

### P0：稳定性止血

- 抽出独立的 Update File 内核模块
- 保持 replacement + 倒序应用为唯一主路径
- 给 `apply_patch` observation 做专门压缩

### P1：结构重构

- 继续把 parser / execute / error 从 `apply_patch_tool.rs` 拆出
- 让入口文件只保留编排和结果封装

### P2：模型体验强化

- 进一步收紧工具描述
- 为 `PATCH_CONTEXT_NOT_FOUND` 输出更稳定的“下一步动作模板”
- 评估是否对 `apply_patch` 增加更强的 dry-run 预检摘要

## 10. 验收标准

### 功能

- 小批量多 hunk 更新可稳定应用
- 前端仍能显示文件级 diff 细节
- 模型 observation 不再携带完整 `diff_blocks`

### 质量

- Rust 编译通过
- 新增/调整单测覆盖 replacement 内核与 observation 压缩
- 不破坏现有 `apply_patch` 错误码与成功骨架
