# 编排页面设计

## 1. 文档定位

本文用于定义用户侧消息页中的“编排”新分区。它不是蜂群页的视觉换皮，而是对现有蜂群运行时的一层固定流程封装，用来把“母蜂派工 -> 工蜂执行 -> 产物落地 -> 母蜂归并”的过程稳定投影成可观察、可回放、可对账的流程界面。

编排页的目标不是替代通用蜂群页，而是补足以下能力：

- 用固定流程约束母蜂和工蜂的协作方式，减少自由派工带来的理解成本。
- 让工蜂结果和产物文件成为一等展示对象，而不只是附着在聊天流里。
- 以“用户轮次”为主轴做时间线切片，支持回看任一轮次对应的产物目录内容。
- 显式隔离旧线程和旧上下文，避免以前对话污染编排链路。

## 2. 与蜂群页的关系

| 维度 | 蜂群页 | 编排页 |
| --- | --- | --- |
| 定位 | 通用蜂群观察与即时协作 | 固定流程的编排工作台 |
| 中栏 | 蜂群列表 | 完全复用蜂群列表 |
| 右侧消息栏 | 可向母蜂或指定工蜂投递 | 视觉复用，但投递目标固定为母蜂 |
| 画布 | 母蜂居中，工蜂围绕 | 画布中的卡片默认按左母蜂、中工蜂、右产物三列组织排布 |
| 线程策略 | 默认复用工蜂当前主线程 | 新建编排时强制全员新主线程；后续用户轮次保持在同一组线程中连续推进 |
| 历史视角 | 当前 mission 为主 | 以用户轮次为主轴的时间线回放 |
| 产物 | 主要通过母蜂工作区查看 | 工蜂产物卡片成为画布主元素，并按用户轮次目录回看历史产物 |

结论：

- 编排页复用蜂群页的中栏、右侧消息区、任务工作流样式和实时链路。
- 编排页新增自己的运行语义、时间线语义、按目录组织的产物回看语义和三列卡片排布布局。
- 编排页不是对现有 `agent_swarm` 默认行为的简单重命名，而是对其进行受控约束的专题页面。

## 3. 固定用户流程

编排页只支持以下主流程，界面和后端语义都围绕这条链路设计：

1. 用户点击左侧栏“编排”按钮进入编排页。
2. 中栏选择一个蜂群。
3. 右侧工作台点击“新建编排”。
4. 系统对该蜂群全部智能体强制创建新的主线程，并绑定到当前编排上下文。
5. 用户点击“态势”按钮，在弹框中填写本轮态势。
6. 用户在右侧消息栏向母蜂发送消息。
7. 母蜂通过 `agent_swarm` 唤起工蜂。
8. 工蜂执行、产出结果与文件产物。
9. 母蜂归并后给出最终回复。
10. 页面在底部时间线生成一个新的“用户轮次切片”，可随时回看。

固定约束：

- 编排页只允许用户对母蜂发起消息，不提供用户直接点对点派工蜂的入口。
- “态势”按用户轮次生效，不允许跨轮次静默沿用。
- 历史普通聊天和历史蜂群记录不会自动回灌到编排时间线。
- 只有用户显式新建编排后，系统才开始记录编排态事件并启用编排产物目录。

## 4. 页面结构

### 4.1 左侧栏

- 在消息页左侧主导航中新增 `编排` 入口。
- 图标建议与蜂群区分，不再复用蜜蜂图标；建议使用 `fa-diagram-project`。
- `MessengerSection` 新增 `orchestrations`（或 `orchestration`）枚举值，不与 `swarms` 混用。

### 4.2 中栏

中栏完全复用蜂群页：

- 数据源继续使用 `beeroomStore.groups / activeGroup / filteredBeeroomGroupsOrdered`。
- 搜索、排序、本地持久化、拖拽排序全部沿用蜂群中栏现有机制。
- 条目样式、活跃高亮、运行态呼吸效果保持与蜂群页一致。

编排页不新增“编排列表”中栏，避免中栏从“蜂群选择”退化成“双层选择”。

### 4.3 右侧工作台

右侧工作台整体骨架复用蜂群页，但有四个关键差异：

- 头部新增 `新建编排`、`态势`、`回到最新` 三个动作。
- 右侧消息输入区目标固定为母蜂，不显示工蜂目标切换下拉。
- 主画布改为“三列卡片排布画布”，但仍保持画布本身的自由浏览语义。
- 画布底部缩略图/小地图移除，改为整条用户轮次时间线。

建议头部动作顺序：

1. `新建编排`
2. `态势`
3. `刷新`
4. `回到最新`（仅历史回放模式显示）

## 5. 画布设计

## 5.1 总体布局

编排画布不是死板的三列流程图，也不是固定泳道。它仍然是一个可平移、可缩放、可容纳动态节点变化的画布，只是默认自动排布规则从蜂群页的“母蜂居中”改成“三列卡片组织”：

- 左列：母蜂列
- 中列：工蜂列
- 右列：产物列

画布目标是让用户一眼看到“谁派工、谁执行、产出了什么”，因此默认阅读顺序服从流程，但实现上仍保留画布的弹性和动态感，而不是做成硬编码的流程图模板。

### 5.1.1 布局规则

- 母蜂节点默认落在左侧区域的视觉主位。
- 工蜂节点默认落在中部区域，并按本轮首次被派发的时间排序，自上而下组织。
- 产物卡片默认落在右侧区域，并与对应工蜂形成清晰的阅读对应关系。
- 连线统一采用正交折线，避免蜂群页那种更自由的放射式视觉。
- 三列是默认排布倾向，不是绝对边界；卡片间距、行高和局部错位可随节点数量动态调整。
- 不参与本轮的工蜂不进入主排布区，统一收纳到画布底部的“待命工蜂”折叠区。
- 子智能体不进入主卡片三列，不作为一等节点展开；它们继续以内嵌 workflow 或详情展开方式附着在工蜂节点上。
- 保留现有画布的平移、缩放、自适应视口和节点定位缓存能力；如后续保留手动拖动，也应是在三列默认排布基础上的微调，而不是取消三列语义。

### 5.1.2 响应式策略

- 宽屏下保持左母蜂、中工蜂、右产物的默认阅读顺序。
- 中屏下压缩工蜂卡和产物卡宽度，但不改变三列语义。
- 窄屏下允许“工蜂卡 + 产物卡”按单行堆叠，但左母蜂、中工蜂、右产物的阅读顺序必须保留。

## 5.2 一等节点

编排页的一等节点只有三类：

| 节点类型 | 是否主节点 | 作用 |
| --- | --- | --- |
| 母蜂节点 | 是 | 展示入口智能体、本轮总状态、归并结果 |
| 工蜂节点 | 是 | 展示每个工蜂的任务、状态、结果摘要与内部 workflow |
| 产物卡片节点 | 是 | 展示该工蜂在本轮产物目录中的文件 |

以下内容不是一等节点：

- 子智能体
- 用户消息
- 态势正文
- 时间线切片

这些内容作为辅助面板、标签或底部时间线出现，不进入主卡片三列。

## 5.3 母蜂节点

母蜂节点必须包含：

- 智能体名称、头像、母蜂标识
- 本轮状态：`ready / running / awaiting_workers / merging / completed / failed / cancelled`
- 本轮用户问题摘要
- 本轮态势摘要标签
- 已派发工蜂数量
- 已归并产物数量
- 最终回复摘要
- 当前轮次的母蜂主线程标识

交互规则：

- 单击：在右侧消息栏聚焦母蜂消息。
- 双击：进入母蜂详情或智能体详情，沿用蜂群页现有行为。
- 历史回看时：节点内容以该轮次的结果摘要和对应产物目录为准，不跟随后续轮次的产物变化。

## 5.4 工蜂节点

工蜂节点必须包含：

- 智能体名称、头像、工蜂标识
- 本轮任务摘要
- 执行状态：`queued / running / awaiting_idle / completed / failed / cancelled`
- 本轮结果摘要
- 产物数量
- 内部 workflow 预览
- 当前轮次工蜂主线程标识

交互规则：

- 单击：弹出该工蜂本轮输出摘要与最近回复。
- 双击：打开智能体详情。
- 展开：可查看该工蜂内部子智能体分支、工具调用轨迹和错误原因。

排序规则：

- 第一关键字：首次被母蜂派发的时间。
- 第二关键字：状态优先级，运行中的排在前面。
- 第三关键字：工蜂名称，作为稳定排序兜底。

## 5.5 产物卡片节点

产物卡片是编排页新增的一等节点，服务于“结果可见”和“历史可回看”。

每张产物卡片至少包含：

- 文件名
- 文件类型
- 所属工蜂
- 文件大小
- 生成时间
- 所属用户轮次
- 预览摘要或缩略信息
- 当前状态：`created / updated / missing`

交互规则：

- 单击：在右侧弹层或对话框中预览该轮次目录下的文件。
- 双击：打开该产物的详情面板。
- 若当前是最新轮次，可提供“打开当前实时目录位置”的二级动作。

显示规则：

- 同一工蜂多份产物按创建时间排序，堆叠在同一行的产物列中。
- 若本轮工蜂没有文件产物，则显示一张占位卡 `无文件产物，仅有文本结果`，保持三列排布下的阅读对齐。
- 历史轮次展示的必须来自该轮次目录，不允许直接读取当前工作区实时文件冒充历史结果。

## 5.6 连线规则

编排页主连线只有两类：

| 连线 | 含义 |
| --- | --- |
| 母蜂 -> 工蜂 | 派工关系 |
| 工蜂 -> 产物 | 产出关系 |

连线标签规则：

- 母蜂到工蜂的连线文案显示该工蜂的任务摘要。
- 工蜂到产物的连线文案显示产物类型，如 `报告`、`代码补丁`、`表格`、`截图`。
- 连线颜色沿用现有状态语义，但饱和度低于节点本体，避免喧宾夺主。

明确不做：

- 不把用户输入也画成流程节点。
- 不把子智能体提升为主连线对象。
- 不把母蜂最终回复再画成右侧第四列节点，最终回复继续留在右侧消息流中。

## 6. 底部时间线

## 6.1 时间线定位

底部时间线取代蜂群页当前的缩略图/小地图，成为编排页的主导航器。

它的切片粒度不是模型轮次，而是用户轮次。

一个时间线切片代表：

- 一条用户消息
- 该条用户消息触发的母蜂工作
- 该轮派发出去的全部工蜂执行
- 该轮所有产物目录内容
- 母蜂最终回复或失败收口

## 6.2 切片内容

每个时间线切片至少显示：

- `第 N 轮`
- 用户问题摘要
- 态势是否已设置
- 工蜂数量
- 产物数量
- 轮次状态
- 开始时间 / 结束时间

状态建议：

- `ready`
- `running`
- `completed`
- `failed`
- `cancelled`

## 6.3 回放模式

用户点击任一时间线切片后，页面进入“历史查看模式”：

- 画布切换到该用户轮次对应的任务结果视图。
- 右侧消息栏仍以当前编排会话为主，不要求回切整段历史消息。
- 产物卡片切换到该轮次目录下的产物文件。
- 发送输入框禁用。
- 头部显示 `回到最新` 按钮。

若用户在回放旧轮次时，最新轮次仍在运行：

- 后台执行继续进行。
- 右侧界面不被实时事件强制打断。
- 最新轮次状态只在时间线切片上做非侵入式更新。

## 6.4 目录要求

为了保证“尤其是产物要能显示那个时候的产物”，编排页不需要额外设计一套数据快照标注，而是直接依赖明确的目录结构。

建议目录结构：

```text
编排产物目录/
└─ 用户轮次1/
   ├─ 智能体1/
   ├─ 智能体2/
   └─ 智能体N/
└─ 用户轮次2/
   ├─ 智能体1/
   ├─ 智能体2/
   └─ 智能体N/
```

展示规则：

- 时间线切到“用户轮次 1”时，只展示 `编排产物目录/用户轮次1/` 下各智能体目录里的内容。
- 时间线切到“用户轮次 2”时，只展示 `编排产物目录/用户轮次2/` 下各智能体目录里的内容。
- 是否把产物放进该目录，交由提示词约束智能体执行，不额外增加系统级标注字段。
- 若某智能体目录为空，则该轮次该智能体显示“暂无产物”。

### 6.4.1 目录命名规范

为避免不同平台和不同数据库形态下的路径歧义，目录命名建议统一收口：

- 编排根目录：`orchestration`
- 编排实例目录：使用稳定 ID，例如 `orch_<id>`
- 用户轮次目录：使用稳定顺序编号，例如 `round_0001`
- 智能体目录：优先使用智能体 ID；若同时需要展示名称，可在前端映射，不建议直接把展示名称写进目录名

推荐结构：

```text
orchestration/
└─ orch_abc123/
   ├─ round_0001/
   │  ├─ mother_agent_id/
   │  ├─ worker_agent_a/
   │  └─ worker_agent_b/
   ├─ round_0002/
   │  ├─ mother_agent_id/
   │  ├─ worker_agent_a/
   │  └─ worker_agent_b/
   └─ meta.json
```

目录规则：

- `round_0001` 这种格式便于排序，不依赖文件系统自然顺序。
- 智能体目录统一使用 agent id，避免中文名、重名和后续改名带来路径漂移。
- 若后续需要少量辅助信息，可只保留一个轻量 `meta.json`，不扩张成复杂持久化体系。

### 6.4.2 前端读取规则

前端读取轮次目录时建议按以下步骤收敛：

1. 根据当前编排实例定位 `orchestration/orch_<id>/`
2. 枚举该目录下全部 `round_*` 子目录，并按轮次编号排序
3. 当前选中某个用户轮次时，只读取该轮次目录
4. 枚举该轮次目录下的智能体子目录
5. 将每个智能体子目录中的文件映射为该智能体右侧的产物卡片

前端显示约束：

- 不扫描整个工作区，只扫描编排根目录下的当前编排实例目录。
- 不做模糊推断，不从普通工作区文件回补到编排卡片。
- 找不到轮次目录时，时间线条目保留，但产物区显示明确空态。
- 找不到智能体目录时，该智能体显示“暂无产物”。
- 文件列表默认按修改时间倒序，其次按文件名排序，保证稳定视觉。

### 6.4.3 提示词约束建议

由于你明确表示产物是否写入目录由提示词约束，因此设计稿里建议把这条约束写进母蜂和工蜂的编排模式提示词：

- 母蜂派工时必须显式告知工蜂当前所属的用户轮次目录。
- 工蜂完成作业后，应将产物写入自己在该轮次下的专属目录。
- 若本轮只有文本结果没有文件，也应在回复中明确“无文件产物”。
- 母蜂归并时，应优先引用各工蜂目录中的文件结果，而不是只口头总结。

这一层约束属于“编排模式提示词协议”，不额外上升为数据库标注系统。

## 7. 运行时语义

## 7.1 新对象：编排会话

建议在蜂群之上新增一个用户态对象：

- `OrchestrationRun`：一次显式启用的编排会话

它不是 hive，也不是 mission 的别名，而是“固定流程模式已开启”的用户态容器。

基本字段建议：

- `orchestration_id`
- `user_id`
- `hive_id`
- `mother_agent_id`
- `status`
- `enabled`
- `created_time`
- `updated_time`
- `latest_round_id`

约束建议：

- 同一用户在同一 hive 下同时只允许一个活动中的 `OrchestrationRun`。
- 新建编排会话时，若已有运行中的会话，必须先结束或显式替换。

## 7.2 新对象：编排用户轮次

编排页的核心账本单位是用户轮次：

- `OrchestrationRound`

基本字段建议：

- `round_id`
- `orchestration_id`
- `user_turn_index`
- `user_message`
- `situation_text`
- `status`
- `worker_total`
- `artifact_total`
- `started_time`
- `finished_time`

## 7.3 线程隔离规则

这是编排页最重要的语义，不允许做模糊处理。

### 7.3.1 新建编排时

用户点击“新建编排”后，系统必须：

- 对该 hive 的母蜂与全部工蜂强制创建新的主线程。
- 清空本次编排对旧主线程的依赖。
- 为当前编排会话记录本次线程绑定。

### 7.3.2 后续用户轮次

一次编排里，只有初次点击“新建编排”时才强制为母蜂和全部工蜂创建新的主线程。

后续用户轮次必须保持在这同一组线程里连续推进，原因很明确：

- 编排需要连续性，母蜂和工蜂要保留前面轮次已经建立的上下文。
- 用户轮次切片主要是为了查看产物，不是为了把线程重新切断。
- 如果每轮重建线程，会把这次编排变成多段互不相连的短任务，不符合你的目标。

因此编排页在线程策略上是：

- 新建编排时：全员新线程
- 编排过程中后续轮次：保持同一线程连续执行

### 7.3.3 工蜂派工约束

母蜂虽然仍使用 `agent_swarm` 工具，但在编排页中不能放任它自由命中任意旧线程。系统必须做受控封装：

- 新建编排时，为每个工蜂绑定本次编排专用的主线程。
- 母蜂后续派工时，运行时优先命中这次编排建立的那组线程。
- 即使外部同名智能体刚好也有别的主线程，编排页也不能漂移过去。

换句话说，编排页里 `agent_swarm` 是“受控派工”，不是“开放派工”。

## 7.4 态势注入规则

态势不是长期记忆，也不是要靠反复重建线程来实现的轮次变量。

编排页对态势的正确处理方式应为：

- 在用户发起该轮消息前，由母蜂读取当前轮态势并将其作为本轮背景继续推进。
- 态势块同时对母蜂与工蜂构成这次轮次的协作背景。
- 态势文本按用户轮次记录，供时间线条目和目录视图识别。
- 该轮开始后，态势锁定，不允许中途修改。

建议语义：

- 态势是 `Round Context`，不是 `Long-term Memory`。
- 态势按轮次填写，但线程不因态势变化而重建。
- 新一轮若未设置态势，系统应阻止发送，要求用户先确认本轮态势。

为兼顾效率，态势弹框可以预填上一轮文本作为草稿，但不应默认生效，用户必须显式确认。

## 7.5 防污染规则

为了避免以前的对话信息污染编排内部，编排页必须同时满足三层防污染：

### 7.5.1 线程防污染

- 只在新建编排时创建编排专用线程
- 编排过程中固定使用这组线程
- 不把普通聊天线程带入编排会话

### 7.5.2 事件防污染

- 只有编排启用后的 mission / task / artifact 才进入编排时间线
- 老蜂群任务不做历史补录
- 普通蜂群页与编排页的事件流分开标识

### 7.5.3 产物防污染

- 各轮产物落在各自的用户轮次目录中
- 历史查看只读取对应轮次目录，不直接读取当前实时目录
- 即使后续轮次生成了新文件，也不会覆盖前面轮次目录中的内容

## 8. 产物目录设计

编排页的产物隔离不依赖额外标注对象，而依赖稳定目录结构。

推荐结构：

```text
<agent_workspace>/orchestration/
└─ <orchestration_id>/
   ├─ 用户轮次1/
   │  ├─ 母蜂/
   │  ├─ 工蜂A/
   │  └─ 工蜂B/
   ├─ 用户轮次2/
   │  ├─ 母蜂/
   │  ├─ 工蜂A/
   │  └─ 工蜂B/
   └─ ...
```

实现约束：

- 编排页只认这套目录，不额外要求数据库里新增“产物快照表”。
- 时间线点击某一轮时，读取该轮目录下各智能体子目录并生成产物卡片。
- 智能体是否把产物放进对应目录，由提示词约束，不额外在系统层补复杂标注。
- 若目录不存在或为空，前端按空态处理即可。

建议补一条最小元信息文件：

- `meta.json` 只用于记录 `orchestration_id / hive_id / mother_agent_id / created_time / latest_round`
- 不记录每个文件的额外索引，不扩张成新的复杂账本
- 其职责只是帮助前端快速确认当前编排目录是否有效

## 9. 实时事件与状态收敛

编排页不应自己发明一套完全平行的运行时，而应建立在现有 `mission / task / beeroom realtime` 之上，主要增加编排会话、用户轮次和目录读取能力。

建议新增的编排事件类型：

- `orchestration_created`
- `orchestration_reset`
- `orchestration_round_prepared`
- `orchestration_round_situation_set`
- `orchestration_round_started`
- `orchestration_round_worker_dispatch`
- `orchestration_round_worker_update`
- `orchestration_round_artifact_indexed`
- `orchestration_round_completed`
- `orchestration_round_failed`

状态收敛原则：

- 实时态由 mission / task / run 推动。
- 编排页只是消费这些事实并按用户轮次重新组织投影。
- 轮次切换时优先读取对应轮次目录，不要求前端拼装一整套历史运行态快照。

## 10. 前端落点建议

考虑到现有蜂群相关文件已经较多，编排页不要继续把大量逻辑堆进现有 `beeroom` 组件中，建议新增独立目录：

```text
frontend/src/components/orchestration/
├─ OrchestrationWorkbench.vue
├─ OrchestrationCanvas.vue
├─ OrchestrationCanvasPane.vue
├─ OrchestrationTimelineBar.vue
├─ OrchestrationSituationDialog.vue
├─ orchestrationCanvasModel.ts
├─ orchestrationCanvasStateCache.ts
└─ useOrchestrationRuntime.ts
```

建议复用但不硬拷贝的现有能力：

- 复用 `BeeroomCanvasChatPanel.vue` 作为右侧消息栏基础组件
- 复用现有 workflow 预览和节点状态色语义
- 复用蜂群中栏 store 与条目排序
- 复用 `WorkspacePanel` 作为实时工作区补充入口

明确新增：

- 编排专用 store：`frontend/src/stores/orchestration.ts`
- 编排专用时间线状态缓存
- 编排专用三列卡片排布画布模型
- 历史轮次目录查看模式控制

`MessengerView.vue` 只做 section 接入，不继续承载编排业务细节。

## 11. 后端落点建议

建议新增独立边界，而不是把编排逻辑散落进现有蜂群 service：

```text
src/api/orchestration.rs
src/services/swarm/orchestration/
├─ mod.rs
├─ service.rs
├─ repo.rs
├─ artifact_dirs.rs
├─ runtime.rs
└─ types.rs
```

职责划分建议：

- `service.rs`：编排会话、轮次、状态机
- `runtime.rs`：初次线程重置、态势管理、受控派工
- `artifact_dirs.rs`：编排目录解析、轮次目录读取、产物索引
- `repo.rs`：编排会话与用户轮次持久化读写
- `types.rs`：编排对象、事件与目录索引结构

数据库侧需要同时覆盖 PostgreSQL 与 SQLite。

## 12. 关键交互细节

### 12.1 新建编排

- 只有选中蜂群后才可点击。
- 若当前 hive 已有运行中的编排，按钮应提示“结束当前编排后再新建”或“替换当前编排”。
- 新建后右侧画布进入 `ready` 状态，但时间线为空。

### 12.2 态势按钮

- 仅在最新轮次可编辑。
- 历史查看模式下只允许查看，不允许修改。
- 若当前轮次尚未设置态势，则发送按钮不可用。

### 12.3 发送消息

- 发送目标固定母蜂。
- 文本区仍沿用蜂群页样式，但去掉目标选择器。
- 运行中不可再次发送，避免一轮内并发双入口破坏固定流程。

### 12.4 回到最新

- 仅当用户选中历史轮次时出现。
- 点击后回到最新轮次，并恢复实时追踪。

### 12.5 产物预览

- 历史轮次打开的是该轮次目录中的文件。
- 最新轮次可以附加“查看实时工作区”动作。
- 目录缺失时必须明确报错，不能静默回退到当前实时文件。

## 13. 边界情况

以下情况必须在设计中提前收口：

- 工蜂本轮只有文本回复，没有文件产物。
- 某智能体在某轮目录下未放置产物时，前端按空目录处理。
- 某些工蜂未被派工，不应占据主流程画布空间。
- 用户切到历史轮次时，最新轮次仍在后台继续运行。
- 用户在未设置态势时尝试发送消息。
- 新建编排后中途切换蜂群，需要阻止脏状态串到别的 hive。
- 编排页关闭或刷新后重新进入，最新编排会话和时间线能从后端恢复。

## 14. 验收标准

- 左侧栏存在独立“编排”入口，中栏与蜂群页保持同构。
- 新建编排后，蜂群全员被强制切到新的主线程。
- 新建编排时，母蜂与工蜂强制切到新的主线程，后续轮次保持连续。
- 态势按轮次填写，但不要求每轮重建线程。
- 用户只能向母蜂发送消息，不能在编排页里直接改派工蜂。
- 画布中的卡片默认按左母蜂、中工蜂、右产物三列组织排布，主连线清晰。
- 子智能体不会把主画布冲散，只作为工蜂内部细节展示。
- 底部时间线以用户轮次切片，点击后能回到对应轮次状态。
- 历史轮次能查看对应轮次目录里的产物，而不是当前工作区里的最新文件。
- 编排未启用时，不记录、不回灌、不污染既有蜂群对话和普通聊天。

## 15. 结论

编排页的本质不是再加一个“蜂群子页面”，而是把蜂群运行时收束成一个固定、可回放、以产物为中心的流程工作台。

它必须同时成立三件事：

- 视觉上有清晰的流程阅读顺序，但仍然保持画布而不是死板模板。
- 语义上按用户轮次切片，而不是按实时消息拼贴。
- 运行时上只在新建编排时切出专用线程，后续保持连续；产物则通过目录按轮次隔离。

只有把这三点同时做实，编排页才会真正比蜂群页更清晰，而不是只是多一个按钮。
