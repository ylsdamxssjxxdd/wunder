# 接入、网关与渠道系统设计

## 1. 设计目标

接入系统负责把 Wunder 的核心能力稳定暴露给不同入口，包括网页、桌面、CLI、外部节点、第三方渠道和自动化系统。目标不是堆更多入口，而是让所有入口共享同一套运行时、存储和实时语义。

## 2. 入口类型

| 入口类型 | 说明 | 主要目录 |
| --- | --- | --- |
| HTTP API | 普通管理与业务入口 | `src/api` |
| WebSocket | 聊天、实时世界、beeroom、core 实时链路 | `src/api/chat_ws.rs` `src/api/user_world_ws.rs` `src/api/beeroom_ws.rs` |
| Gateway | 面向节点、操作端、渠道端的统一接入层 | `src/gateway` `src/api/gateway_ws.rs` |
| Channels | 飞书、微信、QQ、XMPP、WhatsApp 等通讯渠道 | `src/channels` |
| A2A / External | 智能体互联与外部系统嵌入 | `src/api/a2a.rs` `src/api/external_links.rs` |
| Desktop Local | 本地桌面模式的裁剪入口 | `src/api/desktop.rs` `src/api/desktop_lan.rs` |

## 3. API 层原则

`src/api/*` 应长期保持薄层，只承担以下职责：

- 鉴权、参数校验、协议兼容。
- 路由到具体服务或运行时。
- 将结果转换为 HTTP / WS 响应。

API 层不应承载复杂业务逻辑，也不应成为旧逻辑的最终垃圾场。

## 4. Gateway 设计

Gateway 是 Wunder 的统一接入中枢，主要解决多节点、多角色、多路由所有权的接入问题。

其关键对象包括：

| 对象 | 含义 |
| --- | --- |
| Client / Node / Channel | 不同类型的接入角色 |
| Route Owner | 某类目标路由的所有者 |
| Route Lease | 一段时间内某条路由的持有关系 |
| Route Fence | 路由边界保护，防止冲突和越权 |
| Presence | 接入连接的在线状态 |

Gateway 的设计重点是协议稳定、角色清晰、路由安全，而不是只做一个简单 WS 转发器。

## 5. Channels 设计

渠道系统负责把 Wunder 接到外部通讯世界。其设计要点包括：

- inbound 与 outbound 分离。
- 附件、媒体、待处理文件、重试和限流有独立模块。
- 渠道路由与工作区路由统一治理。
- 渠道适配器只负责协议转换，不复制智能体内核。

当前目录中已经形成了较清晰的拆分，如 `inbound_queue`、`outbox`、`media`、`pending_files`、`workspace_routing`、`rate_limit` 等，这个方向应继续保持。

## 6. 用户身份与线程身份

接入层必须长期坚持以下身份规则：

- 实际注册用户由用户管理体系维护。
- 线程创建用户可以是任意传入标识，不要求已注册。
- `/wunder` 接口传入的 `user_id` 只代表本次线程归属，不自动等同于后台账户。

这个规则对外部嵌入、渠道接入和自动化调用都非常关键。

## 7. 通讯原则

系统通讯设计统一遵循以下原则：

- WebSocket 优先，SSE 作为兜底或补充。
- 实时链路尽量复用统一 watch、replay、sync_required 语义。
- 桌面局域网模式和远程 gateway 模式共享核心协议，不共享部署方式。

## 8. 模块落点

| 目录 | 说明 |
| --- | --- |
| `src/api` | 所有对外 API 与 WS 接口 |
| `src/gateway` | Gateway 协议、上下文、调用与 admin 协议 |
| `src/channels` | 外部渠道适配与消息传输 |
| `src/services/bridge` | 桥接服务与身份适配 |
| `src/services/external` | 外部系统接入辅助 |

## 9. 验收标准

- 任一新入口都能复用统一线程、mission 和投影主链。
- Gateway 能稳定管理多角色接入与路由所有权。
- Channels 适配器与核心运行时分层清楚，不会复制智能体内核。
- 用户身份、线程身份、注册身份三者边界清楚。
