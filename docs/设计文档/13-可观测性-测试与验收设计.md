# 可观测性、测试与验收设计

## 1. 设计目标

wunder 的复杂度来自运行时、实时系统、工具执行、协作任务和多入口接入。没有可观测性和测试体系，系统只会在演示时看起来可用，上线后迅速失控。本设计用于定义“如何看到系统正在发生什么”和“如何判断系统已经达到验收门槛”。

## 2. 可观测性对象

| 对象 | 关注点 | 主要目录 |
| --- | --- | --- |
| 线程运行时 | 当前状态、waiting、resume、错误、恢复链 | `src/services/runtime/thread` `src/agent_kernel` |
| Mission 运行时 | 任务推进、agent activity、汇总状态 | `src/services/runtime/mission` `src/services/swarm` |
| 实时投影 | watch 数、replay、lag、sync_required | `src/services/presence` `src/services/projection` |
| 工具调用 | 调用量、失败率、审批等待、超时、重试 | `src/services/tools` `crates/eva/src/tools` |
| 系统吞吐 | token、模型轮次、会话量、错误率 | `src/api/benchmark.rs` `web/modules/monitor.js` `web/modules/throughput.js` |

## 3. 测试层次

建议长期维持五层测试与验证体系：

| 层次 | 目标 |
| --- | --- |
| 单元测试 | 验证模块内部逻辑、边界和恢复语义 |
| 集成测试 | 验证 API、运行时、存储和投影协同 |
| 回归测试 | 锁定历史高频问题，避免重复踩坑 |
| 仿真与压测 | 验证实时链路、并发任务、吞吐与恢复能力 |
| 手工验收 | 验证用户路径、运营路径、异常路径与文档一致性 |

## 4. 代码级验证基线

对 Rust 主工程，推荐长期保持以下代码级验证基线：

- `cargo fmt --all`
- `cargo check -j 8`
- 必要时 `cargo clippy -j 8`
- 与当前改动相关的定向 `cargo test -j 1 ...`

对前端与管理端，应补充与当前改动相关的构建和关键路径验证，重点检查：

- 路由恢复与实时订阅是否稳定。
- 组件卸载后是否仍写状态。
- 大面板切换是否引入重复订阅和明显抖动。

## 5. 可观测性设计原则

可观测性设计必须遵守以下原则：

- 重要状态必须显式建模，而不是只出现在日志里。
- 线程与 mission 的终态、waiting 态、恢复态都必须可观察。
- 实时链路要有 watch、lag、replay 和 sync_required 观测数据。
- token、模型轮次、耗时、失败原因要能对齐到真实对象。
- 监控口径必须稳定，避免前后端对“总消耗”和“上下文占用”口径不一致。

## 6. 验收方法

每个大模块的验收至少要回答两个问题：

1. 正常路径是否稳定可用。
2. 异常路径是否可恢复、可观测、可定位。

建议将验收按以下维度组织：

- 智能体核心：初始化、续跑、waiting、恢复、工具调用。
- 实时投影：首包、增量、重连、慢客户端、列表与详情一致性。
- 工具系统：审批、沙盒、失败、重试、日志。
- 协作系统：任务拆分、并行执行、汇总、公开摘要。
- 接入系统：HTTP、WS、gateway、channels、desktop、cli 统一语义。

## 7. 文档与验收的关系

验收不仅是跑测试，还包括文档与实现的一致性检查。后续每次重要迭代应同步维护：

- `docs/设计` 的骨架说明。
- `docs/API` 的接口与协议事实。
- `docs/功能迭代.md` 的分类记录。
- 若改动涉及使用说明书或图示，还应更新站点构建产物。

## 8. 验收标准

- 关键模块都有明确的观测指标、日志和状态对象。
- 重要改动都有对应的构建、检查或回归验证。
- 高风险路径不仅能运行，还能在失败时被快速定位。
- 文档、接口、监控口径和实际实现保持一致。
