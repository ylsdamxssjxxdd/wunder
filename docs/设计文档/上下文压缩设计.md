# 上下文压缩设计

## 1. 目标

- 对齐 Codex：压缩后提交一份新的 `replacement_history`。
- 压缩后的本次继续执行请求、下一次真实请求、刷新后的历史回放，都基于同一份压缩基线。
- 请求体只保留继续任务真正需要的语义，不回放工具噪音。
- 运行中压缩不能让模型重复执行已经完成的工具，也不能丢失当前轮失败后的修复现场。

## 2. 触发模式

| 模式 | `trigger_mode` | 场景 |
| --- | --- | --- |
| 手动压缩 | `manual` | 用户主动触发 |
| 运行中压缩 | `auto_loop` | 智能体循环过程中历史过长 |
| 溢出恢复 | `overflow_recovery` | 主请求已触发上下文超限 |

## 3. 摘要请求组装

摘要请求只负责生成 `[上下文摘要]`，不直接继续执行用户任务。

模型输入固定按下面顺序组装：

```text
system?
source_messages
user: CONTEXT CHECKPOINT COMPACTION 摘要指令
```

约定：

- `system` 仍来自线程冻结后的 system prompt。
- `source_messages` 会排除原始 `system`。
- 运行中压缩会额外排除当前用户消息，避免摘要模型把当前请求当成待执行任务重跑。
- 当前轮用户消息之后已经发生的模型输出、工具 observation、失败信息仍可留在 `source_messages` 中，作为摘要证据。
- 摘要指令必须作为最后一条 `user` 消息追加；工具失败 observation 不能成为摘要请求尾消息，否则模型可能继续修复任务而不是生成交接摘要。
- 摘要必须包含 `resume_action: final|continue|retry|ask_user`，作为压缩后当前轮续跑的唯一结构化动作建议；续跑层不从 stdout/stderr、代码或任意文本里提取产物路径来推断完成状态。
- 摘要请求会带 `purpose=compaction_summary`，并在 `progress / llm_request / llm_response / compaction` 事件中携带同一个 `compaction_id`。

## 4. 摘要提交物

每次压缩成功后提交两份结果：

1. `summary_text`
2. `replacement_history`

约定：

- `summary_text` 必须带 `[上下文摘要]` 前缀。
- `summary_text` 最多 20000 字符。
- `replacement_history` 写入摘要消息的 `meta.replacement_history`。
- `replacement_history` 不包含线程 `system`。
- `replacement_history` 不包含运行中的当前用户续跑提示。
- 首尾保留按“归一化交互 block”计算：每侧目标保留最早 2 组、最近 2 组用户/模型交互内容。
- 上述交互内容在实现上每侧最多 4 个 block，每个 block 最多 20000 字符。
- 若压缩时可用的归一化交互 block 本身不足 8 个，实际保留数量会小于目标值；此时 `tail` 允许为 0，不视为拼接 bug。

## 5. 本次继续执行请求

压缩完成后，如果当前智能体轮次还要继续执行，会立即重建本次继续请求的上下文。

请求体按下面顺序组装：

```text
system
retained_head_messages
user: summary_text
retained_tail_messages
current_user_replay_message?
```

其中：

- `retained_head_messages` 是压缩前最早的一小段归一化交互。
- `retained_tail_messages` 是压缩前最近的一小段归一化交互。
- `summary_text` 以 `user` 消息形式进入请求，承载被压缩掉的历史。
- `current_user_replay_message` 只用于运行中续跑，不写入 `replacement_history`。
- `system` 必须保持线程冻结后的原始形态，不把动态产物索引、压缩摘要或当前轮信息追加进去，避免模型提示词缓存被破坏。

当前轮续跑按 `current_turn_progress_state` 分类：

| 状态 | 续跑方式 |
| --- | --- |
| `pending` | 追加原始当前用户消息，必要时按预算裁剪 |
| `tool_succeeded` + `resume_action=final/ask_user` | 追加 `user` final continuation note，`current_user_replay_mode=final_continuation`，要求基于摘要直接最终回复或询问用户，禁止重新调用工具、重写文件或重生产物 |
| `tool_succeeded` + `resume_action=continue/retry/unknown` | 追加 `user` continuation note，`current_user_replay_mode=tool_success_continuation`，要求从最新 observation 和摘要继续；工具成功不等于用户任务完成，若摘要明确还有运行、验证、交付等步骤，需要先完成再 final |
| `tool_failed` | 追加 `user` continuation note，要求从保留的失败 observation 继续修复或解释 |
| `in_progress` | 追加 `user` continuation note，要求从当前进展继续，而不是重新开始 |

重要约束：

- continuation note 语义上是在驱动模型继续执行，但协议层必须是 `user` 消息。
- 不允许把 continuation note 作为 `assistant` 尾消息追加；保留尾部可能已经以 assistant 结束，连续 assistant 尾消息会被部分模型服务拒绝。
- `tool_success_continuation` 会写入最新保留的 tool observation 摘要和结构化 `resume_action`。例如 `write_file` 成功只说明文件已写入，只有摘要要求继续时才继续运行或验证；如果摘要标记 `final` 或明确无剩余待办，续跑会直接 final，避免压缩后反复重画/重写。

## 6. `replacement_history` 规则

保留：

- 压缩摘要消息。
- 最早几段用户/模型交互内容。
- 最近几段用户/模型交互内容。
- 非工具噪音的真实用户/模型文本。

不保留：

- 线程 `system`。
- 当前轮临时续跑消息。
- `tool_response:` 观测包装。
- 工具 observation 原始负载。
- `Assistant issued tool call(s): ...`。
- `results_jsonl`、`duration_ms`、`retryable` 这类原始工具回放字段。
- 仅用于发起工具调用的 assistant 占位消息。
- `tool_calls`、`tool_call_id`、`reasoning_content` 等内部执行字段。

原则：

- 优先保留“模型续跑确实需要”的用户意图和已确认事实。
- 不为了“看起来完整”把工具执行过程重新塞回请求体。
- 不把运行中用于承接当前轮的临时指令持久化为历史事实。

## 7. 后续新请求与历史回放

刷新、重进、hydrate，或压缩之后用户发起下一轮真实请求时：

```text
system
latest replacement_history
压缩之后新增的真实历史
current user?
```

约定：

- 只认最新一份 `replacement_history`。
- 更老的压缩前历史不复活。
- 老数据如果没有 `replacement_history`，继续兼容旧逻辑。

## 8. 与 Codex 的主要对齐

当前压缩后的 `replacement_history` 更接近 Codex：

- 以归一化交互窗口和摘要为主。
- 不再依赖 `zero / current / keep`。
- 不再依赖旧的 reset mode 语义。
- 不再依赖首尾锚点预算、字符级上限、前后锚点重建。

Wunder 当前仍保留的差异：

- 有 `manual / auto_loop / overflow_recovery`。
- 有独立的 `context_guard`。
- 有 fresh memory 注入和 compaction 事件诊断字段。
- 有运行中当前轮进展识别与 continuation note。

## 9. Context Guard

压缩后若仍超预算，继续进入 `context_guard`。

可能动作：

- 裁摘要。
- 裁当前用户消息或当前轮续跑提示。
- 极端情况下进一步收缩到最小可运行集。

最终原则：

- 下一次请求和历史回放必须基于同一份已提交基线。
- 不能出现“请求体是一套、刷新回放又是另一套”。

## 10. 边界情况

### 10.1 用户一次输入超长文本

- 允许直接触发压缩。
- 压缩后仍超限则继续走 `context_guard`。
- 禁止落库空摘要、坏摘要、半截摘要前缀。

### 10.2 用户一次输入大量图片或多模态内容

- 当前用户消息仍作为本轮真实输入或 pending 续跑输入保留。
- 压缩基线不再依赖旧的 reset mode 语义。

### 10.3 智能体运行中读取大文件

- 运行中压缩优先保留摘要、首尾交互和当前轮续跑指令。
- 不回放大段工具输出。

### 10.4 工具已经成功

- 压缩后追加 `tool_succeeded` continuation note。
- 模型应基于摘要里的 `resume_action` 继续：`final/ask_user` 直接响应，`continue` 才执行明确剩余步骤；系统不通过解析工具 stdout/stderr、代码文本或动态产物索引来猜测产物是否完成。

### 10.5 工具失败或补丁失败

- 压缩后追加 `tool_failed` continuation note。
- 模型应从保留的失败 observation 改策略继续修复或解释，不应从原始用户请求重新开始。

### 10.6 连续多次压缩

- 只认最新 `replacement_history`。
- 老基线不复活。

### 10.7 摘要失败

- 先尝试 fallback summary。
- fallback 仍无效则不提交坏摘要。
- 必要时只做 guard，不提交污染基线。

## 11. 调试时看什么

重点看 4 处：

1. 摘要阶段 `llm_request`
   - 应满足：`system? + source_messages + user: compaction instruction`。
   - 最后一条必须是摘要指令，不应是工具 observation。
2. `chat.compaction.event`
   - 看 `compaction_id`、`summary_text`、`replacement_history_*`。
   - 看 `source_interaction_block_count` / `source_interaction_messages_debug`，确认压缩输入里一共还有多少可保留交互块。
   - 看 `retained_head_block_count_target` / `retained_tail_block_count_target`。
   - 看 `retained_head_exchange_count_target` / `retained_tail_exchange_count_target`。
   - 看 `retained_head_messages_debug` / `retained_tail_messages_debug`。
   - 看 `current_turn_progress_state`、`current_user_replay_mode`、`current_turn_has_tool_success`、`current_turn_has_tool_failure`。
   - 看 `rebuilt_request_debug`，确认本次继续执行请求尾部没有连续 assistant。
3. 摘要消息 `meta`
   - 看 `meta.replacement_history`。
   - 确认其中没有运行中的 continuation note。
4. 压缩后的第一条普通 `chat.llm_request`
   - 运行中继续执行时应满足：`system + retained head + summary + retained tail + current_user_replay_message`。
   - 新一轮请求或刷新回放后应满足：`system + replacement_history + 压缩后真实历史 + current user`。
