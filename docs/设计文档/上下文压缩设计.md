# 上下文压缩设计

## 1. 一句话

上下文压缩不是“删历史”，而是把旧上下文重组为更短、可续接、可落库的结构。

当前系统区分两份结果：

- 运行时请求结构：发给主模型继续对话
- 持久化回放结构：写入历史，供后续重放

## 2. 三种触发模式

| 模式 | `trigger_mode` | 触发场景 | 当前策略 |
| --- | --- | --- | --- |
| 手动压缩 | `manual` | 用户主动点压缩接口 | 可读性优先，保留前后锚点 |
| 自动压缩 | `auto_loop` | 智能体循环中预检触发 | `summary-first`，优先保摘要 |
| 溢出恢复 | `overflow_recovery` | 主模型报上下文过长后重试 | 最激进的 `summary-first` |

说明：

- 前端只应把 `manual` 视为“手动压缩分割线”。
- `auto_loop` / `overflow_recovery` 属于智能体运行中的自动行为。

## 3. 当前请求体结构

### 3.1 摘要子请求

发给摘要模型的消息结构：

```text
system(冻结 system + artifact index)
history(normalized)
history(normalized)
...
user(compaction instruction)
```

约定：

- observation 不原样透传，会转成短文本
- tool calls 不原样透传，会转成工具摘要
- 多条 system 会合并
- 摘要阶段关闭 reasoning

### 3.2 压缩后继续发给主模型

手动压缩默认结构：

```text
system
head(前 2 轮锚点)
summary
middle(仅 keep 模式、仅运行时)
tail(后 2 轮锚点)
current user
```

自动压缩 / 溢出恢复默认结构：

```text
system
summary
current user
```

如果预算允许，才会附带少量边缘消息；默认不再优先保留 head/tail。

### 3.3 持久化回放结构

历史长期落库统一为：

```text
head -> summary -> tail
```

说明：

- `keep` 模式的 middle window 只存在于“当前这次压缩后立即继续运行”的请求里
- 重新加载历史时，不会长期回放 middle window

## 4. 核心约定

### 4.1 摘要提交规则

- 模型原始输出不直接落库
- 先转成“可提交摘要”再写入历史
- 空摘要、占位摘要、半截摘要禁止落库
- 不允许把 `[上下文`、`...(`、`...(truncated)` 这类坏摘要写进历史

### 4.2 长度预算

- 摘要上限：`20000` 字符
- `head + tail` 合计上限：`20000` 字符
- 当前用户问题单独保留，不占上面两个预算
- 上述上限按字符理解，不按 token 理解
- 仍同时受模型 token 窗口约束

### 4.3 历史重放与再次压缩

- replay 只认 `head -> summary -> tail`
- 再次压缩时，不把旧摘要当普通用户消息再压一次
- fallback 摘要也会跳过已有摘要项，避免“摘要套摘要”

## 5. 不同模式的保留策略

### 手动压缩

目标：可读性、可调试、可人工续接。

特点：

- 默认保留前 2 轮和后 2 轮锚点
- `keep` 模式允许保留运行时 middle recent-user window
- 更适合前端展示“压缩前后发生了什么”

### 自动压缩

目标：让智能体继续跑，不因大文件/大结果把真正有价值的信息挤掉。

特点：

- `summary-first`
- 默认不优先保留 head/tail
- 默认不保留 recent-user window
- 若预算冲突，先牺牲边缘历史，再考虑其它内容
- 重点是保住“已提交摘要 + 当前问题”

### 溢出恢复

目标：先让下一次请求必过，再谈可读性。

特点：

- 走 `summary-first`
- 策略比普通自动压缩更激进
- 用于主模型已明确报上下文过长后的恢复重试

## 6. 压缩后如果还超限

当前处理顺序：

1. 先裁 observation / 大消息
2. 再裁边缘保留消息
3. `summary-first` 模式下，收缩到 `system + summary + current user`
4. 仍超限时再裁当前问题或做最终 budget trim

重要约定：

- 自动压缩和溢出恢复会尽量保住摘要
- 手动压缩更偏可读性，不强保摘要优先级高于一切
- 任何模式都不能把摘要裁成坏片段继续发给模型

## 7. 边界情况

| 情况 | 当前处理 |
| --- | --- |
| 用户一次传入超长文本 | 允许进入压缩链路；必要时 context guard 继续裁当前问题 |
| 用户一次传入大附件/多模态 | `zero` 会提升为 `current` 语义，避免当前多模态输入直接丢失 |
| observation / tool payload 很大 | 不原样保留，只保留摘要 |
| 历史非常长 | 先做摘要，再按字符预算裁 head/tail |
| 摘要模型失败 | 走 fallback summary，不中断主流程 |
| 摘要为空或无效 | 走 fallback summary；若仍无效，本次压缩不提交坏摘要 |
| 连续多次压缩 | 只认最新压缩边界，不复活旧轮次 |
| 自动压缩后再次自动压缩 | 不把旧摘要再压成新摘要内容 |
| 预算极紧 | `summary-first` 模式先退化为 `system + summary + current user` |

## 8. 关键元数据

压缩摘要历史项会写入：

- `type=compaction_summary`
- `trigger_mode`
- `reset_mode`
- `compacted_until_ts`
- `compacted_until`
- `retained_head_until_index`
- `retained_tail_from_index`

说明：

- `retained_head_until_index` / `retained_tail_from_index` 用于恢复 `head -> summary -> tail`
- 自动压缩在极端 `summary-first` 情况下，tail 可能为空

## 9. 调试看哪里

看摘要子请求：

- `llm_request`
- `purpose=compaction_summary`

看压缩后的主请求：

- `compaction`
- `context_usage`
- 下一次主模型 `llm_request`

重点字段：

- `trigger_mode`
- `summary_text`
- `summary_chars`
- `summary_chars_limit`
- `summary_priority`
- `retained_edge_turn_limit`
- `retained_head_turns`
- `retained_tail_turns`
- `recent_user_messages_retained`
- `context_tokens_after`
- `projected_request_tokens_after`

异常判断：

- 如果看到 `[上下文`、`...(`、`...(truncated)` 这类坏摘要参与后续请求，属于实现异常，不是目标行为
