# 上下文压缩设计

## 1. 一句话

上下文压缩不是“删历史”，而是把旧上下文重排成更短、更稳的结构：

```text
system -> 前 2 轮锚点 -> summary -> (keep 的中间窗口) -> 后 2 轮锚点 -> 当前问题
```

注意：

- 上面是“压缩后继续发给主模型”的运行时结构。
- 长期持久化回放不是这个结构，而是：

```text
head -> summary -> tail
```

## 2. 两类触发

| 场景 | 入口 | 压缩后是否继续调用主模型 |
| --- | --- | --- |
| 主动触发 | `POST /wunder/chat/sessions/{session_id}/compaction` | 否 |
| 运行中自动触发 | `execute.rs` 调用 `maybe_compact_messages()` | 是 |
| 上下文溢出恢复 | `call_llm` 报上下文过长后再次压缩 | 是 |

## 3. 两份请求体

### 3.1 摘要子请求：发给“摘要模型”

结构固定为：

```json
[
  { "role": "system", "content": "<system + artifact index>" },
  { "role": "user|assistant", "content": "<normalized history>" },
  { "role": "user|assistant", "content": "<normalized history>" },
  { "role": "user", "content": "<compaction instruction>" }
]
```

约定：

- 最后一条不是用户原问题，而是压缩专用指令。
- observation 不原样透传，会转成短文本。
- tool_calls 不原样透传，会转成工具摘要。
- 多条 system 会合并。
- summary 阶段禁用 reasoning。

### 3.2 自动压缩后：继续发给主模型

结构固定为：

```json
[
  { "role": "system", "content": "<frozen system + artifact index>" },
  "...head anchors...",
  { "role": "user", "content": "<compaction prefix + summary [+ fresh memory]>" },
  "...middle recent-user window, only for keep...",
  "...tail anchors...",
  { "role": "user", "content": "<current question>" }
]
```

稳定顺序：

1. `system`
2. `head`
3. `summary`
4. `middle`
5. `tail`
6. `current user`

### 3.3 手动压缩后：长期回放

手动压缩内部也会重建一份临时上下文做统计，但长期落库只保留：

```text
head -> summary -> tail
```

所以：

- `keep` 的中间窗口只存在于“压缩后的即时运行时上下文”里。
- 重新加载历史时，不会看到 `middle` 被长期保留下来。

## 4. 核心约定

### 4.1 不变项

- 线程首次确定后的 system prompt 保持冻结。
- 压缩摘要统一写成一条 `role=user` 消息。
- 压缩摘要必须带 `history.compaction_prefix`。
- fresh memory 追加到 summary 文本内部，不单独占一个 role。

### 4.2 首尾锚点

- 固定保留最开始 2 轮用户轮次。
- 固定保留最近 2 轮用户轮次。
- “轮次”按用户消息起点切，不按 message 条数切。
- 不足 4 轮时不补空、不重复。
- 这 4 轮保留的是“轮次骨架”，不是无限保真；超长时在轮内裁剪，但不能直接退化成无语义占位符。

### 4.3 长度预算

- 压缩摘要 `summary_text` 上限 20000 字。
- `head + tail` 合计上限 20000 字；前 2 轮和后 2 轮共享这一个预算，不是各 20000 字。
- 当前问题单独保留，不占用上面两个预算。
- 这两个上限按“字符数/字数”理解，不按 token 理解。
- 摘要超出 20000 字时，可以裁剪，但最终结果必须仍是可读、可接手的摘要文本。
- 禁止把最终落库或回放的摘要写成 `...(`、`...(truncated)`、空串这类无语义结果。
- 前后 4 轮合计超出 20000 字时，按轮内裁剪，优先保留用户消息、关键 assistant 结论、必要 observation 摘要，仍要保持“前 2 轮 / 后 2 轮”的轮次结构。
- 若模型窗口仍不足以同时容纳 `system + head + summary + tail + current user`，先继续收缩 `head/tail`，再收缩摘要；若再收缩会使摘要失去语义，则本次压缩应判失败，不能把坏摘要持久化到历史。

### 4.4 `history_compaction_reset`

| 模式 | 含义 |
| --- | --- |
| `zero` | 不保留中间窗口，只保留 `head + summary + tail` |
| `current` | 与 `zero` 一样不保留中间窗口，主要用于当前问题保护语义 |
| `keep` | 额外保留 `head` 和 `tail` 之间最近用户窗口；该窗口只存在于运行时，并有独立预算 |

注意：

- 首尾锚点始终存在，不受 `zero/current/keep` 控制。
- `zero/current/keep` 只控制 `summary` 之外的中间窗口。
- `keep` 不会重复回放已经被 `tail` 覆盖的轮次。

### 4.5 历史 replay 与再次压缩

- 历史回放顺序：`head -> summary -> tail`
- 再次压缩候选：`head + tail`
- 不会把旧 summary 当普通消息再压一次，避免“摘要套摘要”。

## 5. 边界情况

| 情况 | 处理 |
| --- | --- |
| 用户一次传入超长文本 | 当前问题会先参与压缩重建；如果仍超预算，context guard 会继续裁剪当前问题与 summary，直到请求可发出 |
| 用户一次传入大量附件或多模态内容 | 当前用户消息含非文本内容时，`zero` 会自动提升为 `current` 语义，避免当前多模态输入直接丢失 |
| observation / tool payload 很大 | 不原样保留，只转短文本或工具摘要 |
| tool 调用很多、参数很长 | 只保留必要工具名与简要信息，不把整段调用载荷塞回上下文 |
| 历史特别长 | 先做摘要；摘要最多 20000 字，`head + tail` 合计最多 20000 字；`middle` 只有 `keep` 才有，而且有独立预算 |
| 摘要模型输出特别长 | 裁剪到 20000 字以内，但必须保持可读，不允许落成 `...(` 这种坏摘要 |
| 首尾 4 轮本身就很长 | 在 `head + tail <= 20000 字` 的前提下做轮内裁剪，仍保留前 2 轮 / 后 2 轮骨架 |
| 会话轮次少于 4 轮 | 只保留实际存在的轮次，不重复、不镜像 |
| 连续多次压缩 | 只认最新摘要边界；早就丢弃的旧轮次不会被“复活” |
| 主动压缩后再次进入会话 | replay 只会看到 `head -> summary -> tail`，不会看到 `keep` 的中间窗口长期残留 |
| 压缩摘要模型失败 | 回退到 fallback summary，而不是整轮中断 |
| 压缩摘要为空 | 回退到 fallback summary |
| 压缩后仍超出预算 | 先收缩 `head/tail`，再收缩摘要；若摘要会退化成无语义文本，则判失败，不持久化坏摘要 |
| 没有可压缩历史 | 返回 `skipped/guard_only`，只做必要的预算保护 |
| system 很多、artifact 很长 | 合并成单个 replay system，避免 system 堆叠 |
| 模型先报上下文过长再恢复 | 记录 limit hint，再次压缩后重试主请求 |

## 6. 持久化元数据

压缩摘要会写入这些关键 meta：

- `type=compaction_summary`
- `compacted_until_ts`
- `compacted_until`
- `retained_head_until_index`
- `retained_tail_from_index`
- `reset_mode`

其中：

- `retained_head_until_index` / `retained_tail_from_index` 用于恢复 `head -> summary -> tail`
- 老摘要没有这两个字段时，回退到旧的 timestamp/index 边界逻辑

## 7. 调试看哪里

### 7.1 看摘要子请求

看 `llm_request` 事件：

- `purpose=compaction_summary`
- `payload.messages`

这才是“真正发给摘要模型”的请求体。

### 7.2 看压缩后主请求形状

看：

- `compaction`
- `context_usage`
- 下一次主模型的 `llm_request`

重点字段：

- `summary_text`
- `retained_head_turns`
- `retained_tail_turns`
- `recent_user_messages_retained`
- `context_tokens_after`
- `projected_request_tokens_after`

额外约定：

- `summary_text` 必须是可读摘要；如果看到 `[上下文摘要] ...(` 这类结果，说明实现异常，不是目标行为。
