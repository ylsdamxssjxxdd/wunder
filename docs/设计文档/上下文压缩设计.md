# 上下文压缩设计

## 1. 目标

- 对齐 Codex：压缩后提交一份新的 `replacement_history`
- 压缩后的下一次真实请求，与刷新后的历史回放，都优先使用这份 `replacement_history`
- 请求体只保留继续任务真正需要的语义，不回放工具噪音

## 2. 触发模式

| 模式 | `trigger_mode` | 场景 |
| --- | --- | --- |
| 手动压缩 | `manual` | 用户主动触发 |
| 运行中压缩 | `auto_loop` | 智能体循环过程中历史过长 |
| 溢出恢复 | `overflow_recovery` | 主请求已触发上下文超限 |

## 3. 压缩提交物

每次压缩成功后提交两份结果：

1. `summary_text`
2. `replacement_history`

约定：

- `summary_text` 必须带 `[上下文摘要]` 前缀
- `summary_text` 最多 20000 字符
- `replacement_history` 写入摘要消息的 `meta.replacement_history`
- `replacement_history` 不包含线程 `system`
- `replacement_history` 不包含运行中的当前用户消息

## 4. 压缩后下一次真实请求

请求体固定按下面顺序组装：

```text
system
replacement_history
current user
```

说明：

- `system` 仍来自线程冻结后的 system prompt
- `replacement_history` 直接使用上一次压缩已提交快照
- `current user` 是本轮真实问题；手动压缩且无新问题时可为空

## 5. `replacement_history` 规则

保留：

- 压缩摘要消息
- 真实用户消息窗口

不保留：

- `tool_response:` 观测包装
- `Assistant issued tool call(s): ...`
- `results_jsonl`、`duration_ms`、`retryable` 这类原始工具回放负载
- 仅用于发起工具调用的 assistant 占位消息
- `tool_calls`、`tool_call_id`、`reasoning_content` 等内部执行字段

原则：

- 优先保留“模型续跑确实需要”的用户意图
- 不为了“看起来完整”把工具执行过程重新塞回请求体

## 6. 与 Codex 的主要对齐

当前压缩后的 `replacement_history` 更接近 Codex：

- 以用户消息窗口为主
- 追加一条摘要消息
- 不再依赖 `zero / current / keep`
- 不再依赖首尾锚点预算、字符级上限、前后锚点重建

Wunder 当前仍保留的差异：

- 有 `manual / auto_loop / overflow_recovery`
- 有独立的 `context_guard`
- 有 fresh memory 注入和 compaction 事件诊断字段

## 7. 历史回放

刷新、重进、hydrate 时：

```text
replacement_history
+ 压缩之后新增的真实历史
```

约定：

- 只认最新一份 `replacement_history`
- 更老的压缩前历史不复活
- 老数据如果没有 `replacement_history`，继续兼容旧逻辑

## 8. Context Guard

压缩后若仍超预算，继续进入 `context_guard`。

可能动作：

- 裁摘要
- 裁当前用户消息
- 极端情况下进一步收缩到最小可运行集

最终原则：

- 下一次请求和历史回放必须基于同一份已提交基线
- 不能出现“请求体是一套、刷新回放又是另一套”

## 9. 边界情况

### 9.1 用户一次输入超长文本

- 允许直接触发压缩
- 压缩后仍超限则继续走 `context_guard`
- 禁止落库空摘要、坏摘要、半截摘要前缀

### 9.2 用户一次输入大量图片或多模态内容

- 当前用户消息仍作为本轮真实输入保留
- 压缩基线不再依赖旧的 reset mode 语义

### 9.3 智能体运行中读取大文件

- 运行中压缩优先保留摘要和真实用户消息
- 不回放大段工具输出

### 9.4 连续多次压缩

- 只认最新 `replacement_history`
- 老基线不复活

### 9.5 摘要失败

- 先尝试 fallback summary
- fallback 仍无效则不提交坏摘要
- 必要时只做 guard，不提交污染基线

## 10. 调试时看什么

重点看 3 处：

1. `chat.compaction.event`
   - 看 `compaction_id`、`summary_text`、`replacement_history_*`
2. 摘要消息 `meta`
   - 看 `meta.replacement_history`
3. 压缩后的第一条普通 `chat.llm.request`
   - 应满足：`system + replacement_history + current user`
