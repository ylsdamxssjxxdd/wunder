# 上下文压缩设计

## 1. 当前结论

- Wunder 现在不再用“两套重建逻辑”处理压缩后的上下文。
- 每次压缩提交时都会生成一份已提交基线：`replacement_history`。
- 压缩后下一次真实请求、刷新后的历史回放，都优先基于这份 `replacement_history`。
- 每次压缩都有唯一 `compaction_id`，会贯穿 `progress / llm_request / llm_response / compaction` 事件，以及落库摘要元数据。

这一步已经对齐 Codex 的核心方向：不是“生成一条摘要就完事”，而是“提交一份新的历史基线”。

## 2. 运行模式

| 模式 | `trigger_mode` | 场景 |
| --- | --- | --- |
| 手动压缩 | `manual` | 用户主动触发 |
| 循环内压缩 | `auto_loop` | 智能体运行中达到预压缩阈值 |
| 溢出恢复 | `overflow_recovery` | 主请求已触发上下文超限后的恢复 |

## 3. 压缩提交物

每次压缩成功后，会提交两份核心结果：

1. `summary_text`
2. `replacement_history`

其中：

- `summary_text` 是可读摘要，带 `[上下文摘要]` 前缀。
- `replacement_history` 是压缩后的已提交历史快照。
- `replacement_history` 会写入压缩摘要消息的 `meta.replacement_history`。
- `replacement_history` 不包含线程 system prompt。
- 运行中压缩时，`replacement_history` 不包含本轮“正在处理中的当前用户消息”。

## 4. 下一次请求体

压缩后的下一次真实模型请求，按下面顺序拼接：

```text
system
replacement_history
current user
```

说明：

- `system` 仍来自线程冻结的 system prompt。
- `replacement_history` 直接来自上一次压缩已提交的快照，不再临时再拼一套近似结构。
- `current user` 是当前这次真实问题；如果是手动压缩且没有当前问题，则这一段为空。

## 5. 历史回放

刷新、重进、hydrate 时：

```text
replacement_history
+ 后续新增的真实历史消息
```

说明：

- 如果历史里存在新的 `replacement_history`，只认最新的一份。
- 旧压缩之前的历史不会被重新复活。
- 如果遇到旧版本摘要，没有 `replacement_history`，仍按旧的 `head -> summary -> tail` 兼容回放。

## 6. 与旧策略的关系

旧逻辑里有 `head / summary / tail / current user` 的现场重建。

现在的关系是：

- `head / tail` 仍可能参与本次压缩阶段的候选裁剪与诊断指标。
- 但压缩提交后，真正对外生效的是 `replacement_history`。
- 后续请求与回放不再重新根据 `retained_head_until_index / retained_tail_from_index` 再拼一遍。

因此：

- `retained_head_* / retained_tail_*` 现在是诊断字段，不再是下一次请求体的唯一依据。
- 这比“固定保留前后两轮”更接近 Codex。

## 7. Reset Mode

`history_compaction_reset` 仍保留：

| 值 | 含义 |
| --- | --- |
| `zero` | 尽量不保留当前问题的持久化中间态 |
| `current` | 保留当前问题继续执行 |
| `keep` | 允许保留更多最近用户窗口 |

补充：

- 若当前用户消息包含非文本内容，`zero` 会在运行时提升为 `current`，避免当前多模态输入被直接跳过。

## 8. Context Guard

压缩后如果仍超预算，会继续进入 `context_guard`。

`context_guard` 可能继续：

- 裁摘要
- 裁中间消息
- 裁当前用户消息
- 在极端预算下移除摘要

因此：

- `replacement_history` 是“提交时最终确认的基线”。
- 它不保证一定保留摘要文本或锚点文本。
- 真正保证的是：下一次请求和回放都认同一份基线。

## 9. 边界情况

### 9.1 用户一次输入超长文本

- 允许直接触发压缩。
- 若压缩后仍超限，继续走 `context_guard`。
- 不会提交空摘要或占位摘要。

### 9.2 用户一次输入大量图片/多模态内容

- `zero` 自动提升为 `current`。
- 避免当前多模态输入在压缩时被错误丢失。

### 9.3 摘要模型失败

- 先尝试 fallback summary。
- fallback 仍无效，则不提交坏摘要。
- 必要时仅做 `guard_only`。

### 9.4 连续多次压缩

- 只认最新 `replacement_history`。
- 旧压缩基线不会复活。
- 后续请求与回放都从最新基线继续。

### 9.5 旧会话 / 旧数据

- 如果旧摘要没有 `replacement_history`，继续走兼容逻辑：
  `head -> summary -> tail`

## 10. 关键字段

### 摘要落库 `meta`

- `type=compaction_summary`
- `compaction_id`
- `trigger_mode`
- `reset_mode`
- `replacement_history`
- `compacted_until_ts`
- `compacted_until`
- `retained_head_until_index`
- `retained_tail_from_index`

### 事件字段

- `compaction_id`
- `replacement_history_message_count`
- `replacement_history_tokens`

## 11. 调试时看什么

看 3 处就够：

1. `chat.compaction.event`
   关注 `compaction_id`、`summary_text`、`replacement_history_*`
2. 压缩摘要消息 `meta`
   关注 `meta.replacement_history`
3. 压缩后的下一次普通 `chat.llm.request`
   应满足：`system + replacement_history + current user`

## 12. 当前与 Codex 的差距

已经对齐：

- 有压缩事务标识 `compaction_id`
- 有已提交基线 `replacement_history`
- 下一次请求与回放都认同一份基线

还没完全对齐：

- 当前 `replacement_history` 的构建策略仍保留 Wunder 现有的压缩候选裁剪习惯
- 还不是 Codex 那种更彻底的“只围绕 replacement history 做 reverse reconstruction / rollback checkpoint”

结论：

当前实现已经从“摘要 + 临时重拼”升级到“摘要 + 已提交基线”，稳定性和可解释性明显更接近 Codex。
