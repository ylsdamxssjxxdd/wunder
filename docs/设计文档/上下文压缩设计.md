# 上下文压缩设计

## 1. 目标

上下文压缩的目标不是“单纯缩短历史”，而是在不打断当前任务的前提下，把会继续影响推理的上下文重排成一个更小、结构更稳定、便于调试的请求体。

当前实现同时服务两类场景：

- 主动触发上下文压缩：用户或管理员显式要求只做压缩，不继续回答新问题。
- 智能体运行中自动触发上下文压缩：在正常模型调用前预压缩，或在模型返回上下文溢出后做恢复压缩，再继续当前这轮推理。

## 2. 触发入口

### 2.1 主动触发

- Chat 入口：`POST /wunder/chat/sessions/{session_id}/compaction`
- 内部实现：`Orchestrator::force_compact_session`
- 代码入口：
  - `src/api/chat.rs`
  - `src/orchestrator/memory.rs`

主动触发会：

1. 读取当前 session 的系统提示词与历史回放。
2. 发起一次“摘要子请求”。
3. 持久化新的压缩摘要与压缩后的上下文。
4. 返回压缩结果与最终上下文统计。

主动触发不会在压缩完成后立刻继续调用主模型回答问题。

### 2.2 智能体运行中自动触发

- 正常预压缩：`execute.rs` 在正式 `call_llm` 之前调用 `maybe_compact_messages`
- 溢出恢复压缩：`call_llm` 返回上下文过长错误后，再次调用 `maybe_compact_messages`
- 代码入口：
  - `src/orchestrator/execute.rs`
  - `src/orchestrator/memory.rs`

自动触发的压缩完成后，会立刻使用压缩后的 `messages` 继续当前这轮主模型调用。

## 3. 压缩时真正发给“摘要模型”的请求结构

上下文压缩本身也是一次模型调用，但它调用的是“摘要子请求”，不是最终回答用户的主请求。

### 3.1 稳定结构

无论是主动触发还是运行中自动触发，摘要子请求在 `messages` 层的稳定结构都是：

```json
[
  { "role": "system", "content": "<session system prompt + artifact index>" },
  { "role": "user|assistant", "content": "<normalized historical message>" },
  { "role": "user|assistant", "content": "<normalized historical message>" },
  { "role": "user", "content": "<compaction instruction>" }
]
```

其中最后一条 `compaction instruction` 不是用户原始问题，而是压缩专用指令，内容由以下信息拼接：

- 压缩提示模板 `prompts/compact_prompt.txt`
- 当前 artifact index
- 当前用户问题或签名
- 对“本轮真实请求不能丢”的约束说明

### 3.2 自动触发与主动触发的差异

#### 自动触发

- `exclude_current_user=true`
- 当前用户问题不会直接参与 head/tail 锚点选择
- 摘要子请求会把当前用户消息位置替换成压缩指令

可理解为：

```json
[
  { "role": "system", "content": "<system>" },
  "...旧历史...",
  { "role": "user", "content": "<压缩指令，替代当前用户问题>" }
]
```

#### 主动触发

- `exclude_current_user=false`
- 不存在“当前还没回答的新问题”
- 压缩指令直接追加到历史尾部

可理解为：

```json
[
  { "role": "system", "content": "<system>" },
  "...当前 session 历史回放...",
  { "role": "user", "content": "<压缩指令>" }
]
```

### 3.3 摘要子请求的约定

- 使用同一模型配置族，但会切换到压缩专用配置：
  - 关闭 reasoning
  - 限制摘要输出上限
  - 不携带工具调用
- observation 与 tool_calls 不会原样透传：
  - observation 会抽成简短文本摘要
  - assistant 的 tool_calls 会抽成工具名摘要
- 多条 system 不会原样堆叠：
  - system prompt 与 artifact index 会合并

因此，调试压缩质量时，不要只看 session 原始历史；要看 `llm_request` 事件里压缩子请求的真实 payload。

## 4. 压缩后重建的上下文结构

压缩子请求成功后，系统不会把“摘要文本”单独存着就结束，而是会重建一个新的上下文。

### 4.1 自动触发后，继续发给主模型的 `messages`

运行中自动触发压缩后，真正继续喂给主模型的 `messages` 结构为：

```json
[
  { "role": "system", "content": "<frozen system prompt + artifact index>" },
  "...head anchors...",
  { "role": "user", "content": "<compaction prefix + summary text [+ fresh memory block]>" },
  "...middle recent-user window (keep 模式才有)...",
  "...tail anchors...",
  { "role": "user", "content": "<current user question>" }
]
```

顺序是稳定契约：

1. `system`
2. 前置锚点 `head`
3. 压缩摘要 `summary`
4. 中间最近用户窗口 `middle`
5. 尾部锚点 `tail`
6. 当前问题 `current user`

只有第 4、6 段会因场景变化而变化：

- `middle` 只在 `history_compaction_reset=keep` 时出现
- `current user` 只在存在当前待回答问题时出现

### 4.2 主动触发后，压缩完成的上下文

主动触发压缩内部也会重建一份临时上下文，用于：

- 计算压缩后的 `final_context_tokens`
- 输出 compaction 事件中的调试统计

这份临时上下文通常为：

```json
[
  { "role": "system", "content": "<frozen system prompt + artifact index>" },
  "...head anchors...",
  { "role": "user", "content": "<compaction prefix + summary text [+ fresh memory block]>" },
  "...middle recent-user window (keep 模式才有)...",
  "...tail anchors..."
]
```

主动触发场景下一般没有单独的 `current user` 尾项，因为没有一条“正在等待继续回答”的新问题。

但要注意：

- 这份临时重建上下文不会直接作为新的长期历史逐条持久化
- 主动压缩真正长期落库的是“压缩摘要 + 摘要边界元数据”
- 因此主动压缩完成后，后续重新加载历史时看到的仍然是 `head -> summary -> tail`
- `keep` 模式的中间 recent-user window 在主动压缩场景下主要体现在 compaction 事件统计里，不会成为后续长期 replay 的一部分

## 5. 历史持久化与后续回放结构

压缩完成后，历史层不会把完整旧历史保留给下次直接重放，而是持久化为：

- 一条压缩摘要记录
- 摘要对应的边界元数据
- 之后的新消息

下次 `load_history_messages()` 回放给 orchestrator 的顺序是：

```json
[
  "...head anchors...",
  { "role": "user", "content": "<compaction prefix + summary text>" },
  "...tail anchors and later messages..."
]
```

也就是：

- 历史回放顺序：`head -> summary -> tail`
- 再次压缩候选顺序：`head + tail`

这里的“再次压缩候选”刻意不再把旧 summary 当普通消息重新塞回去，避免摘要套摘要。

## 6. 当前实现的核心约定

### 6.1 system prompt 冻结

- 线程首次确定后的 system prompt 不会在压缩时被改写
- 压缩只会把 artifact index 合并到 replay 的 system message
- 压缩不会再次注入长期记忆到 system prompt

### 6.2 压缩摘要的角色与前缀

- 压缩摘要统一落成一条 `role=user` 的消息
- 内容必须带 `history.compaction_prefix`
- fresh memory block 会追加到 summary 文本内部，不单独作为新 role

### 6.3 首尾锚点策略

- 固定保留最开始 2 轮用户轮次
- 固定保留最近 2 轮用户轮次
- 这里的“轮次”以用户消息起点切分，不是按单条 message 计数

### 6.4 首尾锚点不是无限保真

首尾锚点是“原始交互锚点”，但不是无限长度原样拷贝。它们会被做以下处理：

- 角色归一化为 `user / assistant`
- system 不进入首尾锚点
- observation 转简短文本
- tool_calls 转工具名摘要
- 单消息 token 上限裁剪
- head / tail 各自还有总预算

这样做是为了避免“保留前后各两轮”本身再次把上下文撑爆。

### 6.5 `history_compaction_reset` 的真实语义

- `zero`
  - 不额外回放中间用户窗口
  - 仍然保留首尾锚点
- `current`
  - 与 `zero` 一样不保留中间窗口
  - 仍然保留首尾锚点
- `keep`
  - 额外保留 head/tail 之间最近用户窗口
  - 不会重复回放已经进入 tail 锚点的轮次

注意：

- 首尾锚点是固定策略，不由 `zero/current/keep` 控制
- `zero/current/keep` 只控制“摘要之外的中间窗口”

### 6.6 非文本当前问题的保护

如果当前用户消息包含非文本内容（例如图片），配置为 `zero` 时会自动提升到 `current` 语义，避免当前多模态输入在压缩后丢失。

### 6.7 摘要元数据契约

压缩摘要持久化时会写入以下关键 meta 字段：

- `type=compaction_summary`
- `compacted_until_ts`
- `compacted_until`
- `retained_head_until_index`
- `retained_tail_from_index`
- `reset_mode`

其中：

- `retained_head_until_index` / `retained_tail_from_index` 用于历史层在回放时恢复 `head -> summary -> tail`
- 老摘要没有这两个字段时，历史层会回退到旧的 timestamp/index 边界逻辑

## 7. 调试时应该看什么

### 7.1 看压缩子请求

关注事件：

- `llm_request`

重点字段：

- `purpose=compaction_summary`
- `payload.messages`

这是“真正发给摘要模型”的请求体。

### 7.2 看压缩后主请求形状

关注事件：

- `compaction`
- `context_usage`

重点字段：

- `summary_text`
- `recent_user_messages_retained`
- `retained_head_turns`
- `retained_tail_turns`
- `context_tokens_after`
- `projected_request_tokens_after`

如果要确认“继续发给主模型的最终上下文形状”，应结合：

1. `compaction` 事件中的统计与摘要文本
2. 下一次主模型 `llm_request` 的真实 `payload.messages`

## 8. 当前测试覆盖

当前回归测试分成三层：

- `src/orchestrator/memory.rs`
  - 首尾锚点选择
  - 排除当前轮时不吞并下一轮消息
  - 超长锚点预算裁剪
- `src/services/history.rs`
  - `head -> summary -> tail` 回放
  - 再压缩候选只取 `head + tail`
  - 多次压缩后不复活早已被抛弃的旧轮次
- `tests/context_overflow_recovery_regression.rs`
  - 真实压缩链路在多轮对话后仍可恢复
  - 回放结构中前 2 轮位于 summary 前，后 2 轮位于 summary 后
  - `keep` 会保留中间窗口，`zero/current` 不会

- 回归测试：
  - `src/orchestrator/memory.rs`
