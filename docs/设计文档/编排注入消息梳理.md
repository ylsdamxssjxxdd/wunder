# 编排注入消息梳理

本文档记录当前编排系统真实注入给母蜂、工蜂的消息格式，并说明本次收敛后的设计边界。

## 1. 注入链路

当前有两条注入链路：

- 母蜂注入：前端在编排页面拼装，入口是 `frontend/src/components/orchestration/OrchestrationWorkbench.vue`，拼装函数是 `frontend/src/components/orchestration/orchestrationPrompting.ts`。
- 工蜂注入：后端在 `agent_swarm` 派发工蜂时拼装，入口在 `src/services/tools.rs`，核心函数是 `src/services/orchestration_context.rs` 的 `build_worker_dispatch_message(...)`。

母蜂和工蜂不是同一处拼装。后续重构如果要彻底统一，应优先把“模型实际收到的注入包”收敛到同一套结构定义。

## 2. 母蜂注入

### 2.1 首轮格式

首轮仅在 `motherPrimerInjected !== true` 时注入 `[编排运行说明]`，并在存在态势时附加 `[态势背景]`。

```text
[编排运行说明]
你是本次编排的母蜂“{{mother_name}}”。
职责：理解当前态势，按需调用 智能体蜂群 工具调度工蜂，汇总工蜂结果与产物，给出最终回复。
连续性：本次编排会沿用同一批主线程推进，不要当作一次性临时会话。
派发规则：给工蜂的消息只写清楚子任务目标、交付要求和判断标准；不要手写产物目录路径，系统会在派发时自动给工蜂注入本轮产物目录和态势。
读取产物：只有确实需要核对产物时，才对当前轮次产物区做一次有界递归列目录，再批量读取相关文件；不要逐个枚举工蜂目录。

当前可调度工蜂如下：
{{worker_directory_lines}}

[态势背景]
{{situation}}
```

### 2.2 后续轮次格式

后续轮次不再注入 `[编排运行说明]`，也不注入 `[本轮编排上下文]`。

当前实际格式：

```text
[态势背景]
{{situation}}
```

如果态势为空，后续轮次可能没有额外注入内容。

### 2.3 母蜂不再负责产物目录

母蜂首轮说明明确要求：

- 母蜂只需要告诉工蜂“做什么、交付什么、判断标准是什么”
- 母蜂不要手写产物目录
- 工蜂产物目录由后端派发时自动注入

这样可以避免母蜂从历史上下文里沿用旧目录、手抄错目录、或把相对路径和绝对路径混在一起。

## 3. 工蜂注入

工蜂注入由后端完成。只要母蜂所在 session 有编排上下文，`agent_swarm` 派发工蜂时会自动叠加工蜂编排注入。

### 3.1 首次派发格式

某个工蜂在当前线程首次收到编排派发时，格式是：

```text
[编排角色]
你是本次编排的工蜂“{{worker_name}}”。
职责：执行母蜂派发给你的子任务，产出结论或文件，并把结果反馈给母蜂。
回复要求：说明核心结论；如果本轮确实写入了文件，再简要说明文件名称与放置位置；如果只是文本结论，不需要声明已产出文件。

[本轮产物目录]
产物目录：{{artifact_path}}
本轮新增或修改的文件只写入此目录；不要写入其他轮次或其他工蜂目录。
提示：只有在确认文件已经写入后，才把它作为“已产出文件”汇报。

[态势背景]
{{situation}}

[SWARM_CONTEXT]
{
  "swarm": {
    "hive_name": "...",
    "mother_agent_name": "..."
  }
}
[/SWARM_CONTEXT]

任务指令：{{mother_original_message}}
```

### 3.2 后续派发格式

同一个工蜂线程后续轮次不再重复 `[编排角色]`。

```text
[本轮产物目录]
产物目录：{{artifact_path}}
本轮新增或修改的文件只写入此目录；不要写入其他轮次或其他工蜂目录。
提示：只有在确认文件已经写入后，才把它作为“已产出文件”汇报。

[态势背景]
{{situation}}

[SWARM_CONTEXT]
{
  "swarm": {
    "hive_name": "...",
    "mother_agent_name": "..."
  }
}
[/SWARM_CONTEXT]

任务指令：{{mother_original_message}}
```

### 3.3 工蜂目录只保留一处

当前已移除工蜂首次说明里的产物目录句子。

目录约束只存在于 `[本轮产物目录]`：

```text
[本轮产物目录]
产物目录：{{artifact_path}}
本轮新增或修改的文件只写入此目录；不要写入其他轮次或其他工蜂目录。
提示：只有在确认文件已经写入后，才把它作为“已产出文件”汇报。
```

这样可以避免首次派发时出现：

- `[编排角色]` 里说一次目录
- `[本轮产物目录]` 里又说一次目录

## 4. 模板使用状态

### 4.1 当前实际使用

- `config/prompts/zh/orchestration/mother_runtime.txt`
- `config/prompts/zh/orchestration/situation_context.txt`
- `config/prompts/zh/orchestration/worker_first_dispatch.txt`
- `config/prompts/zh/orchestration/worker_round_artifacts.txt`

英文同路径模板也已同步收敛：

- `config/prompts/en/orchestration/mother_runtime.txt`
- `config/prompts/en/orchestration/situation_context.txt`
- `config/prompts/en/orchestration/worker_first_dispatch.txt`
- `config/prompts/en/orchestration/worker_round_artifacts.txt`

### 4.2 当前保留但不应作为母蜂每轮注入

- `config/prompts/zh/orchestration/round_artifacts.txt`
- `config/prompts/en/orchestration/round_artifacts.txt`

这类模板当前不应再拼给母蜂每轮消息。它们只能作为后续设计或调试参考，避免把工蜂目录重新塞回母蜂上下文。

### 4.3 当前存在但未实际接入发送链路

- `config/prompts/zh/orchestration/worker_guide.txt`
- `config/prompts/en/orchestration/worker_guide.txt`
- `config/prompts/zh/orchestration/user_message.txt`
- `config/prompts/en/orchestration/user_message.txt`

其中 `worker_guide.txt` 已改为说明“系统会自动注入工蜂角色、态势和产物目录”。但当前前端的 `buildMotherWorkerPrimerGuide(...)` 没有实际调用方。

## 5. 当前仍然存在的问题

### 5.1 母蜂没有显式 `[用户消息]` 块

当前母蜂触发运行主要依赖 `[态势背景]`。

这是当前系统的产品语义选择：编排页面已把右侧栏改成“当前态势 + 开始/下一轮次/分支”模式，不再把普通聊天输入作为主触发源。

如果后续重新引入“用户指令”和“态势”并存，就必须明确两者边界：

- 态势：背景事实、约束和环境
- 用户指令：本轮要做什么

不能再让两者在同一轮里重复表达。

### 5.2 工蜂仍然有 `SWARM_CONTEXT`

工蜂消息除了编排注入，还会有蜂群工具自己的：

```text
[SWARM_CONTEXT]
...
[/SWARM_CONTEXT]
```

这层来自 `src/services/swarm/beeroom.rs`，不是编排模板。

后续如果要进一步压缩上下文，应评估是否把 `SWARM_CONTEXT` 改为更短的一行元信息，或在编排态下省略其中非必要字段。

### 5.3 母蜂与工蜂拼装仍分散在前后端

母蜂由前端拼，工蜂由后端拼，这仍然会带来维护成本。

后续理想方向：

- 母蜂、工蜂都由后端统一生成注入块
- 前端只传 `run_id / round_index / situation / action`
- 所有实际进入模型的提示词都能在后端日志或调试接口中复现

## 6. 当前收敛原则

当前提示词优化遵循以下原则：

- 角色说明只说角色和职责
- 目录说明只出现在 `[本轮产物目录]`
- 态势只出现在 `[态势背景]`
- 母蜂不要替系统转述目录路径
- 工蜂每轮必须拿到明确产物目录
- 能由系统确定的信息，不要求模型手抄

这版不是最终形态，但已经比之前少了主要重复：工蜂首次说明不再重复产物目录，母蜂首轮说明不再要求母蜂手写目录给工蜂。
