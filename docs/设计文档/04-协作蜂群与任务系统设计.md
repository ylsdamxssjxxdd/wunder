# 协作蜂群与任务系统设计

## 1. 设计目标

本册吸收原 `docs/蜂巢协议设计.md` 的正式内容，用于把“蜂群协作运行时”与“蜂群资产封装协议”统一到一个设计母本中。

协作蜂群系统负责把“一个用户目标”拆解为“多个可执行角色与任务实例”，使 wunder 不只是单线程助手，而是可并行协作、可汇总结果、可公开观察、可导入导出的多智能体系统。

## 2. 核心对象

| 对象 | 说明 | 主要目录 |
| --- | --- | --- |
| Hive | 某个业务目标的协作容器 | `src/services/swarm` `src/storage` |
| Mother Agent | Hive 中承担入口和协调职责的主智能体 | `src/services/swarm` |
| Mission / Team Run | 一次实际协作任务实例 | `src/services/swarm` |
| Team Task | Mission 下的分任务单元 | `src/services/swarm` `src/storage` |
| Subagent | 由线程或 mission 派生的子智能体执行单元 | `src/services/subagents.rs` |
| Beeroom | Hive 在实时层的房间别名和投影附着点 | `src/services/beeroom_realtime.rs` |
| HivePack | 蜂群级资产封装单位 | `src/services/hive_pack.rs` |
| WorkerCard | 单工蜂标准资产描述单位 | Hive 导入导出协议对象 |
| SkillPack | 技能级资产描述单位 | Hive 导入导出协议对象 |

## 3. 设计边界

协作蜂群系统必须坚持以下边界：

- Hive 是协作域，不是单智能体线程。
- Mission 是任务实例，不是永久配置对象。
- Beeroom 是投影房间，不是业务真相源。
- 子智能体控制是协作调度能力，不是前端私有行为。
- 协作态和公开态不能反向污染线程冻结 prompt。
- `WorkerCard` 只声明工蜂期望配置，不直接授予运行权限。

## 4. 模块组成

| 模块 | 作用 |
| --- | --- |
| `src/services/swarm` | Team run、任务拆解、汇总、事件和策略 |
| `src/services/hive_pack.rs` | Hive 聚合打包、协议对象、导入导出辅助 |
| `src/services/subagents.rs` | 子智能体派生、输入传递和控制 |
| `src/api/admin_swarm.rs` `src/api/team_runs.rs` | 管理与用户入口 |

## 5. 执行主链

一次标准的蜂群任务应遵循以下主链：

1. 用户在某个 hive 或 agent 入口提交目标。
2. 系统决定是单智能体执行还是创建 mission / team run。
3. 入口 agent 或 mother agent 生成协作策略、任务拆分和参与者。
4. 各子任务以线程或子智能体形式推进执行。
5. 运行过程不断产出 mission events、agent activity 和公开摘要。
6. 结果被汇总到 mission public board，并同步到 hive / beeroom 投影。

## 6. 状态模型

协作蜂群系统至少需要稳定维护三层状态：

- 配置态：hive、agent、角色关系、默认入口等。
- 运行态：team run、task、agent activity、subagent wait。
- 公开态：mission summary、board、beeroom projection、用户侧摘要。

这三层状态不能混写到同一对象里，否则系统很快会重新回到“大表 + 大 payload + 页面特判”的失控状态。

## 7. 蜂巢协议定位

Hive Packaging Protocol, HPP 是 wunder 蜂群资产化的标准协议。它负责定义蜂群如何被封装、传输、导入、导出，但不直接替代运行时治理。

协议中的核心对象为：

- `HivePack`：蜂群级资产封装单位。
- `WorkerCard`：单工蜂资产描述单位。
- `SkillPack`：技能资产描述单位。
- `WorkerCardBundle`：多张工蜂卡的批量传输单位。

形式化表达如下：

- `HivePack = HiveMeta + WorkerCard[] + SkillPack[]`
- `WorkerCard = Metadata + PromptProfile + AbilityProfile + InteractionProfile + RuntimeProfile + HiveRef + Extensions`
- `SkillPack = SkillMeta + SkillEntry + SkillAssets[]`
- `WorkerCardBundle = WorkerCard[]`

## 8. HivePack 封装结构

### 8.1 推荐目录结构

```text
example.hivepack/
├─ hive.yaml
├─ workers/
│  ├─ planner/
│  │  └─ worker-card.json
│  ├─ coder/
│  │  └─ worker-card.json
│  └─ reviewer/
│     └─ worker-card.json
└─ skills/
   ├─ task_breakdown/
   │  ├─ skill.yaml
   │  └─ SKILL.md
   └─ risk_review/
      ├─ skill.yaml
      └─ SKILL.md
```

### 8.2 文件约定

- 每个工蜂目录 `MUST` 包含一份 `worker-card.json`。
- `hive.yaml` 用于描述蜂群级元信息。
- 每个技能目录 `SHOULD` 包含 `skill.yaml`。
- 技能正文、模板、脚本、静态资源 `MAY` 随技能目录一起封装。

## 9. WorkerCard 规范

### 9.1 顶层结构

一个 `WorkerCard` `SHOULD` 包含以下顶层字段：

- `schema_version`
- `kind`
- `metadata`
- `prompt`
- `abilities`
- `interaction`
- `runtime`
- `hive`
- `extensions`

当前版本标识为 `wunder/worker-card@1`，且 `kind` 固定为 `WorkerCard`。

### 9.2 字段语义

| 字段 | 要求 | 说明 |
| --- | --- | --- |
| `metadata.id` | `SHOULD` | 工蜂标识，建议在当前 hive 内唯一 |
| `metadata.name` | `MUST` | 工蜂名称，不能为空 |
| `prompt.extra_prompt` | `SHOULD` | 标准附加提示词字段 |
| `prompt.system_prompt` | `MAY` | 旧版兼容字段，新导出不应再写入 |
| `abilities.tool_names` | `MAY` | 工具声明数组，元素应为字符串且应去重 |
| `abilities.skills` | `MAY` | 技能声明数组，元素应为字符串且应去重 |
| `interaction.preset_questions` | `MAY` | 工蜂预设问题入口 |
| `runtime.approval_mode` | `MUST` | 审批模式，允许值 `suggest / auto_edit / full_auto` |
| `runtime.sandbox_container_id` | `MUST` | 容器编号，整数，取值范围 `1..10` |
| `runtime.is_shared` | `MAY` | 是否共享 |
| `extensions` | `MAY` | 扩展字段容器，未识别字段应透传 |

### 9.3 `preset_questions` 语义

`interaction.preset_questions` 用于声明工蜂的可复用交互入口：

- `MUST` 为字符串数组。
- 每个元素 `SHOULD` 为可直接发起对话的完整短句。
- 元素顺序 `SHOULD` 保留，消费端应按声明顺序展示。
- 协议只定义声明语义，不限定具体界面实现。

### 9.4 示例

```json
{
  "schema_version": "wunder/worker-card@1",
  "kind": "WorkerCard",
  "metadata": {
    "id": "planner",
    "name": "规划工蜂",
    "description": "负责拆解任务、制定执行计划",
    "icon": "",
    "exported_at": "2026-03-13T08:30:00Z"
  },
  "prompt": {
    "extra_prompt": "你负责先理解目标，再给出分步方案。输出时优先使用编号列表，并标出风险点。"
  },
  "abilities": {
    "tool_names": ["read_file", "write_file", "memory_manager"],
    "skills": ["task_breakdown", "risk_review"]
  },
  "interaction": {
    "preset_questions": [
      "先帮我拆解这个任务，并给出三步方案",
      "帮我列出当前方案的主要风险",
      "请根据现有上下文生成执行清单"
    ]
  },
  "runtime": {
    "approval_mode": "auto_edit",
    "sandbox_container_id": 1,
    "is_shared": false
  },
  "hive": {
    "id": "product-team",
    "name": "产品蜂群",
    "description": "面向需求分析与交付协同"
  },
  "extensions": {}
}
```

## 10. WorkerCardBundle、HivePack 与 SkillPack

### 10.1 `WorkerCardBundle`

`WorkerCardBundle` 用于在一个传输对象中承载多张 `WorkerCard`：

- `schema_version`：版本标识，示例 `wunder/worker-card-bundle@1`
- `kind`：固定为 `WorkerCardBundle`
- `items`：`WorkerCard` 数组

### 10.2 `HivePack`

`HivePack` 是蜂群级分发单元，负责组织以下内容：

- 蜂群元信息
- 多个 `WorkerCard`
- 多个 `SkillPack`
- 可选扩展元数据

`workers/*/worker-card.json` 是工蜂标准描述文件。

### 10.3 `SkillPack`

`SkillPack` 是技能级分发单元，通常包含：

- `skill.yaml`：技能元信息与入口声明
- `SKILL.md`：技能说明或模型使用说明
- 可选脚本、模板、数据文件与静态资源

协议只要求 `SkillPack` 可被蜂群封装引用，不强制限定技能内部资源的具体组织方式。

## 11. 校验与安全语义

消费端接收 `WorkerCard` 或 `WorkerCardBundle` 时，`SHOULD` 至少完成以下校验：

- JSON 结构合法。
- `kind` 与 `schema_version` 合法。
- 必填字段存在且类型正确。
- `metadata.name` 非空。
- `approval_mode` 属于允许枚举值。
- `sandbox_container_id` 在允许范围内。
- `tool_names`、`skills`、`preset_questions` 的元素类型正确。
- 可枚举数组完成去重或在消费阶段按去重策略处理。

同时必须明确以下安全语义：

- `WorkerCard` 只声明工蜂期望配置，不直接授予运行权限。
- `abilities.tool_names` 与 `abilities.skills` 表示声明能力，不等同于目标环境的实际可用能力。
- 目标系统必须根据自身权限模型、运行环境与安全策略决定最终可启用的工具、技能与审批级别。
- 未识别字段或扩展字段应保持前向兼容，避免无关字段导致整体对象不可解析。

## 12. 与其他系统的关系

- 与智能体核心：单个 agent 的执行由内核保证，蜂群系统只负责任务编排和汇总。
- 与实时投影：mission public board 和 beeroom 由投影系统发布，不由 swarm 直接控制前端。
- 与工具系统：子智能体控制、线程控制和 A2A 可作为协作工具，但治理仍属于工具系统。
- 与存储系统：team run、task、activity、hive 资产元数据要有清晰的 durable state。

## 13. 验收标准

- 一个复杂目标能够被稳定拆成 mission、task 与参与 agent。
- 协作过程可以被外部观察，而不必深入每个线程内部。
- mission、hive、beeroom 的状态边界清晰，没有循环依赖。
- 子智能体控制具备明确生命周期，而不是临时脚本拼接。
- `HivePack / WorkerCard / SkillPack` 的导入导出语义稳定，协议对象可被安全校验。
