# Desktop / CLI 本地形态设计

## 1. 设计目标

Desktop 与 CLI 是 wunder 的两种本地运行形态。它们的意义不是“给 server 再包两层壳”，而是在复用同一核心语义的前提下，为个人用户和自动化场景提供不同的人机接口。

## 2. 形态定位

| 形态 | 面向对象 | 设计重点 |
| --- | --- | --- |
| Desktop | 普通个人用户、本地执行场景 | 易用、低门槛、可分发、可授权控制本地环境 |
| CLI | 开发者、自动化和脚本场景 | 可脚本化、可批处理、可纳入流水线 |

二者都必须复用 Wunder 的线程运行时、工具治理和实时投影语义，不能因为本地形态而单独生长出另一套智能体系统。

## 3. 模块组成

| 模块 | 作用 | 主要目录 |
| --- | --- | --- |
| Tauri 主体 | 本地桌面运行与桥接入口 | `desktop/tauri` |
| Electron 壳 | 分发、资源与外壳包装 | `desktop/electron` |
| Desktop API | 本地模式裁剪后的接口集合 | `src/api/desktop.rs` `src/api/desktop_lan.rs` |
| 本地前端 | 用户侧主界面复用 | `frontend` |
| CLI 程序 | 命令行入口与 TUI | `wunder-cli` |

## 4. Desktop 架构

Desktop 长期应稳定为“本地前端 + 本地服务 + 本地桥接 + 本地存储”的组合：

- 前端仍使用用户侧统一界面骨架。
- 后端路由使用裁剪后的 desktop router，减少无关管理面暴露。
- 数据库采用 SQLite。
- 桌面控制、浏览器控制、工作区与本地文件能力通过授权后接入。
- Electron 与 Tauri 的职责以“壳层和桥层”为主，不复制业务主链。

## 5. CLI 架构

CLI 的长期定位是“同一运行时的脚本化入口”：

- 复用同一线程、tooling、prompt、recovery 语义。
- 支持终端流式输出、批处理和自动化管线集成。
- 可以作为本地开发和运维的低成本入口。

CLI 不应为了终端交互便利，绕开主运行时或重写一套私有流式协议。

## 6. 设计原则

本地形态必须遵守以下原则：

- 本地形态与服务端共享同一核心语义。
- 本地接口面可以裁剪，但线程、mission、tool、projection 语义不能裁剪错位。
- Desktop 与 CLI 的差异只应体现在壳层、交互层和本地权限接入层。
- 本地模式仍需保留恢复、日志、工作区和审计能力。

## 7. 与其他系统的关系

- 与用户侧前端：Desktop 复用前端主界面，避免出现第二套用户界面体系。
- 与存储系统：Desktop 使用 SQLite，但数据模型应与服务端共享抽象。
- 与工具系统：本地文件、桌面控制、浏览器控制等能力更强，但仍受统一治理。
- 与实时投影：Desktop 和 CLI 都应消费统一投影语义，而不是私有事件结构。

## 8. 当前演进重点

本地形态后续重点应放在：

- 对齐 `eva` 主链切换，避免继续依赖旧运行时事实来源。
- 对齐实时投影系统的 watch、replay 和恢复语义。
- 让 bridge、desktop_lan、CLI 输出都围绕统一 runtime_api 和 stream 语义收口。

## 9. 验收标准

- Desktop 与 CLI 能在不依赖完整 server 的情况下独立运行。
- 二者消费的是同一套线程与实时语义，而不是私有变体。
- 本地模式依然具备工作区、恢复、工具治理和日志能力。
- 外壳层与业务层职责清晰，不把业务逻辑写进打包壳中。
