# 蜂巢广场设计

## 1. 背景与目标

蜂巢广场是用户侧 Messenger 中的共享资产入口，用来把用户已经沉淀好的蜂群配置、工蜂配置和技能目录发布成可复用资产，再被其他用户直接引入自己的环境。

这一能力服务三个目标：

- 降低复用门槛：用户不必手工导出文件、发给别人、再去本地导入。
- 收敛协议入口：蜂群包、工蜂卡、技能包继续复用现有导入导出协议，不引入第二套分享格式。
- 保持运行时干净：广场只负责资产共享与落盘，不直接参与蜂群运行态、调度态和权限提升。

蜂巢广场不是公共执行环境，也不是在线编辑器。它不负责评论、点赞、审核流、版本历史或跨用户实时协作。

## 2. 用户价值与非目标

### 2.1 用户价值

- 用户可以把自己的蜂群包发布到广场，其他用户一键引入整个蜂群。
- 用户可以把单个工蜂发布为工蜂卡，其他用户直接导入为自己的智能体。
- 用户可以把自定义技能目录发布为技能包，其他用户直接落盘到自己的技能根目录。
- 同一个 Messenger 壳内完成浏览、发布、下架和引入，避免跳出当前工作流。

### 2.2 非目标

- 不做跨租户公开发现与审核平台。
- 不做细粒度共享授权，当前发布即对当前可见广场开放。
- 不做在线 diff、版本回滚和协同编辑。
- 不把广场条目变成运行时绑定对象。引入后得到的是新的本地资产快照。

## 3. 资产模型

蜂巢广场当前支持三类资产：

| kind | 来源 | 发布产物 | 引入结果 |
| --- | --- | --- | --- |
| `hive_pack` | 用户蜂群 | `.hivepack` | 调用现有蜂群包导入链路，重建 hive、agent、技能引用 |
| `worker_card` | 用户智能体 | `.worker-card.json` | 解析工蜂卡并创建新的用户智能体 |
| `skill_pack` | 用户自定义技能目录 | `.skill` | 解压到目标用户 skill root |

广场条目元数据统一为 `UserPlazaItemRecord`，核心字段包括：

- `item_id`：广场条目唯一标识。
- `owner_user_id` / `owner_username`：发布者身份。
- `kind` / `source_key`：资产类别与源对象定位键。
- `title` / `summary` / `icon` / `tags`：展示层字段。
- `artifact_filename` / `artifact_path` / `artifact_size_bytes`：实际工件定位与大小。
- `metadata`：不同资产类型的附加信息，例如 hive 名称、agent 名称、skill 根目录等。

## 4. 核心流程

### 4.1 发布

1. 前端选择要发布的蜂群、工蜂或技能。
2. 后端先执行 `inner_visible.sync_user_state`，确保用户侧 worker-card、技能目录等内外可见状态同步。
3. 按资产类型生成可导入工件：
   - `hive_pack` 走 `run_export_job`
   - `worker_card` 走 `build_worker_card`
   - `skill_pack` 走 `create_skill_archive`
4. 工件复制到共享目录，并写入广场元数据。

同一用户、同一 `kind + source_key` 再次发布时，沿用原 `item_id` 覆盖旧工件，避免重复堆积多个逻辑相同的条目。

### 4.2 浏览

浏览接口只返回“元数据存在且工件文件仍在”的条目。这样可以自动屏蔽工件丢失、手工删文件、历史脏数据等异常条目。

列表支持两个基础过滤条件：

- `mine_only=true`
- `kind=...`

其中 `kind` 同时兼容别名：

- `skill` -> `skill_pack`
- `agent` -> `worker_card`
- `swarm` / `beeroom` -> `hive_pack`

### 4.3 引入

引入流程始终面向“当前登录用户的新资产副本”，不会把源用户的对象直接挂接给目标用户。

- `hive_pack`：直接复用现有蜂群包导入链路，生成目标用户的 hive 与 agents。
- `worker_card`：解析卡片内容，按当前用户权限过滤工具，并保留 `silent`、`prefer_mother`、头像、预设问题等配置。
- `skill_pack`：解压到目标用户技能根目录，并清理技能缓存。

### 4.4 下架

只有条目 owner 可以下架。下架会同时删除：

- 广场工件目录
- 对应 meta 记录

非 owner 的删除请求统一按“条目不存在”处理，避免额外泄露权限信息。

## 5. 后端设计

### 5.1 复用 `meta + artifact` 模型

蜂巢广场没有新建专用数据库表，而是复用现有 `meta` + 文件工件模型：

- 元数据存到 `meta`
- 二进制或文本工件存到工作区共享目录

这样做有两个直接收益：

- server 的 PostgreSQL 与 desktop 的 SQLite 都可以沿用现有存储抽象，无需新增表迁移。
- 广场条目天生符合“元数据 + 文件工件”的系统现状，发布与下架成本低。

### 5.2 存储布局

广场工件统一落到：

```text
<workspace_root>/_shared/hive_plaza/items/<item_id>/<artifact_filename>
```

临时导出文件统一落到：

```text
<workspace_root>/_shared/hive_plaza/tmp/
```

共享目录与用户私有工作区分离，避免把单个用户的私有目录直接暴露成共享来源。

### 5.3 鉴权与所有权

- 列表、详情、引入：登录用户可访问。
- 发布：只允许基于当前用户自己的源对象生成条目。
- 删除：只允许 owner 删除。

广场引入不会绕过原有导入器的校验逻辑。蜂群包、工蜂卡、技能包仍然在各自的协议边界内做合法性检查。

## 6. 前端设计

### 6.1 接入位置

蜂巢广场作为 Messenger 新分区接入，而不是另起一个孤立页面壳。这样可以复用：

- 左侧导航与 section 切换
- 中栏选择态
- 主区详情与操作面板
- 用户当前会话环境与缓存体系

### 6.2 状态职责

- `api/plaza.ts`：封装 `/wunder/plaza/items*` 请求。
- `stores/plaza.ts`：承载列表、发布源、加载态、引入态。
- `MessengerHivePlazaPanel.vue`：负责展示、发布和引入操作。

技能包引入成功后，前端需要：

- 失效用户技能缓存
- 广播工具更新事件

否则 UI 可能继续停留在旧的技能列表快照上。

## 7. 可靠性设计

### 7.1 缺失工件自动隐藏

广场条目并不是只有 meta 就算有效。列表和详情都会校验 `artifact_path` 对应文件是否仍然存在；不存在则视为无效条目。

这样可以避免：

- 历史残留 meta 继续展示脏条目
- 手工删工件后 UI 仍然可点可导入
- desktop/server 双端迁移后工件断链

### 7.2 技能归档安全边界

技能包导入只接受 `.zip` / `.skill`，并额外限制：

- 不能包含绝对路径
- 不能包含 `..` 级路径穿越
- 不能直接把文件扔到 skill root 根上，必须带独立顶层目录
- 不能覆盖内置技能目录

这几条约束保证导入动作不会变成任意文件写入入口。

### 7.3 工蜂配置保真

工蜂卡导入时需要保留与蜂群行为直接相关的关键字段，尤其是：

- `silent`
- `prefer_mother`
- `icon`
- `approval_mode`
- `preset_questions`

否则条目在广场中导入后会和发布前的蜂群语义脱节。

### 7.4 HivePack 结果回报兼容

蜂群包导入结果中的 hive id 需要兼容导入器历史报告字段差异。广场层对 `target_hive_id` 与当前的 `hive_id` 都做兼容读取，避免前端因为报告字段漂移拿不到导入后的目标 hive。

### 7.5 缓存失效

技能包引入后除了落盘，还要同步处理两层缓存：

- 后端 `user_tool_manager` skill cache
- 前端用户工具缓存与刷新事件

否则会出现“文件已导入，但能力列表仍显示旧状态”的伪失败体验。

## 8. 测试与验收

本轮补充了蜂巢广场后端集成测试，覆盖以下高风险场景：

- 列表过滤、别名 kind 过滤、`mine_only` 过滤与缺失工件隐藏。
- 技能包跨用户发布与导入闭环。
- 工蜂卡导入后 `silent` / `prefer_mother` 等字段保留。
- 蜂群包导入后 hive id 回报与目标 agents 重建。
- 非 owner 删除拒绝、owner 删除后工件与 meta 同步清理。
- 恶意技能归档中的非法路径拦截。

验收口径：

- 后端接口返回的数据与落盘结果一致。
- 导入后能在目标用户真实存储中看到对应 hive、agent 或 skill 文件。
- 异常归档不会写出非法路径文件。

## 9. 已知限制与后续演进

当前实现仍有几个明确边界：

- 没有引入公开审核、举报、评分、评论等社区能力。
- 没有版本历史与增量 diff，重复发布按同源覆盖。
- 没有下载次数、引入次数等广场统计指标。
- 没有做跨组织可见性配置。

后续若广场继续演进，优先级应是：

1. 条目版本与修订历史。
2. 组织/租户范围的发布可见性控制。
3. 更细的前端回归与端到端操作测试。
