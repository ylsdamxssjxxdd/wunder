# wunder 设计总纲

## 1. 文档定位

`docs/设计文档` 是 wunder 的正式设计母本，用于回答四类问题：

- 系统为什么这样分层，哪些边界不能被短期实现反向侵蚀。
- 新能力应该落在哪个层级、由哪个模块负责、如何验收。
- 多人并行开发时，哪些结构可以快速迭代，哪些结构必须谨慎收口。
- 持续演进里，哪些语义必须保持稳定。

## 2. 系统总述

wunder 是一个面向组织与个人的智能体调度系统，统一承载三种运行形态：

- `server`：多租户、多用户、多渠道、可治理的云端核心。
- `desktop`：个人用户主推的本地桌面形态。
- `cli`：适合自动化和脚本化任务的本地命令行形态。

系统的核心不是一次问答，而是可持续执行、可恢复、可观察、可治理的智能体运行时。围绕这个核心，wunder 的长期特色已经稳定为：

| 特色能力 | 设计含义 | 关键代码域 |
| --- | --- | --- |
| 智能体核心 | 线程、turn、prompt、context、tool、recovery 拥有稳定语义 | `src/orchestrator` `src/services/runtime` |
| 实时事件系统 | 前端和外部系统只消费公开事件流，不直接定义后端真相 | `src/services/stream_events.rs` `src/services/beeroom_realtime.rs` `src/api/*_ws.rs` |
| 蜂群协作与资产化 | hive、mission、WorkerCard、SkillPack 可编排、可导入、可导出 | `src/services/swarm` `src/services/hive_pack` |
| 工具化运行时 | 对开发者来说一切都是接口，对大模型来说一切皆工具 | `src/services/tools` `src/services/skills.rs` `src/services/mcp.rs` |
| 长会话能力 | 冻结 prompt、上下文压缩、一次性记忆注入、恢复链统一治理 | `src/orchestrator/prompt.rs` `src/orchestrator/context.rs` |
| 多入口共核 | HTTP、WS、gateway、channels、desktop、cli 共享同一运行时主链 | `src/api` `src/gateway` `src/channels` `desktop` `wunder-cli` |
| 双存储形态 | 服务端 PostgreSQL，桌面端 SQLite，共享统一存储抽象 | `src/storage` |

## 3. 设计基线

本目录所有分册共同遵守以下基线：

- 以职责边界组织模块，而不是以调用方便组织模块。
- 线程首次确定后的 `system prompt` 必须冻结，后续轮次不得改写。
- 长期记忆只允许在线程初始化时注入一次，不能在中途反复污染线程认知态。
- 实时系统遵循 `Snapshot + Delta + Replay` 契约，前端只消费事件流，不直接重定义后端状态。
- 当前发布边界固定为"一人一会话、各自分散聊天"。
- 运行时治理必须依赖程序结构保证，不能只靠 prompt 约定。
- 多租户、多用户、多连接访问是前提，不是附加特性。
- `server / desktop / cli` 必须共享核心语义，只允许接入面差异，不允许分裂出三套独立系统。
- 原型阶段可以主动删除旧实现，不为低价值兼容性背负长期债务。

## 6. 与其他文档的关系

本目录重点回答"设计定稿"和"模块边界"问题；其他目录承担更细的施工与事实说明职责：

- `docs/方案/*`：阶段性实施路径、拆解方案、过渡计划。
- `docs/API文档.md`：接口、协议、事件与对外约定。
- `docs/技术说明书/*`：部署、启动、排障、扩展和验收手册。
- `docs/使用说明书/*`：面向最终用户和管理员的说明材料。

后续若系统主骨架、模块边界、核心协议或状态模型发生变化，应优先更新本目录，再回补方案、API 与实现文档。
