# 安全、权限与隔离设计

## 1. 设计目标

wunder 不是单机玩具，而是会执行工具、操作文件、访问外部系统、接入渠道和管理多用户资源的智能体平台。安全设计必须内建在系统结构里，而不能只靠提示词或人工约束。

## 2. 安全边界

系统的核心安全边界包括：

- 用户与组织权限边界。
- 线程与工作区访问边界。
- 工具审批与沙盒边界。
- Gateway、渠道与外部接入认证边界。
- 管理员控制面与普通用户操作面边界。

## 3. 安全组成

| 组成 | 说明 | 主要目录 |
| --- | --- | --- |
| 鉴权与用户访问 | 登录、token、用户权限、管理员权限 | `src/api/auth.rs` `src/services/user_access.rs` `src/services/external_auth.rs` |
| 工具审批 | 高风险能力需要显式确认 | `crates/eva/src/tools/approvals.rs` `src/agent_kernel/tools/approvals.rs` |
| 沙盒与权限策略 | 文件、命令、网络等执行边界 | `crates/eva/src/tools/sandbox.rs` `src/sandbox` |
| 工作区隔离 | 容器、目录 fence、路由保护 | `src/services/directory` `src/services/workspace.rs` |
| 网关与渠道安全 | token、角色、路由所有权、限流 | `src/gateway` `src/channels/rate_limit.rs` |
| 配额与审计 | 用户资源限制、日志、调用记录 | `src/storage` `src/services/runtime/metrics.rs` |

## 4. 设计原则

安全设计必须坚持以下原则：

- 最小权限原则：默认不给，按需开放。
- 显式审批原则：高风险动作必须可确认、可拒绝、可追踪。
- 结构约束原则：安全策略由代码结构保证，不依赖 prompt 兜底。
- 审计可追踪原则：关键动作必须能落日志、落状态、可回放。
- 租户隔离原则：多用户和多组织场景下必须明确资源归属。

## 5. 工具安全

工具安全是 Wunder 的核心安全主题，重点包括：

- 工具可见性控制。
- 审批模式控制。
- 沙盒容器与路径边界控制。
- 命令执行、文件修改、浏览器控制等高风险能力控制。
- 动态工具和外部工具的可信边界控制。

任何新工具，如果没有明确 approval、sandbox、timeout、error model，就不应直接接入可生产使用的工具面。

## 6. 身份与归属

系统必须区分以下身份：

- 注册用户身份。
- 线程归属身份。
- 管理员操作身份。
- gateway / channel / node 接入身份。

这四类身份可以关联，但不能混同，否则外部系统嵌入和渠道接入会迅速出现越权和审计失真问题。

## 7. 高风险场景

设计和验收时应重点关注以下高风险场景：

- 模型执行文件删除、命令执行、补丁应用等高危操作。
- 外部渠道消息伪造或越权路由。
- Gateway 路由所有权冲突。
- 工作区路径逃逸和临时目录误用。
- 管理员界面误操作造成系统级配置污染。

## 8. 验收标准

- 高风险工具具备明确审批与沙盒策略。
- 用户、线程、组织和管理员权限边界清楚。
- Gateway 与渠道接入有认证、限流和路由保护。
- 审计日志与关键状态足以定位安全问题来源。
