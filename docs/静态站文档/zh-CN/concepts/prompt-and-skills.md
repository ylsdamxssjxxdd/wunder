# 提示词与技能

在 Wunder 里，提示词和技能都不是“补充说明”，而是系统主链路的一部分。

## 先分清两类东西

### 系统提示词

决定模型在当前线程里怎么理解自己、工具协议、工程约束和记忆上下文。

### 技能

决定模型在当前会话里还能额外读到什么操作手册、资源路径和执行约定。

两者都会影响模型行为，但职责不同。

## 为什么线程提示词必须冻结

Wunder 当前有一条非常重要的约束：

- 线程的 system prompt 在首次确定后必须冻结

这样做不是保守，而是为了稳定性：

- 避免同一线程越聊越漂
- 避免长期记忆反复改写缓存命中
- 降低多轮会话里“前后人格不一致”的问题
- 让问题定位更可重复

所以你要理解：

- 新线程会重新构造 system prompt
- 老线程不会因为你改了模板包就被重写

## 提示词模板包怎么组织

当前提示词按分段文件组织，例如：

- `role`
- `engineering`
- `tools_protocol`
- `skills_protocol`
- `memory`
- `extra`

系统支持多套模板包，由 `prompt_templates.active` 指定当前启用包。

## 改模板后什么时候生效

提示词模板不是热改当前线程，而是优先影响：

- 新建线程
- 新会话
- 新的系统提示词预览结果

已经建立的线程不追写。

如果你改了模板包却发现当前对话没变化，先不要怀疑系统没保存，先确认是不是还在旧线程里。

## 技能是什么

技能可以理解成“面向模型的结构化操作说明包”。

它通常包含：

- `SKILL.md`
- 脚本
- 示例文件
- 资源目录

技能被启用后，模型会拿到技能协议和技能入口信息，需要时还可以通过 `skill_call` 读取完整技能内容。

## 技能分层

Wunder 当前把技能分成三层：

- `builtin`
- `custom`
- `external`

这背后的意义是治理边界：

- 内置技能默认只读
- 用户和管理员上传的内容写入 custom 目录
- external 更像外部引入能力

所以“技能能不能改”并不是统一答案，要看它属于哪一层。

## `skill_call` 为什么重要

很多系统只给模型一个技能名，但不给完整技能文档，这会导致模型反复猜。

Wunder 内置的 `skill_call` 则是显式能力：

- 按技能名返回完整 `SKILL.md`
- 返回技能目录结构
- 不对结果做过度裁剪

这使得技能从“提示词里的一个名字”变成“模型可以主动读取的操作手册”。

## `{{SKILL_ROOT}}` 是干什么的

技能文档里的资源路径统一建议写成：

- `{{SKILL_ROOT}}`

原因很实际：

- 技能可能在 server、desktop、cli 下落在不同真实路径
- 如果直接写死本地路径，迁移后就会失效

`skill_call` 返回技能内容时，会把这个占位符替换成模型当前可见的绝对路径。

## 提示词、技能、记忆三者怎么配合

可以这样理解：

- 系统提示词定义长期稳定边界
- 技能定义某类任务的附加执行协议
- 记忆定义当前线程初始化时带入的长期事实快照

三者都重要，但不要混用职责。

## 常见误解

### 误解一：改提示词模板会立即改变当前线程

不会。

它优先影响的是新线程。

### 误解二：技能只是一个说明文档

不准确。

在 Wunder 里，技能是模型可以显式读取的运行资源。

### 误解三：记忆会每轮自动重写 system prompt

不会。

当前约定是初始化注入一次，之后线程提示词保持冻结。
