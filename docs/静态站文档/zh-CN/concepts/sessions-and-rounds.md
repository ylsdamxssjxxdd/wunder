# 会话与轮次

wunder 不是把一次对话看成单一请求，而是看成一条持续演进的会话。

## 会话是什么

一个会话通常包含：

- 用户输入
- 模型响应
- 工具调用
- 中间事件
- 最终结果

在复杂任务中，一个会话可能持续很多轮，并包含多次模型调用和工具调用。

## 线程是什么

线程是会话的更高一层组织方式。

当前系统要求：

- 智能体线程的 system prompt 在首次确定后必须冻结
- 长期记忆只允许在线程初始化时注入一次
- 后续轮次继续复用同一条线程上下文

所以“线程”不是一个 UI 标签，而是系统稳定性和上下文一致性的边界。

## 用户轮次与模型轮次

Wunder 明确把轮次拆成两层。

### 用户轮次

用户每发送一条消息，记 1 轮用户轮次。

它代表一次明确的业务输入边界。

### 模型轮次

模型每执行一次动作，记 1 轮模型轮次。

动作包括：

- 一次模型调用
- 一次工具调用
- 一次最终回复

所以一次用户轮次里，通常会包含多轮模型轮次。

## 为什么这个区分重要

如果不把两种轮次拆开，就很容易把这些事情混在一起：

- 用户到底问了几次
- 模型为了回答这一次问题走了几步
- 某次工具爆炸发生在第几步
- 当前速度、耗时和 token 统计应该挂到哪一层

把用户轮次和模型轮次分开后，时间线、回放、告警和计费口径都会清楚很多。

## token 统计口径

Wunder 记录的是 token 占用量，也就是当前上下文实际占了多少 token。

它不是：

- 平台账单总消耗
- 某个供应商返回的所有计费字段简单累加

你在看聊天页、调试页或回放记录时，应该优先把 token 理解为“当前线程上下文负载”。

## 流式事件与终结

一次用户轮次的执行过程，通常会在事件流里看到：

- 输入进入
- 模型开始工作
- 工具调用与观察结果
- 中间增量输出
- 终结事件

当前链路里，是否真正结束应以终结语义为准，而不是只看有没有流式文本。

如果你在做接入或排障，至少要分清：

- 会话是否还活着
- 当前线程 runtime 是否还在 active
- 当前这轮是否已经发出终结事件
