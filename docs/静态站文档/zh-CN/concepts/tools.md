# 工具体系

Wunder 的核心不是“模型回答”，而是“模型调度工具完成任务”。

## 这页解决什么

- Wunder 里的工具到底从哪里来
- 为什么不同来源的工具还能统一调度
- 为什么同一会话里工具可见性会变化

## 先记这几条

- 对模型来说，能力尽量统一成工具，而不是散落在不同隐式机制里。
- 对开发者来说，很多系统能力最终也要落成工具，才能被编排、治理和观察。
- 工具来源可以不同，但调度层、展示层和权限层尽量统一。

## 工具从哪里来

当前至少包括这些来源：

- 内置工具
- MCP 工具
- Skills
- 知识库工具
- 用户工具

## 内置工具先看什么

内置工具承载最基础、最稳定的执行面。

高频例子包括：

- 文件与工作区：读取文件、写入文件、应用补丁、搜索内容、列出文件
- 运行态：执行命令、`ptc`、休眠等待
- 网页与桌面：`web_fetch`、浏览器、桌面控制
- 会话与协作：线程控制、子智能体控制、智能体蜂群、记忆管理

如果你现在就想知道某个工具怎么选，直接去看 [工具总览](/docs/zh-CN/tools/)。

## MCP 负责什么

MCP 负责把 Wunder 之外的系统能力接进来。

它不是替代内置工具，而是把额外服务纳入统一调度，比如：

- 数据库
- 外部服务
- 第三方系统桥接

## Skills 负责什么

Skill 更像流程化经验包。

它封装的通常不是“一个动作”，而是：

- 一套方法
- 一组步骤
- 一批约束
- 某类任务最佳实践

## 知识库和用户工具分别解决什么

- 知识库提供的是受控上下文来源，适合制度、长文档和领域资料。
- 用户工具允许不同用户按自己的工作场景扩展能力，但仍在系统治理边界内运行。

## 为什么一定要统一工具体系

如果不同来源的工具各自为政，会出现这些问题：

- 模型不知道该优先调用哪一类能力
- 前端无法统一展示和搜索
- 权限、审计和治理被打散
- 排障时很难看清到底是哪一层出了问题

Wunder 的做法是：来源不同，但调度和展示尽量统一。

## 最容易搞错的点

- `web_fetch`、浏览器、桌面控制都算工具，但它们不是同一种能力边界。
- Skill 不是工具别名，它更像“模型可读的操作手册”。
- 工具是否可见，不只由代码决定，还受运行形态、配置和会话上下文影响。
