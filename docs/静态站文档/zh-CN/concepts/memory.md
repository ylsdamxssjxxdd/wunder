# 长期记忆

wunder 的长期记忆设计不是“每轮都往 system prompt 里追加内容”。

相反，它非常强调线程稳定性。

## 核心规则

当前系统有两条必须遵守的约束：

1. 智能体线程的 system prompt 一旦首次确定后必须冻结
2. 长期记忆只允许在线程初始化时注入一次

后续轮次不能反复改写线程 system prompt。

## 为什么这样设计

原因很直接：

- 避免破坏提示词缓存
- 降低上下文漂移
- 保证线程行为更稳定
- 让“这一条线程到底基于什么提示词运行”可追踪

## 记忆的形态

当前长期记忆采用结构化记忆碎片思路。

它不是一大段自由文本，而是更适合：

- 命中
- 替代
- 失效
- 追踪版本关系

的结构化记录。

## 记忆与 recall

线程初始化时会注入一部分记忆。

如果当前注入不足，调度层和工具层可以再通过 recall 类能力继续补充检索，而不是反复重写线程 system prompt。

## 记忆与上下文压缩

压缩发生后，系统仍然需要解决“记忆是否足够”的问题。

因此应把“线程冻结的初始注入”与“压缩后的补充 recall”区分开看。

## 理解这套机制时最容易犯的错

- 把长期记忆理解成每轮动态 system prompt
- 把 recall 结果理解成永久写回线程提示词
- 把 token 占用理解成总账单
