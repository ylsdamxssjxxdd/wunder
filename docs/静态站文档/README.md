# 静态站文档目录说明

本目录用于存放 wunder 正式静态文档站的内容源文件。

## 目录约定

- 正式文档统一放在 `docs/静态站文档/`
- 中文文档放在 `docs/静态站文档/zh-CN/`
- 后续如需英文文档，放在 `docs/静态站文档/en-US/`
- `docs/方案/` 继续作为内部方案与 RFC，不直接作为静态站主内容
- `docs/wunder补全文档落地方案.md` 作为实施方案文档保留在 `docs/` 根目录

## 页面风格

静态站文档的布局、内容形式、说法方式尽量参考 OpenClaw 文档工程：

- 目录分层清晰，按 `start / concepts / tools / integration / help / reference` 组织
- 单篇页面只解决一个核心问题，避免写成超长总设计文档
- 页面开头先给出简短 summary，再进入正文
- 多使用“适用场景 / 操作步骤 / 相关文档”这种直读结构
- FAQ、troubleshooting、hub 页面单独组织，不把所有说明堆到首页

## 写法约定

- 先写“这页解决什么”，不要一上来讲大背景。
- 标题尽量使用 OpenClaw 风格的小节：`是什么 / 先看这些 / 按问题找页面 / 最容易搞错的点 / 相关文档`。
- 二级高频页面优先按 `这页解决什么 / 先记结论 / 一条典型链路 / 最容易搞错的点 / 相关文档` 展开。
- 工具页优先写“什么时候先用这个工具”，并明确补出英文工具别名与关键配置键。
- 句子尽量短，先给判断，再给解释。
- 多写“什么时候用这页、什么时候不要看这页”，少写泛泛而谈的系统口号。
- 列表项尽量直给，不要一条里塞太多并列概念。
- 工具页、接入页、运维页优先写“怎么判断、怎么选择、怎么排查”，而不是长篇抽象说明。
- 页面里如果出现“体系、能力、协同关系”这类词，尽量继续往下具体化，不要停在抽象层。

## 推荐模板

- 入口页：一句话判断这页给谁看，再写 `先看这些 / 按问题找页面 / 最容易搞错的点 / 相关文档`
- 接入页：先写“什么时候该接这个接口”，再写最短链路、关键字段和误区
- 运维页：先写部署判断，再写检查顺序，不要一上来堆环境变量

## Frontmatter 约定

- 可选使用 `updated_at: YYYY-MM-DD` 指定页面顶部“最后更新”日期
- 如果不写 `updated_at`，静态站生成脚本会默认使用源文件修改时间

## 不建议放入本目录的内容

- 临时讨论稿
- 一次性实施清单
- 对标分析原稿
- 演示脚本与视频提示词
- 与静态站无关的内部备忘
