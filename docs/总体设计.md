# 总体设计

11 个核心

| 核心 | 设计目标 | 核心能力 | 关键约束 |
| --- | --- | --- | --- |
| 智能体循环 | 让智能体能够稳定执行“思考-行动-观察-继续”的连续回合 | 线程状态机、模型调用、工具调用、重试治理、终态收敛、恢复续跑 | 线程语义必须稳定；不能因为前端展示或临时逻辑破坏运行时主链路 |
| 工具 | 让模型更容易正确调用能力并可靠消费结果 | 清晰工具描述、结构化参数、统一结果截断、工具工作流展示、失败反馈 | 对大模型来说一切皆工具；工具返回必须精简明确；工具事实与展示投影必须分离 |
| 蜂群 | 支持母蜂调度多个工蜂并完成协作任务 | 任务拆解、工蜂派发、节点状态同步、汇总结果、父子会话关联 | 母蜂与工蜂职责分离；不能把蜂群误当普通子智能体链路；协作过程必须持续可见 |
| 上下文压缩 | 在长会话中控制上下文规模并保留有效信息 | 手动压缩、自动压缩、溢出恢复、压缩摘要回注、压缩前后对比与回放 | 压缩结果必须可追溯；压缩不能伪造事实；压缩过程与终态要可观测 |
| 记忆 | 支持长会话与长期资料利用，但不污染线程核心认知 | 线程初始化记忆注入、知识库、工作区文件、长期资料读取 | 长期记忆只允许在线程初始化注入一次；线程首次确定后的 system prompt 必须冻结 |
| 渠道 | 支持多入口接入同一运行时能力 | HTTP、WebSocket、Desktop、CLI、第三方渠道、网关适配 | 多入口共核，只允许接入面差异，不允许演化成多套独立系统 |
| 定时任务 | 支持系统级周期执行与后台治理 | 定时触发、计划任务、后台巡检、自动维护、异步执行链路 | 定时任务必须与在线线程语义区分；失败重试和执行记录要可追踪 |
| 多用户管理 | 支持组织、用户、租户和权限治理 | 用户体系、单位与租户隔离、权限控制、Token 账户治理、管理后台 | 默认按多用户并发设计；数据隔离、权限边界和治理能力不能后补 |
| 实时性 | 让前端和外部系统及时感知线程与任务变化 | WebSocket 事件流、快照补偿、增量同步、断线重连、回放恢复 | 前端消费事件流但不定义后端真相；实时展示允许投影但不能篡改状态语义 |
| 稳定性 | 让系统在长会话、高并发、多工具、多入口场景下持续可运行 | 错误隔离、超时与重试、恢复续跑、资源治理、线程终态收敛、回归验收 | 稳定性依赖程序结构保障，不能只靠 prompt 和人工操作；必须优先控制高风险链路的故障扩散 |
| 可观测性 | 让系统能解释“发生了什么、为什么、如何复盘” | 线程事实流、时间线回放、管理员画像、吞吐评测、调试导出 | 区分事实层、回放层、画像层；区分请求、结果、观察结果；指标口径统一 |

## 总体原则

| 原则 | 说明 |
| --- | --- |
| 一切围绕线程 | 会话、工具、压缩、蜂群、回放最终都要落回线程语义上统一治理 |
| 一切围绕事件 | 实时同步、排障、回放、监控都应建立在事件与状态变更之上 |
| 一切围绕约束 | prompt 冻结、记忆一次性注入、指标口径统一、事实与画像分离都是硬约束 |
| 一切围绕性能 | 高并发访问、长会话、工具链执行和多前端同步必须优先考虑速度与资源成本 |
