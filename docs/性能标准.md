# 性能标准

本文定义 wunder 的性能优化验收标准。任何以“性能、速度、吞吐、卡顿、内存、IO、并发、启动耗时、渲染耗时”为目标的改动，都必须按本文留下可复查的前后对比，避免主观判断和后续回退。

## 核心原则

- 性能优化必须有基线、有对照、有结论。没有前后数据，不在功能迭代中描述为“性能提升”，只能描述为“结构调整”或“待验证”。
- 对比必须使用同一环境、同一配置、同一数据规模、同一入口和同一操作步骤。环境变了，只能建立新基线，不能直接和旧结果比较。
- 测试必须贴近真实运行方式。Docker 文件操作、工作区 IO、数据库、浏览器渲染、WebSocket 长链路等场景，不得为了跑分切到非真实路径、临时挂载或简化链路。
- 性能结果要看延迟、吞吐、错误率、资源占用和队列背压，不能只看单个平均耗时。
- 噪声较大的本地测试至少重复 3 次，取中位数；明显异常的一次必须说明原因，不能只挑最好结果。

## 什么时候必须做性能对比

以下改动必须提交性能对比记录：

- 明确声称优化性能、吞吐、延迟、内存、CPU、IO、卡顿、启动速度或构建速度。
- 触及高频链路：聊天发送、模型流式输出、工具执行、文件工具、工作区扫描、WebSocket、实时投影、会话列表、消息渲染、管理端监控。
- 触及运行时边界：blocking、bounded queue、long task、cron、channel outbox、thread runtime、mission runtime。
- 触及存储热路径：会话列表、消息分页、工具日志、monitor、benchmark、用户/单位列表、workspace 文件列表。
- 触及 Docker、部署目录、卷挂载、缓存目录、构建产物目录、前端依赖目录等运行环境。
- 触及用户侧前端大页面、长列表、Markdown 渲染、虚拟列表、hover/scroll 高频交互。

## 标准工具与场景

| 场景 | 标准入口 | 必看指标 |
| --- | --- | --- |
| 管理端性能采样 | 管理端性能测试页或 `/wunder/admin/performance/sample` | `prompt_build`、`file_ops`、`command_exec`、`tool_access`、`vector_flow`、`log_write`；`file_ops` 必须看 list/write/read/search/patch 子步骤 |
| 系统吞吐 | 管理端吞吐压测或 `/wunder/admin/throughput/*` | 成功率、RPS、平均延迟、p95/p99、最大并发、失败原因 |
| 运行时边界 | `python scripts/runtime_boundary_stress.py ...` | blocking 等待/执行耗时、队列深度、long task 耗时、告警数量、热点 label |
| 后端代码正确性 | 定向 `cargo test -j 8 ...` + `cargo check -j 8 -p <crate>` | 错误、告警、关键回归测试是否通过 |
| 用户侧前端交互 | 对应 Playwright/回归脚本、浏览器 Performance trace | 长任务、首屏可交互时间、滚动/hover 帧耗时、接口重复请求、内存峰值 |
| 模型能力质量 | WunderBench | 任务得分、完成率、工具成功率、稳定性；注意这不是吞吐压测 |

## 首批可建立的标准性能

按当前代码和入口，建议把标准性能分成三档：

### P0 立即可建

- 管理端性能采样基线：`prompt_build`、`file_ops`、`command_exec`、`tool_access`、`vector_flow`、`log_write`。
- 系统吞吐基线：`/wunder/admin/throughput/*`，看成功率、RPS、平均延迟、p95/p99、失败原因。
- 运行时边界基线：`python scripts/runtime_boundary_stress.py ...`，看 blocking、bounded queue、long task、告警和热点 label。
- 后端宏观回归基线：`cargo check -j 8` + 定向 `cargo test -j 8`，用于守住共享逻辑和关键契约。
- 统一 benchmark 基线：`powershell -ExecutionPolicy Bypass -File scripts\benchmark.ps1 -Label baseline`，默认收口 `backend_sim` 标准场景并生成 `benchmark.json/csv/md`。
- backend_sim 标准回归：由统一 benchmark 入口调用 `python scripts/run_backend_sim_workflow.py baseline|compare`，用于把系统回归固定成可比较的基线。
- WunderBench 质量基线：`quick` / `core` / `full` 三档，守住模型完成质量和工具成功率。

### P1 先补脚本再纳入标准

- 用户侧前端交互基线：首屏可交互、消息渲染、切页恢复、长列表滚动、workspace 面板刷新。
- 管理端大表格和历史记录基线：监控页、性能页、吞吐页、benchmark 页的渲染和刷新。
- Docker workspace IO 专项基线：真实挂载语义下的 `file_ops` 子步骤耗时，重点看 list/write/read/search/patch。

### P2 先观察后定门槛

- 真实线上模型网络体验。
- 浏览器复杂页面的长链路 trace 和内存峰值。
- 外部渠道长链路和弱网场景的稳定性。

## 基线采集规则

1. 记录环境：操作系统、Docker/本地、compose 文件、数据库后端、浏览器、Rust/Node 版本、关键环境变量、是否 debug 日志。
2. 记录代码状态：基线来源、优化后来源、是否存在未提交改动；如果工作区有其它开发者改动，要说明可能影响范围。
3. 先热身再采样：服务启动后先跑一次不计入结果的预热，避免冷启动、首次编译、首次缓存填充污染结果。
4. 固定数据规模：会话数、消息数、文件数、并发数、测试用户数、压测时长要一致。
5. 固定并发序列：没有特殊理由时，性能采样至少覆盖 `1 / 5 / 10 / 20`；吞吐压测至少覆盖目标上线并发附近的一个点。
6. Docker 文件操作必须使用真实工作区挂载。不得把 `file_ops` 改到 `/tmp`、内存盘或临时挂载来规避真实部署成本。
7. 外部模型会带来网络和服务端波动。验证系统链路时优先使用稳定 mock、本地模型或固定模型配置；验证真实线上体验时必须明确模型和网络条件。

## 回退判定

同一环境下满足任一条件即视为性能回退，除非报告中明确说明这是有意 tradeoff 且收益大于损失：

- 错误率、失败数、超时数高于基线。
- 核心链路 p95 或平均耗时劣化超过 10%，且绝对增加超过 50ms。
- 用户可感知交互耗时劣化超过 5%，且绝对增加超过 100ms。
- 吞吐 RPS 下降超过 5%，或同等并发下成功率下降。
- 内存峰值或稳定态内存增加超过 10%，且无法用功能新增解释。
- blocking/queue/long task 告警数量增加，或热点 label 出现新的秒级等待。
- Docker 文件操作中任一子步骤劣化超过 15%，且该步骤已经是主要瓶颈。

轻微波动处理：

- 劣化在阈值附近时，必须重复 3 次取中位数。
- 如果基线本身噪声很大，要在报告中标记“不稳定基线”，并先建立稳定采样方法。
- 如果只优化了某一子步骤但总耗时未改善，不能写成整体性能提升，只能写明局部改善。

## 性能报告存放

性能对比报告统一放在 `docs/性能基线/`：

- 文件名：`YYYY-MM-DD-<scope>-<topic>.md`
- 大型原始 JSON、trace、截图放在 `docs/性能基线/assets/`，报告中引用路径。
- 运行态自动生成的大文件可以只保留摘要和导出位置，不强制提交到仓库。

报告必须包含：

- 目标：本次要优化或防回退的链路。
- 环境：足以复现比较条件。
- 命令/入口：采样命令、管理端页面操作或 API payload。
- 基线结果：优化前关键指标。
- 优化后结果：优化后同口径指标。
- 对比表：百分比变化和绝对变化。
- 结论：通过、回退、待复测或有意 tradeoff。
- 后续动作：需要继续观察的指标或补充测试。

## 开发验收要求

- 最终回复必须写明跑了哪些性能/回归命令、报告文件路径，以及是否发现回退。
- 如果因环境限制无法跑性能对比，必须明确写“未验证性能提升”，并说明缺失的环境或数据。
- `docs/功能迭代.md` 中的“性能”记录必须能追溯到性能报告、测试命令或明确的观测数据。
- 涉及 `docs/使用说明书` 的用户文档更新时，仍按仓库要求运行 `python scripts/build_docs_site.py`。
