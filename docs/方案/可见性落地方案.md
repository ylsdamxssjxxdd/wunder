# 可见性落地方案

## 结论
- 可见性要做成服务端策略，不做纯前端隐藏。
- 默认全部可见。
- 预设智能体可见时，允许它挂着一部分当前单位不可见的工具，UI 只提示“少工具正常”。
- 用户单位变更后，下一次请求必须按新单位重新算可见性；WS 活跃连接要补刷新。

## 现状
- 单位树和 `scope_unit_ids` 已有实现，管理员侧也已经按单位做范围过滤。
- 预设智能体已有稳定的 `preset_id` / `revision` / `preset_binding` 同步链路。
- `/wunder/user_tools/tools`、`/wunder/user_tools/catalog`、`/wunder/tools` 都是服务端实时组装工具清单。
- `resolve_user` 每次 HTTP 请求都会重新读用户表，但 `wunder_ws` 在连接建立时会把 `user` 快照传进 `handle_ws`，单位变更会在活跃 WS 上滞后一轮。
- 当前用户单位变更接口只改 `unit_id` 和 `updated_at`，没有自动触发 `sync_user_state`。

## 设计原则
- 只存显式允许单位，不存排除单位。
- 单位树变动时按当前树实时展开，不提前固化补集。
- 预设智能体和工具的可见性是同一套单位策略，但读写接口分开。
- 配置态负责保存，运行态负责实时求交。

## 数据落点
- `src/core/config.rs`：给预设智能体配置补 `visible_unit_ids`。
- 工具可见性建议独立成轻量配置块，不继续塞进主 `wunder.yaml`。
- 可见性值统一是单位 ID 列表，空值表示全量可见。
- 单位选择支持父节点，读路径按当前组织树展开到子树。

## 后端落点
- `src/api/admin.rs`：预设智能体列表、保存、同步接口补可见性字段。
- `src/api/admin.rs`：新增工具可见性接口，供技能管理右侧按钮使用。
- `src/services/user_agent_presets.rs`：保存/同步预设时保留隐藏工具，不要因为可见性把 `declared_tool_names` / `declared_skill_names` 洗掉。
- `src/services/user_access.rs`：把单位可见性并入 `compute_allowed_tool_names` / `compute_allowed_tool_names_for_catalog` 的实时求交。
- `src/api/user_tools.rs` 和 `src/api/core.rs`：返回给前端的工具清单带可见性摘要。
- `src/api/auth.rs`、`src/api/admin.rs`：用户单位变更落库后补一次 `sync_user_state(user_id)`。
- `src/api/core_ws.rs`：WS 运行前或每轮开始时重新确认用户快照，不能永久相信连接初始快照。

## 前端落点
- `web/modules/preset-agents.js`：预设智能体编辑区加“可见性”按钮，默认“全部可见”。
- `web/modules/skills.js`：技能管理右侧加“可见性”按钮，弹窗展示所有已开放工具并按单位勾选。
- `web/modules/tools.js`：工具选择页和预设页显示“当前有 N 个工具因可见性未展示”。
- `web/modules/i18n.js`：补齐可见性、不可见工具数量、全部可见、按单位可见等文案。

## 处理规则
- 智能体可见但部分工具不可见时，智能体仍可见，工具列表只显示当前单位能看到的部分。
- 不可见工具不要伪装成删除，只能显示为隐藏/受限。
- 保存时如果用户当前看不到某些既有工具，要保留原值，不要因为编辑界面少选项就把它擦掉。

## 即时生效
- 读接口：下一次 HTTP/WS 重新拉取时，按新 `unit_id` 直接生效。
- 活跃 WS：需要在每轮前重读用户记录，或用 `updated_at` / 版本号判断快照失效。
- 缓存：如果给可见性做缓存，缓存键必须包含 `user_id + 当前单位版本`。

## 节点
- P0: 后端可见性模型与实时求交。
- P1: 管理端预设智能体/工具可见性 UI。
- P2: 用户单位变更后的重同步与 WS 刷新。
- P3: 回归测试和提示文案。

## 测试
- 单测：全量可见、子树继承、空列表语义。
- 集成：用户换单位后重新拉取工具清单，结果立刻变化。
- 回归：预设智能体存在隐藏工具时仍显示，并提示“少工具正常”。
