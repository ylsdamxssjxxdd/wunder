# 舰桥中心落地方案

## 1. 产品定位

`舰桥中心` 是管理员侧的统一桥接入口，用来把多个外部用户通过同一个渠道账号接入 wunder。

当前落地模型已经统一为：

- 页面总称：`舰桥中心`
- 单个实体：`舰桥节点`
- 管理端职责：`节点配置 + 接入渠道 + 运行监控`
- 外部用户落点：真实 `user_accounts`
- 智能体落点：真实 `user_agents`
- 渠道路由落点：`bridge_user_routes`

它不是第二套用户系统，也不是管理员替桥接用户改智能体配置的控制台。

---

## 2. 标准业务链路

管理员侧流程：

1. 在舰桥中心新建一个舰桥节点。
2. 在“中心配置”弹窗中填写节点名称、状态、默认预设智能体、目标单位和说明。
3. 在“接入渠道”弹窗中，为该节点直接创建或维护节点专属渠道账号。
4. 渠道表单由管理员侧单独实现：公共字段单独填写，渠道专属字段通过 JSON 配置补齐。

运行时流程：

1. 外部用户通过节点绑定的渠道账号发来消息。
2. 系统按渠道身份规则提取 `external_identity_key`。
3. 如果没有命中既有桥接路由，则自动创建 wunder 用户，默认密码固定为 `123456`。
4. 系统根据舰桥节点默认预设，自动确保该用户拥有目标智能体。
5. 系统建立 `外部身份 -> wunder 用户 -> 智能体` 的稳定桥接路由。
6. 消息进入 wunder 正常线程链路处理。
7. 智能体回复后，系统沿原渠道账号把结果发回外部用户。

---

## 3. 页面原则

舰桥中心主页必须是监控页，不是大表单页。

当前页面原则已经固定为：

- 主页面只放节点列表、概览指标、接入渠道监控、自动分配路由、投递日志。
- `中心配置` 和 `接入渠道` 都通过弹窗承载。
- 不再展示 `中心编码`、`用户名策略`、`默认身份策略` 这些提高心智负担的字段。
- 不再暴露独立的“共享账号绑定”产品概念；节点下的渠道由“接入渠道”弹窗直接维护。
- 不在舰桥中心中直接提供桥接用户的线程、定时任务、工作区、智能体高级设置操作。

---

## 4. 接入渠道设计

舰桥中心现在支持所有已进入渠道目录的 provider，而不是只支持 XMPP。

当前管理员侧接入渠道实现为：

- 前端：原生 JS 单独实现，不复用用户侧 Vue 页面。
- 公共字段：`channel / account_id / account_name / peer_kind / enabled / thread_strategy / status_reason / default_preset_agent_name_override`
- 渠道专属字段：统一写入 JSON 配置。
- 交互补强：
  - 根据所选渠道展示 `display_name / description / docs_hint / provider_caps`
  - 提供“填入模板”按钮，给管理员快速生成当前渠道的 JSON 配置骨架
  - 在只读场景下禁用模板和 JSON 编辑，避免误改外部账号

这套设计的目的不是复制用户侧所有动态表单，而是在管理员侧用更轻的 JS 维护一套足够清晰、覆盖所有渠道的配置面。

---

## 5. 后端对象模型

当前落地对象如下：

- `bridge_centers`：舰桥节点主表
- `bridge_center_accounts`：节点接入渠道表
- `bridge_user_routes`：外部身份到 wunder 用户/智能体的稳定路由
- `bridge_delivery_logs`：入站/出站投递日志
- `bridge_route_audit_logs`：路由治理审计日志
- `channel_accounts`：真实渠道账号配置
- `user_accounts`：真实 wunder 用户
- `user_agents`：真实 wunder 用户智能体

核心原则：

- 共享物理渠道账号不复制到每个用户。
- 外部用户不落到虚假 runtime 用户，而是落到真实用户账号。
- 桥接失败不静默回退到 owner 用户，避免串号。

## 7. 当前结论

这套方案是可行的，而且已经按这个方向落地：

- 舰桥中心页面已经收敛为监控主页。
- 接入渠道已经改为管理员侧原生 JS 单独实现。
- 默认预设智能体来源已修正，不再出现列表为空但系统实际有预设的情况。
- 所有外部用户都可以通过单一渠道账号被自动开户、自动挂预设、自动建路由并回消息。
