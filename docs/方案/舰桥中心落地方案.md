# 舰桥中心落地方案

## 1. 定位

`舰桥中心` 不是 XMPP 专用页，也不是第二套用户体系，更不是管理员替桥接用户编辑智能体的控制台。

它的正确定位是：

- 挂在共享渠道账号之上的统一入口；
- 复用现有 `external launch / external embed` 能力的自动开户与自动挂接层；
- 负责“识别外部发送者 -> 创建/复用 wunder 真实用户 -> 自动确保默认预设智能体 -> 将回复发回原渠道”的桥接调度层。

最终形态应当是：

- 外部发送者仍然落到真实的 `user_accounts`；
- 智能体仍然落到真实的 `user_agents`；
- 线程、工作区、长期记忆、定时任务仍走现有链路；
- `舰桥中心页面` 只做中心配置、接入渠道配置、路由观测、异常排查；
- `共享账号绑定不再作为独立产品概念出现，而是收敛到中心配置里统一保存`；
- `舰桥中心页面不提供这些用户智能体的设置操作功能`。

一句话概括：

> 舰桥中心是“全渠道版 external launch”，不是第二套 bridge runtime。

---

## 2. 渠道范围

舰桥中心按 `src/channels/catalog.rs` 的渠道目录统一设计，不再只针对 `xmpp`。

当前目录中的 provider 包括：

- `feishu`
- `qqbot`
- `whatsapp`
- `weixin`
- `wechat`
- `wechat_mp`
- `xmpp`
- `telegram`
- `discord`
- `slack`
- `line`
- `dingtalk`

当前已经在注册表中明确接入的运行时适配器包括：

- `whatsapp`
- `feishu`
- `qqbot`
- `wechat`
- `wechat_mp`
- `weixin`
- `xmpp`

这意味着舰桥中心的设计必须满足两点：

1. 表结构、API、页面模型必须对所有渠道统一抽象，不能把渠道特性写死在 XMPP 语义里。
2. 即使某些 provider 目前仍处于 generic catalog 或预留阶段，舰桥中心也要在模型上原生支持，后续只通过 adapter readiness 打开运行态即可。

因此，舰桥中心中的“共享渠道账号”应被理解为：

- 任意一条来自 `channel_accounts` 的共享接入账号；
- 它可以来自 webhook 模式、长连接模式、长轮询模式或 generic callback 模式；
- 它的 provider 只是 `bridge_center_accounts.channel` 的一个取值，不影响桥接主模型。

## 3. 标准业务链路

以“张三从任意共享渠道发消息进入”为例，标准链路应为：

1. 管理员在 `舰桥中心` 中创建一个 `舰桥节点`。
2. 管理员在中心配置中选择一个或多个共享渠道账号。
3. 管理员为中心设置默认对外预设智能体。
4. 外部用户从共享渠道发消息进入。
5. 系统按该渠道的身份提取策略生成 `external_identity_key`。
6. 系统在桥接路由表中查找该身份是否已存在稳定映射。
7. 若不存在，则自动创建/复用 wunder 真实用户，默认密码固定为 `123456`。
8. 系统自动确保该用户拥有默认预设智能体对应的用户智能体。
9. 系统写入“外部身份 -> wunder 用户 -> agent”桥接路由。
10. 消息被转发给该用户的目标智能体处理。
11. 回复沿原共享渠道账号投递回外部发送者。

这个流程对 `xmpp`、`weixin`、`feishu`、`qqbot`、`wechat`、`whatsapp` 等 provider 都成立；差异只体现在：

- 外部身份怎样提取；
- 是否支持线程/子会话；
- 是否允许主动出站；
- 回复是否依赖 `context_token` 或原会话窗口。

桥接主模型本身不能被这些 provider 差异打散。

## 4. 核心设计决策

### 4.1 真实用户账号，不再引入 bridge member runtime

外部用户最终落到真实的 `user_accounts`，不再设计 `bridge_member / runtime_user_id` 作为第二套运行身份。

理由：

- 用户身份统一；
- 智能体、线程、工作区生命周期统一；
- 与现有用户管理、预设智能体、外链机制完全兼容；
- 后续 bridge 用户转为普通用户无需迁移运行身份。

### 4.2 默认密码沿用外链语义

自动开户沿用现有 `external launch` 语义：

- 若用户不存在，则自动创建；
- 默认密码固定为 `123456`；
- 若用户已存在，则复用原用户；
- 后续安全增强可再通过 `bridge_managed`、首次改密、禁用密码登录等策略追加。

### 4.3 共享渠道账号不复制到每个用户

“默认预设智能体中自动配置好渠道”的正确实现不是复制物理渠道凭证，而是：

- 共享渠道账号只在舰桥中心配置一次；
- 每个外部身份生成一条只读桥接路由；
- 这条路由声明“该外部身份的消息进入哪个 wunder 用户、哪个 agent”。

因此：

- `channel_accounts` 继续只保存物理账号凭证；
- 舰桥中心只新增逻辑路由关系；
- 用户侧不应把它展示成一个可编辑的个人渠道账号。

### 4.4 页面只做桥接治理，不做智能体设置

舰桥中心页面只负责：

- 中心配置；
- 共享账号挂接；
- 用户发现与路由观测；
- 投递日志与异常排查；
- 跳转到现有用户页、线程页、渠道页。

舰桥中心页面不负责：

- 编辑桥接用户的智能体提示词；
- 切换桥接用户的预设；
- 管理桥接用户线程；
- 管理桥接用户定时任务；
- 浏览/编辑桥接用户工作区。

### 4.5 路由优先级固定

推荐顺序：

1. 显式手工绑定 `channel_user_bindings / channel_bindings`
2. 舰桥中心桥接路由
3. 账号 owner fallback

### 4.6 桥接失败禁止 silent fallback

一旦某个共享渠道账号已经挂到舰桥中心：

- 若桥接解析失败；
- 若自动开户失败；
- 若默认预设缺失；
- 若 provider 不支持当前回复方式；

都不允许静默回退到 `owner_user_id`。

否则会发生：

- 多个外部用户串到同一个内部用户；
- 会话与工作区串号；
- 渠道 owner 被误当成统一接收者。

## 5. 与当前代码的复用关系

舰桥中心落地时，应优先复用现有能力，而不是重造。

### 5.1 复用用户自动开户

来自 `src/api/auth.rs`：

- `provision_external_launch_session(...)`
- `DEFAULT_EXTERNAL_LAUNCH_PASSWORD`

它已经实现了：

- 不传密码时自动创建真实用户；
- 默认密码为 `123456`；
- 已有用户则复用。

### 5.2 复用默认预设智能体自动确保

来自 `src/api/auth.rs`：

- `resolve_or_create_external_embed_agent(...)`

它已经实现了：

- 找到默认预设智能体；
- 若用户名下已有同名智能体则优先复用；
- 否则按预设自动创建。

### 5.3 复用渠道目录与适配器注册表

来自：

- `src/channels/catalog.rs`
- `src/channels/registry.rs`
- `ChannelHub`
- `channel_sessions`
- `channel_outbox`

这意味着舰桥中心不应该重写渠道系统，只需在渠道主链路中增加一层：

- 共享账号识别
- 外部身份解析
- 自动开户/自动挂接
- 路由写回与只读观测

## 8. 分期建议

### P1

- 全渠道统一桥接模型落地
- 可创建中心
- 可挂接共享渠道账号
- 可选择默认预设智能体
- 首包自动开户
- 自动确保默认预设智能体
- 稳定桥接路由
- 回复回投原渠道

### P2

- 路由观测与投递日志完善
- 中心账号状态、异常原因、最近活跃统计
- 与用户详情、线程详情、渠道运行日志联动

### P3

- 更细粒度的身份提取策略
- bridge 用户安全策略增强
- provider capability 矩阵细化
- 批量封禁、批量暂停、批量恢复等运营能力

---

## 9. 最终结论

舰桥中心必须按“所有渠道统一桥接”的思路来做，而不是继续沿着“XMPP 特例页”扩张。

最稳的实现路径是：

- 不做第二套 bridge 用户体系；
- 不给每个用户复制共享渠道凭证；
- 直接复用 `external launch + external embed`；
- 用桥接路由表稳定外部身份与 `wunder_user_id + agent_id` 的关系；
- 让管理员侧只处理共享渠道、路由观测和异常排查。

这样才能同时支持现在目录中的所有 provider，并给后续新渠道接入留下统一入口。
