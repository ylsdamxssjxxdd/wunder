# 上下文压缩优化方案

## 1. 目标

本方案不是重写 wunder 的上下文压缩，而是在现有能力上补齐托底能力，重点解决下面几类问题：

- 压缩成功但结果不可用，例如摘要被写成 `...(`。
- 压缩事件、压缩结果、历史回放、下一次真实请求体四者不一致。
- 连续压缩后语义逐轮劣化，甚至“坏摘要套坏摘要”。
- 手动压缩、自动压缩、溢出恢复三条路径行为不一致。
- 前端调试看到的是一种形状，后端真正发给模型的是另一种形状。

目标不是“尽量压进去”，而是：

1. 要么产出可继续工作的压缩结果。
2. 要么明确判定本次压缩失败，不污染历史，不伪装成功。

## 2. 可借鉴的 codex 设计

参考项目：`C:\Users\sjxx\Desktop\参考项目\codex-main`

重点借鉴点：

### 2.1 把压缩视为独立事务，而不是一段临时字符串

codex 会把压缩作为独立 turn/item 处理：

- 手动压缩有单独入口：`thread/compact/start`
- 压缩期间有 started / completed 事件
- 压缩完成后写入的是“可替换历史”，不是只塞一条摘要文本

可借鉴结论：

- wunder 也应该把压缩视为一次明确事务，拥有唯一 `compaction_id`
- 压缩结果必须有明确状态：`started / completed / failed / skipped`
- 不能只依赖一条普通 user 消息承载全部语义

### 2.2 压缩结果和下一次回放形状必须是同一个真相源

codex 的核心不是“生成一条摘要”，而是构建 `replacement_history`，然后直接用这份替换历史继续运行。

可借鉴结论：

- wunder 需要区分：
  - `summary_model_output`：模型原始摘要
  - `committed_summary_text`：通过校验后允许落库的摘要
  - `replacement_history`：真正用于 replay / 下一次请求拼接的历史骨架
- 下一次发给模型的请求体，必须来自同一份已提交的 `replacement_history`，不能再次临时拼一套近似结构

### 2.3 先修剪高噪音历史，再发摘要请求

codex 在压缩前会优先修剪函数调用历史、工具噪音、codex 自生成项目，避免“压缩请求本身先炸掉”。

可借鉴结论：

- wunder 的压缩前置裁剪顺序应固定：
  1. 超长 observation / tool payload
  2. 纯系统生成的低价值运行态消息
  3. 重复的老压缩标记 / 老摘要镜像
  4. 再进入摘要模型

### 2.4 压缩失败时不能写坏结果

codex 的实现更偏“失败就失败”，不会把明显无语义的占位摘要伪装成成功结果长期落库。

可借鉴结论：

- wunder 必须新增“提交前校验”
- 若摘要退化为占位文本、空文本、极短乱码摘要，则直接失败回滚，不允许写入历史

### 2.5 测试关注“请求形状快照”，不只测字段

codex 大量测试不是只断言几个计数值，而是直接快照：

- 压缩请求长什么样
- 压缩后的历史长什么样
- 多次压缩后下一轮请求长什么样
- 手动压缩 / 中途压缩 / 远端压缩后的布局长什么样

可借鉴结论：

- wunder 也要把“最终发给模型的完整请求体形状”纳入测试主对象
- 不能只测 `summary_text`、`retained_head_turns` 这种孤立字段

## 3. wunder 现阶段主要风险

结合现有日志和实现，当前风险集中在这几类：

### 3.1 摘要生成成功，但提交结果被后处理裁坏

现象：

- `summary_model_output` 完整
- `summary_text` 却是 `...(`
- 后续 replay 和正常请求真的带着 `[上下文摘要]\n...(` 继续跑

问题本质：

- 提交阶段缺少“最低语义质量校验”
- 裁剪函数允许生成无语义结果

### 3.2 压缩结果、历史落库、前端水合、下一次请求体不是同一份产物

现象：

- 调试看到一种内容
- hydrate 看到另一种内容
- 真正主请求又是第三种内容

问题本质：

- 缺少单一真相源
- 各阶段都在“重新推导压缩结构”

### 3.3 压缩预算策略过于偏 token 防守，缺少语义保底

现象：

- 为了塞进预算，先把摘要裁没
- 结果 token 是安全了，但语义已经断了

问题本质：

- 当前实现更关注“请求可发出”
- 但缺少“结果仍可继续工作”的硬约束

### 3.4 手动压缩、自动压缩、溢出恢复缺少统一状态机

现象：

- 手动压缩和自动压缩的事件链、调试链、回放链不完全一致
- hydrate 会把历史压缩重新投影成新事件，前端容易误判

问题本质：

- 缺少统一 compaction transaction 模型

## 4. 目标方案总览

目标链路统一成下面 9 个节点：

```text
触发判定
-> 压缩快照冻结
-> 候选历史清洗
-> 摘要请求构建
-> 摘要执行
-> 提交前校验
-> replacement_history 构建
-> 原子提交
-> 下一轮请求重建 / 调试输出
```

核心原则：

- 任何时刻只认一份 `replacement_history`
- 任何阶段发现摘要不可用，都不写坏结果
- 压缩日志、持久化结果、下一次请求体必须能一一对上

## 5. 节点设计

### 节点 A：触发判定

输入：

- 当前会话历史
- 已记录的 context token
- 当前用户消息
- 触发来源：`manual / presampling / overflow_recovery / history_threshold`

输出：

- `compaction_plan`

要求：

- 为每次压缩生成唯一 `compaction_id`
- 明确是否属于：
  - 手动压缩
  - 正常下一轮调用前压缩
  - 溢出恢复压缩
- 若会话已有进行中的压缩事务，拒绝重复进入

失败处理：

- 重复触发时返回 `skipped_already_running`
- 不得创建第二套并行压缩状态

### 节点 B：压缩快照冻结

输入：

- 会话实时历史

输出：

- `compaction_snapshot`

内容至少包括：

- 快照时刻的消息列表
- 当前用户轮次
- 当前模型轮次
- 当前 system prompt 指纹
- artifact/fresh memory 指纹
- 触发前 context token 估计

要求：

- 从这里开始，后续所有压缩步骤都只基于这份快照
- 不允许一边压缩一边继续从 live 历史里重新取消息

失败处理：

- 快照失败则直接压缩失败，不进入摘要模型

### 节点 C：候选历史清洗

输入：

- `compaction_snapshot`

输出：

- `normalized_history_for_summary`

清洗顺序固定：

1. 合并 system 与 artifact 上下文
2. observation/tool payload 文本化、摘要化
3. 剔除旧压缩遗留的镜像摘要
4. 剔除重复的运行态噪音事件
5. 对单条超大工具输出做局部裁剪

要求：

- 这一阶段只允许减少噪音，不允许丢当前用户真实意图
- 清洗后要产出可调试的 `normalization_report`

失败处理：

- 若清洗后已无有效历史，则转为 `fallback_compaction_input`

### 节点 D：摘要请求构建

输入：

- `normalized_history_for_summary`

输出：

- `summary_request_payload`

硬约束：

- 摘要模型请求体必须完整记录到 debug 日志
- 最后一条必须是压缩专用指令，不得和用户原问题混淆
- 多 system 合并为单一 system

要求：

- 增加 `summary_request_hash`
- 增加 `summary_input_chars`
- 记录本次构建是否经过裁剪

失败处理：

- 构建失败直接终止本次压缩

### 节点 E：摘要执行

输入：

- `summary_request_payload`

输出：

- `summary_model_output`

要求：

- 完整保存模型原始输出
- 记录模型、耗时、重试次数、输入/输出 token
- 输出不能为空字符串

失败处理：

- 模型失败时可走 fallback summary
- 但 fallback 也必须进入后续校验，不能直接落库

### 节点 F：提交前校验

输入：

- `summary_model_output`
- 当前文档规定：
  - `summary_text <= 20000 字`
  - `head + tail <= 20000 字`

输出：

- `committed_summary_text`
- `summary_validation_report`

这是最关键的托底节点，必须新增。

校验规则：

1. 不能为空
2. 不能是 `...(`、`...(truncated)`、`...`、空前缀摘要等占位文本
3. 不能只有标题没有正文
4. 不能低于最小有效长度阈值
5. 必须保留可读的任务进展 / 待办 / 关键引用中的至少两类信息
6. 裁剪后仍需满足以上规则

推荐新增状态：

- `valid`
- `fallback_used`
- `invalid_placeholder`
- `invalid_too_short`
- `invalid_empty`
- `invalid_structure`

失败处理：

- 若校验失败：
  - 本次压缩标记 `failed`
  - 不写坏摘要
  - 不改写历史 replay
  - 前端展示“压缩失败，历史保持原状”

### 节点 G：replacement_history 构建

输入：

- `committed_summary_text`
- `head`
- `tail`
- `current_user`
- `trigger_mode`

输出：

- `replacement_history`

构建规则：

- 持久化结构固定为：`head -> summary -> tail`
- 运行时结构固定为：`system -> head -> summary -> (keep middle) -> tail -> current user`
- `head` 固定最前 2 轮用户轮次
- `tail` 固定最后 2 轮用户轮次
- `summary_text` 最多 20000 字
- `head + tail` 合计最多 20000 字

关键托底：

- 若 `head + tail + summary + current user + system` 仍塞不进模型窗口：
  1. 先收缩 head/tail 的轮内长 observation
  2. 再收缩次要 assistant 描述
  3. 最后才收缩摘要
  4. 若摘要再收缩会失去语义，则压缩失败，不提交

不允许行为：

- 为了过预算，把摘要裁成不可读占位符后仍提交
- 运行时请求体和持久化回放结构来源不同

### 节点 H：原子提交

输入：

- `replacement_history`
- `committed_summary_text`
- `summary_model_output`
- `summary_validation_report`

输出：

- 已提交压缩事务

提交内容建议拆开存：

- `compaction_record`
- `replacement_history`
- `committed_summary_text`
- `summary_model_output`
- `validation_report`

要求：

- 提交必须原子化
- 不能出现：
  - 事件写成功但 replacement_history 没写
  - summary_text 写坏但 validation_report 没写
  - 前端看到 completed，历史实际上没替换

失败处理：

- 任一写入失败则整体回滚

### 节点 I：下一轮请求重建 / 调试输出

输入：

- 已提交的 `replacement_history`

输出：

- 下一次真实发给模型的请求体
- 对应 debug snapshot

要求：

- 下一轮真实请求体必须直接从 `replacement_history` 派生
- 调试导出必须包含：
  - 原始摘要请求体
  - 原始摘要输出
  - 提交后的 committed summary
  - replacement_history
  - 下一次真实主请求体

验收标准：

- 任何一次线上问题，都能从日志中完整还原：
  1. 为什么压缩触发
  2. 发了什么给摘要模型
  3. 模型回了什么
  4. 为什么这个摘要能或不能提交
  5. 最终发给主模型的请求体到底是什么

## 7. 错误处理矩阵

### 7.1 摘要模型输出为空

处理：

- 先尝试 fallback summary
- fallback 仍为空则压缩失败
- 历史保持原状

### 7.2 摘要模型输出过长

处理：

- 允许裁剪到 20000 字以内
- 裁剪后重新做语义校验
- 校验不过则失败，不提交

### 7.3 摘要被裁成无语义占位符

处理：

- 直接判失败
- 不允许写成 `summary_text`
- 不允许进入 replay

### 7.4 首尾四轮太长

处理：

- 保留“前 2 轮 / 后 2 轮”骨架
- 在轮内裁 observation 和次要 tool 回显
- 优先保留用户消息和关键 assistant 结论

### 7.5 手动压缩时线程仍在运行

处理：

- 接受请求后创建单独压缩事务
- 若当前轮不可安全压缩，则排队到安全点
- 前端显示 running / waiting，不得假完成

### 7.6 模型流式输出中途命中阈值

处理：

- 不做中途热切换压缩
- 允许当前轮先完成
- 下一次采样前再压缩
- 若本轮直接报上下文溢出，则走 overflow recovery

### 7.7 多次连续压缩

处理：

- 永远只以最新 `replacement_history` 为基线
- 不允许把旧 summary 当普通历史再摘要一次
- 必须保证“摘要套摘要”不会发生

### 7.8 hydrate / replay 重放旧压缩事件

处理：

- 调试事件需带 `source=live|hydrate|replay`
- 前端不得把 hydrate 事件误判为新执行的 live 压缩

## 8. 借鉴 codex 后建议优先落的 6 条硬规则

### 规则 1

压缩必须有独立事务 id，不能靠 round + 若干松散事件拼语义。

### 规则 2

`summary_model_output` 和 `committed_summary_text` 必须分开存。

### 规则 3

任何坏摘要一律不得落库，不得 replay，不得进入下一轮主请求。

### 规则 4

下一轮请求体必须来自已提交的 `replacement_history`，不能临时再拼一套。

### 规则 5

前端调试导出必须能完整还原“摘要请求 -> 摘要输出 -> 提交结果 -> 下一轮主请求体”。

### 规则 6

所有高风险路径必须有快照测试，测试对象是“请求体形状”，不是少数字段。

## 9. 测试补强清单

### 9.1 单元测试

- 摘要占位符识别：`...(`、`...(truncated)`、空串、只有标题
- `summary_text <= 20000 字`
- `head + tail <= 20000 字`
- 首尾 2 轮骨架裁剪顺序
- 校验失败不提交

### 9.2 集成测试

- 手动压缩成功后，下一次真实请求体包含有效 summary
- 手动压缩失败后，历史不变
- 自动压缩成功后，replay 与下一次请求体一致
- overflow recovery 后，坏摘要不会落库
- 连续两次压缩后，不出现摘要套摘要

### 9.3 快照测试

- 摘要模型请求体快照
- 提交后的 replacement_history 快照
- 下一次真实主请求体快照
- hydrate 事件快照
- manual / auto / overflow_recovery 三路径快照

### 9.4 前端联调测试

- live 压缩与 hydrate 回放不串态
- 调试导出不截断
- 手动压缩分割线状态正确
- 压缩失败时展示失败，不展示成功

## 10. 实施顺序

### 第一阶段：先补托底

目标：

- 坏摘要绝不落库

实施项：

- 新增提交前校验
- 新增 placeholder 判定
- 新增失败回滚
- 新增 `source=live|hydrate|replay`

### 第二阶段：统一真相源

目标：

- 压缩结果、replay、下一次请求体完全一致

实施项：

- 引入 `replacement_history`
- 下一次请求体改为从 `replacement_history` 重建
- debug 导出串联完整链路

### 第三阶段：统一状态机

目标：

- manual / auto / overflow_recovery 三路径共用同一事务模型

实施项：

- 引入 `CompactionTransaction`
- started/completed/failed/skipped 统一
- 前端只消费事务态，不猜测状态

### 第四阶段：补齐测试矩阵

目标：

- 上线前能稳定拦住退化

实施项：

- 请求体快照测试
- 连续压缩回归测试
- hydrate / live 区分测试
- 超长输入与超长 observation 测试

## 11. 验收标准

达到下面条件，才算这轮优化完成：

1. 线上任何压缩结果都不会再出现 `[上下文摘要]\n...(` 进入真实主请求。
2. 压缩失败时，历史和下一轮请求保持压缩前状态，不出现半提交。
3. 任意一次压缩都能从日志还原完整链路。
4. 手动压缩、自动压缩、溢出恢复三条路径的最终请求体结构一致。
5. 多次连续压缩后，请求体仍然可读、可接续，不出现摘要套摘要。
