# 编排态系统落地方案

## 1. 目标

本方案用于把 [编排态系统设计](C:\Users\sjxx\Desktop\wunder\docs\设计文档\编排态系统设计.md) 落成可开发、可联调、可回归的实施路径。

本次落地不追求一次性把所有抽象都做满，而是优先把以下四件事做实：

1. 让母蜂为准的蜂群级编排态真正成为系统事实。
2. 让聊天页对编排线程进入强保护，避免用户误操作破坏编排连续性。
3. 让解除编排态与历史恢复闭环，不再只有“新建编排”而没有正式退出与回到过去。
4. 让工蜂线程缺失时具备可控补偿能力，避免编排 run 因局部线程丢失而崩坏。

## 2. 现状问题

当前系统已经有一部分编排能力，但仍停留在“编排页局部运行态”阶段，主要问题如下：

- 编排运行态主要保存在前端，后端只保存母蜂 session 上的一小段上下文，尚未形成蜂群级编排态对象。
- 聊天页与编排页对同一个母蜂是否处于编排态没有统一判断，存在语义裂缝。
- 母蜂编排线程已经可被固定使用，但聊天页仍可能对同一 agent 做新建线程、切换线程、发送消息等普通操作。
- 工蜂缺失线程时，只有零散的注入逻辑，没有正式的“编排线程补偿”模型。
- 当前没有正式的“解除编排态”与“历史恢复编排态”系统入口。

所以本方案的重点不是继续修补页面，而是先把编排态下沉到系统层。

## 3. 总体实施原则

- 母蜂是编排态唯一真相源。
- 编排态是蜂群级状态，不是页面私有状态。
- 前端只负责展示与交互阻断，真正的状态判断必须后端可判定。
- 线程保护要做前后端双保险，不能只靠 UI 禁用。
- 分阶段实施，先做能保护现实的一期，再做恢复与补偿的二期。

## 4. 范围划分

## 4.1 一期必须完成

- 蜂群级编排态对象
- 母蜂进入编排态时全蜂群线程升级
- 聊天页编排线程保护
- 编排页“解除编排态”按钮
- 解除后全蜂群自动新建普通主线程
- 编排线程特殊 logo 标记
- 编排态历史列表基础结构

## 4.2 二期完成

- 历史编排态恢复
- 工蜂线程缺失自动补偿
- 恢复时工蜂缺失线程自动补建
- 更完整的编排态事件流与 UI 状态恢复

## 4.3 三期优化

- 编排态统计与管理员观测
- 编排线程保护原因提示优化
- 历史编排态筛选与搜索
- 更细粒度的恢复策略与冲突提示

## 5. 数据与状态模型

## 5.1 新增后端对象

建议新增 `orchestration_state` 视图对象，对应存储可先走现有 meta / 轻量表，再决定后续是否独立成表。

最小字段：

- `orchestration_id`
- `user_id`
- `hive_id`
- `mother_agent_id`
- `mother_session_id`
- `status`
- `enabled`
- `created_time`
- `updated_time`
- `latest_round_index`

状态值：

- `active`
- `recovering`
- `exiting`
- `closed`

## 5.2 新增成员绑定对象

建议新增 `orchestration_member_binding`：

- `orchestration_id`
- `agent_id`
- `role`
- `session_id`
- `is_missing`
- `updated_time`

作用：

- 表达当前 run 中每个智能体对应的编排线程
- 为线程保护、派工命中、历史恢复提供统一依据

## 5.3 轮次目录继续保留文件真相源

目录结构不改，继续使用：

```text
orchestration/
└─ orch_<id>/
   ├─ round_0001/
   │  ├─ situation.txt
   │  ├─ <worker_agent_id_a>/
   │  └─ <worker_agent_id_b>/
   └─ meta.json
```

这里的核心原则仍然成立：

- 轮次态势和产物历史由目录结构给出真相
- 不额外设计复杂快照体系

## 6. 后端实施方案

## 6.1 编排态查询与写入接口

建议新增或扩展以下接口：

- `GET /wunder/beeroom/orchestration/state?hive_id=...`
- `POST /wunder/beeroom/orchestration/state/create`
- `POST /wunder/beeroom/orchestration/state/exit`
- `GET /wunder/beeroom/orchestration/history?hive_id=...`
- `POST /wunder/beeroom/orchestration/history/restore`

最小职责：

- `state`：返回当前蜂群是否处于编排态，以及母蜂 / 工蜂当前编排线程绑定
- `create`：创建新编排态并初始化全员线程
- `exit`：解除编排态并触发全员普通主线程重建
- `history`：返回母蜂历史编排线程列表
- `restore`：按指定母蜂历史编排线程恢复编排态

## 6.2 编排态与 session-context 关系调整

当前已经有按 `user_id + session_id` 保存的编排 session-context。

一期方案不要求完全推翻，但要调整语义：

- session-context 继续作为“某线程当前处于编排态”的直接判定依据
- 新增蜂群级编排态对象作为更高层真相

关系：

- 蜂群级编排态对象决定“整个蜂群是否处于编排态”
- session-context 决定“这个具体线程是否是当前编排线程的一部分”

## 6.3 线程保护拦截

后端必须拦截以下动作：

- 普通聊天消息发送到编排线程
- 对编排线程执行新建普通线程覆盖主线程
- 切换某 agent 当前主线程到非编排线程
- 在编排线程上执行不受控的普通线程管理操作

建议拦截点：

- `src/api/chat.rs`
- `src/services/runtime/thread_runtime` 或主线程设置链路
- `setDefaultSession` 对应后端路径

返回语义建议：

- `409 Conflict`
- 错误码示例：`ORCHESTRATION_THREAD_LOCKED`

返回信息：

- 当前线程处于编排态，请前往编排页操作

## 6.4 蜂群工具命中与补偿

扩展 `agent_swarm` 调度逻辑：

1. 先读取母蜂当前 session 是否属于活跃编排态。
2. 若不是，维持普通蜂群工具行为。
3. 若是，则按 `orchestration_member_binding` 优先命中目标工蜂编排线程。
4. 若工蜂绑定不存在或目标线程缺失，则自动新建工蜂编排线程。
5. 新建后更新成员绑定，并将当前轮态势与产物目录自动注入。

建议改造点：

- [tools.rs](C:\Users\sjxx\Desktop\wunder\src\services\tools.rs)
- [orchestration_context.rs](C:\Users\sjxx\Desktop\wunder\src\services\orchestration_context.rs)
- 蜂群 beeroom service 主线程解析逻辑

## 6.5 解除编排态

后端 `exit` 接口逻辑：

1. 根据 `hive_id` 找到活跃 `OrchestrationState`
2. 校验母蜂绑定仍存在
3. 清除母蜂 session 上的编排态上下文
4. 标记当前 run 为 `closed`
5. 为母蜂和全部工蜂新建新的普通主线程
6. 更新各 agent 的默认主线程
7. 返回新的普通线程映射

关键点：

- 这是显式系统动作，不是简单清一个前端状态

## 6.6 历史恢复

后端 `restore` 接口逻辑：

1. 用户选择某个母蜂历史编排线程
2. 校验该线程属于该用户、该 hive、且为历史编排线程
3. 建立新的活跃 `OrchestrationState`
4. 母蜂直接绑定到该历史线程
5. 工蜂逐个恢复：
   - 历史线程仍存在则复用
   - 缺失则新建编排线程
6. 重建成员绑定
7. 返回恢复后的活跃编排态

- `orchestration_state_created`
- `orchestration_state_exiting`
- `orchestration_state_closed`

作用：

- 让线程锁定与 logo 状态实时变化

## 9. 分阶段实施

## 9.1 阶段 A：系统骨架

目标：

- 建立蜂群级编排态对象
- 建立成员绑定
- 打通查询 / 创建 / 解除接口

完成标准：

- 后端可回答“这个 hive 当前是否处于编排态”
- 后端可回答“这个 agent 当前主线程是否为编排线程”

## 9.2 阶段 B：聊天页保护

目标：

- 前后端双层阻断编排线程误操作

完成标准：

- 用户在聊天页打开编排线程时，只能看，不能发、不能切、不能新建

## 9.3 阶段 C：解除编排态

目标：

- 正式支持整体退出编排态

完成标准：

- 点击“解除”后，整个蜂群切到新的普通主线程
- 编排线程变成历史编排线程，不再参与当前运行

## 9.4 阶段 D：历史恢复

目标：

- 以母蜂历史编排线程恢复编排态

完成标准：

- 选中历史后，母蜂回到对应历史线程
- 工蜂缺失线程可自动补建

## 9.5 阶段 E：补偿闭环

目标：

- 工蜂线程缺失时不破坏当前编排态

完成标准：

- 母蜂仍处于编排态时，工蜂被唤起即可自动回到编排线程现实

## 10. 风险与注意事项

## 10.1 最大风险

最大的风险不是 UI，而是“主线程真相冲突”：

- 聊天页认为当前主线程是 A
- 编排页认为当前主线程是 B

本方案就是要消灭这类冲突。

## 10.2 历史线程恢复风险

恢复历史时要防止：

- 恢复到不属于该 hive 的母蜂线程
- 恢复到已经被归档或不可读的线程
- 工蜂线程不存在但前端误以为已恢复成功

所以恢复接口必须返回完整的恢复结果，而不是只返回一个成功布尔值。

## 10.3 前端缓存风险

当前编排页已经有一套 runtime cache。

落地时必须避免：

- 旧 run 的缓存污染新 run
- 恢复历史后仍然看到上一套线程绑定
- 解除编排后聊天页还残留编排锁定样式

## 11. 回归测试清单

必须覆盖以下场景：

- 新建编排后，全蜂群线程都带编排标识
- 编排态期间，聊天页发送被阻断
- 编排态期间，聊天页新建线程被阻断
- 编排态期间，聊天页切线程被阻断
- 编排页解除后，全蜂群自动切到新普通主线程
- 解除后，旧编排线程只读保留
- 恢复历史编排态后，母蜂绑定到指定历史线程
- 恢复历史编排态时，缺失工蜂线程能补建
- 母蜂编排态仍在时，缺失工蜂线程在下次派工时自动补偿
- 母蜂编排态解除后，工蜂不再自动补偿到旧编排线程

## 12. 建议实施顺序

建议严格按以下顺序推进：

1. 后端编排态对象与查询接口
2. 前端线程锁定态与聊天页保护
3. 解除编排态
4. 历史恢复
5. 工蜂线程缺失补偿
6. 事件流与观测完善

原因很简单：

- 如果先做恢复和补偿，而线程保护没立住，系统语义还是会乱。
- 先把“不能破坏”做实，再做“怎么恢复”。

## 13. 结论

编排态系统的落地关键，不在于再做多少页面，而在于把“母蜂为准的蜂群级线程现实”真正写进系统。

只有做到下面这四条，编排态才算真正落地：

- 系统知道谁是当前活跃编排态
- 系统知道哪些线程是编排线程
- 系统能阻止用户破坏这些线程
- 系统能在需要时正式退出或恢复这套现实
