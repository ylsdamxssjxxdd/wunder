# 工具结果裁剪与观测链路

本文用于统一说明 wunder 中“工具结果为什么会被截断/压缩”，并明确不同层级是否会影响模型实际输入，避免把前端显示裁剪误判为模型侧裁剪。

## 1. 先看结论

按优先级区分为 5 层：

1. 工具自身裁剪（会影响模型、事件、日志）
2. 编排层全局裁剪 `finalize_tool_result`（会影响模型、事件、日志）
3. 模型观测压缩 `model_observation`（只影响模型输入，不改原始 `tool_result.data`）
4. 监控/时间线裁剪（影响监控记录与时间线 API，不直接改模型输入）
5. 前端显示裁剪（只影响 UI 展示，不改后端真实数据）

## 2. 端到端链路

一次工具调用的关键链路如下：

1. 工具执行返回原始结果（各工具可能先做自身预算裁剪）
2. `normalize_tool_result_payload`：统一成功/失败语义与错误元信息
3. `finalize_tool_result`：全局字符串/数组/大结果裁剪
4. `to_event_payload`：写入 `tool_result` 事件（用于前端工作流与排障）
5. `to_compact_payload`：生成 `model_observation`（给模型的压缩 observation）
6. 将 `model_observation` 作为 `tool_response` 追加到模型上下文

注意：第 4 步和第 5 步是双通道，不同用途。

## 3. 工具层裁剪（Layer 1）

以下为常见高频工具：

### 3.1 `read_file`

- 单文件最大读取字节：1MB
- 默认读取行窗口：1..2000
- `files` 模式默认最多取 5 个文件
- 可配置 `output_budget_bytes`，会被钳制到 1KB..2MB
- 超预算时在 `content` 追加 `...(truncated read output, omitted N bytes)...`
- 会返回 `meta.read.output_budget_hit/output_budget_omitted_bytes` 与 `continuation_required`

### 3.2 `list_files`

- 主要是分页，不是字符串截断
- `limit` 默认 500，范围 1..500
- 返回 `has_more/next_cursor/next_offset` 供续取

### 3.3 `search_content`

- `max_matches` 默认 200，最大 2000
- `max_candidates` 默认 4000，最大 20000
- 可配置 `output_budget_bytes`（钳制 2KB..4MB）
- 命中过多时按预算裁掉尾部 hits，并在 summary 给出 compact 提示

### 3.4 `execute_command`

- 默认输出护栏：
  - stdout 保留 head 24KB + tail 40KB
  - stderr 保留 head 8KB + tail 56KB
- 总预算缺省约 128KB；可配置 `output_budget_bytes`，钳制 4KB..4MB
- 截断标记：`...(truncated command output, omitted N bytes)...`

### 3.5 `web_fetch`

- 响应体字节上限默认 2MB（配置最终钳制 32KB..8MB）
- 内容字符默认 12000，cap 30000
- 截断时返回 `truncated=true`，并通过 `warning` 标明截断信息

### 3.6 浏览器工具（`browser`）

- `snapshot/read_page` 的 `max_chars` 受 `browser.limits.max_snapshot_chars` 限制（默认 40000）
- 截图下载受 `max_download_bytes` 限制（默认 100MB）

## 4. 编排层全局裁剪（Layer 2）

`finalize_tool_result` 会对所有工具结果做统一保护：

- 字符串：head 10000 + tail 10000，中间 `...(truncated)...`
- 数组：默认最多 128，超过后保留头尾并插入 `{"__truncated":true,"omitted_items":N}`
- 对存在分页/续取信号的对象，数组上限放宽到 500
- 如果结果整体仍超大，会包装为：
  - `truncated=true`
  - `original_chars`
  - `preview`
  - `truncation_reasons`
  - 可续取时带 `continuation_required/continuation_hint`

这一层会直接影响后续 `tool_result.data` 与模型侧观测基线。

## 5. 模型观测压缩（Layer 3，仅模型）

`model_observation` 使用 `to_compact_payload` 单独压缩：

- 总字符预算 20000
- 数组上限 32
- 搜索结果最多 10 hits，且每条命中内容只保留头部
- 读取文件内容只保留头部（默认 3200 chars），文件列表最多 8
- 密集数组会转成 `*_jsonl` + `*_count`
- 去掉大量噪声字段（如 `trace_id/user_round/model_round/meta` 等）

关键点：这层不用于前端“原始工具回包”展示，仅用于模型继续推理。

## 6. 监控/时间线裁剪（Layer 4）

- `monitor` 在 normal 画像下会按 `observability.monitor_payload_max_chars` 裁剪字符串，并追加 `...(truncated)`
- 默认 `monitor_payload_max_chars=0` 时走内置默认值（当前 4000）
- 时间线详情接口读取的是 monitor 记录，可能已被监控层裁剪

所以“时间线看到 truncated”不等于“模型输入一定被截断”。

## 7. 前端显示裁剪（Layer 5）

- 聊天工具流式缓冲、结构化预览、列表标题等都可能做显示裁剪
- 这类裁剪是 UI 体验优化，不会回写后端结果
- 当前前端工具详情已优先展示原始 `tool_result` 负载，`model_observation` 仅作为排障对照

## 8. 排障顺序（推荐）

1. 先看 `tool_result.data`：确认工具层/全局层是否已截断
2. 再看 `model_observation`：确认模型实际接收的压缩内容
3. 若时间线与上面不一致，再检查 monitor 画像与 `monitor_payload_max_chars`
4. 最后排查前端本地展示裁剪（预览/流式缓存/标题截断）

按照这个顺序可以快速定位：到底是“真实输入被截断”，还是“监控/前端显示被截断”。
