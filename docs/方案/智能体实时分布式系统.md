# wunder 智能体实时分布式系统

## 1. 定义“智能体实时分布式系统”

本文此前使用“网游化”作为类比术语，强调向标准网游服务器靠拢。  
现在统一将该方向正式命名为：**智能体实时分布式系统**。  
它不是指把 wunder 的网页端简单做成多人房间，也不是把前端交互做得像在线游戏界面，而是指：**把智能体平台的运行时、调度、实时同步、热点隔离、断线恢复、背压控制、观测体系，按标准网游服务器的思路重新设计。**

对 wunder 来说，第一优先级永远不是页面，而是智能体本身：

- 智能体线程必须稳定运行。
- 智能体并发必须可控且高吞吐。
- 智能体执行不能被 UI 同步链路拖慢。
- 智能体在多人围观、多人协作、热点空间下仍要保持响应性。
- 智能体的线程提示词、长期记忆注入、上下文边界必须严格稳定。

因此，本方案的核心立场是：

**房间、场景、同步、前端、社交、可视化，全部只是智能体运行时的投影层；不能反过来支配智能体运行时。**

## 1.1 当前已落地进展（2026-03-27）

当前代码已经完成第一批主轴收口，重点不是“概念命名”，而是把控制面和运行时边界真正切出来：

- `AppState` 已分层为 `kernel / projection / control`，`ThreadRuntime`、`MissionRuntime`、projection、presence、gateway 不再继续平铺堆在同一层。
- `ThreadRuntime` 已切到 route lease 模型：线程提交 lease 与 thread route lease 已成为单写者边界，旧的本地 `pending_sessions/running_threads` 语义已收敛。
- `MissionRuntime` 已开始接入 mission route lease：运行中的 `team_run` 现在拥有明确的 mission owner guard，为后续真正的 instance shard 化打基础。
- `BeeroomProjectionService` 已开始接入 projection route lease：活跃 beeroom stream 在创建 channel 时会登记 projection owner，为 projection shard 化和热点观测提供基线。
- 控制平面 `presence` 已拆成 `connection presence + projection watch presence`，四条 WS 入口都在连接建立、watch、取消、断开时同步登记和清理。
- 已新增 `/wunder/realtime/metrics`、`/wunder/realtime/sessions/{session_id}`、`/wunder/realtime/missions/{team_run_id}` 三个 inspection 接口，用于直接观察 route lease、watch 热点、session/thread/mission/projection 当前权威状态。

这意味着本方案已经不再停留在“建议这么设计”，而是开始形成可观测、可验证、可继续扩展的 agent-first 控制面基线。

## 2. 本方案要解决的问题

当前 wunder 网页版已经有明显的“多人在线系统”特征：

- beeroom 像多人协作场景。
- user_world 像社交大厅。
- chat session 像私有实例。
- WebSocket/SSE、事件回放、慢客户端告警、实时状态流已经初步存在。

但从后端结构上看，系统仍然更接近“会话驱动的智能体编排平台”，而不是“智能体优先的在线世界运行时”。

当前最主要的问题不是“没有实时推送”，而是下面四个结构性问题：

### 2.1 智能体执行平面不是主架构中心

从代码来看：

- `src/services/runtime/thread/runtime.rs`
- `src/services/swarm/runner.rs`
- `src/orchestrator/limiter.rs`

当前智能体执行仍以会话队列、锁轮询、后台扫描为主。  
这套机制保证了原型可用，但它更像“后台任务编排器”，而不是网游中的“角色/副本/AI 运行时内核”。

直接后果：

- 热点任务多时，调度延迟会上升。
- 同一空间内多个智能体的状态推进缺少强一致的权威时序。
- 任务、线程、群组、空间之间的边界不够清晰。

### 2.2 当前实时系统更像旁路观测，不是智能体主状态总线

现有实时基础已经不弱：

- `src/services/projection/beeroom.rs`
- `src/services/stream_events.rs`
- `src/api/chat_ws.rs`
- `src/api/user_world_ws.rs`
- `src/api/beeroom_ws.rs`
- `src/api/ws_helpers.rs`

这些模块已经提供：

- 事件流持久化
- cursor 回放
- lag recovery
- `slow_client(reason=queue_full_resume_required)`
- WS multiplex

但这些能力当前更多是“执行后的广播/回放层”，还不是“智能体执行状态的主总线”。  
也就是说，实时系统现在更偏旁路观察者，而不是主运行时骨架。

### 2.3 beeroom 更像看板，不像智能体在线战场

beeroom 是当前最像“在线场景”的部分，但它仍主要依赖：

- `src/api/beeroom.rs` 的即时聚合
- `frontend/src/stores/beeroom.ts` 的本地补丁
- `frontend/src/views/messenger/beeroomRealtimeSync.ts` 的健康检查式刷新

这意味着 beeroom 当前更像：

- 后端：多张表即时拼装的管理视图
- 前端：实时补丁 + 定时整包修正的仪表板

而不是：

- 后端：智能体场景的权威运行时
- 前端：只消费场景快照和增量复制的在线客户端

### 2.4 智能体认知态与共享态尚未被严格分层

你仓库里有一条非常关键的约束：

- 智能体线程的 system prompt 一旦首次确定必须冻结
- 长期记忆只允许在线程初始化时注入一次

这意味着 wunder 的智能体线程不是普通 UI 房间状态，而是**高价值、不可被多人在线状态反向污染的认知内核**。

如果未来只强调共享作用域权威，而没有把这层边界写死，就会出现严重问题：

- 房间实时状态误写回智能体线程 system prompt
- 共享场景变化污染 agent 认知上下文
- 前端同步逻辑反向影响 agent 行为

所以，真正的 agent-first 智能体实时分布式系统必须先做“认知态隔离”。

## 3. 代码现状的关键判断

基于当前代码，先给出几个必须明确的判断。

### 3.1 wunder 已具备实时分布式系统基础，但核心还没换轴

已经具备的基础：

- durable event log
- replay / resume
- WebSocket 多路复用
- 慢客户端恢复语义
- beeroom 专属实时 runtime
- 聊天会话的稳定流式事件链路

缺少的核心轴：

- 智能体内核分片
- mission 实例权威调度
- 智能体线程与共享空间的严格解耦
- 面向 agent 的执行预算与实时预算
- 将 UI/房间视为 projection，而不是智能体本体

### 3.2 当前最接近网游“主循环”的不是房间，而是任务执行链

如果以智能体为中心看系统，那么最接近“网游服务器主循环”的其实不是 beeroom 房间，而是：

- agent thread 的串行执行
- team_run / mission 的实例推进
- tool call / model call 的异步结果回传
- stream events 的事件流记录

也就是说，正确的主架构不是：

`Room -> Agent`

而应该是：

`Agent Kernel -> Mission Instance -> Projection Room`

### 3.3 beeroom 应该是智能体世界的可视化投影，不是智能体执行本体

beeroom 未来当然要有权威房间，但这个房间的职责应该是：

- 展示当前有哪些 agent
- 谁在忙、谁空闲
- mission 进展如何
- 哪些结果已经公开
- 多人用户当前看到的共享场景是什么

它不应该直接承担：

- agent 线程内部认知态
- 完整 token 级推理主路径
- 长期记忆注入逻辑
- 线程 system prompt 管理

这条边界一旦模糊，智能体平台就会被“房间架构”反向绑架。

## 4. 目标架构：Agent Kernel First

### 4.1 总体架构

未来的 wunder 网页版建议采用以下主结构：

```text
Browser / Desktop Client
  -> Realtime Gateway
    -> Agent Directory / Shard Router
      -> Agent Kernel Shard
        -> Thread Runtime
        -> Mission Instance Runtime
        -> Tool Execution Adapter
        -> Result Event Bus
        -> Projection Publisher
  -> HTTP Control API

Projection Publisher
  -> Beeroom Projection Room
  -> User World Projection Room
  -> Session Projection Room

Kernel / Projection common dependencies
  -> Event Log
  -> Snapshot Store
  -> Projection Store
  -> Presence Service
  -> Metrics / Trace / Replay
```

这里最重要的变化是：

- **Agent Kernel Shard** 成为系统核心。
- 房间不是核心本体，而是 Kernel 状态的可视化投影和协作订阅面。

### 4.2 三层核心模型

整个系统建议明确拆为三层：

| 层 | 角色 | 是否权威 | 说明 |
| --- | --- | --- | --- |
| Agent Kernel Layer | 智能体线程、任务执行、工具运行、推理状态 | 是 | wunder 的第一核心 |
| Mission / Instance Layer | 多智能体协同实例、任务编排、依赖推进 | 是 | 智能体之间协同的副本层 |
| Projection Room Layer | beeroom、user_world、session 可视化与协作态 | 否，或局部权威 | 面向用户和多人协作的共享投影层 |

这三个层次里，真正必须被强保护的是第一层。

### 4.3 五个平面

为了避免职责污染，建议把系统严格拆成五个 plane：

| Plane | 核心职责 | 典型节点 |
| --- | --- | --- |
| Kernel Plane | 线程、上下文、模型调用、工具执行、任务推进 | Thread Runtime、Mission Runtime |
| Projection Plane | 房间快照、增量状态、多人共享场景 | Beeroom Room、Social Room、Session View |
| Event Plane | 事件日志、回放、审计、恢复 | Stream Events、Replay Log |
| Control Plane | 管理接口、配置、冷启动、控制命令 | HTTP API、Admin |
| Persistence Plane | 快照、投影、历史记录、索引 | Postgres/SQLite |

后续所有重构都必须遵守：

- Kernel Plane 不依赖 Projection Plane 才能运行
- Projection Plane 可以丢、可以重建、可以延迟
- Kernel Plane 不能被 UI 背压拖垮

## 5. 核心原则：智能体优先的实时分布式系统约束

### 5.1 线程认知态必须冻结

任何“智能体实时分布式系统”改造都不能破坏以下约束：

- thread system prompt 只在初始化时确定
- 长期记忆只注入一次
- 后续房间同步、多人协作、界面事件不得反向改写线程 system prompt

这条必须写进架构，不是实现细节。

### 5.2 共享态和认知态必须分离

建议明确三类状态：

| 状态 | 是否共享 | 例子 |
| --- | --- | --- |
| 认知态 | 否 | prompt、memory bootstrap、当前工具计划、推理中间态 |
| 协作态 | 是 | mission 状态、任务归属、agent 忙闲、公开产物 |
| 表现态 | 是，但可丢 | 画布布局、动画、hover、typing、可见面板 |

房间层最多只应该持有“协作态 + 表现态”。  
认知态永远留在 Kernel Plane。

### 5.3 Token 主路径必须快于房间同步主路径

智能体平台的核心体验不是只有 UI 状态，而是：

- 首 token 速度
- 工具启动速度
- 中间结果反馈速度
- agent 接管速度

因此建议区分两条链路：

- **执行快路径**
  - token delta
  - tool start / tool result
  - model call lifecycle
- **投影同步路径**
  - beeroom summary
  - mission panel
  - group counters
  - social room updates

结论：

- Token 流不能被 room actor 批处理阻塞。
- room delta 可以合帧。
- token delta 只能轻度聚合，不能为了 UI 一致性牺牲首 token。

### 5.4 单线程单写者，单实例单裁决者

建议引入两类单写者：

- Thread Runtime 单写者
  - 某个 agent thread 的认知态只允许一个内核实例推进
- Mission Runtime 单裁决者
  - 某个 team_run / mission 的状态推进和任务归属只允许一个实例做权威裁决

这比“全局 session lock + 轮询”更符合在线游戏里“角色服/副本服”的模式。

### 5.5 UI 是投影，不是调度核心

无论 beeroom 页面多复杂，都必须遵守：

- 不以页面是否打开来决定 agent 是否继续执行
- 不以房间刷新是否成功来决定任务状态是否推进
- 不以某个面板的缓存一致性来决定线程权威状态

即：

**界面断了，智能体仍继续；界面恢复后只做重放与对齐。**

## 6. wunder 领域对象的实时分布式语义映射

这里要把 wunder 的对象重新按“智能体在线世界”来映射。

| wunder 对象 | 建议网游语义 | 核心定位 |
| --- | --- | --- |
| `agent thread` | 角色实例 / Character Runtime | 最小认知执行单元 |
| `session` | 私有战斗日志 / Private Combat Log | 用户与某智能体的一条交互实例 |
| `team_run` / `mission` | 副本实例 / Dungeon Instance | 多智能体协作运行时 |
| `team_task` | 实例内行为单元 / Action Slot | mission 的细粒度工作项 |
| `beeroom group` | 主城/作战大厅投影 | agent 世界的共享观察面 |
| `user_world conversation/group` | 社交房间/频道 | 用户协作与社交投影 |
| `stream_events` | 战斗日志 / Replay Log | 恢复、审计、续传 |
| `presence` | 在线与订阅态 | 网关与投影路由依据 |

重点不是把 `beeroom group` 当作唯一核心，而是把：

- `agent thread`
- `mission instance`

视为真正的运行时主体。

## 7. 关键子系统设计

## 7.1 Agent Directory / Shard Router

网游里玩家、角色、副本都要被路由到正确的 shard。  
wunder 未来也要先解决“哪个线程/任务/实例由哪个节点负责”。

建议新增 `AgentDirectoryService`，职责：

- 计算 `thread_id` 所属 kernel shard
- 计算 `mission_id` 所属 instance shard
- 记录当前 owner node 与 epoch
- 支持热点线程/热点实例迁移
- 为 Gateway、Control API、Worker 提供统一路由结果

建议至少维护以下逻辑主键：

- `thread_routes`
  - `thread_id`
  - `epoch`
  - `owner_node_id`
  - `lease_deadline`
- `mission_routes`
  - `mission_id`
  - `epoch`
  - `owner_node_id`
  - `lease_deadline`
- `projection_routes`
  - `realm`
  - `room_id`
  - `epoch`
  - `owner_node_id`

第一阶段可以先用 Postgres lease。  
只要 lease + epoch 做对了，后续再升级分布式协调组件即可。

## 7.2 Thread Runtime：智能体线程内核

这部分才是 wunder 真正的“角色服”。

### 7.2.1 Thread Runtime 的职责

- 维护单线程上下文
- 持有冻结后的 system prompt
- 持有单次注入后的 memory bootstrap
- 串行执行模型调用、工具调用、取消、重试
- 生成 thread event log
- 向外发布公开事件，不暴露内部 scratchpad

### 7.2.2 Thread Runtime 的内部状态

建议 Thread Runtime 的内存态最少包含：

- `thread_meta`
  - `thread_id`
  - `user_id`
  - `agent_id`
  - `epoch`
  - `created_at`
- `prompt_snapshot`
  - system prompt 冻结快照
  - memory bootstrap 摘要
- `execution_state`
  - `idle / queued / running / waiting_tool / cancelled / failed / completed`
- `active_call`
  - 当前模型调用
  - 当前工具调用
  - 当前 cancel token
- `message_tail`
  - 最近上下文窗口索引
- `public_projection_state`
  - 当前外部可见状态摘要

### 7.2.3 Thread Runtime 的单写者原则

一个线程在任意时刻只能由一个 runtime 推进：

- 不允许多个节点同时给一个 thread 发模型请求
- 不允许一个 thread 的状态既被 UI 驱动又被后台 worker 驱动
- 所有 thread command 都必须先路由到 thread owner

这能替代当前对 session lock 的大量轮询式等待。

## 7.3 Mission Runtime：多智能体副本实例

`team_run` / `mission` 在 agent-first 方案里应被当成“副本实例服”。

### 7.3.1 Mission Runtime 的职责

- 接收 `start_mission / claim_task / cancel_task / merge_result / finalize`
- 维护任务依赖、归属、超时、重试
- 协调多个 thread runtime 参与同一实例
- 汇总实例公共状态
- 将结果发布到 projection 层

### 7.3.2 为什么要把 Mission 独立成实例

当前 `src/services/swarm/runner.rs` 更像：

- 全局扫描器
- 全局队列驱动器
- 后台轮询调度器

这在原型期没有问题，但一旦要进入高并发多人协作，就会遇到三个问题：

- 热点 mission 与普通 mission 互相影响
- 任务推进与 UI 观察耦合
- 多智能体协作缺少清晰的实例边界

Mission Runtime 化之后，才更像网游里常见的：

- 主城和副本分离
- 副本内权威推进
- 副本外只看摘要和战报

### 7.3.3 Mission Runtime 的状态

建议至少包含：

- `mission_meta`
  - `mission_id`
  - `strategy`
  - `owner_user_id`
  - `epoch`
- `task_graph`
  - task 列表
  - 依赖关系
  - 超时与重试策略
- `assignment_state`
  - 哪个 agent 领取了哪个 task
  - 当前是否等待空闲 agent
- `runtime_budget`
  - token 预算
  - tool 并发预算
  - external IO 预算
- `public_summary`
  - 对外可见的 mission 进度与结果摘要

## 7.4 Projection Room：共享场景投影层

beeroom、user_world、session 页面未来都应该统一成 projection room。

### 7.4.1 Projection Room 的职责

- 订阅 kernel / mission 发布的公开事件
- 维护用户可见的共享状态
- 负责快照、增量、兴趣订阅、回放
- 为前端提供稳定的多人在线视图

### 7.4.2 Projection Room 不应该做什么

- 不负责 agent 认知态
- 不负责工具执行裁决
- 不负责模型调用生命周期的完整管理
- 不直接决定 task 归属

Projection Room 是“战场大屏”，不是“智能体大脑”。

### 7.4.3 beeroom 的新定位

beeroom 未来应当是：

- Agent World Lobby
- Mission Observatory
- Shared Command Surface

也就是：

- 用户在 beeroom 发出命令
- 命令进入 mission/thread runtime
- runtime 推进后把公开事件投影回 beeroom

这样 beeroom 看起来会像多人在线作战空间，但不会绑住智能体运行时。

## 7.5 Realtime Gateway

Gateway 继续负责：

- 统一连接
- 鉴权
- protocol negotiation
- multiplex
- ping/pong
- backpressure

但在 agent-first 架构里，Gateway 的路由目标不再只是 room，而是三类目标：

- `thread`
- `mission`
- `projection_room`

建议统一 envelope：

```json
{
  "type": "connect | join | command | event | replay | ack | resync_required | backpressure | cancel | ping | pong",
  "request_id": "req_xxx",
  "target_kind": "thread | mission | projection_room",
  "target_id": "xxx",
  "channel": "tokens | public_events | summary | mission_canvas | social | logs",
  "epoch": 12,
  "seq": 1024,
  "ack_seq": 1018,
  "payload": {}
}
```

### 7.5.1 关键点

- 同一浏览器连接可以同时看多个 projection room
- 同时 watch 自己的 thread 或 mission
- 同时接收 `public_events` 和 `tokens`
- token 通道与 summary 通道必须支持不同背压策略

## 7.6 事件日志与快照

当前 `src/services/stream_events.rs` 已经是很好的起点。  
但在 agent-first 方案里，需要从“session stream”扩展到“多层事件日志”。

建议分三类日志：

- `thread_events`
  - thread 级公开事件和恢复点
- `mission_events`
  - mission 级调度、归属、结果事件
- `projection_events`
  - 房间增量、摘要变化、社交事件

以及两类快照：

- `thread_snapshots`
  - 仅保存恢复执行所需的最小状态，不暴露内部 scratchpad
- `projection_snapshots`
  - 服务前端冷启动和重同步

### 7.6.1 事件分类

建议把事件按优先级分层：

| 事件层级 | 例子 | 是否必持久化 |
| --- | --- | --- |
| A 强一致执行事件 | message accepted、task claimed、tool finished、mission finalized | 必须 |
| B 可回放公开事件 | summary changed、progress updated、artifact ready | 建议 |
| C 临时表现事件 | typing、focus、hover、光标状态 | 不必 |

### 7.6.2 为什么不能所有 token 都写同一条主日志

如果所有 token、所有 panel update、所有 summary 变化都走同一条热日志，会有三个问题：

- 日志写放大
- replay 恢复过慢
- token 流被公开投影链路绑住

建议：

- token delta 优先走 thread 快路径
- 只对关键切点做 checkpoint
- 最终结构化结果再进入 projection

## 7.7 Presence 与兴趣管理

当前基线已经切到 `src/services/presence/`：`connection presence` 与 `watch presence` 分层落地，`AppState.control.presence` 负责在线态、订阅态与降级观测；后续需要继续把它扩展到多实例 directory / route lease。

在 agent-first 架构里，presence 不只是“用户在线没在线”，而是整个投影路由和资源降级基础。

建议至少区分：

- `connection_presence`
  - 用户在哪个 gateway、有几个连接、最后心跳
- `watch_presence`
  - 用户正在看哪些 thread / mission / room
- `control_presence`
  - 当前谁对某 mission 或某 projection 拥有操作权限

兴趣订阅建议做语义分层：

- `tokens`
- `public_events`
- `summary`
- `mission_canvas`
- `social`
- `logs`

隐藏页面默认只保留：

- `summary`
- `critical public_events`

这样能优先保护智能体执行预算，而不是把后台页的 UI 刷新当作最高优先级。

## 8. 并发、性能、实时性与稳定性设计

## 8.1 真正需要保护的热路径

agent-first 的情况下，最应该保护的热路径不是页面刷新，而是：

1. 用户命令进入 thread / mission runtime
2. runtime 接管并开始执行
3. 首 token / 首公开事件可见
4. 工具结果回流并继续推进

因此，优化顺序应当是：

1. thread command latency
2. mission scheduling latency
3. first token latency
4. public event latency
5. projection room summary latency
6. 非关键 UI 状态同步 latency

如果顺序反了，系统会变成“面板很顺滑，但智能体本身慢”。

## 8.2 单写者模型

建议采用三层单写者：

- 单 thread 单写者
- 单 mission 单裁决者
- 单 projection room 单汇总者

这三层并不冲突：

- thread 决定认知态和模型调用
- mission 决定实例任务和协作归属
- projection room 决定用户看到什么共享状态

### 8.2.1 这比当前模式强在哪里

当前系统在一些链路上仍依赖：

- 全局扫描
- session lock 轮询
- DB 读写后再刷新 UI

新的单写者模型可以把它改成：

- 命令路由到 owner
- owner 串行推进
- 生成公开事件
- projection 按需同步

## 8.3 背压策略：优先保智能体，后保面板

建议背压按优先级降级：

| 优先级 | 通道 | 背压时策略 |
| --- | --- | --- |
| P0 | cancel / tool result / mission finalize | 必达 |
| P1 | 首 token / 关键公开事件 | 强保 |
| P2 | 普通 token delta / progress delta | 轻度合并 |
| P3 | summary / panel diff | 强合并 |
| P4 | typing / hover / 非关键视觉态 | 直接丢弃 |

这条规则非常重要。  
智能体平台绝不能在压力上来时优先保住动画和面板，而牺牲执行主链路。

## 8.4 批量与合帧

建议只对 projection 层重度合帧，对 kernel 层轻度合帧：

- Thread token
  - 轻度字节窗/时间窗聚合
- Mission public events
  - 小批次合并
- Projection room summary
  - 每 `16ms ~ 33ms` 合帧
- 列表页摘要
  - 每 `50ms ~ 100ms` 合批

合帧位置错了，就会直接拖慢智能体的“活感”。

## 8.5 数据库使用原则

Postgres 对网页版的定位应该是：

- durable store
- route lease store
- snapshot/projection store
- history/query store

而不应该是：

- 高频实时广播队列
- 每个实时面板状态的权威内存替代品

### 8.5.1 当前代码的风险点

基于现状，有几个明显风险点：

- `src/api/beeroom.rs` 的 detail/list 聚合在热点 group 下会很重
- `src/api/user_world.rs` 的 700ms polling 会放大数据库读取
- `src/services/stream_events.rs` 当前 append 走单流锁 + 阻塞持久化，正确但热路径吞吐有限

### 8.5.2 新原则

- 热执行态只在 runtime 内存里
- projection summary 读 projection store
- replay 读 event log
- 冷详情才走历史数据拼装

## 8.6 分片与热点隔离

建议分三类 shard：

- `kernel shard`
  - 负责 thread runtime
- `instance shard`
  - 负责 mission runtime
- `projection shard`
  - 负责 room 汇总与多人 fanout

初期这些 shard 可以部署在同一个进程中，只要抽象边界先建立即可。  
后续再按流量和热点拆分到多节点。

### 8.6.1 热点隔离原则

- 热 mission 不影响普通 session thread
- 热 beeroom 不影响后台 thread 执行
- 社交房间风暴不影响模型调用主链路

这才是网游式“世界服/副本服/聊天服”分治的价值。

## 9. 实时体验设计：智能体平台视角

## 9.1 最重要的不是房间流畅，而是“智能体在工作”

用户对智能体平台最强的感知不是列表刷新，而是：

- 发出命令后有没有被接管
- 智能体有没有开始处理
- 是哪个智能体在处理
- 处理到哪一步
- 失败后能不能恢复

因此建议定义以下核心体验指标：

- `Command Accepted`
  - 用户动作被 thread/mission runtime 接管
- `Agent Claimed`
  - 某 agent 已领取任务
- `First Token Visible`
  - 第一个有效 token 或结构化公开事件可见
- `Mission Progress Visible`
  - 第一个任务推进事件可见
- `Recovery Continue Visible`
  - 断线后恢复到连续状态的时间

## 9.2 beeroom 应该呈现什么

beeroom 不应只是显示静态任务列表，而应优先呈现：

- 哪些 agent 正在运行
- 当前 mission 的权威状态
- 哪个 agent 正在占用什么资源
- 是否在等待工具/审批/用户输入
- 哪些结果已经公开，哪些还在内部处理中

这样用户会真正感觉自己在看一个“活着的智能体世界”，而不是看一个后台管理页。

## 9.3 断线恢复原则

断线恢复要按三层恢复：

1. thread watch 恢复
2. mission watch 恢复
3. projection room 恢复

恢复顺序也应按优先级：

- 先恢复 thread / mission 的关键公开事件
- 再恢复投影面板
- 最后恢复低优先级视觉态

这与“智能体优先”的原则完全一致。

## 10. 分阶段迁移路线图

本次重构不能大爆炸，建议按以下阶段推进。

## P0：基线与口径统一

目标：

- 先把当前系统按 agent-first 口径量化。

动作：

- 增加 `thread_accept_latency`
- 增加 `mission_schedule_latency`
- 增加 `first_token_latency`
- 增加 `public_event_latency`
- 增加 `projection_resync_total`
- 扩展 `src/bin/beeroom_projection_baseline.rs` 为通用 realtime 基线工具

验收：

- 能清楚看到当前瓶颈是线程、任务实例、还是投影层

## P1：抽出 Thread Runtime 抽象

目标：

- 先把智能体线程内核从当前会话执行链中明确出来。

动作：

- 在 `src/services/` 下新增 `runtime/thread/`
- 明确 thread owner、epoch、public event、checkpoint
- 把 system prompt 冻结和 memory bootstrap 约束写入 runtime

验收：

- 任一线程都能明确被一个 runtime owner 接管

## P2：Mission Runtime 实例化

目标：

- 把 `team_run` 从全局 runner 体系演进为副本实例。

动作：

- 为 mission 建立实例 runtime
- task claim / result merge / finalize 进入实例裁决链
- 逐步减少对全局 scan loop 的依赖

验收：

- 热 mission 的推进不再主要依赖全局后台扫描

当前状态：

- 已完成 `MissionRuntime` 主语收口，并为 active mission 补上 route lease owner。
- 尚未完成真正的 per-mission actor / shard 化；当前仍保留 scan loop 作为调度基线。

## P3：Projection Room 化

目标：

- 把 beeroom / user_world / session 页面统一成 projection room。

动作：

- 引入 `projection room snapshot + delta + replay`
- `frontend/src/stores/beeroom.ts` 逐步改为消费 room snapshot/delta
- `frontend/src/views/messenger/beeroomRealtimeSync.ts` 降级为 transport fallback

验收：

- steady state 下页面主要靠 projection delta 驱动

## P4：Directory / Presence / Route Lease

目标：

- 为多实例部署建立 agent-first 路由基础。

动作：

- 新增 thread / mission / projection route lease
- 新增 presence 双层模型
- Gateway 支持按 target kind 路由

验收：

- 多节点下 thread 与 mission 不会双写

当前状态：

- `presence` 双层模型已落地。
- `RouteLeaseService` 已覆盖 `thread / mission / projection` 三类 target 的 owner/epoch 基线。
- `realtime_world` inspection API 已可直接观察 session 与 mission 的 route/watch 快照。
- 仍未完成持久化 lease 表结构与真正的多节点 gateway 路由切流。

## P5：执行快路径与投影慢路径彻底分离

目标：

- 确保 UI 再繁忙也不拖慢智能体执行。

动作：

- token 与关键执行事件走快路径
- summary 与面板事件走慢路径
- 背压优先丢弃 P3/P4 级事件

验收：

- 压力下首 token 和关键任务推进仍稳定

## P6：统一客户端运行时

目标：

- 前端统一 watch/replay/backpressure 逻辑。

动作：

- 新增 `frontend/src/realtime/agentWorld/`
- 统一 thread、mission、projection room runtime
- 页面层只做 selector 和 UI state

验收：

- chat / beeroom / user_world 共享统一 runtime 语义

## P7：压测、故障演练、容量验收

目标：

- 达到可长期维护的在线系统级别。

动作：

- 热 thread 压测
- 热 mission 压测
- 热 beeroom fanout 压测
- reconnect storm 压测
- route lease 抖动与 owner 切换演练

验收：

- 有清晰的容量边界、恢复时间和降级策略

## 11. 指标与验收

建议至少定义以下 agent-first SLO：

| 指标 | 建议目标 |
| --- | --- |
| thread command accepted p95 | `< 80ms` |
| mission scheduling p95 | `< 120ms` |
| first token visible p95 | `< 300ms` |
| tool result apply p95 | `< 150ms` |
| projection room snapshot join p95 | `< 200ms` |
| replay gap 500 条恢复 p95 | `< 500ms` |
| critical event delivery under backpressure | 不丢 |

这些指标里，前三项最关键。  
因为它们直接决定 wunder 是“智能体平台”还是“面板平台”。

## 12. 风险与防御

| 风险 | 说明 | 防御策略 |
| --- | --- | --- |
| 房间架构反客为主 | projection 反向支配智能体执行 | 明确 Kernel Plane 独立于 Projection Plane |
| 线程认知态污染 | room/public state 写回 thread prompt | system prompt 冻结，memory 仅初始化注入 |
| token 路径被 summary 合帧拖慢 | UI 为了顺滑牺牲首 token | token 与 projection 双通道 |
| 多节点双写 | thread 或 mission 同时被两个节点推进 | lease + epoch + owner fencing |
| 日志写放大 | 所有 token 和 UI diff 全落盘 | 事件分层，token checkpoint 化 |
| 热 mission 抢占全部资源 | 全局 scan loop 或单队列热点化 | mission runtime、budget、shard 隔离 |

## 14. 最终结论

如果 wunder 要走“智能体实时分布式系统”，正确的重构方向不是先把 beeroom 做成更强的房间，而是先把：

- 智能体线程
- mission 实例
- 执行快路径

做成真正的在线运行时核心。

然后再把：

- beeroom
- user_world
- session 页面

统一收敛为这些运行时核心的共享投影层。

最终要达到的状态是：

- 智能体像网游里的角色实例一样稳定存在
- mission 像副本实例一样独立推进
- beeroom 像主城和作战大厅一样实时可见
- 用户断线、切页、多人围观都不会打断智能体本体运行
- 背压和热点优先保护智能体执行，而不是优先保护 UI

一句话概括：

**wunder 的智能体实时分布式系统，不应该是“页面像网游”，而应该是“智能体运行时像网游服务器”。**
