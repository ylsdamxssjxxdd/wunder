# 实时性优化方案（蜂群 / Beeroom）

## 1. 背景与问题

当前用户侧蜂群页面存在两类实时性问题：

1. 画布状态滞后：TeamRun/Task 状态变化后，节点与连线无法及时反馈运行态。  
2. 中栏蜂群列表滞后：母蜂触发的在运行任务与活跃成员统计不能快速出现在列表。

根因拆解：

- 实时通道仅覆盖 `chat_message/chat_cleared`，缺少 `team_*` 任务态势事件。  
- 蜂群页使用轮询兜底，实时链路缺乏“事件优先 + 增量更新 + 回源校准”的统一策略。  
- 大量状态刷新依赖全量接口回拉，缺少前端 Store 的事件增量应用层。

---

## 2. 目标与验收指标

### 2.1 目标

1. 构建蜂群“事件优先、轮询兜底”的实时链路。  
2. 将画布与中栏从“被动刷新”升级为“事件触发 + 增量更新 + 校准回源”。  
3. 在保证稳定性的前提下，降低无效请求与前端重绘抖动。

### 2.2 指标（上线验收）

1. 蜂群任务状态变更到前端画布可见：P95 <= 1.5s。  
2. 蜂群列表运行中任务数/活跃成员数变更到可见：P95 <= 2.0s。  
3. WS 正常时，蜂群页轮询请求量较改造前下降 >= 40%。  
4. WS/SSE 中断时，60s 内自动恢复并完成状态一致性补齐。

---

## 3. 总体方案

### 3.1 事件优先链路

后端在 TeamRun 生命周期关键节点向 `beeroom_realtime` 广播 `team_*` 事件：

- `team_start`
- `team_task_dispatch`
- `team_task_update`
- `team_task_result`
- `team_merge`
- `team_finish`
- `team_error`

事件通过现有通道下发：

- WS：`/wunder/beeroom/ws`（主通道）  
- SSE：`/wunder/beeroom/groups/{group_id}/chat/stream`（兜底）

### 3.2 前端实时消费策略

蜂群画布组件统一消费 `chat_* + team_* + sync_required`：

1. `chat_*` 走原有协作对话增量渲染。  
2. `team_*` 先进入 `beeroom` store 做增量 patch（任务/成员/摘要统计），实现即时视觉更新。  
3. 同步启动节流回源校准（`loadActiveGroup`），确保事件丢失或字段不全时最终一致。  
4. `sync_required` 强制触发回源补齐。

### 3.3 轮询兜底与节流

- 保留蜂群专用实时同步器作为兜底。  
- 对 `groups` 拉取采用热态/空闲分档最小间隔，避免高并发下无意义全量刷新。  
- 单飞执行（single-flight）防止并发重入导致抖动。

## 4. 详细节点（含落地范围）

## 节点 N1：后端任务态势事件广播（已落地）

### 目标

把 TeamRun 关键状态同步到蜂群实时通道，打通“任务态势 -> 前端”链路。

### 变更

1. `TeamRunRunner` 注入 `BeeroomRealtimeService`。  
2. `emit_team_event` 在写 monitor 事件的同时，向 `beeroom_realtime` 广播 `team_*`。  
3. 修正事件 payload 的 `hive_id`：由固定值改为真实 `run.hive_id`。
4. `/wunder/chat/team_runs` 创建/取消路径同步接入 `beeroom_realtime`，保证 API 触发与 Runner 触发路径事件语义一致。

### 文件

- `src/services/swarm/runner.rs`
- `src/services/beeroom_realtime.rs`
- `src/core/state.rs`

### 风险控制

1. 保持 monitor 事件逻辑不变，新增广播为旁路能力。  
2. 事件推送失败不影响主执行链路（异步发送，不阻塞 TeamRun）。

---

## 节点 N2：前端 Store 增量更新（已落地）

### 目标

在收到 `team_*` 后即时更新蜂群任务与成员状态，降低对全量回拉的依赖。

### 变更

1. `beeroom` store 新增 `applyRealtimeEvent(groupId, eventType, payload)`。  
2. 支持增量维护：
   - 任务（mission）生命周期状态
   - 子任务（task）状态/结果
   - 组内成员 `idle/active_session_total`
   - `running_mission_total/active_agent_total/latest_mission`
3. 保留最终一致回源机制（由组件节流触发）。

### 文件

- `frontend/src/stores/beeroom.ts`

### 风险控制

1. 增量 patch 仅作用于当前 active group，避免跨组误写。  
2. 每次 patch 后仍允许回源纠偏，避免事件丢失导致长期偏差。

---

## 节点 N3：前端实时消费与回源校准（已落地）

### 目标

把 `team_*` 纳入蜂群画布实时监听，做到“先快后准”。

### 变更

1. 画布组件 WS/SSE 同时监听 `team_*` 事件。  
2. 收到 `team_*`：
   - 先调用 store 增量更新（即时 UI 反馈）
   - 再节流触发 `refresh`（回源校准）  
3. `sync_required` 场景强制回源。  
4. 组件卸载/切组时清理定时器，避免泄漏。

### 文件

- `frontend/src/components/beeroom/BeeroomMissionCanvas.vue`

### 风险控制

1. 回源触发做节流（默认 700ms），防止任务高频事件造成接口风暴。  
2. 仍保留轮询兜底链路，WS/SSE 异常不阻断可用性。

---

## 节点 N4：文档与接口说明同步（本次已同步）

### 变更范围

1. API 文档补充 `beeroom/ws` 与 SSE 的 `team_*` 事件类型说明。  
2. 设计文档与系统介绍同步“任务态势实时事件 + 前端增量 patch + 回源校准”策略。  
3. 本方案文档沉淀实施节点与后续扩展计划。

---

## 节点 N5：母蜂工具链实时事件补齐与任务汇总回写（已落地）

### 目标

解决 `agent_swarm send/batch_send` 路径“只写库不广播、run 汇总不及时更新”导致的中栏与画布滞后。

### 变更

1. 新增 `src/services/tools/swarm_realtime.rs`，统一处理：
   - `team_start/team_task_dispatch/team_task_update/team_task_result/team_finish/team_error` 广播；
   - monitor 事件与 beeroom_realtime 事件双写；
   - `team_run` 汇总状态（`running/completed/failed/cancelled`）与统计字段同步回写。
2. `agent_swarm_send`、`agent_swarm_batch_send` 接入上述模块：
   - 创建 run/task 后即时下发 `team_start/team_task_dispatch`；
   - 任务状态变化后下发 `team_task_update`，终态补 `team_task_result`；
   - run 达终态时按失败/成功发 `team_error + team_finish` 或 `team_finish`。
3. `ToolContext` 注入 `beeroom_realtime`，工具执行链可直接复用蜂群实时通道。

### 文件

- `src/services/tools/swarm_realtime.rs`（新增）
- `src/services/tools.rs`
- `src/services/tools/context.rs`
- `src/orchestrator/mod.rs`
- `src/orchestrator/execute.rs`
- `src/core/state.rs`
- `src/ops/performance.rs`

### 风险控制

1. 事件发送依旧异步，不阻塞工具主执行链路。  
2. run 汇总状态由任务快照推导，避免“只更新 task 不更新 run”造成长期不一致。

---

## 5. 后续优化节点（待排期）

1. 观测闭环：增加“事件产生时间 -> 前端渲染完成时间”埋点，输出 P50/P95。  
2. 后端摘要缓存：`/beeroom/groups` 提供更轻量 summary 路径，进一步降低列表刷新开销。  
3. 画布渲染细化：继续从整图刷新向局部节点/边更新演进（减少高频重排）。

---

## 6. 回滚策略

若实时链路出现异常，可按以下顺序快速回退：

1. 前端关闭 `team_*` 增量消费入口，仅保留回源刷新。  
2. 保留蜂群专用同步器，恢复“轮询主导”模式。  
3. 后端保留 monitor 事件，关闭 `beeroom_realtime` 的 `team_*` 广播。
