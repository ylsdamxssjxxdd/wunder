# 外部系统嵌入团队接入指南

## 1. 适用范围

本指南用于团队成员将第三方业务系统接入 wunder 用户侧聊天页（iframe 或新窗口），并满足以下约束：

- 外链仅允许进入预制智能体（管理员配置）。
- 若用户已有同名自建智能体，优先进入用户自建版本。
- 用户不存在时自动创建。
- 不在前端暴露 `WUNDER_EXTERNAL_AUTH_KEY`。

---

## 2. 接入目标（一句话）

外部系统后端调用 `POST /wunder/auth/external/launch`，拿到 `entry_path`，前端用该地址打开 wunder 嵌入聊天页。

---

## 3. 双方职责分工

### 3.1 wunder 侧（我们）

- 配置 `security.external_auth_key`。
- 配置 `security.external_embed_preset_agent_name`。
- 保证 `POST /wunder/auth/external/launch` 可访问。

### 3.2 外部系统侧（对方）

- 后端新增一个“获取 wunder 入口”接口（推荐）。
- 在该后端接口中调用 wunder 的 `external/launch`。
- 前端拿到返回地址后，以 iframe `src` 或 `window.open` 打开。

---

## 4. 先决配置

## 4.1 wunder 配置

```yaml
security:
  external_auth_key: ${WUNDER_EXTERNAL_AUTH_KEY:-} # 未显式配置时自动回退到 security.api_key
  external_embed_preset_agent_name: 文稿校对
```

说明：

- `external_auth_key` 为空时会自动回退到 `security.api_key`，因此外链鉴权默认启用；如需独立轮换，建议显式配置 `WUNDER_EXTERNAL_AUTH_KEY`。
- `external_embed_preset_agent_name` 必须能匹配 `user_agents.presets` 里的某个预制智能体名称。

## 4.2 网关与域名

- 外部系统后端必须能访问 wunder API 域名。
- 外部系统前端必须能访问 wunder 前端域名（用于 iframe）。
- 如果网关设置了 `X-Frame-Options` 或 CSP `frame-ancestors`，需放行外部系统域名。

---

## 5. 接口契约（外部系统后端调用）

### 5.1 请求

- 方法：`POST`
- 路径：`/wunder/auth/external/launch`
- Body：

```json
{
  "key": "WUNDER_EXTERNAL_AUTH_KEY",
  "username": "oa_u1001",
  "unit_id": "sales",
  "password": "optional"
}
```

字段说明：

- `key`：必填，只能在后端保存和使用。
- `username`：必填，建议加业务系统前缀避免重名（如 `oa_u1001`）。
- `unit_id`：可选，传入可自动同步用户单位。
- `password`：可选。推荐不传；仅用于兼容老链路。

### 5.2 返回

```json
{
  "data": {
    "code": "wundc_xxx",
    "expires_at": 1760000000.0,
    "entry_path": "/app/embed/chat?wunder_code=...&agent_id=...&embed=1",
    "agent_id": "agent_xxx",
    "agent_name": "文稿校对",
    "created": false,
    "updated": true
  }
}
```

关键字段：

- `entry_path`：直接拼接 wunder 前端域名后打开。
- `created`：是否新建用户。
- `updated`：是否更新了既有用户资料（如 unit）。

## 6. 外部系统最小改动示例

### 6.1 外部系统后端（Node/Express 示例）

```ts
import express from "express";
import fetch from "node-fetch";

const app = express();

app.get("/api/wunder-entry", async (req, res) => {
  const loginUser = req.user; // 外部系统已登录用户
  const resp = await fetch("https://wunder-api.example.com/wunder/auth/external/launch", {
    method: "POST",
    headers: { "content-type": "application/json" },
    body: JSON.stringify({
      key: process.env.WUNDER_EXTERNAL_AUTH_KEY,
      username: `oa_${loginUser.id}`,
      unit_id: loginUser.deptId || undefined
    })
  });

  const data = await resp.json();
  if (!resp.ok) {
    return res.status(resp.status).json(data);
  }

  return res.json({
    entry_url: `https://wunder-frontend.example.com${data.data.entry_path}`
  });
});
```

### 6.2 外部系统前端（iframe）

```html
<iframe id="wunder-frame" style="width:100%;height:100%;border:0;"></iframe>
<script>
  fetch("/api/wunder-entry")
    .then((r) => r.json())
    .then((r) => {
      document.getElementById("wunder-frame").src = r.entry_url;
    });
</script>
```

## 7. 业务规则确认清单

- [ ] 是否已配置 `security.external_embed_preset_agent_name`。
- [ ] 预制智能体名称是否与配置完全一致（含空格）。
- [ ] 外部系统 username 是否已做命名空间前缀（避免多系统重名）。
- [ ] 是否确认“同名用户智能体优先于预制模板”。
- [ ] 是否允许用户首次访问时自动建号。

---

## 8. 联调步骤（建议）

1. 管理员在 wunder 侧确认预制智能体已配置。
2. 外部系统后端本地调用 `external/launch`，检查返回中有 `entry_path`。
3. 浏览器直接打开 `https://<wunder-frontend-host><entry_path>`，确认可进入聊天页。
4. 检查页面是否隐藏左栏和中栏（仅保留聊天主区）。
5. 用“新用户”和“已有同名自建智能体用户”各测一次。

---

## 9. 常见错误与处理

- `401 api_key_invalid`：`key` 错误或有空白字符。
- `403 external auth disabled`：`security.external_auth_key` 与 `security.api_key` 同时未配置。
- `403 external embed preset agent is not configured`：未配置预制智能体名称。
- `404 preset agent 'xxx' not found`：配置名与预制列表不匹配。
- `401 code expired`：`entry_path` 使用过慢（一次性码默认 60 秒）。

---

## 10. 安全基线

- `WUNDER_EXTERNAL_AUTH_KEY` 只放服务端环境变量，不进入前端、URL、日志。
- 外部系统到 wunder 使用 HTTPS。
- 对 `username`、`unit_id` 做审计日志，避免记录 `key` 和 `wunder_code`。
- 若外部系统做页面分享，禁止分享包含 `entry_path` 的原始链接。

---

## 11. 验收标准

- 用户从外部系统点击入口后，无需再次登录即可进入 wunder 聊天页。
- 进入的是预制智能体或同名用户智能体（优先用户同名）。
- 用户不存在时自动创建，且流程对最终用户无感。
- 安全扫描中不出现前端泄露 `WUNDER_EXTERNAL_AUTH_KEY`。
