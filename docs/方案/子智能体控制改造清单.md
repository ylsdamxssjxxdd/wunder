# 子智能体控制改造清单

## 0. 文档定位
- 本文不是抽象愿景，而是 wunder 当前这轮子智能体控制改造的实施清单与交付节点。
- 对齐目标是 `C:\Users\sjxx\Desktop\参考项目\codex-main` 的“子智能体 control plane 能力形态”，不是照搬其内部线程实现。
- wunder 保持自身架构约束：
  - 线程首次确定后的 system prompt 必须冻结
  - 长期记忆只允许在线程初始化时注入一次
  - 不新增一套平行于现有会话体系的 rollout/thread 主模型
  - 子智能体能力继续服务于“父会话内协作”，不吞并 `agent_swarm`

---

## 1. 这次改造的完成目标

### 1.1 目标结果
- `subagent_control` 从单点派生工具升级为完整控制面。
- 父会话可以一次派发多个子智能体，并按批次追踪、等待、关闭、恢复、中断。
- 子智能体状态不再只依赖零散事件，而是具备稳定查询对象：
  - `session_id`
  - `run_id`
  - `dispatch_id`
- 前端工作流区可以稳定显示子智能体批次与单项状态。

### 1.2 当前已对齐的关键能力
- 可 `spawn`
- 可 `send`
- 可 `list/history`
- 可 `batch_spawn`
- 可 `status`
- 可 `wait`
- 可 `interrupt`
- 可 `close`
- 可 `resume`
- 可通过流式事件恢复与回放协作过程

### 1.3 本轮明确不做
- 不引入新的 `subagent_dispatches` / `subagent_dispatch_items` 主表。
- 不把 `agent_swarm` 合并进 `subagent_control`。
- 不在本轮引入 DAG 编排、优先级调度器、复杂重试策略。
- 不允许 `resume/close` 改写线程冻结后的 system prompt。

## 2. 实际落地方案

### 2.1 控制面入口
- 工具名：`subagent_control`
- 当前动作集合：
  - `list`
  - `history`
  - `send`
  - `spawn`
  - `batch_spawn`
  - `status`
  - `wait`
  - `interrupt`
  - `close`
  - `resume`

### 2.2 多子智能体调度语义
- `batch_spawn` 一次接收 `tasks[]`，逐项创建子会话并立即派发执行。
- 每批调度生成稳定 `dispatch_id`，后续 `status/wait/interrupt/close/resume` 可按批操作。
- 每个任务保持独立：
  - 子会话：`session_id`
  - 运行记录：`run_id`
- 批量等待与批量状态聚合统一通过 `dispatch_id` 完成，不额外引入新的批次表。

### 2.3 状态控制语义
- `status`
  - 支持按 `runId/runIds/sessionId/sessionIds/dispatchId/parentId` 查询
  - 未显式传任何目标时，默认落到当前父会话的子会话集合
- `wait`
  - 支持按 `run/session/dispatch/parent` 等待
  - 返回聚合结果与单项快照
- `interrupt`
  - 基于 `monitor.cancel(session_id)` 发起取消
- `close`
  - 仅切换子会话状态为 `closed`
  - 如目标当前仍在运行，可先走 monitor cancel
- `resume`
  - 仅把子会话状态改回 `active`
  - 不重建线程，不重写 system prompt，不重复注入长期记忆
- `cascade=true`
  - 递归作用于目标子树

## 3. 数据模型决策

### 3.1 复用现有结构
- `chat_sessions`
  - 继续承载父子会话树关系
- `session_runs`
  - 继续承载每次子智能体运行账本
- `monitor`
  - 继续承载 loaded runtime 实时状态
- `stream_events`
  - 继续承载流式事件持久化与回放

### 3.2 本轮新增字段
- `session_runs.dispatch_id`
  - 用于批次追踪
- `session_runs.run_kind`
  - 当前至少区分 `subagent` / `swarm`
- `session_runs.requested_by`
  - 当前用于标记来源，例如 `subagent_control`

### 3.3 明确不新增的结构
- 不新增独立 dispatch 主表。
- 不新增独立 dispatch item 表。
- 不复用 `team_runs/team_tasks` 强行承载子智能体批次，因为它们属于蜂群协作语义，不属于父会话内子线程控制语义。

## 6. 关键边界

### 6.1 必须保持的语义
- 子线程 system prompt 一旦首次物化后不可被 `resume` 改写。
- 长期记忆只允许在线程初始化时注入一次。
- token 统计记录上下文占用量，不混入总消耗量。
- 子智能体控制优先走现有会话树与流式事件链路，不另起炉灶。

### 6.2 本轮修正过的易错点
- `status` 默认父会话选择器只在“未显式传任何目标”时生效，避免用户传了 `runId/sessionId/dispatchId` 后仍被当前父会话结果污染。
- 批量等待结果会把启动失败项并入最终聚合结果，而不是只统计成功派发的运行项。
- `close/resume` 只做状态切换，不做线程再初始化。

## 7. 验收标准
- 后端存在独立的 `subagent_control` 模块，而不是继续堆在 `src/services/tools.rs`。
- `subagent_control` 可完成多子智能体派发与按批次控制。
- `session_runs` 可按 `dispatch_id` / `parent_session_id` / `session_id` 查询。
- `subagent_*` 事件可进入前端工作流视图。
- Rust 侧 `cargo check --release` 通过。
- 子智能体关键边界存在单测覆盖。
