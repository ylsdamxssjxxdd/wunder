# 子智能体控制改造清单

## 0. 文档说明
- 本文是 `docs/方案/子agent落地方案.md` 的升级版执行清单，目标不是只补一个 `spawn` 工具，而是把子智能体能力升级为可调度、可观察、可恢复的正式控制层。
- 本文默认保持 `agent_swarm` 的蜂巢/业务协作定位不变；新增能力聚焦“同一父会话下的内部子智能体协作”。
- 本文明确要求把现有堆积在 `src/services/tools.rs` 的子会话逻辑拆出，避免继续向维护态文件追加复杂分支。

---

## 1. 改造目标

## 1.1 最终结果
- `subagent_control` 从四动作工具升级为完整控制面：`list/history/spawn/send/status/wait/interrupt/close/resume/batch_spawn`。
- 父会话不仅能派生一个子智能体，还能一次调度多个子智能体并收敛结果。
- 子智能体从“临时子会话”升级为“可持续追踪的协作对象”，具备稳定标识、状态、运行记录、事件流和前端卡片。
- 前端从“事件日志视角”升级到“协作工作台视角”，至少能看见：
  - 哪些子智能体已派生
  - 当前谁在运行/等待/失败/完成
  - 哪个子智能体对应哪个子线程
  - 批量派发的整体进度和汇总结果

## 1.2 对齐基线
- 对齐目标不是照搬 codex-main 内部线程模型，而是对齐其关键行为：
  - 可 spawn
  - 可 send
  - 可 wait
  - 可 interrupt
  - 可 close
  - 可 resume
  - 可在历史中重建协作调用
  - 可由前端稳定展示协作状态
- wunder 内部仍以 `chat_sessions + session_runs + stream_events + monitor` 为核心，不强制改造成另一套 `thread_id + rollout` 架构。

## 1.3 非目标
- 不把 `agent_swarm` 删除或并回 `subagent_control`。
- 不在首期把子智能体调度扩展到跨用户、跨蜂巢、跨渠道。
- 不在首期引入复杂 DAG 编排器；先支持有限策略的多子智能体调度。

## 2. 现状问题

## 2.1 已有能力
- 已有 `subagent_control(action=list|history|send|spawn)`。
- 已有 `thread_control` 管理会话树和主线程。
- 已有 `session_runs` 记录子运行状态。
- 已有 `subagent_announce` 回写父会话。
- 已有 `agent_threads/agent_tasks` 与 `team_runs/team_tasks` 的持久化模型，可借鉴其队列与聚合思路。

## 2.2 现状缺口
- `subagent_control` 还只是“会话工具”，不是“控制层”。
- 没有正式的 `status/wait/interrupt/close/resume` 动作。
- 没有“多个子智能体并行派发”的统一入口。
- `subagent_announce` 只有父会话回写，没有前端专门模型与稳定投影。
- 没有父会话视角的“调度批次”对象；多个子智能体同时运行时缺少统一的批量状态。
- 文档里提到的 `allowlist/subagent_max_active` 等策略还没有完整实现闭环。

## 3. 领域边界与核心原则

## 3.1 与 `thread_control` 的边界
- `thread_control` 负责线程树结构：
  - create
  - switch
  - back
  - archive
  - restore
  - set_main
- `subagent_control` 负责子智能体运行控制：
  - 派生
  - 发送任务
  - 等待结果
  - 中断
  - 关闭
  - 恢复
  - 批量调度

## 3.2 与 `agent_swarm` 的边界
- `subagent_control`：
  - 作用域是“当前父会话内部”
  - 默认亲和当前父会话上下文
  - 子智能体天然属于父会话树
  - 用于深挖、校验、并行分工、快速子任务
- `agent_swarm`：
  - 作用域是“同蜂巢/同业务组”
  - 偏长期工蜂、业务编排、蜂群协作
  - 适合独立智能体之间的群体任务

## 3.3 关键原则
- 子智能体的系统提示词冻结规则必须继承现有约束：
  - 首次确定后冻结
  - 长期记忆只允许线程初始化时注入一次
  - `resume` 不得重写线程 system prompt
- token 统计继续记录上下文占用量，不记录总消耗量。
- 通讯优先走 WS，SSE 为兜底；所有新增子智能体状态事件必须进入统一流式事件链路。
- 后端结构必须拆模块，不能继续把子智能体能力堆到 `src/services/tools.rs`。

## 4. 目标能力模型

## 4.1 单子智能体控制
- `spawn`：创建一个子智能体并立即启动任务。
- `send`：向已存在的子智能体继续发送输入。
- `status`：读取一个或多个子智能体的当前状态。
- `wait`：等待一个或多个子智能体进入终态。
- `interrupt`：中断子智能体当前轮执行。
- `close`：关闭子智能体及其未显式脱离的子树。
- `resume`：从已存在子线程恢复运行态，并允许继续 `send`。
- `history`：查看指定子智能体会话历史。
- `list`：列出当前父会话下的子智能体及最近状态。

## 4.2 多子智能体调度
- `batch_spawn`：一次性派发多个子智能体任务。
- `wait` 支持等待批次内任一完成或全部完成。
- `collect` 不单独暴露动作，首期由 `wait` 返回聚合结果；后续如需要再拆。
- 支持三种首发调度策略：
  - `parallel_all`：全部并发启动，等待全部完成
  - `first_success`：全部并发启动，任一成功即可提前收敛
  - `review_then_merge`：先并发产出，再由父会话继续汇总
- 每个批次都有稳定的 `dispatch_id`，便于前端和 API 查询。

## 4.3 协作对象模型
- 稳定对象 1：`child_session_id`
  - 作为子智能体稳定标识
  - 用于 `send/history/resume/close/interrupt`
- 稳定对象 2：`run_id`
  - 表示某次具体运行
  - 用于 `wait/status`
- 稳定对象 3：`dispatch_id`
  - 表示一次多子智能体调度批次
  - 用于批量查询、取消和前端面板聚合

## 10. 调度策略设计

## 10.1 首期支持的多子智能体策略
- `parallel_all`
  - 适合并行探索、并行检索、并行校验
  - 批次完成条件：全部子项进入终态
- `first_success`
  - 适合容错检索、候选解生成
  - 批次完成条件：任一 `required=true` 子项成功，其他子项可选择中断
- `review_then_merge`
  - 适合父会话汇总多个子结果
  - 批次完成条件：全部完成后由父会话继续收口

## 10.2 首期不做的策略
- 完整 DAG 依赖图
- 自动重平衡
- 抢占式优先级调度
- 跨父会话共享子智能体池

## 10.3 并发控制
- `subagent_max_active_per_parent`
  - 单个父会话允许同时运行的子智能体数量
- `subagent_max_dispatch_items`
  - 单批次最多派发数量
- `subagent_max_depth`
  - 子智能体树最大深度
- `subagent_queue_enabled`
  - 达到上限时是否入队

## 10.4 调度收敛规则
- `parallel_all`
  - 汇总每项 `status/result_summary/error`
  - 若存在失败但并非全部失败，批次状态为 `partial_failed`
- `first_success`
  - 找到第一个 success 后触发剩余项 `interrupt`
  - 若全部失败，批次状态为 `failed`
- `review_then_merge`
  - 不自动合并内容，仅生成结构化收敛摘要给父会话下一轮使用

## 13. 测试与验收门槛
- 后端必须覆盖单子智能体与多子智能体两条链路。
- 前端必须能在 WS 中断后通过 SSE/重放恢复子智能体批次状态。
- 管理端必须能查询批次和 item 的最终状态。
- 不允许出现：
  - 子智能体关闭后仍被前端当作运行中
  - `resume` 后 system prompt 被重建
  - 父会话看不到批量调度进度
  - 多子智能体并发导致主运行时被阻塞

---

## 14. 落地优先级建议
- 第一优先级：R1-R4
  - 先让后端真正具备完整 control plane 和多子智能体调度能力
- 第二优先级：R5-R6
  - 再让前端和管理端可见、可控
- 第三优先级：R7-R9
  - 最后补齐策略、测试、文档和灰度

---

## 15. 一句话结论
- 本次改造的核心不是“再给 `subagent_control` 多加几个动作”，而是把它提升为 wunder 会话内协作的正式调度层。
- 真正的完成标准不是“能 spawn 一个子会话”，而是“父会话可以稳定调度多个子智能体，并能等待、打断、关闭、恢复、汇总、展示它们的全过程”。
