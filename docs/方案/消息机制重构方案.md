# wunder 消息机制重构方案

## 1. 背景与目标

当前用户反馈的核心问题是：渠道入站消息、蜂群内部通信、聊天页与蜂群页的实时更新存在明显延迟，且经常需要手动刷新才能看到新内容。  
本方案目标不是“继续补丁式修复”，而是将消息机制重构为统一的、可持续演进 10 年的实时事件架构。

目标指标（Server/Desktop 共用思路，按环境分档）：

- 端到端可见延迟（消息入站到前端可见）：
  - p50 < 150ms
  - p95 < 800ms
  - p99 < 2000ms
- 会话断线恢复成功率（30s 内恢复到最新游标）：>= 99.9%
- “需要手动刷新才看到新内容”的用户反馈率：阶段目标降至 < 0.1%
- 关键事件丢失率（final/error/task_result/channel_message/chat_message）：0

## 2. 现状代码分析（基于当前仓库）

### 2.1 Chat 链路（相对成熟，但仍有多路径补偿）

后端：

- `src/api/chat_ws.rs`：支持 `watch/resume`，底层复用 `resume_stream_events`。
- `src/api/ws_helpers.rs`：`send_ws_event` 对 `llm_output_delta` 在背压时是丢弃策略；非 delta 严格发送，队列满时返回 `queue_full_resume_required`。
- `src/orchestrator/event_stream.rs` + `src/orchestrator/stream_persist.rs`：事件持久化到 `stream_events`，含 event_id，支持溢出补偿回放。
- `src/storage/sqlite.rs` / `src/storage/postgres.rs`：`stream_events(session_id,event_id)` 已有 durable log。

前端：

- `frontend/src/stores/chat.ts`：同时存在 `watch`、`resume`、`watchdog(getSessionEvents)`、`reconcile(loadSessionDetail)` 多套补偿逻辑。
- 结果是“可用性强但链路复杂”，故障时能兜底，但状态收敛依赖多个定时器与补偿流程。

结论：

- Chat 已有“事件ID+持久化+续传”基础。
- 仍有“多机制并存”造成行为复杂、延迟抖动和维护成本高的问题。

### 2.2 Beeroom/Swarm 链路（问题最集中）

后端：

- `src/services/beeroom_realtime.rs`：
  - 使用内存 `broadcast::channel`（512 容量）推送；
  - `sequence: AtomicI64` 在进程内自增；
  - 无持久化、无历史回放、无跨实例同步。
- `src/api/beeroom_ws.rs`：
  - `watch` 只订阅内存广播；
  - 发现游标差距或 `Lagged` 时发送 `sync_required`，不做事件补发。
- `src/services/tools/swarm_realtime.rs` / `src/api/team_runs.rs`：
  - 蜂群事件最终通过 `beeroom_realtime.publish_group_event` 进入内存广播。

前端：

- `frontend/src/components/beeroom/BeeroomMissionCanvas.vue`：
  - 同时维护 WS、SSE、轮询（`CHAT_POLL_INTERVAL_MS = 2000`）、sync_required 后历史重载、团队状态 reconcile。
- `frontend/src/views/messenger/beeroomRealtimeSync.ts`：
  - 还在做页面级周期刷新（1.6s/4.2s/12s）。
- `frontend/src/views/MessengerView.vue`：
  - `refreshBeeroomRealtimeGroups/refreshBeeroomRealtimeActiveGroup` 继续周期拉取。

结论：

- Beeroom 目前是“内存广播 + 丢了就全量刷新”的模式。
- 用户感知到“必须手动刷新”，本质是链路缺少 durable replay 与统一游标恢复。

### 2.3 渠道入站链路（已写入 stream_events，但没有统一推送编排）

- `src/channels/service.rs` 的 `append_channel_stream_event_message` 会将 `channel_message` 写入 `stream_events`。
- 当前实现是单点写库，前端是否快速可见依赖 chat watch 的轮询/补偿机制，缺少“同一事件总线上的统一 fanout + ack + 续传”。

### 2.4 User World 链路（可作为标杆）

- `src/services/user_world.rs` + `src/api/user_world_ws.rs`：
  - 有 `user_world_events` 持久化；
  - watch 前先 `list_events(after_event_id)` 回放，再接入广播；
  - `Lagged` 后可用 DB replay 补齐。

结论：

- User World 的模式已经接近目标架构：durable + cursor + replay。

## 3. 根因归纳

R1：消息机制按业务域分裂（chat / beeroom / user_world 各一套），协议语义和恢复策略不一致。  
R2：Beeroom 缺 durable event log，lag/reconnect 后无法精确补发。  
R3：前端存在多重轮询与重载兜底，导致“最终一致但不实时”。  
R4：内存广播天然不支持多实例一致推送（server 横向扩展会加剧问题）。  
R5：Beeroom 事件ID为进程内计数，进程重启后游标语义不稳。  
R6：背压策略在不同域不一致，关键事件与非关键事件没有统一 QoS 分级与可观测性。

## 4. 目标架构（统一实时事件总线）

### 4.1 架构原则

- WS 优先，SSE 兜底（符合现有要求）。
- 统一事件信封（envelope）与统一游标（cursor）协议。
- 以“追加写事件日志（append-only log）”为唯一真相源。
- 推送语义为至少一次（at-least-once），前端幂等应用。
- 顺序保证范围：同一 `stream_key` 内严格按 event_id 递增。
- 关键事件绝不丢弃，非关键高频 delta 可合并但必须可回放。

### 4.2 统一事件信封

```json
{
  "event_id": 10241,
  "stream_key": "beeroom:u_123:hive_alpha",
  "domain": "beeroom",
  "event_type": "team_task_result",
  "occurred_at": 1774450000.123,
  "persisted_at": 1774450000.126,
  "producer": "swarm_runner",
  "request_id": "req_xxx",
  "trace_id": "trace_xxx",
  "payload": {},
  "qos": "critical"
}
```

建议 QoS：

- `critical`：`final/error/team_finish/team_error/team_task_result/channel_message/chat_message/...`
- `normal`：`progress/tool_call/tool_result/...`
- `delta`：`llm_output_delta/think_delta/...`

### 4.3 统一 Stream Key

- Chat：`chat:{user_id}:{session_id}`
- Beeroom：`beeroom:{user_id}:{group_id}`
- User World：`uw:{conversation_id}`
- Channel inbound（映射到 chat 会话）：`chat:{user_id}:{session_id}`（event_type=channel_message）

### 4.4 写入路径（单写入面）

统一写入流程：

1. 业务模块只负责构建领域事件（不直接操作 ws/sse）。
2. `RealtimeEventService::publish(stream_key, event_type, payload, qos)` 统一处理。
3. 在单事务内分配该 stream 的下一个 event_id 并写入事件表。
4. 写入成功后再 fanout 给在线订阅者。
5. fanout 失败不影响 durable log，客户端可通过 resume 补齐。

### 4.5 读取路径（watch/resume 合一）

客户端统一流程：

1. `watch(stream_key, after_event_id)`。
2. 服务端先 replay `[after_event_id+1, current_tail]`。
3. replay 结束后切到 live push。
4. 任意断线重连仍从本地 cursor 恢复。
5. 若服务端检测到不可恢复窗口（例如保留期外），返回 `SYNC_SNAPSHOT_REQUIRED`，由客户端拉快照后继续 watch。

### 4.6 背压策略

- 连接侧维护固定大小发送队列 + 按 QoS 分级。
- `critical` 永不主动丢弃；队列满时触发“强制 resume 模式”而不是静默丢失。
- `delta` 可以压缩合并（coalesce），但必须可在 replay 中补全。
- 对每个连接输出 `slow_client` 指标与事件，不再让前端依赖猜测性 watchdog。

### 4.7 多实例（Server）

推荐：`DB Durable Log + Broker Fanout` 双层架构。

- Durable：Postgres 事件表（强一致存储）。
- Fanout：Redis Streams 或 NATS JetStream（低延迟跨实例广播）。
- 某节点错过广播时，可按 cursor 从 DB replay 补齐。

说明：即使 broker 瞬时不可用，系统仍能靠 DB replay 保证一致性。

### 4.8 单机（Desktop/CLI）

- 仅保留 `SQLite Durable Log + 本地内存 fanout`。
- 不引入额外 broker，避免本地部署复杂度与资源占用上升。

## 5. 存储模型设计

建议新增统一表（保留旧表做迁移兼容）：

### 5.1 realtime_events

- `id`（可选全局ID，便于审计）
- `stream_key` (TEXT, NOT NULL)
- `event_id` (BIGINT, NOT NULL)  // stream内单调递增
- `domain` (TEXT, NOT NULL)
- `event_type` (TEXT, NOT NULL)
- `qos` (TEXT, NOT NULL)
- `payload` (JSON/TEXT, NOT NULL)
- `occurred_at` (DOUBLE, NOT NULL)
- `persisted_at` (DOUBLE, NOT NULL)
- PK(`stream_key`,`event_id`)

索引：

- `idx_realtime_events_stream_tail(stream_key, event_id DESC)`
- `idx_realtime_events_persisted_at(persisted_at)`
- `idx_realtime_events_domain_time(domain, persisted_at DESC)`

### 5.2 realtime_stream_offsets

- `stream_key` PK
- `last_event_id` BIGINT
- `updated_at`

用途：在事务里安全分配 per-stream event_id，避免 `MAX(event_id)+1` 并发热点与竞态。

### 5.3 保留与归档

- Hot retention（在线恢复窗口）：建议 3~7 天（按域可配）。
- 冷归档：按天导出压缩（可选）。
- delta 事件支持离线压缩，不影响关键事件。

## 6. 协议与接口重构

### 6.1 统一 WS 协议

客户端请求：

```json
{
  "type": "watch",
  "request_id": "req_abc",
  "payload": {
    "stream_key": "beeroom:u_123:hive_alpha",
    "after_event_id": 10200
  }
}
```

服务端响应序列：

1. `watching`（确认参数与当前游标）
2. `replay_begin`
3. `event`（0..N）
4. `replay_end`
5. `event`（live）

### 6.2 SSE 兜底语义

- 使用同样的 `stream_key + after_event_id`。
- 不再把 SSE 作为另一套业务语义，只是 transport fallback。

### 6.3 错误码统一

- `STREAM_NOT_FOUND`
- `FORBIDDEN_STREAM`
- `CURSOR_EXPIRED`
- `SYNC_SNAPSHOT_REQUIRED`
- `BAD_REQUEST`
- `INTERNAL_ERROR`

## 7. 前端重构要点

### 7.1 统一 Stream Runtime（替代多重轮询）

新增公共运行时模块（建议放新文件，避免继续膨胀大文件）：

- `frontend/src/realtime/streamRuntime.ts`
- `frontend/src/realtime/streamCursorStore.ts`
- `frontend/src/realtime/streamProtocol.ts`

统一能力：

- 连接管理（WS/SSE 自动切换）
- watch/resume 状态机
- cursor 持久化
- 事件幂等与去重
- 背压/断线恢复

### 7.2 Chat 页

- 移除对 `getSessionEvents + watchdog + 多重 reconcile` 的主链路依赖。
- 将其降级为低频健康检查（例如 30~60s），而不是实时来源。

### 7.3 Beeroom 页

- 移除 `chatPollTimer(2s)` 作为主链路。
- 移除 `beeroomRealtimeSync.ts` 高频全量刷新作为主链路。
- `sync_required` 不再触发全量刷新优先，而先走 cursor replay；仅在 `CURSOR_EXPIRED` 时快照重拉。

### 7.4 Vue 稳定性约束

- 所有异步回调写状态前检查 `disposed`。
- `watch` 默认不 `immediate` 拉起网络订阅，改在 `onMounted` 启动。
- 页面卸载只取消自身订阅，不全局关闭共享连接。

## 9. 性能与容量设计

容量假设（可调整）：

- 单 stream 峰值：200~1000 event/s（delta 高峰）
- 单节点在线订阅：5000~20000
- 高峰下策略：delta 合并、critical 优先、分域限流

关键实现点：

- fanout 与写库解耦（写库成功后异步推送）
- replay 批量读取（limit 默认 200，可按连接动态调节）
- 连接发送队列按 QoS 调度

## 12. 风险与回滚

风险：

- 协议切换期间新旧客户端并存。
- Beeroom 事件体多样，规范化初期可能出现字段不一致。
- 多实例 fanout 引入新组件（Redis/NATS）后运维复杂度上升。

控制策略：

- Feature Flag：按域开关（chat/beeroom/user_world/channel）。
- 双写期：旧链路与新链路并行，做事件对账。
- 灰度发布：先内测用户，再按租户扩容。
- 一键回滚：保留旧 watch 实现直到新链路稳定两周。

## 14. 结论

本次重构必须以“统一事件总线 + 持久化事件日志 + 游标续传”作为核心，先补齐 Beeroom 的结构性短板，再将 chat/user_world/channel 收口到同一套协议与运行时。  
只要做到“写入必持久化、消费可回放、推送可恢复、前端幂等应用”，用户侧“手动刷新才更新”的问题可以从机制层面被彻底消除，而不是持续依赖补丁。
