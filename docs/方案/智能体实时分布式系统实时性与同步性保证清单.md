# 智能体实时分布式系统实时性与同步性保证清单

## 1. 文档目标

- 本文专门回答“wunder 能否做到网游级别的实时性与同步性，以及做到什么程度”。
- 本文关注的是智能体平台场景，不是 FPS/强动作游戏的硬实时。
- 本文重点覆盖：
  - 慢客户端可用性
  - 用户直发消息的第一时间呈现
  - 渠道入站消息的第一时间呈现
  - 蜂群/mission 内部调用消息的第一时间呈现
  - 智能体状态、任务状态、工具状态的同步
  - 断线恢复、重放、幂等、顺序保证

## 2. 能达到的实时性边界

### 2.1 能达到的目标

- [ ] 做到“协作型网游/大型在线协作系统”级别的实时反馈。
- [ ] 关键状态变化在前端上几乎立刻可见。
- [ ] 多端看到同一份权威公开状态。
- [ ] 断线后能快速恢复连续状态。
- [ ] 慢客户端不掉线、不假死，只降级低优先级细节。

### 2.2 不应承诺的目标

- [ ] 不承诺像 FPS 一样的固定帧确定性。
- [ ] 不承诺外部渠道进入 wunder 之前的链路延迟可控。
- [ ] 不承诺慢客户端在压力下还能完整实时显示所有 token、日志和细节动画。

## 4. 统一入站与“第一时间呈现”保证清单

## 4.1 统一入站模型

以下三类来源必须统一进入同一套 ingress 语义：

- [ ] 用户前端直接发送
- [ ] 渠道 webhook / IM 入站
- [ ] 蜂群/mission 内部调用或子任务产生的新消息

每个入站事件必须统一生成：

- [ ] `ingress_id`
- [ ] `source_kind = user | channel | mission | system`
- [ ] `logical_message_id`
- [ ] `thread_id`
- [ ] `mission_id`（如有）
- [ ] `created_at_ingress`
- [ ] `public_event_seq`

## 4.2 用户前端直接发送

- [ ] 前端必须本地立即回显。
- [ ] 服务端一旦 accepted，必须立刻发 `message_accepted` 公共事件。
- [ ] 若后续 owner/mission 决定了更精确的状态，允许权威覆盖，但不能让消息消失再重现。
- [ ] 本地回显和服务端确认必须能通过 `logical_message_id` 去重。

## 4.3 渠道入站

- [ ] 以“wunder ingest 时间”为实时性起点，而不是渠道用户点击发送的时间。
- [ ] 渠道消息一进入 wunder，就必须发出 `message_accepted` 或 `inbound_message_accepted` 公共事件。
- [ ] 该事件必须能够映射到相关 `thread / mission / projection`。
- [ ] 前端必须能够在 `p95 < 100-200ms` 内看到“消息已进入系统”的结果。

## 4.4 蜂群/mission 内部调用

- [ ] mission 内部派生消息必须和用户消息使用同一 public event 机制。
- [ ] 子任务调用、agent 间转发、merge 消息不能只停留在内部日志里。
- [ ] 面向用户可见的内部消息必须尽快进入 projection。

## 5. 智能体状态与工具状态同步清单

## 5.1 必须统一的公开事件

至少统一以下事件名和载荷结构：

- [ ] `thread_accepted`
- [ ] `thread_running`
- [ ] `thread_waiting`
- [ ] `thread_failed`
- [ ] `thread_completed`
- [ ] `task_claimed`
- [ ] `task_started`
- [ ] `task_progress`
- [ ] `task_result`
- [ ] `task_failed`
- [ ] `mission_progress`
- [ ] `mission_finalized`
- [ ] `agent_busy`
- [ ] `agent_idle`
- [ ] `tool_started`
- [ ] `tool_finished`
- [ ] `tool_failed`
- [ ] `tool_waiting_approval`

## 5.2 事件载荷最小字段

每类公开事件至少要带：

- [ ] `target_kind`
- [ ] `target_id`
- [ ] `event_type`
- [ ] `epoch`
- [ ] `seq`
- [ ] `created_at`
- [ ] `source_agent_id`（如有）
- [ ] `mission_id`（如有）
- [ ] `thread_id`（如有）

## 5.3 同步顺序

关键状态同步必须遵守：

- [ ] 先 `thread`
- [ ] 再 `mission`
- [ ] 后 `projection`

不能出现：

- [ ] beeroom 已显示任务完成，但 thread 实际还没结束
- [ ] 面板已显示 agent 空闲，但 mission 内部仍认为其忙碌
- [ ] 工具已完成，但前端只能在整包刷新后才看到

## 6. 顺序、一致性与幂等保证清单

## 6.1 单写者

- [ ] 单 `thread` 单写者
- [ ] 单 `mission` 单裁决者
- [ ] 单 `projection` 单汇总者

## 6.2 顺序保证

- [ ] 所有 target 都要有 `epoch + seq`
- [ ] 客户端按 `epoch/seq` 应用状态
- [ ] 旧 epoch 的事件必须被忽略
- [ ] 跨 target 不要求全局严格顺序，但各自目标内必须严格有序

## 6.3 幂等保证

- [ ] 用户消息按 `logical_message_id` 去重
- [ ] 渠道入站按渠道消息唯一键或映射 key 去重
- [ ] mission 内部派生事件按 runtime 内部 id 去重
- [ ] 前端应用 delta 时必须支持幂等重放

## 7. 慢客户端保证清单

## 7.1 基本原则

- [ ] 慢客户端不追求“所有细节完整实时”
- [ ] 慢客户端必须保证“关键状态不断流”
- [ ] 慢客户端必须保证“可恢复”

## 7.2 事件优先级

- [ ] `P0` 必达
  - `cancel`
  - `tool_result`
  - `mission_finalized`
  - `fatal_error`
- [ ] `P1` 强保护
  - `message_accepted`
  - `task_claimed`
  - `task_started`
  - `tool_started`
  - `first_token`
- [ ] `P2` 可轻度合并
  - token delta
  - progress delta
- [ ] `P3` 可强合并
  - summary diff
  - panel diff
- [ ] `P4` 可直接丢弃
  - typing
  - hover
  - 非关键视觉态

## 7.3 降级流程

- [ ] `warn`
  - 开始合并低优先级事件
- [ ] `degrade`
  - 暂停部分 summary/panel 通道
- [ ] `resync_required`
  - 直接要求客户端走快照重同步

## 7.4 客户端行为

- [ ] 客户端收到 `resync_required` 后优先恢复 `thread / mission` 关键状态
- [ ] 恢复完成后再恢复 beeroom/user_world 等投影面板
- [ ] 页面隐藏时自动降订阅或降频

## 8. 快照、回放与恢复清单

## 8.1 Thread 恢复

- [ ] 恢复 thread 的公共状态
- [ ] 不恢复内部 scratchpad
- [ ] 允许通过 checkpoint 快速续接

## 8.2 Mission 恢复

- [ ] 恢复 task 归属、任务进度、终态结果
- [ ] mission replay 不依赖页面重新请求整包详情才能看见最新状态

## 8.3 Projection 恢复

- [ ] beeroom/user_world/session 投影优先用 `snapshot + delta`
- [ ] gap 小时 replay
- [ ] gap 大时 resync

## 9. 前端行为保证清单

## 9.1 Chat

## 9.3 User World

- [ ] 多个 conversation watch 时可按可见性和优先级降级

## 10. SLO 与验收清单

- [ ] 用户直发消息本地回显：体感即时
- [ ] `message_accepted` 可见 `p95 < 80ms`
- [ ] 渠道/mission 入站自 wunder ingest 起可见 `p95 < 100-200ms`
- [ ] `task_claimed/task_started/tool_started` 可见 `p95 < 120ms`
- [ ] `mission_finalized/tool_finished` 可见 `p95 < 150ms`
- [ ] `500` 条以内 replay 恢复 `p95 < 500ms`
- [ ] 慢客户端下 `P0/P1` 事件不丢

## 10.2 联调验收

- [ ] 用户直发、渠道入站、mission 派生消息三类路径都能第一时间显示
- [ ] thread/mission/projection 三层状态无明显回跳
- [ ] 工具开始、完成、失败、审批等待都能在前端及时同步
- [ ] 断线恢复后状态连续，不靠手动刷新修正
- [ ] 隐藏页和慢客户端仍可稳定工作
