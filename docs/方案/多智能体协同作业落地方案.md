# 多智能体协同作业落地方案

## 1. 本版结论

这次方案按你的新约束彻底改线，核心结论如下：

1. 用户侧产品名统一为 **蜂巢**，英文技术名统一为 **`beeroom`**。
2. 多智能体协同的核心实体不是群聊，也不是独立页面，而是：
   - **智能体分组**
   - **母蜂机制**
   - **组内协同任务**
   - **组员私有工作线程**
3. 用户**不必进入蜂群页**才触发协作。用户在任何一个智能体会话里聊天，只要该智能体具备蜂群工具且属于某个分组，就可以发起组内协作。
4. **蜂群页只是更强的可视化与调度观察界面**，不是唯一入口。
5. 同组智能体通过 `agent_swarm` 发现与交互，但发现范围必须收缩为**同组成员**。
6. **第一个真正发起组内 fan-out 协作的智能体自动成为母蜂**。
7. 一次蜂群任务的完成判定，不是“母蜂发了总结就算完”，而是：
   - 该任务的所有子任务进入终态；
   - 且该组相关组员全部回到空闲状态；
   - 这时任务才被系统标记为 `completed`。
8. 通过蜂群工具发送的任务消息，必须自动附带：
   - 当前 `beeroom` 基本信息；
   - 发送者基本信息；
   - 当前任务来源和上下文摘要；
   这样组员之间协作才稳定。

一句话概括：

> **蜂巢（beeroom）不是聊天系统，而是“任意入口触发、统一工作台观测、组内并行协作、母蜂统一收口”的多智能体协同系统。**

---

## 2. 与上一版思路相比，真正变化了什么

上一版重点是“把多个智能体放进群聊”。

这一版重点变成了：

- **分组优先**，不是房间优先；
- **任意会话可触发**，不是页面优先；
- **工作台可视化**，不是消息流可视化；
- **并行任务板**，不是串行聊天气泡；
- **协作上下文自动附带**，不是靠模型自己猜上下文。

所以本版不再把“群聊后端”当主轴，而是改成：

- 智能体创建时绑定分组；
- 组内工具作用域隔离；
- 母蜂进行编排；
- 蜂群页做观测与交互增强。

## 4. 新的核心业务模型

## 4.1 `beeroom_group`：蜂群分组

它是系统里的主实体，不是聊天房间。

建议字段：

- `beeroom_group_id`
- `owner_user_id`
- `group_name`
- `description`
- `status`
- `mother_agent_id`
- `created_at`
- `updated_at`

说明：

- 一个用户可以有多个蜂群分组；
- 一个智能体第一版只属于一个蜂群分组；
- 母蜂属于组级状态，不是页面状态。

## 4.2 `user_agents` 扩展字段

建议扩展：

- `beeroom_group_id`
- `swarm_duty`
- `allow_auto_dispatch`
- `allow_swarm`

字段含义：

- `beeroom_group_id`：该智能体属于哪个蜂群；
- `swarm_duty`：职责标签，如 `情报/法理/舆情/行动/红队/纪要`；
- `allow_auto_dispatch`：系统自动分配时是否可被选中；
- `allow_swarm`：是否启用蜂群能力。

## 4.3 `beeroom_mission`：蜂群任务

这是用户触发的一次完整协同任务。

建议字段：

- `mission_id`
- `beeroom_group_id`
- `source_user_id`
- `source_agent_id`
- `source_session_id`
- `trigger_type`
- `entry_agent_id`
- `mother_agent_id`
- `status`
- `linked_team_run_id`
- `summary`
- `error`
- `created_at`
- `updated_at`
- `finished_at`

`trigger_type` 建议值：

- `chat_session`
- `beeroom_page`
- `api`
- `system`

这能明确回答：

- 任务是从哪儿来的；
- 是谁先接住了它；
- 最终由谁担任母蜂。

## 4.4 `beeroom_mission_task`：组员子任务

建议字段：

- `mission_task_id`
- `mission_id`
- `assignee_agent_id`
- `directive`
- `expected_schema_json`
- `linked_session_id`
- `linked_session_run_id`
- `status`
- `result_summary`
- `result_payload_json`
- `error`
- `priority`
- `created_at`
- `updated_at`
- `finished_at`

## 4.5 `beeroom_agent_thread`：组员长期工作线程

每个智能体在每个蜂群分组里保留一个工作线程。

建议字段：

- `beeroom_group_id`
- `agent_id`
- `control_session_id`
- `last_used_at`
- UNIQUE(`beeroom_group_id`, `agent_id`)

这样可以保证：

- 组员有自己的长期上下文；
- 协作发生时不是临时乱开 session；
- 同一组内上下文稳定积累；
- 不同组之间上下文天然隔离。

## 5. 母蜂机制

## 5.1 母蜂认领规则

规则定义建议精确到这一句：

> **某组内第一个真正成功发起 fan-out 协作的智能体，自动被原子性登记为该组母蜂。**

这里的关键词是：

- 第一个
- 真正成功
- fan-out 协作

不是：

- 第一个被用户聊天的智能体；
- 第一个拥有蜂群工具的智能体；
- 第一个被创建进组的智能体。

## 5.2 母蜂职责

母蜂负责：

- 判断是否需要协作；
- 选择参与组员；
- 拆解任务；
- 派发指令；
- 汇总结果；
- 输出最终结论；
- 管理任务结束状态。

## 5.3 母蜂不是唯一入口，但它是协作总控

用户可以和组内任意智能体聊天。

但是：

- 若当前任务要进入蜂群协作，最终编排权要落到母蜂；
- 若还没有母蜂，则发起协作的那个智能体成为母蜂；
- 若已有母蜂，而某普通成员收到复杂任务，成员可以先做初判，再将编排转交母蜂。

## 5.4 母蜂需要支持手动移交

建议支持：

- 手动设为母蜂；
- 重置母蜂；
- 母蜂失效后自动重新认领；
- 某次任务临时代理母蜂。

第一版可简化为：

- 组级只有一个 `mother_agent_id`；
- 支持手动重置；
- 支持重新认领。

## 6. 任意入口触发协作

这是本版最重要的变化之一。

## 6.1 触发协作的入口不止 `beeroom` 页面

建议明确支持以下入口：

### A. 任意智能体聊天页

用户在某个 agent 聊天时：

- 若该 agent 启用了蜂群工具；
- 且该 agent 有 `beeroom_group_id`；
- 则它可以在当前会话中触发组内协作。

这是主入口。

### B. `beeroom` 蜂群页

用户也可以在蜂群页直接发任务，但它只是增强入口，不是唯一入口。

### C. API 或系统任务

后续也可以支持：

- 定时任务触发；
- 外部 API 投递；
- 系统自动重试与补问。

## 6.2 `beeroom` 页面在系统中的正确定位

`beeroom` 页面不是一个单独“聊天频道”，它更像：

- 该分组的**任务态势总览页**；
- 该分组所有协作任务的**可视化控制台**；
- 用户查看谁在忙、谁完成了什么、最终结论是什么的地方。

所以应当明确：

> **用户可在任何智能体会话中引发协作，而 `beeroom` 负责把这些协作统一聚合显示。**

## 7. 蜂群工具协议增强

你这次提的一个关键点非常正确：

> **通过蜂群工具发送的消息指令，要自动附带当前蜂群以及发送者的基本情况。**

这应该做成协议层能力，而不是依赖模型自己记住上下文。

## 7.1 为什么必须自动附带上下文

如果不自动附带，组员 agent 容易出现这些问题：

- 不知道自己属于哪个蜂群；
- 不知道当前母蜂是谁；
- 不知道指令来自谁；
- 不知道发起任务的上游来源；
- 不知道本轮任务是新任务、补问还是复核；
- 不知道其他成员的分工，导致重复劳动。

所以 `agent_swarm send/batch_send` 不应该只是裸发一句话，而应该自动包装为**蜂群信封（Swarm Envelope）**。

## 7.2 推荐的自动附带字段

建议统一封装：

```json
{
  "swarm_context": {
    "beeroom_group": {
      "group_id": "brg_xxx",
      "group_name": "战役研判组",
      "mother_agent_id": "agent_mother_x",
      "member_count": 6
    },
    "sender": {
      "agent_id": "agent_intel_x",
      "name": "情报收集",
      "swarm_duty": "情报",
      "is_mother": false
    },
    "mission": {
      "mission_id": "mission_xxx",
      "source_user_id": "user_001",
      "source_session_id": "sess_xxx",
      "entry_agent_id": "agent_intel_x",
      "trigger_type": "chat_session"
    },
    "routing": {
      "dispatch_mode": "auto",
      "priority": 80,
      "reply_to_mother": true
    },
    "group_snapshot": {
      "active_agents": ["agent_legal_x", "agent_public_x"],
      "idle_agents": ["agent_red_x", "agent_note_x"],
      "duties": [
        {"agent_id": "agent_legal_x", "duty": "法理"},
        {"agent_id": "agent_public_x", "duty": "舆情"}
      ]
    }
  },
  "directive": "请补充蓝军行动方案中的法理边界与潜在争议点"
}
```

## 7.3 这些信息如何进入目标智能体上下文

建议不要作为 UI 裸展示文本，而是：

- 由工具层自动拼进目标智能体的系统前缀或任务前缀；
- 同时写入 `mission_task.result_payload_json` / 监控事件；
- 前端在需要时再结构化展示。

这样可以同时满足：

- 模型协作稳定；
- 数据可追踪；
- UI 不混乱。

## 7.4 对 `agent_swarm` 的实际改造建议

当前 `src/services/tools.rs` 中的 `agent_swarm_send` / `agent_swarm_batch_send` 建议增加内部步骤：

1. 解析当前 agent 的 `beeroom_group_id`；
2. 读取该组基本信息；
3. 读取当前 sender agent 基本信息；
4. 读取当前 mission 基本信息；
5. 自动组装 `swarm_context`；
6. 将 `directive + swarm_context` 一起发送给目标 agent。

## 8. 任务完成定义

你这次新增的完成条件非常关键，建议明确成系统级规则，而不是 UI 判断。

## 8.1 任务完成不等于母蜂发出总结

如果只以“母蜂发出总结”为完成，会有风险：

- 某个组员 session 还在跑；
- 某个子任务处于 queued/running；
- 某个组员刚补发了一轮复核；
- 母蜂已输出阶段性总结，但实际上整个组并未回空闲。

## 8.2 推荐完成判定

一个 `beeroom_mission` 只有在以下条件同时满足时才进入 `completed`：

### 条件 A：所有 mission task 进入终态

终态包括：

- `success`
- `failed`
- `timeout`
- `cancelled`

### 条件 B：该任务涉及到的组员全部回到空闲

建议组员空闲定义为：

- 该组员在本组对应 thread 上没有 `queued/running` 的 `session_run`
- 且没有未完成的 `beeroom_mission_task`

### 条件 C：母蜂线程没有挂起中的合并动作

即：

- 母蜂没有处于 `merging/waiting_for_more` 状态。

满足 A+B+C 后，任务状态才变为：

- `completed`

否则即使已有 summary，也只能是：

- `summarized`
- 或 `awaiting_idle`

## 8.3 为什么这个规则好

它能保证：

- UI 上“完成”是真的完成；
- 便于后期回放和统计；
- 不会因为母蜂提前说了一句总结就误收口。

## 10. 蜂群可视化建议

这部分是本次方案最值得重点设计的地方。

因为你的场景不是“看聊天记录”，而是“看协作态势”。

## 10.1 总原则：不要做成串行聊天流

建议默认主视图不是：

- A 说一句
- B 回一句
- C 再回一句

而应该是：

- **任务摘要流**
- **并行任务板**
- **成员状态图**
- **结果汇总层**

换句话说：

> **让用户看到“谁在做什么、做到哪一步、结果如何”，而不是“谁先说了哪句话”。**

## 10.2 推荐默认布局：四区工作台

### 区块 A：顶部任务输入条

支持：

- 输入任务
- `@任意组员`
- 不 `@` 走自动分配
- 附加材料/附件
- 选择是否强制走蜂群协作

顶部输入条旁边建议显示：

- 当前默认母蜂
- 自动分配状态
- 本组当前忙闲概览。

### 区块 B：左上摘要流 `Mission Feed`

这里只保留高价值信息：

- 用户原始任务
- 入口 agent 接单
- 母蜂认领/转交
- 阶段性汇总
- 风险告警
- 最终结果

它的作用不是记录所有细节，而是让用户快速理解：

- 当前任务走到了哪一阶段；
- 发生了什么关键转折。

### 区块 C：中部主区 `Swarm Board`

这是蜂群页的核心。

建议以**并行任务卡片**来展示子任务，而不是消息气泡。

每个任务卡显示：

- 执行成员
- 职责标签
- 当前状态
- 已耗时
- 当前摘要
- 置信度
- 是否需要补充输入
- 最后更新时间

状态颜色建议：

- `queued`：灰
- `running`：蓝
- `merging`：紫
- `completed`：绿
- `failed`：红
- `blocked`：橙

布局建议：

- 默认按“成员列”展示
- 可切换为“状态列”展示

### 区块 D：右侧状态面板 `Roster + Graph`

分两块：

#### D1. 成员状态面板

每个成员展示：

- 名称
- 职责
- 是否母蜂
- 当前状态：idle/busy/waiting/merging/offline
- 当前处理任务数
- 最近完成时间

#### D2. 协同图

一个简化的关系图：

- 母蜂在中间
- 每个组员是外围节点
- 当前有任务派发时，高亮连线
- 连线粗细表示任务强度或优先级
- 节点颜色表示忙闲状态

这个图的价值不是炫技，而是让用户一眼看出：

- 母蜂当前扇出到哪些人；
- 是否某个成员长期过载；
- 当前协作是集中式还是分散式。

## 10.3 推荐的 4 种视图模式

同一个 `beeroom` 页面建议提供四种视图切换，而不是一页塞满。

### 模式 1：任务板模式（默认）

适合日常使用。

看点：

- 当前有哪些任务
- 哪些成员在忙
- 哪些结果已完成

### 模式 2：时间线模式

适合复盘。

内容：

- 任务何时发起
- 母蜂何时认领
- 任务何时派发
- 各成员何时完成
- 最终何时全组空闲

### 模式 3：证据/结果模式

适合看最终结论。

内容：

- 各成员的结果摘要
- 关键证据
- 风险点
- 母蜂总结
- 最终状态

### 模式 4：资源态势模式

适合看组运行状态。

内容：

- 忙碌成员数
- 空闲成员数
- 任务峰值并发
- 平均完成时间
- 最近失败率

## 10.4 推荐的微交互

- `merging`
- `offline`

## 13.3 任务完成转换

推荐状态流：

```text
queued -> routing -> dispatched -> running -> merging -> summarized -> awaiting_idle -> completed
```

其中：

- `summarized`：母蜂已有结论
- `awaiting_idle`：还在等全组真正回空闲
- `completed`：全任务终态 + 全组空闲

这个设计特别适合你强调的“任务完成必须以全组空闲为准”。

## 15. 最终建议

如果要一句话总结这版方案，我的建议是：

> **蜂巢（beeroom）应该被设计成“多智能体分组协作的观测与控制层”，而不是新的聊天层。**

真正正确的落地方式是：

1. 智能体创建时绑定 `beeroom_group_id`
2. 任何会话中只要当前 agent 有蜂群能力，都可以引发组内协作
3. 第一个真正 fan-out 的智能体自动成为母蜂
4. `agent_swarm` 自动附带组与发送者上下文
5. `beeroom` 页面统一可视化所有来源的蜂群任务
6. 全部子任务终态且全组回空闲，任务才算真正完成

---

## 16. 一句话版本

> 本版多智能体协同方案不再以群聊为中心，而改为 `beeroom` 蜂群分组模型：创建智能体即分组，用户在任意智能体会话中都可触发组内协作，`agent_swarm` 默认只发现同组成员并自动附带当前蜂群与发送者基本信息，第一个真实发起 fan-out 的智能体成为母蜂，母蜂负责编排与收口，而 `beeroom` 页面只负责将所有来源的协同任务以摘要流、任务板、协同图和资源态势方式统一可视化，且只有当全部组员任务完成并全组恢复空闲时，任务才被判定为真正完成。
