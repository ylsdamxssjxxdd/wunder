## 5. Replay 与恢复清单

### 5.1 Thread

- [ ] 支持按 `thread_id + epoch + after_seq` replay。
- [ ] 仅 replay 公开事件，不 replay 内部 scratchpad。

### 5.2 Mission

- [ ] 支持按 `mission_id + epoch + after_seq` replay。
- [ ] 任务归属与终态事件必须可恢复。

### 5.3 Projection

- [ ] 支持 `snapshot + delta` 模式。
- [ ] gap 小时直接 replay，gap 大时返回 `resync_required`。

## 6. Backpressure 清单

- [ ] 在 [ws_helpers.rs](C:/Users/sjxx/Desktop/wunder/src/api/ws_helpers.rs) 中区分 target/channel 优先级。
- [ ] 增加 `warn -> degrade -> resync` 三段策略。
- [ ] 记录每类通道的 queue 占用与丢弃次数。
- [ ] client 端能识别 `backpressure.level`。

## 7. 数据保留与清理清单

- [ ] `thread_checkpoints` 保留最近关键状态。
- [ ] `mission_checkpoints` 保留最近 N 个完成实例。
- [ ] `projection_snapshots` 可按时间和 realm 清理。
- [ ] 高频 token 明细不应无限长期保留。
- [ ] PG 需要补索引和必要的分区策略；SQLite 需要保留简化版实现。

## 8. 协议与存储验收

- [ ] 任一 target 的 owner 迁移不会导致双写。
- [ ] replay 不会把数据库当作唯一实时消息队列。
- [ ] 慢客户端时，关键执行事件仍必达。
- [ ] 断线恢复顺序符合 `thread -> mission -> projection`。
