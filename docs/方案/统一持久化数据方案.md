# 统一持久化数据方案

## 1. 背景与目标

当前仓库里“项目差异化内容”分散在多个顶层目录：

- `config/wunder.yaml`
- `config/org_units.json`
- `knowledge/`
- `fonts/`
- `config/fonts.conf`
- `config/matplotlibrc`
- `config/wunder.desktop.preconfig.yaml`

这会带来两个直接问题：

1. 外部代码更新时，项目差异内容混在源码目录里，不方便按项目打包和替换。
2. server / cli / desktop / docker / desktop 打包脚本对这些路径存在分散的硬编码，后续维护成本高。

本方案的目标是把“项目差异化持久化内容”统一归口到 `data` 体系下，形成明确的路径约定、兼容策略与迁移节点，使后续项目交付和代码升级更可控。

## 2. 先给结论

### 2.1 推荐结论

不要只把若干文件机械地挪到 `data/` 后就结束；推荐把 `data` 明确拆成两层：

- `data/project/`：项目差异化内容，作为对外交付和项目切换的主载体。
- `data/runtime/`：程序运行期写入的持久化状态，不建议被项目包直接覆盖。

同时保留：

- `temp_dir/`：纯临时目录。
- `config/`：代码内置示例、构建期配置、评估样例等非项目差异内容。

### 2.2 为什么不建议“整个 data 目录直接平铺替换”

当前 `data/` 已经承载了运行期写入内容，例如：

- `data/config/wunder.override.yaml`
- `data/user_tools/`
- `data/admin_skills/`
- `data/throughput/`
- `data/wunder.db`（SQLite 默认）

如果再把项目差异化资源直接平铺塞进同一个 `data/` 根目录，后续“整目录替换”很容易把现场运行过程中产生的配置、工具、知识库或数据库状态一起覆盖掉。

因此，本方案推荐：

- **项目包可替换**：`data/project/`
- **运行态可保留**：`data/runtime/`

如果业务场景明确允许“项目包覆盖运行态状态”，可以退化为平铺版 `data/` 方案；但不建议作为主方案。

## 3. 现状扫描

### 3.1 已经走 `data` 的部分

现有代码里，以下内容已经默认落在 `data` 或可由环境变量指向 `data`：

- 配置覆盖：`data/config/wunder.override.yaml`
- 管理端提示词模板：`data/prompt_templates/`
- 用户工具：`data/user_tools/`
- SQLite：`data/wunder.db`
- server docker 默认向量知识目录：`/app/data/vector_knowledge`

这说明“把可持久化内容归口到 `data`”在方向上与现有工程并不冲突，问题主要是还没有统一规划。

### 3.2 仍然存在的硬编码入口

本次扫描确认的关键路径点如下：

- `src/core/config.rs`
  - 默认基础配置：`config/wunder.yaml`
  - 默认覆盖配置：`data/config/wunder.override.yaml`
- `src/services/org_units.rs`
  - 组织架构种子：`config/org_units.json`
- `src/api/admin.rs`
  - 管理端新建字面知识库时，空路径默认写为 `./knowledge/<名称>`
- `src/core/i18n.rs`
  - 默认 i18n 文案：`config/i18n.messages.json`
  - 同时保留 `include_str!` 内置回退
- `wunder-cli/runtime.rs`
  - repo 根识别依赖 `config/wunder.yaml`
  - base config 默认优先读取 `config/wunder.yaml`
- `desktop/tauri/runtime.rs`
  - 运行资源布局识别依赖 `config/wunder.yaml`
  - desktop 预配置默认查找 `config/wunder.desktop.preconfig.yaml`
- `desktop/electron/scripts/prepare-resources.js`
  - 目前整包复制 `config/`
- `docker-compose-arm.yml` / `docker-compose-x86.yml` / `packaging/docker/docker-compose-ubuntu20.yml`
  - 默认 `WUNDER_CONFIG_PATH=/app/config/wunder.yaml`
  - `FONTCONFIG_FILE=/app/config/fonts.conf`
  - `MATPLOTLIBRC=/app/config/matplotlibrc`
  - 挂载字体目录为 `./fonts`
- `web/index.html` / `web/modules/knowledge.js` / `web/modules/i18n.js`
  - 管理端 UI 仍提示默认目录为 `./knowledge/<名称>`

### 3.3 需要特别注意的语义点

#### 3.3.1 `org_units.json` 不是实时配置

`org_units.json` 只在组织架构表为空时用于初始化种子数据；一旦数据库已有内容，后续替换该文件不会自动更新现有组织结构。

这意味着：

- 把它迁到 `data` 是合理的；
- 但不能把“替换 `org_units.json`”误认为“实时修改组织架构”。

#### 3.3.2 把 `wunder.yaml` 挪位置后，相对路径不会自动跟着变

当前代码读取 YAML 后，不会把相对路径解释为“相对于配置文件目录”，而是仍然按进程工作目录解析。

因此如果把基础配置改成 `data/project/config/wunder.yaml`，配置文件内涉及知识库、工作区、资源目录等路径时，必须显式写成：

- `./data/project/knowledge/...`
- `./data/runtime/...`
- `./workspaces`

不能简单假设把文件挪过去之后，`./knowledge/...` 会自动变成相对于新目录。

## 4. 目标目录结构

## 4.1 推荐结构（主方案）

```text
data/
  project/
    config/
      wunder.yaml
      org_units.json
      fonts.conf
      matplotlibrc
      wunder.desktop.preconfig.yaml
      i18n.messages.json          # 可选：项目定制文案时启用
    knowledge/
      plan/
      <base_name>/
    fonts/
      ...

  runtime/
    config/
      wunder.override.yaml
    prompt_templates/
    user_tools/
    admin_skills/
    vector_knowledge/
    throughput/
    wunder.db
```

### 4.2 保留在代码侧的内容

以下内容不建议迁入 `data/project/`：

- `config/wunder-example.yaml`
  - 作为开发示例和缺省回退模板保留在代码侧。
- `config/app_version.json`
  - 这是构建/打包链路的版本单一来源，不属于项目差异化数据。
- `config/benchmark/**`
  - 属于内置 PinchBench 任务、素材与研发资产，不属于交付项目数据。
- `prompts/`
  - 继续作为系统内置默认提示词。
- `skills/`
  - 继续作为系统内置技能。
- `scripts/`
  - 继续作为运行支撑与构建脚本。

### 4.3 如果必须走“平铺 data”简化版

可选但不推荐的简化结构：

```text
data/
  config/
    wunder.yaml
    wunder.override.yaml
    org_units.json
    fonts.conf
    matplotlibrc
    wunder.desktop.preconfig.yaml
  knowledge/
  fonts/
  prompt_templates/
  user_tools/
  admin_skills/
  vector_knowledge/
  wunder.db
```

这个方案改动最小，但缺点也很明确：

- 后续如果直接覆盖整个 `data/`，会把运行期生成状态一起覆盖。

## 5. 路径策略

## 5.1 建议新增统一解析模块

建议新增一个小型路径解析模块，例如：

- `src/core/data_paths.rs`

职责只做一件事：提供统一的“项目数据根 / 运行数据根 / 兼容回退路径”解析，避免路径字符串继续散落在 `config.rs`、`runtime.rs`、`admin.rs`、`org_units.rs` 里。

建议提供的核心能力：

- `resolve_project_data_root()`
- `resolve_runtime_data_root()`
- `resolve_base_config_path()`
- `resolve_org_units_seed_path()`
- `resolve_shared_literal_knowledge_root(base_name)`
- `resolve_project_fonts_root()`
- `resolve_fontconfig_path()`
- `resolve_matplotlibrc_path()`
- `resolve_desktop_preconfig_path()`

## 5.2 建议环境变量

建议补充以下环境变量（全部可选）：

- `WUNDER_PROJECT_DATA_ROOT`
  - 默认：`./data/project`
- `WUNDER_RUNTIME_DATA_ROOT`
  - 默认：`./data/runtime`
- `WUNDER_ORG_UNITS_PATH`
  - 默认：`<project_data_root>/config/org_units.json`
- `WUNDER_I18N_MESSAGES_PATH`
  - 默认：`<project_data_root>/config/i18n.messages.json`，找不到时回退旧路径与内置文案

已有环境变量继续保留：

- `WUNDER_CONFIG_PATH`
- `WUNDER_CONFIG_OVERRIDE_PATH`
- `WUNDER_USER_TOOLS_ROOT`
- `WUNDER_VECTOR_KNOWLEDGE_ROOT`

## 5.3 建议默认搜索顺序

### 基础配置 `wunder.yaml`

推荐搜索顺序：

1. `WUNDER_CONFIG_PATH`
2. `./data/project/config/wunder.yaml`
3. `./data/config/wunder.yaml`（兼容过渡）
4. `./config/wunder.yaml`（兼容过渡）
5. `./config/wunder-example.yaml`

### 组织架构种子 `org_units.json`

推荐搜索顺序：

1. `WUNDER_ORG_UNITS_PATH`
2. `./data/project/config/org_units.json`
3. `./data/config/org_units.json`（兼容过渡）
4. `./config/org_units.json`（兼容过渡）

### 共享字面知识库默认根目录

推荐默认值：

- `./data/project/knowledge/<base_name>`

兼容旧值：

- 允许继续读取 `./knowledge/<base_name>`
- 管理端新增知识库时默认填充新路径，不再写旧路径

### 字体资源

推荐默认值：

- 字体目录：`./data/project/fonts`
- fontconfig：`./data/project/config/fonts.conf`
- matplotlibrc：`./data/project/config/matplotlibrc`

## 6. 具体改造节点

## 6.1 后端 server 节点

### 节点 A：基础配置路径切换

涉及文件：

- `src/core/config.rs`
- `src/main.rs`
- `src/core/config_store.rs`

目标：

- 默认从 `data/project/config/wunder.yaml` 读取基础配置；
- `data/runtime/config/wunder.override.yaml` 作为默认覆盖配置；
- 保留旧路径回退至少一个迁移周期。

### 节点 B：组织架构种子路径切换

涉及文件：

- `src/services/org_units.rs`

目标：

- 把常量路径替换为统一解析；
- 新增 `WUNDER_ORG_UNITS_PATH`；
- 文档明确“仅在表为空时生效”。

### 节点 C：共享字面知识库默认路径切换

涉及文件：

- `src/api/admin.rs`
- `src/services/knowledge.rs`

目标：

- 管理端新建共享字面知识库时，默认路径改为 `./data/project/knowledge/<名称>`；
- 兼容读取旧路径；
- 新 UI 文案同步更新。

### 节点 D：i18n 项目覆盖支持

涉及文件：

- `src/core/i18n.rs`

目标：

- 默认先找 `data/project/config/i18n.messages.json`；
- 找不到则回退旧路径 `config/i18n.messages.json`；
- 再不行则回退 `include_str!` 内置内容。

## 6.2 CLI / Desktop 节点

### 节点 E：CLI 运行时根识别与 base config 查找

涉及文件：

- `wunder-cli/runtime.rs`
- `wunder-cli/args.rs`
- `tests/wunder_cli_line_chat_smoke.rs`

目标：

- repo 根识别同时支持 `data/project/config/wunder.yaml`；
- CLI 帮助文案默认路径同步改为新路径；
- 测试显式覆盖新路径或走兼容查找。

### 节点 F：Desktop 运行时资源布局识别

涉及文件：

- `desktop/tauri/runtime.rs`
- `desktop/electron/scripts/prepare-resources.js`

目标：

- Desktop 资源包内支持 `data/project/config/wunder.yaml`；
- `wunder.desktop.preconfig.yaml` 改为优先从 `data/project/config/` 查找；
- Electron 打包时从 `data/project/` 收集项目资源，而不是整包复制旧 `config/`。

## 6.3 前端与管理端节点

## 7. 分阶段实施建议

## 7.1 第一阶段：兼容接入（推荐先做）

目标：先让新路径可用，不立即破坏旧项目。

动作：

1. 新增统一路径解析模块。
2. server / cli / desktop / docker 全部支持新路径优先、旧路径回退。
3. 管理端新增知识库默认路径切换为新路径。
4. 文档与分发说明同步更新。

完成标准：

- 新项目只放 `data/project/` 就能跑；
- 老项目仍可继续使用 `config/`、`knowledge/`、`fonts/`；
- 运行日志里最好打印一次“当前命中的实际路径”，方便排查。

## 7.2 第二阶段：运行态目录归一

目标：把现有平铺在 `data/` 下的运行态内容收敛到 `data/runtime/`。

动作：

1. `wunder.override.yaml` 切到 `data/runtime/config/`
2. `user_tools/`、`admin_skills/`、`vector_knowledge/`、`throughput/` 切到 `data/runtime/`
3. SQLite 默认路径切到 `data/runtime/wunder.db`
4. Docker 默认环境变量同步调整

完成标准：

- `data/project/` 可单独作为项目差异包交付；
- `data/runtime/` 作为现场运行态数据保留；
- 项目切换时无需覆盖现场运行态。

## 7.3 第三阶段：清理旧路径

目标：在确认内部项目都完成迁移后，逐步移除旧硬编码。

动作：

1. 移除 `config/wunder.yaml`、`config/org_units.json`、`knowledge/`、`fonts/` 的默认读取逻辑。
2. 保留显式环境变量覆盖，但不再默认回退旧目录。
3. 更新 README、分发文档、运维脚本、测试基线。

## 8. 风险与处理

### 风险 1：整目录替换覆盖运行态数据

处理：

- 推荐采用 `data/project` / `data/runtime` 双层结构；
- 如果短期内坚持平铺 `data/`，必须在发布说明中明确“不要直接覆盖运行态目录”。

### 风险 2：`org_units.json` 替换后没有生效

处理：

- 文档明确它只是 seed；
- 后续如有需要，再补单独的“组织架构导入/重建”能力。

### 风险 3：Desktop / CLI 在新目录布局下无法识别 repo 根

处理：

- `looks_like_repo_root()` / `has_runtime_resource_layout()` 必须同步调整；
- 测试覆盖新旧两套目录布局。

### 风险 4：配置文件迁移后，相对路径仍然指向旧目录

处理：

- 第一阶段直接把 `data/project/config/wunder.yaml` 内的路径写成完整的项目数据相对路径；
- 不在第一阶段引入“相对于配置文件目录解析全部路径”的大改动。

### 风险 5：字体在 docker / 桌面端渲染异常

处理：

- compose 挂载路径和 `FONTCONFIG_FILE` 必须一起改；
- 桌面端打包资源要实际复制 `data/project/fonts/` 与 `data/project/config/fonts.conf`。

## 9. 验收标准

满足以下条件即可认为方案落地完成：

1. server 在只保留 `data/project/config/wunder.yaml` 的情况下可以正常启动。
2. 管理端新建共享字面知识库时，默认目录落在 `data/project/knowledge/`。
3. 组织架构种子默认读取 `data/project/config/org_units.json`。
4. docker 默认挂载和环境变量全部切到 `data/project` / `data/runtime`。
5. desktop / cli 不再依赖 `config/wunder.yaml` 才能识别运行资源布局。
6. 旧项目在迁移窗口内仍可用旧路径启动。

## 10. 最终建议

如果这次要正式开做，我建议按下面顺序推进：

1. 先补 `src/core/data_paths.rs`，统一路径策略。
2. 第一批改 server / cli / desktop 的基础配置与组织架构路径。
3. 第二批改知识库默认路径、docker、desktop 打包与管理端提示文案。
4. 第三批再收敛 `data/runtime/`，把现有 `data/` 下的运行态内容分层。

这样做的好处是：

- 改动路径清晰；
- 兼容窗口明确；
- 不会一次性把所有分发、docker、desktop、知识库逻辑同时打散；
- 后续内部项目可以把“项目差异包”逐步固化到 `data/project/`，实现代码与项目数据的稳定解耦。
