# 定时任务调度重构方案（2026-03）

## 1. 背景

当前 Wunder 定时任务已经具备基础可用性：支持 `at/every/cron`、自然语言 `schedule_text`、任务持久化、运行记录、管理员/用户侧查询，以及基于 Postgres/SQLite 的任务抢占执行。

随着定时任务需求增大，现有实现暴露出三个主要问题：

1. **空转轮询偏重**：调度循环默认按 `poll_interval_ms=1000` 醒来，即使下个任务在数小时后也会持续轮询。
2. **失败风暴缺少缓冲**：周期任务一旦外部依赖异常，当前只会累加连续失败次数并最终停用，缺少中间退避层。
3. **多实例稳定性仍需增强**：当前 DB claim 模型方向正确，但需要补齐 lease/owner/token 与心跳续租，避免长任务、实例抖动和重复 claim 场景下出现悬挂或重复执行。

## 2. 与 openclaw 的对比结论

### 2.1 openclaw 的可借鉴点

- 单定时器驱动，按下一次到期时间唤醒，空闲成本低。
- 周期任务错误退避、调度异常隔离、cron stagger、run log、session reaper 都比较完善。
- 测试覆盖非常高，很多行为以回归用例固化。

### 2.2 openclaw 不适合直接照搬的点

- 核心存储是 `jobs.json` + 进程内锁，天然偏单机 gateway/daemon。
- 不适合作为 Wunder server 的最终方案，尤其不适合多租户、多实例共享调度。

### 2.3 Wunder 的正确方向

- 保留 **数据库持久化 + claim due jobs** 的核心模式。
- 吸收 openclaw 的 **轻量调度策略、错误退避、观测能力与边界处理**。
- 最终演进为 **DB lease + 唤醒通知 + 公平调度** 的生产级任务系统。

## 3. 目标架构

### 3.1 保留的核心

- `cron_jobs` / `cron_runs` 继续作为统一事实来源。
- `claim_due_cron_jobs(now, limit)` 继续承担跨进程抢占职责。
- `CronScheduler` 继续作为 server/desktop 共用的调度入口。

### 3.2 需要演进的核心能力

#### A. 调度策略层

- 从“固定轮询”升级为“**下一次到期驱动 + 空闲上限睡眠**”。
- 进一步演进到“**DB 唤醒通知 + 保底轮询**”。

#### B. 执行保护层

- 对周期任务增加 **错误退避**，避免瞬时故障导致重试风暴。
- 后续增加 **执行超时、卡死回收、misfire policy**。

#### C. 运行权属层

- 后续为 `running_at` 补充 `runner_id / run_token / lease_expires_at / heartbeat_at`。
- 禁止通过“启动时全表清空 running”恢复现场，改成 lease 过期回收。

#### D. 观测与治理层

- 增强 `/wunder/cron/status` 输出调度策略参数。
- 后续增加 run retention、按用户/组织公平调度、WebSocket 事件流。

## 4. 分阶段落地

### P0：低风险增量优化（本轮开始落地）

1. 增加空闲睡眠上限 `cron.max_idle_sleep_ms`，降低无效轮询成本。
2. 为周期任务引入错误退避，优先保护系统稳定性。
3. 把调度策略抽成独立模块，避免 `src/services/cron.rs` 继续堆积。

### P1：生产级稳定性

1. 引入 lease/owner/token，替代当前单纯 `running_at`。
2. 增加执行超时与 stuck-run 回收。
3. 引入 scheduler wakeup 机制，减少新增/更新任务后的等待时间。

### P2：大规模任务能力

1. 支持 misfire policy（`coalesce/skip/fire_once`）。
2. 支持 per-user / per-org / per-agent 并发与公平调度。
3. 支持运行日志保留策略、历史归档与聚合查询。

## 5. 本轮已落地

本轮先完成 P0 的第一批低风险改造：

1. **策略模块化**：新增 cron policy 子模块，封装调度睡眠和错误退避规则。
2. **空闲睡眠上限**：新增 `cron.max_idle_sleep_ms`，空闲时最长按该值休眠，降低空转轮询。
3. **错误退避**：周期任务失败后不再立刻按自然周期重试，而是取“自然下一次执行时间”和“退避时间”中的较大值。

当前版本已落地 DB lease + wakeup：新增 `runner_id / run_token / heartbeat_at / lease_expires_at`，启动时不再全表清空 `running_at`，改为依赖租约过期回收。

补充：本轮第二批实现已先行落地 **scheduler wakeup**。任务新增、更新、删除、启停后会主动唤醒调度循环，避免新任务必须等待下一次固定 sleep 周期才能生效；任务运行完成后也会主动唤醒调度器，以便更快释放并发槽位并检查新的到期任务。

## 6. 风险说明

- `max_idle_sleep_ms` 会带来少量“新任务创建后等待下次调度循环”的冷启动延迟；当前用较保守默认值控制影响。
- 错误退避会让异常任务恢复速度变慢，但能显著降低错误风暴对系统吞吐和数据库的冲击。
- 由于本轮未改动 DB schema，属于低风险兼容优化，适合先上线观察。

## 7. 下一步

下一轮建议直接推进以下事项：

1. 已完成：为 `cron_jobs` 补 lease 字段并改造 claim / finish / recovery / heartbeat 流程。
2. 为 scheduler wakeup 增加更细粒度测试，覆盖“休眠中新增任务立即生效”的场景。
3. 在 Postgres server 形态下进一步评估 `LISTEN/NOTIFY` 方案，减少兜底轮询压力。
