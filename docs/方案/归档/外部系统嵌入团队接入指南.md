# 外部系统嵌入团队接入指南

## 1. 适用范围

本文用于团队把第三方业务系统中的已登录用户，直接嵌入到 wunder 用户侧聊天页（`iframe` 或新窗口）。当前支持两种接入模式：

- 推荐：团队系统前端直接带 `token + user_id` 打开 wunder 登录页，wunder 自行校验并跳转到嵌入聊天页。
- 兼容：团队系统后端调用 `POST /wunder/auth/external/launch`，拿到 `entry_path` 后再让前端打开。

## 2. 推荐接入目标（一句话）

团队系统直接打开：

```text
http://<wunder-host>/login?token=<team_jwt>&user_id=<team_user_id>
```

wunder 会自动：

1. 校验团队 JWT；
2. 自动创建/复用 wunder 用户；
3. 自动定位到预制智能体聊天页；
4. 最终进入 `/app/embed/chat`。

这条链路不要求团队系统新增一个“launch 后端接口”。

---

## 3. wunder 侧需要做什么

### 3.1 必配项

```yaml
security:
  external_embed_preset_agent_name: 文稿校对
  external_embed_jwt_secret: ${WUNDER_EXTERNAL_EMBED_JWT_SECRET:-}
  external_embed_jwt_user_id_claim: ${WUNDER_EXTERNAL_EMBED_JWT_USER_ID_CLAIM:-sub}
```

说明：

- `external_embed_preset_agent_name`：外链默认进入的预制智能体名称。
- `external_embed_jwt_secret`：团队系统 JWT 的 HS256 签名密钥，wunder 用它校验 `token`。
- `external_embed_jwt_user_id_claim`：JWT 中代表用户 ID 的 claim，默认取 `sub`。

### 3.2 当前校验规则

wunder 目前对直达模式做以下校验：

- 仅支持 `HS256`；
- 会校验 JWT 签名；
- 会校验 `exp` 是否过期；
- 会校验 `external_embed_jwt_user_id_claim` 对应值是否等于 URL 里的 `user_id`。

如果以上任一条件不满足，则不会放行。

---

## 4. 团队系统需要做什么

### 4.1 最小改动方案

团队系统前端只需要把 iframe 或新窗口地址改成：

```html
<iframe
  src="http://192.168.79.67:18001/login?token=<TEAM_JWT>&user_id=1"
  style="width:100%;height:100%;border:0;"
></iframe>
```

或者：

```js
window.open(
  `http://192.168.79.67:18001/login?token=${teamToken}&user_id=${currentUserId}`,
  '_blank'
);
```

### 4.2 参数约定

- `token`：团队系统已经签发给当前登录用户的 JWT。
- `user_id`：团队系统当前用户 ID，必须与 JWT 中配置的 claim 一致。

建议：

- 若团队系统用户 ID 可能与其他系统冲突，建议后续统一做命名空间，例如 `oa_1`、`erp_1`。
- 如果后续需要映射部门，可追加改造 wunder，支持 `unit_id` 一并透传；当前这版先以 `token + user_id` 直达为主。

---

## 5. wunder 内部实际发生了什么

当用户打开：

```text
/login?token=<TEAM_JWT>&user_id=1
```

用户侧前端会自动调用：

```text
POST /wunder/auth/external/token_launch
```

请求体：

```json
{
  "token": "<TEAM_JWT>",
  "user_id": "1"
}
```

wunder 校验成功后会返回：

```json
{
  "data": {
    "code": "wundc_xxx",
    "expires_at": 1760000000.0,
    "entry_path": "/app/embed/chat?wunder_code=...&agent_id=...&embed=1",
    "agent_id": "agent_xxx",
    "agent_name": "文稿校对",
    "created": false,
    "updated": true
  }
}
```

随后前端会自动跳到 `entry_path`，再用一次性码换取 wunder 自己的 `access_token`，整个过程对团队系统透明。

---

## 6. 用户落库与智能体规则

- 若 wunder 中不存在该用户，则自动创建；
- 新创建的外链用户默认密码固定为 `123456`；
- 若已存在，则复用原用户；
- 外链默认仅允许进入 `security.external_embed_preset_agent_name` 指定的预制智能体；
- 若该用户名下已存在同名用户智能体，则优先进入用户自己的同名智能体。

---

## 7. 兼容旧方案：`external/launch`

如果某些团队后端愿意改造，仍可继续使用：

```text
POST /wunder/auth/external/launch
```

这种模式依赖团队后端保存 `WUNDER_EXTERNAL_AUTH_KEY`，由后端先换取 `entry_path`，再把地址交给前端打开。

适用场景：

- 团队系统没有现成 JWT；
- 不方便共享 JWT 签名密钥给 wunder；
- 希望继续由团队后端集中控制接入。

---

## 8. 验收步骤

### 8.1 wunder 侧

- [ ] 已配置 `security.external_embed_preset_agent_name`
- [ ] 已配置 `security.external_embed_jwt_secret`
- [ ] 已确认 `security.external_embed_jwt_user_id_claim` 与团队 JWT 一致（默认 `sub`）

### 8.2 联调

1. 在团队系统中拿到一个真实登录用户的 JWT；
2. 手工打开：

```text
http://192.168.79.67:18001/login?token=<真实JWT>&user_id=<真实用户ID>
```

3. 观察是否自动跳转到 `/app/embed/chat`；
4. 检查用户是否被自动创建/复用；
5. 检查进入的是否是预制智能体或同名用户智能体。

---

## 9. 常见问题

### 9.1 打开后回到登录页

通常是以下原因：

- `external_embed_jwt_secret` 未配置；
- JWT 签名不对；
- `user_id` 与 JWT 中的 `sub`（或配置的 claim）不一致；
- JWT 已过期。

### 9.2 用户能进，但不是目标智能体

检查：

- `security.external_embed_preset_agent_name` 是否配置正确；
- wunder 中是否已经存在同名用户智能体（这种情况下会优先进入用户自己的同名智能体）。

### 9.3 团队系统只能给 token，不能给 username

没有问题，这版就是按 `user_id` 直接建/复用 wunder 用户。

---

## 10. 安全提醒

- 不要把 wunder 的 `external_auth_key` 暴露到前端；
- `token + user_id` 直达模式的安全边界在于 wunder 对团队 JWT 的签名校验，请妥善保管 `external_embed_jwt_secret`；
- 建议团队系统与 wunder 全链路使用 HTTPS；
- 若使用 iframe，请同步检查网关的 `X-Frame-Options` 与 CSP `frame-ancestors`。
