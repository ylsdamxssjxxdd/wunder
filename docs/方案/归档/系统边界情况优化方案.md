# wunder 系统边界情况优化方案

## 1. 文档目标

本文基于 wunder 当前实现，对照参考项目 `C:\Users\32138\Desktop\参考项目\codex-rust-v0.113.0` 的关键设计，给出一份面向落地的系统边界优化方案。分析重点不是“界面或单点功能”，而是四个决定长期稳定性的底层问题：

- 智能体边界是否清晰，线程/轮次/工具/事件是否各归其位。
- 稳定性控制是否在正确层次，能否中断、恢复、卸载、重放且不留脏状态。
- 运行速度优化是否是端到端的，而不是零散补丁。
- 兼容性设计是否能同时覆盖多模型、多平台、多端接入与旧接口平滑演进。

本文与 [docs/基础工具优化方案.md](./基础工具优化方案.md) 的关系如下：

- 那份文档聚焦“基础工具运行时”。
- 本文聚焦“线程/轮次/事件/协议/兼容层”的系统边界。
- 两者应最终汇合为同一套 runtime + orchestrator + protocol 体系。

## 2. 核心结论

### 2.1 当前 wunder 已经具备的正确方向

从当前代码看，wunder 已经有几条非常关键的正确设计：

- 已明确 `user_round + model_round` 作为运行刻度，见 `src/orchestrator/execute.rs`、`src/ops/monitor.rs`。
- 已落实线程级 system prompt 冻结与首轮记忆快照注入，见 `src/orchestrator/prompt.rs`、`src/orchestrator/memory.rs`。
- 已具备 WebSocket 优先、SSE 兜底、事件持久化与回放能力，见 `src/api/chat_ws.rs`、`src/orchestrator/event_stream.rs`。
- 已开始向 Codex 靠拢做预判压缩、工具输出护栏、搜索双引擎、补丁拦截和 freeform tool 兼容，见 `src/orchestrator/memory.rs`、`src/services/tools.rs`、`src/services/tools/search_content_tool.rs`、`src/services/tools/apply_patch_tool.rs`、`src/services/llm.rs`。

这说明 wunder 不需要推倒重来，真正需要做的是“把已经存在的好设计从局部能力提升为系统边界”。

### 2.2 与 Codex 对照后的四个主要缺口

#### 缺口一：持久线程、加载线程、执行轮次还没有彻底分层

wunder 现在主要以 `session_id` 为主索引，再叠加 `AgentRuntime` 的主会话/队列逻辑。它能工作，但“会话线程”和“当前正在执行的一轮”仍然耦合较重。

Codex 参考实现把这件事拆成了三层：

- 持久线程：`codex-rs/core/src/thread_manager.rs`
- 加载态/订阅态：`codex-rs/app-server/src/thread_state.rs`、`codex-rs/app-server/src/thread_status.rs`
- turn 级临时状态：`codex-rs/core/src/state/turn.rs`

这三层拆开后，中断、恢复、卸载、审批清理、订阅取消、错误回收都会自然很多。

#### 缺口二：稳定性控制还偏“请求链路内解决”，没有形成线程运行时外壳

wunder 现在的大部分稳定性逻辑都在 `Orchestrator::execute_request` 周边完成。这样做短期快，但长期会出现两个问题：

- 断线、取消、审批、工具执行、流式重试都在请求链路内互相影响。
- 前端、渠道、A2A、cron、desktop/cli 复用时，很难保证同样的运行语义。

Codex 的做法是把稳定性控制放到 thread runtime 和 turn state 外层壳里，执行器只是“被壳调度”。

#### 缺口三：速度优化已经不少，但仍偏点状，缺少 warm thread 和统一执行核

wunder 已经有 prompt cache、压缩、搜索双引擎、命令流式输出、工具并行执行。但从系统视角看，热路径仍然缺两块：

- 缺少“线程加载后常驻”的 warm runtime，导致每轮都重复装配较多上下文。
- 缺少统一的 tool runtime kernel，导致本机命令、sandbox、MCP、A2A、补丁、审批在不同分支里仍有语义差异。

#### 缺口四：兼容性已经覆盖模型 provider，但协议与平台兼容层还不够独立

`src/services/llm.rs` 已经做了较强的 OpenAI/Responses/OpenAI-compatible 兼容，但当前兼容逻辑仍然集中在一个大文件和 Orchestrator 逻辑附近。协议兼容、模型兼容、平台兼容还没有彻底解耦。

## 3. 对照分析

### 3.1 边界对照

| 维度 | wunder 现状 | Codex 参考设计 | 结论 |
| --- | --- | --- | --- |
| 线程主边界 | `AgentRuntime` 用 `user_id + agent_id + session_id` 管理主会话、队列、忙闲 | `ThreadManager` 管理持久 thread；app-server 只管理 loaded thread | wunder 线程边界可用，但持久 thread 与 loaded runtime 未彻底分离 |
| 轮次边界 | `user_round + model_round` 已有，且监控已持久化 | Thread / Turn / Item 三层协议非常明确 | wunder 的轮次概念正确，应继续保留并制度化 |
| 提示词冻结 | 首条用户消息后冻结 system prompt；AGENTS.md 快照只注入一次 | 线程上下文与 turn state 分离，历史重建依赖冻结基线 | wunder 在提示词冻结上已经接近 Codex，应作为不可回退约束 |
| 审批归属 | WS 层维护 `pending_approvals`，以连接态为主 | turn state 自己持有 pending approvals / user input / elicitation | wunder 目前审批更像“传输层状态”，需要提升为“turn 状态” |
| 线程卸载 | 有取消、回放、队列，但没有“最后订阅者离开后卸载线程”的统一语义 | `thread/unsubscribe`、`thread/closed`、`notLoaded` 状态完整 | wunder 需要把“运行中线程”和“持久线程”彻底分开 |

### 3.2 稳定性对照

| 维度 | wunder 现状 | Codex 参考设计 | 结论 |
| --- | --- | --- | --- |
| 取消语义 | 以 `session_id` 取消为主，执行链路内部轮询取消 | `turn/interrupt` 精确到 `(thread_id, turn_id)`，pending request 自动清理 | wunder 需要 turn 级中断句柄 |
| 流式失败恢复 | LLM 层已有流式重试；上下文溢出恢复和 compaction 已落地 | 有明确回归：stream error 后允许下一轮继续 | wunder 应增加“失败后释放 active turn”回归矩阵 |
| 工具并发 | `execute_tool_calls_parallel` 并行执行，统一超时与审批 | mutating tool 通过 gate 串行，read-only 才并行 | wunder 当前没有 mutating tool gate，存在文件/命令竞态风险 |
| 线程订阅管理 | WS/SSE 可回放事件，但连接断开后的线程加载状态没有独立层 | loaded/unloaded/status changed 都是独立协议事件 | wunder 需要独立的 thread watch/status manager |
| 审批清理 | 连接结束时清理 pending approvals | interrupt/unsubscribe 时同时 resolve server request | wunder 需要把连接清理提升为 turn 生命周期清理 |

### 3.3 速度对照

| 维度 | wunder 现状 | Codex 参考设计 | 结论 |
| --- | --- | --- | --- |
| Prompt 热路径 | 已做 prompt cache；冻结 prompt 复用 | thread resume + context reconstruction + token breakdown | wunder 还缺 loaded thread 常驻与 resume warm hit |
| 工具执行 | 命令流式输出、搜索 auto=rg/rust、apply_patch 拦截已经不错 | `ToolOrchestrator` + `UnifiedExec` + approval cache + runtime split | wunder 需要把已有优化统一到同一执行核 |
| 搜索/读取 | `search_content` 已有双引擎；`read_file` 有结构化读取增强 | `grep_files`、`read_file`、`apply_patch` 都是稳定的标准工具 | wunder 已走在正确方向上，重点是纳入统一 tool runtime 协议 |
| 编排冷启动 | 每轮仍要重新装配 tool bindings、skills、roots、approval state | loaded thread 下不少状态可复用 | wunder 应增加 thread residency 和 lazy refresh |

### 3.4 兼容性对照

| 维度 | wunder 现状 | Codex 参考设计 | 结论 |
| --- | --- | --- | --- |
| 模型 provider | `services/llm.rs` 已兼容 chat/responses/openai-compatible/freeform | provider、wire api、schema 兼容较清晰，token usage 细分更完整 | wunder 需把 provider 兼容从大文件中拆出独立层 |
| 协议 | `/wunder` 与 chat/ws 事件可用，但事件语义偏业务型 | app-server JSON-RPC schema、thread/turn/item 生命周期明确 | wunder 不必照抄 JSON-RPC，但应内化同等清晰的状态机 |
| 平台 | server/desktop/cli 共栈，sandbox 也已存在 | Linux/Windows sandbox、unified exec、capability gating 独立成模块 | wunder 需把平台差异从业务层抽走，收口到 runtime adapter |
| 旧接口兼容 | 当前前端和渠道都依赖 `session_id` 与既有事件名 | Codex 有版本化 protocol/schema/experimental gate | wunder 后续应新增 `thread_id` 但保留 `session_id` 兼容别名 |

## 4. wunder 应保持不变的设计原则

对齐 Codex 不等于复制 Codex。wunder 的以下原则必须保持：

- `/wunder` 仍然保留“传 `user_id + question` 即可工作”的简洁入口。
- `user_id` 仍然允许是虚构标识，不要求是注册用户。
- WebSocket 优先、SSE 兜底的通讯策略不变。
- 线程 system prompt 首轮冻结、长期记忆只在初始化注入一次的规则不变。
- token 统计继续以“上下文占用量”为主，不改成单纯总消耗。
- server 使用 Postgres、desktop 使用 SQLite 的双存储设计不变，但状态机语义必须一致。
- desktop/cli/server 三形态共享同一内核，不允许分叉出三套不同的线程/工具语义。

## 5. 目标设计

## 5.1 新的系统边界模型

建议将 wunder 运行时拆成四层边界：

### A. 持久线程层

职责：

- 表示“一个可以被恢复、归档、分叉的智能体线程”。
- 保存冻结后的 prompt 快照、首条用户消息摘要、模型/审批/sandbox 快照、线程来源。

建议最小字段：

- `thread_id`
- `session_id_compat`：兼容旧接口，初期可与 `thread_id` 1:1
- `user_id`
- `agent_id`
- `status_persisted`：`idle/archived/system_error`
- `model_name/provider/tool_call_mode/approval_mode/sandbox_mode`
- `first_user_message`
- `frozen_system_prompt_hash`
- `context_tokens_last/context_tokens_peak`
- `last_turn_id`

### B. 加载线程层

职责：

- 表示“当前被某个连接、渠道、任务执行器加载到内存中的线程实例”。
- 负责订阅者数量、loaded/unloaded、thread closed、线程级缓存和 lazy refresh。

这是 Codex `ThreadStateManager + ThreadWatchManager` 最值得借鉴的一层。

建议运行时状态：

- `loaded_generation`
- `subscriber_count`
- `active_turn_id`
- `pending_shutdown`
- `last_error`
- `last_token_usage`
- `prompt_cache_hit`
- `tool_bindings_generation`

### C. turn 层

职责：

- 表示一次用户轮次中的活动执行。
- 拥有自己的 cancellation token、pending approvals、pending user input、pending tool outputs、token baseline。

建议状态：

- `turn_id`
- `user_round`
- `model_round_max`
- `status`：`pending/in_progress/waiting/completed/interrupted/failed`
- `token_usage_at_start`
- `pending_approvals`
- `pending_user_inputs`
- `pending_dynamic_requests`
- `running_tasks`

### D. item 层

职责：

- 统一描述模型消息、推理片段、工具调用、命令执行、补丁变更、文件变化、compaction 等可流式消费的最小单位。

这一层是前端稳定性的关键。没有 item 层，事件就只能“看起来能跑”，但很难做稳定 UI。

## 5.2 事件协议重构目标

建议在不破坏当前 `/wunder` 接口的前提下，把内部事件面重构为稳定协议，再由适配层映射给现有前端。

目标事件集：

- `thread_started`
- `thread_status_changed`
- `thread_closed`
- `turn_started`
- `turn_completed`
- `turn_interrupted`
- `item_started`
- `item_delta`
- `item_completed`
- `approval_request`
- `approval_resolved`
- `token_usage_updated`
- `workspace_update`
- `context_compaction`
- `error`

注意：

- 现有 `progress/tool_call/tool_result` 不需要立刻删掉。
- 新协议先内用，旧事件作为 compatibility facade 保留 1~2 个版本周期。

## 5.3 工具运行时边界重构目标

当前 wunder 已经有工具并行、预算、输出护栏和 patch 拦截，但仍建议进一步对齐 Codex 的运行时分层：

### 当前问题

- `execute_tool_calls_parallel` 会并行跑所有 exec calls，没有 mutating tool 串行门。
- 审批逻辑嵌在 Orchestrator 中，连接和工具执行器都感知审批细节。
- 本机命令、sandbox 命令、A2A、MCP、补丁、技能工具仍有多条执行分支。
- 命令白名单仍以前缀字符串匹配为主，且非 allow-all 场景按 `content.lines()` 拆分执行。

### 目标分层

建议新增：

- `src/services/tools/runtime/`
  - `orchestrator.rs`：统一审批、sandbox 选择、重试、超时、输出封装
  - `registry.rs`：工具注册、并行能力、mutating 标记
  - `gate.rs`：mutating tool gate
  - `events.rs`：tool item 生命周期事件
- `src/services/tools/runtime/exec/`
  - `local_exec.rs`
  - `sandbox_exec.rs`
  - `unified_exec.rs`
- `src/services/tools/runtime/approval/`
  - `approval_cache.rs`
  - `request_state.rs`

### 行为要求

- 只允许只读工具并行；写文件、补丁、命令、目录重写、线程控制等 mutating tool 默认串行。
- 审批归属 turn，而不是连接。
- 同一工具调用无论本机、sandbox、MCP、A2A，返回体都收口为统一 envelope：
  - `ok`
  - `data`
  - `error`
  - `meta`
  - `sandbox`
  - `timing`
  - `continuation`

## 5.4 提示词与记忆边界继续固化

wunder 当前最接近 Codex 且最值得保护的设计，是“首轮冻结 + 只注入一次”。

建议继续强化为明确规则：

- 线程创建后，在首条用户消息真正落地时才物化 `frozen_prompt_snapshot`。
- `AGENTS.md`、长期记忆、技能可见性说明、工具清单都进入首轮快照。
- 后续轮次禁止再改写线程 system prompt，只允许在消息层或 compaction summary 中增量补充信息。
- prompt 预览接口必须明确区分：
  - `pending`
  - `frozen`
  - `archived`
- compaction、resume、fork 都不能破坏冻结基线。

## 5.5 兼容层设计

兼容层需要分三块做，而不是只放在 `services/llm.rs`：

### A. 模型兼容层

建议将当前 `src/services/llm.rs` 拆为：

- `provider_normalizer.rs`
- `request_builder_chat.rs`
- `request_builder_responses.rs`
- `stream_decoder_chat.rs`
- `stream_decoder_responses.rs`
- `tool_schema_normalizer.rs`
- `usage_aggregator.rs`

目标：

- 把 provider/wire_api/tool_call_mode/openai schema 差异从 Orchestrator 中拿掉。
- 统一保存 `input_tokens/output_tokens/cached_input_tokens/context_tokens`。
- 为后续接入更多 OpenAI-compatible provider 留出稳定扩展点。

### B. 协议兼容层

建议保留当前用户接口，但增加内部标准字段：

- 请求继续支持 `session_id`
- 响应与事件新增 `thread_id`
- 新前端优先使用 `thread_id`
- 旧前端继续读取 `session_id`

迁移期建议：

- `session_id` = 兼容别名
- `thread_id` = 新主键
- 直到所有前端/渠道/desktop 适配完成，再考虑降级 `session_id` 的内部地位

### C. 平台兼容层

建议把 Linux/Windows/macOS、server/desktop/cli 的差异统一收敛到 runtime adapter：

- `platform_caps.rs`
- `sandbox_policy_resolver.rs`
- `process_adapter_linux.rs`
- `process_adapter_windows.rs`
- `process_adapter_macos.rs`

业务层只看能力：

- 是否支持 PTY
- 是否支持隔离网络
- 是否支持文件 ACL 精细授权
- 是否支持后台进程托管

而不直接在业务逻辑中分支判断平台。

## 6. 分阶段落地方案

## Phase 0：先立边界，不改外观

目标：

- 不改变 `/wunder` 外部接口。
- 先把内部 thread / loaded thread / turn 三层结构建起来。

交付：

- 新增线程元数据模型与 `thread_id`
- 新增 in-memory `LoadedThreadRuntime`
- 新增 `TurnState`
- 现有 `session_id` 全量映射到 `thread_id`
- 为现有 WS/SSE 事件补发 `thread_id/turn_id/item_id`

验收：

- 前端零改动也能继续工作
- 后端日志里能看到完整 `thread_id + turn_id + user_round + model_round`

## Phase 1：稳定性内核重构

目标：

- 让取消、审批、断线、失败恢复都归到线程运行时和 turn state。

交付：

- turn interrupt 机制
- pending approvals / user inputs 迁移到 turn state
- loaded thread unsubscribe / shutdown / close 机制
- active turn 结束后统一清理 pending 状态
- 补齐 thread status：`idle/running/waiting/system_error/not_loaded`

建议新增回归测试：

- stream error 后下一轮仍可继续
- interrupt 运行中 turn 会终止工具并清理审批
- 最后订阅者离开后 thread 进入 `not_loaded`
- system_error thread 在 resume 后恢复 `idle`

## Phase 2：工具运行时统一

目标：

- 把工具优化从“单点增强”变成统一 tool runtime。

交付：

- mutating tool gate
- approval cache
- local/sandbox exec 统一 envelope
- apply_patch / execute_command / write_file / search_content / read_file 全部接入统一 runtime
- 将现有 `tool_runtime_v2_enabled` 开关扩展到真正的 runtime v2

验收：

- 写入类工具不再与其它 mutating tool 并发
- sandbox 与本机命令返回结构完全一致
- 高风险工具审批链路断线后无挂起请求

## Phase 3：速度优化走向 warm runtime

目标：

- 将速度优化从 point optimization 升级为 warm path optimization。

交付：

- loaded thread 常驻缓存
- tool bindings / skills snapshot lazy refresh
- prompt snapshot 复用
- warm resume 命中率指标
- 命令执行 approval canonical key

重点指标：

- thread resume P95
- 首字节延迟 P95
- tool 首次执行与二次执行时延差
- compaction 后恢复请求时延

## Phase 4：兼容层与协议收口

目标：

- 内部协议标准化，外部接口保持兼容。

交付：

- provider compat 模块拆分
- thread/turn/item 事件协议统一
- `thread_id` 在 frontend/web/desktop/渠道侧全面落地
- 旧 `session_id` 兼容层保留

验收：

- 新旧客户端可同时工作
- OpenAI / OpenAI-compatible / Responses API 行为差异被隔离到 compat 层
- Linux/Windows/desktop/server 的运行语义一致

## 7. 关键测试矩阵

建议直接参考 Codex 的测试思路，补齐 wunder 自己的矩阵：

- 线程边界
  - 首轮冻结 prompt，后续记忆变化不改写线程快照
  - fork 新线程后继承历史但重新建立 turn runtime
  - unloaded thread resume 后状态恢复
- turn 稳定性
  - stream error 不阻塞下一轮
  - interrupt 正在执行的命令/补丁/审批
  - waiting 状态下 user input 恢复继续
- 工具并发
  - read-only 工具可并行
  - mutating tool 自动串行
  - mutating/read-only 混合时顺序可预测
- 协议兼容
  - 旧 `session_id` 客户端与新 `thread_id` 客户端同时联调
  - WS 与 SSE 事件载荷等价
  - 渠道、cron、chat API、desktop 本地模式使用同一事件语义
- provider 兼容
  - chat/responses 返回 tool_calls
  - freeform tool 和 function tool 混用
  - cached_input_tokens / context_tokens 聚合正确
- 平台兼容
  - Linux sandbox
  - Windows sandbox / 本机执行
  - desktop 本地模式工作目录与 server 模式工作目录语义一致

## 8. 建议验收指标

### 边界清晰度

- 100% 的运行事件都能关联 `thread_id + turn_id + item_id`
- 100% 的审批请求都能关联到唯一 turn
- 100% 的 turn 中断都能清理 pending state

### 稳定性

- stream error 后下一轮成功率 >= 99%
- pending approval 泄漏数 = 0
- unloaded thread 后内存残留 thread 数量收敛到 0
- 连续 72h 压测无 loaded thread 泄漏

### 速度

- warm thread resume P95 降低 30% 以上
- 工具首字节延迟 P95 降低 20% 以上
- 搜索/读取/补丁进入统一 runtime 后，工具结果结构一致率 100%

### 兼容性

- 旧前端无需一次性切换即可继续工作
- desktop/server/cli 三形态线程协议一致
- OpenAI / OpenAI-compatible / Responses API 回归样例通过率 100%

## 9. 最终建议

这次优化最重要的不是再补一批工具能力，而是把 wunder 的内核运行边界正式制度化：

- 持久线程是持久线程，加载线程是加载线程，turn 是 turn。
- 审批、取消、工具执行、流式事件都必须挂靠到 turn runtime，而不是散落在连接层和业务层。
- prompt 冻结、记忆单次注入、user_round/model_round 这些已经正确的设计必须继续保留，并上升为架构约束。
- 速度优化要围绕 warm thread、统一 tool runtime、mutating tool gate、approval cache 做，而不是继续零散地在单个工具上修修补补。
- 兼容层要独立出来，避免未来模型 provider、平台和前端协议变化再次穿透到 Orchestrator 主链路。

如果按这个路线推进，wunder 会从“已经能跑、功能很多的原型系统”，升级到“线程边界清晰、可恢复、可扩展、可长期维护的智能体运行平台”。这一步是后续做多智能体、渠道大规模接入、desktop/server 长期并行演进的前置条件。
