# wunder 自动化测试大纲（v2.0）

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 文档版本 | v2.0 |
| 更新日期 | 2026-03-21 |
| 目标 | 建立覆盖 server / cli / desktop / frontend / admin 的自动化回归体系 |
| 核心关注 | 功能正确性、隐性链路一致性、性能、稳定性、管理员治理能力 |

## 2. 自动化覆盖原则

- 分层执行：静态检查 -> 单元/组件 -> 集成/接口 -> E2E/回放 -> 性能稳定性。
- 先快后全：PR 跑最小阻断集；Nightly 跑全量回归和 Soak。
- 覆盖隐性规则：线程提示词冻结、长期记忆仅初始化注入、token 占用口径、队列/重试幂等。
- 管理员侧优先：管理员治理链路必须有独立回归集，不依赖用户侧间接覆盖。

## 4. 自动化分层大纲

## 4.1 L0 静态与编译门禁（PR 必跑）

- Rust：
  - `cargo fmt --all -- --check`
  - `cargo check --lib`
- Frontend：
  - `npm run typecheck:frontend`
- 基础脚本：
  - `python scripts/check_mojibake.py`

目标：
- 杜绝语法、类型、编码类低级回归进入主分支。

## 4.2 L1 单元/模块回归（PR 必跑）

- Rust `--lib` 单测：
  - 渠道适配器纯函数、URL 识别、附件推断、日志聚合、上下文压缩逻辑。
- Frontend 脚本回归：
  - `test:chat-pending`
  - `test:chat-timing`
  - `test:message-render`
  - `test:realtime-pulse`

重点补强项（新增）：
- token 占用统计口径单测（上下文占用 vs 总消耗）。
- 线程提示词冻结与记忆单次注入规则单测。
- QQ 出站文本引用转换（含 `temp_dir/download?filename=`）单测。

## 4.3 L2 路由与集成回归（PR/Release 必跑）

- 建议分组执行并行化：
  - `cargo test --test preset_agent_model_sync`
  - `cargo test --test admin_channel_accounts_routes`
  - `cargo test --test channel_runtime_logs_routes`
  - `cargo test --test qqbot_webhook_runtime_logs`
  - `cargo test --test memory_routes`
  - `cargo test --test agent_model_routes`
  - `cargo test --test inner_visible_agent_routes`
  - `cargo test --test monitor_log_profile_regression`
  - `cargo test --test cron_tool`
  - `cargo test --test cron_lease`
  - `cargo test --test cron_finish_job`
  - `cargo test --test cron_session_delete`
  - `cargo test --test wunder_cli_line_chat_smoke`

目标：
- 覆盖管理员治理主链路、用户核心能力、渠道和计划任务生命周期。

## 4.4 L3 E2E/协议回放（Nightly 必跑）

- 网关协议与抗异常：
  - `cargo test --test gateway_regression -- --nocapture`
- 事件顺序：
  - `node tests/watch_event_order.mjs`
- 额外 Python 集成：
  - `python -m pytest tests/test_extra_mcp_database_unicode.py`

重点补强项（新增）：
- 用户侧 + 管理员侧浏览器 E2E（Playwright）：
  - 管理员登录、用户治理、预设同步、渠道账号治理、monitor cancel/compaction。
  - 用户聊天流式、审批、WS/SSE 切换、工作区文件闭环。
- Desktop E2E：
  - 本地/远程切换、本地 Python 版本展示与修改、容器映射持久化。

## 4.5 L4 性能与稳定性（Nightly/周跑）

自动化专项：
- 吞吐与延迟基线：调用 `/wunder/admin/throughput/*` 与 `/wunder/admin/performance/sample`。
- Benchmark 基线：`/wunder/admin/benchmark/*` 持续跑固定任务集。
- Soak（8h/24h）：长会话 + 渠道出站 + 文件操作混合负载。

建议阈值（可按环境微调）：
- 功能错误率：`< 1%`
- P95 延迟：较基线退化不超过 `15%`
- Soak 期间资源：内存曲线无持续单调上升；线程与队列无持续堆积

## 5. 覆盖矩阵（重点补全“隐性 + 管理员 + 稳定性”）

| 域 | 自动化级别 | 当前状态 | 补齐要求 |
| --- | --- | --- | --- |
| 管理员治理（用户/权限/工具授权） | L2/L3 | 已有部分 | 补 E2E：批量操作、并发保存、越权拦截 |
| 预设智能体与默认智能体同步 | L1/L2 | 已有较完整 | 增加 UI E2E 与冲突并发回归 |
| 用户聊天与传输 | L1/L3 | Frontend 脚本覆盖部分 | 增加 WS/SSE 断连恢复端到端回放 |
| 线程提示词冻结/记忆注入时机 | L1/L2 | 已有部分 | 固化为发布阻断断言 |
| token 占用统计一致性 | L1/L2 | 有回归历史 | 前后端双断言（接口 + UI） |
| 渠道入出站（含 QQ） | L1/L2/L3 | 已有基础 | 增加“文本引用->媒体/下载链接”回归集 |
| 工作区与容器映射 | L2/L3 | 覆盖不足 | 补 desktop + API 双链路自动化 |
| CLI 体验稳定性 | L2 | 有 smoke | 补附件、工具调用、错误恢复 |
| 管理员监控/调试/日志画像 | L2/L3 | 已有部分 | 补 cancel/compaction/调试画像联动断言 |
| 性能、吞吐、Soak | L4 | 缺系统化流水线 | 建夜间任务 + 退化报警 |

## 6. 推荐流水线（CI）

| 阶段 | 触发 | 任务 |
| --- | --- | --- |
| PR Gate（10~20 分钟） | 每次 PR | L0 + L1 + 关键 L2（预设/渠道/记忆/cron） |
| Release Gate（30~90 分钟） | 发版前 | 全量 L2 + 关键 L3（gateway + event order） |
| Nightly（2~8 小时） | 每晚 | L3 全量 + L4 性能基线 + Soak 短跑 |
| Weekly（8~24 小时） | 每周 | 长 Soak + 全 Benchmark + 趋势报表 |

## 7. 发布阻断规则（自动化）

- PR Gate 必须 100% 通过。
- Release Gate 失败项中，涉及管理员治理、聊天主链路、渠道投递、工作区写入任一项失败即阻断发布。
- 性能退化超过阈值且无豁免说明时阻断发布。

## 8. 自动化维护规则

- 每个线上缺陷至少补 1 条自动化用例（先后端，再前端 E2E）。
- 每次功能升级同步更新本大纲与执行命令，不允许“文档与流水线脱节”。
- 回归资产按模块归档，避免单文件过大；新增测试优先放独立文件并保持职责清晰。
