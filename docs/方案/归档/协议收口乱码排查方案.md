# 协议收口乱码排查方案

## 1. 目标

- 收口 `worker-card` 与默认智能体相关协议，减少后端、导入导出链路、前端之间的重复定义。
- 建立可重复执行的乱码扫描机制，避免 UTF-8/GBK 混用或恢复文件损坏再次进入主线。
- 优先修复影响真实功能的协议分叉和乱码问题，再处理文档与图资产的存量清理。

## 2. 现状

### 2.1 协议重复点

| 范围 | 位置 | 现状问题 |
| --- | --- | --- |
| `worker-card` 提示词协议 | `src/services/inner_visible/worker_card.rs` | 有独立 `WorkerCardPrompt` 与 prompt 合并逻辑 |
| `worker-card` 导入导出 | `src/services/hive_pack.rs` | 有另一套 `WorkerCardPrompt` 与 prompt 合并逻辑 |
| `worker-card` 前端读写 | `frontend/src/utils/workerCard.ts` | 前端再实现一套 `extra_prompt/prompt.extra_prompt` 兼容逻辑 |
| 默认智能体快照 | `src/api/user_agents.rs` / `src/services/default_agent_sync.rs` / `src/services/user_store.rs` / `src/services/inner_visible/mod.rs` | 多份快照结构并存，后续易继续分叉 |

### 2.2 乱码现状

- `src/api/user_agents.rs` 文件头注释、`frontend/src/views/ChatView.vue` 缺失资源判定正则、`src/services/user_store.rs` / `src/services/hive_pack.rs` 首批测试字符串乱码均已修复。
- `docs/diagrams/system-intro/07-deployment-topology.svg` 已用 `web/docs` 下的干净副本回填，`png` 也已同步。
- 当前基于 `python scripts/check_mojibake.py` 的全仓扫描结果为通过，后续新增文件继续以脚本为准做回归守门。

## 3. 分阶段推进

### Phase 1：先收口 `worker-card` prompt 协议

- 新增后端共享模块，统一 `WorkerCardPrompt` 结构与 prompt 解析函数。
- 对外保持顶层 `extra_prompt` 为主格式。
- 兼容读取旧格式 `prompt.extra_prompt` 与 `prompt.system_prompt`。
- 先让 `inner_visible` 与 `hive_pack` 共用这一套逻辑，避免继续分叉。

### Phase 2：再收口默认智能体状态快照

- 抽出默认智能体共享快照结构，统一 API、内见同步、用户存储的读写模型。
- 明确“文件外改 -> 设置页回显 -> 后端持久化”的单一事实来源和回写时机。
- 补默认智能体外改同步、首次加载、损坏回退的回归测试。

### Phase 3：建立乱码排查基线

- 新增 `scripts/check_mojibake.py`，扫描 UTF-8 解码失败、`NUL` 字节和常见乱码特征。
- 首批扫描范围：`src/`、`frontend/src/`、`tests/`、`docs/`、`web/`。
- 优先修复运行时源码、设置页、导入导出和高频测试中的乱码。

### Phase 4：处理文档与图资产

- 清理 `docs/开发进度修复方案.md` 等可直接恢复的 Markdown。
- 清理 `svg/html/json` 中的错误编码文本。
- 对无法可靠恢复的内容，直接按当前实现重写，不保留损坏副本继续流转。

## 4. 验证口径

- Rust：`cargo fmt`、`cargo check --lib`，必要时补相关单测。
- Frontend：`npm run typecheck:frontend`。
- 乱码扫描：`python scripts/check_mojibake.py <paths...>`。
- 业务回归：
  - 默认智能体手工修改 `worker-card` 后无需手动点保存即可在设置页回显。
  - `worker-card` 导出保持紧凑结构，默认只输出 `abilities.tool_names/skills` 与顶层 `extra_prompt`。
  - HivePack 导入导出仍兼容旧 `prompt.*` 字段。

## 5. 当前落实情况

- Phase 1 第一批已完成：后端已抽出共享 `worker-card` prompt 协议模块，`inner_visible` 与 `hive_pack` 已复用。
- Phase 2 第一批已完成：默认智能体共享快照结构、meta key、record/config 转换已抽到统一模块，`api`、`default_agent_sync`、`inner_visible`、`user_store` 已接入。
- Phase 3 已完成首轮闭环：新增 `scripts/check_mojibake.py`，并已对代码、文档与图资产完成一轮全仓扫描。
- Phase 4 已完成首个资产修复：部署拓扑图的 SVG/PNG 已与干净发布副本对齐。

## 6. 下一步

- 继续收口默认智能体的归一化逻辑，减少 `api/default_agent_sync/inner_visible/user_store` 之间剩余的行为分叉。
- 评估是否将 `frontend/src/utils/workerCard.ts` 的 prompt/abilities 兼容逻辑继续向协议层规则对齐，避免前后端再次漂移。
- 将乱码扫描脚本接入固定自检流程，避免恢复文件或外部拷贝再次把坏编码带回仓库。
