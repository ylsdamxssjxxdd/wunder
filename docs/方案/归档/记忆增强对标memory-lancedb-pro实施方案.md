# 记忆增强对标 `memory-lancedb-pro` 实施方案

## 1. 目标

本文用于对标参考项目 `C:\Users\32138\Desktop\参考项目\memory-lancedb-pro-master`，明确 `wunder` 当前记忆系统与参考实现的差距、约束条件、分阶段实施路线，以及本轮已经开始落地的第一阶段改造。

本文是对 `docs/记忆增强落地方案.md` 的补充，不替代原总方案。

## 2. 对标结论

参考项目并不是单一的“记忆工具”，而是一套完整的长期记忆子系统，核心上可以拆成五层：

1. **运行时 Hook 层**
   - 在 `index.ts` 里通过 `before_agent_start`、`agent_end`、`command:new` 等生命周期接入自动召回、自动捕获、会话记忆与反思流程。
2. **工具面层**
   - 在 `src/tools.ts` 中提供 `memory_recall`、`memory_store`、`memory_update`、`memory_forget` 等专用工具，面向模型直接使用。
3. **检索内核层**
   - 在 `src/retriever.ts` 中实现向量检索、BM25、融合、重排、MMR、多样性控制与生命周期打分。
4. **提炼内核层**
   - 在 `src/smart-extractor.ts` 中实现 6 类记忆分类、L0/L1/L2 结构化表达，以及 `create / merge / supersede / contradict` 等决策。
5. **生命周期与运维层**
   - 在 `src/decay-engine.ts`、`src/tier-manager.ts`、CLI 命令与迁移脚本中实现分层、衰减、升级、迁移、重建等能力。

它的强点不只是“能记住”，而是把 **写入质量、召回质量、用户可控性、运维可持续性** 做成了一套体系。

## 3. `wunder` 的原则边界

`wunder` 不能直接照搬参考项目，必须遵守当前产品原则：

- **线程级 system prompt 一旦在线程建立时确定，就保持冻结。**
- **长期记忆只允许在线程初始化时注入一次快照。**
- **同线程后续轮次不再自动改写 system prompt。**
- **后续对最新记忆的读取，必须由智能体主动调用记忆工具完成。**

这意味着：

- 参考项目的“每轮主推理前自动 recall 并再次注入 prompt”机制，`wunder` 不能直接采用。
- `wunder` 的增强重点应放在：
  - 线程初始化时的记忆快照质量
  - 智能体主动 recall 的工具体验
  - 自动提炼质量
  - 用户侧可视化与纠偏能力

## 4. 当前已具备能力

相对参考项目，`wunder` 当前已经形成了第一层可用闭环：

- **结构化记忆碎片模型**：记忆以 `memory_fragments` 为中心，而不是只有整段会话摘要。
- **用户可视化管理**：用户侧已有“记忆碎片”页面，可按卡片浏览、编辑、删除、置顶、作废/恢复。
- **记忆命中说明**：已有 `memory_hits` 记录与最近命中原因展示基础。
- **后台提炼作业**：已有 `memory_jobs` 记录自动提炼状态。
- **线程提示词冻结**：线程初次物化时注入记忆快照，后续轮次不再自动改写 system prompt。
- **工具主动 recall**：同线程后续记忆读取依赖记忆工具主动检索。

这说明 `wunder` 已经具备“产品闭环”，但在“记忆内核”上仍然偏轻。

## 5. 与参考项目的主要差距

### 5.1 检索内核仍偏轻

当前 `wunder` 记忆召回以关键词、结构化字段、置顶与近期性为主，`semantic_score` 仍未形成真正的向量语义召回闭环。

差距主要包括：

- 缺少真正的向量召回候选面
- 缺少 BM25 + 向量融合
- 缺少 rerank / MMR / diversity 控制
- 缺少基于生命周期的排序调节

### 5.2 自动提炼仍偏规则型

当前自动提炼已经能识别：

- 回复语言偏好
- 回复风格偏好
- 回复格式偏好
- 用户身份/背景
- 近期计划
- 显式记忆备注

但它本质上仍然是规则型抽取，不是参考项目那种“结构化智能提炼器”，差距包括：

- 还没有 LLM 参与的结构化分类与判定
- 还没有 `merge / supersede / contradict / skip` 决策面
- 还没有从多轮窗口里提炼出更高质量的 L0/L1/L2

### 5.3 生命周期能力还没真正跑起来

虽然 `wunder` 已经有 `tier`、`importance`、`confidence`、`supersedes_memory_id` 等字段，但当前仍偏静态字段，尚未形成完整的：

- 衰减引擎
- 分层晋升/降级
- 事实替代链自动闭环
- 后台重整与清理策略

## 6. 分阶段实施路线

### 6.1 第一阶段：先升级提炼输入质量

第一阶段不先引入 embedding 强依赖，也不恢复“每轮自动 recall 注入 prompt”。

本阶段重点是：

- 让自动提炼不再只看当前问题
- 改为读取“当前问题 + 近期用户轮次窗口”
- 保持完全异步，不阻塞主对话链路
- 不破坏线程 system prompt 冻结原则

这样做的原因：

- 能显著提升“跨轮偏好/计划”提炼质量
- 不增加每轮推理 prompt 抖动
- 不破坏 KV cache 友好性
- 不引入新的模型依赖与部署负担

### 6.2 第二阶段：增强 recall 内核，但只用于工具 recall

在保持线程提示词冻结的前提下，后续可增加：

- 可选 embedding 支持
- 工具 recall 侧的 hybrid 检索
- recall 结果 rerank 与多样性去重
- 更细的命中原因拆解

注意：这一步只增强 **工具 recall 的质量**，而不是恢复每轮自动注入 prompt。

### 6.3 第三阶段：补生命周期内核

后续再逐步增加：

- tier manager
- decay engine
- supersede / contradict 自动链路

当前进展（已完成第一版）：

- 已落地 `supersede` 自动闭环：同 `fact_key` 的新版本事实会保留新旧双记录，并写入 `supersedes_memory_id / superseded_by_memory_id`。
- 已落地 tier 惰性重算：`core / working / peripheral` 会在创建、编辑、列表读取、recall 命中时按访问、时间衰减、重要度与置顶状态重算。
- `confirmed_by_user` 当前仅作兼容旧数据保留，不再作为用户侧操作入口，也不再参与 recall 排序权重。
- 已落地 superseded 过滤：旧版本默认退出 recall，但仍在用户可视化卡片墙中保留查看、编辑、删除能力。
- 维护任务、重建、导入导出、审计与统计

## 7. 本轮已开始实施内容

本轮已经开始落地第一阶段，即“提炼输入质量升级”。

### 7.1 已完成改造

已在 `src/services/memory_auto_extract.rs` 中完成以下调整：

- 新增近期用户消息提炼窗口构建逻辑
- 自动提炼从“只看当前 question”升级为“优先看近期用户消息窗口”
- 将窗口统计信息写入 `memory_jobs.request_payload`
- 增加窗口裁剪、去重与字符预算控制
- 补充覆盖该链路的测试

当前窗口策略：

- 最多读取最近 `16` 条历史消息用于筛选
- 仅提取其中的用户消息
- 最终保留最多 `6` 条最近且去重后的用户文本
- 总字符预算上限 `2400` 字符

### 7.2 触发链路

当前链路如下：

1. 模型产出 `final` 回复
2. 后端异步触发 `capture_turn`
3. `capture_turn` 读取当前问题 + 近期用户消息窗口
4. 自动提炼候选记忆
5. 与现有碎片做规则去重/保护判断
6. 写入 `memory_fragments`
7. 将过程摘要写入 `memory_jobs`

### 7.3 不影响冻结提示词原则

这一阶段的改造只发生在 **回复完成后的异步提炼链路**，不会：

- 修改当前线程已冻结的 system prompt
- 影响当前线程后续轮次的 prompt 结构
- 恢复每轮自动 recall 注入

因此它与 `wunder` 当前“线程提示词冻结”的设计原则完全兼容。

## 8. 下一步实施建议

建议按以下顺序继续推进：

1. **先把提炼窗口方案跑稳**
   - 继续观察真实对话中对“跨轮偏好/计划”的提升效果。
2. **补 recall 工具质量**
   - 强化工具 recall 的排序、解释与去重。
3. **再决定是否接入 embedding**
   - embedding 只作为 recall 增强项，不作为基础依赖。
4. **最后补生命周期引擎**
   - 做 tier、decay、supersede 自动闭环。

## 9. 总结

参考项目值得借鉴的核心，不是“每轮都自动把记忆塞回 prompt”，而是：

- 把记忆做成结构化数据平面
- 把 recall 做成专用工具能力
- 把提炼做成独立内核
- 把生命周期做成持续运维能力

`wunder` 当前的正确路线应当是：

- **坚持线程 system prompt 冻结原则不动摇**
- **优先增强自动提炼质量与工具 recall 质量**
- **embedding 作为可选增强，不作为硬依赖**
- **逐步补齐生命周期与运维层，而不是一次性堆叠复杂度**

这条路线更符合 `wunder` 当前的产品边界、性能目标和长期可维护性。
