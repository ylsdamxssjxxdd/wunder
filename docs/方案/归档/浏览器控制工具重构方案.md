# 浏览器控制工具重构方案

## 0. 目标与约束

- 目标不是把当前 `navigate/click/type/screenshot/read_page/close` 做成更长的动作列表，而是把浏览器能力升级为可长期维护的独立子系统。
- 设计目标对齐 openclaw 的总体思路：
  - 浏览器控制服务独立于模型工具层。
  - 底层以 Chromium/CDP 为主链路，高级能力叠加 Playwright。
  - 模型交互主路径从“猜 CSS selector”切换为“snapshot -> ref -> act”。
  - 支持 profile/tab/session 管理，而不是单页脚本。
  - 从第一阶段就为 Docker headless 运行预留部署模型。
- Wunder 需要同时兼容三种运行形态：`server`、`desktop`、`cli`。其中：
  - `desktop` 是首个完整落地目标。
  - `server/docker` 需要具备 headless 方案。
  - `cli` 先复用控制服务，不单独做浏览器宿主。
- 注意仓库约束：
  - 不继续向 `src/services/tools.rs` 堆浏览器逻辑。
  - 不继续把复杂逻辑留在 `src/services/tools/browser_tool.rs` 里。
  - API 层保持薄层，核心逻辑进入 `src/services/browser/`。

---

## 1. 现状与核心差距

## 1.1 Wunder 当前实现

- 当前浏览器工具本质是“Rust 管理 Python 子进程 + Playwright 脚本”：
  - `src/services/tools/browser_tool.rs`
  - `src/services/tools/browser_bridge.py`
- 当前能力边界：
  - 仅支持 `navigate/click/type/screenshot/read_page/close`。
  - 交互主要依赖 CSS selector，点击失败时再退回文本匹配。
  - 一条 session 对应一个 Python 子进程、一个 browser context、一个 page。
  - 无 profile 管理、无 tab 管理、无 targetId、无 console/network 观察。
  - 无稳定的 UI snapshot/ref 协议，模型需要自己猜 selector。
  - 当前可见性被硬编码限制为 `server.mode=desktop`。
- 当前 Docker 阻塞点：
  - 默认 `server.mode=api`，工具直接不可见。
  - Dockerfile 默认不安装 Playwright Chromium 二进制。
  - 即使强行改成 `desktop`，也只是“容器内直接起 Python+Playwright”，没有专门的容器化策略、运行探针、资源隔离和安全边界。

## 1.2 对齐 openclaw 后需要补齐的关键能力

- 控制面：
  - 独立 browser control service。
  - profile、tab、target 级别的生命周期管理。
  - 控制层与模型工具层解耦。
- 交互面：
  - `snapshot` 输出可供模型消费的结构化 UI 视图。
  - `ref` 成为后续点击、输入、拖拽的主定位方式。
  - `act` 成为统一动作执行面。
- 运行面：
  - Chromium/CDP 为基础能力。
  - Playwright 为高级动作、截图、PDF、role snapshot 提供实现。
  - Docker/headless 有专门模式，不依赖 `desktop` 特判。
- 工程面：
  - 配置拆分清晰。
  - 测试可做单测、集成、Docker 回归。
  - 能在未来支持远程 CDP、existing session attach，但不作为第一阶段强依赖。

## 2. 重构后的总体架构

```text
LLM Tool Layer
  -> browser tool / browser_* aliases
  -> only speaks structured action contracts

Browser Tool Adapter Layer
  -> src/services/tools/browser/
  -> 参数校验、兼容旧别名、结果包装

Browser Control Service
  -> src/api/browser_control/*
  -> src/services/browser/*
  -> profile/tab/session/target lifecycle

Browser Driver Layer
  -> Managed Chromium driver
  -> Existing-session driver (later)
  -> Remote CDP driver (later)

Browser Automation Layer
  -> CDP base ops
  -> Playwright advanced ops
  -> snapshot / refs / act / screenshot / pdf / upload / dialog

Deployment Layer
  -> desktop embedded
  -> server embedded (headless)
  -> docker sidecar (recommended for container deployment)
```

### 2.1 分层原则

- 工具层不再直接启动 Python 脚本。
- 浏览器服务层负责：
  - browser profile 管理
  - tab/target 管理
  - driver 选择
  - Playwright/CDP 会话复用
  - SSRF 与导航策略
  - 下载、截图、临时文件路径治理
- API 层只做薄路由、鉴权和参数解包。
- Docker 场景优先走 sidecar，而不是把浏览器宿主直接塞进主 server 进程。

## 3. 模块拆分方案

## 3.1 新增目录建议

```text
src/
  api/
    browser_control/
      mod.rs
      routes.rs
      request.rs
      response.rs
      auth.rs
  services/
    browser/
      mod.rs
      config.rs
      errors.rs
      runtime.rs
      control_service.rs
      session_registry.rs
      profile_registry.rs
      target_registry.rs
      managed_chromium.rs
      cdp_client.rs
      playwright_runtime.rs
      snapshot.rs
      refs.rs
      act.rs
      screenshot.rs
      observe.rs
      downloads.rs
      uploads.rs
      navigation_policy.rs
      docker_runtime.rs
      storage.rs
  services/
    tools/
      browser/
        mod.rs
        tool.rs
        schema.rs
        legacy_aliases.rs
```

## 3.2 现有文件的处理方式

- `src/services/tools/browser_tool.rs`
  - 第一阶段保留为兼容入口。
  - 第二阶段改成薄适配器，只转发到 `src/services/tools/browser/tool.rs`。
  - 旧的 Python bridge 管理逻辑逐步删除。
- `src/services/tools/browser_bridge.py`
  - 不再作为长期主实现。
  - 第一阶段可保留作为临时回退实现。
  - 当 Rust 服务层的 Playwright/CDP 方案稳定后删除。
- `src/services/tools.rs`
  - 只保留模块注册与 dispatch 接入，不新增实质逻辑。

## 3.3 模块职责

- `runtime.rs`
  - 初始化浏览器运行时。
  - 根据部署形态选择 embedded / sidecar client。
- `control_service.rs`
  - 启动控制服务。
  - 管理生命周期、鉴权、健康检查。
- `session_registry.rs`
  - 维护 `browser_session_id -> runtime`。
  - 处理 idle timeout、并发上限、清理。
- `profile_registry.rs`
  - 管理 profile 配置与状态。
  - 区分 managed / existing-session / remote-cdp。
- `target_registry.rs`
  - 维护 tab/targetId 映射。
  - 支持 focus/open/close/select default target。
- `managed_chromium.rs`
  - 负责拉起 Chromium、管理 user-data-dir、CDP port、launch args。
- `cdp_client.rs`
  - 提供基础 CDP 探活、target 查询、ws 连接。
- `playwright_runtime.rs`
  - 基于 CDP 连接到浏览器。
  - 提供 page 级高级操作。
- `snapshot.rs`
  - 实现 `ai/aria/role` 三类 snapshot。
- `refs.rs`
  - 负责 `ref` 分配、缓存、恢复与失效策略。
- `act.rs`
  - 统一 click/type/press/hover/drag/select/fill/wait/evaluate/close/batch。
- `observe.rs`
  - 记录 console/pageerror/request/response。
- `navigation_policy.rs`
  - URL 校验、SSRF 防护、私网访问策略。
- `docker_runtime.rs`
  - 处理容器场景下的默认 launch args、`--no-sandbox`、`--disable-dev-shm-usage` 等策略。

## 4. 配置重构方案

## 4.1 配置拆分原则

- 保留 `tools.browser.enabled` 作为“是否对模型暴露浏览器工具”的开关。
- 新增顶层 `browser` 配置，作为浏览器子系统运行配置。
- 不再把运行态细节塞进 `tools.browser`。

## 4.2 建议配置结构

```yaml
tools:
  browser:
    enabled: false

browser:
  enabled: false
  deployment: auto          # auto | desktop-embedded | server-embedded | sidecar
  default_profile: managed
  control:
    host: 127.0.0.1
    port: 18791
    auth_token: ${WUNDER_BROWSER_AUTH_TOKEN:-}
    public_base_url: ${WUNDER_BROWSER_BASE_URL:-}
  limits:
    max_sessions: 8
    idle_timeout_secs: 300
    max_tabs_per_session: 16
    max_snapshot_chars: 40000
    max_download_bytes: 104857600
  playwright:
    enabled: true
    driver: node            # node | python
    command: node
    browsers_path: ${PLAYWRIGHT_BROWSERS_PATH:-}
  security:
    allow_private_network: false
    hostname_allowlist: []
    deny_file_scheme: true
  docker:
    enabled: false
    force_headless: true
    use_no_sandbox: true
    disable_dev_shm_usage: true
    browser_data_root: /app/config/data/browser
    downloads_root: /app/config/data/browser/downloads
  profiles:
    managed:
      driver: managed
      headless: true
      cdp_port: 18792
      viewport_width: 1440
      viewport_height: 960
      user_data_dir: ./config/data/browser/profiles/managed
      launch_args: []
    user:
      driver: existing-session
      enabled_in_desktop_only: true
      attach_only: true
    remote:
      driver: remote-cdp
      cdp_url: ""
```

## 4.3 配置语义

- `tools.browser.enabled`
  - 决定模型是否能看到工具。
- `browser.enabled`
  - 决定浏览器子系统是否启动。
- `browser.deployment`
  - `desktop-embedded`：桌面端内嵌控制服务。
  - `server-embedded`：server 进程内嵌 headless 浏览器控制。
  - `sidecar`：server 只调用外部 browser service。
  - `auto`：根据当前运行环境选择。
- `browser.playwright.driver`
  - 第一阶段优先推荐 `node + playwright-core`。
  - 如果后续仍需复用 Python 打包链，可保留 `python` 适配层，但不作为长期核心协议。

## 5. 浏览器控制服务设计

## 5.1 服务职责

- 对外提供稳定控制协议。
- 管理 profile、target、snapshot、action。
- 对模型工具层和前端调试层提供统一入口。
- 在 Docker 模式下作为可独立部署的 sidecar 服务存在。

## 5.2 接口草案

### 基础状态

- `GET /browser/health`
- `GET /browser/status`
- `GET /browser/profiles`
- `POST /browser/start`
- `POST /browser/stop`

### 标签页与 target

- `GET /browser/tabs`
- `POST /browser/tabs/open`
- `POST /browser/tabs/focus`
- `DELETE /browser/tabs/{targetId}`

### 快照与动作

- `GET /browser/snapshot`
- `POST /browser/screenshot`
- `POST /browser/navigate`
- `POST /browser/act`
- `GET /browser/console`
- `GET /browser/network`
- `POST /browser/pdf`
- `POST /browser/upload`
- `POST /browser/dialog`

### Docker / sidecar 运行辅助

- `GET /browser/runtime/info`
- `POST /browser/runtime/reload`

## 5.3 鉴权

- `desktop-embedded`
  - 仅监听 `127.0.0.1`。
  - 使用 runtime token。
- `server-embedded`
  - 仅监听 `127.0.0.1` 或内部 service 网络。
  - 走内部 token。
- `sidecar`
  - 仅暴露给 Docker internal network。
  - 使用 `Authorization: Bearer <WUNDER_BROWSER_AUTH_TOKEN>`。

## 6. 模型工具协议设计

## 6.1 新主工具

- 工具名：`browser`
- 动作集合：
  - `status`
  - `start`
  - `stop`
  - `profiles`
  - `tabs`
  - `open`
  - `focus`
  - `close`
  - `snapshot`
  - `screenshot`
  - `navigate`
  - `console`
  - `network`
  - `pdf`
  - `upload`
  - `dialog`
  - `act`

## 6.2 `snapshot` 设计

### 输入

- `profile`
- `targetId`
- `format`: `ai | aria | role`
- `refs`: `role | aria`
- `interactive: bool`
- `compact: bool`
- `depth: number`
- `selector: string`
- `frame: string`
- `labels: bool`
- `maxChars: number`
- `mode: efficient`

### 输出

```json
{
  "ok": true,
  "profile": "managed",
  "targetId": "tab-1",
  "url": "https://example.com",
  "format": "role",
  "snapshot": "- button \"Search\" [ref=e1]",
  "refs": {
    "e1": { "role": "button", "name": "Search" }
  },
  "stats": {
    "lines": 12,
    "chars": 324,
    "refs": 4,
    "interactive": 3
  },
  "truncated": false
}
```

### 约束

- 模型后续动作优先使用 `ref`，不再以 CSS selector 为主路径。
- `targetId + ref` 组合为稳定动作定位主键。
- `ref` 在导航后默认失效，需重新 snapshot。

## 6.3 `act` 设计

### 动作种类

- `click`
- `type`
- `press`
- `hover`
- `drag`
- `select`
- `fill`
- `wait`
- `evaluate`
- `close`
- `batch`

### 输入示例

```json
{
  "action": "act",
  "profile": "managed",
  "request": {
    "kind": "click",
    "targetId": "tab-1",
    "ref": "e1",
    "doubleClick": false,
    "timeoutMs": 8000
  }
}
```

### 设计要求

- `click/type/hover/select/fill/drag` 主路径使用 `ref`。
- `selector` 只做兼容入口，不再在提示词中推荐。
- `wait` 支持：
  - `timeMs`
  - `selector`
  - `text`
  - `textGone`
  - `url`
  - `loadState`
- `batch` 需要限制最大层级与最大动作数，防止失控。

## 6.4 兼容旧工具

- 保留：
  - `browser_navigate`
  - `browser_click`
  - `browser_type`
  - `browser_screenshot`
  - `browser_read_page`
  - `browser_close`
- 处理方式：
  - 内部全部转发到新 `browser` 工具。
  - 在工具描述中标记为兼容别名。
  - 文档层逐步弱化，不再作为主推荐接口。

## 7. Profile / Tab / Session 设计

## 7.1 Profile 类型

- `managed`
  - Wunder 自己启动 Chromium。
  - 第一阶段主力 profile。
- `existing-session`
  - 连接已存在的本机 Chromium 会话。
  - 桌面端增强能力，后置阶段实现。
- `remote-cdp`
  - 连接外部 CDP endpoint。
  - 为 Browserless / Browserbase / 远程浏览器池预留。

## 7.2 Session 键设计

- 不再沿用“一个用户会话直接对应一个 Python page”的模型。
- 建议拆分：
  - `browser_runtime_id`
  - `browser_session_id`
  - `profile_name`
  - `targetId`
- 推荐映射：

```text
user_id + session_id + agent_id(optional) -> browser_session_id
browser_session_id + profile_name -> active target set
targetId -> page/tab
```

## 7.3 Tab 管理

- 必须支持：
  - 列出 tabs
  - 打开 tab
  - 关闭 tab
  - 聚焦 tab
  - 选择默认 target
- 工具返回必须显式带上 `targetId`，便于后续连续动作。

## 8. Snapshot / Ref 体系设计

## 8.1 为什么必须上 ref

- 当前 CSS selector 路径有三个根本问题：
  - 模型不擅长稳定生成 selector。
  - 现代前端 class/id 经常不稳定。
  - iframe、shadow DOM、动态列表定位容易漂移。

## 8.2 `ref` 设计规则

- `role refs`
  - 形如 `e1/e2/e3`。
  - 来自 role snapshot。
  - 可读性高，适合模型驱动。
- `aria refs`
  - 对齐 Playwright/ARIA 快照。
  - 适合稳定恢复和跨轮定位。
- `ref` 存储策略
  - 以 `browser_session_id + targetId + snapshot_generation` 为命名空间。
  - 每次导航后 generation 自增。
  - 旧 generation 自动失效。

## 8.3 Snapshot 类型

- `ai`
  - 供模型直接阅读的文本快照。
  - 可截断，注重低噪声。
- `aria`
  - 更结构化的无障碍树。
  - 便于程序消费和调试。
- `role`
  - 面向模型动作的主快照。
  - 支持 `interactive/compact/depth/frame/selector`。

## 8.4 第一阶段范围

- 必做：
  - `role` snapshot
  - `ai` snapshot
  - `ref` 缓存与恢复
- 可后置：
  - `aria` 原始树全量导出
  - 带截图标注的 labels 输出

## 10. Docker 运行准备方案

## 10.1 设计结论

- Docker 场景不应该继续依赖 `server.mode=desktop`。
- 推荐设计是：
  - `wunder-server` 正常运行 `api` 模式。
  - 新增 `wunder-browser` sidecar，负责 Chromium/CDP/Playwright。
  - server 通过内部 browser control API 调用 sidecar。

## 10.2 为什么优先 sidecar

- 浏览器依赖重，和主 server 解耦更稳。
- Chromium 升级、Playwright 升级不影响主服务镜像。
- 容器内 `--no-sandbox`、`/dev/shm`、字体、GPU、下载目录这些运行参数更容易独立治理。
- 未来可横向扩展为多个 browser worker，而不是把所有浏览器进程塞进主 server。

## 10.3 sidecar 结构

```text
wunder-server
  -> 调度工具、鉴权、会话持久化
  -> 调用 browser control service

wunder-browser
  -> Chromium runtime
  -> CDP + Playwright
  -> profile/tab/target 管理
  -> downloads / screenshots / browser user-data 持久化
```

## 10.4 Docker 运行要求

- 镜像层面
  - 安装 Chromium 或 Playwright browser binaries。
  - 安装必要字体。
  - 预留 `PLAYWRIGHT_BROWSERS_PATH`。
- 运行参数
  - 默认 headless。
  - 容器内默认附加：
    - `--no-sandbox`
    - `--disable-dev-shm-usage`
    - `--disable-gpu`
    - `--no-zygote`
  - 若宿主机支持更完整沙箱，可再按配置覆盖。
- 存储卷
  - `config/data/browser/profiles`
  - `config/data/browser/downloads`
  - `config/data/browser/screenshots`
  - `config/data/browser/cache`
- 健康检查
  - `/browser/health`
  - 浏览器二进制存在性检查
  - CDP ready 检查

## 10.5 Docker 配置建议

### 新增环境变量

- `WUNDER_BROWSER_ENABLED`
- `WUNDER_BROWSER_DEPLOYMENT=sidecar`
- `WUNDER_BROWSER_CONTROL_URL=http://wunder-browser:18791`
- `WUNDER_BROWSER_AUTH_TOKEN`
- `PLAYWRIGHT_BROWSERS_PATH=/ms-playwright`
- `WUNDER_BROWSER_PROFILE_ROOT=/app/config/data/browser/profiles`
- `WUNDER_BROWSER_DOWNLOAD_ROOT=/app/config/data/browser/downloads`

### 新增 compose 服务

```yaml
wunder-browser:
  image: wunder-browser
  restart: always
  shm_size: "1gb"
  environment:
    - WUNDER_BROWSER_ENABLED=1
    - WUNDER_BROWSER_DEPLOYMENT=sidecar
    - WUNDER_BROWSER_AUTH_TOKEN=${WUNDER_BROWSER_AUTH_TOKEN}
    - PLAYWRIGHT_BROWSERS_PATH=/ms-playwright
  volumes:
    - ./config/data/browser:/app/config/data/browser
  healthcheck:
    test: ["CMD-SHELL", "curl -fsS http://127.0.0.1:18791/browser/health || exit 1"]
```

## 10.6 Docker 第一阶段不做的事

- 不在 Docker 中支持 `existing-session`。
- 不做可视化 GUI 浏览器，只做 headless。
- 不在第一阶段支持多租户浏览器池调度。

## 11. 安全边界

## 11.1 SSRF 与私网访问

- 浏览器导航不能默认无约束访问容器内网段、metadata endpoint、Docker internal service。
- 必须新增导航策略：
  - 默认禁止私网地址。
  - 显式 allowlist 才允许访问内部域名。
  - 禁止 `file://`。
  - 对最终跳转 URL 再校验一次。

## 11.2 文件系统边界

- 下载、上传、截图、PDF 全部限制在 browser runtime root 下。
- 不允许通过浏览器工具直接访问工作区任意路径。
- 上传动作必须走 allowlist 根目录校验。

## 11.3 资源保护

- 限制：
  - 最大 session 数
  - 最大 tab 数
  - 最大截图大小
  - 最大下载大小
  - 最大 snapshot chars
  - 最大 evaluate timeout
- idle session 自动回收。

## 12. 分阶段落地节点

## 12.1 R0：方案冻结与兼容边界确认

### 目标

- 冻结协议方向，不再继续扩写旧 Python bridge。

### 交付

- 配置迁移草案。
- Browser tool schema 草案。

### 验收

- 后续实现按本方案拆模块，不再新增大段逻辑到旧文件。

## 12.2 R1：Browser 子系统骨架

### 目标

- 引入 `src/services/browser/` 与 `src/api/browser_control/`。

### 交付

- 浏览器运行时配置与 registry。
- 控制服务启动、health/status 接口。
- 旧浏览器工具改为调用新 service client。

### 验收

- `browser status/start/stop` 可跑通。
- 旧工具路径仍可编译。

## 12.3 R2：Managed Chromium + Tab/Profile

### 目标

- 支持 managed profile、CDP 探活、tab open/focus/close。

### 交付

- managed Chromium launch 模块。
- profile registry。
- tabs API。

### 验收

- 可打开页面并返回 `targetId`。
- 同一会话可连续管理多个 tab。

## 12.4 R3：Snapshot / Ref / Act 主链路

### 目标

- 建立模型可稳定使用的 `snapshot -> ref -> act` 协议。

### 交付

- role/ai snapshot。
- ref 缓存与失效机制。
- act: click/type/press/hover/select/fill/wait/close。

### 验收

- 模型在不写 CSS selector 的情况下完成基本网页交互。
- 导航后旧 ref 自动失效。

## 12.5 R4：高级能力补齐

### 目标

- 补齐观察与文件类动作。

### 交付

- screenshot/pdf/upload/dialog/console/network/evaluate/batch。

### 验收

- 文件上传、下载、截图、PDF 导出链路可用。
- console/network 能辅助排障。

## 12.6 R5：Docker sidecar 落地

### 目标

- 在 `server/docker` 模式下正式支持 headless 浏览器。

### 交付

- `wunder-browser` Dockerfile / compose service。
- server -> sidecar internal client。
- browser health/readiness/cleanup。

### 验收

- `docker compose up` 后，server 中模型可调用浏览器工具。
- 无需把 `server.mode` 改成 `desktop`。

## 12.7 R6：增强驱动

### 目标

- 对齐更完整的 openclaw 能力，但不阻塞主链路。

### 交付

- existing-session attach（desktop only）。
- remote CDP profile。
- 可选 browser pool。

### 验收

- 桌面端可连接现有浏览器会话。
- 远程 CDP profile 可用。

## 15. 非目标

## 16. 结论

- Wunder 浏览器工具的重构方向必须从“脚本工具”升级为“浏览器子系统”。
- 第一优先级不是增加更多动作，而是建立稳定的控制服务、profile/tab 体系、snapshot/ref/act 协议。
- Docker 支持必须在架构层直接设计出来，不能继续依赖 `desktop` 模式兜底。
- 推荐落地顺序是：
  - `R1-R3` 先把 browser subsystem 和模型主链路做稳。
  - `R5` 再把 Docker sidecar 正式落地。
  - `R6` 再补 existing-session 和 remote CDP。
