# 新微信渠道落地方案（openclaw-weixin / iLink）

> 目标：将 `@tencent-weixin/openclaw-weixin` 对接到 wunder，作为“新微信渠道”，并与现有 `wechat` / `wechat_mp` 明确区分，支持用户快速选择与定位。

## 0. 快速入口（30秒选型）

- 用企业微信应用（公司内部应用消息）: 选 `wechat`（旧）
- 用公众号（关注后收发）: 选 `wechat_mp`（旧）
- 用 iLink 协议个人微信能力（`openclaw-weixin`）: 选 `weixin`（新）

建议在管理端与用户端统一显示名：

- `wechat` → `企业微信（旧）`
- `wechat_mp` → `公众号（旧）`
- `weixin` → `微信 iLink（新）`

## 1. 三渠道差异矩阵（必须先看）

| 维度 | `wechat`（旧） | `wechat_mp`（旧） | `weixin`（新） |
| --- | --- | --- | --- |
| 主要场景 | 企业微信应用 | 微信公众号 | iLink 微信渠道（openclaw-weixin） |
| 入站方式 | 平台回调 Webhook（GET/POST） | 平台回调 Webhook（GET/POST） | 长轮询拉取 `ilink/bot/getupdates` |
| 核心路由 | `/wunder/channel/wechat/webhook` | `/wunder/channel/wechat_mp/webhook` | 无专用 webhook，运行时 worker 主动拉取 |
| 鉴权要点 | token + signature + AES（可选） | token + signature + AES（可选） | Header: `AuthorizationType=ilink_bot_token` + `Authorization: Bearer <bot_token>` + `X-WECHAT-UIN` |
| 会话关键字段 | `agent_id` / XML 字段 | `original_id` / XML 字段 | `context_token`（回复必须回传）+ `get_updates_buf`（游标） |
| 登录方式 | 配置 corp_id/agent_id/secret | 配置 app_id/app_secret | 扫码登录获得 `bot_token` + `ilink_bot_id` |
| 媒体链路 | 微信官方接口 | 公众号官方接口 | CDN + AES-128-ECB（上传/下载都涉及） |
| 对 wunder 的接入形态 | webhook + adapter | webhook + adapter | 长连接 supervisor/worker + adapter |

结论：`weixin` 不是旧两条微信渠道的“参数变种”，而是新的协议族与运行模型，必须独立 provider 接入。

## 2. 命名与可发现性规范（避免用户找错）

### 2.1 Provider 命名

- 固定新增 provider key：`weixin`
- 严禁复用 `wechat` 或 `wechat_mp` 以“偷换”新协议

### 2.2 列表展示规范

- 在渠道列表把三个微信项放在相邻位置，顺序建议：`weixin`、`wechat`、`wechat_mp`
- 每项增加显著标签：`新` / `旧`
- `docs_hint` 不再复用 webhook 地址语义，`weixin` 应指向本方案文档或账号配置入口

### 2.3 搜索关键词

- `weixin` 关键词：`iLink`、`openclaw-weixin`、`扫码登录`、`长轮询`
- `wechat` 关键词：`企业微信`、`corp_id`、`agent_id`
- `wechat_mp` 关键词：`公众号`、`original_id`

## 3. 对 wunder 的落地范围与分期

### P0（先可用，1~2天）

- 文本消息闭环
- 多账号长轮询收消息
- 出站文本回复（强依赖 `context_token`）
- 账号状态与错误可观测

### P1（增强，2~4天）

- 图片/文件/视频出站
- 入站媒体下载与转码
- typing 状态同步（`getconfig` + `sendtyping`）

### P2（完善，2~3天）

- 扫码登录流程内嵌（`get_bot_qrcode` / `get_qrcode_status`）
- 管理端一键重登、token 失效恢复

## 4. 代码落点（按仓库结构拆分）

- `src/channels/types.rs`
- `src/channels/weixin.rs`（建议按职责再拆：api/client、inbound、outbound、media）
- `src/channels/registry.rs`（注册 `WeixinAdapter`）
- `src/channels/catalog.rs`（新增 `weixin` 条目，显示名含“新”）
- `src/channels/mod.rs`（导出模块）
- `src/channels/service.rs`（新增 weixin supervisor/worker）
- `src/api/user_channels.rs`（账号配置入参、校验、预览）
- `src/api/admin.rs`（账号运行态与长连接健康展示）

说明：`weixin` 采用长轮询，不应复制 `wechat/wechat_mp` 的 webhook 专用路由实现。

## 5. 账号配置设计（建议）

在 `channel_accounts.config` 中新增：

```json
{
  "weixin": {
    "api_base": "https://ilinkai.weixin.qq.com",
    "bot_token": "<required>",
    "ilink_bot_id": "<required>",
    "ilink_user_id": "<optional>",
    "long_connection_enabled": true,
    "allow_from": ["<optional weixin user id>"],
    "poll_timeout_ms": 35000,
    "api_timeout_ms": 15000,
    "max_consecutive_failures": 3,
    "backoff_ms": 30000,
    "typing_enabled": false,
    "media_enabled": false,
    "route_tag": "<optional>"
  }
}
```

校验最小集合（P0）：

- `api_base` 非空
- `bot_token` 非空
- `ilink_bot_id` 非空

## 6. 核心链路设计

### 6.1 入站（长轮询）

1. supervisor 扫描 `active` 且 `weixin.long_connection_enabled=true` 的账号
2. 每账号起一个 worker，维护 `get_updates_buf` 游标
3. 调用 `ilink/bot/getupdates`
4. 将消息映射为 `ChannelMessage`，关键字段写入 `meta.weixin`：
   - `context_token`
   - `session_id`
   - `message_id`
   - `from_user_id` / `to_user_id`
5. 通过 `ChannelHub.enqueue_inbound("weixin", ...)` 入总线
6. 连续失败按阈值退避，`session expired`（如 `-14`）进入长退避并告警

### 6.2 出站（文本）

1. 从绑定会话取目标 `peer_id`
2. 从最近入站消息或会话上下文取 `context_token`
3. 缺少 `context_token` 直接失败并写明错误（不发送）
4. 调用 `ilink/bot/sendmessage`
5. outbox 标记 sent/retry/failed

### 6.3 媒体（P1）

上传链路：`getuploadurl` → 本地 AES-128-ECB 加密 → 上传 CDN → `sendmessage` 携带 media 引用。  
下载链路：下载 CDN 对象 → AES 解密 → 统一转存 wunder 附件目录。

## 7. 与旧两条的替换策略（平滑切换）

### 阶段A：并行期（建议 3~7 天）

- 三条渠道同时保留
- 新接入账号只允许走 `weixin`
- 旧渠道仅处理历史存量

### 阶段B：灰度期

- 按账号/用户组迁移到 `weixin`
- 对比指标：
  - 入站成功率
  - 出站成功率
  - 平均首包时延
  - 失败原因 TopN

### 阶段C：替换完成

- 管理端将 `wechat` / `wechat_mp` 标记为“旧（维护态）”
- 默认创建入口只展示 `weixin`
- 旧渠道保留只读与应急回滚开关

## 8. 验收清单（上线前）

- 文本收发 24h 稳定（无 worker 泄漏）
- `context_token` 缺失场景可观测且不误发
- token 失效可自动退避并恢复
- 多账号并发下消息不串会话
- 管理端可看到每账号最近轮询时间、最近错误、连续失败次数

## 9. 常见故障与定位

- 报错“context_token is required”
  - 原因：回复未携带入站 `context_token`
  - 处理：从会话最近入站消息恢复 token，再重试

- 长时间无消息
  - 检查 `long_connection_enabled`、`bot_token` 是否有效、账号是否 `active`

- 频繁 session expired
  - 说明登录态失效，需重新扫码获取新 `bot_token`

- 媒体发送失败
  - 优先检查 `media_enabled`、AES 加密参数、CDN 上传响应

## 10. 实施结论

`openclaw-weixin` 应在 wunder 中作为全新 `weixin` 渠道落地，不应并入 `wechat`/`wechat_mp`。  
先完成 P0 文本闭环和可观测性，再逐步补齐媒体与扫码登录，最后执行旧渠道维护态切换，能在风险可控的前提下完成“新微信渠道替换旧两条”的目标。
