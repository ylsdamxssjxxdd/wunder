# 子智能体对齐实施清单

## 1. 目标

将 wunder 的子智能体能力从“可用的批量派发与自动回流”继续提升到更接近 OpenClaw / Codex 的成熟控制面，重点补齐以下四层：

1. 显式编排原语
2. 持久化运行注册与恢复
3. 分层能力治理
4. 长生命周期子智能体会话与运维控制

当前判断：

- wunder 已具备 `spawn/batch_spawn/status/wait/interrupt/close/resume`、父线程自动唤醒、聊天气泡内子智能体面板。
- 主要差距已从“有没有能力”转为“编排契约是否显式、运行时是否可靠、嵌套控制是否严格、长期会话是否成熟”。

## 2. 里程碑

### P0 显式编排原语对齐

#### N0.1 `sessions_yield` 一级原语

- 状态：已落地
- 目标：让模型可以显式表达“本轮先结束，等子智能体回流后再继续”，而不是只能依赖隐式 auto-wake。
- 已实现内容：
  - 新增内置工具 `会话让出`，英文别名 `sessions_yield` / `yield`
  - 工具结果会写入结构化 `turn_control.kind=yield`
  - orchestrator 在工具结果阶段识别 `yield`，将本轮收束为 `stop_reason=yield`
  - 本轮结束前会追加一条简短助手消息，避免用户看到空白收尾
- 当前文件：
  - `src/services/tools/sessions_yield_tool.rs`
  - `src/services/tools/dispatch.rs`
  - `src/services/tools/catalog.rs`
  - `src/orchestrator/execute.rs`

#### N0.2 Codex 风格推荐调用路径

- 状态：已落地（第一阶段）
- 目标：让模型更稳定采用 `spawn/batch_spawn -> sessions_yield -> auto wake`，减少“派发后继续空转”。
- 已实现内容：
  - 在 `subagent_control` 的工具描述中明确后台任务优先走“派发后让出”的路径
  - 在中英 `tools_protocol` 提示词中加入 Codex 风格编排约束：
    - 后台子智能体任务不需要主线程阻塞等待
    - 需要结束当前轮时优先调用 `sessions_yield`
    - `status/wait` 退化为按需路径，不鼓励忙轮询
  - 为提示词渲染增加 `HAS_SUBAGENT_CONTROL_TOOL` / `HAS_SESSIONS_YIELD_TOOL` flag，避免无关模型拿到多余编排说明
- 验收：
  - 同类任务下模型触发 `sessions_yield` 的稳定率明显提高
  - `wait/status` 从默认串行路径退为按需路径

#### N0.3 前端 stop reason 可视化

- 状态：待做
- 目标：前端能识别 `stop_reason=yield`，在本轮结束态上显示“后台继续中”而不是误判为异常或未完成。
- 要做：
  - 聊天态和调试态识别 `yield`
  - 如有必要，在气泡或工作流中补一个“已让出，等待子智能体回流”的轻提示
- 验收：
  - 用户可区分 `question_panel` / `yield` / `normal final`

### P1 子智能体运行注册与恢复

#### N1.1 专用运行注册表

- 状态：进行中（基础元数据已落地）
- 目标：把当前 `session_runs + chat_sessions + monitor + stream_events` 的组合能力，提升为更明确的子智能体运行注册表。
- 已实现内容：
  - `session_runs` 新增结构化 `metadata`
  - 单子派发与批量派发会写入 `controller_session_id / parent_turn_ref / depth / role / control_scope`
  - 批量派发额外写入 `dispatch_index / dispatch_size / dispatch_label / strategy / completion_mode / remaining_action`
  - `list/status/wait` 与聊天侧子智能体运行项已透出上述字段，前端不再需要从文本推断调度关系
- 建议字段：
  - `run_id`
  - `session_id`
  - `parent_session_id`
  - `controller_session_id`
  - `dispatch_id`
  - `role`
  - `depth`
  - `control_scope`
  - `created_at / started_at / ended_at`
  - `accumulated_runtime_ms`
  - `ended_reason`
  - `announce_retry_count`
  - `wake_on_descendant_settle`
  - `frozen_result_text`
  - `cleanup_state`
- 验收：
  - 不依赖聊天历史文本，也能完整还原一条子智能体生命周期
  - `status/list/info/log` 有稳定单一数据源

#### N1.2 服务重启后的 orphan recovery

- 状态：待做
- 目标：服务重启后恢复仍应继续的子智能体，而不是丢失在后台。
- 要做：
  - 启动时扫描“未终态但上次运行中断”的子智能体
  - 重新投递恢复消息或标记终止原因
  - 避免重复恢复和重复 announce
- 验收：
  - 重启后长任务子智能体可恢复或被明确标记为失败

#### N1.3 announce 重试与回送队列

- 状态：待做
- 目标：父线程唤醒与子结果回送从“尽力而为”升级为“可重试、可追踪”。
- 要做：
  - 为 parent wake / announce 增加 retry/backoff
  - 增加 idempotency key
  - 对失败中的回送任务保留重试记录
- 验收：
  - 短暂故障不会导致子智能体结果静默丢失

### P2 嵌套能力治理

#### N2.1 角色化能力模型

- 状态：待做
- 目标：显式区分 `main / orchestrator / leaf`。
- 要做：
  - spawn 时写入 `role/depth/control_scope`
  - 存储 `can_spawn / can_control_children`
  - 统一能力解析函数，避免只靠 session 层级字符串推断
- 验收：
  - leaf 子智能体无法控制兄弟或父节点

#### N2.2 深度与扇出上限

- 状态：待做
- 目标：防止 runaway fan-out。
- 要做：
  - `max_spawn_depth`
  - `max_children_per_agent`
  - `max_active_children_per_parent`
  - 深度超限时的明确工具报错
- 验收：
  - 多层调度可控，错误清晰

#### N2.3 按深度裁剪工具面

- 状态：待做
- 目标：不同层级只暴露合理工具集。
- 要做：
  - depth 1 orchestrator 保留 `subagent_control/status/wait/...`
  - depth 2 leaf 只保留执行型工具，不再给会话控制能力
- 验收：
  - 子智能体提示词和工具清单均体现该约束

### P3 长生命周期子智能体会话与运维面

#### N3.1 thread-bound 子智能体 session mode

- 状态：待做
- 目标：支持“不是一次性 run，而是长期子会话”。
- 要做：
  - 为 `sessions_spawn` 增加 `mode=run|session`
  - 增加 thread/session 绑定元数据
  - 后续消息可继续命中原子智能体会话
- 验收：
  - 用户可长期与某个子智能体线程持续交互

#### N3.2 richer control surface

- 状态：待做
- 目标：补齐 `info/log/steer/send` 等控制面，而不只停留在 `status/wait/interrupt/close/resume`。
- 要做：
  - `subagent_control.action=info|log|steer`
  - steer 的速率限制、幂等重启、权限校验
  - log 输出按摘要/工具调用/最终结果分层
- 验收：
  - 调度者能看、能控、能纠偏，不必通过猜测状态操作

#### N3.3 聊天页子智能体工作区继续增强

- 状态：待做
- 目标：把现有消息底部子智能体区域继续做成“轻工作流面板”。
- 要做：
  - 条目级运行时间、终止原因、最近事件
  - 详情框内分 tab 展示：概况 / 事件 / 工具 / 日志
  - 支持对单条子智能体执行 `terminate / close / steer`
- 验收：
  - 不离开聊天页即可完成主要调度操作

## 3. 当前推进顺序

按优先级建议：

1. P0.1 `sessions_yield`
2. P0.2 推荐路径与提示词收敛
3. P1.1 专用运行注册表
4. P1.2 orphan recovery
5. P2.1 角色化能力模型
6. P3.1 长生命周期 session mode

- 已新增 `sessions_yield` 工具并接入 orchestrator `stop_reason=yield`
- 已把该能力写入工具目录与基础文档
- 已补齐模型侧推荐路径：`spawn/batch_spawn -> sessions_yield -> auto wake`
- 已将子智能体调度元数据落入 `session_runs.metadata`，并透传到 `subagents/status/wait` 结果
- 下一落实节点建议进入 `N0.3 + N1.2`
