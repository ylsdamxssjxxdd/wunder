# 蜂群页面重构方案

## 1. 结论

可以。

当前用户侧蜂群页面并不需要通用图引擎，`@antv/g6` 在这里承担的主要是：

- 固定蜂窝布局后的节点/边绘制
- 视口缩放、平移、适配视图
- 节点拖拽
- 小地图
- 边高亮与虚线流动
- 命中、悬浮、双击事件

这些能力完全可以用“原生 `SVG + DOM` 混合渲染”实现，最多只保留一个很轻的视口手势库作为可选依赖。  
真正的性能飞跃，不是单纯把 `G6` 替换掉，而是同时完成这两件事：

- 去掉重图引擎与 HTML-in-graph 的额外运行时
- 把当前“任意任务变化 => 整张图全量 render”改成“脏节点/脏边增量更新”

推荐方案：

- 不再使用图引擎
- 采用原生 `SVG + DOM` 混合渲染
- 视口手势优先自研；如果评审更看重跨浏览器手势细节，可只引入一个轻量 panzoom 依赖

不推荐直接切纯 Canvas v1。原因很简单：当前节点卡片是富 HTML 结构，纯 Canvas 会把文本排版、图片裁剪、命中检测、无障碍和调试成本重新做一遍，第一阶段不划算。

## 2. 现状代码分析

### 2.1 当前实现范围

现有蜂群画布核心集中在一个文件：

- `frontend/src/components/beeroom/BeeroomMissionCanvas.vue`

该文件当前约 `4485` 行，已经同时承担：

- 图模型投影
- G6 初始化与销毁
- 画布状态缓存
- tooltip
- minimap
- 聊天面板
- demo 运行态
- 聊天实时流
- team realtime 回补
- workflow 预览轮询
- dispatch stop/resume/approval

相关证据点：

- G6 只在这一个文件中使用：`frontend/src/components/beeroom/BeeroomMissionCanvas.vue:304,526,556`
- 图投影与蜂窝布局在同文件中完成：`frontend/src/components/beeroom/BeeroomMissionCanvas.vue:822-1061`
- render 签名把 mission/tasks/workflow 全拼到一起：`frontend/src/components/beeroom/BeeroomMissionCanvas.vue:2243-2267`
- G6 初始化与行为注册集中在同文件：`frontend/src/components/beeroom/BeeroomMissionCanvas.vue:2780-3010`
- 每次 render 都是 `setData + render + fit/restore`：`frontend/src/components/beeroom/BeeroomMissionCanvas.vue:3014-3088`

外围容器也有可继续收敛的空间：

- `frontend/src/components/beeroom/BeeroomWorkbench.vue` 当前约 `863` 行
- 它已经被硬编码为纯画布模式：`frontend/src/components/beeroom/BeeroomWorkbench.vue:305-306`
- 但仍保留了大量 text view 的旧结构和样式，增加维护成本

### 2.2 当前图形复杂度其实不高

当前页面虽然使用了 G6，但图本身并不复杂：

- 节点布局不是通用图布局，而是固定蜂窝槽位 `buildHoneycombSlots + resolveHoneycombPosition`
- 边关系不是复杂网络，而是“母蜂 -> 工蜂”的辐射型 dispatch 边
- 节点不是图元绘制，而是 HTML 卡片字符串 `buildCanvasNodeCardHtml`
- 边动画不是复杂动画系统，而是定时修改 `lineDashOffset`

这意味着当前场景更像“可缩放的信息面板”，而不是“复杂图编辑器”。

### 2.3 当前性能瓶颈

当前性能问题主要来自 6 类叠加：

1. 引擎与场景错配  
   G6 更适合通用图、复杂布局和大规模图元；当前页面是固定布局 + 富 HTML 卡片，G6 的收益不高，但运行时和调试复杂度很高。

2. HTML 节点放进 G6  
   当前节点用的是 `html` node，内部还是 DOM/CSS。也就是说，既没拿到纯 Canvas/SVG 的轻量优势，又额外引入了图引擎层。

3. 全量 render 策略  
   `renderSignature` 把任务状态、workflow 预览、成员在线状态全部串起来，任何一个 worker 的任务变化，都会触发整张图 `setData + render`。

4. workflow 轮询频繁推高重绘  
   `frontend/src/components/beeroom/useBeeroomMissionWorkflowPreview.ts` 默认每 `1200ms` 为每个“每工蜂最新任务”拉取一次 session events；结果不是只更新一个节点，而是把整个图的 render 签名打脏。

5. 边动画使用 JS 定时器  
   当前 dispatch 活跃边通过 `setInterval -> updateEdgeData -> draw` 驱动，`frontend/src/components/beeroom/BeeroomMissionCanvas.vue:2589-2598` 会持续唤起图层绘制。

6. 组件耦合过重  
   画布、聊天、实时流、审批、demo 都在一个组件里，任何一块 reactive 状态变化，都更容易波及整棵组件树。

### 2.4 结论性判断

如果只是“把 G6 换成另一个图库”，但保留：

- 统一大组件
- renderSignature 全量拼接
- workflow 轮询直接驱动整图
- 聊天与画布耦合

那么性能只会变好一部分，不会出现真正的飞跃。

## 3. 重构目标

### 3.1 目标

- 移除 `@antv/g6`
- 保持现有用户可见能力不倒退
- 把图形渲染改成增量更新
- 把蜂群画布拆成长期可维护的模块
- 避免继续产生 2000+ 行前端单文件

### 3.2 目标架构

推荐采用“`SVG + DOM` 混合渲染”：

- `SVG` 负责边、边标签、小地图框、网格辅助层
- `DOM` 负责节点卡片、tooltip、工具栏、聊天面板
- 整个 world 只做一层统一 transform：`translate + scale`
- 节点位置采用绝对定位
- 选中、悬浮、状态色全部走普通 Vue/CSS

结构草图：

```text
BeeroomSwarmCanvas
├─ SwarmViewportHost
│  ├─ SwarmGridLayer        (CSS/SVG)
│  ├─ SwarmEdgeLayer        (SVG)
│  ├─ SwarmNodeLayer        (DOM absolute cards)
│  └─ SwarmTooltipLayer     (DOM)
├─ SwarmToolbar
├─ SwarmMinimap             (SVG)
└─ BeeroomCanvasChatPanel   (DOM)
```

### 3.3 选型结论

推荐顺序如下：

1. 首选：原生实现 viewport/controller，不引入图引擎
2. 备选：仅引入轻量 panzoom 库处理缩放和平移
3. 不推荐：再换一个重图形库
4. 不推荐 v1：纯 Canvas 全自绘节点

原因：

- 当前布局是固定蜂窝布局，自研成本很低
- 当前节点是富信息卡片，DOM 天然合适
- 当前边关系简单，SVG path 足够
- 当前需要的是“更少的渲染层”和“更清晰的状态边界”

## 4. 详细设计

### 4.1 模块拆分

建议新增目录：

```text
frontend/src/components/beeroom/canvas/
  BeeroomSwarmCanvas.vue
  BeeroomSwarmViewport.vue
  BeeroomSwarmNodeLayer.vue
  BeeroomSwarmNodeCard.vue
  BeeroomSwarmEdgeLayer.vue
  BeeroomSwarmTooltip.vue
  BeeroomSwarmToolbar.vue
  BeeroomSwarmMinimap.vue
  swarmCanvasModel.ts
  useBeeroomSwarmProjection.ts
  useBeeroomSwarmViewport.ts
  useBeeroomSwarmNodeDrag.ts
  useBeeroomSwarmPerf.ts

frontend/src/components/beeroom/chat/
  BeeroomCanvasChatPanel.vue
  useBeeroomCanvasChat.ts
  useBeeroomDispatchRuntime.ts
```

现有文件处理建议：

- `BeeroomMissionCanvas.vue` 第一阶段保留为壳组件，后续缩减为 300-500 行
- `beeroomMissionCanvasStateCache.ts` 保留，但补版本字段
- `useBeeroomMissionWorkflowPreview.ts` 保留，但改为按 task 粒度输出 dirty 信息
- `BeeroomWorkbench.vue` 在切换完成后删除 text 模式遗留代码

### 4.2 图模型抽象

需要先把“画布数据”和“渲染引擎”脱钩。

统一输出模型建议：

```ts
type SwarmViewportState = {
  scale: number;
  offsetX: number;
  offsetY: number;
};

type SwarmNode = {
  id: string;
  agentId: string;
  x: number;
  y: number;
  width: number;
  height: number;
  role: 'mother' | 'worker';
  status: string;
  selected: boolean;
  workflowTaskId: string;
};

type SwarmEdge = {
  id: string;
  sourceId: string;
  targetId: string;
  label: string;
  active: boolean;
  selected: boolean;
  kind: 'dispatch';
};

type SwarmProjection = {
  nodes: SwarmNode[];
  edges: SwarmEdge[];
  nodeMetaMap: Map<string, CanvasNodeMeta>;
  motherNodeId: string;
  bounds: { minX: number; minY: number; maxX: number; maxY: number };
};
```

关键原则：

- layout 数据和 node content 数据分离
- 节点位置变化不应该依赖 workflow 文本变化
- workflow 变化只更新对应节点卡片
- edge active/selected 变化只更新对应边

### 4.3 视口与坐标系

统一世界坐标：

- 布局阶段给出 world coordinates
- `SwarmViewportState` 只负责 `scale + offset`
- 视口变换通过单层 CSS transform 或单层 SVG transform 完成

建议保留现有能力：

- 放大
- 缩小
- 重置 100%
- 适配视图
- 自动整理
- 全屏

实现方式：

- `fitView` 根据 `projection.bounds` 和容器尺寸计算最佳 `scale`
- `autoArrange` 只清空节点 override，再重新投影
- `fullscreen` 继续使用 DOM Fullscreen API
- 缓存状态从“graph position + zoom”改为“viewport transform”

### 4.4 节点层

节点层使用 DOM 绝对定位：

- 每个节点都是独立 Vue 组件 `BeeroomSwarmNodeCard.vue`
- 节点卡片不再拼 HTML 字符串
- 头像、角色、状态、workflow 摘要直接走 template
- hover、selected、running、danger 等状态全部通过 class 控制

这样可以直接获得：

- Vue 的局部更新能力
- 更好调试性
- 更稳定的图片/文本布局
- 更自然的 tooltip / pointer 事件

### 4.5 边层

边层使用单个 `SVG`：

- 边 path 根据 source/target 的世界坐标计算
- 标签直接走 `text` 或 `foreignObject`，优先 `text`
- 活跃边动画改成 CSS `stroke-dashoffset` 动画
- 选中态、活跃态、空闲态只改 class，不再走 JS 定时 draw

收益：

- 去掉 `setInterval -> graph.draw()`
- 边动画交给浏览器合成层
- 单边更新成本远低于整图 render

### 4.6 小地图

小地图不再依赖插件：

- 复用 `projection.nodes` / `projection.edges`
- 只渲染简化矩形和细线
- 当前 viewport 作为一个矩形框覆盖在 minimap 上
- 点击 minimap 可跳转中心

这样能避免当前 minimap 插件对节点 shape 的 clone 开销。

### 4.7 节点拖拽

节点拖拽改成原生 pointer 事件：

- `pointerdown` 进入 drag
- `pointermove` 更新目标节点 override
- `pointerup` 持久化到 cache
- 节点拖拽与画布平移要做互斥

建议规则：

- 点击节点主体默认选中
- 长按/拖动阈值超过 4px 再进入 drag
- 鼠标滚轮始终作用于 viewport

### 4.8 状态缓存

现有 `beeroomMissionCanvasStateCache.ts` 可以复用，但建议升级 schema：

- 增加 `version`
- 旧字段兼容读取一次后迁移
- viewport 统一为 `{ scale, offsetX, offsetY }`

缓存内容继续保留：

- 节点 override
- 当前选中节点
- 聊天面板折叠状态
- 视口状态

### 4.9 聊天面板与实时链路

这一块必须与画布彻底拆开。

当前问题不是聊天逻辑错误，而是聊天和画布共处一个大组件，会放大 reactive 扰动。

建议：

- `BeeroomCanvasChatPanel.vue` 独立承载聊天 UI
- `useBeeroomCanvasChat.ts` 承载消息列表、发送、清空、滚动到底部
- `useBeeroomDispatchRuntime.ts` 承载 stop/resume/approval/demo 运行态
- 画布组件只接收和发出最小事件，不直接拥有整套聊天流程

### 4.10 workflow 预览刷新

这是第二大性能点，必须一起收口。

现状问题：

- workflow hook 每 `1200ms` 拉取一次 active task session events
- hook 输出的是聚合签名
- 签名变化后整张图重绘

建议改法：

- hook 输出 `workflowPreviewByTask`、`workflowItemsByTask` 之外，再输出 `dirtyTaskIds`
- 节点卡片按 `taskId` 精确订阅
- 只让对应节点更新，不触发布局层更新
- layout 与 workflow content 拆成两个独立 signature

进一步优化建议：

- 超过一定节点数时只展示 1 行 workflow 摘要
- 缩放比例低于阈值时隐藏 workflow 细节
- 非活跃任务停止轮询

### 4.11 分级渲染策略

为了保证后续扩容，建议加 LOD（Level of Detail）：

- 节点数 `<= 32`：展示完整卡片 + workflow 多行
- 节点数 `33-72`：默认只展示 1-2 条 workflow
- 节点数 `> 72`：默认收起 workflow 细节，只在 hover/selected 时展开
- 缩放比例 `< 0.78`：隐藏 workflow 详情，仅保留状态与名称

这样可以显著降低大群场景下的 DOM 压力。

## 6. 性能目标

建议把以下指标作为上线门槛：

- 24 节点：首次可交互时间 `<= 200ms`（数据到达后）
- 72 节点：首次可交互时间 `<= 450ms`
- 72 节点：单节点状态更新 `p95 <= 16ms`
- 72 节点：缩放/平移连续操作保持 `55fps+`
- 120 节点：页面可用，不出现明显输入阻塞
- 10 分钟 realtime 场景：内存曲线稳定，无持续性上涨

注意：

- 这里的真正关键指标不是“首次打开快一点”，而是“高频 team realtime + workflow 更新时仍然稳定”

## 7. 测试计划

建议覆盖以下测试：

- 单元测试：蜂窝布局、fitView 计算、minimap viewport 映射
- 组件测试：节点选中、tooltip、拖拽、缩放、自动整理
- 回归测试：聊天发送、stop/resume、approval、demo 运行态
- 压测测试：24 / 72 / 120 节点 + 高频任务更新
- 手工测试：桌面端窗口缩放、全屏切换、切组、切 mission、长时间挂机

## 8. 风险与回滚

主要风险：

- 拖拽与平移手势冲突
- 缩放后 DOM 文本抗锯齿表现变化
- minimap 同步精度问题
- workflow 轮询未同步收敛，导致“引擎变轻但数据层仍然重”

回滚策略：

- 在 N2-N4 保留 `g6/native` 双引擎开关
- native renderer 默认灰度一段时间
- 任何阻塞问题都可以快速切回 G6 renderer

## 9. 最终建议

建议按下面的顺序推进，而不是一次性大爆炸替换：

1. 先拆壳，切出画布边界
2. 再抽共享 projection/model
3. 再上 native renderer 双轨并行
4. 最后做增量更新和 G6 下线

一句话总结：

当前蜂群页面完全可以不用 antv，而且值得换；但要想拿到“性能飞跃”，重点不是“换个库”，而是“把画布从重图引擎 + 全量 render 的思路，改成原生轻渲染 + 增量更新的思路”。
