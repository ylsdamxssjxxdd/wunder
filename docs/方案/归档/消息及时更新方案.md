# 消息及时更新方案

## 1. 问题结论

当前“微信/其他渠道消息进入后，聊天页不能立刻显示、顺序偶发错乱、刷新后恢复”的核心问题，不在渠道接入层，也不在后端是否收到消息，而在用户侧前端聊天页的实时事件合并策略。

根因可以概括为两点：

1. `thread_status` 进入 `running` 后，会立即触发聊天页 `reconcile`，进而调用 `loadSessionDetail(...)`。
2. `loadSessionDetail(...)` 会中断当前 watcher，并用服务端返回的 `last_event_id` 直接推进本地游标，但此时服务端 chat history 里的用户消息不一定已经完成落库或返回给详情接口，导致这条真实存在的 `channel_message` 被跳过，或者在消息列表重建时被覆盖。

因此页面上会先看到“智能体运行中”，但用户消息没有及时出现；后续如果又发生 watcher 重启、快照覆盖、assistant 占位复用，就会进一步造成顺序错乱。手动刷新后恢复，是因为此时服务端 history 往往已经补齐。

一句话定性：

这是聊天前端的“事件游标推进 + 详情重建覆盖”问题，不是渠道消息接入慢。

## 2. 现有链路中的致命点

### 2.1 当前正确的后端意图

渠道消息入站后，后端已经先写入 `channel_message` stream event，再开始模型执行。这意味着系统设计目标本来就是：

1. 用户渠道消息先显示到聊天页。
2. 然后线程状态进入运行。
3. 然后模型开始回复。

所以当前体验与设计目标相违背，主故障点在前端消费阶段。

### 2.2 当前前端错误行为

当前聊天页 watcher 存在以下高风险路径：

1. 收到 `thread_status=running`
2. 立即触发 `scheduleWatchReconcile(0)`
3. `reconcile` 调用 `loadSessionDetail(...)`
4. `loadSessionDetail(...)` 先 `clearSessionWatcher()`
5. 再根据 `eventsPayload.last_event_id` 更新本地 runtime 游标
6. 再重建消息列表
7. 最后重新启动 watcher

这里有两个直接问题：

### 2.2.1 游标推进过早

`last_event_id` 表示“服务端已经存在更晚事件”，不等于“这些事件已经被前端物化进当前消息列表”。

如果把本地 watcher 游标直接推进到这个值，而消息列表里又没有那条对应的 `channel_message`，后续 watcher 就会从更大的 `after_event_id` 开始，导致该消息永久跳过。

### 2.2.2 详情重建会覆盖本地实时消息

`loadSessionDetail(...)` 重建消息主要依赖：

- `sessionDetail.messages`
- `eventsPayload.rounds`
- 本地 snapshot

但当前 snapshot merge 主要偏向 assistant 消息合并，对“刚刚通过 watch 实时插入、但服务端 history 尚未带回的 user 渠道消息”保护不够，导致：

- 用户消息丢失
- 用户消息落点被重排
- 旧的 pending assistant 被错误复用

## 3. 彻底解决的目标

彻底修复必须同时满足以下目标：

1. 渠道用户消息一旦进入 stream event，就应立即出现在聊天页。
2. `thread_status` 变化不能抢在用户消息之前破坏 UI 顺序。
3. 同会话 `reconcile` 不能中断活跃 watcher。
4. 任何一次详情刷新都不能覆盖尚未被服务端 history 物化的本地实时消息。
5. 本地游标只能推进到“已被 UI 消费并安全落地”的位置。
6. 消息顺序必须以 `stream_event_id` 为主序，而不是依赖不稳定的异步详情返回时机。

## 4. 总体方案

建议分为四层处理。

### 4.1 第一层：拆分“传输游标”和“物化游标”

当前只有一个 `lastEventId`，既承担“流已经看到哪里”，也承担“消息列表已经渲染到哪里”，职责混在一起，这是本问题的核心结构性缺陷。

应拆成至少两个游标：

- `transportLastEventId`
  含义：watch 通道已接收到的最大 event id。
- `materializedLastEventId`
  含义：已经成功反映到当前消息列表的最大 event id。

规则：

1. watcher 收到事件时，先更新 `transportLastEventId`。
2. 只有当该事件已经安全合并到当前消息列表后，才推进 `materializedLastEventId`。
3. watcher 重连、resume、详情 reconcile 的 `after_event_id`，必须基于 `materializedLastEventId`，不能直接用 `transportLastEventId` 或远端快照 `last_event_id`。

这样即使出现：

- 先收到 `thread_status`
- 再触发 reconcile
- history 尚未包含渠道用户消息

也不会把真实消息跳过去。

### 4.2 第二层：同会话 reconcile 改为“原地合并”，禁止打断 watcher

对当前激活会话，`reconcile` 不应再走“`clearSessionWatcher()` -> `loadSessionDetail()` -> `startSessionWatcher()`”这条重路径。

应改为：

1. watcher 持续保持连接。
2. reconcile 只拉取 `sessionDetail + eventsSnapshot`。
3. 将服务端返回内容与当前内存消息列表做原地合并。
4. 不重置 round state，不清空本地 watcher 上下文，不重启 watch 流。

只有以下情况才允许重建 watcher：

1. 会话切换。
2. 明确 slow client 恢复。
3. 连接已断开。
4. 服务端返回明确的 resume required。

这一步可以直接消除“状态变 running 导致 watcher 中断”的主要抖动源。

### 4.3 第三层：引入“实时消息保护层”

需要为来自 watch 的 `channel_message` 建立一层短生命周期保护，直到该消息被服务端 history 正式带回。

建议增加 `realtimePendingMessages`，按 session 维护：

- key：`stream_event_id`
- value：`role/content/created_at/hiddenInternal/...`

处理规则：

1. 收到 `channel_message` 后，先写入消息列表。
2. 同时登记到 `realtimePendingMessages`。
3. 详情刷新或快照合并时，如果服务端消息列表里还没有这条消息，则继续保留并重新合并回 UI。
4. 一旦服务端 history 中出现了相同 `stream_event_id` 或可确认等价的消息，再从 `realtimePendingMessages` 移除。

这层的作用是防止服务端详情暂时“看不到”时把用户消息冲掉。

### 4.4 第四层：消息排序统一改为事件序排序

对于有 `stream_event_id` 的消息，排序规则必须统一为：

1. `stream_event_id` 升序
2. `created_at` 升序
3. 原始插入顺序兜底

尤其是以下组合必须稳定：

1. `channel_message(user)`
2. `thread_status(running)`
3. `round_start/received/progress`
4. `assistant delta/final`

其中 `thread_status` 不应直接参与消息排序，它只更新线程状态，不改变消息链表顺序。

## 5. 关键改造点

### 5.1 禁止 `thread_status=running` 立即触发重量级详情重载

当前策略过于激进。建议调整为：

1. `thread_status` 只更新 runtime。
2. 不直接 `scheduleWatchReconcile(0)`。
3. 仅在满足以下条件时才触发轻量 reconcile：
   - 本地存在 pending assistant，但长时间没有后续事件
   - 远端 `last_event_id` 明显大于 `materializedLastEventId`
   - 检测到事件链断档

也就是说，`thread_status` 只应视为运行信号，不能视为“立刻重建整页消息”的信号。

### 5.2 `loadSessionDetail(...)` 只能推进安全游标

`loadSessionDetail(...)` 在拿到 `eventsPayload.last_event_id` 后，不能直接写入 watcher 的继续消费起点。

应遵守：

1. `runtime.remoteObservedLastEventId = eventsPayload.last_event_id`
2. `runtime.materializedLastEventId` 由当前消息列表真实最大已物化 event id 决定
3. watcher resume 一律从 `materializedLastEventId` 开始

这样即使详情接口比 history 先返回更大的 event id，也不会吞掉中间消息。

### 5.3 snapshot merge 不能只偏向 assistant

当前 merge 策略重点处理 assistant，需要补足 user 渠道消息保护逻辑：

1. 合并时优先按 `stream_event_id` 匹配 user message。
2. 如果本地存在 user `channel_message`，服务端详情暂时不存在，不允许删除。
3. 如果 assistant 占位存在，user message 必须插入在对应 assistant 之前。
4. 本地 `hiddenInternal` 等展示元信息也要保留，直到远端显式覆盖。

### 5.4 round state 不得复用到错误用户轮次

当前顺序错乱与 assistant placeholder 复用也有关。需要新增约束：

1. 来自渠道的 `channel_message(user)` 必须先入列。
2. assistant shell 只有在确认轮次匹配时才允许复用。
3. 若已有 pending assistant 属于旧轮次，必须终止或分离，不能作为新一轮承载容器。

## 6. 推荐实施顺序

### 阶段一：止血修复

目标：先解决“消息不及时出现”和“被吞”的主问题。

实施项：

1. 去掉 `thread_status=running` 的即时重量级 reconcile。
2. 同会话 reconcile 改为原地合并，不再 `clearSessionWatcher()`。
3. watcher resume 基于 `materializedLastEventId`，不再直接使用远端 `last_event_id`。
4. 为 `channel_message(user)` 增加本地保护缓存。

预期结果：

1. 渠道消息能立即显示。
2. 消息不会因详情重载被吞。
3. 手动刷新前后结果一致。

### 阶段二：顺序稳定化

目标：解决偶发错位和占位复用错误。

实施项：

1. 消息链表统一事件序排序。
2. user/assistant merge 统一走 `stream_event_id`。
3. round state 与 pending assistant 的复用增加轮次校验。

预期结果：

1. 不再出现用户消息跑到 assistant 后面。
2. 不再出现刷新前后顺序不同。

### 阶段三：结构性加固

目标：避免未来同类问题反复出现。

实施项：

1. 将 watch runtime 从 `chat.ts` 中继续拆分，单独封装游标状态机。
2. 明确区分 runtime state、message materialization、reconcile state。
3. 增加针对 `channel_message + thread_status + round_start` 组合时序的自动化测试。

## 7. 验收标准

以下场景全部通过，才算彻底解决。

### 7.1 单条渠道文本消息

步骤：

1. 微信发来一条文本消息。
2. 聊天页保持打开。

期望：

1. 200ms 级别内出现该用户消息。
2. 随后线程状态切为运行。
3. 模型回复开始流式出现。

### 7.2 渠道消息与状态事件几乎同时到达

期望：

1. 用户消息仍先可见。
2. 不会只显示“运行中”而无用户消息。

### 7.3 reconcile 期间不丢消息

步骤：

1. 渠道消息到达。
2. 同时触发详情刷新。

期望：

1. watcher 不被打断，或即使刷新也不会丢失该消息。
2. 刷新前后消息数量与顺序一致。

### 7.4 多条连续渠道消息

期望：

1. 多条 user message 按 `stream_event_id` 严格递增显示。
2. 不串轮次。
3. 不覆盖前一条消息。

### 7.5 手动刷新一致性

期望：

1. 刷新前看到的消息顺序与刷新后完全一致。
2. 不存在“刷新才正常”的现象。

## 8. 测试建议

建议补充以下自动化测试：

1. `channel_message(user)` 先到，`thread_status(running)` 后到。
2. `thread_status(running)` 先到，但 `channel_message(user)` 已在 stream storage 中可回放。
3. `loadSessionDetail(...)` 返回更大的 `last_event_id`，但 history 尚未包含该用户消息。
4. 同会话 reconcile 不应重启 watcher。
5. 本地实时 user message 在下一次详情拉取中必须被保留，直到远端 history 接管。

## 9. 最终建议

如果要彻底解决，不建议继续在现有逻辑上做“小补丁”式修修补补。最少要完成下面三件事：

1. 拆分 `transportLastEventId` 与 `materializedLastEventId`。
2. 同会话 reconcile 改为原地合并，禁止因为 `thread_status` 打断 watcher。
3. 为 `channel_message(user)` 增加本地保护层，直到服务端 history 确认接管。

这三项做完，才能真正把“渠道消息先显示，然后模型开始回复”的体验稳定下来。
