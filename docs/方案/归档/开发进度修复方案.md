## 1. 审查结论

本次对比以当前仓库 `C:\Users\sjxx\Desktop\wunder` 为基线，以恢复目录 `C:\Users\sjxx\Desktop\wunder-` 为事故后残留样本，重点审查了三条开发主线：

- 静态文档
- 工具技能统一
- 错误处理对齐 Codex

结论很明确：

1. 静态文档不是“没开始”，而是已经做出了较完整内容，但恢复目录里的构建脚本和发布产物损坏较重，当前仓库尚未接回这套成果。
2. 工具技能统一目前主要停留在方案和局部收敛阶段，运行时的共享链路已经被压缩，但数据模型、接口协议和前端消费方式还没有统一。

- 恢复目录缺少整个 `frontend/` 目录，因此事故发生后如果有用户侧前端改动，这份恢复样本无法帮助找回。

## 2. 静态文档

### 2.1 当前进度判断

这一条线已经形成了“方案 + 内容 + 一部分发布产物”，但没有形成可直接合并的稳定结果。

当前仓库里已经有这些基础：

- `docs/wunder补全文档落地方案.md` 已经写明正式文档源应落在 `docs/静态站文档/`
- `src/main.rs` 已经把 `web/` 作为根静态目录挂载，因此 `/docs/` 本身没有后端阻塞
- 当前 `web/docs/` 仍主要是旧的 `api-docs.json`、`paper.md` 和图示资源

恢复目录里额外找到了这些成果：

- `docs/静态站文档/site.json`
- `docs/静态站文档/zh-CN/**` 大批正式文档页
- `web/docs/**` 静态站发布页
- `scripts/build_docs_site.py`

### 2.2 已确认的问题

1. 文档源不完整。
   当前 `site.json` 导航一共引用了 72 个页面，但恢复目录里只找回了 61 个可用页面源文件，缺了整组 `zh-CN/concepts/*` 共 11 页。

2. 文档源有损坏。
   `docs/静态站文档/zh-CN/integration/wunder-api.md` 含大量 `NUL` 字节，不能作为可信源文件直接合并。

3. 构建脚本已损坏。
   `scripts/build_docs_site.py` 含 `4096` 个 `NUL` 字节，不能作为可执行脚本继续维护。

4. 发布产物大面积损坏。
   恢复目录 `web/docs/` 下共有 86 个文本类文件，其中 24 个带 `NUL` 字节，包含：
   - `web/docs/site.js`
   - `web/docs/site.css`
   - `web/docs/manifest.json`
   - `web/docs/search.json`
   - 部分页面 `index.html`

5. 方案文档与实现命名已经漂移。
   `docs/wunder补全文档落地方案.md` 里规划的是 `web/docs/app.js`、`web/docs/app.css`，恢复目录实际产物是 `web/docs/site.js`、`web/docs/site.css`，说明这条线在实现阶段没有回写文档。

6. 管理端入口还没有接完。
   当前仓库 `web/index.html` 仍只有 API 文档和论文入口，没有正式“静态文档”入口，说明即使文档站生成成功，后台导航也还没收尾。

### 2.3 可恢复度判断

静态文档这一条线是可以救回来的，但不能“整包复制”恢复目录内容。

可信部分：

- `docs/静态站文档/site.json`
- `docs/静态站文档/` 中绝大多数 Markdown 源文件
- `web/docs/` 里未损坏页面的 `docs-page-data` JSON，可反向提取正文

不可信部分：

- `scripts/build_docs_site.py`
- 大量生成后的 `web/docs/*`
- 已损坏的单页源文件

### 2.4 修复建议

P0 做法：

1. 先把恢复目录里未损坏的 `docs/静态站文档/` 源文件与 `site.json` 提取回当前仓库。
2. 从恢复目录 `web/docs/zh-CN/concepts/**/index.html` 的 `docs-page-data` 中反提 11 个缺失概念页源码。
3. 同样从 `web/docs/zh-CN/integration/wunder-api/index.html` 反提 `wunder-api.md`，替换损坏源文件。
4. 不要复用恢复目录的 `scripts/build_docs_site.py`，按照当前方案文档和实际需要在当前仓库重写一份新脚本。
5. 用当前仓库重新生成 `web/docs/`，不要直接复制恢复目录产物。
6. 生成链路稳定后，再决定是否在 `web/index.html` 增加“静态文档”入口。

## 3. 工具技能统一

### 3.1 当前进度判断

这条线不是丢失了“快完成的代码”，而是明显停在“方案已成形，局部治理已开始，但核心模型还没换掉”的阶段。

恢复目录里新增了：

- `docs/工具技能统一优化方案.md`

但没有找到与方案相匹配、且可可信合并的完整实现增量。

当前仓库已经做过的收敛主要是：

- 共享工具运行时链路已停用，`shared_tools` 大多只作为兼容字段保留
- 用户自建技能在智能体设置中不可见的问题已修复
- worker-card 与依赖声明做过一轮整理

### 3.2 尚未收尾的核心问题

1. 统一能力模型还不存在。
   方案里提出的 `AbilityDescriptor`、`runtime_name`、`display_name`、`abilities_json`、`worker-card@2`，当前仓库都还没有成为正式主模型。

2. `/user_tools/catalog` 仍然是多桶返回，不是统一能力目录。
   当前 `src/core/schemas.rs` 中 `AvailableToolsResponse` 仍保留：
   - `builtin_tools`
   - `skills`
   - `knowledge_tools`
   - `user_tools`
   - `user_skills`
   - `shared_tools`
   - `shared_tools_selected`

3. 智能体持久化仍是旧三段式。
   当前 `src/api/user_agents.rs`、`src/storage/*`、`src/services/inner_visible/worker_card.rs` 仍围绕：
   - `tool_names`
   - `declared_tool_names`
   - `declared_skill_names`
   运转，没有切到统一 `abilities_json`。

4. worker-card 仍是旧能力结构。
   当前前后端仍以 `abilities.tool_names` 和 `abilities.skills` 双数组处理，不是方案里的结构化 `abilities[]`。

5. 共享能力 UI 仍然残留。
   虽然运行时已经基本停掉共享能力，但当前仓库仍保留：
   - `/wunder/user_tools/shared_tools` 接口
   - `frontend/src/components/user-tools/UserSharedToolsPanel.vue`
   - `frontend/src/components/user-tools/UserSharedToolsModal.vue`
   - `web/modules/tools.js`
   - `web/modules/user-tools.js`
   这会继续拖住模型统一。

6. 前后端仍在兼容旧分组与旧字段。
   当前这些位置还在显式处理 `shared_tools`、`user_skills`、`user_tools`、`declared_*`：
   - `frontend/src/utils/toolSummary.ts`
   - `frontend/src/utils/promptHighlight.ts`
   - `frontend/src/utils/workerCard.ts`
   - `frontend/src/utils/agentDependencyStatus.ts`

### 3.3 进度判断

这条线的真实状态应定义为：

- 方案已明确
- 运行时共享链路已收缩
- 统一目录模型、统一存储模型、统一前端消费模型尚未落地

也就是说，当前没有“可直接从恢复目录捞回并合并的大块实现”，需要在当前仓库基于已有方案继续做完。

### 3.4 修复建议

P1 做法：

1. 新建 `src/services/abilities/`，把能力目录、名称解析和运行时解析收口到单一模块。
2. 新增统一目录输出 `items[]`，旧接口字段短期继续兼容映射，避免一次性打断前端。
3. 为智能体和 worker-card 引入统一结构化能力字段，逐步替换 `tool_names/declared_*`。
4. 清理共享能力残留接口与前端组件，至少先把它们降级为只读兼容，再逐步移除。
5. 同步更新 `docs/API文档.md`、`docs/设计方案.md` 和工具/技能相关说明，避免继续一边写旧模型一边做新模型。

## 4. 错误处理对齐 Codex

### 4.1 当前进度判断

这条线和前两条不同，当前仓库已经明显前进，不是事故后完全丢失。

当前仓库已落地的部分包括：

- `src/orchestrator/error.rs` 已有统一 `error_meta`
- `src/orchestrator/execute.rs` 已有 `turn_terminal`、`approval_resolved`
- `src/orchestrator/event_stream.rs` 已持久化这些终结事件
- `src/core/approval_registry.rs` 已引入共享 `PendingApprovalRegistry`
- `docs/功能迭代.md` 已记录异常处理第一批、第二批落地

### 4.2 与恢复目录对比后的判断

恢复目录里没有发现可信的后续增量：

- `src/api/errors.rs`
- `src/orchestrator/error.rs`
- `src/api/gateway_ws.rs`
- `src/services/skills.rs`
- `web/modules/skills.js`

这些能正常读取的关键文件，与当前仓库在规范化文本后内容一致，没有额外实现可捞。

而真正可能承载后续改造的文件，如：

- `src/api/chat_ws.rs`
- `src/orchestrator/execute.rs`
- `src/orchestrator/event_stream.rs`
- `tests/gateway_regression.rs`

在恢复目录中都存在大量 `NUL` 字节，不能作为可信来源。

### 4.3 还没完成的部分

按照现有 `docs/系统异常处理优化方案.md`，当前还缺至少三块：

1. 线程运行时状态机还没建完。
   方案里要求的 `not_loaded / idle / active / system_error` 显式 runtime 状态，以及 `thread_status / thread_closed` 事件，目前仓库里还没有真正落地。

2. turn 级临时状态所有权还没真正收口。
   现在虽然有 `PendingApprovalRegistry`，但还没有方案里定义的 `ActiveTurnState`，也还没把 pending approval / user input / dynamic request 全部统一归到 turn 生命周期。

3. 恢复语义测试不够。
   当前有事件持久化和错误映射测试，但缺少方案中强调的端到端回归，例如：
   - 流式失败后下一轮仍可继续
   - interrupt 必须 resolve 掉待审批
   - unsubscribe/unload 后 runtime 回到干净态
   - tool timeout 后 active turn 被彻底清理

### 4.4 修复建议

P2 做法：

1. 先补 `ActiveTurnState`，把 turn 临时状态的所有权真正统一。
2. 再补线程 runtime 状态机与 `thread_status/thread_closed` 事件。
3. 最后补一组高价值异常回归测试，把恢复语义固化下来。

这条线不建议从恢复目录继续“扒代码”，建议直接在当前仓库接着做。

## 5. 不建议直接合并的恢复内容

以下内容不应作为源代码直接回灌：

- 恢复目录 `data/` 下的运行时数据、用户工具数据、throughput 结果
- 带 `NUL` 字节的脚本、Markdown、JSON、HTML、JS、CSS
- 恢复目录里的 `web/docs/` 生成产物
- `__pycache__`、演示输出、临时导出物

这些内容最多只适合作为“人工抽取线索”，不适合作为代码或文档真源。

## 6. 推荐执行顺序

建议按下面顺序恢复，避免三条线互相干扰：

1. 先修静态文档。
   这是最容易从恢复目录里提纯的成果，而且对后续开发也能反向提供参考文档。

2. 再收尾工具技能统一。
   因为它牵涉接口、存储、worker-card、前端目录模型，改动面最大，需要在文档恢复后按统一模型推进。

3. 最后继续错误处理第三阶段。
   当前这条线已经有明确基线，不依赖恢复目录，适合在模型统一之后做状态机和回归测试。

## 7. 最终判断

如果以“事故前这三条线做到了哪一步”来定义当前状态，可以这样总结：

- 静态文档：内容生产已经做了大半，但构建链路和发布产物损坏，当前仓库还未接入。
- 工具技能统一：方案完成，局部收敛已开始，但真正的统一模型还没有落地。
- 错误处理对齐 Codex：前两批已在当前仓库落地，后续阶段仍待继续实现，恢复目录没有额外可信增量。

因此，最合理的修复策略不是“把恢复目录整个覆盖回来”，而是：

- 只提纯静态文档的可用源内容
- 把工具技能统一和错误处理都继续放在当前仓库中按方案完成
- 严格丢弃恢复目录中所有损坏或运行时性质的文件
