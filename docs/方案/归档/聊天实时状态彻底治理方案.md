# 聊天实时状态彻底治理方案

## 1. 目标

本方案用于根治用户侧聊天页长期反复出现的实时刷新、状态不一致、气泡错序、重复 assistant、头像卡运行中、按钮假死等体验问题。

本次治理不以兼容旧实现为目标。聊天实时链路统一收敛到 WebSocket，SSE 聊天流式与 watch 兜底链路直接移除。HTTP 仅保留会话列表、历史详情、配置管理等普通请求，不再承担聊天运行态流式传输。

最终目标：

- 用户发送消息后，页面立即进入可解释的运行态，不再闪已完成。
- 同一用户轮次下最多只有一个最终 assistant 气泡。
- assistant 气泡永远不会跑到所属用户气泡上方。
- 终态后头像、中栏状态、按钮 busy、消息统计同时收口。
- 断线、重连、页面隐藏恢复后，前端通过 WS replay 恢复，不需要刷新页面。
- 前端组件不再各自判断运行态，只消费统一投影状态。
- 任何乱序、重复、丢事件、旧快照覆盖 live 状态的问题都能被不变量测试发现。

## 2. 基本判断

当前问题不是“刷新频率不够”，而是聊天运行态缺少统一事实源。

现有链路中，同一条会话可能同时被以下入口修改：

- 本地 optimistic user / assistant 占位消息。
- 发送请求返回的 WS/SSE 增量事件。
- watcher / resume 补偿事件。
- channel sideband 用户消息或 assistant 消息。
- `loadSessionDetail` hydrate 快照。
- 页面组件自己的 busy、avatar、stats、button 计算。

这些入口没有统一 reducer，也缺少稳定的 `turn_id/message_id/event_seq` 协议约束，前端只能用 `created_at`、`stream_round`、内容相似、stream flag、pending 状态猜测消息归属。只要事件乱序或 hydrate 落后，就会出现 UI 状态分叉。

因此根治方向是：

- 通信层：聊天实时只走 WebSocket。
- 协议层：所有运行事件必须带稳定轮次和消息标识。
- 状态层：前端只允许一个 reducer 修改消息和 runtime。
- 渲染层：所有 UI 只读统一 selector。
- 验证层：用乱序事件回归和运行时不变量堵住复发。

## 3. 非目标

- 不继续保留聊天 SSE fallback。
- 不继续保留 `server.chat_stream_channel=ws/sse` 这种运行时选择。
- 不继续支持前端按内容或时间猜测 assistant 属于哪一轮作为主路径。
- 不把问题继续下放给 `MessengerView.vue` / `ChatView.vue` 组件层修补。
- 不追求在旧协议上做小修小补。

### 3.1 执行红线

为避免治理过程中再次走回“多通道兼容 + 组件层补丁”的老路，本方案执行期必须遵守以下红线：

- 任何新增聊天实时能力只能接入 WS 事件协议，不允许新增 SSE fallback、polling fallback 或组件私有补偿链路。
- 遇到旧 SSE 分支导致的体验问题，只允许删除、隔离或迁移到 WS，不再继续修补旧分支。
- `MessengerView.vue`、`ChatView.vue`、消息气泡组件不得新增长期运行态字段，只能读取 selector 输出。
- `loadSessionDetail`、重连补偿、sideband 消息不得直接覆盖 active session 的 visible messages。
- 发布前必须通过反向搜索确认聊天域没有有效 SSE 入口，并通过乱序、重复、断线、旧 snapshot 覆盖 live 的回归用例。
- 如果某阶段无法满足上述红线，必须停在 shadow mode 或灰度环境，不能把半套新协议和半套旧协议同时推给用户。

## 5. 目标架构

### 5.1 单一事实源

聊天页事实源为：

```text
后端持久化事件日志
  -> WebSocket replay/live push
  -> 前端 chatRuntimeReducer
  -> 前端 ChatRuntimeProjection
  -> UI selector
  -> 组件渲染
```

任何组件、watcher、hydrate 回调都不能绕过 reducer 直接修改消息运行态。

### 5.2 WebSocket 单通道

聊天 WebSocket 承担三类能力：

- `command`：客户端发起动作，例如 start、cancel、ack、watch、unwatch、sync_request。
- `event`：服务端推送持久化事件和 live 事件。
- `control`：连接健康、重放、补齐、背压、重连协商。

HTTP 仍可用于：

- 会话列表。
- 会话历史首次加载。
- 用户设置。
- 工具配置。
- 非实时管理请求。

HTTP 不再用于：

- 聊天流式输出。
- 聊天运行态 watch。
- 聊天增量 resume。

### 5.3 前端运行态投影

新增前端模块：

```text
frontend/src/realtime/chat/
  chatWsClient.ts
  chatRuntimeTypes.ts
  chatRuntimeReducer.ts
  chatRuntimeSelectors.ts
  chatRuntimeInvariants.ts
  chatRuntimeReplay.ts
  chatRuntimeDebug.ts
```

职责边界：

- `chatWsClient.ts` 只负责连接、认证、重连、发送命令、接收原始事件。
- `chatRuntimeReducer.ts` 只负责根据事件生成状态。
- `chatRuntimeSelectors.ts` 只负责给 UI 输出状态。
- `chatRuntimeInvariants.ts` 只负责开发期和测试期校验。
- `chat.ts` 逐步退化为会话管理 store，不再直接承载实时状态机。
- `MessengerView.vue` 和 `ChatView.vue` 只订阅 selector。

## 6. 后端事件协议

### 6.1 事件基础字段

所有聊天实时事件必须包含：

```json
{
  "schema_version": 2,
  "event_id": "evt_xxx",
  "event_seq": 1024,
  "session_id": "session_xxx",
  "user_id": "user_xxx",
  "agent_id": "agent_xxx",
  "user_turn_id": "ut_xxx",
  "model_turn_id": "mt_xxx",
  "message_id": "msg_xxx",
  "event_type": "assistant_delta",
  "runtime_status": "running",
  "is_terminal": false,
  "created_at": "2026-04-30T10:12:05.000+08:00",
  "payload": {}
}
```

字段约束：

- `event_seq` 是 session 内单调递增整数。
- `event_id` 全局唯一。
- `user_turn_id` 在用户每发送一条可见消息时生成。
- `model_turn_id` 在模型每轮动作开始时生成。
- `message_id` 在消息创建时生成，并在后续 delta/final/tool 事件中保持不变。
- `runtime_status` 只能来自后端状态机。
- `is_terminal=true` 表示该事件对所属 turn 或 session 具备终态意义。

### 6.2 客户端命令

客户端发送：

```json
{
  "type": "command",
  "command": "start",
  "request_id": "req_xxx",
  "session_id": "session_xxx",
  "client_seq": 12,
  "last_seen_event_seq": 1023,
  "payload": {
    "content": "用户输入",
    "attachments": []
  }
}
```

命令类型：

- `watch`：订阅一个 session，并声明 `last_seen_event_seq`。
- `unwatch`：取消订阅。
- `start`：发送用户消息并启动智能体。
- `cancel`：取消当前运行。
- `ack`：确认已处理到某个 seq。
- `sync_request`：主动要求服务端补齐事件或返回 snapshot。
- `ping`：连接保活。

### 6.3 服务端事件类型

最小事件集：

- `session_snapshot`：会话基线快照。
- `session_runtime`：会话运行态变化。
- `user_message_created`：用户消息创建。
- `assistant_message_created`：assistant 占位消息创建。
- `assistant_delta`：assistant 内容增量。
- `assistant_reasoning_delta`：reasoning 增量。
- `tool_call_started`：工具调用开始。
- `tool_call_delta`：工具调用过程增量。
- `tool_call_completed`：工具调用完成。
- `tool_call_failed`：工具调用失败。
- `assistant_final`：assistant 最终内容。
- `turn_completed`：模型轮次完成。
- `turn_failed`：模型轮次失败。
- `turn_cancelled`：模型轮次取消。
- `session_idle`：会话已无活动运行。
- `sync_required`：客户端状态落后，需要拉取 snapshot 并 replay。

### 6.4 Snapshot 与 Replay

snapshot 不能直接覆盖 live 状态。必须采用：

```text
snapshot(seq=N) + replay events where event_seq > N
```

规则：

- 前端进入 session 时，先 `watch(last_seen_event_seq)`。
- 服务端若能 replay，直接推 `event_seq > last_seen_event_seq`。
- 服务端若不能完整 replay，先推 `session_snapshot(snapshot_seq=N)`，再推 N 之后事件。
- 前端 reducer 只接受 `event_seq` 大于当前已应用 seq 的事件。
- 旧 snapshot 不允许覆盖新 live event。

## 7. 状态机

### 7.1 Session 状态

```text
not_loaded
  -> idle
  -> queued
  -> running
  -> waiting_approval
  -> waiting_user_input
  -> finalizing
  -> completed
  -> idle

running -> cancelling -> cancelled -> idle
running -> failed -> idle
running -> reconnecting -> running
reconnecting -> completed -> idle
```

UI busy 只由以下状态决定：

- `queued`
- `running`
- `waiting_approval`
- `waiting_user_input`
- `finalizing`
- `cancelling`
- `reconnecting`

### 7.2 User Turn 状态

```text
created
  -> accepted
  -> dispatched
  -> model_running
  -> waiting_user_input
  -> completed

model_running -> failed
model_running -> cancelled
```

### 7.3 Model Turn 状态

```text
created
  -> waiting_first_output
  -> streaming
  -> tool_running
  -> finalizing
  -> completed

waiting_first_output -> failed
streaming -> failed
tool_running -> failed
streaming -> cancelled
tool_running -> cancelled
```

### 7.4 Message 状态

```text
placeholder
  -> waiting_first_output
  -> streaming
  -> tooling
  -> final

placeholder -> failed
waiting_first_output -> failed
streaming -> failed
tooling -> failed
placeholder -> cancelled
streaming -> cancelled
```

消息状态由 reducer 根据事件计算，不允许组件直接组合 flag。

## 8. 前端状态结构

### 8.1 Store 投影结构

```ts
type ChatRuntimeProjection = {
  sessions: Record<string, SessionProjection>;
  activeSessionId: string | null;
};

type SessionProjection = {
  sessionId: string;
  agentId: string;
  appliedSeq: number;
  snapshotSeq: number;
  connectionState: 'connected' | 'reconnecting' | 'offline';
  runtimeStatus: SessionRuntimeStatus;
  busyReason: BusyReason | null;
  userTurns: string[];
  modelTurns: string[];
  messages: string[];
  messageById: Record<string, MessageProjection>;
  userTurnById: Record<string, UserTurnProjection>;
  modelTurnById: Record<string, ModelTurnProjection>;
  invariantViolations: RuntimeInvariantViolation[];
};
```

### 8.2 Selector

必须统一提供：

- `selectSessionBusy(sessionId)`
- `selectSessionBusyReason(sessionId)`
- `selectSessionRuntimeLabel(sessionId)`
- `selectMessageRuntime(messageId)`
- `selectMessageStats(messageId)`
- `selectVisibleMessages(sessionId)`
- `selectLatestAssistantForTurn(userTurnId)`
- `selectCanSend(sessionId)`
- `selectCanStartNewSession(agentId)`
- `selectCanSwitchSession(sessionId)`

UI 禁止直接读取以下字段判断运行态：

- `stream_incomplete`
- `workflowStreaming`
- `reasoningStreaming`
- `waiting_updated_at_ms`
- `threadStatus`
- `loadingBySession`

这些字段只能在 reducer 或兼容迁移适配器内部出现。

## 9. Reducer 规则

### 9.1 事件应用规则

- `event_seq <= appliedSeq` 的事件直接丢弃。
- 同一 `event_id` 重复到达直接丢弃。
- 同一 `message_id` 的 delta/final 只能更新同一个 message。
- 同一 `model_turn_id` 只能有一个最终 assistant message。
- `assistant_final` 必须把所属 message 状态置为 `final`。
- `turn_completed/failed/cancelled` 必须收口所属 model turn。
- `session_idle` 必须收口 session busy。
- 任何事件缺少必要 ID，进入 quarantine，不参与 UI 投影。

### 9.2 Hydrate 规则

旧的 `loadSessionDetail` 不再直接替换 active session 消息数组。

新规则：

- 非 active session 可用 snapshot 快速构建 projection。
- active session 必须走 `session_snapshot` 事件进入 reducer。
- snapshot 只作为基线，不允许覆盖 `appliedSeq` 之后的 live 状态。
- snapshot 中缺失 live message 时，不删除 live message，除非收到明确 `message_deleted` 或 `turn_compacted` 事件。

### 9.3 Sideband 规则

sideband 用户消息和 assistant 消息不能直接按到达顺序 append。

必须按以下优先级归属：

1. `message_id`
2. `user_turn_id`
3. `model_turn_id`
4. `event_seq`

缺少这些字段的 sideband 事件视为协议错误，不再进入正常消息流。

## 10. WebSocket 连接策略

### 10.1 连接生命周期

```text
connect
  -> authenticate
  -> watch(active_session)
  -> replay
  -> live
  -> reconnecting
  -> replay from last_seen_event_seq
  -> live
```

### 10.2 重连策略

- 断线后立即进入 `reconnecting`，UI 显示可解释状态。
- 第一次重连延迟 300ms，之后指数退避，最大 8s。
- 重连成功后必须发送 `watch(last_seen_event_seq)`。
- 如果服务端返回 `sync_required`，客户端请求 snapshot，再 replay snapshot 之后事件。
- 重连期间禁止把 session 标记为 completed。

### 10.3 背压策略

- 服务端检测客户端处理落后时发送 `sync_required`，不再尝试无限推 delta。
- 前端收到 `sync_required` 后暂停渲染高频 delta，进入 snapshot + replay。
- 大量工具输出或模型 delta 必须支持合并事件，但合并后仍保留最终 message_id 和 seq。

- 为模型轮次增加 `model_turn_id`。
- 为消息增加稳定 `message_id`。

- 后端聊天域不再返回 `text/event-stream`。
- 断线重连通过 WS replay 恢复。

### 阶段 C：前端 reducer 投影

目标：前端只有一个地方能修改聊天运行态。

任务：

- 新建 `frontend/src/realtime/chat/` 模块。

- 头像、消息状态、中栏、按钮 busy 读取同一 selector。

### 阶段 D：组件瘦身

目标：组件只渲染 projection。

任务：

- 移除 `MessengerView.vue` 内的运行态拼接逻辑。
- 移除 `ChatView.vue` 内重复消息状态计算。
- `MessageToolWorkflow.vue` 只展示 selector 输出的 workflow projection。
- 新建或整理消息组件输入结构。

交付：

- Messenger 组件层运行态逻辑减少到 selector 调用。
- 消息气泡不直接修改 message 对象。

验收：

- 组件层没有 `stream_incomplete/workflowStreaming/reasoningStreaming` 组合判断。
- 状态展示差异只能来自 selector，不来自组件私有逻辑。

### 阶段 E：不变量和乱序测试

目标：让体验类 bug 在开发期直接失败。

任务：

- 实现事件乱序生成器。
- 编写 reducer 不变量测试。
- 编写 WS reconnect/replay 集成测试。
- 编写前端页面级 Playwright 回归。

必须覆盖：

- assistant 事件先到，用户 sideband 后到。
- final 先到，snapshot 后到。
- snapshot 旧于 live event。
- cancel 后服务端补 failed/error。
- 工具调用期间断线重连。
- 页面隐藏后恢复。
- 同一事件重复到达。
- event_seq 跳号。
- sync_required 后 snapshot + replay。
- 慢客户端背压。

不变量：

- assistant 不得出现在所属 user turn 之前。
- 同一 `model_turn_id` 最多一个 final assistant。
- terminal 后 session busy 必须为 false。
- `appliedSeq` 只能递增。
- active session snapshot 不得删除未终态 live message。
- UI busy 必须有 `busyReason`。
- 缺少必要 ID 的事件不得进入 visible messages。

### 阶段 F：删除旧链路和文档同步

目标：旧实现不可再被调用。

任务：

- 更新管理端设置说明。

验收：

- `rg "chat_stream_channel|sendMessageStream|resumeMessageStream|text/event-stream"` 不再命中聊天域有效代码。
- 用户侧聊天运行态只存在 WS path。
- 文档不再说聊天 SSE 兜底。

## 12. 前端实施节点

### 节点 1：建立新模块但不接管 UI

- 新建 `frontend/src/realtime/chat/`。
- 把现有 WS 事件转换为新事件类型。
- reducer 在 shadow mode 下运行，只记录 projection 和 invariant。
- 不影响现有 UI。

验收：

- debug 面板可导出 shadow projection。
- shadow projection 与当前 UI 大体一致。
- invariant 能捕获已知错序样例。

### 节点 2：发送链路切到 WS command

- `sendMessage` 不再走 HTTP/SSE。
- 本地 optimistic 消息由 reducer 的 `client_message_submitted` 事件生成。
- 服务端确认后用 `user_message_created` 回填 ID。

验收：

- 发送消息只出现一次 user bubble。
- 首包前状态为 `waiting_first_output`。
- 按钮 busy 有明确原因。

### 节点 3：接管 assistant 流式渲染

- assistant placeholder、delta、tool、final 全部由 reducer 管理。
- tool workflow 只挂在 `message_id` 下。

验收：

- 同轮 assistant 不重复。
- 工具调用、reasoning、final 内容不分裂到多条气泡。

### 节点 4：接管 hydrate 和重连

- `loadSessionDetail` 对 active session 只产生 snapshot event。
- WS reconnect 统一 replay。

验收：

- 刷新、断线、切后台恢复后无需手动刷新。
- snapshot 不再覆盖 live 状态。

### 节点 5：删除旧状态路径

- 删除旧 pending helper 中仅为兼容旧 SSE 的分支。
- 删除旧 watch channel message runtime 或降级为协议适配器。
- 删除组件内运行态判断。

验收：

- 状态相关 bug 只需看 reducer 和 selector。
- `MessengerView.vue` 不再是实时状态主战场。

## 13. 后端实施节点

### 节点 1：事件身份字段落库

- PostgreSQL 和 SQLite 均支持 session 内 event seq。
- 新事件写入必须在事务中分配 seq。
- 同一消息多次 delta 不新建 message_id。

### 节点 2：Orchestrator 发结构化事件

- 用户轮次创建时生成 `user_turn_id`。
- 模型动作开始时生成 `model_turn_id`。
- 最终回复、工具调用、错误、取消都绑定 model turn。
- final_response 工具类事件不再制造额外 assistant 消息。

### 节点 3：WS gateway 接管 start/watch/replay

- WS 连接后必须 authenticate。
- `watch` 根据 `last_seen_event_seq` replay。
- `start` 命令返回 ack，然后事件流继续推送。
- `cancel` 命令生成取消事件并收口 runtime。

### 节点 4：删除聊天 SSE

- 删除或禁用聊天 SSE endpoint。
- 删除聊天 SSE 测试。
- 删除配置项。
- 管理端不再暴露聊天流式通道选择。

## 16. 回归测试矩阵

| 场景 | 事件顺序 | 预期 |
| --- | --- | --- |
| 正常发送 | user_created -> assistant_created -> delta -> final -> idle | 一条 user，一条 assistant，最终 idle |
| 首包慢 | user_created -> assistant_created，3 秒后 delta | 显示等待首包，不显示完成 |
| 用户 sideband 晚到 | assistant_created -> final -> user_created | reducer 按 turn_id 排序，user 在 assistant 前 |
| snapshot 旧 | live seq=20 后收到 snapshot seq=15 | snapshot 不覆盖 live |
| final 早于 hydrate | final seq=30，snapshot seq=25 | 不重复 assistant |
| cancel 后补 failed | cancel seq=40，failed seq=41 | 同一 assistant 收口，不新增重复气泡 |
| 重复事件 | final seq=50 到达两次 | 第二次丢弃 |
| seq 跳号 | 收到 seq=60，当前 58 | 请求 sync |
| 断线重连 | disconnect，watch last_seen=80 | replay 81+，状态恢复 |
| 背压 | 服务端发 sync_required | snapshot + replay 后恢复 |

## 18. 风险和处理

| 风险 | 处理 |
| --- | --- |
| 旧客户端连接新服务失败 | 本次允许破坏兼容，发布时要求前后端同版本 |
| WS 连接被代理中断 | 通过 replay 和 sync_required 恢复，不再回退 SSE |
| event_seq 迁移复杂 | 新事件使用 seq，旧历史只作为 snapshot，不参与 live replay |
| reducer 初期和旧 UI 不一致 | shadow mode 先跑 invariant，不直接接管 UI |
| 文件过大继续难维护 | 新逻辑落到 `frontend/src/realtime/chat/`，不继续堆进 `chat.ts` |

## 19. 最终判断

这次治理的核心不是把刷新做得更勤，而是让聊天页从“多个入口共同修改 UI”变成“后端事件日志驱动前端单一投影”。只要 SSE、watch、hydrate、本地 optimistic、组件私有状态继续并存，这类体验问题就会反复出现。

彻底方案必须接受一次破坏兼容的收敛：聊天实时只保留 WebSocket，所有事件带稳定身份，前端统一 reducer，组件只读 selector，再用乱序测试和不变量守住边界。
