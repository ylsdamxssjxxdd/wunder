# 聊天按钮状态收敛与刷新自愈守则

## 1. 目标

避免再次出现以下问题：

- 模型已结束、线程已终态，但发送按钮仍显示“中止”。
- 页面内刷新无效，必须整页刷新才恢复。
- 新建线程/刷新存在延迟窗口，用户抢先发送导致消息被吃掉。

## 2. 必须长期满足的状态不变量

1. 终态优先
- 当 `runtime` 已终态（`idle/not_loaded/system_error`）且交互控制器已清空（`send/resume/compact` 均无）时，前端不得继续显示“中止”。

2. busy 判定三信号一致
- `busy = loadingBySession || runtimeBusy || runningAssistantMessage`。
- 但若满足“终态优先”条件，允许抑制消息残留 running 标记，强制回落为 idle。

3. 终态清理必须全量
- 终态收敛不得只清一条 pending assistant，必须清理当前用户轮次尾部所有 assistant 的 streaming 标记（`stream_incomplete/workflowStreaming/reasoningStreaming`）。

4. 刷新自愈必须可校准
- 刷新流程遇到 `busy=true` 时，不得先验信任 `runtime=running`；必须执行至少一次强制 `loadSessionDetail(...forceHydrateForeground)` 校准后再判定。

5. 延迟动作必须阻塞前台交互
- 新建线程/刷新期间必须启用全屏交互阻塞（遮罩 + 焦点回收 + 发送门禁），避免输入框回车抢发。

## 3. 代码落点（后续改动必须同步检查）

- `frontend/src/stores/chat.ts`
  - `isSessionBusy` 聚合与终态兜底

  - watchdog `running=false` 收敛
- `frontend/src/stores/chatPendingMessage.ts`
  - `findPendingAssistantMessage`

## 5. 调试与排障标准流程

### 5.1 开关

浏览器控制台：

```js
localStorage.setItem('wunder:chat-debug', '1');
location.reload();
```

关闭：

```js
localStorage.removeItem('wunder:chat-debug');
```

### 5.2 关键日志事件

- `messenger.busy snapshot-change`
- `chat.store.loading set-session-loading`
- `chat.store.runtime terminal-runtime-event`
- `chat.store.runtime settle-terminal-state`
- `messenger.refresh *`
- `chat.store.busy suppress-stale-assistant-busy-after-terminal`
- `messenger.busy force-idle-after-terminal-runtime`

### 5.3 最小排障证据

出现“按钮卡中止”时，必须同时提供：

1. 问题发生前后 20 秒 `wunder-chat-debug` 日志。
2. 当下 `messenger.busy snapshot-change` 展开对象。
3. `window.__wunderChatDebugDump()` 输出片段（可脱敏）。

## 6. 发布验收清单（聊天页）

- 第一轮回复结束后按钮可从“中止”回落为“发送”。
- 第二轮连续发送场景按钮仍可回落。
- 点击页面刷新可修复 stale busy，不依赖整页刷新。
- 新建线程/刷新期间不能发送消息，遮罩期间无“消息被吃掉”。
- thread 终态（`idle/not_loaded/thread_closed`）后 1 秒内按钮状态与会话状态一致。
