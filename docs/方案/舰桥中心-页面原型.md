# 舰桥中心页面原型

## 1. 页面定位

`舰桥中心` 是管理员侧的桥接治理页面，不是聊天页，不是用户页，也不是智能体设置页。

页面目标只有三件事：

1. 配置共享渠道入口
2. 观察外部用户如何被自动桥接到 wunder 用户与智能体
3. 快速排查失败路由、失败投递与 provider 能力问题

页面必须显式避免：

- 让管理员在这里直接编辑桥接用户智能体
- 让管理员在这里直接管理线程、定时任务、工作区
- 复制一套 `frontend` 的聊天/智能体控制界面到 `web`

---

## 2. 页面入口与导航

建议在管理员侧新增一级导航：

- `舰桥中心`

位置建议：

- 放在 `渠道监控` 附近
- 但作为独立面板，不与渠道账号列表直接混写在同一页

原因：

- 它依赖渠道能力，但治理对象不是单条渠道账号，而是“共享渠道 -> 外部身份 -> wunder 用户/agent”这条桥接链路。

---

## 3. 信息架构

建议采用现有管理端最稳的布局：

- 左侧列表
- 右侧详情
- 详情内使用 tab

页面分为三层对象：

| 层级 | 对象 | 页面职责 |
|---|---|---|
| L1 | `bridge_center` | 展示中心列表与概览 |
| L2 | `bridge_center_account` | 展示共享账号绑定与 provider 状态 |
| L3 | `bridge_route` | 展示外部身份发现、路由命中、最近投递 |

---

## 5. Tab 设计

## 5.1 `概览`

目标：

- 一眼看懂这个中心当前是否健康
- 一眼看懂接了哪些 provider
- 一眼看懂最近有没有新增外部用户和失败投递

建议模块：

### 概览卡片

- 中心状态
- 默认预设智能体
- 共享账号数
- 活跃路由数
- 最近 24h 入站数
- 最近 24h 出站失败数

### 最近事件流

- 自动开户成功
- 自动确保默认预设成功
- provider 回复失败
- 路由被暂停/封禁

### 风险提示区

- 默认预设不存在
- 某共享账号缺少凭证
- provider adapter 未注册
- 某共享账号禁止主动出站但当前策略要求主动出站

---

## 5.2 `共享账号`

目标：

- 管理中心挂了哪些共享渠道账号
- 看见每个账号当前 provider 能力与运行态
- 不在这里编辑账号凭证本身，只做挂接与策略覆盖

列表字段建议：

- 渠道 provider
- `account_id`
- 运行状态
- 是否启用
- 身份提取策略
- 回复策略
- 默认预设覆盖值
- 最近活跃时间

操作建议：

- `[绑定共享账号]`
- `[编辑挂接策略]`
- `[暂停接入]`
- `[解除挂接]`
- `[查看渠道详情]`

不提供：

- 直接编辑 `channel_accounts.config` 大表单

如果管理员要改真实凭证，应跳转现有渠道账号页面。

原型示意：

```text
┌─ 共享账号 ────────────────────────────────────────────────────────────┐
│ [绑定共享账号]                                                       │
├─────────┬────────────────────┬────────┬────────────┬───────────────┤
│ provider│ account_id         │ 状态   │ 身份策略   │ 最近活跃       │
├─────────┼────────────────────┼────────┼────────────┼───────────────┤
│ xmpp    │ shared_xmpp_01     │ running│ sender_in… │ 15:32          │
│ weixin  │ service_bot_a      │ running│ platform…  │ 15:30          │
│ feishu  │ cs_entry           │ paused │ sender_in… │ 14:11          │
└─────────┴────────────────────┴────────┴────────────┴───────────────┘
```

## 5.3 `用户发现`

这是页面核心 tab。

它展示的不是“用户管理列表”，而是“桥接路由列表”。

列表字段建议：

- 外部显示名
- provider
- 外部身份键
- wunder 用户
- 目标智能体
- 路由状态
- 首次接入时间
- 最近消息时间
- 最近错误

筛选建议：

- provider
- 共享账号
- 路由状态
- 搜索词（外部显示名 / wunder 用户 / agent）

行操作建议：

- `[查看路由详情]`
- `[暂停]`
- `[恢复]`
- `[封禁]`
- `[查看用户]`
- `[查看线程]`

原型示意：

```text
┌─ 用户发现 / 路由 ────────────────────────────────────────────────────┐
│ provider[全部▼]  状态[全部▼]  搜索[张三________]                    │

│渠道  │外部名  │ wunder 用户           │ agent      │状态    │最近消息│
├──────┼────────┼──────────────────────┼────────────┼────────┼───────┤
```

## 5.4 `路由详情抽屉`

点击某条路由后，右侧弹出详情抽屉，而不是跳大页。

建议内容：

### 基础信息

- `route_id`
- provider / account_id
- 外部身份键
- 外部显示名
- 路由状态

### 外部身份快照

- `peer_id`
- `sender_id`
- `thread_id`
- provider 原始 profile 摘要

### wunder 映射

- `wunder_user_id`
- `agent_id`
- `agent_name`
- 是否自动开户
- 是否自动创建 agent

### 最近活动

- 最近 session
- 最近入站
- 最近出站
- 最近错误

### 操作区

- `[暂停路由]`
- `[恢复路由]`
- `[封禁外部身份]`
- `[查看用户详情]`
- `[查看线程详情]`

明确不放：

- `编辑智能体`
- `切换预设`
- `改 system prompt`
- `管理 cron`
- `浏览 workspace`

这些按钮不该出现在这个抽屉里。

---

## 5.5 `投递日志`

目标：

- 看最近的入站/出站行为
- 快速定位 provider 失败、context_token 缺失、adapter 不可用等问题

列表字段建议：

- 时间
- provider
- 方向（入站/出站）
- route_id
- session_id
- 状态
- 阶段
- 摘要

点击日志行后，可展开：

- provider message id
- payload 摘要
- 错误正文

原型示意：

```text
┌─ 投递日志 ───────────────────────────────────────────────────────────┐
│ 方向[全部▼] 状态[失败▼] 渠道[全部▼] 搜索[context_token____]         │
├────────────┬───────┬─────────┬──────────┬──────────┬───────────────┤
│时间        │方向   │provider │状态      │阶段      │摘要            │
├────────────┼───────┼─────────┼──────────┼──────────┼───────────────┤
│15:12:33    │outbound│weixin  │failed    │outbox    │context_token…  │
│15:12:09    │inbound │qqbot   │ok        │resolved  │route matched   │
└────────────┴───────┴─────────┴──────────┴──────────┴───────────────┘
```

---

## 5.6 `运行日志`

这里展示的是共享账号级运行日志，而不是 route 粒度日志。

适合排查：

- 长连接未启动
- webhook 验签失败
- 凭证缺失
- adapter 未注册
- 配置不完整

建议字段：

- 时间
- 账号
- provider
- 事件类型
- 事件摘要

这部分最好和现有渠道监控日志的 UI 风格保持一致。

## 6. 创建 / 编辑流程原型

## 6.1 新建中心抽屉

字段建议：

- 中心名称
- 稳定编码 `code`
- 默认预设智能体
- 默认单位
- 默认身份提取策略
- 用户名策略
- 描述

原型示意：

```text
┌─ 新建舰桥中心 ─────────────────────────────┐
│ 名称                 [客服总入口_________] │
│ 编码                 [service_hub_______] │
│ 默认预设智能体       [客服助手▼]           │
│ 默认单位             [客服部▼]             │
│ 默认身份提取策略     [sender_in_peer▼]     │
│ 用户名策略           [namespaced_generated]│
│ 描述                 [..................] │
│                               [取消][创建]│
└───────────────────────────────────────────┘
```

## 6.2 绑定共享账号抽屉

字段建议：

- provider
- 账号 ID（从现有 `channel_accounts` 选择）
- 身份提取策略覆盖
- 默认预设覆盖
- 回复策略

关键约束：

- 这里只允许选择已有共享渠道账号；
- 不直接在这里编辑 provider 凭证；
- 若账号已被其他中心占用，直接给出冲突提示。

---

## 7. 视觉与交互建议

由于这是管理员侧页面，建议遵循现有管理端语言：

- 信息优先，不做聊天式视觉；
- 左侧列表 + 右侧详情的稳定布局；
- 状态用 badge 明确标出：`active / paused / blocked / error`；
- 风险项统一放顶部警示区；
- 日志与路由列表支持快速过滤。

前端实现上应注意：

- 使用 `web/modules/bridge-center.js` 独立承载逻辑；
- 不往 `web/app.js` 里继续堆大段逻辑；
- 不使用 `backdrop-filter`；
- 不做重型动画；
- 优先保证大量日志、列表场景下的可读性和响应速度。

---

## 8. 明确不提供的页面能力

为避免边界失控，页面上明确不提供：

- “编辑该用户智能体”按钮
- “切换该用户预设”按钮
- “查看/修改 system prompt”面板
- “管理定时任务”面板
- “查看工作区文件”面板
- “会话聊天模拟器”

如果管理员真的需要这些能力，应通过：

- `[查看用户详情]`
- `[查看线程详情]`
- `[查看渠道详情]`

跳转到既有页面，而不是在舰桥中心复制一套功能。

---

## 9. 最终建议

这个页面做得优雅与否，关键不在组件多少，而在是否守住边界：

- 它是桥接治理页，不是用户操作台；
- 它处理共享渠道，不处理个人凭证；
- 它显示 route，不直接编辑 agent；
- 它负责观测与排障，不负责替代现有用户页和线程页。

只要这个边界守住，舰桥中心页面就会清晰、稳定、长期可维护。
