# wunder 系统异常处理优化方案

## 1. 文档目标

本文基于 wunder 当前实现，并参考 `C:\Users\32138\Desktop\参考项目\codex-rust-v0.113.0` 中 codex-rs 的异常处理设计，给出一份面向长期运行的系统异常处理优化方案。分析重点不是“多补几个错误码”，而是把异常处理从零散分支提升为系统级边界能力，确保 wunder 在未来 10 年内仍能满足以下要求：

- 异常发生后线程仍然可恢复、可继续、可追踪，而不是只能重建新会话。
- 失败路径与成功路径一样有明确生命周期，不留下脏状态、挂起审批、僵尸 turn 或锁泄漏。
- server、cli、desktop 三种运行形态共享同一套异常语义，不因接入方式不同而行为漂移。
- 对外协议、前端展示、运维监控、内部调度共用同一套异常分类与恢复语义，减少层间歧义。

本文重点对照以下代码与测试：

- wunder：
  - `src/orchestrator/error.rs`
  - `src/api/errors.rs`
  - `src/orchestrator/execute.rs`
  - `src/orchestrator/llm.rs`
  - `src/orchestrator/event_stream.rs`
  - `src/api/chat_ws.rs`
  - `src/services/tools/tool_error.rs`
  - `src/services/llm.rs`
  - `src/ops/monitor.rs`
- codex-rs：
  - `codex-rs/app-server/README.md`
  - `codex-rs/core/src/state/turn.rs`
  - `codex-rs/core/src/tools/orchestrator.rs`
  - `codex-rs/app-server/src/thread_status.rs`
  - `codex-rs/core/tests/suite/stream_error_allows_next_turn.rs`
  - `codex-rs/app-server/tests/suite/v2/turn_interrupt.rs`
  - `codex-rs/app-server/tests/suite/v2/thread_unsubscribe.rs`

## 2. 核心结论

### 2.1 wunder 已有基础并不差

wunder 当前并不是“没有异常处理”，而是已经有几块不错的基础，但尚未统一成系统边界：

- `src/api/errors.rs` 已经提供了较完整的 HTTP 错误包，包含 `code/message/status/hint/trace_id/timestamp`，适合作为对外统一格式基础。
- `src/services/tools/tool_error.rs` 已经为工具失败引入了 `error_meta`，包含 `code/hint/retryable/retry_after_ms`，说明工具层已经开始表达“可否重试”和“如何恢复”。
- `src/orchestrator/llm.rs` 与 `src/services/llm.rs` 已经有流式失败识别与重试逻辑，尤其对上下文超限、SSE 错误解析、provider 差异做了不少兼容。
- `src/orchestrator/event_stream.rs` 已经把 `error/final/tool_result/approval_request/approval_result` 等事件纳入持久化范围，说明系统已有“异常也是事件”的方向。
- `src/ops/monitor.rs` 已经能追踪取消、阶段变化、轮次与部分资源状态，为异常可观测性留出了入口。

这些都是值得保留的基础。

### 2.2 主要问题不在“缺少 if/else”，而在异常所有权不清晰

与 codex-rs 对照后，wunder 当前异常处理的核心问题主要有四个：

1. `OrchestratorError` 过于扁平，无法表达异常来源、严重级别、可恢复性、重试策略、清理动作。
2. 审批、取消、等待输入等“异常相关临时状态”仍偏向传输层持有，尤其 `src/api/chat_ws.rs` 中 `pending_approvals` 归属于连接，而不是 turn。
3. 缺少显式的线程运行时状态机，异常后“谁负责清理 active turn、谁负责切回 idle、谁负责释放等待项”没有收拢到单一所有者。
4. 缺少被测试固化的失败语义，例如“流式失败后下一轮必须还能继续”“interrupt 必须 resolve 掉待审批项”“unsubscribe 后线程必须卸载且清理 system_error 缓存”。

### 2.3 codex-rs 做得好的不是“错误提示更漂亮”，而是失败路径有确定的收尾

codex-rs 的关键设计不在于错误文案，而在于以下几条稳定性原则已经落成代码与测试：

- turn 是异常相关临时状态的唯一所有者，见 `codex-rs/core/src/state/turn.rs`。
- thread 的 loaded/notLoaded/systemError/active/idle 是显式状态，不靠调用方猜测，见 `codex-rs/app-server/src/thread_status.rs`。
- 流式失败后必须发出终结性 turn 事件并释放 active turn，下一轮继续可用，见 `stream_error_allows_next_turn.rs`。
- 中断不仅要停掉执行，还要 resolve pending approval/request，见 `turn_interrupt.rs`。
- unsubscribe 不只是断连接，而是 unload thread，并清空运行态缓存，见 `thread_unsubscribe.rs`。

这正是 wunder 下一阶段要对齐的重点。

## 3. 当前现状分析

## 3.1 wunder 当前异常模型的优点与短板

### 3.1.1 API 对外错误包已经相对成熟

`src/api/errors.rs` 已经把对外错误响应统一成：

- `code`
- `message`
- `status`
- `hint`
- `trace_id`
- `timestamp`

这个方向是正确的，因为它已经区分了：

- 面向用户和调用方的错误码
- 面向排障的 trace_id
- 面向恢复动作的 hint

问题在于，这套结构主要存在于 API 层，尚未上溯到 orchestrator、tool runtime、thread runtime 内部。因此系统内部虽然能报错，但异常语义仍然分散。

### 3.1.2 OrchestratorError 仍然过平

`src/orchestrator/error.rs` 当前的 `OrchestratorError` 只有：

- `code`
- `message`
- `detail`

目前已有 `INVALID_REQUEST / USER_BUSY / CANCELLED / LLM_UNAVAILABLE / CONTEXT_WINDOW_EXCEEDED / INTERNAL_ERROR / USER_QUOTA_EXCEEDED` 等构造函数，但它仍然缺失几个决定系统健壮性的字段：

- `category`：输入错误、策略拒绝、provider 错误、工具错误、传输错误、内部错误。
- `severity`：info / warning / error / critical。
- `retryable`：能否自动重试，还是必须用户修正后重试。
- `retry_after_ms`：用于过载、限流、依赖冷却。
- `source_stage`：发生在请求解析、turn 启动、LLM 流、tool 执行、审批等待、状态持久化哪个阶段。
- `recovery_action`：建议动作是 compact、retry、interrupt cleanup、thread unload、provider fallback 还是直接 fail-fast。

没有这些字段，会带来两个问题：

- orchestrator 只能“知道失败了”，却不能统一驱动后续收尾策略。
- monitor 和前端只能看到表面 message，无法稳定区分“暂时失败”和“线程已损坏”。

### 3.1.3 工具错误比编排错误更成熟，但两套语义还没有合并

`src/services/tools/tool_error.rs` 已经支持：

- `code`
- `hint`
- `retryable`
- `retry_after_ms`

这说明工具层已经具备更好的恢复表达能力。但当前问题是：

- tool error 和 orchestrator error 没有同构。
- API 层错误包与 tool error 也没有同构。
- 不同层都能表达一部分恢复语义，但字段不统一，事件名也不统一。

结果是：

- 同一类失败，在工具事件、最终错误响应、监控记录中可能变成三种风格。
- 前端和外部调用方很难建立稳定的自动恢复逻辑。

### 3.1.4 审批等待状态的所有权仍偏连接层

`src/api/chat_ws.rs` 当前以 `pending_approvals` 维护待审批项，并在连接断开或 cancel 时执行 `clear_pending_approvals(...)`。这个做法短期可行，但从系统边界看是脆弱的：

- approval 是 turn 生命周期的一部分，不是 websocket 连接的一部分。
- 同一线程可能以后接入 websocket、SSE、桌面 IPC、CLI，本不应把等待态绑死在某个 transport 上。
- 一旦连接与线程运行态解耦不彻底，就容易出现“连接断了但 turn 仍在等”“换连接后拿不到旧 approval”“断开清理和 turn 收尾重复执行”的问题。

codex-rs 在 `core/src/state/turn.rs` 中把 pending approvals / request permissions / user input / elicitation / dynamic tools 全部放进 `TurnState`，这比当前 wunder 的边界更稳。

### 3.1.5 事件流已持久化，但缺少“终结事件唯一真相”

`src/orchestrator/event_stream.rs` 已经将 `error` 和 `final` 这类事件持久化，这是好事。但当前仍有一个语义空缺：

- 哪个事件才是“一轮执行已终结”的权威结论？

如果系统同时可能发：

- `error`
- `final`
- `approval_result`
- `tool_result`
- 传输结束

那么前端、monitor、resume 逻辑仍然需要自己猜“这轮是不是已经结束”。这会直接影响：

- 是否能安全开始下一轮
- 是否应该释放 active turn
- 是否应保留 waiting 状态
- resume 后是否应继续回放还是重建

codex-rs 用 `turn/completed` 作为权威终结事件，这是一个很关键的稳定性设计。

## 3.2 与 codex-rs 的关键差异

### 3.2.1 codex-rs 把失败状态建模成 runtime 状态机

`codex-rs/app-server/src/thread_status.rs` 中的 thread runtime 状态能明确表达：

- `notLoaded`
- `idle`
- `active`
- `systemError`

同时它还会追踪是否有 pending permission request、pending user input request。也就是说，thread status 不是简单的“忙/闲”，而是运行态概括。

wunder 当前虽然在 monitor 里有 `cancelled/cancelling/approval_pending` 等阶段，但它们更偏观测字段，还不是 thread runtime 的正式语义。

### 3.2.2 codex-rs 的 turn 是审批、输入、动态请求的唯一收口点

`codex-rs/core/src/state/turn.rs` 中 `TurnState` 管的不是单一 approval，而是整轮执行的等待面：

- pending approval
- pending request_permissions
- pending user_input
- pending elicitation
- pending dynamic_tools
- pending_input

并提供 `clear_pending()` 统一收尾。这种设计的价值是：

- interrupt 时不需要逐模块猜谁还有 pending state。
- stream error 后可以一次性清理所有待决句柄。
- transport 断连、thread unload、system error 都可以共享同一收尾逻辑。

当前 wunder 还没有这样一个 turn-scoped state owner。

### 3.2.3 codex-rs 用测试固化失败恢复语义

codex-rs 的几个测试非常值得直接借鉴：

- `stream_error_allows_next_turn.rs`
  - 目标不是“正确提示错误”，而是“错误后 active turn 必须被释放，下一轮可以继续”。
- `turn_interrupt.rs`
  - 要求 interrupt 不只是停止执行，还要让 `turn/completed` 落成 `Interrupted`。
  - 若有审批请求挂起，还必须发出 `serverRequest/resolved`。
- `thread_unsubscribe.rs`
  - unsubscribe 后如果没有订阅者，线程应被 unload，并发出 `thread/closed`。
  - 如果线程处于 `systemError`，unsubscribe + resume 后要能回到 `Idle`，而不是保留脏缓存。

wunder 当前缺的不是“处理异常的 if 分支”，而是这类能保证 10 年后仍不漂移的回归矩阵。

### 3.2.4 codex-rs 的工具编排不是“执行失败就报错”，而是有审批、沙盒、升级重试链路

`codex-rs/core/src/tools/orchestrator.rs` 的执行顺序很值得借鉴：

1. 先判断是否需要 approval。
2. 再选择初始 sandbox。
3. 执行第一次 attempt。
4. 若是 sandbox denial 且策略允许，则根据原因再次申请 approval，并执行升级尝试。
5. 重试时不重复污染状态，approval 和 network policy 语义都清晰。

这套设计的价值在于，异常不是简单地“失败结束”，而是：

- 可预期地进入某种恢复分支。
- 恢复分支有独立错误语义。
- 用户拒绝、策略拒绝、沙盒拒绝、真正执行失败这几类情况被区分开。

wunder 当前的工具执行已经有 approval 和 timeout，但仍偏“流程内嵌”，缺少统一的工具异常编排器。

## 4. wunder 当前的主要缺口

## 4.1 缺少统一异常分类体系

目前 wunder 不同层的异常大致分成了三套：

- API 错误
- Orchestrator 错误
- Tool 错误

缺点是：

- 分类标准不同。
- 字段不同。
- 恢复语义不同。
- 对监控的暴露粒度不同。

这会导致“一个 provider 429”可能在不同路径下被包装成：

- `SERVICE_UNAVAILABLE`
- `LLM_UNAVAILABLE`
- `UPSTREAM_TIMEOUT`
- `INTERNAL_ERROR`

从运维、前端、自动重试角度看，这种不一致会直接伤害稳定性。

## 4.2 缺少 turn 级异常清理中心

当前取消、审批、工具超时、连接断开、流式失败等清理动作散落在：

- `src/orchestrator/execute.rs`
- `src/api/chat_ws.rs`
- `src/orchestrator/llm.rs`
- `src/services/llm.rs`

这意味着异常清理是“路径依赖”的：

- 走 A 分支会清理 approval。
- 走 B 分支会清理 cancel。
- 走 C 分支会发 error 但未必归还 active state。

这种结构在规模扩大后很难长期稳定。

## 4.3 缺少 loaded thread 与 active turn 的分层

目前 wunder 以 `session_id` 为主索引是可行的，但需要进一步拆出三层概念：

- 持久线程：存历史、配置、工作区、冻结提示词。
- 加载线程：驻留内存、可订阅、可恢复、可卸载。
- 活跃 turn：当前这轮正在跑的 LLM/tool/approval/input/cancel 状态。

没有这三层分离，异常后容易出现：

- 线程本身没坏，但当前 turn 卡死，用户却只能重建线程。
- transport 已断开，但 loaded thread 仍残留 waiting 状态。
- system error 被错误缓存到线程级，导致 resume 后仍像“坏线程”。

## 4.4 缺少协议级 retryable/backpressure 语义

codex-rs 在 `app-server/README.md` 中明确规定：

- 当 ingress 饱和时，返回 JSON-RPC `-32001`。
- 客户端应视为 retryable，并采用 backoff + jitter。

wunder 当前已经有 `USER_BUSY`、`RATE_LIMITED`、`USER_QUOTA_EXCEEDED` 等语义，但系统缺少一套覆盖以下情况的统一过载语义：

- 单用户/单线程忙
- 全局队列饱和
- provider 429
- 工具池拥塞
- 写入队列拥塞
- 审批等待资源不足

没有统一的 retryable/backpressure 语义，就无法把“异常处理”提升为“系统弹性控制”。

## 5. 目标设计

## 5.1 统一异常模型

建议在 orchestrator 层引入统一异常模型 `RuntimeError`，并逐步替代当前分散的包装方式。核心字段建议如下：

- `code`
- `category`
- `severity`
- `message`
- `public_message`
- `hint`
- `detail`
- `retryable`
- `retry_after_ms`
- `source_stage`
- `recovery_action`
- `trace_id`
- `session_id`
- `thread_id`
- `turn_id`
- `item_id`
- `provider`
- `tool_name`
- `timestamp`

其中建议的 `category` 至少包括：

- `input`
- `auth`
- `quota`
- `policy`
- `approval`
- `tool`
- `provider`
- `stream`
- `transport`
- `storage`
- `state`
- `internal`
- `recovery`
- `overload`

建议的 `severity`：

- `info`
- `warning`
- `error`
- `critical`

建议的 `recovery_action`：

- `none`
- `retry_same_turn`
- `retry_next_turn`
- `compact_context`
- `await_user_action`
- `resolve_pending_and_interrupt`
- `unload_thread`
- `fallback_provider`
- `rebuild_runtime`
- `fail_fast`

这样做后，API 错误、tool error、event error、monitor error 都应基于同一模型投影，而不是各自定义一套字段。

## 5.2 区分三层状态：持久线程、加载线程、活跃 turn

建议明确引入以下状态边界：

### 5.2.1 PersistedThread

负责：

- 历史 turn/items
- 冻结 system prompt 基线
- 线程级配置快照
- 记忆注入快照
- 工作区绑定与目录元数据

不负责：

- 当前有没有 pending approval
- 当前是不是正在流式输出
- transport 是否在线

### 5.2.2 LoadedThreadRuntime

负责：

- 当前线程是否已装载到内存
- 当前 thread runtime 状态
- 当前订阅计数
- 当前是否存在 active turn
- 是否需要 lazy refresh 配置

建议状态：

- `not_loaded`
- `idle`
- `running`
- `waiting_approval`
- `waiting_user_input`
- `interrupting`
- `system_error`

注意：

- `cancelled` 不建议作为 thread runtime 常驻状态，它更适合作为 turn terminal status。
- `system_error` 表示“当前 loaded runtime 不可信，需要 rebuild/unload”，而不是“持久线程数据不可用”。

### 5.2.3 ActiveTurnState

负责：

- pending approvals
- pending request_user_input
- pending tool continuations
- cancellation token
- timeout handles
- deferred cleanup actions
- 当前轮异常上下文

它应成为异常清理的唯一所有者。

## 5.3 turn 终结语义必须唯一

建议为 wunder 增加明确的 turn 终结事件语义。即使不照抄 codex 的 `turn/completed` 命名，也必须有一个唯一权威终态事件，例如：

- `turn_terminal`

该事件至少包含：

- `thread_id/session_id`
- `turn_id`
- `status`
- `error`
- `retryable`
- `cleanup_applied`
- `pending_resolved_count`
- `usage`

建议的 `status`：

- `completed`
- `failed`
- `cancelled`
- `interrupted`
- `rejected`

系统规则应为：

- `error` 事件只表达诊断，不表达生命周期结束。
- 只有 `turn_terminal` 才表示这轮已经正式结束。
- 任何路径离开 active turn 前，都必须先发出终结事件，再释放 turn 资源。

## 5.4 审批/输入/动态请求必须从 transport 迁移到 turn state

建议把 `src/api/chat_ws.rs` 中当前连接级 `pending_approvals` 的真正所有权迁移到 turn 层：

- transport 只负责转发请求和回传决策。
- request registry 归 `ActiveTurnState` 持有。
- 连接断开时，不是直接“自己清理一遍”，而是向 turn runtime 发起 `transport_detached`。
- turn runtime 决定：
  - 若该 turn 仍有其它订阅者，则保持等待。
  - 若无订阅者且策略要求回收，则 resolve 为 deny/abort 并终结 turn。
  - 若线程改为 unload，则统一清理所有 pending。

这样可以让 websocket、SSE、CLI、桌面 IPC 共用同一等待/清理语义。

## 5.5 工具异常统一收口

建议将工具异常统一为四层语义：

1. `approval_rejected`
2. `sandbox_denied`
3. `tool_runtime_failed`
4. `tool_output_invalid`

并要求所有 mutating tool 在异常结果中明确：

- 是否已产生副作用
- 副作用是否完整
- 是否存在补偿动作
- 是否允许自动重试

例如：

- `write_file` 半写入
- `apply_patch` 部分命中
- `shell_command` 已启动但被中断
- `browser` 请求已发出但响应不完整

对用户来说，这比单纯返回“工具失败”更重要，因为它直接决定下一步是重试、回滚还是人工确认。

同时建议引入 mutating tool gate：

- 读操作可并行。
- 写操作默认串行或按工作区资源锁分组。
- 避免多个工具在同一轮并行修改同一文件或目录，异常时更难收尾。

## 5.6 过载与退避语义统一化

建议新增统一过载错误分类：

- `USER_BUSY`
- `THREAD_BUSY`
- `SERVER_OVERLOADED`
- `TOOL_POOL_SATURATED`
- `PROVIDER_RATE_LIMITED`
- `PROVIDER_OVERLOADED`

这些错误都应至少带：

- `retryable = true`
- `retry_after_ms`
- `backoff_policy`

这样前端、CLI、桌面端都能统一做自动退避，而不是把所有错误都当成“用户自己再点一次”。

## 6. 关键异常场景优化方案

## 6.1 流式失败后下一轮必须继续可用

参考 `stream_error_allows_next_turn.rs`，wunder 应明确以下规则：

1. LLM stream 出错后，必须写入本轮失败终结事件。
2. 必须清空 active turn。
3. 必须 resolve/close 所有本轮 pending request。
4. 必须释放 session/thread 运行锁。
5. 必须允许同一线程直接开始下一轮，而不是要求新建线程。

当前 wunder 已经对 stream error 有识别和一定重试，但还需要把“释放 active turn 并允许下一轮”写成强约束和回归测试。

## 6.2 interrupt 必须先清理再结束

参考 `turn_interrupt.rs`，wunder 的 interrupt 目标不应只是“尽量打断执行”，而应是：

- tool/llm/cancel token 被触发
- 所有 pending approval / user input 统一 resolve 为终止态
- 发出 `approval_resolved` 或等价事件
- 产出本轮 `interrupted` 终结事件
- runtime 切回 `idle`

否则就会出现“看上去停了，但审批项还挂着”“下一轮仍提示忙碌”的典型脏状态。

## 6.3 unsubscribe/断线不应只是连接层行为

参考 `thread_unsubscribe.rs`，建议将 wunder 的“断开订阅/断线”语义升级为：

- transport detach
- thread unsubscribe
- thread unload

三层概念：

- `transport detach`：仅连接断开。
- `thread unsubscribe`：当前观察者主动不再订阅线程。
- `thread unload`：最后一个订阅者离开且无线程级驻留要求时，清除 loaded runtime。

这样才能做到：

- 连接断了但线程仍可由其它客户端继续观察。
- 最后一个观察者离开时自动卸载，降低内存占用。
- `system_error` 缓存不再长期粘在线程运行态里。

## 6.4 system_error 要定义清楚边界

建议明确：

- `failed turn` 不等于 `system_error`。
- `system_error` 只在 runtime 自身状态不再可信时使用，例如：
  - active turn 清理失败
  - pending registry 状态损坏
  - runtime 与存储快照冲突
  - 内部 panic 恢复后状态不确定

普通 provider 500、tool timeout、用户取消都不应把 thread 置为 `system_error`。

一旦进入 `system_error`，恢复动作应是固定的：

1. 拒绝继续在当前 loaded runtime 上执行新 turn。
2. 允许 `resume/rebuild` 或 `unsubscribe -> unload -> reload`。
3. rebuild 成功后切回 `idle`。

## 6.5 审批请求必须有 resolved 语义

参考 codex 的 `serverRequest/resolved`，wunder 建议新增等价事件，例如：

- `approval_resolved`
- `user_input_resolved`

它们不只是为了前端 UI，而是为了把“所有挂起请求最终一定闭环”变成系统约束。无论请求是：

- 正常被批准
- 被拒绝
- 因 interrupt 终止
- 因 disconnect/unload 被清理

都必须有 resolved 事件和终态原因。

## 7. 建议的实现落点

## 7.1 异常模型

建议新增：

- `src/orchestrator/runtime_error.rs`
- `src/orchestrator/error_codes.rs`
- `src/orchestrator/recovery.rs`

并逐步让以下模块只依赖统一错误模型：

- `src/orchestrator/execute.rs`
- `src/orchestrator/llm.rs`
- `src/services/tools/*`
- `src/api/errors.rs`

`src/orchestrator/error.rs` 可以保留一段时间作为兼容入口，但不再继续扩展。

## 7.2 turn 状态所有权

建议新增：

- `src/orchestrator/turn_state.rs`

负责收口：

- pending approvals
- pending user inputs
- cancel token
- deferred cleanup
- terminal status
- turn trace metadata

`src/api/chat_ws.rs` 只保留转发桥接逻辑，不再拥有真正的等待态。

## 7.3 thread 运行时状态机

建议新增：

- `src/orchestrator/thread_runtime.rs`
- `src/orchestrator/thread_watch.rs`

负责：

- loaded/unloaded 状态
- subscriber count
- active turn registry
- waiting/running/system_error 状态切换
- unload/rebuild 入口

## 7.4 工具异常编排器

建议新增：

- `src/services/tools/runtime/`
- `src/services/tools/runtime/orchestrator.rs`
- `src/services/tools/runtime/error.rs`
- `src/services/tools/runtime/policy.rs`

逐步把当前分散在 `execute.rs` 中的 approval + timeout + retry + mutating gate 逻辑迁入工具编排器，避免继续膨胀 orchestrator 主流程。

## 7.5 事件协议

建议在 `src/orchestrator/event_stream.rs` 增补并固化以下事件：

- `turn_terminal`
- `approval_resolved`
- `user_input_resolved`
- `thread_status`
- `thread_closed`
- `runtime_error`

其中：

- `runtime_error` 用于诊断。
- `turn_terminal` 用于生命周期闭环。
- `thread_status/thread_closed` 用于 loaded runtime 边界。

## 8. 分阶段实施建议

## 8.1 Phase 0：先统一 taxonomy，不急着大改行为

目标：

- 把 API、orchestrator、tool 的错误字段统一。
- 建立 `RuntimeError` 和标准错误码。
- monitor 与 event stream 开始记录 `category/severity/retryable/recovery_action`。

这一阶段收益最大、风险最低，因为大部分是结构统一，不需要大动执行链路。

## 8.2 Phase 1：建立 ActiveTurnState，迁移 pending ownership

目标：

- 新增 `turn_state.rs`。
- approval/user input/pending request 改为 turn 持有。
- chat_ws 改为桥接层。
- interrupt、cancel、stream fail 统一走 `turn.cleanup_terminal(...)`。

这是异常处理从“散落逻辑”变成“中心所有权”的关键阶段。

## 8.3 Phase 2：引入 LoadedThreadRuntime 与 unload/rebuild 机制

目标：

- 明确 thread loaded status。
- 引入 subscriber count 和 unload 策略。
- 分离 failed turn 与 system_error。
- 支持 unload 后清理 stale runtime，并允许 resume/rebuild。

这一阶段完成后，wunder 的异常边界会从“请求级处理”升级为“线程运行时处理”。

## 8.4 Phase 3：把工具异常编排从 execute.rs 中拆出

目标：

- 引入 tool runtime orchestrator。
- 实现 mutating gate。
- 统一 approval/sandbox/retry/timeout 语义。
- 为副作用型工具补充 partial effect 标记。

这一阶段直接提升异常时的稳定性和可解释性。

## 8.5 Phase 4：协议与监控强化

目标：

- 增加 `turn_terminal/approval_resolved/thread_status/thread_closed` 事件。
- 引入 overload/backpressure 统一错误。
- 前端、desktop、cli 对 retryable/backoff 做统一处理。
- monitor 看板增加异常分类和恢复动作维度。

## 8.6 Phase 5：测试与混沌回归

目标：

- 把 codex-rs 那种“异常后系统仍可继续”的场景转成 wunder 集成测试。
- 对 websocket 断连、provider 500、tool timeout、审批中断做稳定回归。
- 增加随机故障注入，验证 active turn 释放和 pending 清理。

## 9. 测试矩阵建议

必须新增的回归用例至少包括：

1. `stream_error_then_next_turn_succeeds`
   - 第一轮流式失败。
   - 第二轮在同一线程正常成功。
2. `interrupt_resolves_pending_approval`
   - approval 发出后 interrupt。
   - pending request 被 resolve。
   - turn 终态为 `interrupted`。
3. `disconnect_does_not_leave_pending_approval_leak`
   - transport 断开。
   - turn cleanup 或 unload 之后无 dangling approval。
4. `unsubscribe_unloads_thread_runtime`
   - 最后一个订阅者离开。
   - thread runtime 进入 `not_loaded`。
5. `system_error_can_rebuild_to_idle`
   - 人工注入 runtime 状态损坏。
   - rebuild 后恢复 `idle`。
6. `provider_rate_limit_returns_retryable_error`
   - 429 明确返回 `retryable = true` 和 `retry_after_ms`。
7. `tool_timeout_clears_active_turn_state`
   - 工具超时后 turn 结束。
   - 不保留 active turn 与等待态。
8. `parallel_mutating_tools_are_serialized`
   - 同轮多个写工具不会并发踩踏同一资源。
9. `approval_resolution_emitted_once`
   - 无论 approve/deny/interrupt/unload，resolved 事件只能发一次。
10. `panic_in_runtime_marks_system_error_and_prevents_dirty_continue`
   - 内部 panic 恢复后不会带脏状态继续执行。

## 10. 验收指标

建议将异常处理改造的验收指标量化，而不是只看“用户说好像稳定了”：

- 流式失败后同线程下一轮成功率 `>= 99%`
- interrupt 后 1 秒内 pending request 泄漏数为 `0`
- tool timeout 后 active turn 清理完成率 `>= 99.9%`
- `system_error` 进入后未经 rebuild 的继续执行次数为 `0`
- 过载场景中含 `retryable + retry_after_ms` 的错误覆盖率 `>= 95%`
- 线程 unload 后内存驻留线程数与订阅数偏差长期接近 `0`
- 所有终结状态均有唯一 `turn_terminal` 事件覆盖率 `= 100%`

## 11. 最终建议

对 wunder 而言，下一阶段异常处理优化最重要的不是“继续补更多错误码”，而是完成三件事：

1. 建立统一异常 taxonomy，把 API、orchestrator、tool、monitor 的错误语义打通。
2. 让 turn 成为审批、输入、取消、清理的唯一所有者，把异常收尾从 transport 中剥离出来。
3. 建立 loaded thread runtime 状态机，把失败恢复、interrupt、unsubscribe、system_error 全部纳入明确边界。

如果只做第一步，文案会更整齐，但系统仍然可能卡死。
如果只做第二步，局部会稳定，但线程级状态仍然混乱。
如果只做第三步，没有统一异常模型，前端和运维仍然无法稳定理解系统状态。

因此建议按“taxonomy -> turn ownership -> thread runtime -> tool orchestrator -> protocol/tests”的顺序推进。这样改造风险最低，但对稳定性、恢复性、兼容性和运行效率的提升最大，也最符合 wunder 当前架构继续演进的方向。
