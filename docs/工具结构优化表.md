# 工具结构优化表

## 目标

本表用于收敛 wunder 的模型侧工具结构。  
设计取向遵循以下原则：

- 保留“成套多动作工具”风格，不强行拆散。
- 模型侧接口优先简洁、稳定、低歧义。
- 运行时兼容层、历史别名、后台管理字段不直接暴露给模型。
- 同一个工具允许动作多，但每个动作的最小必填和下一步语义必须清楚。

## 总原则

| 项 | 现状问题 | 优化原则 |
| --- | --- | --- |
| 工具形态 | 多动作工具里混入过多兼容字段、历史别名、后台字段 | 保留多动作，但只向模型暴露 canonical 参数 |
| schema 体积 | `description` 过长，`schema` 过深，`allOf/anyOf` 较多 | 模型侧优先“短 schema + 短规则”，运行时保留完整校验 |
| 动作表达 | 同一工具下不同动作的必填差异大，但模型不易抓住重点 | 每个动作都应有最小参数集、简短规则、示例 |
| 参数命名 | 同义参数并存，如 `sessionId/session_id/sessionKey` | 模型侧只保留 1 组名字，别名仅在后端解析层兼容 |
| 结果衔接 | 工具定义未明确下一步 follow-up 语义 | 在描述中明确“本动作完成后下一步通常调用什么” |

## 推荐结构

模型侧推荐采用三段式：

| 层 | 作用 | 内容 |
| --- | --- | --- |
| `name` | 稳定工具名 | 不使用历史别名，不混中文/英文多套名称 |
| `description` | 说明用途和动作规则 | 1 句用途 + 2~6 条动作约束 |
| `parameters` | 仅保留 canonical 参数 | 不暴露兼容别名、不暴露后台细节 |

不建议把以下内容直接暴露给模型：

- 历史兼容别名
- 纯前端字段
- 纯管理员字段
- 复杂预算对象的全部细节
- 与当前动作弱相关的大对象

## 多动作工具的推荐写法

既然倾向保留“一个工具一整套动作”，则建议统一写法如下：

| 项 | 推荐做法 |
| --- | --- |
| 动作入口 | 固定使用 `action` |
| 参数命名 | 仅暴露 1 套 canonical 字段名 |
| 动作约束 | 不依赖大段 `allOf/if/then` 才能读懂；在 `description` 里直接列出 |
| 示例 | 每个高频动作给 1 个短示例 |
| 后续动作 | 明确写出 “本动作后通常接 wait/status/history/none” |

推荐描述模板：

```text
用途：一句话说明这个工具负责什么。
动作：
- list: ...
- send: 需要 ...
- wait: 需要 ...
规则：
- ...
- ...
```

## 关于 `allOf / if / then`

| 问题 | 结论 |
| --- | --- |
| 是否必须 | 不是必须 |
| 对运行时校验是否有帮助 | 有一定文档价值，但不是唯一真源 |
| 对模型是否友好 | 通常不如直接的动作规则和短示例友好 |
| 推荐做法 | 运行时可保留；模型侧 schema 应尽量去掉，改成短规则描述 |

原因：

- 后端代码本来就在做真正的动作级校验。
- 模型更擅长理解“动作 + 必填字段 + 示例”，不擅长稳定消化长条件 schema。
- 在 `tool_call` 模式下，这些条件 schema 会直接膨胀系统提示词。

## 工具返回内容优化原则

工具返回给模型的内容，目标不是“完整还原后端世界”，而是“让模型知道这次调用是否成功、得到了什么、下一步该怎么做”。

| 项 | 反模式 | 推荐做法 |
| --- | --- | --- |
| 成功标识 | 同时返回 `ok=true`、`success=true`、`status=done/success/completed` | 只保留一套主信号，建议统一 `ok`，必要时补 `state` |
| 动作回显 | 返回整份原始请求体、兼容字段、默认值展开结果 | 只回显真正影响结果理解的 `action` 与关键目标标识 |
| 结果正文 | 同时给 `message/summary/result/output/data/detail` 多套近义字段 | 保留 1 个摘要字段 + 1 个结构化主结果字段 |
| 调试细节 | 默认回传内部路径、缓存命中、耗时拆分、SQL/调度细节 | 默认不返回；仅 debug 模式或管理员模式开放 |
| 大对象 | 把全文、全量列表、全量日志、整段 HTML/JSON 都塞回模型 | 只返回摘要、截断片段、计数和下一步可继续读取的锚点 |
| 错误表达 | 只有一段自然语言报错，或者把内部异常直接透出 | 返回结构化错误：`code/message/retryable`，必要时补 `missing_fields/example/next_step_hint` |
| 下一步指引 | 工具调用结束后没有说明后续如何继续 | 对需要串联的工具补 `next_step` 或 `next_step_hint` |
| 冗余计量 | 默认返回大量 token/耗时/预算/监控字段 | 仅保留对下一步决策有帮助的少量指标 |

## 推荐的通用返回骨架

推荐把大多数工具返回收敛为下面两类：

### 成功返回

```json
{
  "ok": true,
  "action": "xxx",
  "summary": "一句话说明本次调用得到的关键结果",
  "data": {},
  "next_step": null
}
```

### 失败返回

```json
{
  "ok": false,
  "action": "xxx",
  "error": {
    "code": "missing_fields",
    "message": "缺少必要参数",
    "retryable": true,
    "missing_fields": ["session_id"],
    "example": {
      "action": "wait",
      "session_id": "sess_xxx"
    },
    "next_step_hint": "先用 list/status 获取可等待目标"
  }
}
```

说明：

- `summary` 面向模型理解，应该短、准、可直接用于下一步推理。
- `data` 面向结构化消费，避免和 `summary` 重复。
- `next_step` 只在确实存在高概率后续动作时返回，不要滥用。
- 如果 `error.code` 已经足够明确，`message` 不要再写成长段解释。

## 工具返回内容优化表

| 工具/工具族 | 当前常见问题 | 推荐返回形态 | 应去掉或下沉的信息 | 模型侧收益 |
| --- | --- | --- | --- | --- |
| `final_response` | 有时会混入过多元信息、说明性包装 | 只保留最终给用户的内容本体 | 内部执行痕迹、工具统计、调度解释 | 减少模型把“最终回复”当成结构化结果继续处理 |
| `search_content` | 容易返回过多匹配、上下文和兼容字段 | `ok/summary/data.hits[]/data.truncated/data.next_hint` | 原始 rg 兼容参数回显、超长上下文、无关文件统计 | 让模型快速决定“继续读哪个文件、哪一段” |
| `read_file` | 容易把大段文本与附加元信息一起回传 | `ok/summary/data.files[]`，每项只含 `path/start_line/end_line/content/truncated` | 重复的路径解释、无关元数据、过长全文 | 降低上下文占用，鼓励模型精确继续读取 |
| `list_files` | 容易返回过多无用属性 | `ok/summary/data.entries[]/data.next_cursor`，每项只含 `name/type/path`，必要时补 `size` | 重复 parent path、低价值 stat 字段、兼容分页字段 | 模型更容易据此选择下一步读/搜 |
| `execute_command` / `ptc` | stdout/stderr/budget/命令计划容易一起堆回去 | `ok/summary/data.stdout/data.stderr/data.exit_code/data.truncated` | 预算解析细节、默认参数展开、重复命令回显 | 让模型围绕命令结果而不是工具内部预算推理 |
| `web_fetch` / `browser` | 容易回传整页文本、HTML、调试状态 | `ok/summary/data.title/data.url/data.excerpt/data.links?` | 完整 HTML、浏览器内部状态、缓存/回退细节 | 让模型优先消费页面结论和可继续探索的锚点 |
| `memory_manage` | list/recall 容易返回字段过多；写入动作容易回太啰嗦 | list/recall 返回 `items[]` 的精简视图；写入动作只回 `memory_id/summary` | 存储层内部字段、排序细节、重复正文镜像 | 模型更容易判断“引用哪条记忆/是否需要更新” |
| `schedule_task` | 任务对象层级深、状态字段多 | `ok/summary/data.job_id/data.state/data.schedule_text` | 完整内部 job 树、底层 deliver 结构、低频控制字段 | 模型只关注任务是否创建/更新成功，以及后续如何查状态 |
| `thread_control` / `subagent_control` / `智能体蜂群` | 返回内容最容易又大又乱，混入太多会话对象 | `ok/summary/data.session_id/run_id/dispatch_id/state/next_step` | 完整 session 快照、所有历史消息、内部派发细节 | 提高多智能体链路可纠错性，降低模型被大对象干扰 |
| `channel_tool` / `用户世界工具` | 目标对象与发送结果容易同时返回很多无用字段 | 查询类返回精简目标卡片；发送类只回投递结果与目标标识 | 渠道内部字段、冗余联系人属性、原始回包镜像 | 模型更容易继续“选人 -> 发消息 -> 等待” |
| `节点调用` | 容易把节点原始回包整段透出 | `ok/summary/data.node_id/data.command/data.result?` | 内部 metadata、大段调试信息、重复请求回显 | 保持节点工具结果简短直达 |
| `a2a观察` / `a2a等待` | 容易返回整份任务对象和轮询噪声 | `ok/summary/data.task_ids/data.state/data.completed/data.pending` | 全量任务明细、轮询过程日志、重复快照 | 模型能更快判断“继续等 / 改做别的” |
| `desktop_controller` / `desktop_monitor` | 默认返回过多桌面细节会污染上下文 | `ok/summary/data.action/data.target/data.observation?` | 低层鼠标键盘细节、内部截图/采样路径 | 模型聚焦动作是否成功和观察到什么 |
| `read_image` | 返回内容若太多会与真正视觉分析回合重复 | 只回 `ok/summary/data.path/data.mime_type/data.size_bytes` | 原始图片内容之外的重复解释、无关元信息 | 保持它只是“把图送进下一轮”，而不是提前替模型分析 |
| `LSP查询` | 容易混入过多诊断或协议层字段 | `ok/summary/data.items[]`，按 operation 返回最小结果集 | LSP 原始协议字段、低价值诊断细节 | 提高定位定义/引用/符号时的信噪比 |
| `self_status` | 调试价值高，但最容易把监控大对象灌给模型 | 默认只回精简状态；`full` 才放开更多字段 | 原始事件流、低层监控快照、对当前任务无帮助的系统指标 | 让它成为“诊断工具”而不是“上下文污染源” |
| `a2ui` / `计划面板` / `问询面板` | 客户端型工具如果回太多确认信息会显得啰嗦 | 只回 `ok/summary` 和必要标识，如 `panel_id/surface_id` | 前端渲染细节、重复 payload 镜像 | 防止模型把 UI 回显当成新的任务数据 |

## 返回内容硬约束

建议后续统一执行以下约束：

- 默认不要同时出现 `summary/message/result/output/detail` 这类近义字段。
- 默认不要回显完整请求参数，除非它们本身就是结果的一部分。
- 默认不要返回内部路径、缓存命中、数据库主键以外的实现细节。
- 默认不要把列表工具做成“全量导出”；要有 `truncated/next_cursor/next_hint`。
- 默认不要把观察类工具做成“流水账”；要输出当前结论而不是全过程。
- 默认不要把错误做成纯自然语言；要可纠错、可重试、可继续。

## 高优先级返回改造建议

| 优先级 | 工具 | 主要问题 | 推荐动作 |
| --- | --- | --- | --- |
| P0 | `subagent_control` | 返回对象过大，易混入会话与运行态噪声 | 统一成 `session_id/run_id/state/next_step` 主骨架 |
| P0 | `智能体蜂群` | 派发、等待、汇总结果混在一起 | 将 dispatch、wait、result 三类返回分别收敛成短骨架 |
| P0 | `search_content` | 命中结果和附加统计过多 | 强制只保留命中片段、截断标志、下一步建议 |
| P1 | `read_file` | 文件正文和元信息比例失衡 | 统一单文件片段骨架，弱化无关元数据 |
| P1 | `browser` / `web_fetch` | 容易把整页或完整 HTML 塞回模型 | 以摘要、标题、摘录、可继续访问锚点为主 |

| P1 | `channel_tool` / `用户世界工具` | 联系人和消息结果字段偏重 | 查询与发送返回分开收敛 |

## 落地改造清单

这一节不再讨论原则，而是直接给出可落代码的改造目标。  
约定写法：

- `现状 schema`：指当前实际暴露给模型的主要形态，不追求列出全部字段。
- `建议 schema`：指建议暴露给模型的 canonical 形态。
- `兼容迁移`：指后端如何继续兼容旧字段、旧别名、旧调用方式。
- `预期收益`：指改完后对提示词体积、调用准确率、纠错性和维护性的直接收益。

| 工具 | 现状 schema | 建议 schema | 兼容迁移 | 预期收益 |
| --- | --- | --- | --- | --- |
| `subagent_control` | 一个大平面对象，`session_id/sessionId/sessionKey`、`runId/runIds`、`timeoutSeconds/waitSeconds/runTimeoutSeconds` 并存 | `action, session_id, run_id, task, message, tasks, limit, wait_seconds, strategy, cleanup` | 保留旧字段解析；入口统一映射到 canonical 字段；对冲突字段返回结构化警告而不是静默吞并 | 明显降低模型选错字段概率，是当前收益最大的收敛项 |
| `智能体蜂群` | `agentId/agentName/name/sessionKey` 并存，`send/spawn/batch_send/wait` 靠 `allOf` 读规则 | `action, agent_id, agent_name, session_id, message, task, tasks, run_ids` | 后端继续兼容 `name/agentName/sessionKey`；批量任务对象也做同样归一化；缺参错误统一返回 example + next_step_hint | 让模型更容易修正错误调用，减少蜂群误投递、误等待、误建线程 |
| `search_content` | `query/pattern` 双入口，`-A/-B/-C/-F/-i` 和语义字段并存，还叠加 `budget.*` | `query, query_mode, path, glob, context_before, context_after, max_matches, timeout_ms, case_sensitive` | 旧别名全部留在解析层；若同时传 `query` 与 `pattern`，后端按统一优先级归一化 | 直接缩小 schema 体积，减少模型把 ripgrep 兼容开关当主要接口学习 |
| `browser` | `action` 很多，且 `request` 泛对象会让模型倾向走万能入口 | `action, browser_session_id, target_id, url, selector, text, key, format, max_chars` | 保留 `act/request`，但作为高级兼容模式；模型侧文档只主推显式动作 | 降低浏览器工具“会用但不稳定”的情况，减少一把梭到 `request` 的模糊调用 |
| `channel_tool` | 联系人查询、消息发送混在同层，`text/content` 双正文入口，另有 `meta/refresh/wait_timeout_s` | `action, channel, account_id, keyword, limit, contact, to, peer_kind, thread_id, content, attachments, wait` | 后端把 `text` 归一到 `content`；`meta/refresh` 留在兼容层；必要时仅管理员/高级模式开放 | 提高渠道工具的一致性，减少模型在正文和目标对象上来回猜字段 |
| `read_file` | `path/file_path/files/offset/limit/start_line/end_line` 多种读取入口混用 | `path, start_line, end_line, line_ranges, mode, indentation` | 保留 `file_path -> path`、`offset+limit -> start/end`、`files` 批量模式；但模型侧文档只展示单文件主路径 | 让模型先学会“精准读一个窗口”，避免盲目批量翻文件 |
| `schedule_task` | 主要信息包在 `job` 大对象中，层级深，新增/更新/启停共用同一棵树 | `action, job_id, schedule_text, message, enabled` | 后端将简化字段装配回完整 `job`；保留高级 `job` 直传模式 | 大幅降低定时任务首轮调用难度，尤其适合自然语言创建任务 |
| `memory_manage` | 字段本身还好，但说明过长，工具使用策略和字段语义混在一起 | `action, memory_id, title, summary, content, category, tags, query, limit, order` | 不动主要字段；重点改 description 结构，把 recall/write 规则拆成短句 | 不一定减少字段数量，但能显著减少模型“看不完说明”的问题 |
| `thread_control` | `session_id/parent_session_id/agentId/title/label/switch/setMain` 共存，命名风格不统一 | `action, session_id, parent_session_id, title, scope, status, limit, set_main` | 旧 camelCase 字段在后端兼容；`switch` 语义并入 `action=switch` | 提升线程治理接口的一致性，减少线程管理动作误配布尔开关 |
| `节点调用` | `action` 可省略，`node_id+command` 也能直接触发 invoke | `action, node_id, command, args, timeout_s` | 旧式“省略 action 直接 invoke”仅放在解析层兼容；模型侧一律显式 `action` | 降低歧义，避免模型把 list 和 invoke 的输入形态混为一谈 |
| `用户世界工具` | `list_users/send_message` 用同一平面对象，依赖条件 schema 理解必填 | `action, keyword, offset, limit, user_id, user_ids, content, content_type, client_msg_id` | 旧结构原样兼容；description 中直接列明两种动作最小参数 | 简化用户世界类动作的学习成本，提升“查人后发消息”的连贯性 |
| `a2a观察` / `a2a等待` | 两个工具参数高度重叠，同时支持 `task_ids` 和 `tasks` 大对象 | `task_ids, endpoint, service_name, refresh, timeout_s`；等待工具额外保留 `wait_s, poll_interval_s` | `tasks` 作为高级兼容输入继续解析出 `task_ids` | 降低 A2A 类工具的双接口心智负担，促使模型稳定围绕 task_id 操作 |
| `desktop_controller` | `description` 仍是必填，且容易和 `bbox/action` 形成重复语义 | `action, bbox, to_bbox, key, text, delay_ms, duration_ms, scroll_steps` | 暂时保留 `description` 兼容；模型侧文档改为非必填或仅调试使用 | 降低桌面控制工具的冗余描述负担，减少动作参数相互矛盾 |

## 落地顺序建议

建议按“先收敛最影响模型正确率的工具，再处理重工具”的顺序推进：

1. 先做 `subagent_control`、`智能体蜂群`、`search_content`
2. 再做 `browser`、`channel_tool`、`read_file`
3. 然后处理 `schedule_task`、`thread_control`、`节点调用`
4. 最后再清理 `memory_manage`、`用户世界工具`、`a2a*`、`desktop_controller`

这样安排的原因：

- 第一批直接影响多智能体协作和高频文件检索，错误成本最高。
- 第二批属于大模型“会调用但不稳定”的重工具，收敛后体感提升最明显。
- 第三批偏治理与系统能力，重要但频次略低。
- 最后一批更多是规范化和统一体验，适合在主干收敛后做。

## 改造约束

真正落代码时，建议强制遵守以下约束：

- 模型侧 schema 只保留 canonical 字段，不再暴露兼容别名。
- 兼容别名全部下沉到后端解析层，不影响运行时兼容。
- 每个多动作工具都补一份“动作 -> 最小必填 -> 常见下一步”的短规则。
- 错误返回优先给出 `missing_fields`、`example`、`next_step_hint`，不要只给自然语言报错。
- 不要再让 `allOf/if/then` 承担主要教学职责，它们最多作为运行时校验补充。

## 建议落地顺序

| 优先级 | 工具 | 原因 | 建议动作 |
| --- | --- | --- | --- |
| P0 | `subagent_control` | 参数面过大、别名过多、动作复杂 | 先做 canonical 参数收敛 |
| P0 | `智能体蜂群` | 目标标识多套并存，且直接影响多智能体执行正确性 | 收敛 agent/session/run 标识命名，并补强短示例与缺参提示 |
| P0 | `search_content` | 兼容 rg 风格导致模型侧过载 | 只保留模型真正会用的查询参数 |
| P1 | `browser` | 动作面大，且 `act/request` 过于泛化 | 建立模型侧主路径动作集，高级动作下沉 |
| P1 | `channel_tool` | 联系人检索与消息投递参数混杂 | 统一消息正文字段，收敛大对象 |
| P1 | `read_file` | 兼容形态多 | 收敛为单文件主形态，批量读作为高级用法 |
| P1 | `schedule_task` | `job` 树偏深 | 提供模型侧简化参数视图 |
| P1 | `memory_manage` | description 过长 | 收紧描述，保留稳定字段 |
| P2 | `thread_control` | 命名风格不统一但风险相对可控 | 收敛布尔与会话标识字段命名 |
| P2 | `desktop_controller` | 桌面动作强依赖外部视觉上下文 | 收紧必填字段，减少模型自由拼装空间 |

## 结论

wunder 当前工具结构的核心问题，不是“动作太多”，而是：

- 模型侧暴露了太多兼容层
- 多动作工具缺少 canonical 参数收敛
- description 与 schema 都承担了过多职责

在保留“成套多动作工具”风格的前提下，最合理的方向是：

- 不拆散工具
- 但将模型侧 schema 缩成最小必要集
- 将复杂动作规则收敛成短描述和短示例
- 将兼容别名全部下沉到后端解析层
