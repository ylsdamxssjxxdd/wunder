# 前端聊天渲染根治方案

> 针对用户侧前端聊天页面长期反复出现的三类症状：**气泡重复**、**顺序颠倒**、**假死不更新**。
> 本文给出根因分析（附代码证据）与根治路线。结论先行：这三类症状是同一个架构病灶的三种表现，
> 单独修任何一个症状都只是打补丁，必须消灭病灶本身——**双真相源 + 双向同步**。

---

## 1. 结论摘要

当前聊天渲染链路存在**两套并行的状态机**，且二者**互相喂数据**：

- **legacy 管线**：`store.messages` 数组，由 4 个写入者并发原地 mutate，靠启发式（轮次猜测、内容匹配、时间窗判断）决定气泡归属；
- **canonical 管线**：`runtimeProjection`（`frontend/src/realtime/chat/chatRuntimeReducer.ts`），有严格的 event_id 去重、seq 乱序缓冲、turn 结构。

两套管线的耦合方式决定了问题不可能靠修补收敛：

1. legacy 每次 mutate 都会通过 `legacy_messages_reconciled` 事件**反向灌入** projection（把启发式错误"洗"进严格管线）；
2. projection 渲染时又**重用 legacy 的 raw 对象**、projection 为空时**回退 legacy**（渲染源动态翻转）；
3. 响应式更新靠**手动版本号 + 签名比较**驱动，签名漏判即假死。

根治 = 确立 projection 为**唯一真相源**，legacy 数组降级为只读历史输入，全链路单向数据流，以后端已具备的 **per-session 单调 event_id** 为唯一权威序。

## 2. 现状架构盘点

### 2.1 legacy `messages` 数组的 4 个并发写入者

| 写入者 | 位置 | 写入方式 |
|---|---|---|
| send 流 | `stores/chatSendActions.ts:342`（workflow processor 挂到乐观 assistant 占位上） | 原地 mutate + `notifySessionSnapshot` |
| watch 流 | `stores/chatWatcher.ts:146` `startSessionWatcher`（roundStates 状态机 + `ensureRoundState` 匹配/新建气泡，`:348` push） | 原地 mutate + push |
| watchdog 快照 | `stores/chatWatcher.ts:579` `runWatchdogTick` → `applyCanonicalSessionEventsSnapshot` + 终态整理 | 快照对账 |
| 会话打开/历史加载 | `stores/chatSessionOpenLoadActions.ts`（hydration）、`chatMessageHydration.ts` | 数组替换（`replaceMessageArrayKeepingReference`） |

外加合并/保护补丁层：`chatMessageDedup.ts`（404 行）、`chatAssistantMatch.ts`（内容锚定匹配）、
`chatPendingMessage.ts`、`chatRealtimeMessageProtection.ts`、`chatFinalTranscriptMerge.ts`、
`chatMessageArraySync.ts`、`chatCompactionMarker.ts`、`chatWatchdogRecovery.ts`……
这些文件本身就是历史上每次"修一下"留下的地层。

### 2.2 双向同步的证据

- **legacy → projection 反灌**：`stores/chatRuntimeState.ts:1820` `notifySessionSnapshot` →
  `:1866` `syncChatRuntimeProjectionFromLegacy` → `buildLegacyMessagesReconciledEvent` →
  reducer `applyLegacyMessagesReconciled`（`realtime/chat/chatRuntimeReducer.ts:880`）。
  即：legacy 每 mutate 一次，projection 就按 legacy 的样子"对账"一次。
- **legacy 消息进 projection 靠猜**：`chatRuntimeReducer.ts:1145-1152`，turn 绑定强度包含
  `'timestamp'`、`'adjacent_previous'`、`'nearest_existing'` 三种**弱绑定**——同一逻辑消息两次
  reconcile 可能绑到不同 turn，生成不同 messageId。
- **projection → legacy 渲染回借**：`realtime/chat/chatRuntimeRenderAdapter.ts:171`
  `canReuseLegacyMessage` 命中时直接返回 legacy raw 对象；`:129-133` projection 为空时
  `projection-empty-fallback` 回退 legacy 渲染。
- **渲染源动态翻转**：`views/messenger/controller/messengerControllerRenderableMessages.ts:732-785`，
  每次重算先构建 legacyRenderable，再按 decision 在 projection/legacy 间选择。

### 2.3 可以依赖的后端不变量（根治的地基）

- **per-session 单调 event_id**：`crates/wunder-runtime/src/services/stream_events.rs:41`
  `next_event_id = cached_tail.max(storage_tail) + 1`；
- **任意点位重放**：`:57` `load_stream_events(after_event_id, limit)`；watch 协议已支持
  `{ after_event_id }` 续传（`chatWatcher.ts:998`）。

即：**事件序 + 重放**这两个最难的服务端能力已经存在，前端却没有把它们作为唯一权威。

### 2.4 协议缺口（唯一需要后端补的东西）

`client_message_id` 在 `crates/` 全仓 **零命中**——后端不接收、不回带客户端消息 ID。
前端 reducer 却在依赖它做乐观合并（`chatRuntimeReducer.ts:1574,1867`），实际永远匹配不上，
只能退化为内容匹配 + 时间窗去重（`WATCH_USER_MESSAGE_DEDUP_MS`），这正是重复气泡的温床。

## 3. 根因分析：三症状 → 一个病灶

### 3.1 气泡重复

| # | 触发路径 | 证据 |
|---|---|---|
| 1 | **两套乐观占位 ID 体系**：send 路径造 `local-assistant:{modelTurnId}`（`chatSendActions.ts:301`），watch 路径造 `local-assistant:watch:{key}:round:{n}`（`chatWatcher.ts:328`）。send 流中断后 watch 接管同一轮 → 两个 shell 各自成泡 | 两处 ID 无法互认 |
| 2 | **轮次归属靠猜**：事件缺 round 字段时 `resolveWatchRoundNumber`（`chatWatcher.ts:401-476`）用 activeRound/maxKnownRound 兜底猜测；`ensureRoundState`（`:189-367`）再用 pending 候选 + 内容 + 时间戳(±1500ms) 启发式匹配，猜错即新开气泡 | 归属错误 → push 新泡 |
| 3 | **legacy 反灌 projection 时弱绑定**：`'timestamp'/'adjacent_previous'/'nearest_existing'` 绑定使同一消息两次 reconcile 产生不同 turn/messageId | `chatRuntimeReducer.ts:1145-1152` |
| 4 | **用户消息回显无精确键**：后端无 client_message_id，`insertWatchUserMessage` 只能按事件 id + 时间窗去重（`chatWatcher.ts:909`），重连重放路径绕过时窗即重复 | 协议缺口 |

### 3.2 顺序颠倒

| # | 触发路径 | 证据 |
|---|---|---|
| 1 | **未匹配消息全部甩到末尾**：`resolveProjectedVisibleMessagesFromStore`（`chatWatcher.ts:1282-1318`）按 projection 排序后，凡 raw **对象引用**匹配不上的 legacy 消息一律 append 到列表尾部。hydration 重建对象/数组元素替换导致引用断裂时，旧消息瞬间跳到最后 | `:1312-1316` |
| 2 | **turn 排序主键是墙钟时间**：`chatRuntimeSelectors.ts:177-180` 按 `createdAtMs` 排 turn，乐观消息用本地时钟、服务器消息用服务端时间戳，时钟偏差直接翻转顺序 | 墙钟不可作序 |
| 3 | **渲染源翻转时排序规则不同**：projection 排序（turn 结构）与 legacy 排序（数组自然序 + 插入锚点启发式）不一致，`projection-source ↔ projection-empty-fallback` 翻转瞬间整列重排 | `messengerControllerRenderableMessages.ts:753-784` |

### 3.3 假死不更新

| # | 触发路径 | 证据 |
|---|---|---|
| 1 | **签名比较漏判**：响应式靠手动 bump `messageMutationVersion`，是否 bump 由 `hasVisibleMessageMutation` 的签名决定，而签名只取 content/reasoning 的**长度**、最后一个 workflowItem 摘要（`chatRuntimeState.ts:1760-1798`）。同长度内容替换、中间 workflowItem 状态变化 → 签名不变 → 不 bump → UI 冻结 | 长度≠内容 |
| 2 | **send 期间 watch 静默**：`chatWatcher.ts:739-744`，只要 `runtime.sendController || runtime.resumeController` 存在，watch 事件（除 sideband）直接 return。send 流挂死而 controller 未清理 → watch 永久沉默 | 单点闸门 |
| 3 | **event_id 单调闸门误杀**：`chatWatcher.ts:814-885`，`normalizedEventId <= lastEventId` 即丢弃；一旦 lastEventId 被快照/其他路径推得过前，后续 live 事件全部被判重丢弃，只能等 watchdog（带 `WATCH_RECONCILE_COOLDOWN_MS` 冷却）兜底 | 多写 lastEventId |
| 4 | **seq gap 缓冲无超时**：reducer 小 gap（≤4）缓冲等待 `appliedSeq+1`（`chatRuntimeReducer.ts:370-476`），缺失事件永不到达时缓冲不 drain，projection 停止前进；`syncRequired` 触发的 reconcile 又有冷却与竞态 | 无限等待 |
| 5 | **双管线全量重算**：每次变更 legacyRenderable 与 projectionRenderable 都构建 + shadow 深比较，长会话下卡顿被用户感知为假死 | `messengerControllerRenderableMessages.ts:736` |

### 3.4 为什么"修很多次都不行"

每次修复都发生在**症状所在的层**（加一个 dedup、加一个 protection、加一个 watchdog、加一个时间窗），
但产生脏数据的写入者和双向同步还在。补丁之间还会互相制造新竞态
（如 protection 保住的消息与 authoritative prune 冲突、watchdog resume 与 send 流并发）。
`stores/` 下 40+ 个 chat*.ts 文件即是证明。**补丁层越厚，写入路径越多，竞态越多。**

## 4. 根治方案：单一真相源、单向数据流

### 4.1 目标架构

```
                        ┌─────────────────────────────────────────┐
  WS(send/watch/resume) │                                         │
  HTTP snapshot/history ├─→ 规范化(canonical events) ─→ reducer ──→ projection ──→ renderAdapter ──→ UI
  本地乐观(发送时)       │      (唯一入口)            (唯一写入者)  (唯一真相源)     (唯一渲染源)
                        └─────────────────────────────────────────┘
  权威序 = 后端 event_id（单调）        乱序 = 有界缓冲 + 超时强制 replay(after_event_id)
  乐观合并 = client_message_id 精确键   渲染 key = runtime:{role}:{messageId}（永不回退 index）
```

**不变量（必须全部成立，任何 PR 不得破坏）：**

1. 只有 reducer 能改变会话消息状态；UI、watchdog、store action 一律不得直接 mutate 消息对象。
2. 消息排序键 = `(turn 首个 event_id, message 首个 event_id)`；时间戳仅用于显示，永不参与排序。
3. 每条事件按 `event_id` 幂等：重放、重连、重复投递均安全。
4. gap 只允许有界等待（建议 800ms 或收到 8 条后续事件，取先到者），超时立即 `after_event_id` replay，**永不无限等待**。
5. projection 每次 `applied === true` 统一 bump 一次版本号；**删除签名比较机制**。
6. 渲染源永远是 projection；不存在 legacy fallback，也不存在渲染源翻转。

### 4.2 协议层（后端小改，是精确合并的前提）

1. **`client_message_id` 贯穿**：发送消息 payload 携带 `client_message_id`；后端在该轮的
   `received`/`round_start` 及后续所有事件的 payload 中原样回带。
   落点：`crates/wunder-runtime/src/api/`（入参）+ `orchestrator/`（事件产出），存储进 stream event payload。
2. **所有聊天流事件必带** `event_id`（已有）、`user_round`（补齐缺失的事件类型，消灭前端 round 猜测）。
3. 文档同步 `docs/API文档.md`。

> 这是本方案唯一的后端改动，工作量小，但它直接消灭 3.1-#1/#2/#4 三条重复路径。

### 4.3 事件层：统一入口，切断反灌

1. send / watch / resume / watchdog / 快照 / 历史分页，全部只做一件事：
   转成 canonical events → `applyChatRuntimeEvent`。删除各自的 workflow processor 与 roundStates 状态机。
2. **删除 `syncChatRuntimeProjectionFromLegacy` 的实时调用**。`legacy_messages_reconciled`
   仅保留一条用途：会话打开时把 HTTP 历史转成**一次性权威快照**（authoritative transcript），
   之后该会话的 projection 只接受 canonical events。
3. `lastEventId` 只由 reducer 内部维护（单写者），watch 续传的 `after_event_id` 从 projection 读取；
   删除 `chatStreamIds.ts` 中多处独立维护的 event id 状态。
4. send 与 watch 不再互斥静默：两路事件都进 reducer，由 event_id 幂等去重天然解决双流并发
   （删除 `chatWatcher.ts:739-744` 闸门；send 流仅用于拿到本轮错误/终态回执，不再独占写入）。

### 4.4 reducer 层：修正排序与 gap 策略

1. 排序键改为 event_id 序（见不变量 2），改 `chatRuntimeSelectors.ts:75-88、177-180`。
2. gap 缓冲加超时：`pendingSequentialEvents` 入队时登记 deadline，到期未 drain 即置
   `syncRequired` 并**立刻**触发 replay（复用现有 `onSyncRequired` 通道，去掉冷却对首次触发的抑制）。
3. 乐观 user/assistant 占位统一由 `client_message_submitted` 事件产生，键为 `client_message_id`；
   服务端事件回带同 ID 时精确升级占位，不再有两套 local ID 体系。
4. `final`/`thread_status(idle)` 之后到达的迟到 delta：按 event_id 判断——小于等于已终结轮次
   最大 event_id 的 delta 直接丢弃，大于的按新轮次处理（消灭"终态后复活"类竞态）。

### 4.5 渲染层：唯一渲染源与稳定更新

1. `agentRenderableMessages` 只走 `buildChatRuntimeRenderableMessages(projection)`；
   删除 legacyRenderable 构建、decision 翻转、shadow 对照（shadow 保留为 debug-only 开关，默认关闭）。
2. `materializeChatRuntimeMessage` 去掉 `canReuseLegacyMessage`/raw 回借——渲染对象由 projection
   物化并按 messageId 缓存复用（内容未变时返回同一对象引用，保证虚拟窗口高度缓存命中）。
3. 渲染 key 固定 `runtime:{role}:{messageId}`；messageId 由 `client_message_id` 或服务端
   message_id 决定，占位升级时 key 不变（气泡不重挂载）。
4. 版本驱动只留 `runtimeProjectionVersion` 一个；`messageMutationVersion`、
   `hasVisibleMessageMutation` 签名机制整体删除。
5. 空态/加载态：projection 会话未知时渲染骨架，而不是回退 legacy 数组。

### 4.6 待删除/收编的补丁层（root cure 的红利）

迁移完成后以下文件失去存在必要，直接删除（原型阶段保持项目干净）：

```
stores/chatMessageDedup.ts            stores/chatAssistantMatch.ts
stores/chatRealtimeMessageProtection.ts  stores/chatPendingMessage.ts（匹配部分）
stores/chatMessageArraySync.ts        stores/chatFinalTranscriptMerge.ts
stores/chatWatchdogRecovery.ts        stores/chatMessageLookup.ts（round 查找部分）
stores/chatWorkflowProcessor.ts(2089行) → 收编为 reducer 内的 workflow 事件归约
chatWatcher.ts 中 roundStates/ensureRoundState/resolveWatchRoundNumber 整段（~500行）
```

watchdog 简化为一条规则：`本地 appliedSeq < 服务端 tail_event_id 且 N 秒无事件 → replay(after_event_id)`。
不再做任何内容级恢复/终态整理。

## 5. 迁移步骤（每步可独立验收、可回退）

| 阶段 | 内容 | 回退方式 |
|---|---|---|
| P1 后端协议 | client_message_id 贯穿 + 事件补齐 user_round；更新 API 文档 | 字段向后兼容，无回退成本 |
| P2 reducer 硬化 | 排序键改 event_id 序；gap 超时 replay；乐观合并走精确键 | flag `wunder:chat-runtime-render` 已有 legacy 值可回退 |
| P3 切渲染 | 渲染只走 projection，legacy fallback 删除；版本 bump 收敛为单一 | 同上 flag |
| P4 切写入 | send/watch 停止写 legacy 数组，删除反灌；legacy 只剩历史 hydration 输入 | 保留一个发布周期的 shadow 对照日志 |
| P5 清理 | 删除 4.6 清单文件；`docs/功能迭代.md`、`docs/经验教训.md` 落账 | — |

P2/P3 期间用现有 shadow 机制（`chatRuntimeRenderShadow.ts`）收集 drift 指标，drift 归零后进 P4。

## 6. 验收标准

1. **重复**：同一轮次在任意重连/重放/双流并发下，DOM 中 assistant 气泡数 ≤ 1（e2e：断网重连、send 中断由 watch 接管、快照与 live 交叠三个场景）。
2. **顺序**：任意事件乱序注入（fuzz：打乱 event_id 到达序），最终渲染序 = event_id 序；本地时钟偏移 ±5min 不影响顺序。
3. **假死**：人为吞掉任意一条中间事件，UI 在 gap 超时 + replay 后 2s 内恢复推进；同长度内容替换必须触发重渲染。
4. **性能**：千条消息会话流式期间，每事件渲染路径不做全量深比较；虚拟窗口高度缓存命中率不下降。
5. 回归脚本：`frontend` 的 `test:chat-realtime`、`test:chat-*`、Playwright e2e 全绿；
   新增 reducer 乱序/重放/gap 超时单测。

## 7. 风险与边界

- **蜂群/子智能体、审批、压缩标记**等 sideband 事件当前散在 watch 分支里，收编进 reducer 时
  需逐类补 canonical 映射（`chatCanonicalEvents.ts` 已覆盖大半，缺口以 shadow drift 日志为准）。
- 桌面端（SQLite/本地 bridge）无 watchdog 分支（`desktopMode` 早退），P4 需单独验证桌面链路。
- 历史消息无 event_id（老会话），hydration 快照以 transcript 序作为该会话的初始权威序，
  新事件从 tail 接续，不与老数据混排。
