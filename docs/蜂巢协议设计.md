# 蜂巢协议设计（Hive Package Protocol, HPP）

## 1. 背景与目标

为解决「能力沉淀难、跨环境迁移慢、复用成本高」问题，wunder 引入“蜂群包/工蜂包”协议，用于把可运行的多智能体业务能力打包为标准资产并一键导入。

目标：

- 把“蜂群协作能力”从一次性配置变为可分发资产。
- 导入后自动创建蜂群成员智能体，自动注册并启用技能。
- 让包内能力快速适配目标系统：内置工具、MCP 工具、知识库工具默认挂载目标系统当前可用集合。
- 支持 server/desktop/cli 统一资产格式，降低迁移与协作成本。

---

## 2. 核心概念与分层模型

### 2.1 概念定义

- **蜂群包（HivePack）**
  - 业务分发的最小“组织单元”。
  - 组成：`组员构成说明 + 蜂群角色定位说明 + N 个工蜂包`。
- **工蜂包（WorkerPack）**
  - 业务分发的最小“执行单元”。
  - 组成：`工蜂角色定位说明 + N 个技能包`。
- **技能包（SkillPack）**
  - 可执行能力单元，兼容 wunder 现有 SKILL 目录规范（`SKILL.md + scripts/assets/...`）。

### 2.2 逻辑表达式

- `HivePack = HiveMeta + HiveRoleCard + WorkerPack[]`
- `WorkerPack = WorkerMeta + WorkerRoleCard + SkillPack[]`

### 2.3 设计原则

1. **逻辑与运行时解耦**：包声明“业务角色与能力”，不固化目标系统的基础设施细节。
2. **默认可用优先**：导入后默认挂载目标系统的内置/MCP/知识库工具，不要求逐项手工配置。
3. **可验证可回滚**：导入流程必须可预检、可追踪、可失败回滚。
4. **可演进**：协议版本化，向后兼容，允许字段扩展与渐进升级。

## 3. 包格式与目录规范

### 3.1 分发文件

- 推荐扩展名：`.hivepack`（本质为 zip）。
- MIME（建议）：`application/vnd.wunder.hivepack+zip`。
- 单包建议上限：`200MB`（可配置）。

### 3.2 标准目录（v1）

```text
hr-shared-service-1.0.0.hivepack
├─ hive.yaml                      # 蜂群包总清单（必需）
├─ HIVE_ROLE.md                   # 蜂群整体角色定位（必需）
├─ checksums.sha256               # 全包校验摘要（必需）
├─ signatures/
│  └─ package.sig                 # 可选：签名
├─ workers/
│  ├─ recruiter/
│  │  ├─ worker.yaml              # 工蜂清单（必需）
│  │  ├─ WORKER_ROLE.md           # 工蜂角色定位（必需）
│  │  └─ skills/
│  │     ├─ jd_writer/
│  │     │  ├─ skill.yaml         # 技能元数据（建议）
│  │     │  ├─ SKILL.md           # 技能说明（必需）
│  │     │  ├─ scripts/
│  │     │  └─ assets/
│  │     └─ candidate_screening/
│  └─ interviewer/
├─ fixtures/                      # 可选：样例输入/输出、验收用例
└─ README.md                      # 可选：导入说明
```

### 3.3 关键约束

- `hive.yaml`、`worker.yaml`、`SKILL.md` 必须存在且 UTF-8 编码。
- 包内路径必须是相对路径，禁止 `..`、绝对路径与软链逃逸。
- 技能包执行脚本仅允许落在技能目录内。

## 4. 清单字段设计（协议 v1）

### 4.1 `hive.yaml`（蜂群包清单）

```yaml
protocol: hpp/1.0
kind: hive_pack
pack:
  id: hivepack_hr_shared_service
  name: 人力资源共享服务蜂群
  version: 1.0.0
  description: 覆盖招聘、面试、入转调离主流程
  author: wunder-hr-team
  created_at: 2026-03-09T18:00:00+08:00
  tags: [hr, recruitment, interview, onboarding]
compatibility:
  wunder_version: ">=0.1.0"
  runtime_modes: [server, desktop, cli]
mount_policy:
  builtin_tools: system_all
  mcp_tools: system_all
  knowledge_tools: system_all
  imported_skills: package_only
import_policy:
  default_conflict: versioned_copy
  create_hive_if_missing: true
workers:
  - worker_id: recruiter
    path: workers/recruiter
    role: coordinator
  - worker_id: interviewer
    path: workers/interviewer
    role: specialist
integrity:
  checksum_file: checksums.sha256
  signature_file: signatures/package.sig
```

说明：

- `mount_policy` 定义导入后默认挂载策略，满足“目标系统快速适配”。
- `workers[]` 决定导入后自动创建哪些智能体。

### 4.2 `worker.yaml`（工蜂包清单）

```yaml
protocol: hpp/1.0
kind: worker_pack
worker:
  id: recruiter
  display_name: 招聘专员
  description: 负责 JD 生成、渠道建议与候选人初筛协同
  duty: recruiter_specialist
agent_profile:
  system_prompt_file: WORKER_ROLE.md
  model_hint: gpt-5-mini
  approval_mode: suggest
  icon: fa-user-tie
skills:
  - skill_id: jd_writer
    path: skills/jd_writer
    required: true
  - skill_id: candidate_screening
    path: skills/candidate_screening
    required: true
dependencies:
  optional_mcp_services: [talent_db, org_chart_service]
  optional_knowledge_bases: [hr_policy]
```

说明：

- `dependencies` 仅用于导入预检与告警，不阻断默认挂载策略。
- `agent_profile` 会映射为 `user_agents` 创建参数（名称、提示词、图标、审批模式等）。

### 4.3 `skill.yaml`（技能元数据，建议）

```yaml
protocol: hpp/1.0
kind: skill_pack
skill:
  id: jd_writer
  name: JD生成器
  version: 1.2.0
  entry: SKILL.md
  language: zh-CN
  tags: [hr, jd]
```

## 5. 导入协议（Import Protocol）

### 5.1 状态机

`uploaded -> validating -> planning -> installing -> creating_agents -> activating -> completed|failed|rolled_back`

### 5.2 导入阶段

1. **上传与解包**
   - 接收 `.hivepack`，写入临时隔离目录。
   - 校验 zip 结构与路径安全。
2. **完整性校验**
   - 校验 `checksums.sha256`。
   - 若存在签名，执行签名验证（可配“强制签名”策略）。
3. **预检规划**
   - 解析 `hive.yaml/worker.yaml/skill.yaml`。
   - 输出导入计划：新建/复用蜂群、待创建智能体、待安装技能、冲突项。
4. **技能安装**
   - 将技能包注册为导入用户的 `custom` 技能源。
   - 对同名技能执行冲突策略（见 5.4）。
5. **智能体物化**
   - 按 `workers[]` 自动创建智能体并归属目标蜂群。
   - 写入角色提示词与默认配置。
6. **默认能力挂载**
   - 对每个新建工蜂应用 `mount_policy`：
     - `builtin_tools = system_all`
     - `mcp_tools = system_all`
     - `knowledge_tools = system_all`
     - `imported_skills = package_only`
7. **激活与验收**
   - 技能启用检查、智能体可见性检查、蜂群成员关系检查。
   - 产出导入报告（成功项、告警项、失败项）。

### 5.3 回滚策略

- 任一关键阶段失败时触发回滚：
  - 删除本次创建的智能体；
  - 回滚本次安装的新技能（不影响历史已存在技能）；
  - 若蜂群为本次新建且无历史成员，则删除该蜂群；
  - 保留失败报告与审计日志。

### 5.4 冲突策略（默认 `versioned_copy`）

- **技能冲突**（同 `skill_id`）：
  - 相同 hash：跳过安装并复用；
  - 不同 hash：生成版本化副本（如 `jd_writer@1.2.0+ab12`）。
- **智能体命名冲突**：
  - 默认追加后缀（如 `招聘专员 (2)`）。
- **蜂群冲突**：
  - 优先按 `hive_id`/名称匹配复用；
  - 用户选择“总是新建”时强制创建新蜂群。

## 6. 导出协议（Export Protocol）

### 6.1 导出来源

- 从现有蜂群实体导出：
  - 蜂群元信息；
  - 成员智能体角色信息；
  - 每个成员绑定的技能目录（可选：仅导出引用或导出完整内容）。

### 6.2 导出模式

- `full`（默认）：导出完整技能内容，适合离线迁移。
- `reference_only`：仅导出技能引用与依赖声明，适合同源环境快速复刻。

### 6.3 导出产物

- 标准 `.hivepack`；
- 自动生成 `checksums.sha256`；
- 可选签名 `signatures/package.sig`；
- 导出报告（包含包大小、技能数量、哈希、导出时间）。

## 7. API 与事件协议（建议）

> 传输策略：WebSocket 优先，SSE 兜底。

### 7.1 API 设计（用户侧）

1. `POST /wunder/beeroom/packs/import`
   - `multipart/form-data`：`file` + `options`
   - 返回：`job_id`
2. `GET /wunder/beeroom/packs/import/{job_id}`
   - 返回导入状态与详细报告
3. `POST /wunder/beeroom/packs/export`
   - 入参：`group_id`、`mode=full|reference_only`
   - 返回：下载任务 `job_id`
4. `GET /wunder/beeroom/packs/export/{job_id}/download`
   - 下载 `.hivepack`

### 7.2 进度事件

- `beeroom.pack.import.progress`
- `beeroom.pack.import.completed`
- `beeroom.pack.import.failed`
- `beeroom.pack.export.progress`
- `beeroom.pack.export.completed`

事件体建议统一字段：

- `job_id`
- `phase`
- `progress`（0-100）
- `summary`
- `detail`（可选）
- `timestamp`

## 8. 数据模型建议

建议新增三张表：

1. `beeroom_pack_jobs`
   - 记录导入/导出任务状态机与报告摘要。
2. `beeroom_pack_registry`
   - 记录已导入包（`pack_id/version/hash/source/owner_user_id`）。
3. `beeroom_pack_items`
   - 记录任务级明细（技能安装项、智能体创建项、冲突与回滚项）。

索引建议：

- `idx_pack_jobs_user_status (user_id, status, updated_at desc)`
- `idx_pack_registry_owner_pack (owner_user_id, pack_id, version)`
- `idx_pack_items_job (job_id, item_type, status)`

## 9. 前端交互设计（蜂群页中栏“+”）

### 9.1 导入流程

1. 中栏 `+` 点击后出现菜单：`新建蜂群 / 导入蜂群包`。
2. 选择 `.hivepack` 后先显示“预检结果”：
   - 将创建几个工蜂；
   - 将安装几个技能；
   - 是否存在命名冲突；
   - 目标挂载策略（系统默认全挂载）。
3. 用户确认后开始导入，进度实时展示。
4. 导入成功后自动跳转到新蜂群详情并高亮新建工蜂。

### 9.2 导出流程

1. 在蜂群详情页点击“导出蜂群包”。
2. 选择导出模式（`full`/`reference_only`）。
3. 生成后直接下载 `.hivepack`。

## 10. 安全与治理

1. **包来源可信**
   - 支持签名验证与签发者白名单。
2. **内容安全**
   - 阻断路径穿越、超大文件、非法扩展名。
3. **执行安全**
   - 技能仍受现有工具审批、沙盒与权限策略约束。
4. **审计追踪**
   - 记录导入人、来源、摘要、创建资产清单、失败原因。
5. **租户隔离**
   - 导入资产默认归属于当前 `user_id`（或当前租户域），不可越权注入他人空间。

## 11. 兼容策略与版本演进

### 11.1 版本规范

- 协议版本：`protocol: hpp/<major>.<minor>`
- 包版本：`pack.version`（语义化版本）

### 11.2 兼容规则

- 同 `major`：向后兼容，未知字段忽略。
- 跨 `major`：必须通过迁移器转换后再导入。
- 建议提供 `hpp migrate` 工具用于离线升级旧包。

---

## 12. 分阶段落地建议

### Phase 1（最小可用）

- 支持 `.hivepack` 导入/导出；
- 自动创建工蜂智能体；
- 自动安装并启用技能；
- 默认挂载系统内置/MCP/知识库工具；
- 具备基础回滚与导入报告。

### Phase 2（增强）

- 签名校验、发布者信任链；
- 冲突策略可视化配置；
- 导入前模拟演练（dry-run）。

### Phase 3（生态）

- 包仓库与版本市场（组织内共享）；
- 蜂群包依赖解析与自动升级建议；
- 标准化测试样例与质量评分。

## 13. 示例：人力资源蜂群包落地效果

导入“人力资源蜂群包”后，系统自动完成：

1. 创建蜂群：`人力资源共享服务蜂群`；
2. 创建工蜂智能体：`招聘专员`、`面试官`、`入职专员`；
3. 安装技能：`JD生成`、`简历初筛`、`面试问题设计`、`入职材料校验`；
4. 每个工蜂默认挂载目标系统当前可用的内置/MCP/知识库工具；
5. 导入结果在蜂群工作台可即时查看并直接发起协作。

## 14. 结论

该协议把“蜂群协作”升级为可打包、可迁移、可审计、可复用的标准资产。  
对使用者而言是“一键导入即可用”，对平台而言是“能力沉淀可持续演进”，兼顾了工程优雅性与业务落地效率。

## 15. 当前实现状态（2026-03-09）

### 15.1 已落地（Phase 1）

- 后端已提供 HivePack 导入导出接口：
  - `POST /wunder/beeroom/packs/import`
  - `GET /wunder/beeroom/packs/import/{job_id}`
  - `POST /wunder/beeroom/packs/export`
  - `GET /wunder/beeroom/packs/export/{job_id}`
  - `GET /wunder/beeroom/packs/export/{job_id}/download`
- 导入已支持：
  - zip 结构与路径安全校验（阻断绝对路径与 `..` 穿越）；
  - 解析 `hive.yaml`、`worker.yaml`、`WORKER_ROLE.md`；
  - 技能安装与启用、工蜂自动创建、蜂群自动归组；
  - 失败回滚（智能体删除、技能回滚、蜂群归档）。
- 导出已支持：
  - 从现有蜂群组装标准包目录；
  - 生成 `hive.yaml/worker.yaml/skill.yaml/checksums.sha256`；
  - 输出 `.hivepack` 文件并提供下载。

### 15.2 当前约束

- 任务记录暂存于 `meta`（键：`beeroom_pack_job:{job_id}`），后续可迁移到独立任务表。
- 导出模式 `reference_only` 当前会写入技能占位说明（不含完整脚本资产）。
- 签名验签、发布者信任链、dry-run 预演仍处于后续阶段。
