# 蜂巢协议设计（Hive Packaging Protocol, HPP）

> 本文只定义蜂巢协议本身，包括协议对象、目录结构、字段语义、交互语义与校验规则。

---

## 0. 约定

本文使用以下规范关键字：

- `MUST`：必须满足
- `SHOULD`：建议满足
- `MAY`：可选支持

协议中的核心对象为：

- `HivePack`：蜂群级资产封装单位
- `WorkerCard`：单工蜂资产描述单位
- `SkillPack`：技能资产描述单位
- `WorkerCardBundle`：多张工蜂卡的批量传输单位

---

## 1. 协议对象与关系

### 1.1 三层抽象

- **`HivePack`**：描述蜂群元信息、成员清单与技能集合
- **`WorkerCard`**：描述单个智能体的身份、提示词、能力、交互入口与运行策略
- **`SkillPack`**：描述可复用技能及其附属资源

### 1.2 形式化表达

- `HivePack = HiveMeta + WorkerCard[] + SkillPack[]`
- `WorkerCard = Metadata + PromptProfile + AbilityProfile + InteractionProfile + RuntimeProfile + HiveRef + Extensions`
- `SkillPack = SkillMeta + SkillEntry + SkillAssets[]`
- `WorkerCardBundle = WorkerCard[]`

### 1.3 单工蜂资产单位

协议以 `WorkerCard` 作为单工蜂的标准资产单位。一个 `WorkerCard` 用于完整描述一个工蜂的可交换配置。

---

## 2. 封装结构

### 2.1 `HivePack` 推荐目录结构

```text
example.hivepack/
├─ hive.yaml
├─ workers/
│  ├─ planner/
│  │  └─ worker-card.json
│  ├─ coder/
│  │  └─ worker-card.json
│  └─ reviewer/
│     └─ worker-card.json
└─ skills/
   ├─ task_breakdown/
   │  ├─ skill.yaml
   │  └─ SKILL.md
   └─ risk_review/
      ├─ skill.yaml
      └─ SKILL.md
```

### 2.2 文件约定

- 每个工蜂目录 `MUST` 包含一份 `worker-card.json`
- `hive.yaml` 用于描述蜂群级元信息
- 每个技能目录 `SHOULD` 包含 `skill.yaml`
- 技能正文、模板、脚本、静态资源 `MAY` 随技能目录一起封装

---

## 3. `WorkerCard` 规范

### 3.1 文件格式

- 标准文件名：`worker-card.json`
- 标准对象类型：`kind = "WorkerCard"`
- 标准版本字段：`schema_version`
- 当前版本标识：`wunder/worker-card@1`

### 3.2 顶层结构

一个 `WorkerCard` `SHOULD` 包含以下顶层字段：

- `schema_version`
- `kind`
- `metadata`
- `prompt`
- `abilities`
- `interaction`
- `runtime`
- `hive`
- `extensions`

### 3.3 字段语义

#### `schema_version`

- 类型：`string`
- 要求：`MUST`
- 含义：协议版本标识
- 示例：`"wunder/worker-card@1"`

#### `kind`

- 类型：`string`
- 要求：`MUST`
- 固定值：`"WorkerCard"`

#### `metadata`

- 类型：`object`
- 要求：`MUST`
- 含义：工蜂基础身份信息

`metadata` 子字段：

- `id`：工蜂标识，`SHOULD` 在当前蜂群内唯一
- `name`：工蜂名称，`MUST` 非空
- `description`：工蜂描述，`MAY` 为空
- `icon`：图标或图标引用，`MAY` 为空
- `exported_at`：导出时间，`MAY` 为 ISO 8601 时间字符串

#### `prompt`

- 类型：`object`
- 要求：`MUST`
- 含义：工蜂提示词配置

`prompt` 子字段：

- `system_prompt`：主系统提示词，`MUST`
- `extra_prompt`：附加提示词，`MAY`

#### `abilities`

- 类型：`object`
- 要求：`MUST`
- 含义：工蜂声明的工具与技能能力

`abilities` 子字段：

- `tool_names`：工具名数组，`MAY`
- `skills`：技能名数组，`MAY`

约束：

- `tool_names` 与 `skills` 中的元素 `SHOULD` 去重
- 元素类型 `MUST` 为字符串

#### `interaction`

- 类型：`object`
- 要求：`MUST`
- 含义：工蜂对外暴露的交互入口配置

`interaction` 子字段：

- `preset_questions`：预设问题列表，`MAY`

#### `runtime`

- 类型：`object`
- 要求：`MUST`
- 含义：运行时策略声明

`runtime` 子字段：

- `approval_mode`：审批模式，`MUST`
- `sandbox_container_id`：容器编号，`MUST`
- `is_shared`：是否共享，`MAY`

`approval_mode` 允许值：

- `suggest`
- `auto_edit`
- `full_auto`

`sandbox_container_id` 约束：

- `MUST` 为整数
- 取值范围 `MUST` 为 `1..10`

#### `hive`

- 类型：`object`
- 要求：`MAY`
- 含义：蜂群归属引用信息

`hive` 子字段：

- `id`：蜂群标识
- `name`：蜂群名称
- `description`：蜂群描述

#### `extensions`

- 类型：`object`
- 要求：`MAY`
- 含义：扩展字段容器
- 约束：未识别扩展字段 `SHOULD` 按透传处理

### 3.4 示例

```json
{
  "schema_version": "wunder/worker-card@1",
  "kind": "WorkerCard",
  "metadata": {
    "id": "planner",
    "name": "规划工蜂",
    "description": "负责拆解任务、制定执行计划",
    "icon": "",
    "exported_at": "2026-03-13T08:30:00Z"
  },
  "prompt": {
    "system_prompt": "你负责先理解目标，再给出分步方案。",
    "extra_prompt": "输出时优先使用编号列表，并标出风险点。"
  },
  "abilities": {
    "tool_names": ["read_file", "write_file", "memory_manager"],
    "skills": ["task_breakdown", "risk_review"]
  },
  "interaction": {
    "preset_questions": [
      "先帮我拆解这个任务，并给出三步方案",
      "帮我列出当前方案的主要风险",
      "请根据现有上下文生成执行清单"
    ]
  },
  "runtime": {
    "approval_mode": "auto_edit",
    "sandbox_container_id": 1,
    "is_shared": false
  },
  "hive": {
    "id": "product-team",
    "name": "产品蜂群",
    "description": "面向需求分析与交付协同"
  },
  "extensions": {}
}
```

---

## 4. `preset_questions` 语义

`interaction.preset_questions` 是 `WorkerCard` 的标准字段，用于声明工蜂的可复用交互入口。

约束如下：

- `preset_questions` `MUST` 为字符串数组
- 每个元素 `SHOULD` 为可直接发起对话的完整短句
- 元素顺序 `SHOULD` 保留，消费端 `SHOULD` 按声明顺序展示
- 消费端读取该字段后，`MAY` 以快捷入口、命令列表或模板问题形式呈现
- 协议仅定义声明语义，不限定具体界面实现

## 5. `WorkerCardBundle` 规范

### 5.1 用途

`WorkerCardBundle` 用于在一个传输对象中承载多张 `WorkerCard`。

### 5.2 结构

- `schema_version`：版本标识，示例 `wunder/worker-card-bundle@1`
- `kind`：固定为 `WorkerCardBundle`
- `items`：`WorkerCard` 数组

### 5.3 示例

```json
{
  "schema_version": "wunder/worker-card-bundle@1",
  "kind": "WorkerCardBundle",
  "items": [
    {
      "schema_version": "wunder/worker-card@1",
      "kind": "WorkerCard",
      "metadata": { "id": "planner", "name": "规划工蜂" },
      "prompt": { "system_prompt": "负责拆解任务" },
      "abilities": {},
      "interaction": {},
      "runtime": { "approval_mode": "suggest", "sandbox_container_id": 1 }
    },
    {
      "schema_version": "wunder/worker-card@1",
      "kind": "WorkerCard",
      "metadata": { "id": "coder", "name": "执行工蜂" },
      "prompt": { "system_prompt": "负责执行任务" },
      "abilities": {},
      "interaction": {},
      "runtime": { "approval_mode": "auto_edit", "sandbox_container_id": 2 }
    }
  ]
}
```

## 6. `HivePack` 与 `SkillPack`

### 6.1 `HivePack`

`HivePack` 是蜂群级分发单元，负责组织以下内容：

- 蜂群元信息
- 多个 `WorkerCard`
- 多个 `SkillPack`
- 可选的扩展元数据

`HivePack` 中的 `workers/*/worker-card.json` 为工蜂标准描述文件。

### 6.2 `SkillPack`

`SkillPack` 是技能级分发单元，通常包含：

- `skill.yaml`：技能元信息与入口声明
- `SKILL.md`：技能说明或模型使用说明
- 可选脚本、模板、数据文件与静态资源

协议只要求 `SkillPack` 可被蜂群封装引用，不强制限定技能内部资源的具体组织方式。

---

## 7. 校验规则

消费端接收 `WorkerCard` 或 `WorkerCardBundle` 时，`SHOULD` 至少完成以下校验：

- JSON 结构合法
- `kind` 与 `schema_version` 合法
- 必填字段存在且类型正确
- `metadata.name` 非空
- `approval_mode` 属于允许枚举值
- `sandbox_container_id` 在允许范围内
- `tool_names`、`skills`、`preset_questions` 的元素类型正确
- 可枚举数组完成去重或在消费阶段按去重策略处理

---

## 8. 安全语义

- `WorkerCard` 只声明工蜂期望配置，不直接授予运行权限
- `abilities.tool_names` 与 `abilities.skills` 表示声明能力，不等同于目标环境的实际可用能力
- 目标系统 `MUST` 根据自身权限模型、运行环境与安全策略决定最终可启用的工具、技能与审批级别
- 未识别字段或扩展字段的处理方式 `SHOULD` 保持前向兼容，避免无关字段导致整体对象不可解析
