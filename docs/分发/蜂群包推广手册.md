# 蜂群包推广手册（平台无关版）

> 适用范围：任意智能体平台的蜂群资产推广、实施与运营。  
> 配套规范：`docs/蜂巢协议设计.md`

---

## 1. 手册目标

本手册用于把“蜂群能力分发”从一次性项目交付，升级为可规模复制的标准化运营动作。

面向三类角色：

- **业务负责人**：判断价值、边界和 ROI。
- **实施工程师**：按标准导入、验收、回滚。
- **平台运营**：建立版本治理和复用机制。

---

## 2. 对外主张（统一口径）

1. **导入即用**：一个 `.hivepack` 文件可快速物化蜂群能力。  
2. **跨平台复用**：同一包可迁移到不同智能体系统。  
3. **冲突可控**：默认自动改名，避免覆盖已有资产。  
4. **风险可管**：任务化导入导出，失败可回滚。  
5. **资产可持续**：经验沉淀为版本化、可审计资产。

---

## 3. 推广交付标准包

每次推广交付建议包含：

1. `*.hivepack` 制品（支持极简手工包）  
2. 版本说明（`CHANGELOG` 摘要）  
3. 适用场景说明（行业/部门/前置能力）  
4. 验收清单（3~8 条关键指标）  
5. 回滚预案（触发条件与执行步骤）

## 5. 实施操作模板（平台无关）

### 5.1 导入模板

1. 打开平台“蜂群资产导入”入口。  
2. 选择 `.hivepack` 文件并提交导入任务。  
3. 跟踪任务状态：`uploaded → validating → planning → installing → creating_agents → binding_capabilities → activating`。  
4. 导入完成后查看 `report`，确认：
   - 目标蜂群 ID/名称；
   - 新建工蜂数量；
   - 技能安装结果；
   - 冲突改名映射（`from -> to`）。

### 5.2 导出模板

1. 选择目标蜂群并创建导出任务。  
2. 选择导出模式：`full` 或 `reference_only`。  
3. 导出完成后下载并归档制品。  
4. 将导出报告纳入版本记录（大小、摘要、时间、作者）。

### 5.3 手工构建极简蜂群包（推荐）

最小目录如下（其余文件均可选）：

```text
your-pack.zip
├─ hive.yaml
├─ skills/
│  └─ requirement_analyzer/
│     └─ SKILL.md
└─ workers/
   └─ planner/
      ├─ WORKER_ROLE.md
      └─ skills.yaml
```

导入器自动处理：

1. `hive.yaml.workers[]` 为空时自动扫描 `workers/*`；  
2. `skills.yaml` 记录技能名称，技能原始文件统一从包根 `skills/*` 读取；  
3. `skills.yaml` 缺失时兼容旧版 `worker.yaml` 或工蜂目录内 `skills/*/SKILL.md`；  
4. `checksums.sha256`、`package.sig`、`skill.yaml` 缺失不阻塞导入。

## 6. 现场验收清单（最小闭环）

导入后至少验证：

1. 蜂群结构正确（成员数量、角色分工符合预期）。  
2. 工蜂实体可见可用（能进入会话并响应）。  
3. 技能安装成功（可调用、可执行、可观测）。  
4. 能力挂载符合策略（系统能力 + 导入技能）。  
5. 任意工蜂可完成一次端到端样例任务。  
6. 冲突处理符合策略（默认自动改名，不复用）。  
7. 失败场景下回滚有效（无脏资产残留）。

## 7. 版本与发布策略

建议采用双轨：

- **LTS**：稳态维护，面向生产。  
- **Latest**：快速迭代，面向试点。  

发布门槛建议：

1. 试点环境导入成功率 ≥ 99%。  
2. 验收样例通过率 = 100%。  
3. 回滚演练通过率 = 100%。  
4. 协议与推广文档同步更新。  
5. 关键风险项（权限、合规、安全）已签字确认。

## 8. 常见问题（FAQ）

### Q1：导入失败会影响现网吗？

应通过任务化回滚保证“本次新增、本次可撤销”。  
建议在导入前后执行资产快照对比，确保可审计。

### Q2：为何不同平台导入后能力不完全一致？

协议定义的是能力类别与挂载策略，不绑定具体工具实现；  
最终能力由“目标平台当前可用能力 + 权限策略”共同决定。

### Q3：如何避免导入覆盖线上已有资产？

建议默认 `auto_rename_only`：自动改名后新建，不复用、不覆盖。  
同时把改名映射写入导入报告，便于追踪。

### Q4：必须提供 checksums、签名、skill.yaml 吗？

不是必须。  
协议支持“极简手工包”直接导入，上述文件均为可选增强项，建议在正式发布时再补齐。

## 9. 运营机制建议

### 9.1 资产目录治理

建议按以下维度管理蜂群包：

- 业务域（HR/客服/法务/研发等）  
- 难度等级（L1/L2/L3）  
- 版本（SemVer）  
- 负责人（Owner）  
- 状态（试点/生产/弃用）

### 9.2 指标体系

至少跟踪：

- 导入成功率  
- 样例任务完成率  
- 资产复用次数  
- 升级成功率  
- 回滚次数与原因  
- 平均导入耗时

推荐结构：

1. **1分钟**：问题与目标（为什么要资产协议）  
2. **2分钟**：协议对象与价值（HivePack/WorkerPack/SkillPack）  
3. **2分钟**：流程图讲解（第4节）  
4. **2分钟**：风险控制（冲突策略+回滚+审计）  
5. **2分钟**：试点数据与 ROI  
6. **1分钟**：下一步路线（LTS/Latest 与资产运营）
