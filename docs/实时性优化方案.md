# 实时性优化方案

## 1. 目标

本方案聚焦 wunder 聊天主链路的“用户感知实时性”，目标不是单纯压低单点耗时，而是让用户在以下场景都能持续收到及时、可信、可恢复的状态反馈：

- 请求进入排队后，界面不能悬空等待，必须立即进入“已接管、可继续追踪”的状态。
- WebSocket 断线、切换、慢客户端积压后，不能出现“服务端还在跑，但前端像卡死”的错觉。
- 刷新页面、切换页面、重连 WS/SSE 后，排队态、执行态、完成态都能从持久化事件中恢复。
- 实时链路在高频增量输出下，优先保证“可恢复”而不是“静默丢事件”。

## 2. 当前链路概览

当前聊天实时链路由四层组成：

1. 提交流量层
   - `/wunder`、`/wunder/chat/*` 负责会话创建、消息提交、流式输出入口。
   - 传输层以 WebSocket 为主，SSE 作为兜底。

2. 执行编排层
   - `Orchestrator` 负责模型调用、工具调用、事件流输出。
   - `AgentRuntime` 负责忙时排队、重试、串行化同会话任务。

3. 状态与回放层
   - `monitor` 负责内存态运行状态与工作流事件。
   - `stream_events` 负责持久化事件、断点续传、刷新恢复。

4. 前端消费层
   - 前端 WS 多路复用器负责请求-响应配对。
   - chat store 负责把实时事件映射成消息气泡、工作流节点、补偿 watch/resume。

## 3. 已识别问题

### 3.1 排队接管不完整

WS `start` 请求在收到 `queued` 后，前端请求 promise 之前不会结束，容易出现“请求已排队，但当前流仍像没返回”的悬空状态，用户会误判为发送失败或卡住。

### 3.2 队列事件不可稳定回放

`queue_enter/queue_start/queue_finish/queue_fail` 之前主要停留在 monitor 内存事件里，刷新页面、重连或 watch/replay 时无法稳定恢复，导致排队状态展示断档。

### 3.3 活跃态判定不统一

`waiting`、`queued task pending/retry/running` 没有在所有读取侧统一视为活跃流状态，表现为：

- 会话详情刷新后可能看不出该会话仍在排队。
- `get_session_events` 返回的 `running` 可能偏早变成 `false`。
- 一些恢复路径只把 `running/cancelling` 当作活跃态。

### 3.4 慢客户端恢复语义偏弱

当 WS 出站队列逼近满载时，`llm_output_delta` 之前可能被软丢弃，客户端不一定能及时切到 `resume/watch`，会产生“内容跳字”或“局部缺字”。

### 3.5 同 session 新请求可能绕过已排队任务

如果只依赖 monitor/会话锁判断忙闲，而不检查持久化的 `agent_tasks`，同一 `session` 的后续请求可能在前一个请求已入队但尚未开始执行时直接穿透，破坏顺序性和用户感知一致性。

## 4. 本轮已落地优化

### 4.1 前端排队接管

- WS 请求层新增 `resolveOnQueued`，`start` 收到 `queued` 即结束本次请求并切换到 watcher 模式。
- chat store 新增 `queue_enter/queue_start/queue_finish/queue_fail` 工作流事件消费逻辑。
- `queue_fail` 被视为 watch 路径终态，避免排队失败后界面长期停留在 loading。

### 4.2 队列生命周期事件持久化

- `AgentRuntime` 新增对 `queue_enter/queue_start/queue_finish/queue_fail` 的统一发送入口。
- 队列生命周期事件现在会同时写入：
  - monitor 事件流
  - `stream_events` 持久化流
- 这意味着页面刷新、WS/SSE 重连、`watch/resume` 都能回放完整队列状态。

### 4.3 活跃态读取补强

- `chat.rs` 统一把 `waiting` 视为流式活跃态。
- 会话详情与 `get_session_events` 新增对持久化队列任务的识别：只要同 `session` 仍存在 `pending/retry/running` 任务，就保持活跃态判断。
- SSE/WS 的 `watch/resume` 恢复循环也会识别持久化队列任务，排队期不再因为 monitor 尚未进入 running 而过早断流或频繁重连。
- `compact_session` 在存在排队/执行任务时会明确拒绝，避免与实时运行态冲突。
- `is_workflow_event` 已纳入 `queue_*`，SSE watch 路径不再忽略队列事件。

### 4.4 慢客户端恢复改为显式切换

- WS 实时发送中，当出站队列接近低水位时，不再静默丢弃 `llm_output_delta`。
- 服务端会主动下发 `slow_client(reason=queue_full_resume_required)`，要求客户端改走 `resume/watch`。
- 非增量事件仍采用严格发送，保证终态、错误态和关键状态转换优先可达。

### 4.5 提交顺序保护

- `AgentRuntime` 在判断某个 `session` 是否还能直接执行时，新增对 `agent_tasks` 中 `pending/retry/running` 的检查。
- 这样可以避免“前一个请求已排队，后一个请求却直接执行”的顺序错乱。

## 5. 下一阶段优化项

### 5.1 统一 monitor 与 queue scheduler 的等待态来源

当前排队生命周期已经进入持久化流，但 monitor 的等待态来源仍然偏执行期视角。下一阶段建议补一个通用 `mark_waiting(reason)`，把“排队等待 / 审批等待 / 用户输入等待”统一收敛到同一状态机。

### 5.2 建立 chat 侧统一 realtime runtime

目前 beeroom 已较完整地走向“持久化流 + watch/replay + live push”，chat 侧仍有部分路径分别维护 WS/SSE/watchdog 逻辑。建议抽出统一 runtime，收敛：

- 连接状态机
- 心跳/重放/补偿逻辑
- 慢客户端降级策略
- 事件幂等合并与断点恢复

### 5.3 队列阶段的首包可见性优化

建议为排队请求定义稳定的“首状态包”指标，保证用户在提交后极短时间内必然看到以下之一：

- `queued`
- `queue_enter`
- `queue_start`
- `error`

并纳入监控告警。

### 5.4 增量事件批量与节流优化

对于高频 `llm_output_delta`，建议继续优化：

- 服务端按时间窗/字节窗合批
- 前端按帧批量落 UI
- 对超慢消费者优先切换回放而不是继续拥塞 live 通道

## 6. 验收指标

建议把以下指标作为实时性验收口径：

- 提交到首个可见状态变化 P95 < 300ms
- `queued -> queue_start` 的状态切换在前端可被稳定观察到，不能出现无状态空窗
- WS 慢客户端降级后，不允许出现静默缺字；必须收到 `slow_client` 或后续 `resume/watch` 补齐
- 页面刷新后，处于排队/执行中的会话能在 1 次会话详情加载 + 1 次 watch 建立内恢复状态
- 同一 `session` 下消息顺序必须与 `agent_tasks.created_at` 一致，不允许后发先执行

## 7. 建议的后续压测与观测项

- 压测 100/300/500 并发会话下的 `queued -> watch -> resume` 正常率
- 监控 `queue_enter` 到 `queue_start` 的等待分布、超时比例、失败比例
- 监控 `slow_client(reason=queue_full_resume_required)` 触发率
- 监控 `watch/resume` 回放事件数、回放耗时、重复事件去重率
- 对前端埋点“首状态可见时间”“最终回复可见时间”“恢复成功率”

## 8. 结论

本轮优化的重点，是把“排队态可感知、断线后可恢复、慢客户端不静默丢事件”先落到实处。这样即使绝对延迟没有立刻大幅下降，用户也能稳定感知到系统在持续推进，实时体验会明显改善。
