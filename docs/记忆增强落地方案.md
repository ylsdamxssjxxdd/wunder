# 记忆增强落地方案

## 1. 背景与目标

当前 `wunder` 的长期记忆能力已经具备“会话总结落盘 + 提示词注入 + 手工 CRUD”的基础闭环，但整体形态更接近“历史摘要附录”，距离“可被智能体稳定使用、可被用户理解和纠偏、可被系统持续演进”的记忆系统仍有明显差距。

当前主要问题如下：

- 记忆粒度过粗：现有记录以 `session_id -> summary` 为核心，无法表达偏好、实体、决策、案例、模式等不同记忆类型。
- 召回方式过弱：当前更偏向“将历史记忆整块拼接到 prompt”，而不是“围绕当前问题检索最相关记忆”。
- 工具可用性不足：现有 `memory_manager` 偏管理型工具，适合增删查改，不适合模型自然使用。
- 用户感知弱：用户侧没有直接可见的记忆入口，无法理解系统记住了什么、为什么记住、为什么这轮被召回。
- 生命周期缺失：缺少重要性、置信度、访问次数、确认状态、失效状态、版本链、关系链等结构信息。
- 扩展性不足：当前结构不利于后续增加全文检索、混合检索、时间线、关系图、回忆解释等增强能力。

本方案目标：

- 将“长期记忆”升级为“结构化记忆碎片系统”。
- 在用户侧智能体页面，为“设置”按钮旁新增 `记忆碎片` 按钮，提供可视化页面与卡片管理体验。
- 将记忆能力升级为 **系统自动能力 + 智能体工具 + 用户可视化页面** 的混合形态。
- 明确支持 **无嵌入模型可用** 的基础方案，并在配置了 embedding 时自动升级为更强的混合检索方案。
- 保持后端高性能、低耦合、可扩展，兼容 server / desktop 两种运行形态。

## 2. 设计原则

### 2.1 三层能力形态

记忆增强不应只做成单一工具，而应分为三层：

- **系统自动层**：在线程初始化时注入一次记忆快照，在最终回复后自动捕获；线程内后续记忆读取改由记忆工具主动检索。
- **工具层**：为智能体提供显式记忆操作入口，用于精确控制。
- **用户可视化层**：通过“记忆碎片”页面向用户展示、编辑、确认、失效、遗忘记忆。

### 2.2 不强制依赖嵌入模型

记忆能力必须支持两种运行模式：

- **Keyword 模式**：仅依赖结构化字段 + 全文/模糊检索，不依赖 embedding。
- **Hybrid 模式**：在 Keyword 基础上叠加向量召回与语义去重。

要求：

- 没有 embedding model 时，记忆系统必须完整可用。
- 有 embedding model 时，系统自动升级为更强的召回与去重策略。
- 前端页面、工具协议、核心服务接口保持一致，不因是否配置 embedding 而分叉。

### 2.3 用户优先与可解释性

- 记忆页面展示的内容必须与模型实际可用的记忆一致。
- 每次自动召回都应能解释“为什么命中”。
- 用户必须可以纠偏，包括确认、编辑、标记失效、忘记、置顶。

### 2.4 性能与稳定性优先

- 自动捕获必须异步执行，不阻塞主对话链路。
- 自动召回默认只注入 Top-N 小批量结果，避免上下文膨胀。
- 前端页面优先分页与增量加载，不允许一次性拉取全部记忆。
- 结构按职责拆分，不继续向 `src/services/tools.rs` 与 `frontend/src/views/MessengerView.vue` 堆叠逻辑。

## 3. 分期范围

## 3.1 P1（本次落地方向）

- 新增“记忆碎片”可视化入口与基础页面。
- 将现有长期记忆模型升级为结构化记忆碎片模型。
- 提供用户侧记忆列表、详情、筛选、搜索、编辑、确认、失效、忘记、置顶。
- 新增自动召回：围绕当前问题检索相关记忆，而不是整块拼接历史摘要。
- 新增回忆解释：展示本轮使用了哪些记忆，以及命中原因。
- 提供无需 embedding 的 Keyword 检索模式。
- 提供新的记忆工具接口，逐步替代 `memory_manager`。

## 3.2 P2（近程增强）

- 新增自动捕获与结构化抽取。
- 新增版本链（supersede）与关系链（relation）。
- 增加访问次数、置信度、重要度、状态流转。
- 配置 embedding 后开启 Hybrid 检索与语义去重。
- 新增时间线视图。

## 3.3 P3（中程增强）

- 增加关系图视图。
- 增加记忆层级（Peripheral / Working / Core）与衰减策略。
- 增加多 Scope 访问控制（用户 / 智能体 / 项目 / 全局）。
- 增加记忆统计面板、清理策略、重建索引与导出能力。

## 4. 用户体验方案

## 4.1 入口位置

用户侧智能体页面在“设置”按钮右侧新增 `记忆碎片` 按钮。

要求：

- 图标建议使用碎片/卡片/脑图语义，不使用过于抽象的管理类图标。
- 按钮文案固定为 `记忆碎片`，避免与“历史记录”“聊天记录”“知识库”混淆。
- 支持浅色/深色双主题，不使用 `backdrop-filter`。

建议交互：

- 点击按钮后打开独立的记忆页面或全高度侧页。
- 优先建议采用 **独立子页面** 方案，以便承载卡片流、时间线、解释面板与后续关系图。
- 推荐路由：`/app/agents/:agentId/memory`。

## 4.2 页面结构

页面建议分为四个区域：

1. **顶部摘要栏**
   - 记忆总数
   - 已置顶数量
   - 最近 7 天新增数量
   - 最近命中次数
   - 当前检索模式：`keyword` / `hybrid`

2. **搜索与筛选栏**
   - 搜索框：支持关键词搜索
   - 分类筛选：`档案 / 偏好 / 实体 / 决策 / 案例 / 模式`
   - 状态筛选：`生效 / 已确认 / 已失效 / 已置顶`
   - 来源筛选：`自动提取 / 工具写入 / 用户编辑`
   - 排序方式：`最近更新 / 最近命中 / 重要度 / 创建时间`

3. **主内容区：记忆卡片流**
   - 默认以卡片流展示
   - 支持切换为列表视图
   - 分页加载，默认每页 20 条，最大 100 条

4. **右侧详情/解释区**
   - 选中卡片后显示完整详情
   - 展示本轮是否命中、命中原因、关联记忆、来源会话、变更历史

## 4.3 卡片字段设计

单张记忆卡片建议展示以下信息：

- `标题`：L0 摘要，一句话索引，适合快速扫描。
- `分类`：档案 / 偏好 / 实体 / 决策 / 案例 / 模式。
- `摘要`：L1 概览，1~3 行。
- `来源`：来源会话 / 来源轮次 / 来源类型。
- `更新时间`。
- `最近命中时间`。
- `重要度`（0~1）与 `置信度`（0~1）。
- `状态`：active / invalid / superseded / archived。
- `标记`：已确认、已置顶、自动提取、用户修正。

字段长度建议：

- L0 标题：不超过 48 个中文字符。
- L1 摘要：不超过 180 个中文字符。
- L2 全文：默认不超过 1200 个中文字符。
- tags：最多 8 个。
- entities：最多 8 个。

## 4.4 卡片操作

每张卡片至少提供以下操作：

- `确认`：用户确认该记忆准确，提升置信度并标记为已确认。
- `编辑`：修改标题、摘要、分类、标签、重要度。
- `置顶`：标记为核心记忆，召回时加权。
- `标记失效`：保留审计痕迹，但不再参与自动召回。
- `忘记`：硬删除，面向用户侧隐私需求。
- `查看来源`：跳转到来源会话/来源消息。

说明：

- `标记失效` 与 `忘记` 必须区分。
- `偏好` 与 `实体` 类记忆在文本被修改时，应优先走“新版本替代旧版本”的 supersede 流程，而不是原地覆盖。

## 4.5 本轮命中解释

这是用户建立信任的关键能力。

页面需提供“本轮使用的记忆”面板，展示：

- 本轮被召回的记忆列表。
- 每条记忆的召回原因：
  - 关键词命中
  - 分类命中
  - 最近频繁使用
  - 已置顶
  - 语义相似（仅 hybrid 模式）
- 召回分数拆解：
  - lexical_score
  - semantic_score
  - freshness_score
  - importance_score
  - final_score

该能力既用于用户可视化，也用于内部调试。

## 5. 记忆能力形态设计

## 5.1 自动召回（系统层）

触发时机：模型主推理前。

输入：

- 当前用户问题
- 当前智能体 ID
- 当前会话 ID
- 当前轮次上下文摘要

处理流程：

1. 解析 query 与结构化约束（分类、实体、近期性关键词）。
2. 在 `user_id + agent_id` 范围内检索可用记忆。
3. 对 Top-N 候选进行排序与去重。
4. 将前 3 条（最大 5 条）注入系统上下文。
5. 记录 recall hit，供解释面板展示。

约束：

- 每轮默认注入 3 条，最大 5 条。
- 注入内容以 L0 + L1 为主，必要时附带短 L2 摘要片段。
- 不允许将全部记忆拼接进 prompt。

## 5.2 自动捕获（系统层）

触发时机：最终回复完成后异步执行。

输入：

- 本轮用户消息
- 本轮最终回复
- 工具执行摘要（如有）
- 当前用户与智能体上下文

处理流程：

1. 过滤噪声：寒暄、拒答、确认词、纯流程消息不入记忆。
2. 使用结构化提示词做记忆抽取，最多生成 3 条候选。
3. 与现有记忆做规则去重/合并。
4. 落盘并记录写入事件。
5. 通过 WS 推送前端刷新。

约束：

- 自动捕获必须异步，不阻塞最终回复。
- 单轮最多写入 3 条。
- 当抽取失败时静默降级，不影响主流程。

## 5.3 工具层

工具层保留，但不再使用“管理型大杂烩工具”作为主入口。

建议新增工具：

- `memory_recall`
- `memory_store`
- `memory_update`
- `memory_forget`
- `memory_confirm`
- `memory_pin`

现有 `memory_manager`：

- 保留兼容一段时间。
- 内部逐步改为转发到新的记忆服务。
- 后续标记为弃用。

## 6. 记忆数据模型

## 6.1 主表：`memory_fragments`

新增结构化记忆表 `memory_fragments`，建议字段如下：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `memory_id` | TEXT/UUID | 主键 |
| `user_id` | TEXT | 用户 ID |
| `agent_id` | TEXT | 智能体 ID，空值表示默认范围 |
| `session_id` | TEXT | 来源会话 |
| `source_round_id` | TEXT | 来源轮次 |
| `source_type` | TEXT | `auto/tool/user/import` |
| `category` | TEXT | `profile/preference/entity/decision/task_case/pattern` |
| `title_l0` | TEXT | 一句话索引 |
| `summary_l1` | TEXT | 结构化摘要 |
| `content_l2` | TEXT | 完整叙述 |
| `fact_key` | TEXT | 同类事实归并键 |
| `tags_json` | TEXT | 标签 JSON |
| `entities_json` | TEXT | 实体 JSON |
| `importance` | REAL | 0~1 |
| `confidence` | REAL | 0~1 |
| `tier` | TEXT | `peripheral/working/core` |
| `status` | TEXT | `active/invalid/superseded/archived` |
| `pinned` | INTEGER/BOOLEAN | 是否置顶 |
| `confirmed_by_user` | INTEGER/BOOLEAN | 是否已确认 |
| `access_count` | INTEGER | 被召回次数 |
| `hit_count` | INTEGER | 在用户可见解释中命中次数 |
| `last_accessed_at` | REAL | 最近召回时间 |
| `valid_from` | REAL | 生效时间 |
| `invalidated_at` | REAL | 失效时间 |
| `supersedes_memory_id` | TEXT | 替代旧版本 |
| `superseded_by_memory_id` | TEXT | 被新版本替代 |
| `embedding_model` | TEXT | 选填 |
| `vector_ref` | TEXT | 选填，向量索引引用 |
| `created_at` | REAL | 创建时间 |
| `updated_at` | REAL | 更新时间 |

说明：

- P1 即可落这张表。
- 原 `memory_records` 可作为迁移来源，不再继续扩展。

## 6.2 关系表：`memory_relations`

用于后续时间线与关系图。

建议字段：

- `id`
- `user_id`
- `agent_id`
- `source_memory_id`
- `target_memory_id`
- `relation_type`（如 `about/support/contradict/supersede/related_to/derived_from`）
- `weight`
- `created_at`

P1 可先建表但只使用 `supersede/related_to` 两种关系。

## 6.3 命中表：`memory_hits`

用于解释面板与排序学习。

建议字段：

- `id`
- `memory_id`
- `user_id`
- `agent_id`
- `session_id`
- `round_id`
- `query_text`
- `reason_json`
- `lexical_score`
- `semantic_score`
- `freshness_score`
- `importance_score`
- `final_score`
- `created_at`

说明：

- P1 必须落这张表，否则“本轮为何命中”无法解释。

## 6.4 作业表：`memory_jobs`

用于记录自动捕获、重建索引、迁移等后台任务。

建议字段：

- `job_id`
- `job_type`（`extract/reindex/migrate/cleanup`）
- `user_id`
- `agent_id`
- `session_id`
- `status`
- `request_payload`
- `result_summary`
- `error_message`
- `queued_at`
- `started_at`
- `finished_at`
- `updated_at`

说明：

- 当前 `memory_task_logs` 可视为旧结构，P1 可新建 `memory_jobs` 承载增强流程。

## 6.5 索引建议

基础索引：

- `idx_memory_fragments_owner_updated`：`(user_id, agent_id, updated_at DESC)`
- `idx_memory_fragments_owner_status`：`(user_id, agent_id, status)`
- `idx_memory_fragments_owner_category`：`(user_id, agent_id, category)`
- `idx_memory_fragments_owner_fact_key`：`(user_id, agent_id, fact_key)`
- `idx_memory_hits_owner_round`：`(user_id, agent_id, session_id, round_id)`

全文索引建议：

- **Desktop / SQLite**：使用 `FTS5` 为 `title_l0 + summary_l1 + content_l2 + tags` 建立全文索引。
- **Server / Postgres**：优先使用 `pg_trgm` + GIN，提升中文/短语匹配能力；如环境允许，再补充 `tsvector`。

## 7. 检索与排序策略

## 7.1 Keyword 模式（默认可用）

不依赖 embedding。

召回来源：

- L0 / L1 / L2 文本匹配
- tags / entities 命中
- category / fact_key 过滤
- 时间新鲜度与置顶权重

排序建议：

`final_score = lexical + freshness + importance + pin_boost - redundancy_penalty`

建议初始权重：

- `lexical_score`: 0.55
- `freshness_score`: 0.15
- `importance_score`: 0.15
- `confirmed_boost`: 0.05
- `pin_boost`: 0.10

## 7.2 Hybrid 模式（可选增强）

启用条件：

- 系统配置了 embedding model
- 对应向量索引可用

召回流程：

1. Keyword 检索获取候选。
2. 向量召回获取候选。
3. 融合两路候选。
4. 去重与多样性重排。

建议权重：

- `lexical_score`: 0.35
- `semantic_score`: 0.30
- `freshness_score`: 0.10
- `importance_score`: 0.10
- `confirmed_boost`: 0.05
- `pin_boost`: 0.10

说明：

- Hybrid 是增强项，不应成为主流程硬依赖。
- `wunder` 应优先复用现有 embedding 与向量知识链路，不新增独立模型调用体系。

## 7.3 去重与版本链

P1 使用规则去重：

- 同 `user_id + agent_id + fact_key + category` 命中时优先合并。
- `preference / entity` 改写时优先创建新版本，并写 `supersedes_memory_id`。
- `decision / task_case` 以追加为主，不轻易覆盖旧记录。

P2 可增加语义去重：

- 向量相似度阈值预筛。
- 对高相似候选做二次判定：`create / merge / supersede / skip`。

## 7.4 噪声过滤

以下内容默认不自动入记忆：

- 问候语、感谢语、寒暄。
- 纯确认词（“好的”“收到”“继续”）。
- Agent 自我介绍、拒绝回答、免责声明。
- 纯工具失败日志、无业务价值的过程信息。

## 8. 分类与层级设计

P1 建议采用以下 6 类：

- `profile`：稳定档案，如身份、角色、长期背景。
- `preference`：偏好，如语言、风格、习惯。
- `entity`：人、项目、文件、组织、系统名等实体。
- `decision`：明确做出的决策。
- `task_case`：阶段性案例、问题、约束与进展。
- `pattern`：长期重复出现的行为模式或协作模式。

P2 引入三层层级：

- `peripheral`：默认层，普通记忆。
- `working`：近期高频、有较强上下文价值。
- `core`：长期稳定且高价值的核心记忆。

初始晋升建议：

- `working`：访问次数 >= 3 且 `final_score_avg >= 0.55`
- `core`：访问次数 >= 8 或被用户置顶/确认，且 `importance >= 0.8`

## 9. 后端模块拆分建议

为避免继续向大文件堆逻辑，建议新增目录：`src/services/memory/`。

建议模块如下：

- `src/services/memory/mod.rs`：统一导出。
- `src/services/memory/model.rs`：数据结构与枚举。
- `src/services/memory/store.rs`：存储抽象与 DB 读写。
- `src/services/memory/retriever.rs`：Keyword / Hybrid 召回。
- `src/services/memory/extractor.rs`：自动捕获与结构化提取。
- `src/services/memory/ranker.rs`：评分、去重、重排。
- `src/services/memory/formatter.rs`：注入 prompt 与前端展示格式化。
- `src/services/memory/jobs.rs`：后台作业与任务日志。
- `src/services/memory/migrate.rs`：旧 `memory_records` 迁移。

工具接入建议：

- 在 `src/services/tools/` 下新增 `memory_tools.rs`。
- `src/services/tools.rs` 仅保留注册入口，不再承载详细实现。

API 接入建议：

- 用户侧新增：`src/api/user_memory.rs`

## 11. API 草案

用户侧 API 建议如下：

### 11.1 列表与搜索

- `GET /api/user/agents/:agent_id/memories`
  - 参数：`q/category/status/source/sort/page/page_size`
  - 返回：分页卡片数据 + 概览统计

### 11.2 详情

- `GET /api/user/agents/:agent_id/memories/:memory_id`
  - 返回：完整详情、来源、关系、版本链、最近命中

### 11.3 创建

- `POST /api/user/agents/:agent_id/memories`
  - 用于用户手工新增

### 11.4 更新

- `PATCH /api/user/agents/:agent_id/memories/:memory_id`
  - 编辑标题、摘要、分类、标签、重要度

### 11.5 确认/置顶/失效/忘记

- `POST /api/user/agents/:agent_id/memories/:memory_id/confirm`
- `POST /api/user/agents/:agent_id/memories/:memory_id/pin`
- `POST /api/user/agents/:agent_id/memories/:memory_id/invalidate`
- `DELETE /api/user/agents/:agent_id/memories/:memory_id`

### 11.6 本轮命中解释

- `GET /api/user/agents/:agent_id/memory-hits`
  - 参数：`session_id/round_id`
  - 返回：本轮命中的记忆与原因拆解

### 11.7 后台任务

- `GET /api/user/agents/:agent_id/memory-jobs`
  - 展示自动抽取与重建索引任务状态

## 13. 实施节点设计

## 13.1 节点 A：入口与只读页面

目标：先让用户“看见记忆”。

范围：

- 新增“记忆碎片”按钮。
- 新增记忆页面路由与基础布局。
- 支持分页列表、卡片展示、搜索、分类筛选。
- 支持记忆详情查看。

后端依赖：

- 最小可用列表接口。

验收：

- 用户能从智能体页点击进入记忆页面。
- 页面在浅色/深色主题下正常显示。
- 不需要 embedding 即可完成搜索与浏览。

## 13.2 节点 B：结构化存储替换

目标：把当前会话摘要模型升级为结构化记忆碎片模型。

范围：

- 增加 `memory_fragments / memory_hits / memory_jobs`。
- 新增存储抽象与读写接口。
- 将旧 `memory_records` 内容迁移为 `task_case` 类基础记录。

验收：

- 新写入的记忆全部进入 `memory_fragments`。
- 列表、详情、编辑 API 可用。
- 旧数据可迁移或可明确丢弃，不再继续扩展旧表。

## 13.3 节点 C：自动召回与解释面板

目标：让系统“真正用上记忆”。

范围：

- 每轮前自动召回 Top-N 相关记忆。
- 注入 prompt 时只注入精选记忆，不拼接全量历史。
- 记录 `memory_hits`。
- 前端新增“本轮使用的记忆”解释面板。

验收：

- 当前问题变化时，命中的记忆会跟着变化。
- 前端可看到本轮命中列表与命中原因。
- 注入 token 量可观测，且不会造成上下文失控。

## 13.4 节点 D：自动捕获与工具升级

目标：让记忆进入“自动生成 + 用户纠偏”的闭环。

范围：

- 最终回复后异步抽取最多 3 条记忆。
- 新增 `memory_recall/store/update/forget/confirm/pin`。
- 页面支持确认、置顶、失效、忘记。

验收：

- 自动捕获不阻塞主回复。
- 工具能被智能体自然调用。
- 用户能在页面中对自动提取结果纠偏。

## 13.5 节点 E：Hybrid 检索、时间线与关系图

目标：从“可用”升级到“更聪明、更可视”。

范围：

- embedding 可用时启用 hybrid 检索。
- 支持语义去重与版本链。
- 新增时间线视图。
- 新增关系图视图。

验收：

- 没有 embedding 时系统仍然稳定可用。
- 有 embedding 时召回质量明显提升。
- 时间线与关系图可用于定位偏好变化与关键决策。

## 14. 风险与回退策略

主要风险：

- 自动抽取质量不稳定，导致脏记忆。
- 召回策略过强，造成 prompt 污染。
- 页面信息过多，影响用户理解。
- 混合检索实现过早引入复杂度。

对策：

- P1 先上 Keyword 模式，Hybrid 延后。
- 自动捕获默认限量、限类、限长度。
- 前端默认卡片流 + 解释面板，不在 P1 引入关系图。
- 引入 `invalid` 与 `forget` 双通道，降低错误记忆长期残留风险。

回退策略：

- 自动捕获可通过配置一键关闭。
- 自动召回可降级为只使用置顶记忆与最近确认记忆。
- Hybrid 检索故障时自动回退到 Keyword 模式。

## 15. 验收标准

- 用户侧智能体页面在“设置”按钮旁新增 `记忆碎片` 按钮。
- 记忆页面支持卡片查看、搜索、筛选、详情、确认、编辑、失效、忘记、置顶。
- 系统在无 embedding 的情况下可完成结构化存储与 Keyword 召回。
- 系统在有 embedding 的情况下可升级为 Hybrid 检索，但不改变外部接口。
- 每轮主回复前最多自动注入 3 条（最大 5 条）相关记忆。
- 每轮主回复后最多自动提取 3 条高价值记忆，且异步执行。
- 前端可看到“本轮使用的记忆”与命中原因。
- 后端结构按模块拆分，不继续向超大文件堆叠实现。

## 16. 推荐落地顺序

建议按以下顺序实施：

1. 节点 A：先做入口与只读页面。
2. 节点 B：再做结构化存储与 API。
3. 节点 C：接入自动召回与解释面板。
4. 节点 D：补自动捕获与新工具。
5. 节点 E：最后做 Hybrid、时间线、关系图。

该顺序的好处是：

- 用户可以尽早看到价值。
- 后端改造可逐层推进，不必一次性切换所有能力。
- 在无 embedding 的条件下即可完成第一轮体验升级。
