# 聊天页面体验优化方案

## 目标
- 长时工具调用/排队执行时，页面持续显示“运行中”，不中途变为已完成。
- 同一线程可稳定对话上千轮，滚动/渲染不卡顿，消息更新保持流畅。
- 断线/切换/后台时能自动恢复，不出现重复或错位气泡。

## 现象复盘
- 长时间工具执行时，页面显示“已完成/不再运行”，但后台实际上仍在跑；工具结束后突然出现新的助手气泡。
- 连续多轮后输出越来越卡，模型输出速度未明显变慢，疑似前端渲染/通信处理瓶颈。
- 刷新页面后状态恢复正常，说明前端状态与后端真实状态存在“收敛过早/不同步”。

## 代码层问题定位（关键证据）

### 1) 运行状态收敛过早（假完成）
- `frontend/src/stores/chat.ts` 的 `sendMessage` / `resumeStream` 在 `finally` 中无条件将 `workflowStreaming=false`，并在未确认收到 `final/error` 事件时将 `stream_incomplete` 置为 `false`（仅 queued 时例外），导致 UI 认为任务已结束。
- 同文件 `setSessionLoading` 在请求结束时直接清空 `loadingBySession`，未与真实运行状态绑定。
- `startSessionWatcher` 仅会“优先复用”`stream_incomplete=true` 的助手消息；若该标记被提前清除，就会新建一条助手消息，从而出现“新气泡突然冒出”。

### 2) WS 队列语义与 SSE 不一致
- `src/api/chat_ws.rs` 在 `AgentSubmitOutcome::Queued` 时发送 `queued` 后立即发送 `final`，导致前端在任务尚未开始时就认定已完成。
- `src/api/chat.rs`（SSE）仅发送 `queued` 事件并结束连接；两种通道语义不一致，客户端难以统一处理。

### 3) 运行状态缺少服务端兜底
- 前端缺少“最终是否完成”的强一致依据，仅靠流式连接的结束判断。
- 当流式连接意外结束或排队等待较久时，前端没有定期向后端“校验状态/补拉事件”，导致假完成与错位气泡。

### 4) 长会话渲染/状态计算成本线性增长
- `frontend/src/views/MessengerView.vue` 与 `frontend/src/views/ChatView.vue` 中存在多处基于 `chatStore.messages` 的全量 `map/compute`，即便启用虚拟滚动也会在每次流式更新时对全量消息进行 O(n) 计算。
- `frontend/src/components/chat/MessageToolWorkflow.vue` 对 `workflowItems` 的解析与结构化渲染成本高，流式工具输出会触发频繁重算与 DOM 更新。
- `frontend/src/stores/chat.ts` 的快照写入（localStorage）在流式更新时频繁触发，进一步增加主线程压力。

## 优化方案（分阶段执行）

### 阶段 1：状态一致性修复（1~2 天）
目标：消除“假完成”和“新气泡突现”。

1. **统一队列语义**
   - WS：删除 `queued` 后立刻 `final` 的行为，或将 `final` 改为明确的 `queued_final`（不代表完成）。
   - SSE/WS：统一对“排队”的事件约定，客户端只在收到真正 `final/error` 后收敛。

2. **前端引入 `finalSeen` / `errorSeen` 标志**
   - `sendMessage`/`resumeStream`：仅在确认 `final/error` 后清理 `stream_incomplete`。
   - 若流结束但未见 `final/error`，保持 `stream_incomplete=true`，并进入“等待恢复”状态。

3. **会话运行状态从“事件驱动”转为“可恢复状态机”**
   - `loadingBySession` 不再仅依赖请求结束，而是基于：
     - 任意消息 `stream_incomplete/workflowStreaming/reasoningStreaming`
     - 或服务端监控状态（running/waiting）。

4. **Watcher 复用逻辑增强**
   - `startSessionWatcher` 除了 `stream_incomplete`，还需尝试按 `stream_event_id/round/时间窗口` 复用最近一条助手消息，避免创建新气泡。

**验收标准**
- 长工具运行时状态持续显示“运行中”。
- 工具结束后不会出现多余助手气泡。
- 刷新前后状态一致。

---

### 阶段 2：通信健壮性与自恢复（1 周）
目标：断线/排队/长静默时自动保持同步。

1. **加入心跳/进度事件**
   - 服务端在工具执行超过阈值时周期性 `progress/heartbeat`，保持前端“活跃”态。

2. **流式 Watchdog**
   - 前端若 N 秒无事件且未见 `final`，自动触发 `resume/watch` 或请求 session 状态。

3. **运行状态轻量查询接口**
   - 增加 `GET /wunder/chat/sessions/{id}/status`（或在现有 session detail 中暴露运行态），前端定期校验。

**验收标准**
- 断网/切换后能自动恢复，状态不漂移。
- 排队执行的请求能在开始时自动进入“运行中”。

---

### 阶段 3：长会话性能优化（2~4 周）
目标：上千轮仍可流畅滚动与更新。

1. **消息渲染去 O(n)**
   - 把 `chatStore.messages` 的全量 map/compute 改为“增量式窗口”计算，避免每次 token 更新遍历全量消息。

2. **工具工作流懒解析**
   - `MessageToolWorkflow` 仅在展开时解析 `detail`，并为 `tool_output_delta` 做节流（例如 100~200ms 更新一次）。

3. **历史消息分页/懒加载**
   - 仅在视口内保留最近 N 轮消息，旧消息按需加载；与服务端历史接口对接。

4. **快照写入降频**
   - `localStorage` 写入改为 `requestIdleCallback` 或更大时间间隔，并限制快照体积。

**验收标准**
- 1000+ 轮对话时，滚动/输入/输出保持可用流畅。
- CPU/内存随轮次增长趋于平稳。

---

## 参考 Codex 插件的可借鉴点
- 运行状态由服务端驱动（状态机式），而不是仅靠流结束判断。
- 断线后自动补拉事件，避免状态“收敛过早”。
- 输出渲染节流与懒展开，避免工具日志导致 DOM 爆炸。

## 里程碑节点
- **节点 A（稳定性热修）**：
  - 修正 WS 队列事件语义
  - `finalSeen/errorSeen` 状态机
  - watcher 复用逻辑增强

- **节点 B（通信健壮）**：
  - 心跳/进度事件
  - Watchdog 自动恢复
  - 轻量状态查询接口

- **节点 C（性能扩展）**：
  - 消息增量窗口
  - 工作流懒解析+节流
  - 历史分页与快照降频
