# 工具返回内容优化表

## 目标

这份文档描述 wunder 当前的工具返回设计，重点不是“理想上应该怎样”，而是“现在实际上怎样组织”。

当前设计的核心原则只有三条：

- 工具层原始回包要保留足够结构，便于前端、重试治理、工作流和排障使用。
- 面向模型的 `model_observation` 必须更短、更稳、更可继续推理，不能把原始回包原样塞回模型。
- 失败信息必须结构化，但不能重复展开同一份错误。

## 总览

当前实际是三层结构，而不是单一返回格式：

1. 工具原始结果
2. `tool_result` 事件负载
3. `model_observation`

这三层用途不同，不能混看。

| 层级 | 面向对象 | 主要用途 | 当前形态 |
| --- | --- | --- | --- |
| 工具原始结果 | 工具实现层 / 编排层 | 保留业务结果与错误元信息 | 成功结果按工具类型而异；失败大量走统一失败包 |
| `tool_result` 事件 | 前端工作流 / 日志 / 排障 | 保留结构化结果，便于展示与关联 | `tool/ok/data` 为主，失败时附带 `error/error_code/retryable`，并保留 `meta` |
| `model_observation` | 大模型 | 继续推理 | 强压缩；成功保留简洁 `data`，失败只保留一条顶层错误 |

## 1. 工具原始结果

### 1.1 成功结果

成功结果并未完全统一成单一骨架。

当前主流趋势是：

```json
{
  "ok": true,
  "action": "search_content",
  "state": "completed",
  "summary": "Found 3 matches in 2 files.",
  "data": {
    "count": 3,
    "hits": []
  }
}
```

但仍然存在例外：

- 某些工具直接返回业务对象，而不是统一成功包。
- `browser`、`web_fetch` 等工具成功结构仍保留其领域特征。
- 因此读取工具成功结果时，业务代码应优先从 `data` 读，再做兼容回退。

### 1.2 失败结果

基础工具失败现在大量走统一失败包，由 `build_failed_tool_result` 生成：

```json
{
  "ok": false,
  "data": {
    "...": "...",
    "error_meta": {
      "code": "TOOL_EXEC_NON_ZERO_EXIT",
      "hint": "命令返回非 0，请先根据 stderr 修正后再重试。",
      "retryable": false,
      "retry_after_ms": null
    }
  },
  "error": "命令退出码 1。",
  "error_meta": {
    "code": "TOOL_EXEC_NON_ZERO_EXIT",
    "hint": "命令返回非 0，请先根据 stderr 修正后再重试。",
    "retryable": false,
    "retry_after_ms": null
  },
  "sandbox": true
}
```

当前约束：

- `error_meta` 必须同时出现在顶层和 `data.error_meta`。
- `error_meta` 当前核心字段是 `code/hint/retryable/retry_after_ms`。
- 工具自身可以在 `data` 里保留必要业务上下文，例如失败命令、退出码、精简 `stderr`。

## 2. 编排层归一化

工具原始结果进入编排层后，会经过归一化与裁剪。

关键动作：

- 统一推导 `error_code`
- 统一推导 `retryable`
- 补齐业务失败语义
- 保留 `meta.error_code` 与 `meta.error_retryable`
- 对大结果做全局裁剪

因此，最终前端事件和模型看到的内容，可能与工具函数刚返回时不完全相同。

## 3. `tool_result` 事件

`tool_result` 是给前端、工作流、日志和排障看的，不是给模型看的。

当前设计重点：

- 尽量保留结构化 `data`
- 保留 `meta`
- 顶层补充 `error/error_code/retryable`
- 附带 `model_observation` 文本，便于直接比对“模型实际看到了什么”

典型形态：

```json
{
  "tool": "执行命令",
  "ok": false,
  "data": {
    "command": "python3 draw_heart.py",
    "returncode": 1,
    "stderr": "IndentationError: unindent does not match any outer indentation level",
    "error_meta": {
      "code": "TOOL_EXEC_NON_ZERO_EXIT",
      "hint": "命令返回非 0，请先根据 stderr 修正后再重试。",
      "retryable": false,
      "retry_after_ms": null
    }
  },
  "error": "命令退出码 1。",
  "error_code": "TOOL_EXEC_NON_ZERO_EXIT",
  "retryable": false,
  "meta": {
    "error_code": "TOOL_EXEC_NON_ZERO_EXIT",
    "error_retryable": false
  },
  "model_observation": "{\"tool\":\"执行命令\",\"ok\":false,\"error\":\"命令退出码 1。 stderr: IndentationError: unindent does not match any outer indentation level\",\"error_code\":\"TOOL_EXEC_NON_ZERO_EXIT\",\"retryable\":false}"
}
```

适合前端展示的内容：

- 原始业务结果：看 `data`
- 失败原因：优先看 `error`，再结合 `error_code`
- 自动恢复判断：看 `retryable`

## 4. `model_observation`

`model_observation` 是真正送回模型继续推理的 observation。

它的目标不是“完整”，而是“足够决策”。

### 4.1 成功时

成功 observation 仍保留 `data`，但会做明显压缩：

- 大数组改写成 `*_jsonl + *_count`
- 大文本截断
- 搜索、读文件、表格等类型做定制 compact
- 续取信号上提到顶层，例如 `continuation_required/continuation_hint`
- `meta` 默认移除

示例：

```json
{
  "tool": "search_content",
  "ok": true,
  "data": {
    "query": "token_balance",
    "hits_count": 3,
    "hits_jsonl": "{\"path\":\"src/a.rs\",\"line\":12,\"content_head\":\"...\"}"
  },
  "continuation_required": true,
  "continuation_hint": "result_truncated_continue_with_pagination_or_narrower_query"
}
```

### 4.2 失败时

失败 observation 现在走单独的精简分支，不再保留 `data`。

当前失败 observation 的目标形态：

```json
{
  "tool": "ptc",
  "ok": false,
  "error": "Script preflight blocked: non-standard Python indentation detected. Line 35 uses indentation width 11.",
  "error_code": "PRECHECK_PYTHON_INDENTATION",
  "retryable": false
}
```

当前明确不再放进 `model_observation` 的失败噪声：

- `data`
- `data.error_meta`
- `data.preflight`
- 顶层 `preflight`
- `sandbox`
- `duration_ms`

这样做的原因：

- 同一份错误不能重复两到三次
- 模型不需要看到完整失败包才能修正下一步
- preflight 拒绝、命令失败、读文件失败都应尽量压成“一条主错误 + 错误码 + 是否可重试”

## 5. 现在推荐怎么设计工具返回

### 成功返回

优先使用：

```json
{
  "ok": true,
  "action": "xxx",
  "state": "completed",
  "summary": "一句话摘要",
  "data": {}
}
```

但要接受现实：

- 不是所有历史工具都已完全统一
- 新工具尽量对齐这个骨架
- 旧工具至少保证 `data` 可稳定读取

### 失败返回

优先使用统一失败包：

```json
{
  "ok": false,
  "data": {
    "...": "...",
    "error_meta": {
      "code": "XXX",
      "hint": "下一步怎么修",
      "retryable": false,
      "retry_after_ms": null
    }
  },
  "error": "一条简洁主错误",
  "error_meta": {
    "code": "XXX",
    "hint": "下一步怎么修",
    "retryable": false,
    "retry_after_ms": null
  }
}
```

失败文案的约束：

- `error` 只写一条主错误
- `hint` 放在 `error_meta`，不要和 `error` 重复大段展开
- `data` 只保留纠错必需上下文

## 6. 常见反模式

| 反模式 | 问题 | 现在的处理原则 |
| --- | --- | --- |
| 同一份错误同时写进 `error`、`data.error`、`data.content`、`preflight.summary` | 模型和前端都被噪声淹没 | 保留一条主错误，其余只留必要结构化字段 |
| 把完整失败包直接送进 `model_observation` | 浪费上下文 | 失败 observation 只保留顶层 `error/error_code/retryable` |
| 为了排障把所有内部字段默认暴露给模型 | 让模型看到无关实现细节 | 原始细节留在 `tool_result.data`，不要默认进入 observation |
| 返回完整大数组/完整大文本 | 下一轮上下文膨胀 | 转 `*_jsonl`、截断、抽样、附带 continuation 信号 |
| 错误只有一句“调用失败” | 模型无法自修复 | 至少给 `error + error_meta.code + retryable` |
| `error_meta` 只放顶层不放 `data` | 前端、重试治理、归一化看到的语义不一致 | 顶层和 `data.error_meta` 同步保留 |

## 7. 不同场景推荐保留什么

| 场景 | `tool_result.data` 推荐保留 | `model_observation` 推荐保留 |
| --- | --- | --- |
| 读取文件成功 | `content/files/read_lines/total_lines` 的必要子集 | 精简 `content`，必要时 `files` 概览和 continuation |
| 搜索成功 | `query/hits/matches/summary` | `hits_jsonl/matches_jsonl` 与计数 |
| 执行命令失败 | `command/returncode/stderr/stdout/error_meta` | 一条 `error`，附 `error_code/retryable` |
| preflight 拒绝 | `preflight/error_meta` | 一条 `error`，附 `error_code/retryable` |
| 二进制文件误读 | `path/error_meta` | 一条 `error`，附 `error_code/retryable` |
| 超长结果被裁剪 | 原始裁剪结果 + 必要标记 | 顶层 `continuation_required/continuation_hint` |

## 8. 排障顺序

排查工具问题时，顺序必须固定：

1. 先看 `tool_result.data`
2. 再看 `model_observation`
3. 最后看前端展示和 monitor 裁剪

不要倒过来。

原因：

- `tool_result.data` 更接近真实工具结果
- `model_observation` 是面向模型的压缩版本
- UI/monitor 可能还有额外展示裁剪

## 9. 当前结论

如果问“wunder 现在是这样设计的吗”，更准确的回答是：

- 成功结果正在逐步统一到 `ok/action/state/summary/data`
- 失败结果在工具层已经大规模统一到 `build_failed_tool_result`
- 前端事件和模型 observation 已明确分层
- `model_observation` 失败场景现已进一步精简，不再保留冗余 `data/preflight`

所以，这份表不应再把“工具返回”理解成单层 JSON，而应理解成：

- 工具原始结果尽量结构化
- `tool_result` 负责可观测性
- `model_observation` 负责低噪声推理
