# 工具返回内容优化表

## 目标

本表用于收敛 wunder 工具执行完成后返回给模型的内容结构。

核心目标只有三点：

- 让模型一眼知道这次调用是否成功。
- 让模型只看到对下一步决策真正有帮助的信息。
- 让错误信息具备纠错性，而不是只暴露实现细节。

## 总原则

| 项 | 现状问题 | 优化原则 |
| --- | --- | --- |
| 成功信号 | `ok/success/status/result` 多套并存 | 统一保留 `ok=true`，再配合 `action/state/summary` |
| 结果正文 | `message/summary/output/result/data/detail` 并存 | 统一保留 `summary` 和 `data` 两层 |
| 动作回显 | 返回整份原始请求体，冗余很高 | 仅回显本次动作判断所需的关键目标字段 |
| 错误表达 | 自然语言报错过散，难以纠错 | 统一结构化错误：`ok=false` + `error` + `meta` |
| 调试细节 | 默认把内部路径、缓存、耗时细节都回传 | 默认不回传，仅在 debug 或管理员场景开放 |
| 大对象回传 | 全量日志、全量列表、完整 HTML/JSON 直接塞回模型 | 默认返回摘要、计数、截断片段和继续读取锚点 |
| 下一步衔接 | 工具返回结束后没有指路 | 需要串联的工具补 `next_step_hint` |

## 推荐骨架

### 成功返回

```json
{
  "ok": true,
  "action": "search_content",
  "state": "completed",
  "summary": "Found 3 matches in 2 files.",
  "data": {
    "count": 3,
    "files": ["src/a.rs", "src/b.rs"]
  }
}
```

### 成功但需要提示下一步

```json
{
  "ok": true,
  "action": "subagent_wait",
  "state": "running",
  "summary": "2 subagents are still running.",
  "data": {
    "running": 2,
    "completed": 1
  },
  "next_step_hint": "Call subagent_control status or wait again after a short delay."
}
```

### 失败返回

```json
{
  "ok": false,
  "error": "Missing required field: session_id",
  "data": {
    "action": "history"
  },
  "meta": {
    "code": "missing_field",
    "retryable": true,
    "missing_fields": ["session_id"],
    "example": {
      "action": "history",
      "session_id": "sess_xxx"
    }
  }
}
```

## 字段设计约束

| 字段 | 作用 | 约束 |
| --- | --- | --- |
| `ok` | 成功/失败主信号 | 必须保留，且只承担成功判断，不复用语义 |
| `action` | 本次动作语义 | 使用 canonical 动作名，不混别名 |
| `state` | 当前结果状态 | 推荐枚举：`completed/accepted/running/yielded/awaiting_input/dry_run` |
| `summary` | 给模型看的短摘要 | 1 句话，描述结果而不是复述请求 |
| `data` | 结构化主结果 | 只保留决策所需字段，不做后台世界全量镜像 |
| `next_step_hint` | 下一步提示 | 仅在链式工具或异步工具里返回 |
| `error` | 失败时的人类可读信息 | 简洁、可纠错，不泄露项目意图 |
| `meta` | 错误补充上下文 | 放 `code/retryable/missing_fields/example/detail` 等结构化内容 |

## 常见反模式

| 反模式 | 问题 | 替代做法 |
| --- | --- | --- |
| 返回整份请求体 | 浪费上下文，还会重复兼容别名 | 只回显关键目标，如 `session_id/action/path` |
| 同时返回 `status=ok` 和 `ok=true` | 让模型需要猜哪一个是主信号 | 只保留 `ok=true`，状态用 `state` |
| 同时返回 `result/output/data/detail` | 结果入口不稳定 | 统一放入 `data` |
| 把后台管理字段直接塞进成功结果 | 模型容易被无关信息干扰 | 管理字段仅保留到管理员 UI 或 debug 输出 |
| 失败时只说 “调用失败” | 模型无法自修复 | 带上 `code/retryable/missing_fields/example` |
| 全量回传长列表或大文本 | 容易造成下一轮上下文膨胀 | 返回 `count + items(head) + truncated` |

## 结果体优化表

| 类型 | 推荐返回重点 | 不建议默认返回 |
| --- | --- | --- |
| 文件读取 | `path/start_line/end_line/content/truncated` | 文件 stat 全量信息、内部根路径解析细节 |
| 搜索 | `query/path/count/hits/truncated` | 搜索计划全部候选、重复预算明细 |
| 命令执行 | `results[].command/returncode/stdout/stderr/truncated` | 过多捕获预算对象、沙箱内部实现细节 |
| 写文件 | `path/bytes_written/created_or_overwritten` | 兼容别名、请求镜像 |
| 补丁编辑 | `files_changed/created/deleted` | 解析中间态、整段 patch echo |
| 会话治理 | `session_id/status/updated` | 原始数据库记录、无关父子链细节 |
| 子智能体 | `items[].session_id/run_id/status/summary` | 兄弟分支内部调度细节、兼容命名字段 |
| 蜂群 | `task_id/run_id/session_id/state` | 母蜂内部实现状态、额外缓存字段 |
| 渠道发送 | `channel/account_id/outbox_id/status` | 大量联系人缓存或渠道底层对象 |
| 长期记忆 | `action/count/items[].id/title/summary` | 数据库存储字段、检索算法细节 |
| 知识检索 | `knowledge_base/count/documents[].content/score/reason` | 向量实现细节、完整原始 chunk 元数据 |
| 桌面控制 | `action/description/screenshot_path/followup_prompt` | 底层设备状态、临时路径上下文 |

## 错误返回优化表

| 错误类型 | 推荐信息 | 示例 |
| --- | --- | --- |
| 缺字段 | `missing_fields + example` | 缺少 `session_id` 时直接给最小可用调用例子 |
| 参数冲突 | `code=invalid_combination` + 冲突字段 | 同时传 `session_id` 与不兼容的 `dispatch_id` |
| 资源不存在 | `code=not_found` + 目标标识 | 会话不存在、技能不存在、文件不存在 |
| 临时失败 | `retryable=true` + 建议下一步 | 网络发送失败、等待超时、锁冲突 |
| 权限失败 | `retryable=false` + 权限边界描述 | 当前用户不能访问该资源 |
| 输出截断 | `truncated=true` + `omitted_bytes` | 让模型知道需要继续读，而不是误判信息不完整 |

## 适用到当前 wunder 的落地方向

| 方向 | 当前建议 |
| --- | --- |
| 统一成功包 | 全部模型侧工具逐步统一到 `ok/action/state/summary/data` |
| 统一失败包 | 全部失败结果统一走 `build_failed_tool_result` |
| 工具摘要 | 摘要统一写“结果”，不要写成提示词或内部实现说明 |
| 大对象压缩 | 命令、知识、渠道、子智能体结果优先做 compact view |
| 异步工具 | `accepted/running/completed` 明确区分，不混成一个 `ok` |
| 调试扩展 | debug 版额外字段不进入默认结果体，避免污染模型上下文 |
