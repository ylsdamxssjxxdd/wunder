# 计划拆解

用于把"定个目标"升级为"可执行、可追踪、可考核"的正式计划体系。

## 核心目标

- 先明确计划目标，再分解任务，不要带着模糊目标盲目拆解。
- 先区分责任归属，再分配任务，不要把责任人和执行人混为一谈。
- 先形成计划文档，再输出交付，不要只给口头总结。
- 最终产物应尽量转换为正式DOCX文档，便于归档和流转。


## 强制原则

- 不要把"计划拆解"理解成"随便列几个任务"。
- 每个任务必须有明确的责任人、截止日期和交付物。
- 结论必须能追溯到上级目标：目标→里程碑→任务→执行项。
- 在开始拆解前，只允许做一轮澄清问询；要把关键问题一次问全，不要多轮追问。
- 计划文档要符合正式公文规范，便于后续转换为DOCX。

## 计划模式选择

收到任务后，先判断属于哪种模式：

### A. 年度计划模式

满足任一条件就进入：

- 用户明确说"年度计划/年度目标"
- 涉及单位/部门的年度战略目标
- 时间跨度覆盖完整年度

此模式下按"年度→季度→月度→周度"四级分解。

### B. 项目计划模式

满足任一条件就进入：

- 用户明确说"项目计划/实施方案"
- 有明确的起止时间和交付物
- 需要多部门/多人协同完成

此模式下按"阶段→里程碑→任务→活动"四级分解。

### C. 专项计划模式

满足任一条件就进入：

- 用户明确说"专项计划/工作计划"
- 针对特定事项的短期计划
- 时间跨度在季度以内

此模式下按"目标→任务→活动"三级分解。

## 标准流程

每次计划拆解按以下阶段推进。

### Phase 0: 任务澄清

先判断信息是否足够。如果不够，先进行且仅进行一轮澄清问询，再开始拆解。

这一轮澄清必须尽量一次问全，至少覆盖以下要素：

- 计划类型：年度计划、项目计划、还是专项计划
- 计划目标：要达成什么结果
- 时间范围：起止时间、关键节点
- 资源约束：人力、预算、外部依赖
- 输出形式：是否需要DOCX格式
- 责任主体：涉及哪些部门/人员

推荐把问题合并成一轮统一提问，例如：

- 这次计划的类型、时间范围和核心目标分别是什么？
- 涉及哪些责任部门或人员？有什么资源约束？
- 是否需要输出正式DOCX文档？

如果用户这一轮回复后仍不完整，不要继续连环追问。应当：

- 基于现有信息做最保守、最明确的工作假设
- 在计划文档中写清楚这些假设
- 在最终交付时明确说明信息缺口

在完成这一轮澄清后，把用户需求压缩成 4 个核心要素：

- 计划类型：年度/项目/专项
- 计划目标：要达成什么
- 时间范围：起止时间
- 输出形式：Markdown + DOCX

然后在工作区先写一个计划概要文件：

- 推荐路径：`plan/00-计划概要.md`
- 内容至少包括：
  - 计划名称
  - 核心目标
  - 时间范围
  - 责任主体
  - 资源约束
  - 预期产物

### Phase 1: 目标规范化

使用 SMART 原则对每个目标进行规范化：

- **S**pecific（具体）：明确目标内涵与边界
- **M**easurable（可衡量）：设定量化指标与验收标准
- **A**chievable（可达成）：评估资源匹配度与可行性
- **R**elevant（相关性）：与上级战略/单位使命对齐
- **T**ime-bound（有时限）：明确起止时间与关键节点

详细原则见 `{{SKILL_ROOT}}/references/smart-principle.md`。

规范化后的目标示例：

```
原目标：提高工作效率
SMART化：在2026年12月31日前，将项目交付周期从30天缩短至20天，通过优化流程和引入自动化工具实现
```

### Phase 2: 层级化分解

根据计划类型选择分解层级：

#### 年度计划分解

```
年度目标（战略层）
├── Q1 里程碑
│   ├── 1月任务
│   ├── 2月任务
│   └── 3月任务
├── Q2 里程碑
│   ├── 4月任务
│   ├── 5月任务
│   └── 6月任务
├── Q3 里程碑
│   └── ...
└── Q4 里程碑
    └── ...
```

#### 项目计划分解

```
项目目标
├── 阶段1：启动阶段
│   └── 里程碑1.1
├── 阶段2：实施阶段
│   ├── 里程碑2.1
│   └── 里程碑2.2
├── 阶段3：收尾阶段
│   └── 里程碑3.1
└── ...
```

#### 专项计划分解

```
专项目标
├── 任务1
│   └── 活动1.1, 活动1.2
├── 任务2
│   └── 活动2.1, 活动2.2
└── 任务3
    └── ...
```

### Phase 3: 责任矩阵

为每个任务明确 RACI 责任：

| 代码 | 含义 | 说明 |
|------|------|------|
| R | Responsible | 执行者，实际完成任务的人/团队 |
| A | Accountable | 审批者，对结果负责并有权决策的人 |
| C | Consulted | 咨询方，提供意见或信息的人/部门 |
| I | Informed | 知会方，需要知晓结果的人/部门 |

核心规则：

- 每个任务必须有且仅有一个 A（审批者）
- R 可以有多个，但建议单一
- C 和 I 可有多个或没有

详细说明见 `{{SKILL_ROOT}}/references/raci-matrix.md`。

### Phase 4: 资源匹配

评估任务所需资源：

| 资源类型 | 需求量 | 可用量 | 缺口 | 解决方案 |
|----------|--------|--------|------|----------|
| 人力 | [量] | [量] | [量] | [方案] |
| 预算 | [量] | [量] | [量] | [方案] |
| 设备 | [量] | [量] | [量] | [方案] |
| 外部依赖 | [项] | [项] | [项] | [方案] |

### Phase 5: 风险预案

识别关键风险并制定应对措施：

| 风险项 | 影响程度 | 发生概率 | 预警指标 | 应对措施 |
|--------|----------|----------|----------|----------|
| [风险1] | 高/中/低 | 高/中/低 | [指标] | [措施] |
| [风险2] | 高/中/低 | 高/中/低 | [指标] | [措施] |

### Phase 6: 文档生成

在些文档前，如果有可用的公文写作技能，务必要先调用它获取写作的规范格式
最终计划文档默认写入：

- `plan/计划文档.md`

先判断计划类型和交付场景，再选择对应模板：

- 年度计划：使用 `{{SKILL_ROOT}}/templates/年度目标模板.md`
- 季度计划：使用 `{{SKILL_ROOT}}/templates/季度里程碑模板.md`
- 月度/专项计划：使用 `{{SKILL_ROOT}}/templates/月度任务模板.md`

报告写作前必须先读取已选定的模板全文。

#### Markdown 版式规则

如果最终产物需要进入正式文稿流转，或后续要交给 `公文写作` 技能转换 DOCX，必须遵守以下 Markdown 版式规则：

- 第一行直接写计划标题，不要在第一行写 `#`。
- 从正文开始，`#` 只用于一级标题，`##` 只用于二级标题；不要手写"一、""（一）"等同层级编号，交给转换器处理。
- 表格前一行和后一行都必须留空行，表格标题应单独占一行，不要和表格写在同一行。
- 不要使用 `---`、`***` 这类分隔线分段。
- 不要把连续元信息写成无序列表，尤其是责任人、时间、资源等字段。
- 严格禁止使用 `-` 来列举干条，要写成自然段落或表格形式。

### Phase 7: DOCX 导出

**重要：最终计划文档应尽量转换为 DOCX 格式。**

- 计划文档写入 `plan/计划文档.md` 后，如果当前环境已挂载或可调用 `公文写作` 技能，**必须**继续调用该技能把最终 Markdown 计划导出为 DOCX。
- DOCX 导出应以最终计划正文为唯一主输入，不要把内部草稿、分析过程混入正文。
- 导出前先检查 Markdown 是否满足公文写作技能的版式约束；不满足时先修正 Markdown，再执行导出。
- 如果用户明确拒绝 DOCX 格式，则保留 Markdown 版本。
- 成功导出后，最终回复同时给出 Markdown 路径和 DOCX 路径。


### Phase 8: 交付检查

提交前自检：

- [ ] 是否回答了用户真正的问题，而不是只堆任务
- [ ] 每个任务是否有责任人、截止日期、交付物
- [ ] 是否建立了责任矩阵（RACI）
- [ ] 是否识别了资源缺口和风险
- [ ] 是否已调用公文写作技能转换为 DOCX
- [ ] 如果未转换 DOCX，是否说明了原因

## 推荐输出物

计划拆解默认落这 2 个文件：

- `plan/00-计划概要.md`
- `plan/计划文档.md`（以及对应的 `.docx`）

除非用户明确要求，不要另写责任矩阵单独文件、资源分析单独文件、风险预案单独文件等。这些内容应整合进最终计划文档。

## 验收标准

- ✅ 目标符合 SMART 原则
- ✅ 任务分解粒度适当（一周内可完成）
- ✅ 每个任务有责任人和截止日期
- ✅ 建立了 RACI 责任矩阵
- ✅ 识别了资源缺口和关键风险
- ✅ 已转换为 DOCX 格式（除非用户拒绝）

## 常见错误

- 把"计划拆解"当成"随便列几个任务"
- 没有澄清就直接开始拆解
- 任务粒度过大或过小
- 责任人不明确或多人同为责任人
- 只给任务清单，不给资源分析和风险预案
- 未调用公文写作技能转换为 DOCX
- Markdown 格式不规范，导致 DOCX 转换后格式混乱

## 最小执行范式

1. 如信息不足，先进行一轮统一澄清，问清计划类型、时间范围和核心目标
2. 写 `plan/00-计划概要.md`
3. 使用 SMART 原则规范化目标
4. 按层级分解任务（年度→季度→月度 或 项目→阶段→里程碑）
5. 建立 RACI 责任矩阵
6. 评估资源匹配和风险预案
7. 写 `plan/计划文档.md`，从写作阶段就确保格式符合 DOCX 转换要求
8. **必须调用 `公文写作` 技能将计划文档导出为 DOCX**（除非用户明确拒绝）
9. 回复用户时说明：
   - 计划文档已保存到哪里
   - DOCX 是否已导出，导出路径是什么；如未导出，说明原因
   - 关键里程碑和风险提示
