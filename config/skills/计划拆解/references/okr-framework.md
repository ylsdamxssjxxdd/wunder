# OKR 框架

OKR（Objectives and Key Results）是目标与关键成果管理框架，适用于战略目标分解与进度追踪。

## OKR 基本结构

```
Objective（目标）
├── Key Result 1（关键成果1）
├── Key Result 2（关键成果2）
└── Key Result 3（关键成果3）
```

### Objective（目标）

- **定性描述**：我们要达成什么？
- **特征**：鼓舞人心、有挑战性、方向明确
- **周期**：通常按季度设定

### Key Result（关键成果）

- **定量描述**：如何衡量目标达成？
- **特征**：可量化、有时限、有挑战但可达
- **数量**：每个目标 2-5 个关键成果

## OKR 示例

### 示例1：产品团队

```
O：打造用户喜爱的产品体验

KR1：用户满意度从 85% 提升至 92%
KR2：核心功能日活跃用户增长 30%
KR3：用户反馈响应时间缩短至 4 小时以内
```

### 示例2：销售团队

```
O：实现市场份额突破性增长

KR1：新增签约客户 50 家
KR2：客户续约率达到 95%
KR3：单客户平均营收提升 20%
```

### 示例3：技术团队

```
O：构建稳定高效的技术底座

KR1：系统可用性达到 99.9%
KR2：核心接口响应时间 < 200ms
KR3：完成 3 项核心技术债务清理
```

## OKR 与 KPI 的区别

| 维度 | OKR | KPI |
|------|-----|-----|
| 性质 | 目标管理工具 | 绩效考核工具 |
| 周期 | 季度/年度 | 月度/季度/年度 |
| 目标值 | 有挑战性（70%达成即成功） | 必须达成 |
| 方向 | 关注增长与突破 | 关注维持与达标 |
| 透明度 | 全员公开 | 通常层级可见 |
| 用途 | 对齐与聚焦 | 考核与激励 |

## OKR 制定原则

### 原则1：聚焦

- 每个周期 3-5 个 Objective
- 每个 Objective 2-5 个 Key Result
- 避免目标过多导致精力分散

### 原则2：对齐

- 自上而下：公司 → 部门 → 个人
- 自下而上：个人目标支撑部门目标
- 横向协同：跨部门目标相互支持

### 原则3：挑战

- 目标应有挑战性（"射月"目标）
- 70% 达成率视为成功
- 100% 达成可能目标过低

### 原则4：透明

- 全员可见所有 OKR
- 定期同步进度
- 促进跨团队协作

### 原则5：追踪

- 周度检查进度
- 月度复盘调整
- 季度评分总结

## OKR 评分

### 评分标准

| 分数 | 含义 | 说明 |
|------|------|------|
| 0.0-0.3 | 未达成 | 目标过高或执行不力 |
| 0.4-0.6 | 部分达成 | 正常挑战水平 |
| 0.7-1.0 | 达成/超额 | 目标可能偏低 |

### 评分公式

```
OKR 得分 = 实际值 / 目标值

示例：
目标：新增客户 100 家
实际：新增客户 75 家
得分：75 / 100 = 0.75（良好）
```

## OKR 制定流程

```
1. 设定周期目标（季初）
   ↓
2. 分解关键成果
   ↓
3. 对齐上下级 OKR
   ↓
4. 周度进度同步
   ↓
5. 月度复盘调整
   ↓
6. 季末评分总结
   ↓
7. 下一周期设定
```

## 常见误区

### 误区1：OKR 当 KPI 用

- OKR 用于目标聚焦，不是考核工具
- 不要直接与绩效奖金挂钩

### 误区2：目标过多

- 10 个目标 = 没有目标
- 聚焦 3-5 个真正重要的目标

### 误区3：关键成果不可量化

- "提升用户体验" 不是合格的 KR
- "用户满意度从 80% 提升至 90%" 才是

### 误区4：目标过低

- 100% 达成说明目标太简单
- 挑战性目标才能激发潜力

### 误区5：设定后不管

- OKR 需要持续追踪
- 周检查、月复盘、季总结

## OKR 与年度计划拆解的结合

年度计划拆解可借鉴 OKR 思路：

1. **年度目标** → 年度 Objective
2. **季度里程碑** → 季度 Key Result
3. **月度任务** → 月度执行计划
4. **周度执行项** → 周度行动清单

通过 OKR 框架，确保目标层层分解、上下对齐、进度可视。
