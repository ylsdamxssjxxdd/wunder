# 深度研究

用于把“随手查一下”升级为“可复核、可追溯、可交付”的正式研究流程。

## 核心目标

- 先明确研究问题，再收集资料，不要带着模糊目标盲搜。
- 先区分资料层级，再下结论，不要把传闻、二手转述、原始材料混为一谈。
- 先形成研究记录，再输出报告，不要只给口头总结。
- 网络不可用时，默认进入离线优先模式：优先吃透工作区、知识库、用户上传文件、现有草稿和历史产物。

## 什么时候触发

出现以下意图时应主动使用本技能：

- “帮我做个深入研究/深度调研/系统研究”
- “先收集资料，再给我完整报告”
- “比较 A 和 B，做得深入一点”
- “研究某行业/公司/人物/技术/政策/产品”
- “帮我写研究报告/分析报告/尽调报告/学术性综述/立项研究”
- “基于我给你的材料整理结论”
- “现在没网，先用本地资料研究”

如果用户只是要一个很短的常识回答，不需要触发。

## 工具优先级

研究时按下面顺序选工具，不要反过来：

1. `skill_call`
   - 先读当前技能和其他已挂载技能，确认有没有更专业的资料入口或专用流程。
2. 工作区文件工具：`list_files` -> `search_content` -> `read_file`
   - 本地资料定位、筛选、定点阅读的主路径。
3. 知识库工具：`kb_query` 或 `kb_query_<name>`
   - 用于制度、技术资料、既有知识沉淀的检索。
4. `write_file`
   - 只落盘研究计划和最终报告；资料盘点、证据台账、分析归纳作为内部工作过程，不另建文件。
5. `web_fetch`
   - 仅在网络可用且确实需要公开网页原文时使用；优先精确 URL 抓取，不把它当搜索引擎。
6. `browser`
   - 仅在 `web_fetch` 抓不到有效正文、页面依赖渲染、需要手动确认页面内容时使用。

## 强制原则

- 不要把“深度研究”理解成“多搜几次网”。
- 没有充分资料时，可以输出“当前证据不足的最佳结论”，但必须明说不足在哪里。
- 结论必须能回溯到资料来源：工作区路径、知识库命中、或网页 URL。
- 在开始研究前，只允许做一轮澄清问询；要把关键问题一次问全，不要多轮追问。
- 报告中要明确区分：
  - 原始材料
  - 二手整理
  - 推断/判断
- 如果用户给了文件，先用文件；如果知识库已有资料，先用知识库；不要一上来就联网。

## 研究模式选择

收到任务后，先判断属于哪种模式：

### A. 离线优先模式

满足任一条件就进入：

- 用户明确说“没网/离线/不能联网”
- 当前环境拿不到网页或联网价值明显低
- 用户已经提供足够多的本地资料
- 研究对象本身主要存在于工作区或知识库

此模式下：

- 主路径必须是 `list_files`、`search_content`、`read_file`、知识库工具。
- 只围绕本地资料做证据组织、缺口识别、交叉验证和总结。
- 如果资料不足，不虚构外部事实，只输出“基于现有材料”的结论与补充资料清单。

### B. 联网增强模式

仅当以下同时满足时使用：

- 环境允许联网
- 本地资料不足以覆盖关键问题
- 公开网页补充能显著提高结论质量

此模式下也要先本地后网络。网络只是补充层，不是主路径。

## 标准流程

每次深度研究按以下阶段推进。

### Phase 0: 任务澄清

先判断信息是否足够。如果不够，先进行且仅进行一轮澄清问询，再开始研究。

这一轮澄清必须尽量一次问全，至少覆盖以下要素：

- 研究对象：研究谁、什么问题、哪个范围
- 研究目标：用户最终要拿这份研究做什么
- 时间范围：历史、当前、近几年、某个阶段
- 输出形式：短简报、完整报告、对比表、立项材料、学术综述
- 资料来源偏好：优先使用网络、知识库、工作目录文件，还是只基于用户给定资料
- 限制条件：是否允许联网，是否有必须排除的来源或必须纳入的文件/知识库

推荐把问题合并成一轮统一提问，例如：

- 你希望我优先基于哪类资料开展研究：网络公开资料、知识库、工作目录文件，还是仅基于你提供的材料？
- 这次研究的对象、时间范围、重点问题和最终交付形式分别是什么？
- 是否有必须纳入或必须排除的文件、目录、知识库或网页来源？

如果用户这一轮回复后仍不完整，不要继续连环追问。应当：

- 基于现有信息做最保守、最明确的工作假设
- 在 `research/00-研究计划.md` 中写清楚这些假设
- 在最终报告中明确说明研究边界和信息缺口

在完成这一轮澄清后，再把用户需求压缩成 4 个核心要素：

- 研究对象：研究谁/什么
- 研究目标：用户最终要拿这份研究做什么
- 时间范围：历史、当前、近几年、某个阶段
- 输出形式：短简报、完整报告、对比表、立项材料、学术综述

如果用户一开始已经把以上信息说得足够清楚，则不需要额外发问，直接进入研究计划编写。

然后在工作区先写一个研究计划文件：

- 推荐路径：`research/00-研究计划.md`
- 内容至少包括：
  - 研究题目
  - 核心问题清单
  - 资料来源计划
  - 预期产物

### Phase 1: 资料盘点

优先检查本地可用资料：

1. `list_files` 看目录结构
2. `search_content` 搜题目、关键词、对象名、别名、缩写
3. `read_file` 只读高价值片段，不整篇盲吞
4. 知识库工具检索已有资料

盘点时把来源分成四层：

- S 级：用户提供的一手材料、官方原文、原始数据、会议纪要、制度原文
- A 级：官方说明、技术文档、正式报告、权威机构材料
- B 级：高质量媒体/研究机构二手分析
- C 级：论坛、评论、零散观点、未证实转述

盘点结果只作为内部工作记录，不单独写 `research/01-资料盘点.md`。必要信息压缩进 `research/00-研究计划.md` 的“资料来源计划/资料缺口”，或进入最终报告的方法说明和参考文献。

内部盘点至少覆盖：

- 资料路径或来源名
- 来源级别
- 主要覆盖问题
- 明显缺口


### 公开网页快速获取入口（实证可用）

当研究对象是近期热点、开源项目、产品或公司时，优先使用 `web_fetch` 抓取以下入口来快速找到一手资料。注意：搜索结果页只是线索，不能当作最终证据；找到候选链接后，必须再抓取官方站、官方文档、GitHub/代码仓库、云厂商官方页或权威机构页面做主证据。

**优先级 1：Bing 搜索结果（稳定返回可读摘要，适合找一手链接）**

- `https://www.bing.com/search?q=<query>`
- `https://www.bing.com/search?q=<query>+official`
- `https://www.bing.com/search?q=<query>+docs`
- `https://www.bing.com/search?q=<query>+GitHub`
- `https://www.bing.com/search?q=<query>+pricing`
- `https://www.bing.com/search?q=<query>+security+risk`
- `https://www.bing.com/search?q=<query>+scam+controversy`
- 中文热点可用 URL 编码组合：`https://www.bing.com/search?q=<query>+%E5%AE%98%E7%BD%91`、`https://www.bing.com/search?q=<query>+%E6%95%99%E7%A8%8B`、`https://www.bing.com/search?q=<query>+%E8%AF%84%E6%B5%8B`、`https://www.bing.com/search?q=<query>+%E5%AE%89%E5%85%A8+%E9%A3%8E%E9%99%A9`、`https://www.bing.com/search?q=<query>+%E9%AA%97%E5%B1%80+%E4%BA%89%E8%AE%AE`。
- Bing 常会跳转到 `https://cn.bing.com/search?...`，这是正常结果，可继续使用。

**已实证可跑通的示例入口（OpenClaw 热点研究日志验证）**

- `https://www.bing.com/search?q=OpenClaw`：可返回官方文档、官方站、GitHub 和社区文章线索。
- `https://www.bing.com/search?q=OpenClaw+%E5%85%BB%E9%BE%99%E8%99%BE`：可返回中文热点、教程和官方/社区来源线索。
- `https://www.bing.com/search?q=OpenClaw+%E5%85%BB%E9%BE%99%E8%99%BE+%E9%AA%97%E5%B1%80`：可用于快速定位争议/风险语境，但必须再打开原始页验证。
- `https://developer.aliyun.com/article/1724368`：阿里云开发者社区文章，本次抓取成功，可用于 OpenClaw 部署、成本、安全风险线索。
- `https://www.aliyun.com/activity/ecs/clawdbot`：阿里云官方活动页，本次抓取成功，适合核验云厂商官方部署、镜像和价格信息。
- `https://developer.aliyun.com/article/1724373`：阿里云开发者社区文章，本次抓取成功，可用于功能场景、成本和权限风险交叉校验。
- `https://openclawlaunch.com/zh/raise`：OpenClaw Launch 指南页，本次抓取成功，适合核验“养龙虾”社区语义、入门流程和 Launch 成本信息。

**优先级 2：官方与一手信息源（证据级别最高）**

- 官方文档：`https://docs.<project-domain>`、`https://<project-domain>/docs`、`https://<project-domain>/zh`。如果直接抓取是客户端渲染壳页，就先用 Bing 结果摘要定位子页或用浏览器打开。
- 代码仓库：`https://github.com/<org>/<repo>`、`https://github.com/search?q=<query>&type=repositories`。必须核对仓库是否真实存在，不要凭搜索摘要猜组织名。
- 云厂商/平台官方页：对云部署、价格、镜像、官方合作等信息，优先抓取平台官方活动页和开发者社区文章。
- 官方博客/发布页：`https://<project-domain>/blog`、`https://<project-domain>/changelog`、`https://<project-domain>/releases`。

**优先级 3：技术社区与二手验证（用于交叉校验，不要单独作为结论来源）**

- CSDN：`https://www.bing.com/search?q=site:blog.csdn.net+<query>`
- 博客园：`https://www.bing.com/search?q=site:cnblogs.com+<query>`
- 知乎专栏：`https://www.bing.com/search?q=site:zhuanlan.zhihu.com+<query>`。知乎直接抓取经常 403，优先用 Bing 摘要和标题做线索，不要把 403 页面当资料。
- 腾讯云开发者社区：`https://www.bing.com/search?q=site:cloud.tencent.com/developer/article+<query>`，搜到后再抓取具体文章，注意校验 404。

**抓取避坑规则**

- 百度搜索页常返回安全验证/验证码页，即使 `web_fetch` 显示 ok 也不能当有效搜索结果。遇到“百度安全验证”、“网络不给力”等内容要立即换 Bing。
- 直接抓取客户端渲染站（如只返回壳页、空内容、大量 JS）不能认定为无信息，先用 Bing 搜索其子页、文档片段或用浏览器工具打开。
- 超时、403、404、证书/连接错误只在内部记录“未取得/失败原因”，不要当作证据使用；除非用户明确要求，不另写失败清单文件。
- 使用搜索结果页时，至少再打开 2-3 个不同域名的原始页做交叉验证。
- 对于数字类信息（Star 数、价格、上线时间、用户规模），如果不能从一手页面实时确认，必须标注“搜索结果/二手资料显示”，不要写成已验证事实。

### Phase 2: 研究取证

围绕核心问题逐个取证，不要按文件顺序机械翻页。

每个问题至少回答：

- 现有证据说了什么
- 哪些来源相互印证
- 哪些来源互相冲突
- 还缺什么证据

离线模式下，重点做：

- 多文件交叉验证
- 同一主题的不同版本对比
- 草稿与正式稿差异梳理
- 本地历史产物时间线重建

联网增强模式下，额外做：

- 对关键公开网页用 `web_fetch`
- 只抓精确 URL，不擅自拼接 URL
- 网页抓不到正文时再切 `browser`

研究证据只作为内部工作记录，不单独写 `research/02-证据台账.md`；关键事实必须在最终报告正文就地引用来源。

内部证据梳理建议结构：

- 问题 1
  - 证据
  - 来源
  - 判断
  - 缺口
- 问题 2 ...

### Phase 3: 分析与归纳

在内部证据梳理基础上做结构化归纳，而不是直接开始写正文。

至少产出以下内容：

- 关键事实清单
- 核心矛盾/分歧点
- 趋势或阶段划分
- 风险与不确定性
- 可以下结论的部分
- 暂时不能下结论的部分

分析归纳不单独写 `research/03-分析归纳.md`；归纳内容直接服务最终报告提纲与正文。

### Phase 4: 报告生成

最终报告默认写入：

- `research/最终研究报告.md`

先判断最终交付场景，再选择对应模板：

- 正式调研、竞品分析、公司分析、政策研究、方案论证、尽调分析：优先使用 `references/report-template-正式调研报告.md`
- 学术综述、课题论证、研究设计、理论分析、方法研究：优先使用 `references/report-template-学术研究报告.md`
- 领导汇报、管理判断、方案选型、风险研判、阶段判断：优先使用 `references/report-template-决策简报.md`
- 如果用户没有明确指定格式，但最终结果要正式留档或转 DOCX，默认使用 `references/report-template-正式调研报告.md`

`references/report-template.md` 仅作为模板选择说明入口，不作为最终正文骨架直接套写。

报告写作前必须先读取已选定的模板全文，并把模板转化为报告提纲后再开写：

- 如果当前环境挂载了 `公文写作` 技能，在写 `research/最终研究报告.md` 前必须先调用或阅读该技能的版式与 Markdown 约束，再按其规范起草最终报告。
- 最终报告要从写作阶段就满足 DOCX 转换要求，不要等写完后再大量修补格式。
- 不能仅凭记忆或凭主题直接写报告。
- 正式调研报告必须覆盖“研究背景与目标”“资料基础与研究方法”“核心发现与分析”“风险与不确定性”“结论与建议”“参考文献”等主体部分。
- 如果任务不需要某个模板段落，可删减，但要保留“为什么这么判断”的论证链。
- 模板中的“结论/证据/来源/影响/限制”不是占位符，必须用实际材料填完，形成完整分析段落。
如果最终产物需要进入正式文稿流转，或后续要交给 `公文写作` 技能转换 DOCX，必须遵守以下 Markdown 版式规则：

- 第一行直接写报告标题，不要在第一行写 `#`。
- 从正文开始，`#` 只用于一级标题，`##` 只用于二级标题；不要手写“一、”“（一）”等同层级编号，交给转换器处理。
- `摘要：`、`关键词：`、`研究日期：`、`研究对象：`、`研究范围：`、`资料基础：` 等字段必须一行一个字段，不要写成列表，也不要多字段挤在同一行。
- 表格前一行和后一行都必须留空行，表格标题应单独占一行，不要和表格写在同一行。
- 不要使用 `---`、`***` 这类分隔线分段。
- 不要把连续元信息写成无序列表，尤其是来源、参数、对象、时间等字段。

### DOCX 导出要求

- 最终报告写入 `research/最终研究报告.md` 后，如果当前环境已挂载或可调用 `公文写作` 技能，并且用户没有明确拒绝 DOCX，应尽量继续调用该技能把最终 Markdown 报告导出为 DOCX。
- DOCX 导出应以最终报告正文为唯一主输入，不要把研究计划、内部证据梳理、草稿过程或内部分析混入正文。
- 导出前先检查 Markdown 是否满足公文写作技能的版式约束；不满足时先修正 Markdown，再执行导出。
- 如果无法导出 DOCX，最终回复中要明确说明原因，并保留 Markdown 报告路径供用户后续转换。
- 成功导出后，最终回复同时给出 Markdown 路径和 DOCX 路径。

报告正文必须做到：

- 必须像报告，不能像干条或清单；清单只能用于归纳要点，每个关键部分后都要有连贯的分析性段落。
- 不要把资料摘录直接堆叠成正文；要对事实之间的关系、因果、矛盾、适用边界和对用户问题的含义做解释。
- 每个主要章节至少包含一段“综合分析”，说明该章节材料对整个研究结论的支撑或限制。
- 先给摘要，再展开分析
- 每个核心结论附近都能找到来源指向
- 引用本地文件时写清路径和必要片段位置
- 引用知识库时写清知识库名称或文档标识
- 引用网页时写标题 + URL

### Phase 5: 交付检查

提交前自检：

- 是否回答了用户真正的问题，而不是只堆资料
- 是否把事实、判断、建议分开
- 是否明确说明了证据边界
- 是否只保存了研究计划和最终报告两个默认文件
- 如果是离线模式，是否显式说明“本结论仅基于当前本地资料”

## 离线优先协议

无网络环境时，必须遵守：

### 1. 先扫全局，再定点读

- 先 `list_files`
- 再 `search_content`
- 最后 `read_file`

不要上来连续 `read_file` 多个长文。

### 2. 以“研究问题”为单位收集证据

不要按文件顺序抄资料。
应该按“问题”聚合证据，例如：

- 研究对象是什么
- 关键时间线是什么
- 主要观点或机制是什么
- 风险或争议在哪里

### 3. 充分利用已有草稿和历史产物

工作区里如果已有：

- 旧报告
- 纪要
- 方案草稿
- 对比表
- 导出的 Markdown
- 上传的 PDF/OCR 文本

都应视为高价值输入，而不是忽略后重新从头写。

### 4. 不因缺网而降级成空泛总结

即使没有网，也要输出：

- 基于本地资料可确认的结论
- 本地资料之间的冲突点
- 当前无法确认的问题
- 后续若补资料，最应该补什么

## 联网增强协议

网络可用时，仍遵守以下规则：

- 先本地后网页
- `web_fetch` 只抓单页正文
- 只抓用户给出的 URL 或搜索/已有资料中明确出现的 URL
- 不抓需要登录、鉴权、私密页面
- 不擅自改写 URL，不擅自加 `www`
- 遇到动态页面或交互页时，优先判断是否真的值得用 `browser`

## 推荐输出物

深度研究默认只落这 2 个文件：

- `research/00-研究计划.md`
- `research/最终研究报告.md`

除非用户明确要求，不要另写资料盘点、证据台账、分析归纳、来源清单、时间线、草稿、日志等过程文件。必要来源应进入最终报告正文就地引用和报告末尾参考文献。

## 报告风格要求

- 默认用中文写
- 偏正式、清晰、克制，不写营销腔
- 文风应是“研究报告”而不是“工具说明书”或“知识点列表”；要用完整句和连贯段落展开论证。
- 少用“有以下几点”“具备以下能力”这类简单列举句；列完要点后必须追加解释这些要点共同说明了什么。
- 优先使用“因此”“这意味着”“从证据看”“但需注意”“其限制在于”等连接语形成分析链。
- 结论句尽量短，论证句尽量具体
- 避免“显而易见”“不言而喻”这类无证据词
- 对不确定信息使用“可能/现有材料显示/尚无法确认”等措辞

## 引用规则

### 本地文件引用

推荐格式：

```markdown
来源：`/workspaces/.../docs/xxx.md`
```

如果能精确到章节或行范围，更好。

### 知识库引用

推荐格式：

```markdown
来源：知识库《xxx》命中片段
```

### 网页引用

推荐格式：

```markdown
来源：[页面标题](https://example.com)
```

### 正文中的引用要求

- 关键事实、关键数字、项目性质、价格、时间、风险判断、竞品对比等不能只在附录统一列来源，必须在正文相应句子或段落后面附上来源。
- 推荐写法：`...。（来源：[页面标题](https://example.com)）`；如同一段使用多个来源，可写：`（来源：[来源一](...), [来源二](...)）`。
- 报告末尾仍需保留“参考文献”；不要另建独立来源清单文件，也不能用末尾清单替代正文就地引用。
- 如某个事实只来自搜索结果摘要或二手文章，要明确写“搜索结果/二手资料显示”，不能写成经官方核验的事实。

## 常见错误

- 把“资料收集”当成“复制粘贴”
- 没有研究计划就直接开写报告
- 为了补细节连续追问多轮，迟迟不开始研究
- 离线场景下还试图强行联网
- 一手、二手、推断混写
- 只给结论，不留来源
- 只列资料，不回答问题
- 不读选定模板就直接写报告，导致报告变成干条和素材清单
- 只在末尾列来源，关键事实后面没有就地引用，读者无法判断信息可信度
- 明明证据不足，却写成确定判断

## 最小执行范式

1. 如信息不足，先进行一轮统一澄清，问清资料来源偏好和必要细节
2. 写 `research/00-研究计划.md`
3. 盘点工作区、知识库和网页资料，但只做内部记录，不另建资料盘点/证据台账文件
4. 如有 `公文写作` 技能，先读取并遵循其 Markdown/DOCX 版式约束，再按要求编写最终报告，顺序不能乱
5. 读取并参照选定报告模板，先形成报告提纲
6. 归纳后写 `research/最终研究报告.md`，从写作阶段就确保主体是连贯分析而不是干条列表，关键事实后要附来源
7. 如可用且用户未拒绝，调用 `公文写作` 技能将 `research/最终研究报告.md` 导出为 DOCX
8. 回复用户时说明：
   - 用了哪些资料层
   - Markdown 报告已保存到哪里
   - DOCX 是否已导出，导出路径是什么；如未导出，说明原因
   - 结论边界在哪里
