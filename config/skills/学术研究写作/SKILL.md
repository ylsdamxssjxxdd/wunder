# 学术研究写作（离线优先）

## 概要
本技能面向“本地资料驱动的研究写作”。它不是一个在线检索代理，也不是多智能体科研流水线，而是一套适合离线环境的中文研究工作台。

目标很明确：把分散的材料、摘录、笔记和草稿，稳定推进成结构清晰、证据边界明确、可继续交付为 DOCX 的正式文稿。

## 适用场景
- 基于本地文献、报告、会议纪要、访谈记录或历史草稿，整理研究计划。
- 基于现有材料撰写文献综述、专题研究报告、实证论文草稿、理论分析论文草稿、开题报告。
- 基于已有稿件做同行评审、问题诊断或审稿意见回复。
- 需要在离线环境中完成学术写作，不希望默认联网检索。
- 需要以 Markdown 为源稿，再导出 DOCX、PDF 或供后续排版使用。

## 不适用场景
- 需要实时查新、广泛在线检索、批量抓取最新论文原文。
- 需要自动执行完整 PRISMA、Meta-analysis、在线引文核验或跨模型核查。
- 需要大型科研数据库、实验平台或外部 API 驱动的工作流。

这类任务应先由其它技能或外部工具补齐资料，再回到本技能完成写作和整合。

## 核心原则
- 资料优先：先确认手里有什么，再决定能写到什么深度。
- 边界优先：能确认的写清楚，不能确认的明确标注，不靠想象补齐。
- 成稿优先：终稿用完整段落和连贯论证，不用项目符号替代正文。
- Markdown 优先：先把结构、关系、图表草案写进 Markdown，再做 DOCX 交付。
- 轻量闭环：尽量在一个工作目录内完成问题拆解、证据整理、图表组织、正文写作和终稿校样。

## 从参考项目迁移进来的增强能力
本技能吸收了两个参考项目里最值得保留、且适合离线中文写作的部分，但做了收缩：

来自学术研究技能套件的能力：
- 常见研究设计模式
- 证据分级与来源评估
- 中文正式文稿模板
- 结构化评审与修订框架
- 研究工作台文件：问题拆解、文献矩阵、证据评估、图表清单

来自 scientific-agent-skills-main 的能力：
- `scientific-writing` 的成稿规则：终稿写完整段落，不用清单代替论证
- `literature-review` 的证据整合逻辑：先做矩阵，再做主题归纳，再写共识与分歧
- `peer-review` 的评审逻辑：问题、材料、方法、结论四层检查
- `scientific-visualization` 的图表规则：图表必须增加信息量，且能被正文解释
- `markdown-mermaid-writing` 的文本图理念：流程、结构、关系优先用 Markdown/Mermaid 表达
- `citation-management` 的离线可迁移部分：正文引用与参考文献对齐、条目核对、缺失项管理
- `docx` 与 `pdf` 的离线交付意识：终稿导出前必须做版式校样，保留 Markdown 源稿与图表原件

未迁移的部分主要是：
- 在线检索与数据库 API
- 大规模科学软件包操作
- 重型自动化科研流水线
- 在线生成图和云端平台集成

## 快速流程
1. 明确文稿类型：研究计划、文献综述、研究报告、实证论文、理论分析论文、开题报告、同行评审报告、审稿意见回复，还是参考文献整理表。
2. 运行 `{{SKILL_ROOT}}/scripts/init_academic_project.py` 初始化工作目录。
3. 先判断是否需要图表；复杂关系优先做 Mermaid 或图表草案。
4. 选择最接近目标的模板作为 `drafts/主稿.md` 的起点。
5. 在 `notes/` 中先完成问题拆解、文献矩阵、证据评估卡、图表清单、参考文献整理表中的必要部分。
6. 如果是综述、研究报告或论文主稿，优先读取“科研写作精要”和“文献综述与证据整合”。
7. 再写正文：先交代研究边界、资料基础和核心问题，再展开分析。
8. 正文写完后，核对正文引用与参考文献条目是否一一对应。
9. 最后按自检清单、章节写作检查表和终稿交付清单做校样。
10. 需要 Word 交付时，用 `{{SKILL_ROOT}}/scripts/export_docx.py` 导出 DOCX；只要 Markdown 中包含图片，必须使用 pandoc 链路（`--use-pandoc`），否则图片可能只能变成占位文字或后处理嵌入不完整。

## 图表优先规则
尽量让文稿带图表，但不要为显得专业而硬塞图。

### 适合加表格
- 文献矩阵
- 来源对比
- 证据等级判断
- 研究方案对照
- 优缺点矩阵
- 章节检查或交付清单

### 适合加图
- 研究流程
- 逻辑链条
- 概念关系
- 系统结构
- 时间线
- 场景切换或模块耦合关系

### 需要主动问用户的情况
- 需要真实数据才能画图，但用户尚未提供。
- 研究对象或范围还没定，图表类型无法判断。
- 用户说要“更专业”，但没有说更偏表格、流程图还是概念图。

### 可以自行生成的图表
- 对比表
- 文献矩阵
- 研究问题拆解表
- 证据评估表
- 研究流程图
- 概念关系图
- Mermaid 流程图或结构图

### 底线
- 图表必须帮助理解。
- 图表必须在正文中被引用。
- 图注必须说明图在表达什么。
- 信息不足时先问用户，不虚构数据。

## Markdown 约束
- 第一行直接写标题，不要在第一行写 `#`
- 用标准标题层级：`#`、`##`、`###`、`####`
- 标题、表格、代码块前后都留空行
- 不用 `---`、`***`、`___` 做视觉分隔
- 正文尽量保持自然段，不要频繁硬换行
- 终稿不允许退化成项目符号堆叠
- 非代码内容不要滥用三反引号代码块

## 正文要求
- 先交代研究对象、资料范围、问题边界，再展开分析
- 每一章都要说明“这些信息共同说明了什么”
- 引文、摘录、统计描述和研究判断要分层写
- 章节之间要有过渡句
- 证据有限时，只写有限结论

## 引用与参考文献要求
- 正文中出现的关键来源，参考文献区必须能找到对应条目
- 参考文献区的条目，最好都能在正文找到实际用途
- 无法核实的信息统一标注：`待补充来源`、`待核对页码`、`待核对年份`
- 本地文件引用优先记录文件名、章节、页码或摘录位置
- 不要编造 DOI、作者顺序、年份或页码
- 终稿前最好用 `templates/14-参考文献整理表.md` 再过一遍

## 审稿与修订要求
- 同行评审报告先做结构化判断，再提建议
- 审稿回复必须逐条回应，不跳题
- 回复中先写是否采纳，再写修改动作和位置
- 不采纳意见时，要给出基于材料或研究边界的理由

## 导出与最终交付
### 初始化脚本
```bash
python {{SKILL_ROOT}}/scripts/init_academic_project.py ./我的研究项目 --template report
```

可选模板键包括：
- `plan`
- `review`
- `report`
- `empirical`
- `theory`
- `response`
- `proposal`
- `peer-review`
- `chart`
- `evidence-matrix`
- `reference-log`

### DOCX 导出
```bash
python {{SKILL_ROOT}}/scripts/export_docx.py \
  ./我的研究项目/drafts/主稿.md \
  --output ./我的研究项目/output/主稿.docx
```

说明：
- 脚本负责把 Markdown 转给本地转换器；检测到本地图片引用时会自动追加 `--use-pandoc`，走 pandoc 图片嵌入链路。
- 含图片文稿也可以显式写：

```bash
python {{SKILL_ROOT}}/scripts/export_docx.py \
  ./我的研究项目/drafts/主稿.md \
  --output ./我的研究项目/output/主稿.docx \
  --use-pandoc
```

- `--use-pandoc` 需要安装 `pypandoc`，并确保 `pandoc` 可用；Windows/Linux 均可通过 PATH 找到 pandoc，或继续向底层转换器透传 `--pandoc <pandoc路径>`。
- Markdown 中的本地图片建议与主稿放在同一项目目录或其子目录内，路径使用相对路径，例如 `![图1 研究流程](../figures/workflow.png)`。
- SVG 图片会由底层转换器尝试先转成 PNG 再嵌入；如果没有 SVG 转换器，请优先提供 PNG/JPG，或安装 `cairosvg`、`resvg`、`rsvg-convert`、Inkscape、ImageMagick 之一。
- 导出前先做终稿交付清单检查
- 导出后要人工检查图片、表格、分页、图注和是否残留 Markdown 痕迹
