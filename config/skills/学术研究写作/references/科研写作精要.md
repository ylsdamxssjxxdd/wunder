# 科研写作精要

## 读取时机
在以下场景优先读取本文件：
- 需要把草稿提升成正式稿
- 需要把综述材料整理成成稿
- 需要判断图表、段落和结构是否专业
- 需要把研究过程写成可交付的 Markdown 文稿

本文件整合了可迁移的写作经验，重点来自科学写作、文献综述、同行评审、可视化与 Markdown/Mermaid 写作几类能力。

## 1. 成稿原则
- 正式稿优先用完整段落，不要把终稿写成检查表或碎片清单。
- 实证类论文默认按 `引言 -> 方法 -> 结果 -> 讨论 -> 结论` 组织。
- 综述、理论分析、方法说明、案例分析可以调整结构，但每章都要回答一个明确问题。
- 每一段至少要完成一个动作：提出判断、给出证据、解释关系、指出边界。
- 终稿里少写“本文将……”“下面将……”这类自我说明句，直接进入内容。
- 引用要和正文判断一一对应；不能核实的内容要明确标记为待核对。

## 2. 综述与证据整合
- 先界定问题，再确定材料范围。
- 先做来源表或文献矩阵，再写正式综述。
- 主题归纳比单纯罗列摘要更重要。
- 综述不是“资料堆叠”，而是比较不同来源之间的共识、冲突、缺口与趋势。
- 如果材料来自本地文件或用户上传内容，要先注明边界，再写结论。
- 证据强度不够时，宁可写保守判断，也不要把可能性写成定论。

### 推荐的证据整理字段
| 字段 | 说明 |
| --- | --- |
| 来源 | 文献、报告、记录或材料名 |
| 年份 | 资料时间 |
| 类型 | 论文、报告、纪要、笔记等 |
| 方法/材料 | 研究怎么做的，或材料如何形成 |
| 主题 | 该来源支持哪一类论点 |
| 结论 | 该来源最核心的判断 |
| 证据等级 | 强、中、弱、待核对 |
| 使用方式 | 主证据、辅证据、背景、排除 |

## 3. 图表与版式
- 只有在图表能让读者更快理解时才使用图表。
- 表格适合精确值、分类对比、矩阵和清单。
- 图适合趋势、关系、流程、结构和层级。
- 每个图表都应在正文中被引用，不能只放在附件里。
- 图题、图注、单位、样本量、图例要完整。
- 图和表不要重复展示同一件事。
- 颜色要考虑黑白打印和色盲阅读，不能只靠颜色区分。
- 复杂图尽量拆成多图或多面板，不要挤成一张看不懂的大图。

### 图表硬规则
- 图必须增加信息量。
- 图注必须让读者离开正文也能看懂图的大意。
- 线稿优先高分辨率或矢量，照片类优先高质量位图。
- 避免 3D、花哨装饰、无意义阴影和无解释的双坐标轴。

## 4. Markdown 与 Mermaid
- Markdown 是工作底稿，不只是导出格式。
- 结构、流程、关系、层级、时间线这类内容，优先用 Mermaid 先写清楚。
- Mermaid 图最好放在 Markdown 源文件里，作为“源文本”保留。
- 如果后续需要 PNG 或 DOCX，再从 Markdown 生成衍生图，而不是反过来。
- 图文档里，先写说明，再写图，不要把图单独堆在最后。

### 适合优先用 Mermaid 的内容
- 研究流程
- 工作流
- 章节结构
- 逻辑链条
- 概念关系
- 比较矩阵
- 时间线

## 5. 评审与修订
- 先看问题是否清楚，再看材料是否足够，再看结构是否成立，最后看结论是否克制。
- 如果要回复外部意见，必须逐条回应，不要跳题。
- 回复里先写是否采纳，再写具体修改动作，最后写修改位置。
- 对不采纳意见，要给出可辩护的原因，且要基于材料或研究边界。

### 评审时的最小检查项
- 研究问题是否清楚
- 方法是否匹配
- 统计或比较是否站得住
- 图表是否清楚且可复核
- 结果是否与结论一致
- 参考文献是否支撑正文
- 是否有明显夸大或越界

## 6. 终稿前自检
- 终稿是不是完整段落，而不是项目符号堆叠。
- 每个核心结论有没有证据落点。
- 图表是不是比正文更清楚。
- 章节之间有没有明显断裂。
- 是否还有待核对信息被写死。
- 是否符合目标文体，而不是像工作笔记。
