# 通用气象知识

这是一个面向气象法规、政策、业务知识和科普解释的本地知识型技能。它不是实时天气查询工具，也不能替代官方预报、预警、法律意见或业务审批结论。

## 核心原则

- 先检索再回答，不凭记忆给出法规条文、发布权限、预警标准或业务要求。
- 先读索引，再按关键词定点读取相关文件，不要一次性读取全部 references。
- 明确区分法规政策、业务规范、气象知识、经验判断和实时资料缺口。
- 涉及现行法规、政策、地方标准或正在生效的业务规则时，必须提示以官方原文和最新版本为准。
- 涉及实时天气、灾害预警或行动建议时，必须提示用户核对当地气象主管机构发布的信息。

## 检索流程

默认策略是“先搜索，后读取”。除 `references/00-检索索引.md` 和用户明确给出的具体文件路径外，不要在未检索命中的情况下直接读取专题 `index.md` 或知识条目。

1. 先读取 `{{SKILL_ROOT}}/references/00-检索索引.md`，只用于确定专题、关键词和候选范围。
2. 使用 `搜索内容` 在 `{{SKILL_ROOT}}/references` 内检索用户原词、同义词、法规名称、灾种名或产品名。用户说“搜搜、检索、查找、找一下”时，这一步是强制动作。
   - 如果使用 `OR`、`|` 这类多词扩展，必须设置 `query_mode=regex`，例如 `预报预警业务|预报|预警|短临|影响预报`。
   - 如果不设置 `query_mode=regex`，不要在查询里写 `OR`；应使用用户原词做一次字面检索，必要时再换同义词检索。
3. 根据 `搜索内容` 命中的文件路径，再使用 `读取文件` 定点读取 1 到 3 个最相关条目。不要只因为索引列出了某个专题，就直接通读该专题所有文件。
4. 如果搜索命中过多，缩小关键词或加专题路径再次搜索；如果搜索无命中，再回到总索引扩展关键词，不要直接猜文件。
5. 如果本地资料不足，说明缺口；只有用户允许且当前工具可用时，才补充官方网页原文。
6. 回答时写清楚依据来源：技能文件路径、法规/文件名称、来源等级和是否需要二次核验。

### 反例

用户说“搜搜预报预警业务”时，错误做法是：

```text
skill_call -> 读取 00-检索索引 -> 直接读取 02-预报预警业务/index.md -> 直接读取多个条目
```

正确做法是：

```text
skill_call -> 读取 00-检索索引 -> 搜索内容(query=预报预警业务 OR 预报 OR 预警 OR 短临 OR 影响预报, path={{SKILL_ROOT}}/references) -> 读取命中条目
```

如果没有实际调用 `搜索内容`，不要在回复中说“已检索”；只能说“已读取”。

## 查询扩展

检索时主动扩展关键词：

- 预警：预警信号、灾害性天气警报、发布、传播、解除、防御指南、属地发布。
- 法规：气象法、气象灾害防御、预报发布、人工影响天气、探测环境、防雷、气象信息服务、涉外探测、AI气象应用。
- 强对流：雷暴大风、冰雹、短时强降水、龙卷、下击暴流、飑线、超级单体、中尺度对流系统。
- 观测：自动站、天气雷达、雷达回波、径向速度、双偏振、卫星云图、闪电定位、资料质量控制。
- 服务：风险提示、不确定性、影响预报、公众表达、决策服务、科普解释。

## 来源分级

| 等级 | 含义 | 使用方式 |
| --- | --- | --- |
| S | 官方原文、法规规章、标准原文、用户给定原始资料 | 可支撑核心结论，但仍要注意版本 |
| A | 官方解读、业务技术指南、权威教材或培训资料 | 可支撑解释和方法判断 |
| B | 专业综述、内部整理、经验性知识卡 | 可用于辅助解释，不单独支撑法规结论 |
| C | 未核验材料、口头经验、推断 | 只能作为线索，必须降级表达 |

## 回答要求

- 对法规政策问题，先给“适用范围/核心要求/使用边界”，再给依据来源。
- 对气象知识问题，先给简洁结论，再解释机理、识别要点和常见误区。
- 对预警或服务表达问题，避免替用户发布预警；应写成“根据规则可关注的风险点”。
- 对缺少实时资料的问题，明确说“当前技能不含实时观测/预报数据”。
- 不要长篇照抄法规或资料原文；只摘取必要短语，主要用自己的话概括。

## 维护方式

知识采用专题目录制：`references/<编号-专题>/index.md` 是专题入口；专题下可继续放多个独立条目、图片或原始资料摘录。不要把所有知识长期堆在一个根级 Markdown 文件中。

### 新增知识

1. 先读取 `references/00-检索索引.md`，判断应落到哪个专题目录。
2. 如果是少量补充，追加到对应专题的 `index.md`；如果是独立法规、标准、案例化知识或长篇资料，按 `{{SKILL_ROOT}}/templates/知识条目模板.md` 在专题目录中新建独立条目文件。
3. 文件命名使用稳定、可检索的中文短名，例如 `气象灾害预警信号发布.md`；不要使用无意义日期或临时标题。
4. 在新增条目中写清楚分类、关键词、来源等级、适用范围、版本或更新时间、官方来源和使用边界。
5. 在 `references/00-检索索引.md` 同步增加关键词和入口；法规政策类资料还要在 `references/01-法规政策/index.md` 记录官方来源、版本状态和核验要求。
6. 对地方性规则，必须标注适用区域和版本日期；不清楚时写“待核验”。

### 更新知识

1. 先定位专题目录和具体条目，不要直接全局改写。
2. 对法规、政策、标准、业务规则等高风险内容，必须核验官方来源和版本日期，再更新摘要。
3. 如果只是新增解释或补充误区，优先追加到原条目对应小节；如果会改变原结论，必须写明“修订说明”。
4. 更新关键词、适用范围或文件路径后，必须同步更新 `references/00-检索索引.md`。

### 移除知识

1. 只有在资料错误、重复、过期且不宜保留、来源不合规或用户明确要求时移除。
2. 删除独立条目前，先用 `搜索内容` 查找引用；同步移除索引入口和交叉引用。
3. 对过期但仍有历史参考价值的法规政策，优先移到条目内“历史版本/已失效说明”，不要直接删除。
4. 如果删除整个专题目录，必须先确认目录内没有仍被索引引用的文件。

### 重命名或迁移知识

1. 目录重命名时保持编号和专题含义稳定；同步更新 `references/00-检索索引.md` 和 `SKILL.md` 的参考文件列表。
2. 跨专题迁移时，在目标条目中说明迁移原因；删除旧路径引用。
3. 不要为了短期方便把法规、灾害知识、观测知识混放到同一专题。
