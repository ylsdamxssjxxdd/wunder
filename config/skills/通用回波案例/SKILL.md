# 通用回波案例

这是一个本地雷达回波案例库技能，用于从典型回波案例中检索相似结构、解释判读逻辑、辅助形成案例对照。它不包含实时雷达资料，不能替代业务值班、官方预报预警或现场核验。

## 使用原则

- 先把用户问题转换成标签，再检索案例，不要只按标题猜。
- 同时看反射率、径向速度、演变趋势、环境场和地面实况，不要用单一雷达图像下结论。
- 案例只用于“相似性对照”和“判读提示”，不能直接推断当前天气一定发生某灾害。
- 输出时必须写清楚：匹配依据、差异、可借鉴点、不能借鉴的部分。
- 如果用户的主要意图是“查案例、找案例、给我案例、有没有类似案例”，必须优先把匹配到的原案例交给用户：先把完整案例目录复制到当前工作目录，再在回复中渲染复制后的示意图、给出复制后的路径、展示原案例正文核心内容，并说明为什么匹配；不要只给二次总结。
- 如果用户提供了实时雷达图或观测资料，应先说明资料时次、产品类型和空间范围，再套用案例。
- 每张内置案例卡都应优先展示同案例目录中的 `diagram.svg`；图片只表达结构要点，不代表真实雷达产品。

## 检索流程

1. 读取 `{{SKILL_ROOT}}/references/00-案例索引.md`，确定候选标签和案例编号。
2. 如需判读方法，读取 `{{SKILL_ROOT}}/references/01-回波判读框架.md`。
3. 使用 `搜索内容` 在 `{{SKILL_ROOT}}/references/cases` 检索标签、灾害类型、回波特征或产品名称。
4. 读取 1 到 3 个最相关案例卡。不要一次读取全部案例。
5. 判断用户意图：
   - 如果用户是在查案例，先把匹配案例目录复制到当前工作目录，再输出复制后的原案例卡内容和可渲染图片，并说明为什么匹配。
   - 如果用户是在判读实时资料，先输出匹配案例摘要，再强调需要核验的实时资料和边界。
6. 输出“原案例/匹配案例 + 为什么匹配 + 应继续核验哪些资料 + 使用边界”。

## 标签体系

检索时优先组合以下标签：

- 天气类型：`强对流`、`暴雨`、`台风雨带`、`层状降水`、`非气象回波`
- 回波形态：`钩状回波`、`弓形回波`、`飑线`、`训练效应`、`强回波核`、`弱回波区`、`亮带`
- 风险类型：`雷暴大风`、`冰雹`、`龙卷`、`下击暴流`、`短时强降水`、`低能见度误判`
- 雷达产品：`反射率`、`径向速度`、`谱宽`、`双偏振`、`相关系数`、`差分反射率`
- 生命周期：`初生`、`增强`、`成熟`、`合并`、`减弱`、`后侧入流`

## 输出模板

### 用户查案例时

必须把匹配案例目录复制到当前工作目录，推荐目标路径：

```text
./回波案例交付/CXX-简短主题/
```

复制整个目录，至少包含：

- `case.md`
- `diagram.svg`
- 该案例目录下的其它补充资料

如果目标目录已存在，不要覆盖用户文件；使用 `CXX-简短主题-2`、`CXX-简短主题-3` 这类后缀。

回复用户时必须遵守：

- 示意图必须用 Markdown 图片语法渲染：`![案例示意图](./回波案例交付/CXX-简短主题/diagram.svg)`。
- 不要把示意图路径只写成反引号代码，例如不要只写 `` `./回波案例交付/.../diagram.svg` ``。
- 案例文件路径可以用 Markdown 链接：`[case.md](./回波案例交付/CXX-简短主题/case.md)`。
- 原案例正文核心内容要直接展示在回复中，不要只让用户自己打开文件。
- 如果图片无法在当前客户端渲染，仍要保留 Markdown 图片语法，并补充普通文件路径。

```markdown
原案例：……

已复制到：……

案例文件：[case.md](./回波案例交付/CXX-简短主题/case.md)

示意图：

![案例示意图](./回波案例交付/CXX-简短主题/diagram.svg)

标签：……

典型场景：……

关键回波特征：……

需要继续核验：……

易混淆点：……

使用边界：……

为什么推荐这个案例：……
```

### 用户判读实时资料时

```markdown
匹配案例：……

匹配依据：……

关键回波特征：……

需要继续核验：……

使用边界：该案例只能说明相似结构和判读思路，不能替代实时预警结论。
```

## 维护方式

案例采用单案例目录制：`references/cases/CXX-简短主题/`，目录内至少包含 `case.md`，有图时放 `diagram.svg`。案例正文、图片、补充资料和后续修订记录都应留在同一个案例目录中。

### 新增案例

1. 先读取 `references/00-案例索引.md`，确定新编号。编号使用连续的 `CXX`，不要复用已移除编号；确需复用时必须在索引备注说明。
2. 新建目录：`{{SKILL_ROOT}}/references/cases/CXX-简短主题/`。
3. 复制 `{{SKILL_ROOT}}/templates/回波案例模板.md` 为 `case.md`，按模板填完整标签、风险、典型场景、关键回波特征、核验资料和使用边界。
4. 如果有图片、截图或示意图，统一放在同一案例目录中；默认文件名为 `diagram.svg`，正文使用 `![...](diagram.svg)` 引用。
5. 在 `references/00-案例索引.md` 增加案例编号、`case.md` 路径、主要标签和适用问题。
6. 如果案例来自真实个例，删除敏感地点、单位、人员、业务专名和不可公开数据，只保留可复用结构、环境、判读逻辑和边界。

### 更新案例

1. 先读取目标案例目录下的 `case.md` 和可能存在的图片或补充资料。
2. 只改该案例目录内文件；除非标签、标题、适用问题或路径变化，否则不要改总索引。
3. 若修改了标签、标题、风险类型、图片文件名或案例编号，必须同步更新 `references/00-案例索引.md`。
4. 若旧判断被新资料推翻，不要静默覆盖；在 `case.md` 的“使用边界”或“易混淆点”中说明修订原因。

### 移除案例

1. 只有在案例错误、重复、来源不合规或用户明确要求时移除。
2. 删除整个 `references/cases/CXX-简短主题/` 目录，避免留下孤立图片或补充资料。
3. 从 `references/00-案例索引.md` 删除对应行；如编号不再使用，可在索引末尾新增“移除记录”简述原因。
4. 移除前用 `搜索内容` 检查是否有其它案例或说明引用该编号，避免断链。

### 重命名或合并案例

1. 重命名时保持编号不变，只改主题目录名；同步更新索引路径。
2. 合并时保留信息更完整的案例编号，另一个案例按“移除案例”处理，并在保留案例的正文中注明合并来源。
3. 不要把多个天气类型差异很大的案例塞进同一个目录；宁可拆成多个编号。
